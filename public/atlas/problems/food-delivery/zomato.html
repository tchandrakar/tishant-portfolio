Design Zomato (restaurant discovery + delivery)

Hard food-delivery search maps-location

1. Problem statement

Build Zomato: a discovery-first food platform that started as restaurant reviews and expanded into delivery, dine-out, and Hyperpure (B2B supplies). The hard problem isn't just delivery — it's a search engine over 1.5 M restaurants ranked by location, cuisine, ratings, photos, ads, and Zomato Gold benefits, then a transactional flow on top of it.

2. Functional requirements

  • Search and browse: cuisine, dish, locality, price, rating, "open now", offers.
  • Restaurant page: photos, menu, reviews, distance/ETA, dine-out booking.
  • Order: cart, checkout, payments (UPI/cards/Zomato Money), delivery tracking.
  • User-generated content: reviews, photos, follows, helpful votes.
  • Ads: sponsored placements, Gold subscription, restaurant analytics dashboard.

3. Non-functional requirements

  • Latency: search p99 < 250 ms; home feed personalised < 400 ms.
  • Scale: 80 M MAU, 1.5 M restaurants, 5 M orders/day, 200 M reviews/photos.
  • Availability: 99.95% search, 99.99% checkout.
  • Freshness: menu/price changes visible < 60 s; new review searchable < 5 min.

4. Capacity estimation

  • QPS: search 30 K peak; feed 25 K peak; orders ~350 peak.
  • Storage: 1.5 M restaurants × 5 KB metadata + 200 M photos × 250 KB ≈ 50 TB photos.
  • Reviews: 200 M × 1 KB = 200 GB; embeddings ~ 200 M × 256 floats = 200 GB.
  • Search index: ~30 GB primary, replicated 3x.
  • CDN egress for photos: ~5 Gbps peak.

5. API design

GET   /v2/search?q=biryani&lat=&lng=&sort=rating  → list[restaurant]
GET   /v2/restaurants/{id}                          → details + menu
POST  /v2/reviews     {restId, rating, text, mediaIds[]} → reviewId
POST  /v2/orders      {cartId, addrId, payMode}     → orderId
GET   /v2/feed?lat=&lng=                         → personalised cards
POST  /v2/ads/impression  {slotId, restId}          → ack

6. High-level architecture

[App/Web] → [CDN (Cloudflare)] → [API GW] ├─→ [Search Svc] ← [Elasticsearch (geo+text)] ← [Indexer] ← Kafka(restaurant.changes) ├─→ [Feed/Reco Svc] ← [Feature Store] ← [Spark/Flink] ├─→ [Restaurant Svc] → [MySQL (sharded)] ─┐ ├─→ [Order Svc] → [Postgres] │ Debezium → Kafka ├─→ [Review Svc] → [Cassandra] │ └─→ [Media Svc] → [S3 + Imgix] │ ▼ [Data Lake / Snowflake]

CDC via Debezium pumps every source-of-truth change into Kafka; the search indexer consumes restaurant + review streams to update Elasticsearch in near-real-time.

7. Data model & storage

7.1 Restaurant (MySQL, shard by city_id)

CREATE TABLE restaurants (
  rest_id   bigint PRIMARY KEY,
  name      varchar(200), city_id int,
  lat decimal(9,6), lng decimal(9,6),
  cuisines  json, avg_cost int, rating_avg decimal(3,2),
  is_delivery tinyint, is_dineout tinyint,
  updated_at timestamp
);

7.2 Reviews (Cassandra)

Partition rest_id, clustering created_at DESC — read-heavy "latest reviews" pattern.

7.3 Search index (Elasticsearch)

Doc per restaurant with geo_point, dense vector for semantic dish search, BM25 fields for name/menu/reviews. Cuisines and tags as keyword filters.

8. Detailed component design

8.1 Search service

Hybrid retrieval: ES BM25 + ANN over dish embeddings (HNSW). Geo-filter (geo_distance ≤ 7 km) is the first pass; ranker fuses BM25 score, distance, rating, "open now", ad bid, personalisation. Final re-rank with LightGBM.

8.2 Feed personalisation

Two-tower model: user tower (history, time-of-day, locality) × restaurant tower (cuisine, price, rating). Candidates from ANN + business rules; ranked by xGBoost CTR/conversion blend.

8.3 Ads / sponsored slots

Real-time auction per slot — second-price among bids constrained by relevance threshold. Pacing service caps daily spend per advertiser using token buckets.

8.4 Photos pipeline

Uploaded via signed S3 URLs; Lambda triggers thumbnailing (Imgix) + NSFW + dish classification; metadata to Cassandra.

9. Scaling & bottlenecks

  • Hot search ("biryani Bengaluru"): Redis edge cache keyed by (geohash5, query), TTL 30 s.
  • Index lag: dual-write tier — synchronous primary + async ES; rely on CDC for replay.
  • Photo egress: CDN with origin shielding; pre-resize on upload.
  • Review write storm post-cricket-match: Cassandra LWT only on dedup of (user, restaurant); rest is plain INSERT.

12. Talking points for the interviewer

  • "I'd separate read-paths (search/feed/ads) from write-paths (orders/reviews) and let Kafka be the join."
  • "Hardest part is ranking quality at 7-km radius density in metros — I'd invest in a learned ranker over hand-tuned scores."
  • "For ad load I'd budget impressions per session to protect organic relevance."

Every technology choice in this design exists for a concrete reason.
