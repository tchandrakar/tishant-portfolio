Design Real-time Order Tracking (driver location → customer in <2s)

Hard food-delivery maps-location messaging

1. Problem statement

Build the live-tracking subsystem that takes a courier's GPS pings and projects them onto every active customer's order screen with end-to-end latency under 2 s. Also surfaces order state transitions (placed → accepted → preparing → picked → delivered), recomputed ETA, and notifications. Must work on flaky mobile networks and scale to millions of concurrent open orders during peak hours.

2. Functional requirements

  • Courier app pushes location at 1 Hz with status (en-route to pickup, with order, dropped).
  • Customer app subscribes to its order's location channel and receives updates < 2 s after the ping.
  • Show order state machine transitions live; show ETA refreshed every 10 s.
  • Allow customer ↔ courier voice/chat (masked numbers) tied to the order.
  • Replay last 5 minutes for late-joining clients.

3. Non-functional requirements

  • Latency: ping → customer screen p99 < 2 s.
  • Scale: 5 M concurrent open orders peak; 600 K courier pings/s.
  • Availability: 99.99% — degraded mode allows polling at 5 s if WS layer fails.
  • Bandwidth: bound mobile traffic to ~10 KB/min/customer on cellular.

4. Capacity estimation

  • Pings: 600 K/s × 200 B = 120 MB/s ingress.
  • Fanout: typical 1 customer + 1 ops dashboard subscriber per order ≈ 1.2x amplification → 720 K msg/s egress.
  • WS connections: 5 M concurrent → ~50 GB RAM across gateway fleet (10 KB/conn).
  • Replay buffer: 5 M orders × 300 pings × 200 B ≈ 300 GB in Redis (sharded).
  • Push notifications: ~50 M/day at peak (state transitions + ETA breaches).

5. API design

WS  /v1/track?orderId=&token=          → {seq, courierLat, courierLng, etaSec, state}
GET /v1/track/{orderId}/snapshot       → last known state (REST fallback)
POST /v1/courier/ping                  {courierId, lat, lng, accuracy, ts, status}
POST /v1/orders/{id}/state             {state}      (internal, from Order Svc)

6. High-level architecture

[Courier]→[Geo Ingest API]→Kafka(courier.pings)→[Flink job] │ ┌───────────────────────┤ ↓ ↓ [Active-Order Joiner] [Geo index Redis] (joins ping → orderId) ↓ Kafka(order.tracking.{shard}) ↓ [WS Gateway fleet (Go/Netty)]──→ [Customer] (sticky by orderId) ↓ [Replay buffer: Redis Streams] [Order Svc]───→ Kafka(order.events) ──→ [WS Gateway] (state transitions) [FCM/APNs] ←── [Notif Svc] ←── [Flink ETA-breach detector]

The crucial join is "which order is this courier on right now?" — done in Flink with keyed state on courier_id, fed by assignment events. Output topic is sharded by orderId hash so a WS gateway only consumes the partitions for connections it owns.

7. Data model & storage

7.1 Tracking message (protobuf)

message Track {
  fixed64 order_id = 1;
  sfixed32 lat_e7  = 2;   // 4 bytes
  sfixed32 lng_e7  = 3;
  uint32   eta_sec = 4;
  uint32   seq     = 5;
  State    state   = 6;
}

~25 B on the wire vs ~200 B for JSON.

7.2 Replay buffer (Redis Streams)

XADD track:{orderId} MAXLEN ~ 600 ... — ~5 minutes at 1 Hz. Customer reconnect sends last-seq; gateway replays from there.

7.3 Cold archive (Cassandra/S3)

Trails downsampled to 1 ping / 5 s for compliance + ML training; partitioned by courier_id+date.

8. Detailed component design

8.1 WS gateway

Go service, single process handles ~100 K WS conns, sticky by orderId via consistent hashing in the LB. Each gateway pod consumes a slice of order.tracking.* partitions matching its connection set; rebalances on scale events.

8.2 Ping → order join (Flink)

Keyed by courier_id; state holds current assignment(s). Inputs: courier.pings, assign.events. Output: per-order Track messages. p99 processing time < 100 ms.

8.3 Bandwidth optimisation

Send delta-encoded protobuf: only changed fields. Throttle to max 1 update / 3 s on cellular when courier is > 1 km away. Polyline-snap to road graph client-side for smooth animation.

8.4 Reconnect & degraded modes

On WS drop, exponential backoff up to 30 s; if 3 attempts fail, fall back to GET /snapshot every 5 s. Server stores last-seq in JWT-style resume token so replay is stateless on gateway.

8.5 Notifications

Flink job watches ETA breaches (current ETA > promise+2 min), state transitions, and emits push messages via FCM/APNs through a notification service.

9. Scaling & bottlenecks

  • Connection storms (post-app-update): rate-limit re-handshake at LB; spread reconnects via jitter.
  • Hot orderId partition: not really a thing here — orders are naturally well-distributed.
  • Cellular dead zones: client buffers last-known and animates; server keeps Redis stream alive.
  • WS gateway memory: shed idle connections after 60 s no activity in delivered state.
  • Cross-region: co-locate gateway with the order's origin region; rare cross-region order goes via global LB.

12. Talking points for the interviewer

  • "I treat ping ingestion and customer fanout as separate concerns connected by Kafka — gateways scale independently of solvers."
  • "The hardest part isn't the happy path; it's reconnect semantics. A stateless resume token keyed by last-seq makes failovers invisible."
  • "On flaky cellular, I prefer a slightly lower update rate over jittery UX; client-side smoothing buys back perceived freshness."
  • "For privacy, courier locations are masked to ±100 m once the order is delivered; raw trails are accessible only to ops with audit."
