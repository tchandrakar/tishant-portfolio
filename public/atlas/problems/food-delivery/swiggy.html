Design Swiggy (food delivery in India: dispatch + routing)

Hard food-delivery ride-sharing maps-location

1. Problem statement

Build Swiggy: a 3-sided marketplace connecting hungry customers, restaurant partners, and a fleet of delivery executives ("DEs") across 500+ Indian cities. The system must surface nearby restaurants, accept orders, route them to the kitchen, dispatch a DE within 60s of food being ready, and stream live ETA to the customer. India-specific constraints dominate: dense pin codes, dark alleys without GPS, 2G fallback, COD payments, monsoon outages, and a peak dinner rush at 8 PM IST that spikes 6x baseline.

2. Functional requirements

  • Discover restaurants within a delivery radius (typ. 5 km), filter by cuisine, ETA, rating.
  • Place order with item-level customisations, apply coupons, pay via UPI/card/COD.
  • Restaurant partner accepts/rejects within 90s; kitchen prep timer starts.
  • Dispatch engine assigns the best DE; courier tracks pickup → drop with live map.
  • Real-time ETA with re-estimation every 10s based on traffic + DE location.
  • Ratings, refunds, in-app chat between customer ↔ DE.

3. Non-functional requirements

  • Latency: home feed p99 < 400 ms; ETA push < 2 s end-to-end.
  • Scale: 50 M MAU, 6 M orders/day, 500 K concurrent DEs, 250 K active restaurants.
  • Availability: 99.95% for order path; dispatch must degrade gracefully on map outages.
  • Consistency: strong for payment, eventual for feed/listings.

4. Capacity estimation

  • QPS: 6 M orders/day → 70 avg, 420 peak (dinner). Browse traffic ~50x → 21 K peak QPS.
  • Storage: order row ~2 KB → 12 GB/day; 5 yr retention ~22 TB. Geo-pings 1/sec/DE → 500 K ev/s.
  • Bandwidth: live tracking 200 B/ping × 500 K DEs = 100 MB/s outbound to customers (fanout 1:1).
  • Cache: top-1000 restaurants per city in Redis ~5 GB.

6. High-level architecture

[App] → [Edge LB] → [API GW] ─┬─→ [Feed Svc] ← [ES + Redis] ← [Restaurant Svc] ├─→ [Cart Svc] ├─→ [Order Svc] → [Postgres (orders)] → [Kafka: order.events] ├─→ [Payment Svc] → [PSP: Razorpay/UPI] └─→ [Track Svc] ← WS gateway [Kafka] → [Dispatch Engine] → [Assignment Store: Redis] ↓ [Routing/ETA Svc] ← [OSRM/Google Maps] ↑ [DE App] → [Geo Ingest] → [Kafka: de.pings] → [Geohash Index in Redis]

Order is the source of truth in Postgres; events fan out via Kafka to dispatch, notifications, analytics. DE pings flow into a geohash-bucketed Redis index used for nearest-DE lookup.

7. Data model & storage

7.1 Orders (Postgres, sharded by city_id)

CREATE TABLE orders (
  order_id   bigint PRIMARY KEY,
  cust_id    bigint, rest_id bigint, de_id bigint,
  city_id    int, state text, total_paise int,
  created_at timestamptz, eta_at timestamptz
);
CREATE INDEX ix_orders_cust ON orders(cust_id, created_at DESC);

7.2 DE locations (Redis GEO + Cassandra archive)

Live state in Redis GEOADD per city geohash bucket; durable trail in Cassandra (partition: de_id+date) for payouts and disputes.

7.3 Why Postgres for orders?

ACID is non-negotiable for money + inventory holds. Per-city sharding keeps a hot Bengaluru shard isolated from Patna. See DDIA Ch. 6.

8. Detailed component design

8.1 Dispatch engine

Every 5s, batched matching: take pending orders in a city, candidate DEs within 2 km via Redis GEOSEARCH, score by dist + load + acceptanceRate − idleTime. Solved as a min-cost bipartite matching (Hungarian, capped at 200 orders × 500 DEs per tick). Falls back to greedy if >500 ms.

8.2 ETA service

Two-stage: OSRM gives base travel time on Indian OSM graph; an XGBoost model adds a residual using time-of-day, weather, restaurant prep variance. p50 error target: ±3 min.

8.3 Live tracking

WebSocket gateway with sticky sessions; DE app pushes 1 ping/sec, server publishes to order:{id} channel; customer sub fans out via Redis pub/sub. On drop, SSE fallback.

9. Scaling & bottlenecks

  • Dinner spike: pre-warm 2x dispatch workers at 7:30 PM via cron-driven HPA.
  • Hot restaurant: token-bucket queue per rest_id protects POS integrations.
  • Geo hotspot (Koramangala): sub-shard the geohash by 1-level finer grid at runtime.
  • 2G/flaky network: protobuf payloads, last-known-position client cache, exponential reconnect.

Every technology choice in this design exists for a concrete reason.
