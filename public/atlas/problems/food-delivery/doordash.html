Design DoorDash (3-sided marketplace at US scale)

Hard food-delivery ride-sharing maps-location

1. Problem statement

Build DoorDash: a 3-sided marketplace serving consumers, merchants (restaurants, grocers, convenience), and Dashers (independent contractors) across the US, Canada, Australia, and Japan. The platform must accept orders, hand them to merchants, dispatch a Dasher with strong batching/efficiency, and track delivery in real-time. Peak load is Super Bowl Sunday at ~10 M orders/day with a 90-minute window taking the bulk. Subscriptions (DashPass) and ad inventory add monetisation surfaces.

2. Functional requirements

  • Consumer: browse merchants, place orders, schedule, tip, track, rate.
  • Merchant: receive orders via tablet/POS, manage menu, mark item ready, throttle inflow.
  • Dasher: go online, accept offers, multi-pickup batches, complete drop, get paid.
  • Marketplace: surge pricing, promised ETAs, refund flows, ad placements, DashPass benefits.

3. Non-functional requirements

  • Latency: feed p99 < 300 ms; offer-to-Dasher push < 1 s.
  • Scale: peak 10 M orders/day, 600 K active Dashers, 1 M merchants, 30 M MAU.
  • Availability: 99.99% order path; degrade ranking before degrading checkout.
  • Correctness: exactly-once payment capture; idempotent retries.

4. Capacity estimation

  • QPS: 10 M/day → ~115 avg, ~1 K peak orders/s during Super Bowl halftime.
  • Browse:order ratio ~80:1 → 80 K peak browse QPS.
  • Storage: order ~3 KB → 30 GB/day; 7-yr retention ~80 TB hot + cold tier.
  • Dasher pings: 600 K × 1 Hz = 600 K events/s, ~120 MB/s into Kafka.
  • Search index (Elasticsearch): ~1 M merchants × 50 menu items ≈ 50 M docs.

6. High-level architecture

[Mobile] → [Cloudflare] → [Envoy/API GW] ├─→ [Feed Svc] ← [ES] ← [Catalog Svc] ← [CockroachDB] ├─→ [Checkout Svc] → [Order Svc] → [Postgres-shards] → [Kafka order.events] ├─→ [Pricing/Surge] ← [Feature Store (Feast)] └─→ [Tracking Gateway (WS)] [Kafka] → [Deep Red Dispatcher] → [Assignment DB (Cassandra)] ↓ ↑ [Routing/ETA] ← [OSRM + ML] ← [Telemetry Lake (S3 + Snowflake)] [Dasher app] → [Geo Ingest] → [Kafka dasher.pings] → [Redis Geo]

The Order Service owns the state machine; everything else subscribes via Kafka. Dispatch ("Deep Red") consumes both order and Dasher streams to produce assignments.

7. Data model & storage

7.1 Orders (Postgres, hash-sharded by order_id; Vitess-style)

CREATE TABLE orders (
  order_id   uuid PRIMARY KEY,
  consumer_id uuid, store_id uuid, dasher_id uuid,
  state      text, subtotal_cents int, tip_cents int,
  promise_at timestamptz, created_at timestamptz
);

7.2 Assignments (Cassandra)

Wide partition by region+hour for write-heavy fan-in from dispatch; trailing scans for ops.

7.3 Why this mix?

Postgres for OLTP money; Cassandra for high-write trails (DDIA Ch. 5 leaderless); CockroachDB for catalog needing multi-region reads with strong consistency.

8. Detailed component design

8.1 Deep Red dispatcher

Solves a Vehicle Routing Problem with Time Windows (VRPTW) per region every 2–5 s. Inputs: open offers, Dasher locations + capabilities (catering, alcohol), promise times. Builds candidate routes (single, batched 2-3 deliveries), scores by Dasher pay + customer ETA + efficiency. Uses LP relaxation + local search; falls back to greedy nearest-neighbor if SLA breached.

8.2 ETA / promise model

Promised ETA at checkout = restaurant prep p80 + drive time + buffer. Continuously re-estimated post-order. Model is a quantile gradient-boosted tree retrained nightly on 90-day pings.

8.3 Surge / pricing

Per-hex (Uber H3 res-8) supply-demand ratio; multiplier published every 60 s to feed and offer engine.

8.4 Tracking

WS gateway sharded by order_id; client receives differential updates ~every 3 s; bandwidth optimised with delta-encoded protobuf.

9. Scaling & bottlenecks

  • Super Bowl spike: capacity-plan 3x; pre-scale Kafka, K8s nodes, ES replicas; brown-out non-essentials (recos).
  • Hot store (Chick-fil-A 12 PM): per-store rate limiter; queue-and-stagger menu reads.
  • Hot region: H3-cell sub-sharding for the dispatcher; multi-region active-active with Cockroach.
  • Hot order_id keys: use UUIDv7 (timestamp-prefixed) for index locality; periodic VACUUM tuned per shard.

12. Talking points for the interviewer

  • "I separate the promise ETA (set at checkout, customer-facing SLA) from the predicted ETA (live, ops-facing) — they have different cost functions."
  • "Batching is the single biggest lever on unit economics; I'd quantify the trade-off vs customer freshness via offline simulator."
  • "For multi-region, I'd colocate consumers with a primary region but make catalog and ad inventory global via Cockroach."
