Design Restaurant Search & Discovery (location + cuisine + ranking)

Hard food-delivery search maps-location

1. Problem statement

Design the search and discovery layer for a food-delivery app: given a user's location and intent (free-text query, cuisine filter, "open now", price band), return a ranked list of restaurants that can deliver to them within an SLA. The system must handle text relevance, geospatial filtering, business-state filters (open, accepting orders, surge), and a learned ranker — all in < 250 ms.

2. Functional requirements

  • Free-text search across restaurant name, cuisine, dish names.
  • Geo radius filter (default 7 km, capped by ETA < 60 min).
  • Filters: cuisine, dietary (veg/halal), price, rating ≥ 4, deals, "fast delivery".
  • Sort: relevance, distance, rating, ETA.
  • Type-ahead autocomplete (cuisine, dish, restaurant suggestions).
  • Personalisation: re-rank by user history; boost subscribed perks.

3. Non-functional requirements

  • Latency: p99 < 250 ms; autocomplete p99 < 60 ms.
  • Scale: 1.5 M restaurants, 50 K peak search QPS, 300 M searches/day.
  • Freshness: open/closed flips visible in < 30 s; price/menu < 5 min.
  • Availability: 99.95% with graceful degradation (drop personalisation before relevance).

4. Capacity estimation

  • QPS: 50 K peak; 80% cache-hit on (geohash5, query, filters) = 10 K to backend.
  • Index size: 1.5 M restaurants × ~3 KB doc = 5 GB; with replicas 30 GB.
  • Embedding store: 1.5 M × 256 floats = 1.5 GB.
  • Click-log stream: 300 M searches/day × ~1 KB ≈ 300 GB/day → S3.

6. High-level architecture

[App] → [CDN edge cache (30s TTL)] → [API GW] ↓ [Search Svc] ├─→ [Redis: hot query cache] ├─→ [Elasticsearch (geo+text)] ├─→ [ANN: HNSW (FAISS/Vespa)] ├─→ [Feature Store: Redis + DynamoDB] └─→ [Ranker Svc (LightGBM, ONNX)] [Catalog DB (MySQL)] → Debezium → Kafka → [ES Indexer] [Order events] → Kafka → [Click logs / training pipeline] → [Spark] → S3

Two-stage retrieval: ES candidate generation + ANN semantic recall; merge → re-rank with learned model.

  "open_hours": [...], "accepting_orders": true,
  "embedding": [0.12, -0.04, ...256d]
}

7.2 Hot state (Redis)

Per-restaurant open/busy/surge flags updated by partner heartbeats; consulted as a post-filter to avoid re-indexing constantly.

7.3 Why ES + ANN?

ES handles geo + boolean filters with predictable performance; ANN handles "spicy chicken" → "chettinad" semantic recall. Combining both beats either alone.

8. Detailed component design

8.1 Retrieval

ES query: function_score with geo_distance filter, BM25 over name/cuisines/menu, gauss decay on distance. Returns top 200. ANN parallel call with query embedding (sentence-transformer, 256d) returns top 200. Union deduped to ~300 candidates.

8.2 Ranking

LightGBM model (~150 features): BM25, ANN sim, distance, ETA, rating, price match, user-cuisine affinity, time-of-day, recent CTR, deal flag, ads bid. Inference via ONNX Runtime; batched prediction for 300 docs in < 25 ms.

8.3 Autocomplete

Trie compiled per city, served by an in-memory Go service. Re-built nightly from query logs + restaurant/dish dictionaries; weighted by city + frequency.

8.4 Personalisation

User feature vector cached at session start; ranker reads from Redis (fallback: cohort-mean to keep latency).

9. Scaling & bottlenecks

  • Hot queries: "pizza", "biryani" — 30 s edge cache cuts ES load 5x.
  • Geohash skew: shard ES by geohash prefix not random; balance with virtual shards.
  • Open-flag freshness: push-based via Kafka into Redis, not re-index.
  • Ranker drift: shadow-deploy new model; compare NDCG@10 online.
  • Long tail of cities: hot-spare ES nodes for Mumbai/Delhi; cold-tier smaller cities on shared shard.

12. Talking points for the interviewer

  • "I'd lean on ES for the 90% case and add ANN only when query intent isn't lexical — measured by query class."
  • "Open/closed is the trickiest freshness problem; I solve it via a fast post-filter rather than a hot index."
  • "For ranking I'd start with hand-tuned weights, log labels for 4 weeks, then ship a learned ranker — never start with ML."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

Elasticsearch (geo+text)50 K peak search QPS over 1.5 M restaurants needs geo_distance + BM25 + boolean filters in one query at p99 < 250 ms; ES function_score with gauss decay on distance is mature, predictable, and supports the cuisine/price facet aggregations.Solr (older ANN story); Vespa (smaller team expertise); Postgres+PostGIS (joins on dense filters too slow at 50 K QPS).

LightGBM (ONNX Runtime)~150 features × 300 candidates ranked in <25 ms; LightGBM's leaf-wise growth + ONNX vectorised inference outperforms XGBoost and DNNs at this latency budget. Re-trains nightly on 300 M searches/day click-stream.XGBoost (slightly slower inference); DNN (10x latency); learning-to-rank (RankNet) (smaller off-the-shelf tooling); hand-tuned (15-25% worse conversion).
