Design TikTok For You feed (heavy ML-driven recommendation)

Very Hard social-network video-streaming recommendation

1. Problem statement

Design TikTok's For You Page (FYP): an infinite vertical feed of short videos personalized per user, where the candidate set is the entire global corpus (not just followed accounts). The system must turn every micro-interaction (watch-time, scroll-away, like, share, replay) into near-real-time signals that retrain ranking, while serving 1B+ DAU at p95 first-frame < 600 ms.

2. Functional requirements

  • Upload short video (≤3 min), client-side trim/effects.
  • FYP feed (algorithmic, global corpus). Following feed (graph-based).
  • Engagement events: watch_time_ms, completion, like, comment, share, follow, "not interested".
  • Search by hashtag, sound, creator. Discover trending sounds.

3. Non-functional requirements

  • Scale: 1.5B MAU, 1B DAU, ~50M uploads/day, ~100B video plays/day, ~500B engagement events/day.
  • Latency: first-frame p95 < 600 ms; ranking budget 100 ms p99 server-side; signal-to-model fresh in <5 min.
  • Availability: 99.95% on read; ranking degrades to popularity if model serving is down.

4. Capacity estimation

  • Uploads: 50M/day = 580/sec avg; avg 5 MB → 250 TB/day raw, ~500 TB/day after ABR.
  • Plays: 100B/day = ~1.2M plays/sec avg, ~5M/sec peak. Each play emits ~5 engagement events → ~25M events/sec peak.
  • Candidate corpus: ~5B videos eligible at any time; ANN index per region must scale to billions.
  • Inference: 1.2M plays/sec × 500 candidates ranked = 600M scoring ops/sec → distributed GPU/TPU farm with batched inference at ~200 QPS/instance.
  • CDN egress: 100B × 4 MB ≈ 400 PB/day egress; cache hit ratio target > 95%.

6. High-level architecture

[Mobile] → [Edge POP/CDN] ──────────► [Video CDN tiers] │ ▲ │ events (batched, every 2s) │ ABR segments ▼ │ [API GW] ──► [Event Collector] ──► [Kafka events] ──► [Flink: feature updater] ──► [Online Feature Store (Redis/Cassandra)] └─► [Realtime Counters] └─► [Training Sample Builder] ──► [Hourly model retrain → Model Registry] │ [FYP Svc] ──► [Candidate Generator (multi-source)] ──► [Ranker (DNN, GPU)] ──► [Diversity reranker] ──► [Pre-fetcher] ──► response ├─ ANN: user-embedding × video-embedding (FAISS/ScaNN) ├─ Trending pool, Locale pool, Creator-affinity pool └─ "Cold-start fresh content" pool (50%-injected for exploration)

Walk-through: client batches engagement events. Flink jobs update online features (recent watch ratio, completion ratio, locale, time-of-day). At feed open, FYP Svc runs candidate generation from ~6 retrieval sources (each returning ~200 ids), unions to ~1000 candidates, sends them to a DNN ranker that emits a single score blending P(complete), P(like), P(share), P(comment), P(follow). Diversity reranker (MMR) deduplicates by sound/creator/topic. Top-N returned with prefetch URLs for the next 2 videos.

7. Data model & storage

7.1 Video metadata (sharded by video_id)

CREATE TABLE videos (
  video_id   bigint PRIMARY KEY,
  creator_id bigint, sound_id bigint, hashtags array,
  duration_ms int, lang text, region text, status enum,
  created_at timestamp
);

7.2 Engagement event log (Kafka → ClickHouse for analytics, Redis for online)

event {
  user_id, video_id, ts_ms,
  type, watched_ms, completion_ratio, device, network
}

7.3 Embeddings & ANN

Two-tower model. User tower output (256-d) cached per user, refreshed every few minutes. Video tower output stored in a sharded ANN index (FAISS HNSW or ScaNN). Index rebuilt incrementally as new videos arrive; videos enter the index within 60 s of upload.

7.4 Feature store

Online: Redis (low-latency point lookup, <2 ms p99). Offline: Parquet on HDFS for training. Same feature definitions via a feature registry to prevent train-serve skew.

8. Detailed component design

8.1 Multi-source candidate generation

No single retrieval covers exploration + exploitation. Sources: ANN-by-embedding (personal taste), Trending (popularity-weighted recent), Locale/Language pool, Creator-affinity, Hashtag-co-occurrence, Fresh-uploads (cold-start exploration). Each contributes a quota; the union is deduped before ranking.

8.2 Ranker

Multi-task DNN: shared bottom + task-specific heads for completion, like, share, comment, long-view. Final score = weighted sum tuned per the business objective (engagement, retention). Inference batched at 64–128 candidates/request on GPU; p99 budget 30 ms ranking + 20 ms feature fetch.

8.3 Cold-start: videos & users

New videos: forced exposure to small "exploration slots" (~5% of feed) to gather initial signal. New users: locale + popularity + onboarding interests bootstrap until ~20 watched videos generate a stable user embedding.

8.4 Real-time signal loop

Watch-time logged every 2 s while user watches. Flink updates the user's recent-watch ratio in Redis within seconds, so the next request reflects current taste — that's why TikTok feels "scary good" after a few swipes.

8.5 Pre-fetch & CDN

Server returns N=3 next videos with signed URLs; client pre-fetches segment 0 of N+1 and N+2 over WiFi to hide latency. CDN keys are content-hashed for cacheability; popular video is shielded with a small origin protect-tier.

9. Scaling & bottlenecks

  • ANN index freshness: shard by video_id range; incremental insert HNSW; full rebuild offline weekly.
  • Hot video: CDN edge cache; same-DC origin shield; pre-warm via predicted virality score.
  • Train-serve skew: single feature definition library, replayable from log.
  • Filter bubble: exploration quota and "not interested" hard-negative signal kept in feature store.
  • Ranker p99 spikes: graceful fallback to last-known-good model, then to popularity baseline.

12. Talking points for the interviewer

  • "FYP isn't a feed problem, it's a recommendation problem; the architecture is dominated by the retrieval-then-ranking pattern, not the social graph."
  • "The differentiator is real-time features — most systems retrain daily; here we update user features in seconds and that's the magic."
  • "Cold-start gets its own quota in candidate generation; without that, new creators starve and the platform stagnates."
  • "I'd discuss the train-serve skew problem because it's the operational footgun that bites every ML team."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Apache Flink (stateful streaming)Watch-time logged every 2 s must update user features in <5 min for the "scary good after a few swipes" effect; Flink's stateful operators with exactly-once semantics handle 25M events/sec peak join (impression × engagement) for training-sample assembly.Spark Streaming (micro-batch latency too coarse for 5 min freshness SLO), Kafka Streams (lacks sophisticated stateful join primitives at this scale), Storm (no exactly-once).
FAISS / ScaNN HNSW (ANN candidate gen)5B-video corpus needs sub-ms nearest-neighbor lookup of user-embedding × video-embedding to fit the 100 ms p99 ranking budget; HNSW gives us tunable recall vs latency and supports incremental insert so new uploads enter the index within 60 s.Exact kNN (impossible at 5B videos), Annoy (lower recall), Elasticsearch dense vector (slower per-query and higher RAM at this scale).
Multi-task DNN ranker (GPU/TPU)600M scoring ops/sec requires batched accelerator inference; shared-bottom DNN with task heads for completion, like, share, comment, long-view lets the policy layer rebalance objectives without retraining for every product change.GBDT (cannot share representations across tasks), per-task models (5× the inference cost), CPU inference (would need ~30K instances vs 2,300 GPU).
Two-tower model (user/video embeddings)Decoupled towers let video embeddings be precomputed at upload and stored in the ANN index, while user embeddings refresh every few minutes from the real-time signal loop — necessary because joint encoding billions of (user, video) pairs is computationally infeasible.Joint cross-encoder (cannot precompute, blows latency), bag-of-features only (loses semantic similarity), single tower (cannot separate retrieval from ranking).
Redis (online feature store)Feature lookup at <2 ms p99 for 1.2M plays/sec is the only way to fit feature fetch + ranking + diversity in the 100 ms budget; recent watch ratio, completion ratio, locale must be point-lookup-able.Cassandra (single-digit-ms point read too slow at this rate), DynamoDB (cost prohibitive), DB direct (latency blowup).
Cassandra (durable feature backup + counters)Online Redis is RAM-bound; Cassandra holds the durable copy and serves cold features when Redis evicts, plus realtime counters that need write throughput at 25M events/sec without single-shard hot keys.Postgres (cannot shard at this rate), Redis-only (loses durability on cluster failure), HBase (operational complexity vs Cassandra OSS).
ClickHouse (engagement analytics)500B engagement events/day need columnar OLAP for ad-hoc queries (creator dashboards, cohort analysis, A/B significance); ClickHouse aggregates billion-row queries in seconds where row-store DBs would crawl.BigQuery (vendor lock + per-query cost), Druid (similar but weaker SQL), Postgres analytics (cannot scale beyond TB).
Apache Kafka (event log + retraining source)Replayable log is the source of truth for hourly model retrain — can replay any time window to rebuild training samples after a feature bug; consumer-lag drives Flink autoscaling and serves as the kill-switch for retraining when upstream is broken.Kinesis (vendor lock + per-day retention limits), RabbitMQ (no replay), in-process queues (lose all of the above).
Parquet on HDFS (offline feature store)Same feature definitions via a feature registry must be readable in batch for nightly retraining; columnar Parquet projects only the features the trainer needs from 5 TB/day of training data, cutting IO 10×.Avro (row-wise, slower scan), JSON (uncompressed bloat), per-feature CSV (no schema evolution).
CDN (multi-tier with origin shield)400 PB/day egress with >95% cache hit ratio is the only way to fit Reels economics; per-region origin shields collapse misses for the same hot video into a single origin pull, and pre-warming via predicted virality avoids the cold-cache spike.Single-tier CDN (cache miss storm on viral video), origin-only (impossible at 400 PB/day), peer-to-peer (mobile network unfriendly).
HLS ABR (adaptive video)First-frame p95 < 600 ms across cellular and WiFi requires the client to start with a low-bitrate ladder and adapt; segment-0 prefetch of N+1 and N+2 reels works only with chunked segmented streams.Progressive MP4 (no adaptation), DASH (interchangeable but HLS has stronger iOS support), WebRTC (overkill for one-way playback).
MMR diversity rerankerFilter-bubble kills retention; greedy λ·score − (1−λ)·sim(picked) over sound/creator/topic similarity is auditable and predictable in the 100 ms budget where heavier global optimization isn't feasible.Determinantal Point Processes (more principled but expensive), no diversity (filter bubble), client-side rerank (cannot use global similarity).
Feature registry (DRY transforms)Train-serve skew is "the silent killer" — the same transform code must run online (Flink) and offline (Spark batch); a registry-versioned library prevents the divergence that destroys model quality silently in prod.Two implementations (skew bugs guaranteed), runtime feature-spec interpretation (slower), feature-as-DB-view (cannot share Flink stateful logic).
Exploration quota (fresh-uploads pool)~5% of feed slots forced to new videos solves the cold-start problem (new creator gets initial signal) and keeps the corpus refreshing — without it, the platform stagnates on a small evergreen set and creator churn rises.No exploration (creator starvation), pure-Thompson-sampling (computationally expensive at 5B corpus), random insert (no popularity prior, hurts UX).
