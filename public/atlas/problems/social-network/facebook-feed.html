Design Facebook News Feed (push/pull/hybrid fanout)

Hard social-network recommendation

1. Problem statement

Build the News Feed for a Facebook-scale social network: when a user opens the app, return a personalized, ranked list of posts authored by friends, pages, and groups they follow. The system must ingest a sustained ~50K writes/sec of new posts and serve ~1M feed reads/sec at p99 < 200 ms while feeling fresh — new posts from close friends should appear within seconds.

2. Functional requirements

  • Compose post (text, image, video, link) with privacy scope (public, friends, custom).
  • Render personalized feed on app open / pull-to-refresh / infinite scroll.
  • Engagement actions: like, comment, share, save, hide. Each is a feedback signal.
  • Edge cases: muted authors, blocked users, privacy filtering at read time.

3. Non-functional requirements

  • Scale: 3B MAU, 2B DAU, ~150 friends median, ~600 p95.
  • Latency: feed open p99 < 200 ms, post-to-visible p95 < 5 s for close friends.
  • Availability: 99.99% on read path; degrade gracefully to cached feed on partial outage.
  • Consistency: read-your-writes for the author; eventual for everyone else.

4. Capacity estimation

  • Writes: 2B DAU * 2 posts/day / 86400 s ≈ 46K posts/sec sustained, ~150K/sec peak.
  • Reads: 2B DAU * 10 feed opens/day = 20B/day ≈ 230K/sec avg, ~1M/sec peak.
  • Fanout amplification: avg 150 followers × 46K writes = 7M timeline inserts/sec for pure push. Celebrities with 50M followers turn one post into 50M inserts — push alone is infeasible.
  • Storage: 2B posts/day × 1 KB metadata = 2 TB/day metadata; media goes to blob store (~500 PB lifetime).
  • Cache: hot timeline working set ~500M users × 800 post-IDs × 16 B = ~6 TB Redis.

5. API design

POST  /v1/posts                  {authorId, body, mediaIds, privacy} → postId
GET   /v1/feed?cursor=...&limit=50                               → [postId, score, reason]
POST  /v1/feed/engagement        {postId, action: like|hide|...}
GET   /v1/posts/{id}                                             → hydrated post
WS    /v1/feed/live              push: new-post-available banner

6. High-level architecture

[Client] → [Edge LB] → [API GW] → [Post Svc] → [Post DB (sharded MySQL/RocksDB)] │ │ ▼ ▼ [Kafka post-events] → [Fanout Workers] │ push for normal users ▼ [Timeline Cache (Redis ZSET)] ▲ pull for celeb posts [Feed Svc] ← [Ranker (GBDT/DNN)] ← [Feature Store] ← [Engagement Stream]

Walk-through: a write hits Post Svc, persists, emits to Kafka. Fanout workers classify the author: normal (≤10K followers) → push post-IDs into each follower's Redis ZSET timeline. Celebrity (>10K) → skip push; the Feed Svc pulls celeb posts at read time and merges. On feed open, Feed Svc reads the user's pushed timeline, merges followed-celebrity pulls, runs the ranker, and returns top-N.

7. Data model & storage

7.1 Posts (sharded by authorId)

CREATE TABLE posts (
  post_id   bigint PRIMARY KEY,   -- Snowflake, time-sortable
  author_id bigint, body text, media_ids json,
  privacy   tinyint, created_at timestamp
) PARTITION BY HASH(author_id);

7.2 Timeline cache (Redis ZSET per user)

ZADD timeline:{userId} {score=created_at} {postId}
ZREVRANGEBYSCORE timeline:{userId} +inf -inf LIMIT 0 1000

Cap at 1000 entries, TTL 7 days. Cold users rebuild from Post DB on first open.

8. Detailed component design

8.1 Hybrid fanout

Threshold 10K followers separates push from pull. Push gives O(1) read but pays write amplification; pull is cheap on write but expensive on read. Hybrid bounds worst case: a user with 200 friends + follows 50 celebrities pulls 50 lists at read (small) and reads 1 ZSET for normal-friend posts.

8.2 Ranker

Two-stage: candidate generation pulls ~2000 posts from timeline + celeb-pull + sponsored. Ranking model (DNN with embeddings for user, author, post) scores each in <30 ms total via batched GPU inference. Features: affinity, recency decay, post-type weights, predicted P(like|comment|share). EdgeRank-style baseline: score = affinity × weight × decay.

8.3 Read path

Feed Svc: 1) fetch ZSET (1 Redis call), 2) parallel fetch celeb pull-lists (Redis MGET), 3) hydrate via Post Svc with batched cache, 4) rank, 5) apply privacy filter, 6) return cursor of (postId, rank-score).

9. Scaling & bottlenecks

  • Celebrity fanout: pull-on-read avoids 50M writes; cached celeb-list at edge.
  • Hot key Redis: shard timeline by userId murmur3, replicate hot keys.
  • Thundering herd on celeb post: request coalescing + soft TTL with stale-while-revalidate.
  • Backpressure: Kafka with consumer-lag SLO, autoscale fanout workers.

10. Trade-offs

12. Talking points for the interviewer

  • "I'd start with pure push, measure the celebrity tail, then introduce the pull path — premature hybrid is overengineering."
  • "The hardest part is the celebrity fanout problem; mitigation is the threshold-based hybrid plus pull-side caching."
  • "For ranking, I'd separate the candidate-gen and ranking stages so we can iterate on ML without touching infra."
  • "I'd discuss read-your-writes via session-pinned cache to make the author always see their own post immediately."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Sharded MySQL/RocksDB (Post DB)2 TB/day metadata across 2B posts/day needs hash-partitioning by author_id for write locality; MySQL gives us transactions for the privacy-scope mutation, RocksDB underneath delivers the LSM write throughput required at 46K writes/sec sustained.Cassandra (weaker secondary indexes for moderation queries), DynamoDB (vendor lock and cost at PB scale), single MySQL (cannot shard 500 PB lifetime).
Redis (Timeline ZSET cache)p99 sub-ms ZREVRANGEBYSCORE reads on a 1000-entry capped ZSET per user is exactly what the 200 ms feed-open SLO needs; the ~6 TB hot working set fits across a Redis cluster with murmur3 sharding.Memcached (no sorted-set primitive), Aerospike (overkill on durability we don't need), DynamoDB (per-op cost too high at 1M reads/sec).
Apache Kafka (post-event log)Decouples the synchronous post-write (must respond fast) from the 7M timeline-insert/sec fanout; consumer-lag SLO + replayability lets us autoscale fanout workers and recover from worker failure without losing posts.RabbitMQ (no durable log replay), AWS SQS (no ordering or partition affinity), direct gRPC to fanout (loses backpressure semantics).
Snowflake IDsTime-sortable 64-bit ID lets us use the ID itself as the ZSET score, so ZADD places posts in chronological order without a separate timestamp field; also supports cross-DC ID generation without coordination.UUIDv4 (not sortable, breaks ZSET semantics), DB auto-increment (single-point bottleneck at 46K writes/sec), ULID (acceptable but Snowflake is the social-media de-facto).
Fanout workers (custom service)Hybrid push-vs-pull at the 10K-follower threshold needs application logic that classifies authors per post; idempotent ZADD makes at-least-once Kafka delivery safe and bounds the celeb worst case from 50M down to a pull-side merge.Pure-push (50M inserts/celeb post is infeasible), pure-pull (every read does N MGETs even for normal users), Spark Streaming (batch granularity too coarse for 5 s SLO).
GBDT / DNN ranker with embeddings30 ms ranking budget across 2000 candidates demands batched GPU inference; a multi-task DNN scoring P(like|comment|share) blended via the EdgeRank-style score = affinity × weight × decay baseline gives us interpretability plus headroom.Logistic regression (plateaus on engagement gain), pure online learning (operationally fragile), per-request ANN-only (no calibration across objectives).
Feature store (online + offline)The ranker depends on user-author affinity, recency decay, and post-type weights — all features that must match between training and serving to avoid train-serve skew at 230K RPS; a registry with shared transforms is the only way to keep them aligned.Direct DB lookup at ranking time (blows the 30 ms budget), client-side feature compute (impossible across 3B users), no feature store (skew bugs become silent quality regressions).
WebSocket (live banner)WS /v1/feed/live push for "new post available" requires a long-lived connection per active user; HTTP polling at 2B DAU would generate billions of wasted requests/day, while WebSocket keeps the open socket cheap on the server side.HTTP long-polling (more bandwidth and connection churn), Server-Sent Events (less client support on older mobile), push notifications (wrong UX surface for an in-app banner).
Edge LB + API GW1M reads/sec peak across multiple regions needs TLS termination and routing at the edge POP closest to the user; the API GW layer enforces auth, rate-limits abusive clients, and emits the per-route metrics the 99.99% read-availability SLO depends on.Direct service exposure (no rate limiting), single-region LB (cross-ocean RTT blows 200 ms p99), service-mesh-only (lacks public-internet TLS+WAF features).
Blob store (S3-class) for media~500 PB lifetime media storage at 11-nines durability is impractical on attached disk; object storage with CDN-fronted reads keeps the post DB to just 1 KB metadata rows and offloads bandwidth from the app tier entirely.Local SAN (cannot scale to PB), HDFS (designed for batch, not low-latency reads), embedded blobs in MySQL (inflates row size, kills cache density).
Consistent hashing (murmur3) for Redis shardsHot-key replication for the top 0.01% of timeline keys requires a hash that distributes evenly and supports virtual nodes for cheap rebalancing when we add capacity; murmur3 is the standard choice with non-cryptographic speed.Modulo hashing (every resize remaps everything), MD5/SHA (slower without security benefit here), random sharding (loses key locality for replicas).
Stale-while-revalidate + request coalescingCeleb-post thundering herd is the predictable failure mode when one author has 50M followers; soft TTL with single-flight refresh prevents 50M cache-miss requests from stampeding the Post DB simultaneously.Hard TTL (causes stampedes), longer TTL (degrades freshness past the 5 s SLO), pessimistic per-key locks at the DB (worse tail latency).
