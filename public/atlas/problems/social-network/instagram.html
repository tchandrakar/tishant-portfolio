Design Instagram (feed + stories + reels)

Hard social-network video-streaming file-storage

1. Problem statement

Design an Instagram-class media-first social network: photo/video feed (permanent), Stories (24-hour ephemeral), and Reels (short vertical video with algorithmic discovery). The system must ingest ~100M media uploads/day, transcode and CDN-distribute globally, and serve personalized feeds at p99 < 200 ms to 2B MAU.

2. Functional requirements

  • Upload photo (≤10 MB) or video (≤100 MB, ≤60 s for reels, ≤90 s for posts).
  • Permanent feed posts; 24h-expiring Stories with seen-by tracking; Reels with looping playback.
  • Likes, comments (threaded one-deep), DMs (out of scope here), follow/follower model.
  • Discovery: Explore tab and Reels feed, both ML-ranked.

3. Non-functional requirements

  • Scale: 2B MAU, 1.2B DAU, 100M uploads/day, 500M Stories views/day, Reels 200B plays/day.
  • Latency: feed open p99 < 200 ms; first-frame Reels p95 < 800 ms; upload-to-visible p95 < 30 s after transcode.
  • Availability: 99.99% read; uploads must accept retries.
  • Durability: 11 nines for media (S3-class).

4. Capacity estimation

  • Uploads: 100M/day = 1160/sec avg, 5K/sec peak. At avg 3 MB → 300 TB/day ingest, ~110 PB/year (raw).
  • After transcoding (5 ABR ladders for video) ≈ 2× → 220 PB/year.
  • Feed reads: 1.2B DAU × 8 sessions × 30 posts = 290B impressions/day = 3.4M/sec avg, ~12M/sec peak.
  • Stories: 500M views/day × ~6 stories/session ≈ peak 100K/sec story fetches.
  • Reels CDN egress: 200B plays × 5 MB avg = 1 EB/day egress (ABR + edge cache halves origin pulls).

5. API design

POST  /v1/media/upload/init     {type, size, mime}             → uploadUrl, mediaId
PUT   {uploadUrl}                                              ← chunked direct-to-blob
POST  /v1/posts                 {mediaId, caption, location}   → postId
POST  /v1/stories               {mediaId, stickers}            → storyId  (TTL 24h)
GET   /v1/feed?cursor=...                                       → [post]
GET   /v1/stories/tray                                          → [user, latestStoryId, unseen]
GET   /v1/reels/feed?cursor=...                                 → [reel]
POST  /v1/reels/{id}/event      {watchedMs, completed}          → ok

6. High-level architecture

[Mobile] → [Edge POP / CDN] → [API GW] │ direct upload │ ▼ ▼ [Blob Store (S3-like)] → [Transcode Workers (FFmpeg, GPU)] → [HLS/DASH ABR] ↓ Kafka │ [Post/Story/Reel Svc] → [Sharded MySQL/RocksDB] ↓ Kafka post-events │ [Fanout Svc] → [Timeline Cache Redis] ▼ ↓ [Reel CDN edges] [Reels Ranker (DNN, 2-stage)] ← [Feature Store] ↓ [Engagement Stream] → [Counters Cassandra]

Walk-through: client uploads to a presigned URL directly to blob store (bypass app servers). A Kafka event triggers transcode workers; outputs are written back to blob and registered in metadata DB. Post/Story/Reel Svc emits create-events; Fanout Svc updates follower timelines (hybrid push/pull). Reels uses a separate ranker because the candidate set is global discovery, not follow-graph.

7. Data model & storage

7.1 Posts (sharded by user_id)

CREATE TABLE posts (
  post_id   bigint PRIMARY KEY,   -- Snowflake
  user_id   bigint, caption text, media_ids json,
  type      enum('photo','carousel','video','reel'),
  created_at timestamp
);

7.2 Stories (TTL via partition rotation)

-- Cassandra: TTL set on insert
CREATE TABLE stories (
  user_id bigint, story_id bigint, media_id text,
  created_at timestamp, PRIMARY KEY(user_id, created_at)
) WITH default_time_to_live = 86400;
-- Seen-by:
CREATE TABLE story_seen (story_id bigint, viewer_id bigint, seen_at timestamp,
  PRIMARY KEY(story_id, viewer_id)) WITH default_time_to_live = 86400;

7.3 Media blobs

Original in cold tier (S3 IA), HLS renditions in hot tier with CDN; immutable URLs keyed by content hash for cacheability.

8. Detailed component design

8.1 Upload + transcode pipeline

Resumable chunked upload (TUS protocol). On finalize, emit Kafka. Worker pool: photo (resize to 7 sizes, WebP+JPEG, EXIF strip) ~150 ms; video (5 ABR ladders + thumbnail extraction) ~0.5–2× real-time. Idempotent by content hash.

8.2 Stories tray

Per-user "tray" cache: Redis HSET storing followed-users → latest story_id. Updated on story create. Read returns the tray sorted by recency × affinity. Seen-bit kept in Bloom filter per viewer (false-positive tolerable: shows seen instead of unseen, recoverable on tap).

8.3 Reels feed

Discovery, not follow-graph driven. Two-stage ML: candidate gen by ANN (FAISS/ScaNN over user-reel embeddings, ~1000 candidates); ranker DNN scoring P(watch_complete), P(like), P(share) blended. Diversity reranking (MMR) prevents repetitive content. Cold-start: trending pool by region.

8.4 CDN strategy

Multi-CDN (Akamai + Cloudflare + own POPs). Origin shielding for hot reels. Pre-warm top 1% predicted reels to edges before they explode.

9. Scaling & bottlenecks

  • Transcode backlog: autoscale GPU pool on Kafka lag; degrade to lower-tier ladder during spikes.
  • Hot reel: consistent-hash + edge shielding; pre-fetch first segment on tray render.
  • Story expiry: TTL-driven Cassandra compaction is cheap; metadata cleanup via tombstones.
  • Counter contention: like counts via sharded counters + async aggregation.

12. Talking points for the interviewer

  • "I'd separate three product surfaces — feed, stories, reels — because each has a different access pattern and different DB requirements; one schema does not fit all."
  • "The hardest part is Reels CDN economics; my mitigation is multi-CDN + predictive pre-warming for the top 1%."
  • "For Stories I'd lean on TTL primitives instead of building a janitor — leveraging the database is cheaper than rolling your own."
  • "I'd discuss client-side prefetch as part of the feed contract — bandwidth-aware, drops to lower bitrates on cellular."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:

TechnologyWhy we chose it (for THIS problem)Alternatives considered
S3-class blob store (media)11-nines durability for 220 PB/year of transcoded media is non-negotiable; presigned-URL direct upload bypasses the app tier so 5K/sec peak uploads don't saturate API bandwidth, and immutable content-hashed keys make the CDN trivially cacheable.Local SAN/NAS (cannot scale to PB), HDFS (designed for batch reads, not media serving), DB BLOBs (would require sharding 300 TB/day of binary).
FFmpeg on GPU (transcode workers)5 ABR ladders per video at ~0.5–2× real-time for 100M uploads/day requires hardware-accelerated encoding; CPU-only would require ~5× the worker fleet, blowing the 30 s upload-to-visible SLO during peaks.CPU-only x264 (too slow at this volume), AWS Elemental MediaConvert (vendor lock + per-minute pricing at 100M/day is brutal), client-side transcoding (cannot trust client output).
HLS / DASH ABR200B Reels plays/day across cellular networks needs adaptive-bitrate streaming so weak connections get the 480p ladder while WiFi gets 1080p; first-frame p95 < 800 ms requires segment-0 prefetch which only works with chunked segmented formats.Progressive MP4 (no adaptation, breaks on weak networks), proprietary streaming (loses CDN compatibility), WebRTC (overkill for one-way playback).
Sharded MySQL/RocksDB (post metadata)2B MAU with hash-partitioning by user_id gives us write locality on 100M post creates/day; MySQL's transactional updates handle privacy/visibility mutations, RocksDB gives us LSM write throughput for the hot index.DynamoDB (cost at 12M reads/sec peak), Cassandra (weaker secondary index for moderation), single MySQL (cannot shard).
Cassandra (Stories with TTL)24 h ephemeral stories use default_time_to_live = 86400 as a database primitive — no janitor job needed; tombstone-based expiry plus per-partition compaction handles 500M views/day cleanup automatically.Redis with TTL (RAM cost too high for 500M views/day footprint), MySQL with cleanup cron (operational overhead and lag), DynamoDB TTL (vendor lock).
Redis (Stories tray HSET + Timeline)Per-user followed-users → latest story_id tray needs sub-ms point-lookup at feed open; 100K story-fetches/sec peak demands an in-memory store and the HSET primitive maps perfectly to (followed_user → story_id).DynamoDB (per-op cost too high), Cassandra (point-read latency too high for 200 ms feed-open SLO), Memcached (no HSET primitive).
Apache Kafka (transcode trigger + post-events)Decouples upload-finalize from transcode (which can take 0.5–2× video length) and from fanout; replayable log lets us reprocess transcodes after worker bug fixes without losing originals, and consumer-lag drives transcode-pool autoscaling.SQS (no replay), RabbitMQ (no durable log), direct RPC (loses backpressure and reprocess capability).
FAISS / ScaNN (Reels candidate ANN)Reels discovery is global-corpus, not follow-graph — billions of candidate videos need sub-millisecond user-embedding × video-embedding nearest-neighbor at 1.2M plays/sec; HNSW indexes hit this at acceptable recall.Exact kNN (impossible at billion scale), Elasticsearch vector (slower per-query, higher RAM), Annoy (lower recall at this scale).
Multi-CDN (Akamai + Cloudflare + own POPs)1 EB/day Reels egress cannot ride a single CDN's economics; multi-CDN gives us peering negotiation leverage, regional fallback when one CDN has an outage, and pre-warming top 1% predicted-viral reels across providers.Single-CDN (cost and outage risk), origin-only (impossible — would need PB/sec origin), no edge cache (defeats the 800 ms first-frame SLO).
Cassandra counters (engagement)Sharded distributed counters for likes at 200B Reels plays/day need write-throughput optimization; Cassandra's counter type with eventual consistency is fine because counts are advisory and per-user vote state lives separately.Redis INCR (would saturate single-shard hot keys), MySQL counter row (lock contention), DynamoDB counters (cost).
Bloom filter (story seen-bit)Per-viewer seen-tracking across 500M views/day with full-fidelity storage would explode; Bloom's space efficiency wins because false-positive (showing seen instead of unseen) is recoverable with one tap, while false-negative cannot happen.Per-user set in Redis (RAM blowup), no tracking (UX regression), exact RoaringBitmap (more space than Bloom for the FP-tolerant case).
TUS protocol (resumable upload)≤100 MB video uploads on flaky cellular networks need resume-from-byte-N; TUS is the open standard with broad client SDK support and avoids re-uploading the first 80 MB after a network blip.Plain PUT (full re-upload on failure), proprietary chunked protocol (client-SDK fragmentation), S3 multipart-only (works but TUS is the cleaner abstraction).
Two-stage DNN ranker (Reels)1000 candidates from ANN ranked into top-N at the feed-open budget needs candidate-gen separated from heavy ranking; multi-task heads scoring P(watch_complete), P(like), P(share) with MMR diversity prevents the filter-bubble that kills retention.Single-stage exhaustive (impossible at billion-corpus scale), pure-popularity (kills personalization), client-side rerank (cannot use server-side counters).
WebP + JPEG (photo renditions)7 photo sizes × WebP (modern) + JPEG (fallback) lets newer clients save 25–35% bandwidth while older devices still render; EXIF strip is mandatory for privacy on the 100M uploads/day surface.JPEG-only (loses bandwidth savings), AVIF-only (decode support gaps on older devices), single-size (kills mobile rendering).
