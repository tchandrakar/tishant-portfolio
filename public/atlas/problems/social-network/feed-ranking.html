Design News Feed Ranking Service (EdgeRank-style)

Hard social-network recommendation

1. Problem statement

Design the ranking service that sits between a candidate generator (the user's pushed timeline + pulled celebrity tweets + sponsored items) and the rendered feed. Given a candidate set of ~2,000 items per request, produce a sorted top-N at p99 < 100 ms while continuously incorporating new engagement signals. This is the classic EdgeRank-style problem: score = affinity × weight × decay, evolved into a multi-task DNN.

2. Functional requirements

  • Accept (userId, candidate[]) and return ranked top-N with scores and reasons.
  • Support multiple objectives: P(like), P(comment), P(share), P(long-dwell), P(report) — combined by a policy.
  • A/B framework: model variants, feature variants, weight variants.
  • Diversity / dedup constraints (no two posts from same author within top-5).
  • Explainability surface: "ranked highly because X."

3. Non-functional requirements

  • Scale: 2B DAU × 10 feed opens/day → 230K ranking RPS avg, ~1M peak. Each request scores ~2K candidates.
  • Latency: p99 < 100 ms per request including feature fetch; ranking-only budget 30 ms.
  • Throughput on inference: 230K × 2K = 460M scoring ops/sec sustained; ~2B peak.
  • Freshness: features < 60 s old for engagement counts; user features < 5 min.

4. Capacity estimation

  • Feature fetch: 1M req/sec × 50 user-features × 1 hop = bulk MGET; Redis cluster sized for 50M ops/sec at p99 < 2 ms.
  • Inference: 460M ops/sec / 200K ops/sec/GPU = ~2,300 GPU instances baseline; double for headroom.
  • Training data: 50B labeled impressions/day → ~5 TB/day Parquet; 30-day retention ≈ 150 TB.
  • Model registry: O(100) models × O(GB) — small footprint; deploy via shadow + canary.

6. High-level architecture

[Feed Svc] ──► [Rank GW] ──► [Feature Fetcher] ──► [Online Feature Store (Redis)] │ │ │ ▼ │ [Inference Servers (GPU/CPU)] ◄── [Model Registry] │ │ ▼ ▼ [Diversity / Policy Layer] ──► sorted top-N │ [Logger] ──► [Kafka impressions+labels] ──► [Flink joiner: impression × engagement] │ ▼ [Training pipeline] │ ▼ [Model Registry] ──► canary ──► prod

Walk-through: Feed Svc collects candidates and calls Rank GW. Rank GW: 1) batch-fetch features for the user (one Redis MGET) and per-item (one MGET keyed by itemId), 2) build the model input tensor, 3) score in batches of 128 on inference servers, 4) blend task heads via the policy weights, 5) apply diversity reranking (greedy MMR), 6) log impressions to Kafka. A separate Flink job joins impressions with engagement events to build training samples nightly + hourly.

7. Data model & storage

7.1 Online feature store (Redis cluster, key sharded)

user:{userId}      → {recent_likes_24h, ctr_30d, locale, device, embedding[256]}
item:{itemId}      → {age_minutes, type, author_id, embedding[256], counters}
edge:{u}:{a}       → {affinity_score, last_interact_ts}    # user × author

7.2 Offline store (Parquet on object storage)

Same feature definitions via a feature registry. Generated by the same code paths used online (DRY) to avoid train-serve skew.

7.3 Model registry

Versioned, immutable. Each version has metadata (training data hash, hyperparams, eval metrics). Promoted via shadow → 1% → 50% → 100%.

8. Detailed component design

8.1 EdgeRank baseline → DNN

Start simple: score = Σ affinity(u,author) × type_weight(post) × time_decay(age). This is interpretable and easy to debug. Once we have engagement labels, swap in a DNN with the same features plus embeddings: shared bottom + per-task heads. The policy layer combines task heads with tunable weights so product can shift toward sharing or commenting without retraining.

8.2 Feature fetch path

Two MGETs (user, items) in parallel. Co-located Redis cluster in the same DC as inference. p99 fetch budget 5 ms; if a feature is missing, default value is recorded as a feature itself (so the model can learn the "missingness" signal).

8.3 Inference

Batched scoring with dynamic batching window (5 ms or 128 items, whichever first). GPU model with INT8 quantization. Fallback CPU path for when GPU pool degrades — runs the same model at 3× latency but doesn't drop traffic.

8.4 Diversity / policy reranking

Greedy MMR: pick the highest-scoring item, then for each subsequent slot pick the item maximizing λ·score − (1−λ)·sim(picked). Author and topic diversity caps applied as hard constraints.

8.5 A/B + shadow

Every request hashes userId into experiment buckets. Shadow inference: traffic is scored by both control and candidate models; only control results return; candidate scores logged for offline comparison. This catches regressions without user impact.

8.6 Train-serve skew prevention

Critical operational concern. Same feature transform code used at train and serve time. Schema-versioned features. Continuous monitoring of feature distributions in prod vs training; alerts on drift.

9. Scaling & bottlenecks

  • Tail latency: hedged feature reads to Redis replicas; cancel-on-first-response.
  • Hot user (e.g. an influencer browsing): per-user cache of recent ranking results for 30 s, invalidated on new candidates.
  • Model rollout risk: auto-rollback if guardrail metrics (CTR, integrity flags) regress > X %.
  • Feature freshness vs cost: different SLOs per feature class; counters refresh on stream, embeddings hourly.
  • Cold candidates: include a global popularity prior so unknown items still rank above noise.

12. Talking points for the interviewer

  • "I'd start with a linear EdgeRank model because it's debuggable and the infra to serve it is the same as for a DNN; you can swap models without rebuilding the system."
  • "The hardest part isn't the model — it's the data pipeline. Train-serve skew is the silent killer; my mitigation is one shared feature transform library."
  • "I'd separate ranking score from policy weights so product can move objectives without ML re-training."
  • "I'd discuss shadow ranking for safe rollout, and guardrail-driven auto-rollback as the safety net."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis cluster (online feature store)1M req/sec × 50 user features = ~50M ops/sec needs sub-2ms p99 point lookups; MGET batches all user features in one round-trip, fitting comfortably inside the 5 ms feature-fetch budget out of the 100 ms total ranking SLO.Cassandra (single-digit-ms point reads too slow), DynamoDB (cost prohibitive at 50M ops/sec), DB direct (latency blowup, no MGET semantics).
Multi-task DNN with shared bottomScoring 460M ops/sec sustained for 5 task heads (P(like), P(comment), P(share), P(long-dwell), P(report)) on a single shared backbone is 5× cheaper than per-task models; policy-layer weighting lets product shift objectives without retraining.GBDT per task (cannot share representations), separate DNNs per task (5× inference cost), single-task scalar (loses tunable objective blending).
EdgeRank linear baselinescore = Σ affinity(u,author) × type_weight(post) × time_decay(age) is interpretable and debuggable, and ships on the same infra as the DNN — so we can stand the system up before engagement labels exist and swap the model later without rebuilding anything.DNN-from-day-one (no labeled data yet), pure-recency (lower engagement), heuristic-only forever (plateaus and blocks ML iteration).
GPU inference with INT8 quantization30 ms ranking budget for 2K candidates demands batched accelerator scoring; INT8 quantization gives ~4× throughput vs FP32 at acceptable quality loss, dropping the GPU fleet from ~9,000 down to the ~2,300 baseline.FP32 (4× more GPUs), CPU-only (3× latency, blows budget), per-request inference without batching (kicks GPU utilization to ~10%).
Apache Kafka (impressions + label log)Replayable, durable log is the source of truth for training-sample assembly; lets Flink rebuild samples after a feature bug, and serves as the cross-team event bus so engagement signals flow into ranking, integrity, and analytics simultaneously.RabbitMQ (no replay for sample rebuild), Kinesis (vendor lock + per-day retention), direct DB writes (no replayability or fan-out to multiple consumers).
Apache Flink (impression × engagement joiner)50B labeled impressions/day (5 TB/day Parquet) need an exactly-once join of impression-time features with later-arriving engagement events; Flink's stateful window-join with watermarks handles late-arriving labels (a like 2 hours after impression) deterministically.Spark Streaming (micro-batch latency, weaker exactly-once for joins), Kafka Streams (stateful join at this scale operationally harder), nightly batch only (loses hourly retraining cadence).
Parquet on object storage (offline training data)5 TB/day with 30-day retention (~150 TB) needs columnar projection — trainer reads only the 80 of 200 features it cares about, cutting IO 2.5× and matching the schema-versioned features served online via the same registry.Avro (row-wise, slower scan), JSON (uncompressed bloat), MySQL training table (cannot scale).
Model registry (versioned, immutable)O(100) model variants in flight with shadow → 1% → 50% → 100% rollout requires immutable artifacts with metadata (training data hash, hyperparams, eval metrics) to make rollback safe and auto-rollback on guardrail regression mechanical.Filesystem-only (no metadata, no rollback discipline), per-team ad-hoc storage (compliance nightmare), inline-deploy from training (cannot canary).
MMR diversity reranker (greedy)Server-side λ·score − (1−λ)·sim(picked) with hard caps on author/topic repeat is consistent across users (auditable and explainable), and "no two posts from same author within top-5" is enforceable as a hard constraint rather than a soft penalty.Determinantal Point Processes (more principled, computationally expensive), client-side dedup (loses cross-feature similarity), no diversity (UX regression with creator domination).
Shadow A/B frameworkDoubling inference cost during shadow tests is the price of safe rollout; both control and candidate score every request, only control returns, candidate scores logged for offline comparison — catches regressions before they touch users.Direct A/B (regression hits real users), offline-only eval (misses serving-time skew), canary-without-shadow (still leaks bad behavior to canary cohort).
Hashed userId bucketing (experiments)Stable experiment assignment must persist across sessions and not require server lookup; deterministic hash of userId into bucketIds gives consistent treatment without state and supports nested/orthogonal experiment layers.Random per-request (breaks longitudinal analysis), DB-stored assignment (lookup latency on hot path), cookie-based (mobile clients don't share cookies cross-session reliably).
Hedged Redis readsTail latency at p99 5 ms feature-fetch budget cannot tolerate one slow shard; hedged read to a replica + cancel-on-first-response trades 2× bandwidth for elimination of long tails — exactly the deal Jeff Dean articulated.Single-replica read (tail latency violates SLO), full duplicate read always (wasted RPC, no cancellation), client-side timeout-and-retry (worse than hedge because retries serialize).
Feature transform library (shared train+serve)Train-serve skew is "the silent killer" — same transform code library used at both training (Spark) and serving (Rank GW) is the only durable fix; schema-versioned features + drift monitors complete the safety net.Two implementations (skew bugs guaranteed and silent), DSL that compiles both ways (yet another DSL to maintain), ad-hoc per-team transforms (compounding skew).
Per-user result cache (30 s TTL)Hot-user case (influencer browsing rapidly) generates repeated identical ranking calls inside the candidate-stable window; 30 s TTL invalidated on candidate-set change is the cheapest way to absorb that without weakening freshness for normal traffic.No cache (wasted GPU on duplicates), longer TTL (stale ranks for fast topics), per-candidate-set memoization (RAM blowup at 2B DAU).
