Design Twitter / X Timeline (with celeb fanout problem)

Hard social-network messaging

1. Problem statement

Design Twitter's home timeline: a reverse-chronological (with light ranking) stream of tweets from accounts a user follows. Must handle the asymmetric follow graph where a single celebrity tweet can fan out to 100M+ followers, while still delivering tweets to typical users in <1 s and surviving 500K+ tweets/sec at peak (e.g. World Cup final).

2. Functional requirements

  • Post a tweet (≤280 chars, optional media, reply, quote, retweet).
  • Home timeline (followed accounts) and User timeline (one author).
  • Mentions, notifications, search hashtag/keyword (out of scope here, separate svc).
  • Edge cases: protected accounts, blocks, mutes, retweet-collapse.

3. Non-functional requirements

  • Scale: 500M MAU, 250M DAU, ~500M tweets/day; peak 500K writes/sec; reads 300K/sec avg, 1.2M/sec peak.
  • Latency: home timeline p99 < 200 ms; tweet-to-follower visible p95 < 5 s.
  • Availability: 99.99% read; write may degrade to async during spikes.
  • Durability: tweet must not be lost once 200 OK is returned.

4. Capacity estimation

  • Tweets: 500M/day = ~5.8K/sec avg, ~50K/sec peak (single-event spikes hit 500K/sec).
  • Fanout: avg 200 followers × 5.8K = 1.16M timeline inserts/sec for pure push.
  • Celeb amplification: a 100M-follower tweet = 100M inserts. Even at 100K inserts/sec/worker, that's 1000 worker-seconds. Pure push cannot meet 5 s SLO — pull needed.
  • Storage: 500M × 300 B avg = 150 GB/day tweets; 5-yr retention ≈ 275 TB on Manhattan (hot) + Hadoop (cold).
  • Cache: timeline cache 250M users × 800 IDs × 16 B ≈ 3.2 TB Redis.

5. API design

POST  /1.1/statuses/update           {text, mediaIds, replyTo}      → tweetId
GET   /1.1/statuses/home_timeline?count=50&max_id=...               → [tweet]
GET   /1.1/statuses/user_timeline/{userId}                          → [tweet]
POST  /1.1/favorites/create          {tweetId}
POST  /1.1/statuses/retweet/{tweetId}
WS    /1.1/streaming/user                                          → live new tweets

6. High-level architecture

[Client] → [TFE Edge] → [API GW] → [Tweet Write Svc] → [Manhattan (sharded KV)] │ ▼ [Kafka tweets] │ ┌───────────────┴───────────────┐ ▼ ▼ [Fanout Worker (push)] [Search Indexer (Earlybird)] │ ▼ [Redis Timeline Cache] ▲ [Timeline Read Svc] ──────┘ + pull celeb tweets from User-tweets cache │ ▼ [Mixer/Ranker] → [Hydration] → response

Walk-through: a tweet is written to Manhattan (durable), emitted to Kafka, then fanout workers push the tweetId into followers' Redis timelines. For accounts above the celebrity threshold (~10K followers, configurable), fanout is skipped. At read time, the Timeline Read Svc reads the user's Redis timeline and unions it with on-demand pulls of recent tweets from each followed celebrity, merges by time, ranks lightly, and hydrates from Manhattan.

7. Data model & storage

7.1 Tweets (Manhattan, sharded by tweet_id)

CREATE TABLE tweets (
  tweet_id   bigint PRIMARY KEY,   -- Snowflake, ms-precision
  user_id    bigint, text varchar(280), media_ids list,
  reply_to   bigint, retweet_of bigint, created_at timestamp
);
-- Secondary by user_id for user_timeline:
CREATE TABLE user_tweets (user_id, created_at, tweet_id) WITH CLUSTERING ORDER BY (created_at DESC);

7.2 Home timeline (Redis ZSET per user)

ZADD ht:{userId} {tweet_id_as_score} {tweet_id}   -- Snowflake encodes time
ZREVRANGE ht:{userId} 0 799                       -- cap 800

7.3 Why Manhattan?

Multi-tenant, multi-DC KV with strong per-key consistency, built for write-heavy social workloads. Wide-row friendly, designed at Twitter for exactly this access pattern.

8. Detailed component design

8.1 Fanout service

Stateless workers, partitioned by user_id of follower for locality. Idempotent ZADD makes retries safe. Backed by Kafka with at-least-once. SLO: p95 fanout-to-Redis latency < 2 s for non-celebs.

8.2 Celebrity pull path

Per-author tweet-list cache (Redis LIST keyed by user_id, last 800). At read, Timeline Read Svc fetches user's Redis ZSET + parallel MGET of followed-celeb lists (typically <50 celebs/user). Merge by Snowflake id, dedup retweets, return top N.

8.3 Asymmetric follow + threshold tuning

Threshold isn't fixed; it's the cost minimum of (push amplification × write rate) vs (pull cost × read rate). Twitter's empirical sweet spot ≈10K followers. Above that, account is "pull-only"; below, "push." Switching is a metadata flag; fanout worker reads it per tweet.

9. Scaling & bottlenecks

  • Spike events: backpressure — if Kafka lag > threshold, Fanout Svc switches mid-flight celebrities to pull-only and shrinks push depth from 800 to 200.
  • Hot Redis shards: consistent-hash with virtual nodes; replicate top 0.01% celeb-list keys to N replicas.
  • Cold cache: rebuild from Manhattan user_tweets clustering by created_at desc; warm in background on first hit.
  • Retweet storm: store retweet as a small pointer record; hydration pulls the original.

12. Talking points for the interviewer

  • "The interesting design tension is asymmetric follow — one celebrity write touches 100M reader timelines, but the median user has 200 followers, so we need two regimes."
  • "I'd build push first, measure the worker queue depth on a celeb tweet, then add the pull path."
  • "I'd argue against ranking the home timeline heavily on the read path — fan-in is already expensive; offload to async ranking jobs."
  • "For the World Cup spike, I'd discuss admission control and degrading retention depth before throughput."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Manhattan (sharded KV, tweet store)Multi-tenant, multi-DC KV with strong per-key consistency built at Twitter for write-heavy social workloads — exactly the access pattern for 500M tweets/day with 5-yr retention (~275 TB hot) where wide-row reads dominate.MySQL (sharding ops crushing at this write rate), Cassandra (weaker per-key consistency for read-your-writes), DynamoDB (vendor lock at this scale).
Redis ZSET (home timeline cache)250M users × 800 IDs × 16 B ≈ 3.2 TB hot working set; ZREVRANGE 0 799 at p99 sub-ms is what the 200 ms home-timeline SLO requires, and Snowflake-as-score gives us free time-ordering.Memcached (no sorted set), in-process LRU (cannot share across read replicas), DynamoDB (per-op cost too high at 1.2M reads/sec peak).
Apache Kafka (tweet event log)500K writes/sec spike events (World Cup) need a buffered, replayable log; Kafka's consumer-lag SLO gates the celeb-pull-switch backpressure mechanism that drops mid-flight celebrities to pull-only when lag exceeds threshold.RabbitMQ (no replay for fanout recovery), AWS Kinesis (vendor lock plus throughput limits), direct RPC (loses backpressure and durability before fanout).
Snowflake IDs64-bit ms-precision time-sortable IDs let us use the ID itself as ZSET score, eliminating clock-skew issues during merge of pushed-timeline + celeb-pull lists; deduplication of retweets becomes trivial integer comparison.UUIDv4 (not sortable, breaks merge-by-time), DB auto-increment (cannot generate 500K IDs/sec across DCs), ULID (Snowflake is the established Twitter convention).
Earlybird (search indexer)Built-at-Twitter Lucene-based real-time index optimized for the temporal locality of tweet search (mostly recent-first); avoids indexing 5-yr archive in the hot path while still serving search queries.Elasticsearch (heavier per-doc cost at 5.8K writes/sec sustained), Solr (similar trade-offs without the Twitter-specific tuning), Postgres FTS (cannot scale to billions of tweets).
Fanout workers partitioned by follower user_idPer-tweet classification into push (≤10K followers) vs pull (>10K) requires application logic; partitioning by follower locality lets each worker write to a contiguous Redis shard set, and idempotent ZADD makes at-least-once retries safe.Pure-push (100M inserts/celeb tweet violates 5 s SLO), pure-pull (every read does up-to-2K author MGETs), Spark Streaming (batch latency too coarse).
WebSocket streaming (/1.1/streaming/user)Live new-tweet delivery to active users at 1.2M peak reads/sec would generate billions of poll requests/day if HTTP-only; persistent WebSocket per active session keeps server cost flat while giving sub-second tweet visibility.HTTP long-poll (more connection churn at 250M DAU), SSE (older mobile client gaps), push notifications (wrong surface for the in-app stream).
TFE Edge (Twitter Front End)1.2M reads/sec peak across regions needs TLS termination, anti-abuse filtering, and request shedding before the API GW; admission control during World Cup spike is the difference between graceful degradation and full outage.Generic LB-only (no app-aware shedding), CDN-only (cannot terminate authenticated streaming), nginx-only (no integrated rate-limit backplane at this scale).
Hadoop (cold tweet archive)5-yr retention at 150 GB/day = 275 TB; only the last few days are hot. Hadoop on commodity disk gives us ≤$0.01/GB/month for the cold tail while Manhattan stays sized for the hot working set only.All-Manhattan (10× cost), S3 (acceptable but Twitter built on Hadoop), tape archive (recovery latency unsuitable for user-timeline backfill).
Consistent hash with virtual nodesHot Redis shards on top 0.01% celeb-list keys need to be replicated to N replicas without remapping the entire keyspace; virtual nodes give us cheap rebalancing when we add capacity for spike events.Modulo hashing (every resize remaps everything, breaks cache during the worst possible moment), per-shard manual placement (operationally brittle).
Idempotent ZADD (at-least-once semantics)Kafka at-least-once + idempotent ZADD is dramatically simpler than exactly-once; duplicates collapse because the score (Snowflake ID) is identical, so retries during fanout-worker restarts cause no user-visible artifacts.Exactly-once via two-phase commit (latency hit on every write), at-most-once (loses tweets on crash, breaks durability promise).
Mixer/Ranker (light read-time ranking)Heavy ranking on read explodes the 200 ms p99 budget; light reranking with offline-precomputed signals (engagement priors, author affinity) keeps the synchronous path cheap while still beating pure reverse-chronological.Heavy DNN at read (blows latency), pure chronological (lower engagement), client-side ranking (cannot use server-only signals like global virality).
