Design Real-time Event Collector (HTTP → Kafka → S3, billions of events/day)

Hard analytics

1. Problem statement

Build a global, public-internet event collector — the front door for clickstream, mobile telemetry, and server-side beacons — that accepts billions of HTTP POSTs per day, durably enqueues them onto Kafka within milliseconds, and lands them as compacted Parquet files on S3 for downstream analytics. This is the "Snowplow / Segment / Iglu" layer underneath Mixpanel, A/B testing platforms, and ML feature pipelines.

2. Functional requirements

  • Public HTTPS endpoint, browser-friendly (CORS, GIF pixel, beacon).
  • Schema validation against a registry; reject malformed events with structured error.
  • Idempotency via client-supplied event_id (UUIDv7).
  • At-least-once delivery to Kafka; exactly-once landing to S3 (file-level).
  • Per-tenant rate limiting and authentication (API key + HMAC).
  • S3 lake organized for efficient downstream batch jobs.

3. Non-functional requirements

  • Scale: 10B events/day baseline, 50B peak (Black Friday); 600K rps p99.
  • Latency: p99 ack < 50 ms after Kafka commit; client-perceived < 200 ms globally.
  • Durability: 99.9999% (1 in 1M lost is the SLO ceiling).
  • Availability: 99.99% (this is a write-only critical path).
  • Cost: < $0.05 per million events landed.

4. Capacity estimation

  • 10e9 ev/day ≈ 116K eps avg, 600K eps peak.
  • Per event: ~1.5 KB JSON → ~15 TB/day raw; ~3 TB/day Snappy/Parquet.
  • Kafka cluster: 1024 partitions @ 100 KB/s avg, 1 MB/s peak → 6 brokers per region with RF=3.
  • Edge fleet: 600K rps / 5K rps-per-pod = ~120 collector pods/region (3 regions).

5. API design

POST /v1/track            Headers: x-api-key, x-signature
Body: { event_id, ts, tenant, type, props }
GET  /pixel.gif?d=<b64-json>        // for sandboxed browsers
POST /v1/batch    [up to 500 events]
→ 202 Accepted { received: 500, rejected: 0 }
→ 4xx for schema/auth errors with field-level details

6. High-level architecture

[Browser/Mobile/Server] --DNS GeoDNS--> [Anycast LB / CloudFront] | v [Edge Collector Pod (Go)] - auth, rate-limit - schema-lite check - write to Kafka idempotent producer | v [Kafka events.raw, RF=3, acks=all, 1024 parts] | +-----------------------------+----------------------------+ v v [Validator Stream (Flink)] [Real-time consumers] - registry schema check - feature pipelines - DLQ for invalid - alerting / fraud | v [S3 Sink (Kafka Connect / Flink)] hourly Parquet partitioned by (tenant, dt, hr)

The collector terminates TLS, authenticates, applies a fast structural check, and produces to Kafka with acks=all and an idempotent producer. Only after the Kafka ack do we 202 the client — this guarantees at-least-once. A separate stream job validates against the schema registry; bad events go to a DLQ, never silently dropped.

_SUCCESS markers per (dt,hr) when all 1024 partitions have committed past hr+1

7.3 Why Parquet?

Columnar Parquet over S3 is the canonical analytical lake format (DDIA Ch. 10). It enables predicate push-down, delivers ~5× compression over JSON, and is consumable by every analytical engine (Trino, Spark, Athena, Snowflake external tables).

8. Detailed component design

8.1 Idempotency

Producers send event_id; downstream consumers de-dup using a Bloom filter + 24h Redis set keyed by (tenant, event_id). Kafka's idempotent producer prevents broker-side dup; consumer-side dedup catches client retries.

8.2 Schema registry

Confluent-style registry keyed by (tenant, event_type, version); stream validator enforces backward-compatible evolution. Rejected events carry a x-iglu-error reason code.

8.3 S3 sink "exactly-once"

Flink's two-phase-commit S3 sink writes temp files, on checkpoint atomically renames to final names — gives effectively exactly-once file landing even though Kafka delivery is at-least-once (DDIA Ch. 11 end-to-end argument).

8.4 Backpressure & shedding

If Kafka producer queue exceeds threshold, the collector returns 429 with Retry-After. Mobile SDKs honor this and exponentially back off; we never silently drop accepted events.

9. Scaling & bottlenecks

  • Connection storm: 600K rps over HTTP/2 = ~50K concurrent conns; tune kernel somaxconn, use connection-reuse on edge LB.
  • Tenant skew: one customer launches a campaign → partition hot-spot; partition by hash(tenant, event_id) instead of tenant alone.
  • Small-files on S3: 1024 partitions * 8760 hours = millions of files/year; nightly compaction batch job (DDIA Ch. 10) coalesces < 64 MB files.
  • Cold start of new region: bootstrap Kafka MirrorMaker before pointing GeoDNS to avoid a brand-new partition leader storm.

11. DDIA references

  • Ch. 11 Stream Processing — Kafka as the unbundled commit log; idempotent producers + transactions; two-phase-commit S3 sinks for end-to-end exactly-once landing; DLQ pattern for poison messages.
  • Ch. 10 Batch Processing — the S3 lake is the canonical immutable batch input; nightly compaction, schema migration via re-write, and downstream MapReduce/Spark jobs all follow Ch. 10 patterns.
  • Ch. 8 Trouble with Distributed Systems — client clock skew makes ts_client unreliable; we retain it but use ts_server for partitioning.
  • Ch. 9 Consistency & Consensus — Kafka ISR with acks=all is a quorum write; choosing acks=1 vs all is the durability/latency knob.

12. Talking points for the interviewer

  • "This is a write-amplified write-only path; my dominant constraint is durability, so I lock in acks=all even though it costs 10 ms."
  • "I'd ack the client only after Kafka commit — ack-then-Kafka is the classic data-loss footgun."
  • "At 50B events/day Black-Friday peak, the small-files problem on S3 kills downstream queries; the compaction batch job is part of the design, not an afterthought."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Kafka with acks=all + min.insync.replicas=2 50B events/day with no tolerance for loss (billing/audit downstream). acks=all costs 5-10 ms latency but guarantees a committed event survives a single-broker failure; ack-then-Kafka is the classic data-loss footgun and is non-negotiable here. acks=1 (loses on broker crash); RabbitMQ mirrored queues (lower throughput, harder ops)
Stateless HTTP collector behind L7 LB At 5M events/sec peak (Black-Friday), only horizontal scale survives. Stateless pods let us autoscale 100→1000 instances on CPU without coordination; any state (auth, dedup) lives in Redis or Kafka, never in the collector. Sticky sessions (breaks autoscaling); stateful dedup in collector (kills horizontal scale)
Protobuf over HTTP/2 for SDK→collector JSON adds 3× payload size; at 50B events/day that's 200+ TB/day of wasted egress. Protobuf shrinks per-event size to ~120 bytes and HTTP/2 multiplexing keeps connection count low for mobile clients on poor networks. JSON over HTTP/1.1 (3× bandwidth, head-of-line blocking); MsgPack (weaker tooling)
Edge regional collectors + cross-region Kafka MirrorMaker Mobile clients in APAC paying 300 ms RTT to us-east-1 will time out under retry storms. Regional collectors give <50 ms ingest latency globally; MirrorMaker async-replicates to the central topic for analytics without putting RTT on the client path. Single-region (high mobile loss); active-active with conflict resolution (overkill)
Idempotency-key dedup via Redis (24h TTL) Mobile SDKs retry aggressively on network blips; without dedup we'd ship 5-10% duplicate events to billing. SHA1(idempotency_key) lookup in a Redis cluster (~1 M ops/sec, 1 ms p99) is the only structure cheap enough at 5M events/sec. DB-level UNIQUE (write contention); Bloom filter (false positives drop real events)
S3 + Parquet sink with hourly compaction job Streaming Kafka→S3 produces millions of tiny files; the small-files problem on S3 kills downstream Spark queries (each file = 1 task). An hourly compaction batch job rewrites to ~512 MB Parquet files, a designed-in part of the pipeline, not an afterthought. No compaction (Spark OOM); Avro (row-oriented, no column pruning)
Schema registry (Confluent) with backward-compat enforcement 50+ client teams ship SDK upgrades on independent timelines. Backward-compat enforcement at registration time prevents one team's rename from silently breaking the warehouse loader, the failure mode that costs days of backfill at 50B events/day. No schema (downstream chaos); Protobuf without registry (no compat checks)
Token-bucket rate limiter per API key One bug in a customer's SDK can DDoS our cluster at 100× their normal rate. Per-key token bucket in Redis gives O(1) check at 5M events/sec and isolates noisy tenants without affecting the long tail of well-behaved customers. Global limit (good actors throttled); per-IP limit (mobile NATs share IPs)
HMAC-signed event payloads Public collector endpoints invite spoofing; a forged event in billing is a $-impacting incident. HMAC with per-tenant rotating keys is <10 µs to verify and gives cryptographic source authentication, far cheaper than mTLS at SDK scale. mTLS (client cert distribution painful for mobile); plain API key (replayable)
Async batch fsync to local disk before Kafka send If the collector crashes between accept and Kafka produce, events are lost. A small WAL on local NVMe with batched fsync (every 5 ms) bounds loss to <10K events even on hard kill, while keeping p99 ingest latency under 20 ms. No WAL (silent data loss); fsync per request (kills throughput)
OpenTelemetry for collector self-observability The collector itself processes 50B events/day; we need spans on its hot path or we're flying blind during incidents. OTel SDK + OTLP export integrates with the same tracing backend the rest of the platform uses, no extra infra. StatsD (metrics only, no traces); custom logging (no causality across hops)
30-day raw event archive on S3 Glacier Compliance and dispute resolution require replaying any historical event. Glacier at $1/TB/month makes 30 days of raw 50B events/day (~6 PB) affordable; retrieval latency is fine because audits are non-interactive. Hot S3 (10× cost); no archive (compliance/legal exposure)
