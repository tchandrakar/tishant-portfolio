Design Funnel & Retention Analytics (sessionization at scale)

Hard analytics database-ddia

1. Problem statement

Build the analytical engine that powers product-team questions: "what % of users who started checkout finished within 30 minutes?", "what's the D7 retention of users acquired via TikTok in March?", "where do users drop off in the onboarding flow?". The substrate is a stream of user events (10B/day); the output is interactive funnel and retention queries that must run in seconds. The crux is sessionization at scale — grouping unbounded events into per-user sequences efficiently.

2. Functional requirements

  • Funnel queries: ordered N-step conversion within a user-supplied window.
  • Retention queries: weekly/daily cohorts vs. an "active" event.
  • Segmentation: filter cohorts by user properties or first-event properties.
  • Sessionization: derive sessions (30-min inactivity gap) used by both funnel and retention.
  • Backfill: re-run historical computations after a definition change.

3. Non-functional requirements

  • Scale: 10B events/day, 500M MAU, 50M DAU.
  • Query p95: < 5 s for a 90-day funnel over 100M-user cohort.
  • Freshness: within 5 min for "today" queries; full daily snapshot by 02:00 local.
  • Backfill SLA: 1 year of history reprocessable in < 24 h.
  • Reproducibility: same query → same number bit-for-bit.

4. Capacity estimation

  • 10e9 events/day * 1 KB = 10 TB/day; 3.6 PB/yr after 3× Parquet compression ≈ 1.2 PB.
  • Sessions/day: 50M DAU * 3 sessions = 150M; avg 30 events/session.
  • Per-user state for streaming sessionization: 50M active * 1 KB = 50 GB Flink RocksDB.
  • Cohort bitmaps: 500M users * 1 bit * 365 days = ~22 GB Roaring (post-compression).

6. High-level architecture

[events.clean Kafka] ----+ | v [Flink Sessionizer] <-- streaming, 30m gap key_by(user_id) RocksDB state window(EventTimeSession) | v [sessions.stream Kafka topic] | +-----------------+--------------------+ v v [Hot Cube (ClickHouse)] [S3 Parquet lake] partitioned by (project, dt) sessions / events history ORDER BY (user_id, ts) | ^ v | [Spark batch sessionizer] (DDIA Ch. 10) | rebuilds full history nightly +------- bulk load (compaction) ----------+ | [Query Planner / API]

Two cooperating sessionizers: a streaming Flink job for fresh data and a batch Spark job that re-derives sessions deterministically from raw events nightly. Queries hit the streaming-built hot tier for "now", the batch-built lake for everything else, and the planner unions results when a window crosses the boundary — the lambda-architecture trade-off discussed in DDIA Ch. 11.

7. Data model & storage

7.1 Sessions table

CREATE TABLE sessions (
  project_id   UInt32,
  dt           Date,
  user_id      String,
  session_id   UUID,
  start_ts     DateTime64(3),
  end_ts       DateTime64(3),
  events       Array(Tuple(name LowCardinality(String), ts DateTime64(3), props Map(String,String))),
  acq_channel  LowCardinality(String),
  device       LowCardinality(String)
) ENGINE=MergeTree
PARTITION BY (project_id, toYYYYMM(dt))
ORDER BY    (project_id, user_id, start_ts);

7.2 Cohort bitmaps

active_bitmap (project, day)  ->  RoaringBitmap of user_id_hash
funnel_step    (project, funnel_def_hash, day, step) -> RoaringBitmap

8. Detailed component design

8.1 Sessionization (the hard part)

Stream sessionization keys events by user_id and uses an event-time session window with 30-min gap (DDIA Ch. 11). Two real-world headaches: (a) late events — mobile reconnects after 7 days — handled by allowed-lateness + side-output for "session corrections"; (b) state size — 50M active users in RocksDB; we TTL state at 7 days.

8.2 Funnel evaluator

Given steps[A, B, C] and window W: walk a user's events in time order, advance a step pointer when matching, reset on window expiry. Emit (user, max_step_reached). Aggregating gives the funnel. This is O(events_per_user) and parallel across users; the clustered (user_id, ts) sort key makes it a sequential scan.

8.3 Retention via bitmaps

Cohort C = users whose first signup was in week W. Retention[C][k] = popcount(C AND active_bitmap[W+k]). Whole 12-week retention matrix for a 100M-user cohort runs in < 1 s using RoaringBitmap intersections (millions of ops/sec).

8.4 Backfill / replay

Sessionization logic and cohort definitions live in versioned config. A backfill spins a Spark job over the immutable Parquet lake (DDIA Ch. 10 batch over immutable inputs) producing a new sessions_v{n} table swapped atomically; queries pin to the version to keep history reproducible.

9. Scaling & bottlenecks

  • Skewed users: bots/test accounts with 1M events/day blow up Flink state — cap per-user events/session, isolate via salted secondary key.
  • Definition churn: when a PM changes the funnel definition, naive recompute is expensive; we precompute step-bitmaps daily and intersect at query time.
  • Hot project: one huge customer dominates the cluster; we isolate them onto a dedicated ClickHouse shard.
  • Stream/batch reconciliation: if streaming and batch disagree on a day, batch wins; planner serves batch from the day boundary.

12. Talking points for the interviewer

  • "I'd run lambda not because I love operating two systems but because product analysts demand reproducible numbers across definition changes — that needs batch-on-immutable-input."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
RoaringBitmap over user_ids per (event, day) An N×M retention grid (e.g., 30 cohorts × 90 return-days) over 50M users would be a 4 TB JOIN; bitmaps store each set in ~6 MB and intersect via popcount in <5 ms. This is the trick that turns retention from O(N×M) to O(N) and makes the UI feel instant. SQL JOIN on user_id (timeouts at 50M users); HyperLogLog (no exact intersection)
ClickHouse for ad-hoc funnel/cohort queries Analysts iterate on funnel definitions interactively; each redefinition triggers a fresh scan of 50B events. ClickHouse's vectorized aggregates and windowFunnel() primitive return 7-step funnels in <3 s on a year of data; a row store would take 10+ minutes. Postgres (row store, OOM); BigQuery (good alt, $10/TB scan cost adds up); Druid (weak JOIN)
Lambda architecture (Spark batch + Flink streaming) Product analysts demand reproducible numbers across funnel-definition changes — that needs batch-on-immutable-input. Streaming gives today's freshness; nightly Spark over Parquet gives reproducibility. We run two systems not for fun but because no single engine satisfies both constraints. Kappa (streaming-only) (no reproducibility for redefinitions); batch-only (24h staleness)
Flink stateful sessionization on Kafka "User session" is the foundation of every funnel; getting it right once at the streaming layer means funnel/retention are cheap derivations on top. Flink's keyed-state + event-time windows + watermarks correctly handle 30-min idle gap with late events at 5M events/sec. Spark Streaming (micro-batch, weaker watermarks); custom (re-implement Flink badly)
Parquet on S3, partitioned by date + bucketed by user_id Funnel batch jobs filter by date range and shuffle on user_id. Date partitioning prunes 99% of the 5 PB lake; user_id bucketing eliminates the shuffle stage in Spark, cutting nightly funnel rebuild from 4 h to 45 min. Date-only partitioning (full shuffle); ORC (smaller ecosystem); CSV (no pushdown)
Iceberg table format over the Parquet lake Nightly recompute rewrites partitions while analysts are still querying yesterday's snapshot. Iceberg's snapshot isolation gives transactional swap so readers never see half-written data, critical when 50+ analysts are hitting the same tables. Plain Parquet + symlink swap (race conditions); Delta Lake (good alt, less multi-engine)
HyperLogLog sketches for top-of-funnel reach "Unique users entering funnel step 1 last 90 days" needs no exact count. HLL at 1% accuracy in 16 KB per cohort is millions of times smaller than the underlying user-id set and allows associative merge across days, so adding a day to the window is sub-second. COUNT(DISTINCT) (full scan); exact bitmap (overkill for top-of-funnel)
Materialized retention cubes (cohort × day matrix) The retention chart is the single most-loaded view in the product. Pre-computing the 30×90 cohort matrix nightly turns a multi-billion-row scan into a single small read; without it the dashboard would scan 50B events on every refresh. Query-time recompute (slow dashboards); always-on streaming (high cost for stale view)
Kafka as the event log feeding both pipelines Lambda needs the same event stream feeding batch and stream paths or the two will diverge. Kafka with 7-day retention lets streaming consume live and the daily Spark job consume from offset 0 for full reprocess on definition change. Kinesis (24h retention default); separate ingest paths (batch vs. stream divergence)
RocksDB for Flink keyed state Sessionization at 5M events/sec keeps state for ~50M active users (~50 GB). RocksDB-backed Flink state spills to local NVMe so we don't pay for 50 GB of RAM per task slot, with 1 ms p99 read on the keyed state path. Heap state (OOM at 50M keys); Redis (network hop kills throughput)
Bitmap caching layer (Redis with binary values) Hot cohorts (last-7-day signups) are queried by every PM dashboard. Caching the RoaringBitmap blob in Redis with a 1h TTL serves repeat queries in <5 ms without re-running the funnel; cache miss falls through to ClickHouse. Memcached (no binary-safe ops); ClickHouse query cache (per-shape, narrower hit rate)
Airflow for nightly orchestration The reconciliation job has 12+ stages with cross-dependencies (compaction→dedup→sessionize→rollup→bitmap-build). Airflow's DAG model with retries and SLAs means a single stage failure doesn't silently corrupt the morning dashboards. Cron + scripts (no dependency management); Dagster (smaller ecosystem at this org's stage)
