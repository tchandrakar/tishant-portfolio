Design Datadog-style Metrics Pipeline (ingest → tsdb → dashboards)

Hard analytics

1. Problem statement

Design a SaaS observability platform like Datadog that accepts numerical time-series metrics from hundreds of thousands of customer hosts, stores them in a purpose-built TSDB, evaluates alert rules in near-real-time, and renders dashboards over arbitrary time ranges (1 minute to 15 months). Datadog publicly reports ingesting on the order of trillions of metric data points per day across millions of hosts, so the design must be horizontally scalable and economically viable at petabyte scale.

2. Functional requirements

  • Ingest metrics via a StatsD-compatible agent (counters, gauges, histograms, distributions) and via API push.
  • Tag-based dimensional model: web.req.latency{env=prod, region=us-east-1, host=h-42}.
  • Query API supporting filters, group-by, rollups (avg, p99, sum) over arbitrary windows.
  • Alerting: evaluate rules every 10s, trigger via PagerDuty/Slack.
  • Multi-tenant isolation, retention tiers (15s for 24h, 1m for 30d, 1h for 15mo).

3. Non-functional requirements

  • Ingest latency: point lands in queryable storage in < 10s p99.
  • Query latency: dashboard panel returns in < 500 ms p99 for 1-day windows.
  • Scale: 5T points/day, 50M unique series active, 200K agents.
  • Durability: 99.999% (no silently dropped points).
  • Availability: 99.95% query, 99.9% alerting.

4. Capacity estimation

  • QPS ingest: 5e12 / 86400 ≈ 58M points/sec globally; per region ~10M pps.
  • Wire size: ~25 B compressed per point → ~125 GB/s → ~10 PB/day raw, ~1 PB/day after Gorilla compression (10× per Facebook's paper).
  • Series cardinality: 50M active * 16 B series-id = 800 MB hot index per shard.
  • Query fan-out: dashboard with 30 panels → 30 parallel queries → each touches ~6 shards.

5. API design

POST  /api/v1/series          { series:[{metric, points:[[ts,v]], tags}] }
GET   /api/v1/query?from=&to=&query=avg:web.req.latency{env:prod}by{host}
POST  /api/v1/monitor         { query, threshold, window, notify }
GET   /api/v1/dashboard/{id}                               → panels

6. High-level architecture

[Agent (statsd)] -- HTTPS --> [Edge Ingest LB] | v [Intake Svc] --> [Kafka: metrics-raw, 1024 parts] | +-------------------------+-------------------+ v v v [Rollup Worker] [Alert Evaluator] [Indexer (tag KV)] | | | v v v [TSDB shards (Gorilla)] [Rule state RocksDB] [Cardinality svc] ^ | [Query Frontend] <-- [Dashboards / API users]

Agents batch every 10s, ship to the nearest region. The intake service authenticates, enriches with tenant ID, and produces to Kafka partitioned by (tenant, metric_hash). Rollup workers consume, write the 15-second resolution to the TSDB, and emit downsampled 1m/1h streams. A separate alert evaluator consumes the same topic so alerting is decoupled from storage availability (DDIA Ch. 11 — stream consumers reading the same log).

7. Data model & storage

7.1 TSDB shard layout

Series-ID = murmur3(metric || sorted(tags))    // 64-bit
Block file: per-shard, 2-hour blocks
  +-- chunks[ ]   Gorilla-compressed (XOR + delta-of-delta on ts)
  +-- index       postings: tag=value -> [series-id...]
  +-- meta        min_ts, max_ts, series count

7.2 Why a custom TSDB?

Generic OLTP DBs fail because every point is an append, points are immutable, and reads scan billions of values. Gorilla's bit-packed format averages ~1.4 B/point, a 10× reduction over raw doubles. We borrow Prometheus/M3DB's inverted index for tag lookups — DDIA Ch. 3 secondary index discussion applies here.

8. Detailed component design

8.1 Cardinality control

The killer failure mode is high-cardinality tags (e.g., user_id). The intake service maintains a per-tenant Bloom + count-min over series-IDs and rejects new series after a threshold (e.g., 1M unique series/metric). This protects neighboring tenants from one bad customer.

8.2 Query frontend

Two-tier fan-out: frontend splits the time range into 2-hour blocks, dispatches sub-queries to the owning shards, and merges. A "split-and-cache" layer (similar to Cortex) memoizes immutable old blocks in Redis for 24h.

8.3 Alerting

Each rule is sharded by hash(rule_id) onto evaluator nodes. State (last-fired, hysteresis) is kept in embedded RocksDB and replicated via Raft for HA — DDIA Ch. 9 quorum trade-off.

9. Scaling & bottlenecks

  • Hot tenants: rebalance Kafka partitions hourly using consistent hashing with bounded loads.
  • Tag-explosion: auto-quarantine offending metric, alert customer, throttle.
  • Long range queries (15mo): served exclusively from 1h rollup tier in S3/Parquet.
  • Backfill: rollup workers checkpoint Kafka offsets in ZooKeeper; on lag > 5 min, autoscale.

12. Talking points for the interviewer

  • "At trillions of points/day the dominant cost is storage, so I'd lead with the Gorilla compression decision and only then talk about query."
  • "The hardest real-world problem is cardinality explosion from a single bad user_id tag — I'd build the rejector before I built the dashboard."
  • "I'd treat alerting as a Kafka consumer not a DB query so a TSDB outage never silences pages."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Gorilla TSDB compression (delta-of-delta + XOR) At trillions of points/day, raw 16-byte (ts, val) tuples cost ~16 PB/day. Gorilla's delta-of-delta on monotonic timestamps and XOR on slowly-changing floats compresses to ~1.37 bytes/point, an >10× reduction that turns a $50M/yr storage bill into a $5M one. Snappy/LZ4 (general-purpose, ~3× only); gzip (CPU-heavy, kills ingest at 10M points/sec)
2-hour immutable in-memory blocks (Prometheus TSDB style) Recent points are queried 100× more than old ones; keeping 2h hot in RAM serves p99 dashboard reads in <50 ms across trillions of points/day. Immutability lets us ship blocks to object storage without locks and rebuild indexes offline. B-tree on disk (random IO crushes ingest); always-RAM (cost-prohibitive past 24h)
Inverted index on label pairs (Lucene-style postings) Queries like cpu{env=prod,region=us-east} at trillion-point scale must resolve to series IDs in <5 ms. Postings lists per k=v intersected via roaring bitmaps are the only structure that scales to 100M+ active series per cluster. Full-table scan (impossible at this cardinality); B-tree on (k,v) (no efficient AND across labels)
Cardinality rejector at ingest One bad user_id tag can balloon series count from 10M to 1B in minutes and OOM the entire fleet. A pre-write Bloom-backed cardinality limiter per tenant is mandatory at trillions of points/day; it must run before the inverted index ever sees the label set. Post-hoc cleanup (cluster already crashed); per-query limits (too late, write path is dead)
Kafka as ingest log (separate topic per shard) Decouples 10M points/sec write spike from TSDB compaction. Lets alert evaluator, downsampler, and storage write all consume independently; a TSDB outage doesn't silence pages because alerts read Kafka, not the DB. Direct gRPC to TSDB (back-pressure pushes to agents); RabbitMQ (no replay, ordering weak)
Object storage (S3) for blocks > 2h $23/TB/month vs. $80/TB on local SSD; at multi-PB scale this saves >$10M/yr. 2-hour immutable blocks are perfect for object stores: write-once, read-many, range-GET friendly. HDFS (operational burden); local SSD (10× cost); Glacier (retrieval latency kills queries)
Downsampling pipeline (1m → 5m → 1h rollups) A year of 1-second resolution per series is 31M points; downsampled to 1h it's 8.7K points (3500× smaller). Long-range dashboards (30d, 1y) read the rollup tier, keeping query latency flat regardless of retention. Query-time aggregation only (slow at 1y range); fixed retention (loses long-term signal)
Raft for alert evaluator state "Is alert X currently firing?" must be consistent across replicas or PagerDuty gets duplicate or lost pages. Raft on a small alert-state group (~1 KB per alert) is cheap and gives linearizable reads at the rate alerts evaluate (~1k/sec). Single leader (SPOF); eventual consistency in DynamoDB (causes alert flapping)
Consistent hashing for series → shard At 100M+ active series, re-sharding on every node add/remove is a non-starter. Consistent hashing on hash(metric+labels) moves only 1/N of the keyspace per topology change, so we can scale ingest from 100 to 1000 nodes without ingestion stall. Modulo hashing (full reshuffle); range partitioning (hot shards on popular metrics)
PromQL-compatible query layer Engineers already know PromQL; a custom DSL would block adoption. PromQL's vector/matrix model maps naturally to the (series → timeseries) layout and lets us reuse Grafana out of the box. SQL on TSDB (subqueries clumsy for rate/irate); custom JSON DSL (zero ecosystem)
RoaringBitmap for label-set intersection Postings lists for popular labels (e.g. env=prod) hold 50M+ series IDs. Roaring's hybrid array/bitmap containers compress sparse and dense ranges, doing AND across 4-label queries in <1 ms versus 50+ ms for sorted-array intersection. Sorted int arrays (slow AND); plain bitmaps (memory blowup on sparse postings)
WAL on local NVMe before in-memory block A node crash 1h into a 2h block would lose 600M points. A simple append-only WAL on NVMe (1 GB/sec write) makes recovery deterministic and lets us ack writes durably within 5 ms p99. Replicate in-memory only (loses on dual-failure); fsync on every point (kills throughput)
