Design Mixpanel / Amplitude Event Analytics (funnel + cohort)

Hard analytics database-ddia

1. Problem statement

Build a product-analytics SaaS where customers instrument their apps with an SDK that emits semi-structured user events (signup, video_played, checkout) with arbitrary properties. Product managers then run interactive funnel, cohort, retention, and segmentation queries over months of data. Mixpanel publicly handles tens of billions of events per day; Amplitude similarly. Queries must return in seconds despite scanning hundreds of billions of rows.

2. Functional requirements

  • Ingest JSON events (5–50 properties each).
  • Funnel query: of users who did view_landing, how many later did signup within 7 days?
  • Cohort: weekly active users by acquisition channel.
  • Retention curve: D1/D7/D30 for any cohort definition.
  • Live "stream" of events for debug.
  • SDKs for web, iOS, Android, server.

3. Non-functional requirements

  • Ingest: 50B events/day (~580K eps), bursty 3× on launches.
  • Query latency: p95 < 5 s for 30-day funnels over 10B events.
  • Freshness: events queryable within 60 s.
  • Retention: 5 years hot.
  • Multi-tenancy: 30K projects, hard isolation.

4. Capacity estimation

  • 50e9 events * 1 KB JSON = 50 TB/day raw; ~5 TB/day after columnar Zstd compression.
  • Active users: 2B distinct user_ids/day → user-state KV ~ 200 GB.
  • Query workload: 10K analyst queries/min; each scans ~1B rows.
  • Shards: target 50 GB/shard → ~100 shards/project for the largest customers.

5. API design

POST /track            { event, distinct_id, time, properties }
POST /engage           { distinct_id, $set:{plan:"pro"} }      // user props
POST /query/funnel     { steps:[...], window:"7d", segment:{} }
POST /query/cohort     { definition, group_by:"acq_channel" }
GET  /export?from=&to= → signed S3 URL

6. High-level architecture

[SDK] -> [Edge Collector] -> [Kafka events.raw] --+--> [Validator] --> [Kafka events.clean] | +--> [User-State Stream Joiner (Flink)] | v [Columnar Lake (Parquet on S3)] | [ClickHouse / Druid hot tier] <-- bulk loader ^ | [Query Engine] <-- [API GW] <-- [Web UI]

The SDK batches events client-side, posts to the edge, which writes to Kafka. Stream processors enrich each event with the latest user-properties (a stream-table join — DDIA Ch. 11), partition by (project_id, user_id), and write Parquet files to S3 every minute. A bulk loader ingests fresh files into a column store (ClickHouse) for sub-second analytics; older partitions are queried directly from S3 via a Trino-like engine.

7. Data model & storage

7.1 Event row

CREATE TABLE events (
  project_id   UInt32,
  event_date   Date,
  user_id      String,
  event_name   LowCardinality(String),
  ts           DateTime64(3),
  props        Map(String, String),
  device_id    String,
  session_id   UUID
) ENGINE = MergeTree
PARTITION BY (project_id, event_date)
ORDER BY    (project_id, user_id, ts);

7.2 Why ORDER BY user_id?

Funnels are user-centric: "did THIS user do step A then step B?" Co-locating a user's events on disk turns funnel evaluation into a sequential scan with O(events_per_user) memory, not a full table sort. This is the same insight as DDIA Ch. 3's clustered indexes.

8. Detailed component design

8.1 Funnel evaluator

For each user, walk events in time order, advance a step pointer when the next required event matches and falls inside the conversion window. Emit (user, max_step_reached). Aggregate. This is a stateful stream operator and runs in O(N) — DDIA Ch. 11 sessionization pattern.

8.2 Schema-on-read

Properties are stored as a Map; per-project a metadata service tracks observed keys for autocomplete. Hot keys are promoted to materialized columns nightly (DDIA Ch. 10 batch job) to skip JSON parsing.

8.3 Cohort retention

Pre-compute a daily bitmap (RoaringBitmap) per (project, event_name, day) of active user_ids. D7 retention = popcount(bitmap[D] AND bitmap[D+7]) / popcount(bitmap[D]). Bitmaps fit in < 1 GB even for 100M-user products.

9. Scaling & bottlenecks

  • Schema explosion: cap distinct property keys per project at 1000; reject the rest with a warning.
  • Skewed users: a "test" user_id with 1B events kills the partition — detect and bucket-shard suffixed keys.
  • Late events: mobile reconnects after 7 days — Flink uses event-time with 7d allowed lateness, then drops; reconciliation batch job repairs Parquet daily.
  • Query concurrency: small per-tenant ClickHouse cluster + global query queue; expensive queries spill to Trino.

12. Talking points for the interviewer

  • "Funnel queries are the single most important workload, so I clustered on user_id even though it hurts time-range scans — design follows the dominant query."
  • "I'd ship with schema-on-read because customers will rebel against rigid schemas; we promote hot keys nightly."
  • "The trickiest bug at scale is late-arriving mobile events; I'd reserve a daily reconciliation batch job from day 1."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
ClickHouse for interactive analytics Funnel and cohort queries scan 50B events/day across 100s of properties. ClickHouse's columnar MergeTree with vectorized execution returns 30-day funnels in <2 s p95; a row store would take minutes and saturate IO at this scale. Postgres (row store, OOM at 1B rows); Druid (weaker JOINs for funnel); Snowflake (10× cost, slower point-queries)
Parquet on S3 for cold/historical tier Past 90 days of events (~5 PB) is queried weekly, not interactively. Parquet's columnar layout + Snappy compression yields 8× size reduction over JSON; partition pruning by date lets ad-hoc Spark jobs read only the relevant slice instead of the full 5 PB. ORC (smaller ecosystem); JSON.gz (no column pruning); Avro (row-oriented, slow scans)
ORDER BY (user_id, event_time) clustering in ClickHouse Funnel queries are the dominant workload — 60% of all queries pull "all events for user X". Clustering on user_id lets the engine read each user's events as a contiguous run, turning a fan-out scan into a single sequential read. ORDER BY (event_time) (kills funnel perf); no clustering (10× slower point lookup)
Schema-on-read with promoted hot keys Customers send unpredictable JSON properties; a rigid schema would block onboarding. Storing the raw JSON blob plus nightly-promoted top-50 properties as typed columns gives schema-on-write performance for hot paths and schema-on-read flexibility for the long tail. Strict schema (slow customer onboarding); pure schema-on-read (10× slower queries)
Kafka as the durable event log 50B events/day = 580K events/sec average, 5M peak. Kafka decouples ingest from ClickHouse, lets the warehouse fall behind safely during compaction, and feeds independent consumers (real-time alerting, ML feature store, audit archive) from one source of truth. Kinesis (lower per-partition throughput); RabbitMQ (no replay for backfills)
Flink for streaming sessionization & real-time funnels Customers expect today's data in dashboards within minutes. Flink's event-time windows + watermarks correctly sessionize at 5M events/sec with exactly-once semantics; Spark micro-batch would add 30-60 s of dashboard staleness. Spark Streaming (micro-batch, weaker event-time); Kafka Streams (no large state)
Spark on S3 for nightly batch reconciliation (Lambda arch) Late-arriving mobile events (up to 7 days late) silently break streaming aggregates. A nightly Spark job over immutable Parquet recomputes "true" daily counts; product analysts demand reproducible numbers across definition changes — that needs batch-on-immutable-input. Streaming-only (Kappa) (no reconciliation for late events); Hive (10× slower than Spark)
RoaringBitmap for cohort/funnel user sets Retention queries intersect "users who did X on day 0" with "users who returned on day N" — sets of 10M-100M user IDs. Roaring stores these in 100 MB and intersects in milliseconds via popcount; without bitmaps a JOIN approach is O(N×M) and times out. JOIN on user_id (slow at 100M users); HyperLogLog (no exact intersection)
HyperLogLog sketches for cardinality (DAU/MAU) "Unique users in last 30d" across 50B events at 1% accuracy fits in 16 KB per cohort; merging is associative so we precompute per-day sketches and union at query time, returning sub-second instead of scanning 1.5T events. COUNT(DISTINCT) (full scan); Theta sketches (heavier, marginal accuracy gain)
Schema registry (Confluent / Avro) 50+ producer teams emit events; without a schema contract a single bad rename breaks every consumer. Centralized schema registry with backward-compat checks prevents the "broken pipeline at 3 AM" failure mode that costs analyst-hours. Free-form JSON (no contracts); Protobuf without registry (no compat enforcement)
Materialized views for top-N dashboards The top 20 dashboards account for 80% of query load. ClickHouse materialized views maintain pre-aggregated daily/hourly rollups so dashboard refresh stays <200 ms even as raw event volume grows; without them the same query would scan billions of rows every refresh. Query-time aggregation (slow); separate Druid cube (extra system to operate)
Iceberg/Hive table format for the data lake Schema evolution and partition compaction over 5 PB of Parquet must be transactional — otherwise readers see half-written partitions. Iceberg's snapshot isolation gives ACID over S3 so daily reconciliation jobs can rewrite without blocking analyst queries. Plain Parquet directories (no atomicity); Delta Lake (good alt, weaker multi-engine)
