Design Distributed Tracing (OpenTelemetry / Jaeger — span propagation)

Hard analytics

1. Problem statement

Build a distributed tracing platform like Jaeger / Honeycomb / Tempo that captures causally-linked spans across hundreds of microservices, joins them into traces, and exposes them via a UI for debugging latency and errors. At Uber/Lyft scale, services emit billions of spans per day. The system must handle bursts during incidents (when traces matter most), yet keep storage costs bounded via sampling.

2. Functional requirements

  • OpenTelemetry-compatible ingest (OTLP/gRPC).
  • Trace lookup by trace-id; trace search by service, operation, tag, duration, error.
  • Service dependency graph derived from spans.
  • Tail-based sampling: keep all error traces, p99-slow traces, and 1% baseline.
  • Retention 7 days hot, 30 days cold.

3. Non-functional requirements

  • Ingest: 5B spans/day baseline, 50B during incidents.
  • Trace assembly latency: a complete trace queryable within 30 s of last span.
  • Lookup p99: < 300 ms by trace-id.
  • Search p99: < 3 s on 24h window.
  • Loss budget: ≤ 0.1% of sampled traces.

4. Capacity estimation

  • 5e9 spans * 1 KB = 5 TB/day raw; with 10% post-tail-sampling kept, ~500 GB/day persisted.
  • Span eps: ~58K baseline, ~580K incident peak.
  • Trace fan-out: avg 30 spans / trace, p99 1500 spans (think Uber dispatch).
  • Hot index: ~24h * 500 GB = 12 TB across cluster.

5. API design

POST  /v1/traces            (OTLP protobuf batch)
GET   /api/traces/{traceId}                            → full trace
GET   /api/traces?service=&op=&minDur=&tag=err:true    → trace list
GET   /api/dependencies?lookback=1h                    → service graph

6. High-level architecture

[OTel SDK in app] -> [OTel Collector sidecar] -> [Regional Gateway] | v [Kafka spans-raw, partitioned by trace-id] | +-------------------------+--------------------------+ v v [Trace Assembler (Flink)] [Tail Sampler] sessionize 30s/trace decide keep/drop per trace | | +-----------------+----------------------------------+ v [Object Store: traces/{traceId}.pb] | [Index: ES/Cassandra (svc,op,tag) -> traceId] | [Query API + UI]

Apps emit spans through OpenTelemetry SDKs. A sidecar Collector batches and forwards to the regional gateway, which produces to Kafka partitioned by trace-id so all spans of a trace land in the same partition and the same Flink operator instance. The assembler windows by trace-id with a 30-second inactivity timeout, then ships the complete trace to object storage and emits index entries.

7. Data model & storage

7.1 Span

{
  trace_id:      [16]byte,        // W3C traceparent
  span_id:       [8]byte,
  parent_span:   [8]byte,
  service:       string,
  operation:     string,
  start_ns:      int64,
  duration_ns:   int64,
  status:        OK|ERROR,
  attrs:         map<string, any>,
  events:        [{ts, name, attrs}],
  links:         [{trace_id, span_id}]
}

7.2 Index

service_op_idx (Cassandra):
  PK = (service, op, day_bucket), CK = (start_ts DESC, trace_id)
tag_idx        (Elasticsearch): inverted index over flattened attrs

8. Detailed component design

8.1 Trace assembly

The Flink operator keys spans by trace-id and uses an event-time session window with 30 s gap. State is RocksDB-backed. On window close, it serializes the trace, computes derived metrics (root duration, error count, critical path) and emits to two sinks: blob store and indexer. This is a textbook stream sessionization (DDIA Ch. 11).

8.2 Tail-based sampling

A tail sampler subscribes to the same Kafka topic and decides per trace whether to keep it. Policies: keep if any span has status=ERROR, keep if root duration > service p99, keep with probability 0.01 otherwise. Decisions stream into a "keep-set" Kafka topic, joined into the assembler so dropped traces never reach storage.

8.3 Service graph

A separate stream job derives (caller_service, callee_service, latency_histogram) from each span pair and writes hourly aggregates — a small classic batch reduction (DDIA Ch. 10) that powers the dependency map.

9. Scaling & bottlenecks

  • Hot trace-id partition: a runaway trace with 10M spans melts one Kafka partition. Mitigation: per-trace span cap at the SDK (e.g., 50K), and a producer-side circuit breaker.
  • Long-running traces: async pipelines may keep a trace open for hours; we close the window at 5 min and stitch later via "trace-fragment" merging in queries.
  • Indexer back-pressure: ES bulk > 80% queue triggers shedding to a slow-path Kafka topic.
  • Cardinality: attribute key allow-list per service prevents tag-explosion.

10. Trade-offs

DecisionProsCons
Tail samplingKeeps interesting traces, drops noiseNeeds whole trace before deciding → latency & memory
Head sampling (alt)O(1) memory, decided at rootUniform random; misses error traces
Partition by trace-idLocal sessionization, no cross-shard joinsSkew if traces are heavy-tailed
Blob store + secondary indexCheap storageTwo systems to keep consistent

11. DDIA references

  • Ch. 11 Stream Processing — sessionization by key (trace-id), event-time windowing, watermarks for the inactivity gap, exactly-once via Kafka offsets + Flink checkpoints, materialized service-graph view as a streaming aggregation.
  • Ch. 10 Batch Processing — daily compaction of sampled traces into Parquet for long-term cold queries; offline rebuild of indexes after schema change is a Hadoop-style batch job over immutable inputs.
  • Ch. 7 Transactions — index + blob write must be atomic-ish; we use the blob as source of truth and reconcile the index in batch.
  • Ch. 8 Trouble with Distributed Systems — clock skew between services makes start_ns ordering unreliable; we use parent_span for causal order, never wall-clock.

12. Talking points for the interviewer

  • "Sampling is the central trade-off: I'd default to tail sampling because head sampling silently drops the traces engineers actually need."
  • "Causality comes from the parent_span pointer, not timestamps — clocks across hosts disagree by tens of ms."
  • "During incidents traffic 10×es; the same incidents are when traces matter most, so back-pressure must shed low-value traces, not random ones."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
OpenTelemetry SDKs (OTel) With 10K+ services in 7 languages, a vendor-locked agent is suicide. OTel gives one wire format (OTLP) and one context-propagation spec (W3C traceparent) so every service emits the same span shape; without this, trace assembly across polyglot fleets is impossible. Zipkin libs (older, weaker context propagation); vendor SDKs (lock-in, no Rust/Go parity)
OTel Collector (agent + gateway tier) 10M spans/sec ingress means we can't push raw spans to backend. The collector batches, compresses, and applies head-sampling close to the source, cutting egress bandwidth ~10× before traces ever cross AZ boundaries. Direct SDK→backend (network amplification); Fluentd (no trace-aware semantics)
Tail sampling (decision after full trace assembled) Head sampling (1%) silently drops the slow/failing traces engineers actually need. Tail sampling buffers spans 30 s, then keeps 100% of errors and p99-latency traces; we throw away cheap healthy traffic instead of expensive incidents. Pure head sampling (loses anomalies); 100% retention (10× cost, no extra signal)
Kafka between collector and backend During incidents trace volume spikes 10× — exactly when traces matter most. Kafka absorbs the burst at 1 GB/sec/topic and lets the indexer fall behind without dropping; without it the indexer becomes the bottleneck and back-pressure shreds production. gRPC direct (no buffering); Kinesis (lower throughput per partition, higher latency)
Cassandra for span storage by trace_id Span lookup is point-query by trace_id (16 bytes); we never scan. Cassandra's LSM + partition-by-trace_id gives single-digit-ms read at PB scale, and tunable consistency (CL=ONE) is fine because traces are advisory, not transactional. Postgres (insert rate too low at 10M spans/sec); HBase (operational pain); S3 (read latency)
Elasticsearch for service/operation/tag indexes Engineers search by service=checkout AND http.status>=500 AND duration>1s. ES inverted indexes resolve this across 100B+ spans/day in <500 ms; Cassandra alone cannot answer attribute-search queries. Cassandra secondary index (high write amp); Lucene-on-HDFS (no live ingest)
Jaeger UI / Tempo-style query layer Engineers expect a flame-graph UI; Jaeger's UI is the de-facto standard and integrates with OTel data models for free. Building a custom UI is months of work that doesn't move the SLO needle. Custom React dashboard (re-invents wheel); Zipkin UI (weaker for tag search)
Parent-span pointer for causality (not timestamps) Clocks across hosts skew by tens of ms; ordering spans by start-time produces wrong parent/child trees. The explicit parent_span_id in OTel context is the only correct causality signal at fleet-wide scale where NTP can't be trusted to ms. Wall-clock ordering (wrong on clock drift); HLC (overkill for read-time assembly)
30-day hot retention + S3 cold tier 95% of trace lookups are within 24 h of the span. 30 days hot in Cassandra/ES (~600 TB) covers all incident postmortems; older traces go to S3 Parquet at 1/10 the cost, restored on demand for rare audits. 90-day hot (3× cost, no extra value); no archive (compliance failure)
Adaptive rate-limiter per service in collector One service emitting 1M spans/sec from a hot loop must not drown the cluster. A token-bucket per service tag in the collector sheds that service's spans first, preserving the long tail of other services' traces during incidents. Global rate limit (noisy neighbor wins); per-pod limit (no aggregate protection)
RoaringBitmap for trace-id set operations in tail sampler The tail sampler holds ~30 s of trace_ids in memory (~30M IDs). Roaring bitmaps store these in <100 MB and let us union "error trace IDs" with "slow trace IDs" in microseconds before flushing the kept set. Hash set (3× memory); sorted array (slow union)
W3C Trace Context propagation HTTP, gRPC, Kafka headers all need the same trace_id format; W3C is the only spec implemented across every language SDK. Without it cross-language calls (Go → Python → Node) lose the trace at the boundary. B3 single-header (Zipkin legacy); custom header (no SDK support)
