Design Log Aggregation Pipeline (Splunk-style, Elasticsearch + Kafka)

Hard analytics search

1. Problem statement

Design a centralized log platform like Splunk / Elastic Stack / Loki for an enterprise running 50K hosts and producing 100–500 TB of logs/day. Engineers need full-text search, structured filters, dashboards, alerting, and long-term archival. The system must keep ingestion alive even when downstream search is degraded — logs are the lifeline during outages.

2. Functional requirements

  • Collect logs via Fluent Bit / Vector / Filebeat agents.
  • Parse (regex/grok), enrich (k8s metadata, geoip), and structure into JSON.
  • Full-text + structured search with sub-second latency on 24h window.
  • Saved searches, alerts, dashboards (Kibana-style).
  • 30-day hot retention, 1-year cold archive in S3, replayable to hot tier.
  • Role-based access by index pattern.

3. Non-functional requirements

  • Ingest: 5 PB/month ≈ 200 TB/day, 5M log lines/sec peak.
  • Search latency: p95 < 2 s for 1h, < 10 s for 24h.
  • Indexing lag: log queryable within 30 s.
  • Durability: zero loss for committed lines; agents retry up to 24 h.
  • Multi-tenancy: 200 teams isolated by index + RBAC.

4. Capacity estimation

  • 200 TB/day raw; ~50 TB/day after gzip on disk; ~80 TB/day Lucene-indexed.
  • Lines: 5M/s baseline, 20M/s peak → Kafka with 2048 partitions @ 10 MB/s each.
  • Hot ES: 30d * 80 TB = 2.4 PB; ~120 nodes × 24 TB SSD with 1 replica.
  • Cold S3: 12 mo * 50 TB compressed = 18 PB / yr.

5. API design

POST /_bulk                       (Elastic _bulk JSON)
GET  /search?q=service:checkout AND status:5xx&from=now-1h
POST /alerts  { query, threshold, schedule }
POST /restore { date_range, dest_index }   // rehydrate cold tier

6. High-level architecture

[App] -> [Fluent Bit agent] --> [Edge Aggregator] --> [Kafka logs.raw, 2048 parts] | +-----------------------------------------------+--------------------------+ v v [Parser/Enricher (Flink)] [Cold Archiver] | | v v [Kafka logs.structured] [S3 Parquet by hour] | +--------> [ES Indexer Workers] --> [Elasticsearch hot tier] | ^ +--------> [Alert Evaluator] | [Kibana / Search API]

Agents tail files, batch every 5s, and ship to a regional aggregator that authenticates and writes to Kafka. Kafka is the durable buffer — if Elasticsearch falls behind or crashes, the pipeline doesn't lose data; it just reads from a lagged offset later. This decoupling is the single most important architectural choice (DDIA Ch. 11 "log as buffer").

7. Data model & storage

7.1 Index strategy

Index pattern: logs-{tenant}-{yyyy.MM.dd}
Shards/index : ceil(daily_size_GB / 30)        # ~30 GB target
Replicas     : 1 (RF=2)
Mapping      : @timestamp, service, host, level, msg(text), trace_id(keyword), labels(flat)

7.2 Cold tier

The cold archiver consumes logs.structured and writes hour-partitioned Parquet to S3 with Hive-style paths s3://logs/tenant=X/dt=2026-05-06/hr=14/. Athena/Trino can query directly; the restore API rehydrates a date range back into a temporary ES index for interactive search.

8. Detailed component design

8.1 Parser/Enricher

Stateless Flink job applying per-tenant grok patterns hot-loaded from a config service. Failed parses go to a logs.parse_dlq topic with the raw line preserved — no log line is silently lost. K8s metadata is enriched via a streaming join against a CDC stream from the API server (DDIA Ch. 11 stream-table join).

8.2 Hot/warm/cold tiering

ILM policy: 0–3 d on NVMe nodes, 3–30 d on HDD warm nodes, >30 d frozen on S3-backed searchable snapshots. Index rolls over at 50 GB or 1 day, whichever first.

8.3 Search performance

Per-shard query parallelism + cross-cluster search for tenants spanning multiple clusters. Time-range pruning on index name keeps fan-out narrow. Frequent saved searches are pre-aggregated into rollup indexes by a nightly batch job.

9. Scaling & bottlenecks

  • Bursty noisy neighbors: per-tenant token bucket at the edge; on overage, we keep ingesting to Kafka but shed indexer throughput — logs survive even if search lags.
  • Hot shard: a single service emitting 1M lines/s overwhelms one ES shard; mitigate with rollover-on-size and index-level routing by hash(host).
  • Mapping explosion: uncontrolled JSON keys blow up the field map; we use the flattened field type for free-form labels.
  • Replay during outage recovery: Kafka retention 72h gives time to rebuild ES; replay throttled to avoid double-burst.

11. DDIA references

  • Ch. 11 Stream Processing — Kafka as the durable log decoupling producers from indexers; backpressure handled by lag, not by drop. Stream-table joins for k8s enrichment. Dead-letter queues for un-parseable records.
  • Ch. 10 Batch Processing — nightly rollup indexes (count/sum per service/level/minute) are MapReduce-style aggregations. Cold-tier Parquet builds + Trino queries are batch-on-immutable inputs, exactly the pattern in Ch. 10.
  • Ch. 3 Storage & Retrieval — Lucene segment files = LSM-style immutable sorted runs with merging; inverted index is the canonical search structure.
  • Ch. 5 Replication — ES primary/replica is single-leader per shard; we accept brief unavailability on primary failover for stronger consistency.

12. Talking points for the interviewer

  • "Logs are the most valuable when the system is broken, so the pipeline must keep accepting them even with ES down — that's why Kafka is non-negotiable."
  • "I'd push back on indexing everything: a 30-day searchable + 12-month grep-able tier saves an order of magnitude in cost."
  • "Mapping explosion has destroyed more ES clusters than disk failure; I'd ship with strict per-tenant field budgets."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Kafka as durable buffer between agents and indexer Logs are most valuable when the system is broken, so the pipeline must keep accepting them even with ES down. Kafka absorbs 200 TB/day at >2 GB/sec sustained and lets the indexer fall arbitrarily behind without losing logs — this decoupling is non-negotiable. Direct agent→ES (back-pressure crashes apps); Fluentd buffer files (no replay across nodes)
Elasticsearch for hot 30-day searchable tier SREs grep recent logs interactively; only an inverted index gives sub-second free-text on 6 PB of recent data (200 TB/day × 30 days). ES with hot/warm/cold node classes is the only mature OSS option that handles this scale with ad-hoc query patterns. Loki (label-only search, slow on full-text); Splunk (10× cost); Solr (smaller ops community)
S3 + Parquet for 12-month grep-able cold tier Indexing every log at full retention would cost ~$50M/yr; a 30-day searchable + 12-month grep-able tier saves an order of magnitude. Parquet on S3 with date partitioning makes Athena/Presto grep over old logs cheap and on-demand. Index everything in ES (10× cost); Glacier-only (retrieval latency too high for ops)
ILM (Index Lifecycle Management) policies 200 TB/day of new indices means manual rotation is impossible. ILM rolls over indices at 50 GB or 1 day, transitions hot→warm at 7 days (cheaper hardware), and force-merges to fewer segments — all automatically, preventing the "1000 tiny shards" failure mode that kills ES clusters. Manual rotation scripts (operational toil, error-prone); Curator (deprecated)
Strict per-tenant field budget (mapping limits) Mapping explosion has destroyed more ES clusters than disk failure: one app emitting error.line_42_var_x-style dynamic field names balloons mappings to 50K+ fields and OOMs masters. A 1000-field-per-tenant cap with rejection at the indexer is mandatory at 200 TB/day. Dynamic mapping (cluster death); pre-defined schema (blocks dev velocity)
Fluent Bit / Vector agent on every host Tens of thousands of hosts emitting logs; we need a 1-2 MB-RSS agent that handles backpressure, multiline, and Kafka output. Fluent Bit's C core does this at <1% CPU per host; a heavy JVM agent would tax production hosts at 200 TB/day fleet emission. Filebeat (heavier, JVM); Logstash on host (200 MB RSS, kills app density)
Structured JSON logging + canonical fields Free-form text logs force regex parsing at index time, doubling indexer CPU. Mandating JSON with 8 canonical fields (ts, level, svc, trace_id, ...) cuts indexer cost ~40% and unlocks join-by-trace_id with the tracing system at zero additional infra. Free text + Grok (CPU-heavy); CEF/syslog (legacy, weaker tooling)
Time-based ES indices (per-day, per-tenant) Most queries are "last 1h on service X"; per-day indices let the query coordinator skip 95% of shards via routing. Per-tenant prefix lets ILM expire one customer's data without touching others, which matters at 200 TB/day shared infra. Single huge index (no shard pruning); per-hour indices (10K+ shard explosion)
OpenTelemetry log SDK + trace_id correlation Engineers debug by jumping from a slow trace span to the logs of that exact request. OTel's log SDK injects trace_id automatically; a free-form logger requires manual MDC plumbing in 7 languages and inevitably misses paths. Per-language MDC (inconsistent coverage); no correlation (debugging is grep-and-pray)
RoaringBitmap inside ES for terms aggregations "Top 10 services by error count last 1h" over billions of docs needs efficient set ops on doc-id lists. ES uses Roaring under the hood for filter caches; this is why filter queries are 10×+ faster than query on the same predicate at our scale. Sorted long arrays (slow AND); plain bitmaps (memory blowup on sparse fields)
Kafka Connect / Logstash to S3 sink with hourly compaction Streaming Kafka→S3 produces millions of tiny Parquet files; Athena tasks-per-file would explode query cost. Hourly compaction batch rewrites to ~512 MB Parquet, which is the file size cold-tier query engines actually run well on. No compaction (Athena query timeouts); Avro (no column pruning for cold grep)
Per-tenant rate limiting at the indexer One application in a debug-spam loop can emit 10× its baseline and starve every other tenant's indexing. A token-bucket per tenant on the indexer guarantees fair share at 200 TB/day fleet-wide and prevents one bad deploy from blinding the rest of the org. Global queue (noisy neighbor wins); host-level limit (no aggregate tenant view)
