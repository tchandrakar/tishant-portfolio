Design Pandora (radio-style personalized streaming)

Hard audio-streaming recommendation

1. Problem statement

Design Pandora, a lean-back personalized radio service. Unlike Spotify, the user does not pick exact tracks — they seed a "station" with an artist/song, and the system streams a never-ending sequence of related tracks. Users give thumbs-up/thumbs-down feedback that immediately shapes the station. The core technical asset is the Music Genome Project: ~450 hand-labeled musicological attributes per track. Skips are rate-limited (DMCA radio compliance). The system must handle station generation, streaming, real-time feedback adaptation, and audio ad insertion for free tier.

2. Functional requirements

  • Create a station from any seed (track, artist, genre, or mix of seeds).
  • Continuous playback of related tracks; no user-chosen track order.
  • Thumbs-up / thumbs-down feedback adjusts station immediately.
  • Skip enforcement: max 6 skips per station per hour (free tier).
  • "My Stations" persistence across devices.
  • Audio ad insertion every ~15 minutes for free users.
  • "Why this song?" explanation surfacing genome attributes.
  • Replay/rewind for paid tier (Pandora Plus/Premium).

3. Non-functional requirements

  • Latency: next-track selection under 100ms (must arrive before current track ends to avoid silence).
  • Scale: 50M MAU, 6M paid, peak 2M concurrent stations.
  • Availability: 99.95% playback.
  • Compliance: DMCA — no on-demand selection on free tier; skip caps; no playing the same artist twice within 3 hours of station start.
  • Personalization quality: thumbs-down adjustment within next 1–2 tracks.

4. Capacity estimation

  • Catalog: 2M tracks × 2 bitrates (64k, 192k AAC) × ~3MB avg = ~12 TB.
  • Tracks played/day: 15M DAU × 30 tracks/day = 450M tracks/day = ~5K plays/sec avg.
  • Egress: 450M × 3MB = 1.35 PB/day from CDN.
  • Feedback events: 15M DAU × 10 thumbs/skips = 150M/day = ~1.7K writes/sec.
  • Station generation calls: next-track called every ~3.5min × 2M concurrent = ~10K QPS, plus instant on skip/thumbs.
  • Genome data: 2M tracks × 450 attributes × 4B = 3.6 GB — entirely in memory across reco fleet.
  • Ad impressions: 30M ads/day for free tier = ~350 QPS to ad server.

6. High-level architecture

[Client] --> [CDN audio] -- [S3] | +--> [API GW] | +--> [Station Svc] --> [Redis: live station state] | | | +--> [Reco Engine] -- [Genome KV (in-mem)] -- [User Profile Store: Cassandra] | +--> [Feedback Svc] --> [Kafka: feedback events] --> user profile updater +--> [Skip Counter] --> [Redis: per-station counters with TTL] +--> [Ad Svc] --> [Ad Server] [Kafka] -> [Spark batch] -> nightly user-taste model retrain -> pushed to Reco Engine

Walk-through: client requests next track. Station service consults the live in-memory station state (last 50 tracks, current taste vector) and asks the Reco Engine for a candidate set, filters by DMCA rules and recently-played, and returns the chosen track + signed CDN URL. Feedback events flow through Kafka and update the user profile within a couple of seconds.

artist_index[artist_id] = list<track_id>
3.6 GB total, replicated to every reco node.

7.4 User taste profile (Cassandra)

8. Detailed component design

8.1 Reco engine (next-track selection)

The station's tasteVector is a 450-dim weighted vector (genome attributes). Candidate generation: cosine similarity between station vector and all 2M track vectors, top-K (~500) returned in ~5ms via SIMD batch dot-products on the in-memory KV. Filters applied: not played in last 50 tracks; same artist not within last 3 hours; not in user's "banned" list (thumbs-down history). Final selection samples weighted by similarity to inject diversity (epsilon-greedy 10%).

8.2 Real-time feedback adaptation

Thumbs-up: station vector += 0.05 × track_genome (normalized).
Thumbs-down: station vector -= 0.10 × track_genome, plus add track to "banned" set in Redis, plus skip current.
Updates land in Redis synchronously so the next /next call sees them immediately. Async path persists to Cassandra for cross-device consistency.

8.3 DMCA-compliant skip enforcement

Skip counter in Redis: INCR skips:{stationId}:{hourBucket} with 1h TTL. If >6 in current hour, return {allowed:false, retryAfterSec:N}. Premium users bypass.

8.4 Ad insertion

Free tier: every ~15 min of listen time, the next-track response includes adBreak: {adId, durationMs, adUrl} instead of (or before) the next track. The Ad Service decides creative based on user demographics and current track context.

8.5 Why this song?

Computed at request time: top-5 genome attributes where station vector and track vector both have high values, e.g., "syncopated rhythm + minor key + female vocals". Cached for 24h since vectors evolve slowly.

9. Scaling & bottlenecks

  • Cosine over 2M tracks per request: SIMD + early-exit beam over 450 dims; or pre-cluster tracks (k-means 1000 clusters) and only score top 5 clusters.
  • Genome memory: 3.6 GB fits one node, but we replicate per region for latency. Updates rare (curators add tracks).
  • Redis station state hot keys: sticky-route session to a partition; Redis Cluster of 64 shards.
  • Skip enforcement under retries: Lua script for atomic INCR+CHECK to avoid double-counts.
  • New-track cold start: curators backfill 450 attributes within 24h; meanwhile the track is shadow-eligible only for stations with high seed similarity.

12. Talking points for the interviewer

  • "Pandora's edge is explainability via the Music Genome — an ML embedding could match accuracy but loses the 'why this song' UX. I'd hybridize: genome for explanation + neural embeddings for candidate gen at scale."
  • "DMCA radio rules are not a footnote — skip caps and artist-repeat windows are baked into the request path, not bolted on."
  • "Feedback latency is the personalization moat — I'd treat sub-second taste-vector updates as a hard SLO."
  • "For multi-seed stations (e.g., Beatles + Radiohead) I'd weight seed influence dynamically based on recent thumbs — users naturally pull stations toward one seed over time."
