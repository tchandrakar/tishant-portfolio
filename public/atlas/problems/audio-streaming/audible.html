Design Audible (audiobook platform with offline)

Hard audio-streaming file-storage payments

1. Problem statement

Design Audible, an audiobook platform where users purchase or subscribe (1 credit/month) to long-form audiobooks (avg 10 hours, 300 MB). Books are bought once and owned forever; offline listening is the primary mode (commutes, flights, runs). The system must handle a large catalog (~1M titles), per-user library entitlements, chaptered playback with bookmark/note sync across devices, Whispersync (resume position synced even mid-chapter across phone/Kindle/Echo), and integrations with the Amazon retail backbone for purchases and Audible Originals.

2. Functional requirements

  • Browse catalog by genre, author, narrator, length; search.
  • Purchase audiobook (one-tap with Amazon account) or redeem monthly credit.
  • Stream and/or download audiobook (chaptered MP4-AAC files).
  • Whispersync: resume position synced across all user devices in seconds.
  • Bookmarks, notes, clip-sharing of 30s snippets.
  • Variable playback speed (0.5x to 3.5x) without pitch shift.
  • Sleep timer, chapter navigation, table of contents.
  • Listening stats, badges, recommendations.
  • DRM-protected downloads bound to Audible/Amazon account.

3. Non-functional requirements

  • Latency: first byte under 300ms (less critical than Spotify — users listen for hours).
  • Scale: 90M MAU, 20M paying members, 5M concurrent at peak.
  • Availability: 99.95% playback; 99.99% library/entitlement (purchases).
  • Durability: 11 nines for master audio; books stay in catalog 50+ years.
  • Offline-first: 80% of listening hours are from downloaded files.
  • Whispersync: position propagates across devices within 5s p95.

4. Capacity estimation

  • Catalog: 1M titles × avg 300 MB × 2 bitrates (32k, 64k AAC) = ~600 TB + lossless masters ~6 PB.
  • Daily downloads: 20M DAU × 0.3 books/day = 6M downloads/day × 300 MB = 1.8 PB/day egress.
  • Streaming bandwidth: 5M concurrent × 64 kbps = 320 Gbps peak.
  • Library/entitlement reads: 20M DAU × 5 reads/day = 100M reads/day = ~1200 QPS avg, 5K peak.
  • Whispersync writes: 20M DAU × 50 position updates/day = 1B/day = ~12K writes/sec avg.
  • Purchases: 1M txns/day = ~12 TPS avg, 200 TPS peak (e.g., Black Friday).

6. High-level architecture

[App: phone, Kindle, Echo, Web] | +-- HTTPS --> [Edge CDN] -- chapter MP4s -- [Origin: S3 + Glacier] | +-- HTTPS --> [API GW] | +--> [Catalog Svc] ---- [DynamoDB: books] +--> [Library Svc] ---- [Aurora MySQL: entitlements] +--> [Purchase Svc] --- [Amazon Pay] -- [Kafka: orders] +--> [DRM Svc] ----- [KMS] -- license tokens +--> [Whispersync Svc]- [DynamoDB Global Tables] +--> [Reco Svc] ----- [SageMaker endpoints] [Kafka: listen-events] -> [Kinesis Analytics] -> [S3] -> [Athena]

Walk-through: purchase flow goes through Amazon Pay; on success, an entitlement row is written and a Kafka event triggers download permission. Download manifest is signed per-chapter and DRM-wrapped. Whispersync uses DynamoDB Global Tables for last-write-wins multi-region replication of position. Listen events stream into a data lake for reco training and royalty payouts.

7. Data model & storage

7.1 Catalog (DynamoDB)

PK: asin   SK: META | CHAPTER#nnn
Attributes: title, authors[], narrator, durationMs,
            chapterTitle, chapterStartMs, audioBlobId, bitrate

7.2 Library/entitlement (Aurora MySQL)

CREATE TABLE entitlement (
  user_id BIGINT, asin VARCHAR(16),
  acquired_at DATETIME, source ENUM('credit','purchase','gift'),
  PRIMARY KEY (user_id, asin),
  KEY idx_user_recent (user_id, acquired_at DESC)
);

7.3 Whispersync (DynamoDB Global Tables)

PK: user_id   SK: asin
Attributes: pos_ms, ts, device_id, version
Conditional update: only accept if incoming.ts > stored.ts

7.4 Why these stores?

DynamoDB Global Tables give multi-region active-active with last-writer-wins — perfect for Whispersync where we want fast cross-device sync and tolerate the rare clock-skew loss. Aurora MySQL for entitlements because purchases must be transactionally consistent with payment. S3 for audio with intelligent tiering — rarely-listened-to long-tail moves to Glacier.

8. Detailed component design

8.1 Whispersync engine

Each client emits position updates every 10s + on pause/seek. The Whispersync service applies a conditional write keyed on monotonic logical timestamp (ts, device_id). Conflict resolution = highest ts wins (Lamport-ish). Other devices subscribe via long-poll or MQTT for sub-5s push. Bookmarks/notes use the same model but stored as a list with CRDT-style merging.

8.2 Download & DRM

On download request, DRM service mints an offline license bound to device_id + customer_id, valid for the entitlement lifetime. Audio chapters are encrypted with AES-128 CTR; content key is wrapped under a device key. Loss of license = book unplayable. Re-authorization is required if user removes device from Amazon account.

8.3 Variable speed without pitch shift

Done client-side using a phase-vocoder (WSOLA on iOS/Android). The server simply ships standard AAC; no per-speed transcoding needed. This is a critical UX feature — 40% of users listen at 1.5x.

8.4 Recommendation

Two-tower over user listen history (last 100 books) and book features (genre, narrator, length). SageMaker batch transform refreshes daily; online ranker uses 24h-recent listens for "Because you finished X". Long-form means signals are sparser than music — we lean on completion rate and 4-week retention as labels.

9. Scaling & bottlenecks

  • Cold storage tiering: 80% of catalog is in long-tail; auto-promote to S3 Standard when an entitlement is granted.
  • Whispersync write hotspot: partition key is user_id, so naturally spread; provisioned WCU autoscaled.
  • Black Friday surge: purchase service scales horizontally behind SQS to absorb spikes; entitlement write is async-confirmed.
  • Download manifest signing: cache signed URLs per (user, asin, device) for 24h.
  • Chapter granularity: fragment into ~5MB chapter slices to enable resumable downloads on flaky mobile networks.

12. Talking points for the interviewer

  • "Audible is fundamentally a digital ownership product, not a streaming product — entitlements outlive subscriptions, so I treat the entitlement DB like a financial system of record."
  • "Whispersync is the moat; I'd over-invest in low-latency cross-device sync because 'losing my place' is the #1 churn driver."
  • "DRM is necessary but adversarial — I'd build a piracy-detection signal pipeline (impossible-listen-rate, geo-jumps) and revoke licenses asynchronously."
  • "For new-narrator discovery I'd run a multi-armed bandit on the homepage rail rather than a static reco model — long-form has high commitment cost so exploration must be deliberate."
