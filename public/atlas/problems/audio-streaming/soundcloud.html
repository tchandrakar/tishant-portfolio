Design SoundCloud (UGC audio + waveform)

Hard audio-streaming file-storage social-network

1. Problem statement

Design SoundCloud, a user-generated-content audio platform where any creator (DJ, podcaster, musician) uploads tracks, others stream/comment/repost. Distinct from Spotify: there is no curated label catalog — the catalog is the upload pipeline. The signature feature is an interactive waveform with timestamped comments. The system must handle high-throughput uploads (long DJ sets up to 6 hours), efficient transcoding to multiple formats, abuse and copyright detection (audio fingerprinting), social graph (follow/repost), and a "Stream" timeline of recent posts from followed users.

2. Functional requirements

  • Upload audio (MP3, WAV, FLAC, up to 4 GB / 6 hours).
  • Server-side transcode to streaming formats; auto-generate waveform PNG/JSON.
  • Stream tracks with adaptive bitrate; embeddable player on third-party sites.
  • Timestamped comments anchored to waveform position.
  • Like, repost, follow, share to social.
  • Personalized "Stream" timeline of new posts from followed creators.
  • Search by track, user, tag.
  • Copyright fingerprint scan on every upload.
  • Creator analytics: plays, geo, referrers.

3. Non-functional requirements

  • Latency: first byte under 250ms; waveform JSON under 100ms.
  • Scale: 175M MAU, 30M creators, 12 hours of audio uploaded per minute.
  • Availability: 99.95% playback; 99.9% upload.
  • Durability: 11 nines for original uploads; transcodes are reproducible.
  • Fairness: small creators must surface in feeds; not a winner-take-all algorithm.

4. Capacity estimation

  • Uploads: 12 hours/min × 60 = 720 hr/hr = 17K hr/day × ~50MB/hr (192kbps WAV/MP3 mix average post-trim) = ~850 GB/day raw + 3 transcodes ~3.4 TB/day.
  • Catalog: 320M tracks, ~2 PB raw, ~6 PB total with transcodes.
  • Daily plays: 80M DAU × 15 plays = 1.2B plays/day = ~14K plays/sec avg, 60K peak.
  • Egress: 1.2B × ~3MB avg = 3.6 PB/day.
  • Comments: 80M DAU × 0.5 comments/day = 40M/day = ~500 writes/sec.
  • Stream timeline: avg 80 follows; fanout-on-write = 40M posts × 80 = 3.2B inserts/day — too many; use hybrid (see below).
  • Fingerprint scans: 720K uploads/day, ~1 sec CPU per minute of audio.

6. High-level architecture

[Creator] --multipart--> [S3 Upload Bucket] --event--> [Transcode Workers (FFmpeg on K8s)] | +--> [S3 Streaming Bucket] (HLS variants) +--> [Waveform Generator] -- [S3 waveform JSON] +--> [Fingerprint Svc] -- [Postgres index] +--> emit Kafka 'track_published' [Listener] --> [CDN] -- audio -- [S3 Streaming] | +--> [API GW] --> [Track Svc] -- [Cassandra: track meta] --> [Comment Svc] -- [Cassandra: comments by track] --> [Social Svc] -- [Postgres: graph] --> [Feed Svc] -- [Redis: pre-built timelines] --> [Search Svc] -- [Elasticsearch]

Walk-through: creator uploads multipart to S3, S3 event triggers a transcode job that emits 4 HLS renditions + a waveform JSON (1024 peaks per minute). When done, a Kafka event lands; the Feed service decides per-follower whether to fanout (active followers) or leave in pull-on-read (inactive). Fingerprint runs in parallel; on positive match the track is shadow-banned pending review.

7. Data model & storage

7.1 Track metadata (Cassandra)

CREATE TABLE tracks (
  track_id timeuuid PRIMARY KEY,
  user_id bigint, title text, tags set<text>,
  duration_ms int, hls_master_url text, waveform_url text,
  visibility text, created_at timestamp
);
CREATE TABLE user_tracks (
  user_id bigint, created_at timestamp, track_id timeuuid,
  PRIMARY KEY (user_id, created_at, track_id)
) WITH CLUSTERING ORDER BY (created_at DESC);

7.2 Comments (Cassandra, partitioned by track_id)

CREATE TABLE comments (
  track_id timeuuid, pos_ms int, comment_id timeuuid,
  user_id bigint, body text,
  PRIMARY KEY (track_id, pos_ms, comment_id)
);

7.3 Social graph (PostgreSQL, sharded)

follows(follower_id, followee_id, created_at) -- bi-directional indexes

7.4 Feed/timeline (Redis)

ZSET feed:{user_id}  score=created_at_ms  member=track_id
Trim to 5000 entries, TTL 30 days for inactive users.

8. Detailed component design

8.1 Upload + transcode pipeline

Resumable multipart S3 upload with 5MB parts. On /upload/complete a Kafka job dispatches FFmpeg workers (K8s autoscaled on queue depth). Outputs: HLS at 64/128/256kbps AAC + master MP3. Transcoding for a 1-hour track ~3 minutes on a c6i.2xlarge. Idempotent on upload_id.

8.2 Waveform generation

FFmpeg pipes raw PCM into a downsampler that emits 1024 peak amplitudes per minute, packed as a JSON array (or a 1px-tall PNG for cheap rendering). Stored to S3, served via CDN, ~5KB per track.

8.3 Hybrid feed (fanout)

Pure fanout-on-write at 3.2B/day is too expensive. Hybrid: active users (logged in past 7 days) get fanout-on-write; inactive users get fanout-on-read at next login (gather posts from N most-recently followed). Celebrity accounts (>100K followers) skip fanout entirely — readers pull their content via a separate "celebrity timeline merge" at read time.

8.4 Audio fingerprinting

Chromaprint-style: extract 30s sliding chromagram, hash to 32-bit codes, query a fingerprint index (Postgres + GIN, 50M reference fingerprints from majors). Match threshold = 70% overlap. Positive matches go to a manual review queue + revenue share to rights holder via ContentID-style flow.

8.5 Timestamped comments

Comments are partitioned by track_id with pos_ms as clustering key, so a SELECT WHERE track_id = ? ORDER BY pos_ms returns the entire timeline in one query. Client renders dots above the waveform in O(N) on draw.

9. Scaling & bottlenecks

  • Transcode queue surge: autoscale on Kafka lag; spot instances for ~70% cost cut.
  • Hot tracks (going viral): CDN absorbs; pre-shard popular HLS playlists across 10 cache keys.
  • Feed celebrity fanout: hybrid avoids 100M-write hits when a megastar posts.
  • Fingerprint hot index: shard by hash prefix; in-memory bloom filters cut 80% of negative lookups.
  • Comment hot tracks: rate-limit + Redis write-buffer; flush in batches to Cassandra.

12. Talking points for the interviewer

  • "UGC platforms live or die on the upload-to-publish latency — I'd target <60s for a 5-min track and surface a 'processing' state in UI rather than blocking."
  • "Hybrid fanout is the only feed strategy that survives both Justin Bieber and a million inactive users; pure pull or pure push fails one extreme."
  • "Copyright is the existential business risk. I'd treat fingerprint as a first-class pipeline with SLAs and human review, not a side-job."
  • "For embeds on third-party sites, CSP and CORS get nasty — I'd ship a sandboxed iframe player with a stable cross-origin postMessage API."
