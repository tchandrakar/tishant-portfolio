Design Clubhouse / Twitter Spaces (live audio rooms with WebRTC)

Very Hard audio-streaming messaging video-streaming

1. Problem statement

Design a live audio room product (Clubhouse / Twitter Spaces / X Spaces) where any user can host a virtual room with multiple speakers and thousands of listeners. Speakers exchange real-time audio with sub-300ms mouth-to-ear latency; listeners receive a near-real-time mix. Users in audience can "raise hand" and be promoted to speaker. Rooms can be public, private, or social-graph-gated. Optional features: recording, captions, moderation (mute, kick), reactions. The system must handle WebRTC media routing at scale, room presence/state, social-graph notifications, and abuse moderation.

2. Functional requirements

  • Create/join room (public, private, follower-only).
  • Up to 12 simultaneous speakers; up to 10K listeners per room.
  • Raise hand → host promotes to speaker; demote/kick.
  • Mute/unmute, push-to-talk option, speaker indicator.
  • Real-time reactions (clap, heart) — non-audio.
  • Optional recording; optional live captions.
  • Room discovery: hallway feed of rooms hosted by people you follow.
  • Notifications when followed creator starts a room.
  • End-of-room recap; replay (recorded rooms).

3. Non-functional requirements

  • Latency: speaker-to-speaker mouth-to-ear under 300ms p95 globally; speaker-to-listener under 1s.
  • Scale: 100K concurrent rooms peak; one room can hit 10K listeners; 50M MAU.
  • Availability: 99.95%; a single SFU node failure must not kill the room.
  • Quality: Opus 32 kbps mono is sufficient; jitter buffer on client.
  • Moderation: abuse signal-to-action under 60s for safety reports.

4. Capacity estimation

  • Concurrent users: 100K rooms × avg 50 listeners + 5 speakers = ~5.5M concurrent participants peak.
  • Speaker uplink: 5 speakers × 32 kbps × 100K rooms = 16 Gbps inbound to SFUs.
  • Listener downlink: 5.5M users × 32 kbps avg = 176 Gbps egress (mostly via SFU/CDN).
  • Signaling QPS: join/leave/mute = ~10 events/user/session × 5.5M = 55M events; over a 30-min avg = ~30K signaling QPS.
  • Recordings: assume 20% rooms recorded × avg 60min × 32 kbps = ~14 MB/room × 20K = 280 GB/day.
  • Notifications fanout: popular host with 1M followers × "started a room" = 1M push-notifs in 30s = ~33K/sec.

6. High-level architecture

[Client] | WebRTC media (UDP/SRTP) WebSocket signaling v v [SFU node] <--cascade--> [SFU peer] [Signaling Svc] -- [Redis pub/sub: room state] | | +--> [Mixer/Recorder] -- [S3] +--> [Room Mgr] -- [Postgres: rooms, roles] | +--> [Presence Svc] -- [Redis] | +--> [Notif Svc] -- [Kafka -> FCM/APNs] | +--> [Caption Svc (ASR)] --> WS to clients [Edge: TURN servers] for NAT traversal [Hallway/Feed Svc] -- [Cassandra: live rooms by follow-graph]

Walk-through: client signals via WebSocket to acquire a SFU URL + ICE creds. WebRTC PeerConnection established to nearest SFU; the SFU forwards each speaker's RTP to other participants (no decode/encode). Across regions, SFUs cascade to one another. Recording forks an RTP copy to a mixer that produces a single Opus file. Signaling state (who's in, who's a speaker) lives in Redis, mirrored to Postgres for durability.

7. Data model & storage

7.1 Room metadata (PostgreSQL)

CREATE TABLE rooms (
  room_id uuid PRIMARY KEY, host_id bigint,
  title text, type text, started_at timestamptz,
  ended_at timestamptz, recording_url text
);
CREATE TABLE room_members (
  room_id uuid, user_id bigint, role text,  -- host|speaker|listener
  joined_at timestamptz, PRIMARY KEY (room_id, user_id)
);

7.2 Live presence (Redis)

HSET room:{id}:presence userId -> "{role,sfuNode,joinedAt}"
SET  room:{id}:speakerCount  (atomic INCR/DECR)
PUBLISH room:{id}:events  {type, payload}

7.3 Hallway feed (Cassandra)

CREATE TABLE follow_rooms (
  user_id bigint,            -- the follower
  started_at timestamp,
  room_id uuid, host_id bigint, title text,
  PRIMARY KEY (user_id, started_at, room_id)
) WITH default_time_to_live = 14400;  -- 4h auto-expire

8. Detailed component design

8.1 SFU media plane

Selective Forwarding Unit (mediasoup or custom Pion-based) receives Opus RTP from each speaker and forwards to all other participants in the room. No mixing on hot path = no encode/decode = lowest CPU. Each SFU node handles ~2K concurrent connections; we shard rooms across nodes by room_id hash and use cascading: large rooms span multiple SFUs that peer with each other (n × m forwarding mesh).

8.2 Cascading for 10K-listener rooms

Single SFU caps at ~2K. For larger rooms, primary SFU forwards to edge SFUs (one per region). Edge SFUs serve listeners locally. Speakers always join the primary; listeners get assigned to the geographically-nearest edge.

8.3 Signaling & presence

WebSocket gateway terminates client connections, delegates room state to Redis pub/sub. Promote/demote = Postgres update + Redis publish + per-client SDP renegotiation (transition from recvonly to sendrecv transceiver).

8.4 Recording & replay

An auxiliary "ghost" participant subscribes to all speaker streams via the SFU and writes mixed Opus to local disk; on room end, the file is uploaded to S3 and re-encoded to AAC for HLS replay. CDN serves replay with adaptive bitrate.

8.5 Live captions

SFU forks raw PCM to a streaming ASR (e.g., Whisper-stream / Google Streaming STT). Captions broadcast over the WS event channel; clients render them with ~1.5s lag.

8.6 Moderation

Abuse reports go to a queue; a parallel ASR + classifier flags hate-speech in near-real-time. Severe matches auto-mute the speaker pending review; the host receives a notification.

9. Scaling & bottlenecks

  • SFU CPU: dominated by SRTP encrypt; bare-metal nodes with AES-NI; 1 vCPU per ~150 forwarded streams.
  • NAT traversal: coturn TURN fleet for ~15% of users behind symmetric NAT; egress costs are real.
  • Cascading mesh failure: if an edge SFU dies, listeners auto-failover to peer edge or primary; speakers see a 1–2s glitch.
  • Hot-host fanout: when a 10M-follower account starts a room, push notifications batched + rate-limited; hallway feed write uses fanout-on-read for celebrities.
  • Signaling spikes on a viral room: WS fleet horizontally scaled; sticky session by room_id.

12. Talking points for the interviewer

  • "This is a real-time media problem at heart, not a CRUD app — the SFU topology and WebRTC details dominate the design. I'd budget mouth-to-ear latency: 30ms encoder + 100ms network + 50ms jitter buffer + 50ms decode = ~230ms target."
  • "Cascading SFUs are how Clubhouse scaled past Agora's per-room cap; the trade-off is cross-region listeners hear speakers ~80ms later than co-region listeners."
  • "For host disconnect I'd promote the longest-tenured speaker via a Raft-backed coordinator — abandoning a 10K-listener room is a UX disaster."
  • "Moderation is the existential product risk — live audio + millions of listeners means hate-speech can go viral in seconds; I'd ship streaming ASR + classifier as Day 1, not Day 90."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:
