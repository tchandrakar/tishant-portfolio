Design Spotify (audio streaming at scale)

Hard audio-streaming cdn recommendation

1. Problem statement

Design an on-demand music streaming service like Spotify. Users browse a global catalog of ~100M tracks, play any track on-demand under 200ms, build playlists, follow artists, and receive personalized recommendations (Discover Weekly, Daily Mixes). The system must handle royalty-grade play accounting, multi-device synchronization (phone, web, smart speaker, car), offline downloads for premium users, and DRM for label compliance. Scope excludes podcasts ingestion pipeline (handled separately) but includes the core audio plane.

2. Functional requirements

  • Browse and search catalog (artist, album, track, playlist) with sub-second results.
  • Stream any track on-demand with adaptive bitrate (96/160/320 kbps).
  • Create, edit, share, collaborate on playlists; follow artists/users.
  • Personalized recommendations (Discover Weekly, Release Radar, Daily Mixes).
  • Offline downloads for Premium subscribers (encrypted at rest on device).
  • Cross-device playback resume (Spotify Connect).
  • Royalty-grade play accounting (every >30s play counted exactly once).
  • Like/save tracks; library sync across devices.

3. Non-functional requirements

  • Latency: first audio byte under 200ms p99 globally; search under 150ms p99.
  • Scale: 600M MAU, 230M premium, 250M DAU, peak 8M concurrent streams.
  • Availability: 99.95% playback; 99.9% catalog/search.
  • Durability: 11 nines for master audio files; loss = legal risk with labels.
  • Consistency: eventual for play counts/recos; strong for billing and library writes.
  • Security: Widevine/FairPlay DRM, signed short-lived stream URLs, TLS 1.3.

4. Capacity estimation

  • Catalog: 100M tracks × 3 bitrates × avg 4MB (160kbps, 3.5min) = ~1.2 PB raw + lossless masters ~10 PB.
  • Daily plays: 250M DAU × 25 plays/day = 6.25B play events/day = ~72K events/sec average, peak ~250K/sec.
  • Streaming bandwidth: 8M concurrent × 160kbps avg = 1.28 Tbps egress at peak.
  • Daily egress: 6.25B plays × 4MB = 25 PB/day from CDN.
  • Search QPS: 250M DAU × 10 searches/day / 86400 = ~30K QPS average, 100K peak.
  • Reco compute: 600M user embeddings (256d × 4B) = ~600 GB hot in feature store.
  • Play-event log: 6.25B × 200B = 1.25 TB/day (compressed ~300 GB) into Kafka.

6. High-level architecture

[Mobile/Web/Speaker] | v [Edge POP / CDN] ----audio bytes---- [Origin Shield] -- [S3/GCS Audio Blobs] | v [API Gateway + Authz] | +--> [Playback Svc] -- [License/DRM Svc] -- [Key Mgmt] | | | +--> emit play-events --> [Kafka] | +--> [Catalog Svc] ---- [Cassandra: tracks, albums] | +--> [Search Svc] ---- [Elasticsearch cluster] | +--> [Library Svc] ---- [PostgreSQL shards] | +--> [Reco Svc] ----- [Feature Store: Redis + Bigtable] ^ | [Kafka] --> [Flink stream jobs] --> [BigQuery] --> [Spark batch ML training]

Walk-through: a play request hits the nearest POP, gets routed to the Playback service which mints a short-lived signed CDN URL after DRM license check. The client streams encrypted MP4/Opus segments directly from CDN. Heartbeats land in Kafka. Flink computes royalty-eligible plays (>30s) and updates leaderboards. Nightly Spark jobs retrain the recommender; embeddings are pushed to a Redis ANN index for online candidate generation.

7. Data model & storage

7.1 Catalog (Cassandra)

CREATE TABLE tracks (
  track_id uuid PRIMARY KEY,
  title text, artist_id uuid, album_id uuid,
  duration_ms int, isrc text,
  audio_blobs map<text, text>,  -- bitrate -> blob_id
  available_markets set<text>,
  release_ts timestamp
);
CREATE TABLE artist_tracks (
  artist_id uuid, release_ts timestamp, track_id uuid,
  PRIMARY KEY (artist_id, release_ts, track_id)
) WITH CLUSTERING ORDER BY (release_ts DESC);

7.2 Library & playlists (PostgreSQL, sharded by user_id)

playlists(playlist_id, owner_id, name, public, version)
playlist_tracks(playlist_id, position, track_id, added_ts)
user_library(user_id, track_id, saved_ts)

7.3 Why these stores?

Cassandra for the catalog: write-once/read-many, multi-DC replication, predictable wide-column scans for "all tracks by artist". PostgreSQL for playlists because users expect strong read-after-write semantics on their own library. Audio bytes live in object storage (S3/GCS) behind a CDN; metadata never inlines bytes.

8. Detailed component design

8.1 Playback service

Stateless. On /playback/start it (a) validates entitlement (region, premium tier), (b) calls License service for a Widevine license bound to device_id, (c) signs a CloudFront/Fastly URL with 30s TTL, (d) emits a play_started event. The client refreshes the URL every 25s while streaming.

8.2 Recommendation pipeline

Two-stage: candidate generation via ANN (HNSW over user/track embeddings, 256-d, retrained daily on collaborative-filtering matrix factorization + audio CNN features) returns top 5K candidates in 10ms. A LambdaMART ranker scores them with 200 features (recency, skip rate, time-of-day, device). Discover Weekly is precomputed every Monday into Cassandra; Daily Mix is online.

8.3 Spotify Connect (cross-device)

Each device opens a long-lived WebSocket to a "device fan-out" service partitioned by user_id. Transfer events push state (track, position, queue) to the new device; old device fades out. State of truth lives in Redis with 24h TTL, persisted to Cassandra on logout.

8.4 Play-event ingestion & royalties

Heartbeats batched on-device every 10s, posted to Kafka via a thin HTTP collector. Flink job dedupes by (session_id, track_id) and emits a single royalty_play when listened >30s. Output sinks to BigQuery for monthly label reporting; idempotency is critical because every play = money.

9. Scaling & bottlenecks

  • Hot tracks (new release): shielded origin, pre-warm CDN POPs the hour before drop, regional traffic-shaping.
  • Cassandra hot partitions: bucket artist_tracks by month to avoid skew on prolific artists.
  • Search: Elasticsearch with completion suggester; tier popular queries in Redis (1-min TTL).
  • Reco serving: embeddings sharded by user_id; ANN index replicated read-only per region.
  • Kafka: partition play events by user_id (256 partitions per topic); retention 7d hot, archived to GCS.

12. Talking points for the interviewer

  • "I'd start with a single-region MVP using Postgres for everything, then split catalog to Cassandra once read QPS exceeds 50K, because adding NoSQL too early adds operational cost without payoff."
  • "The hardest part is royalty-grade exactly-once accounting under client retries; my mitigation is idempotent session_id keys and Flink's two-phase commit sink to BigQuery."
  • "For cold-start recommendations, I'd lean on content-based audio embeddings (CNN over mel-spectrograms) before collaborative signals exist."
  • "Offline downloads need encrypted blobs with device-bound keys rotated every 30 days to satisfy label DRM clauses."
