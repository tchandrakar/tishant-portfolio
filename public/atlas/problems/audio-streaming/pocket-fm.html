Design Pocket FM (audio series platform)

Hard audio-streaming recommendation ad-tech

1. Problem statement

Design Pocket FM, an audio-series platform where creators publish multi-episode "audio shows" (long-form serialized fiction). The product has a freemium model: the first N episodes are free with ads; later episodes are unlocked with virtual coins purchased via in-app payments. Core user flow is binge-listening one show at a time across days. The system must serve episodic catalogs, drive aggressive personalization (because retention hinges on next-episode hooks), monetize via interstitial audio ads + coin unlocks, and handle bursty traffic when a show goes viral on social media.

2. Functional requirements

  • Browse shows by genre/language/trending; play episodes sequentially.
  • Sequential progress tracking per show (resume position, last-played episode).
  • Coin wallet: buy coins, unlock episodes, refunds, gifting.
  • Insert audio ads dynamically at episode boundaries for free users.
  • Push notifications when next episode in followed show drops.
  • Personalized "Made for You" rails and "Continue Listening" rail.
  • Creator dashboard: upload episodes, view listens, revenue split.
  • Multi-language support (Hindi, Tamil, English, Indonesian, Spanish).

3. Non-functional requirements

  • Latency: first byte under 250ms p99 in target geos (India SEA LATAM).
  • Scale: 100M MAU, 25M DAU, 1.5M concurrent at prime time.
  • Availability: 99.95% playback; 99.9% wallet (payments tier).
  • Consistency: strong for coin balance/unlocks; eventual for listen counts.
  • Cost: low ARPU geography — egress cost dominates, must aggressively cache and use cheap codec (Opus 64kbps).

4. Capacity estimation

  • Catalog: 200K shows × avg 200 episodes × 25min × Opus 64kbps = ~12 MB/episode × 40M episodes = ~480 TB raw + 2 bitrates ~1 PB.
  • Daily listens: 25M DAU × 4 episodes = 100M episode-plays/day = ~1.2K plays/sec avg, 6K peak.
  • Egress: 100M plays × 12MB = 1.2 PB/day from CDN.
  • Coin txns: 5M unlock events/day = 60 TPS avg, 500 TPS peak (post-show-launch surge).
  • Ad impressions: 100M plays × 2 ads/episode × 70% free users = 140M impressions/day = ~1.6K QPS to ad server.
  • Notification fanout: when popular show drops, 5M followers × push = 5M sends in 60s = 80K/sec.

6. High-level architecture

[App] -- HTTPS --> [CDN/POP] -- audio segments -- [Origin: S3] | +--> [API Gateway] | +--> [Catalog Svc] --- [MongoDB: shows, episodes] +--> [Playback Svc] -- [Ad Decision Svc] -- [Ad Inventory: Redis+MySQL] +--> [Wallet Svc] --- [PostgreSQL: ledger, idempotent] +--> [Progress Svc] --- [Redis] --- async --- [Cassandra: history] +--> [Reco Svc] --- [Vector DB (Milvus)] + [Feature Store] +--> [Notif Svc] --- [Kafka] -- [FCM/APNs workers] [Kafka: listen-events, ad-events] -> [Flink] -> [ClickHouse] -> creator dashboards

Walk-through: client requests episode start; Playback service checks unlock state, calls Ad Decision for an interstitial pre-roll if user is free-tier, returns a manifest with audio + ad segments stitched. Wallet service is the only strongly consistent path. Listen events flow through Kafka into ClickHouse for creator analytics and into the feature store for next-day reco training.

7. Data model & storage

7.1 Catalog (MongoDB)

show: { _id, title, lang, genres[], creatorId, totalEpisodes,
        freeEpisodeCount, coinPrice, coverArt }
episode: { _id, showId, seq, title, durationMs, audioBlobId,
           releaseTs, isFree }

7.2 Wallet ledger (PostgreSQL, double-entry)

CREATE TABLE wallet_txn (
  txn_id uuid PRIMARY KEY,
  user_id bigint, delta_coins int,
  reason text,           -- 'topup' | 'unlock' | 'refund'
  ref_id text,           -- episode_id or receipt_id
  idempotency_key text UNIQUE,
  created_at timestamptz
);
CREATE TABLE wallet_balance (
  user_id bigint PRIMARY KEY,
  balance_coins int, version bigint
);

7.3 Listen progress (Redis primary, Cassandra archive)

HSET progress:{userId} {showId} "{episodeSeq}:{posMs}:{ts}"
TTL 30 days; nightly Spark archives to Cassandra by user_id.

8. Detailed component design

8.1 Wallet & unlock flow

Idempotency-key on every /unlock; transactional insert into wallet_txn + UPDATE balance with optimistic version check. On payment top-up, the IAP receipt is verified server-side with Apple/Google before crediting. Refunds use a reversing entry — never UPDATE prior rows. This double-entry ledger satisfies finance audit and prevents race-condition exploits during viral surges.

8.2 Ad decision & stitching

Server-side ad insertion: the playback manifest interleaves audio segments with ad segments transcoded to identical codec/bitrate for seamless transitions. Ad selection uses a real-time auction (header bidding fallback to direct campaigns) with frequency capping in Redis. Free users see a pre-roll + mid-roll; coin-unlocked episodes are ad-free.

8.3 Recommendation engine

Show-level reco uses a two-tower model: user tower (history embedding via GRU over last 50 episode listens) and show tower (text + cover-image features). ANN search in Milvus returns top 1000 candidates; an XGBoost ranker re-orders with completion-rate features. "Continue Listening" rail is a simple Redis ZSET sorted by last-touch.

8.4 Episode-drop notification fanout

On creator publish, a Kafka topic episode_published drives a fanout job that batches followers (1000/batch) and dispatches to FCM/APNs. Quiet hours are enforced per-user timezone. Dedup key prevents double-sends on retries.

9. Scaling & bottlenecks

  • Viral show surge: CDN absorbs reads; pre-warm POPs based on social-trend signals.
  • Wallet write hotspot: shard PostgreSQL by user_id hash; per-user single-writer avoids deadlocks.
  • Reco cold start: language + genre fallback rails for first-session users.
  • Push fanout: rate-limit per provider (FCM 600K msg/min) via token bucket on dispatcher.
  • Egress cost: Opus 64kbps cuts bandwidth 60% vs MP3 128 with comparable speech quality.

12. Talking points for the interviewer

  • "Pocket FM's economics are different from Spotify — ARPU is 10x lower, so I optimized for egress cost first (Opus, aggressive CDN, regional POPs)."
  • "The wallet is the riskiest component; I'd treat it as a payments-grade microservice with double-entry ledger and quarterly reconciliation against Apple/Google receipts."
  • "Sequential bingeing means the next-episode hook is everything — I'd A/B test mid-roll ads heavily and use completion-rate as the north-star reco metric, not CTR."
  • "For viral shows I'd also pre-decode coin pricing into a config service so we can dynamically discount episodes 1–5 for cohorts that drop off."
