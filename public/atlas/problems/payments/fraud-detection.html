Design Fraud Detection at Scale (real-time ML scoring)

Hard payments recommendation analytics

1. Problem statement

Score every payment in real time (≤ 50 ms) and decide approve / step-up-auth / decline. Inputs: the txn itself, 90-day user history, device/IP/geo signals, network features (graph of accounts), and merchant context. Outputs: a probability of fraud, an explanation, and a verdict. Volume: 5 K TPS sustained, 50 K TPS Black Friday peak, 5 B txns/year.

2. Functional requirements

  • Inline scoring API: txn in → verdict out, < 50 ms p99.
  • Feature store with point-in-time correctness (no train-serve skew).
  • Rule engine merchants/analysts can edit (e.g., "block if BIN=X and amount > $1k").
  • Graph features: accounts, devices, IPs as nodes; edges = shared use within 30 days.
  • Manual review queue with explanation + feedback loop into training.
  • Backtesting framework on labeled history (chargebacks as ground truth, ~30–60 day label lag).

3. Non-functional requirements

  • Latency: p99 < 50 ms inline; budget split: 5 ms feature fetch, 10 ms model, 35 ms slack.
  • Scale: 50 K TPS peak × ~300 features = 15 M feature lookups/s.
  • Recall vs precision: target 80% recall at 1% false-positive rate (FPR) — every basis point of FPR = real $$$ in lost legitimate sales.
  • Model freshness: features ≤ 5 s stale; model retrained weekly, A/B-rolled.
  • Availability: 99.99% (failure-mode → fallback rule engine, never fail-open on high-risk).

4. Capacity estimation

  • Feature store: 500 M users × 300 features × 8 B = 1.2 TB hot (Redis); cold tier 50 TB (Cassandra).
  • Event ingest: 50 K TPS × 1 KB = 50 MB/s into Kafka; 7-day retention × 86400 s = 30 TB.
  • Graph: ~2 B edges (accounts↔devices↔IPs); JanusGraph on Cassandra, 800 GB.
  • Model inference: 1 GBT model × 50 K QPS / 5 K QPS per node = 10 inference nodes (with headroom: 30).

5. API design

POST /v1/risk/score
  Body: {txnId, userId, amount, currency, ip, deviceFP, merchantId,
         paymentMethod, billingAddr, shippingAddr, ...}
  → {score: 0.0..1.0, decision: APPROVE|STEP_UP|DECLINE,
     reasonCodes: ["high_velocity_24h","new_device","ip_proxy"],
     modelVersion: "gbm_v37"}

POST /v1/risk/feedback
  Body: {txnId, label: FRAUD|LEGIT, source: chargeback|manual_review}

6. High-level architecture

[Payment Svc] ──score──► [Risk Gateway] │ ┌─────────────────┼─────────────────────────┐ ▼ ▼ ▼ [Feature Store] [Model Server] [Rule Engine] Redis (hot) (Triton/TFServ) (decision tree) Cassandra (cold) │ │ ▲ ▼ │ │ [Graph Lookup] │ │ (JanusGraph) │ │ │ │ ▼ │ │ [Verdict Combiner] ◄───────────┘ │ │ │ ▼ │ [Audit log → S3] │ [Feature Pipeline] Flink streaming ▲ │ [Kafka: txn, login, click events]

Two scoring paths run in parallel: ML model and rule engine. The combiner takes max(score_ml, score_rule) and applies thresholds. Rules are the safety net when the model misbehaves on novel attacks.

7. Data model & storage

7.1 Feature store (online)

Redis Cluster:
  KEY user:{user_id}:features → HASH {
    txn_count_1h: 3,
    txn_amount_24h: 412.50,
    distinct_ip_7d: 2,
    avg_txn_amount_30d: 89.21,
    ...
  }
  TTL: 90 days; updated by Flink within 2-5 s of source event.

7.2 Offline store (training)

Parquet on S3, partitioned by date and feature group. Point-in-time joins ensure training labels see only data available at txn time (no leakage). Feast or in-house equivalent ties online + offline definitions to one source.

7.3 Why this split?

Strong consistency on features is wrong: we'd rather see a 3-second-old aggregate than block a payment for 200 ms. Eventual consistency with bounded lag is the right SLA. (DDIA Ch.11: stream-stream joins for velocity features.)

8. Detailed component design

8.1 Feature pipeline (Flink)

Tumbling windows of 1m, 5m, 1h, 24h, 7d, 30d. Each event fans out into ~6 windowed counters. Watermark = max(event_time) − 5 s; allowed lateness 60 s. State backend: RocksDB; checkpoint to S3 every 30 s (DDIA Ch.11 stream stateful processing).

8.2 Model serving

Gradient-boosted trees (XGBoost/LightGBM) for the primary model: 5 ms inference, interpretable via SHAP. A neural side-model handles sequence features (last 50 user txns via Transformer encoder). Both run in Triton; the gateway calls them in parallel and combines via stacking.

8.3 Graph features

Pre-computed nightly: PageRank on the account-device-IP graph; community detection identifies fraud rings. At scoring time, a 1-hop lookup ("how many fraud-flagged accounts share my device fingerprint?") fits in 3 ms.

8.4 Feedback & retraining

Chargebacks arrive 30–60 days after txn; a labeling service joins them back to the original score, computes the label, and writes to the offline store. Weekly retrain on a rolling 180-day window; champion/challenger A/B with 5% shadow traffic before promotion.

9. Scaling & bottlenecks

  • Feature fetch fan-out: mget 300 keys → use field grouping in Redis hash to fetch 1 key returning 300 fields; cuts RTT 50×.
  • Cold-start users: if no history, fall back to merchant + device priors; never block a first-time legitimate user without step-up.
  • Model thundering herd at deploy: gradual rollout 1%→10%→50%→100% with auto-rollback on FPR spike.
  • Adversarial drift: fraudsters probe; monitor feature distributions (KL divergence) and trigger emergency retrain when a feature drifts > 3σ.
  • Graph blowup: a single bad device can connect 10 K accounts; cap node degree, treat super-nodes specially.

12. Talking points for the interviewer

  • "Fraud is an adversarial system, not a static one — the architecture must support feature drift detection and weekly retraining as a first-class operation, not a research exercise."
  • "I'd never let the ML model decline alone; a human-readable rule engine is the off-switch when the model goes bad at 3 a.m."
  • "Train-serve skew is the silent killer; one source of truth for feature definitions, used by both Flink (online) and Spark (offline)."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Flink — feature pipeline core 50 K TPS peak × 6 windowed counters per event = ~300 K state mutations/s with windows from 1m to 30d. Flink's RocksDB state backend, watermark-based event time, and exactly-once checkpointed state (DDIA Ch.11) is the only mature engine that handles this windowing without dropping or double-counting on restart. Kafka Streams (state size limits at 30-day windows); Spark Streaming (micro-batch latency exceeds 5 s feature freshness target).
Kafka — event ingest backbone 50 MB/s × 7-day retention = 30 TB; Kafka's partition-ordered log is what lets the same txn event drive both real-time features (Flink) and offline training (Spark) — DDIA Ch.11 stream-table duality. Replay capability is essential when retraining model on a fixed historical window. Pulsar (less ops familiarity); Kinesis (worse retention economics, weaker replay ergonomics at 30 TB).
Redis — online feature store (hot) 50 K TPS × ~300 features = 15 M lookups/s with a 5 ms budget. Redis Hash-per-user (single key returns all 300 fields in one RTT) cuts fan-out 50×; 1.2 TB hot tier across cluster shards. Stale by < 5 s is acceptable — we'd rather a slightly old aggregate than block a payment. Memcached (no hash data structure for fan-out); DynamoDB (latency floor & cost at this QPS).
Cassandra — cold feature tier 50 TB of warm/cold features (90+ days, infrequent reads) suit Cassandra's write-heavy wide-column model. Eventual consistency is acceptable here because the ML inference is itself probabilistic — a stale feature does not compromise correctness, only marginal recall. HBase (heavier ops); S3 + Athena (latency too high for fallback feature reads).
Postgres (SSI) — model registry & rules The active model version, rule definitions, and audit-of-rule-changes need ACID with strong consistency — a stale rule that says "block all txns > $10" deployed by mistake is a real outage. SSI (DDIA Ch.7) catches conflicting rule edits by analysts. etcd alone (ok for active model pointer; insufficient for full rule history); MySQL (no proper SSI for editor write skew).
JanusGraph (on Cassandra) — fraud graph 2 B edges (account ↔ device ↔ IP); 1-hop lookup at score time ("how many fraud-flagged accounts share this device?") fits in 3 ms. Cassandra storage handles 800 GB graph data with horizontal write scale. Neo4j (single-master write bottleneck at our ingest rate); native graph in Spark (offline only, can't serve inline).
Triton / TF-Serving — model server Gradient-boosted tree inference at 5 ms p99 + transformer side-model in parallel. Triton's dynamic batching + GPU multiplexing fits 50 K QPS across ~30 nodes with headroom; protocol-buffer-based RPC keeps overhead low inside the 50 ms total budget. Sagemaker endpoints (cost & cold-start jitter); custom Python servers (no batching, ~3× more nodes).
etcd (Raft) — model version pointer "Which model is live?" must be linearizable across all 30 inference nodes (DDIA Ch.9): a half-rolled-out version where some nodes serve v37 and some v38 corrupts A/B metrics. Raft-replicated KV is exactly the primitive. ZooKeeper (heavier ops burden); database polling (race window during deploy).
S3 + Parquet — offline training store Point-in-time joins on 5 B txns/yr × 90-day window need columnar scan. Parquet partitioned by date and feature group lets Spark read only relevant columns; matches the durability + cost profile of cold training data perfectly. Cassandra (column reads inefficient for analytics); single Postgres warehouse (cost & scan throughput).
Idempotency keys (verdict binding) The score is read-only, but the verdict's binding to the payment reuses the txnId-keyed idempotency pattern (DDIA Ch.7). We never re-score a replayed payment with stale features — same key returns the same verdict, even if features have moved on. Re-score on every retry (non-deterministic outcomes; merchant disputes); cache score by feature snapshot (storage + miss complexity).
Saga — fail-closed verdict combiner If model server is unreachable inside the 50 ms budget, the rule engine alone is the safety net (verdict = max(score_ml, score_rule)). No 2PC across model + rules — the combiner takes whichever finishes first and applies thresholds. Saga-style compensating step-up auth replaces declines on uncertain scores. Synchronous join with both required (latency + cascading failure); ML-only (no off-switch when model goes bad).
2PC — explicitly rejected Inline scoring is on a strict 50 ms p99 budget; 2PC across feature store + model + rule engine would add coordinator round-trips and tie availability to the slowest participant. Independent calls with combiner is the correct shape for an inference path. Saga / parallel calls (chosen); RPC chaining (latency adds, no gain).
Per-merchant rate limit / circuit breaker 50 K TPS comes unevenly — Black Friday spikes one merchant 100×. Per-merchant token bucket prevents one bad actor from starving inference capacity for everyone else; circuit breaker on model server protects the rule-only fallback path. Global rate limit (one merchant starves all); no limit (cascading inference latency).
Drift monitor (KL divergence on features) Adversarial drift is a first-class threat: fraudsters probe and shift feature distributions. Watching KL > 3σ on the live feature stream and triggering emergency retrain is what keeps the 80% recall / 1% FPR target stable across weeks. Schedule-only retrain (lags real attacks by days); manual review-only (doesn't scale to 5 B txns/yr).
