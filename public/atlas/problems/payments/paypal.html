Design PayPal (P2P payments, multi-currency, fraud)

Hard payments ecommerce

1. Problem statement

Build a global P2P and merchant payments platform: users link bank accounts, debit/credit cards, and balances across 25+ currencies; send money to other users or pay merchants; refunds, chargebacks, holds, and a fraud-aware risk layer keep the platform honest. The core engineering challenge is a system that never loses money, never double-pays, and never books a half-transfer — even when networks fail mid-call.

2. Functional requirements

  • Send/receive money P2P (intra-PayPal) and to external bank accounts via ACH/SEPA/SWIFT.
  • Multi-currency wallets with FX conversion at quoted rate (60-second quote TTL).
  • Merchant checkout: authorize, capture, void, refund. Idempotent against retries.
  • Disputes & chargebacks (180-day window for cards), holds, and reserves.
  • Real-time fraud / risk scoring before commit; manual review queue for soft-decline.
  • Statements, transaction history, exports for tax (1099-K).

3. Non-functional requirements

  • Correctness: zero double-debit, zero phantom credit. Money is conserved.
  • Latency: P2P transfer commit p99 < 800 ms; merchant authorize p99 < 400 ms.
  • Scale: 430 M active accounts, ~25 B transactions/year (~800 TPS avg, 5 K TPS peak Black Friday).
  • Availability: 99.99% for authorize/capture; 99.95% for ledger reads.
  • Compliance: PCI-DSS, AML/KYC, SOX, GDPR; 7-year audit retention.

4. Capacity estimation

  • QPS: avg 800 writes/s × 5 storage rows per txn (ledger entries + journal) ≈ 4 K writes/s; reads 50 K/s.
  • Storage: 25 B txns/yr × 2 KB/row × 7 yrs ≈ 350 TB ledger; replicated 3× in primary + 1 cold = 1.4 PB.
  • Fraud features: 200 features × 4 B × 5 K TPS = 4 MB/s into the model serving tier.
  • FX quote cache: 25² currency pairs × 1 KB × 100 K active quotes = ~62 MB (Redis).

6. High-level architecture

[Client] → [Edge/WAF] → [API GW] ──► [Payment Orchestrator] │ (Saga coordinator) ┌─────────────────────────┼──────────────────────────┐ ▼ ▼ ▼ [Risk/Fraud Svc] [Ledger Svc] [FX Svc] (Flink + ML) (Postgres + 2PC) (Redis quotes) │ ▼ [Kafka: txn-events] │ ┌───────────────┼────────────────┐ ▼ ▼ ▼ [Notifications] [Reporting/DWH] [Settlement Svc] (ACH/SEPA gateways)

The orchestrator implements a saga: reserve funds → fraud check → debit sender → credit recipient → emit event. Each step has a compensating action; we prefer saga over distributed 2PC because cross-region locks would tank latency (DDIA Ch.9: coordinator is a single point of contention).

7. Data model & storage

7.1 Ledger (Postgres, sharded by account_id)

CREATE TABLE ledger_entries (
  entry_id     BIGSERIAL,
  txn_id       UUID NOT NULL,
  account_id   BIGINT NOT NULL,
  amount       NUMERIC(20,4) NOT NULL,  -- signed
  currency     CHAR(3) NOT NULL,
  posted_at    TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (account_id, entry_id)
) PARTITION BY HASH(account_id);

CREATE UNIQUE INDEX ux_idem ON payments(idempotency_key);

Double-entry: every txn writes ≥2 rows summing to zero per currency. We use Postgres for the ledger because we need serializable isolation on the account row (DDIA Ch.7: SSI catches write skew that would let two concurrent withdrawals both pass a balance check).

7.2 Why not Cassandra for ledger?

LWW eventual consistency is unsafe for money: a delayed replica could resurrect a spent balance. We use Cassandra only for append-only event/audit logs.

8. Detailed component design

8.1 Idempotency & exactly-once

Client sends Idempotency-Key. The orchestrator inserts (key, request_hash, txn_id) with UNIQUE constraint before any side effect. Replay returns the cached response. The payment row is committed in the same Postgres transaction as the ledger debit — leveraging single-node ACID instead of cross-service 2PC (DDIA Ch.7: linearizable single-row writes are far cheaper than distributed atomic commit).

8.2 Fraud scoring

Inline path: 30 ms budget. Features come from (a) Redis hot cache (last-1h velocity), (b) feature store (90-day aggregates pre-computed by Flink). Gradient-boosted model returns score 0–1; threshold 0.85 = decline, 0.6–0.85 = step-up auth (3DS / OTP).

8.3 FX

Quotes are signed JWTs with TTL=60s. Settlement at quoted rate; the FX desk holds a hedge book reconciled hourly.

9. Scaling & bottlenecks

  • Hot account contention (e.g. a marketplace seller): shard the account into sub-ledgers by hash(txn_id mod N), aggregate on read.
  • Cross-region writes: pin an account to its home region; cross-region transfers go through a two-phase saga with reservation rows.
  • Kafka backpressure: downstream slowness must never block the synchronous commit; we use a local outbox table drained by a CDC connector (Debezium) — guarantees no event is lost (DDIA Ch.11 transactional outbox pattern).

11. DDIA references

  • Ch. 7 Transactions — Serializable Snapshot Isolation prevents write skew on balance checks; single-object atomic writes underpin the idempotency table; the chapter's discussion of "exactly-once via idempotent operations" is the core of the orchestrator.
  • Ch. 7 Transactions — read-committed isn't enough; we need SSI for the "balance >= amount" predicate.
  • Ch. 9 Consensus — leader election on the orchestrator (Raft via etcd) for in-flight saga state; linearizability arguments for why the ledger commit must be totally ordered per account.
  • Ch. 11 Stream Processing — transactional outbox + CDC for at-least-once event publication.

12. Talking points for the interviewer

  • "I'd start with a single-region Postgres ledger and saga orchestrator; multi-region active-active comes only after we've proved the invariants in one region — distributed money is hard enough without distributed consensus."
  • "The hardest part is the boundary between fraud (eventually consistent ML) and the ledger (strongly consistent SSI) — I treat fraud's verdict as an input, not a participant in the transaction."
  • "Reconciliation is non-negotiable: a daily job re-derives every balance from the entry log and pages on any drift."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Postgres (SSI) — ledger 800 TPS avg / 5 K TPS peak fits a single sharded primary. SSI (DDIA Ch.7) catches write skew on the "balance >= amount" predicate so two concurrent withdrawals can't both pass; same-txn idempotency row + ledger debit gives single-node ACID instead of cross-service 2PC. MySQL (no SERIALIZABLE that catches phantom write skew cleanly); CockroachDB (overkill at this TPS, higher commit latency).
Cassandra — audit/event log only Append-only audit trail (350 TB ledger × 7 yrs) fits Cassandra's wide-column write throughput perfectly — but only because audit is write-once and never asks "is the balance right". LWW eventual consistency would resurrect a spent balance, so it never touches money rows. HBase (operationally heavier, fewer ops in our shop); S3 Parquet (cheaper but worse for time-range scans on hot audit).
Kafka — txn-events bus 4 K writes/s × N consumers (notifications, DWH, settlement) needs durable fan-out with replay. Partition by account_id preserves per-account ordering — required so "debit then credit" events are never reordered downstream (DDIA Ch.11 total order per partition). RabbitMQ (no replay, weaker ordering guarantees); AWS SQS (no partition ordering, lossy at our retention).
Flink — fraud feature pipeline 200 features × 5 K TPS = 4 MB/s of stateful aggregation across 1m–90d windows; Flink's RocksDB state + checkpointed exactly-once (DDIA Ch.11) is the only mature option for windowed velocity features with sub-second freshness. Kafka Streams (state size limits at 90-day windows); Spark Structured Streaming (micro-batch latency floor too high for inline scoring).
Redis — FX quote & fraud hot cache 62 MB FX quote book + 200-feature 1h velocity per user needs sub-ms p99 reads inside the 30 ms fraud budget. JWT-signed quotes mean clients can replay without DB hits. Bounded staleness is acceptable here — money still commits in Postgres. Memcached (no data structures for hash-of-features); local in-process cache (cache stampede + cross-region invalidation problems).
Saga orchestrator (custom + etcd) Reserve → fraud → debit → credit → emit spans 4 services across regions; 2PC would put a coordinator lock on every account row across regions (DDIA Ch.9: coordinator is single point of contention). Saga with compensating reversals lets the rail keep flowing during partial failures. Distributed 2PC (XA) — rejected: cross-region locks blow the 800 ms p99; Temporal/Cadence considered for v2.
2PC — explicitly rejected for cross-region For 430 M accounts spread across regions, 2PC would require synchronous coordinator round-trips per commit — fundamentally incompatible with 99.99% availability when any region flaps. We use it only inside a single Postgres txn (ledger + idempotency row). Saga (chosen); Spanner TrueTime (used by competitors, but we'd inherit the cost & lock-in).
Idempotency keys — UNIQUE constraint Client-generated key + Postgres UNIQUE index gives bulletproof at-most-once semantics for retries (DDIA Ch.7: "exactly-once = at-least-once + idempotent"). Insert before side effect; commit response in same txn as ledger debit. Redis-only dedup (loses on failover); request hashing alone (false negatives when client retries with new timestamp).
Double-entry ledger (sum=0 per currency) Money cannot be created or destroyed — every of our ~25 B txns/year writes ≥2 balanced entries. Deferred SQL constraint enforces it at commit so no application bug can produce a phantom credit. The integrity invariant is in the database, not in code. Single-entry "balance" column (rejected: no audit trail, no point-in-time, no proof of conservation).
Debezium (CDC) — transactional outbox Ledger commit + Kafka publish must be atomic; we write to an outbox table inside the same Postgres txn (DDIA Ch.11 transactional outbox), Debezium tails WAL and republishes. Zero lost events even if Kafka is down at commit time. Dual-write (ledger + Kafka directly) — rejected, classic correctness bug; Postgres LISTEN/NOTIFY (not durable, no replay).
etcd (Raft) — saga & FX desk leader The orchestrator's in-flight saga state needs a linearizable KV with leader election (DDIA Ch.9 Raft). 5 K peak TPS of saga state changes well within etcd's envelope; lease semantics handle node failures cleanly. ZooKeeper (heavier, JVM ops burden); Consul (weaker linearizability story for our use).
Sub-ledgers — hot account contention A marketplace seller can receive 1 K txns/s alone; SSI on a single row would abort half. Hash(txn_id) mod N spreads writes across N sub-rows; the read path sums them. Trade-off: one extra GROUP BY per balance read. Optimistic concurrency on a single row (abort rate too high); Redis counter (loses durability).
