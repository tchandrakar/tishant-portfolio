Design Stripe (payments-as-API for merchants)

Hard payments ecommerce

1. Problem statement

Design a developer-first payments platform: a merchant integrates a single REST API and gets card acceptance across 135 currencies, 3DS, subscriptions, payouts, fraud (Radar), and webhooks. The product is the API surface: idempotent, predictable, and absurdly available. Behind it sits a card-network gateway, a ledger, risk, and a connector mesh to acquirers (Visa/Mastercard/Amex direct or via processors).

2. Functional requirements

  • Create PaymentIntent → confirm → capture (auth/capture or auto-capture).
  • Tokenize card (PAN never touches merchant servers; PCI scope minimization).
  • Subscriptions: recurring billing, dunning, proration.
  • Payouts to connected accounts (Stripe Connect): split, hold, schedule.
  • Webhooks with at-least-once delivery, signed, retried with exponential backoff up to 3 days.
  • Radar: real-time fraud score; rules engine merchants can edit.

3. Non-functional requirements

  • API availability: 99.999% for /charges and /payment_intents (5 nines is the brand promise).
  • Latency: p99 < 200 ms for charge.create (excluding network round-trip to acquirer).
  • Scale: 500 M API requests/day at peak (~6 K RPS sustained, 50 K RPS spikes on Cyber Monday).
  • Idempotency: any request safely retryable for 24 hours.
  • Compliance: PCI-DSS Level 1, SOC 2, regional data residency (EU, IN, BR).

4. Capacity estimation

  • Charges: ~10 B/year ≈ 320/s avg, 50 K/s peak; storage 10 B × 4 KB × 5 yrs ≈ 200 TB.
  • Webhook fan-out: 1 charge → ~6 events × 2 endpoints/merchant = 12 deliveries; ~3 K deliveries/s avg.
  • Token vault: 2 B PCI tokens × 200 B = 400 GB (HSM-backed).
  • Idempotency store: 24 h × 320 RPS × 2 KB = 55 GB (Redis with TTL).

6. High-level architecture

[Merchant SDK] → [Edge/Anycast] → [API GW] → [Idempotency Cache (Redis)] │ ▼ [PaymentIntent Svc] ┌──────────────────┼──────────────────┐ ▼ ▼ ▼ [Token Vault] [Ledger (Postgres)] [Risk/Radar] (HSM) [Mongo for objects] (Flink + ML) │ ▼ [Acquirer Connectors] (Visa/MC/Amex/local rails) │ ▼ [Kafka: events] │ [Webhook Dispatcher]

Stripe's real internal stack uses MongoDB for object storage (objects are largely document-shaped) plus a separate strongly-consistent ledger. We accept that pattern: Mongo for the object (PaymentIntent state machine), Postgres for the money.

7. Data model & storage

7.1 PaymentIntent (Mongo)

{ _id: "pi_xxx", amount: 2500, currency: "usd",
  status: "requires_capture",
  charges: [{id: "ch_xxx", outcome: "approved"}],
  client_secret: "...", created: 1730000000,
  livemode: true, idempotency_key: "...", merchant_id: "acct_xxx" }

7.2 Ledger (Postgres, sharded by merchant_id)

CREATE TABLE postings (
  posting_id BIGSERIAL,
  txn_id     UUID,
  account    BIGINT,    -- merchant balance / fees / reserve
  amount     NUMERIC(20,4),
  currency   CHAR(3),
  posted_at  TIMESTAMPTZ,
  PRIMARY KEY (account, posting_id)
);

Why split? The PaymentIntent doc evolves through 8+ states; doc DB is natural. The ledger needs serializable invariants: every txn balances to zero per currency. DDIA Ch.7 SSI on the merchant_balance row prevents over-payout when two refunds race.

8. Detailed component design

8.1 Idempotency

Two-tier: Redis (hot, 24 h TTL) backed by Postgres unique constraint (durable). On request, lookup key → if present and request hash matches, return cached response; if hash differs, return 409. Critical: we record the response only after the side effect completes, but we record the in-flight marker before, so a crash mid-call resolves to a clean retry.

8.2 Webhook delivery

Events written to Kafka via the transactional outbox in the same Postgres txn as the ledger write. Dispatcher pulls, signs (HMAC-SHA256), and POSTs. Retries: 1 m, 5 m, 30 m, 2 h, 6 h, … capped at 3 days. Per-merchant rate limit prevents one slow endpoint from starving others (DDIA Ch.11: backpressure via partition assignment).

8.3 Acquirer connectors

Each card network has a stateful TCP session with heartbeats. Connectors are stateless, backed by a session pool. Timeouts are asymmetric: we time out at 8 s but treat the txn as UNKNOWN not failed — and reconcile with the network's daily file the next day.

9. Scaling & bottlenecks

  • Hot merchant on Black Friday: shard merchant ledger by (merchant_id, day_bucket) so writes spread across partitions; aggregate balance via a materialized view refreshed every minute.
  • Webhook poison endpoint: per-endpoint circuit breaker; deliveries quarantined to a slow lane.
  • Acquirer outage: retryable error path falls through to a backup connector (different processor for the same scheme).
  • 5-nines API: active-active multi-region with cell-based isolation; idempotency keys are cell-affinitized via consistent hashing so retries land on the same cell.

11. DDIA references

  • Ch. 7 Transactions — idempotency keys are the canonical "idempotent operation" example; SSI on merchant_balance avoids write skew on concurrent payouts/refunds; single-object atomic writes power the in-flight marker.
  • Ch. 7 Transactions — discussion of "exactly-once" reframed as "at-least-once + idempotency" is literally the Stripe API contract.
  • Ch. 9 Consensus — Raft (etcd) for connector leader election so only one node owns a card-network session; linearizable reads of PaymentIntent status.
  • Ch. 11 Stream Processing — transactional outbox + Kafka for webhooks; reprocessing on dispatcher restart.

12. Talking points for the interviewer

  • "The API contract is the system: idempotency, versioning, and predictable errors are first-class architectural concerns, not afterthoughts."
  • "I'd push back on 'exactly-once'; what we deliver is at-least-once events plus deterministic deduping — that's what Kafka and stripe.com both actually do."
  • "Reconciliation with acquirer settlement files runs nightly; it's the safety net for every UNKNOWN."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Postgres (SSI) — money ledger 10 B charges/yr need serializable isolation on merchant_balance: two concurrent refunds would both pass an "available >= refund" check under read-committed (DDIA Ch.7 write skew). SSI aborts the loser; app retries. Single-node ACID for charge + ledger + idempotency in one COMMIT. MySQL (no true SSI); Aurora (close, but we want predictable autovacuum behavior under 50 K RPS spikes).
MongoDB — PaymentIntent objects The PaymentIntent state machine has 8+ states and per-merchant variant fields (3DS challenges, partial captures, statement descriptors). Document model fits naturally; per-document atomicity is sufficient since money lives in Postgres. Matches Stripe's actual production split. Postgres JSONB (works but losing schema-flex evolution speed); DynamoDB (no rich queries on charges array).
Cassandra — explicitly rejected for ledger LWW + tunable consistency cannot enforce sum=0 invariants or detect write skew on merchant_balance. Eventual consistency on money is a correctness bug, not a tradeoff. Used only for high-volume append-only telemetry (request logs). n/a — this is the rejection.
Kafka — webhook event bus 3 K webhook deliveries/s × at-least-once × 3-day retry window needs durable replayable log. Per-merchant partitioning gives ordered delivery and natural backpressure (DDIA Ch.11): one slow endpoint can't starve others, since assignment is per-partition. SQS (no replay, no per-key ordering); Pub/Sub (weaker ordering at our scale).
Flink — Radar fraud features Velocity (txns/min), distinct-card counts, and BIN-level aggregates need windowed stateful joins on the live charge stream. Flink's exactly-once checkpointed state (DDIA Ch.11) keeps features correct even after dispatcher restarts; 5 K avg / 50 K peak TPS is well within envelope. Kafka Streams (state grows past per-task limits at our retention); Spark Streaming (micro-batch latency exceeds our 200 ms p99 budget).
Redis — idempotency hot cache 50 K RPS × 24 h TTL = 55 GB of keys; Redis returns prior responses in < 1 ms so retries don't re-enter the orchestrator. Backed by Postgres unique index for durability — Redis loses on failover, Postgres doesn't. Memcached (no persistence option); in-process cache (no cross-cell consistency).
HSM (token vault) 2 B PCI tokens × 200 B = 400 GB of PANs that are PCI-DSS Level 1 in scope. HSM tokenization isolates the vault to a tiny audited surface so the rest of the platform stays out-of-scope (regulatory, not just technical). Software KMS only (fails PCI Level 1 audit); third-party tokenization (latency + vendor lock).
Saga (cell-based) — over 2PC 5-nines availability requires cell isolation; 2PC across cells would couple their failure domains. Saga with compensating refunds tolerates a cell going dark without losing the API contract. Idempotency keys are cell-affinitized via consistent hashing so retries land where the original lives. Distributed 2PC (XA) — kills cell isolation; Spanner txns (vendor lock, cost).
2PC — used only inside Postgres Single-DB 2PC (prepared txn) is fine to bind charge insert + ledger postings + idempotency update atomically. We never extend it across services or regions because coordinator failure semantics (DDIA Ch.9) destroy the 99.999% target. App-level dual writes (correctness bug); cross-service XA (latency & coordinator risk).
Idempotency-Key — two-tier (Redis + PG) Required for "any request safely retryable for 24 h." Two-tier: hot path through Redis (sub-ms), correctness path through Postgres UNIQUE constraint inside the same txn as the side effect (DDIA Ch.7). Hash mismatch returns 409 to catch client bugs. Redis-only (loses on failover); request-body hash only (no client-controlled retry semantics).
Transactional outbox (Debezium → Kafka) Webhook events must commit atomically with the charge — dual-writing to Kafka risks dropped events. Outbox table written inside the Postgres txn; Debezium tails WAL and publishes (DDIA Ch.11). Zero event loss even on Kafka outage. Direct Kafka publish in handler (lost events); LISTEN/NOTIFY (not durable, no replay).
etcd (Raft) — connector leadership Each card-network TCP session must have exactly one owner (Visa won't tolerate parallel sessions for the same BIN); leader election via Raft (DDIA Ch.9) gives linearizable lease + failover under 5 s. ZooKeeper (heavier ops); Consul (weaker linearizability story).
Double-entry ledger (postings) Stripe takes a fee, the merchant gets net, the platform takes a slice — every charge writes ≥3 balanced entries summing to zero per currency. Deferred constraint enforces conservation at commit; reconciliation against acquirer files is mathematically possible only because of double-entry. Single-row balance (no audit, no proof of conservation, no point-in-time).
