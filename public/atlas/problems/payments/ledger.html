Design Double-Entry Ledger System (immutable, balanced, auditable)

Hard payments database-ddia

1. Problem statement

Build the system of record for money: every value movement is recorded as a balanced double-entry posting (debits = credits per currency), entries are immutable, balances are derivable from entries, and the whole structure is auditable to any point in history. Used by banks, fintechs (Stripe, Square, Wise, Revolut), exchanges, and any system that cannot afford "the database lost a transaction".

2. Functional requirements

  • Post a transaction = atomic group of ≥ 2 entries summing to zero per (currency, partition).
  • Read account balance at now or at any past timestamp (point-in-time).
  • Append-only entries; corrections via reversing entries, never updates.
  • Multi-currency, multi-account-type (asset, liability, equity, revenue, expense).
  • Pending → posted lifecycle (e.g., card auth → capture).
  • Reports: trial balance, P&L, account statement.
  • Audit trail with cryptographic hash chain (optional but increasingly standard).

3. Non-functional requirements

  • Correctness invariant: for every txn, sum(entries) == 0 per currency. Always. Enforced in DB, not just app.
  • Immutability: no UPDATE or DELETE on the entries table — period.
  • Throughput: 10 K postings/s sustained, 100 K bursts; each posting writes ~3 entries.
  • Latency: commit p99 < 30 ms.
  • Auditability: entries reproducible byte-for-byte from the WAL; 7+ year retention.

4. Capacity estimation

  • Entries/year: 10 K × 3 × 86400 × 365 ≈ 95 B entries; ×7 yr retention ≈ 670 B rows.
  • Row size 200 B → ~135 TB raw, ~400 TB replicated (3×).
  • Active accounts: 100 M; balance cache size 100 M × 64 B × 25 currencies = ~160 GB Redis.
  • Hot account write rate: top 0.1% of accounts handle 50% of postings → contention design needed.
  • WAL throughput: 10 K postings × 3 entries × 200 B = 6 MB/s sustained.

5. API design

POST /v1/transactions
  Body: {
    txn_id: "uuid",            // idempotency
    description: "Charge cust_42 for invoice inv_99",
    entries: [
      {account: "user:42:wallet", amount: -2500, currency: "USD"},
      {account: "stripe:fees",    amount:    75, currency: "USD"},
      {account: "merchant:7:rev", amount:  2425, currency: "USD"}
    ],
    posted_at: "2026-05-06T10:00:00Z"
  }
  → {txn_id, status: POSTED}

GET /v1/accounts/{id}/balance?as_of=2026-05-01T00:00:00Z
  → {currency, available, pending}

GET /v1/accounts/{id}/entries?from=...&to=...
  → [{entry_id, txn_id, amount, currency, posted_at}]

7. Data model & storage

7.1 Schema (Postgres, sharded by account_id hash)

CREATE TABLE txn_headers (
  txn_id        UUID PRIMARY KEY,
  posted_at     TIMESTAMPTZ NOT NULL,
  description   TEXT,
  prev_hash     BYTEA,                 -- hash chain
  hash          BYTEA NOT NULL,
  CONSTRAINT immutable CHECK (false) NO INHERIT  -- enforced via revoke
);

CREATE TABLE entries (
  entry_id      BIGSERIAL,
  txn_id        UUID NOT NULL REFERENCES txn_headers,
  account_id    BIGINT NOT NULL,
  amount        NUMERIC(20,4) NOT NULL,    -- signed; integer minor units recommended
  currency      CHAR(3) NOT NULL,
  posted_at     TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (account_id, entry_id)
) PARTITION BY HASH(account_id);

-- Materialized running balance, updated in same txn
CREATE TABLE account_balances (
  account_id    BIGINT,
  currency      CHAR(3),
  available     NUMERIC(20,4),
  version       BIGINT,                    -- monotonic per account
  PRIMARY KEY (account_id, currency)
);

7.2 The zero-sum constraint

Enforced two ways: (a) deferred SQL constraint at commit time checking sum(amount) GROUP BY currency = 0; (b) application-level pre-check. Both, because defense in depth.

7.3 Why integer minor units?

Floats are forbidden — IEEE-754 cannot represent $0.10 exactly. Use NUMERIC(20,4) or store as integer cents (USD), paise (INR), satoshi (BTC).

8. Detailed component design

8.1 Concurrent postings on the same account

Two payments to merchant:7 race. Both want to UPDATE balance. Without isolation, lost-update is possible. Use SSI (Serializable Snapshot Isolation) — DDIA Ch.7 explains exactly this case as write skew: each txn reads a balance, decides it's enough, both commit, balance goes negative. SSI aborts one. Application retries.

8.2 Hot account contention

Stripe fees account, exchange omnibus account: receives a fraction of every txn. Sub-ledger pattern: split into N shards (e.g., stripe:fees:[0..63]), pick at random, aggregate on read. Trade-off: balance reads scan N rows.

8.3 Point-in-time balance

Two strategies:
(a) Replay from entries: SELECT SUM(amount) FROM entries WHERE account_id=X AND posted_at <= T. Correct, slow on hot accounts.
(b) Snapshot table: hourly snapshots; read = nearest snapshot + delta. 99% queries hit fast path.

8.4 Hash chain & tamper-evidence

hash = SHA-256(prev_hash || canonical(txn)). Daily, root hash is published (S3, blockchain, or a notary). Any tampering with old entries invalidates the chain — auditors love it. (DDIA Ch.5: hash-pointer-style log.)

8.5 Pending vs posted

Card authorizations create pending entries on a separate "in-flight" account; capture moves them to "available". Reservations expire via a sweeper that posts compensating entries. Never mutate.

9. Scaling & bottlenecks

  • Sharding: hash by account_id; cross-account postings (most txns) span shards → use a 2-phase mini-protocol or a single coordinator shard per txn.
  • Multi-shard txns: for cross-shard postings, write to a global txn coordinator with a pending marker, then fan out per-shard inserts; finalize via 2PC (Postgres prepared txn) or move to CockroachDB / TiDB which give Spanner-style txns natively.
  • Read replicas for reports: CDC into a columnar store (Clickhouse / Snowflake) for analytics; never run reports on the OLTP primary.
  • Cold storage: entries older than 90 days move to compressed Parquet on S3, queryable via Athena; balance cache stays current via running totals.

11. DDIA references

  • Ch. 7 Transactions — this is the archetypal chapter for ledgers. Write skew on balance checks (Ch.7's "doctors on call" example translated to "two withdrawals on one account") motivates SSI. Single-object atomic writes underpin idempotency. Deferred constraints are exactly the zero-sum check. The chapter's discussion of "what isolation level you actually need" maps 1:1 to ledger requirements.
  • Ch. 7 Transactions — 2PC + its failure modes are the design constraint that pushes us toward single-shard txns where possible, and toward CockroachDB/Spanner when not.
  • Ch. 9 Consensus — total order broadcast is what gives us a globally consistent entry sequence; Raft for the txn coordinator's leader election; linearizable reads for "what's my balance right now".
  • Ch. 9 Consensus — quorum reads/writes for cross-region replication of the ledger; FLP impossibility reminding us that "no false positives, no false negatives, fast" can't all hold under partition.
  • Ch. 11 Stream Processing — entries are an event log; downstream reports, balance projections, and analytics are derived state — classic event sourcing.
  • Ch. 5 Replication — synchronous replicas in the same region for durability; async cross-region with documented RPO.

12. Talking points for the interviewer

  • "Double-entry isn't accounting cosplay; it's an integrity invariant the database can enforce. Sum-to-zero is a constraint, not a guideline."
  • "Immutability is the point. If you ever feel the urge to UPDATE an entry, you write a reversing entry instead. This is what makes audits possible without time travel."
  • "The hardest real-world problem is hot accounts (the omnibus); sub-ledgers solve it but trade write contention for read aggregation."
  • "I'd reach for Postgres + SSI for a single-region 10 K TPS system; for multi-region active-active, CockroachDB or Spanner — the cost of building Spanner yourself is not worth it."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Postgres (SSI) — entries & balances 10 K postings/s with a 30 ms p99 commit fits a sharded primary; SSI (DDIA Ch.7) is essential because two concurrent withdrawals on one account both passing a "balance >= amount" predicate is the textbook write-skew bug that costs banks millions. Single-shard txns commit entries + balance update + outbox atomically. MySQL (no real SSI); Aurora (close, but autovacuum behavior at 95 B entries is unproven for us).
CockroachDB / Spanner — multi-region option For active-active across regions or cross-shard txns spanning 100+ shards, Spanner-style external consistency via TrueTime/HLC (DDIA Ch.9) gives single-system-image txns without app-level 2PC. The 10 K TPS sustained, 100 K bursts envelope is well within both products. Postgres + 2PC across shards (coordinator failure modes); TiDB (kept as Spanner alternative).
Cassandra — explicitly rejected for entries The zero-sum invariant cannot be enforced under LWW eventual consistency: a delayed replica could surface a phantom credit. Cassandra LWTs are too weak (no group atomicity across multi-row entries). It does not touch this system. n/a — this is the rejection.
Kafka — outbox / event sourcing bus Every posting fans out to reporting, downstream services, and audit. 10 K postings/s × 3 entries × multiple consumers needs durable replayable log with per-account ordering (partition key = account_id) so derived projections (DDIA Ch.11 stream-table duality) stay consistent. RabbitMQ (no replay window for retro-projections); SQS (no per-key ordering).
Debezium — CDC outbox publisher Outbox table written inside the same Postgres txn as the entries; Debezium tails WAL and republishes to Kafka (DDIA Ch.11 transactional outbox). Guarantees no posting goes unannounced even if Kafka is down at commit, and never publishes a rolled-back posting. Dual-write from app (lost-event bug); Postgres LISTEN/NOTIFY (not durable, no replay).
Redis — balance read cache 100 M accounts × 25 currencies × 64 B = ~160 GB; sub-ms balance reads for "what's available now?" without hitting the OLTP primary on every check. Cache invalidated by outbox events; eventual consistency is fine because the source of truth (entries) remains the durable arbiter. Memcached (no Hash type for multi-currency per account); in-process (no cross-pod consistency).
Clickhouse / Snowflake — reporting Trial balance, P&L, statement queries scan billions of entry rows; columnar OLAP is 100×+ faster than running on the OLTP primary. CDC into the warehouse keeps it live within minutes; never run analytics on the ledger primary or you'll starve postings. Postgres read replica (row-store too slow for full-scan reports); Druid (more ops complexity for our query mix).
S3 + Parquet — cold storage Entries older than 90 days move to compressed Parquet; 7-year retention × 670 B rows would cost a fortune in OLTP storage. S3 + Athena gives queryable archives at 1/20th the cost while running balance recomputation jobs nightly. Glacier (recall latency too high for occasional audit query); keep all in OLTP (cost-prohibitive at PB scale).
Double-entry (sum=0 per currency) The defining invariant. Every txn writes ≥2 balanced entries; deferred SQL constraint sum(amount) GROUP BY currency = 0 enforces it at commit time so no application bug can create money. This is what makes reconciliation possible at all. Single-entry (no proof of conservation); app-only check (defense-in-depth missing).
Hash chain (SHA-256) — tamper evidence hash = SHA-256(prev_hash || canonical(txn)); daily root hash published externally. Tampering with any old entry breaks the chain. Cheap (32 B per row), expensive to forge — auditors love it. Per-shard chains avoid global serialization. Merkle tree (overkill for sequential audit); HMAC only (no proof of order).
2PC (Postgres prepared txn) — cross-shard For postings that span shards (most txns!), Postgres's PREPARE TRANSACTION + COMMIT PREPARED gives true atomic commit across shards. Used reluctantly (DDIA Ch.9: coordinator failure modes are real) — wrapped in a coordinator with timeout-based recovery. Saga (compensating reversals are messy when zero-sum must hold instant-by-instant); single-shard fanout (impossible for inter-account txns).
Saga — alternative for autonomous-shard mode If we drop 2PC for higher availability, a saga writes a pending-marker txn header, fans out per-shard inserts, then finalizes. Compensating reversals on partial failure restore the sum-to-zero invariant by writing reversing entries — never UPDATE. 2PC (chosen for strict consistency); raw async (loses zero-sum guarantee).
Idempotency keys — txn_id UNIQUE The txn_id is the idempotency key; UNIQUE on txn_headers means a retry never double-posts (DDIA Ch.7 single-object atomic write). Combined with same-txn entry inserts, the entire posting is idempotent end-to-end. Server-generated id (defeats client retry semantics); request hash (false negatives on metadata diff).
Sub-ledger pattern — hot accounts Stripe-fees and exchange-omnibus accounts receive ~50% of postings; SSI on a single row aborts everything. Split stripe:fees:[0..63] randomly; aggregate on read. Trade-off: balance reads scan 64 rows — acceptable because reads are rare relative to writes here. Optimistic concurrency on single row (abort rate too high); Redis counter (no durability, no audit).
Raft (etcd) — txn coordinator leader Cross-shard postings need a single coordinator per txn to prevent double-fanout; Raft-elected leader (DDIA Ch.9) gives clean failover. Lease-based so a partitioned old leader can't double-finalize after a new one takes over. ZooKeeper (older, heavier ops); DB-row-based election (slower failover, weaker semantics).
