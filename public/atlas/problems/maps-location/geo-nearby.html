Design Geo-search Nearby (top-K within radius, geohash + KD-tree)

Hard maps-location search

1. Problem statement

Design the generic primitive that powers Uber driver lookup, Tinder matches, food-delivery couriers, scooter availability: "given (lat, lng), return top-K POIs within radius r, ranked by distance/score, in <30 ms p99". POIs are mobile (drivers move) so writes are 100 K+/s; reads are 10× higher; index must support filters (e.g., car_type=XL, available=true). The point of this problem is the algorithmic core: geohash, S2, R-tree, KD-tree.

2. Functional requirements

  • Insert/update an entity's location (entity_id, lat, lng, attrs).
  • Delete/expire an entity.
  • Query top-K within radius r of (lat, lng), with attribute filters, sorted by distance.
  • Bulk-load (cold start) and snapshot/restore.
  • Range query: all entities inside a polygon (e.g., a city).

3. Non-functional requirements

  • Latency: point query p99 < 30 ms; update p99 < 10 ms.
  • Scale: 50 M live entities, 100 K wps, 1 M rps.
  • Freshness: updates visible to queries < 2 s.
  • Availability: 99.99% with hot-standby per shard.
  • Correctness: radius queries must not miss entities at cell boundaries.

4. Capacity estimation

  • 50 M entities × 100 B (id+lat+lng+attrs) = 5 GB — fits in RAM per region.
  • Sharding: 12 regions × ~4 M entities each → single node serves a region from 1 GB.
  • Reads: 1 M rps / 12 ≈ 83 K rps per shard — comfortably handled by an in-process index.
  • Writes: 100 K wps; an R-tree mutation is O(log n) ≈ 25 nodes touched, ~40 µs CPU.
  • Memory budget: KD-tree node ≈ 40 B → 50 M × 40 B = 2 GB.

6. High-level architecture

[Producers]──updates──▶[Ingest API]──▶[Kafka: loc.updates] │ ▼ ┌─────────────[Geo Shard Router (S2 cell → shard)] ▼ ▼ [Geo Shard 0..N] [Geo Shard N..] ┌───────────────┐ ┌───────────────┐ │ in-RAM index: │ │ in-RAM index: │ │ geohash buckets │ │ KD-tree │ │ + R-tree on attrs│ │ + R-tree │ │ WAL → S3 every 5s│ │ │ └───────────────┘ └───────────────┘ ▲ ▲ └──────[Query Coordinator]──────┘ ▲ │ [Clients]

Updates are partitioned by S2 cell (level 8) so geographically close entities co-locate, making radius queries single-shard ~95% of the time. Cross-shard queries fan out to ≤4 shards (a 4-cell cover at the boundary). Every shard keeps a write-ahead log to S3 + periodic snapshot for recovery in <30 s.

7. Data model & index choices

7.1 Geohash bucketing (simplest)

HashMap<geohash7, List<Entity>>   // 153 m × 153 m cells
Query: cover the search disk with 9 neighbour cells, scan, filter by haversine.

7.2 KD-tree (best for static-ish data)

Recursive median split on alternating lat/lng axes.
Build O(n log n); kNN query O(log n) expected; range O(√n + k).
Pros: tight pruning; Cons: rebalance on heavy churn.

7.3 R-tree (good for moving entities + attribute filter)

Bounding-box hierarchy; supports rectangles, polygons, kNN.
Update is O(log n) but expensive on splits → use R*-tree + bulk reinsertion.

7.4 S2 / Uber H3 (cell-id approach)

Hierarchical, equal-area cells. Index is just a sorted map cell_id → entities; a radius query computes a cell covering. This is what Uber uses for driver dispatch (H3 hex grid).

8. Detailed component design

8.1 Query path

1) Compute the disk's S2 covering at level 12 (≈ a few cells for r=1 km). 2) Route to owning shard(s). 3) Each shard scans candidate buckets, applies haversine, attribute filters, then a min-heap of size K. 4) Coordinator merges results across shards — total candidates rarely >1 K so the heap merge is microseconds.

8.2 Update path

Producer → Kafka (key = entity_id) so updates for one entity are ordered. Shard consumer applies in-RAM, appends to WAL. Tombstones expire after 60 s of staleness — e.g., a driver who stopped pinging is dropped from the index.

8.3 Hot-cell mitigation

Times Square at peak has 50 K entities in one S2 level-12 cell. We split overflowing cells dynamically into a child level (12→13). Queries that land on a split cell descend.

8.4 Boundary correctness

Always cover with cells strictly larger than the radius circumcircle. With S2 the formula is: pick level whose cell width > r, then take a 4-neighbour cover. Validated unit tests at every coastline.

9. Scaling & bottlenecks

  • Re-shard during regional growth: S2 cells subdivide on threshold; consistent-hash ring uses cell prefix.
  • GC pressure (Java/Go): off-heap structures (chronicle map / mmapped flat files) for the entity table.
  • Write amplification: coalesce updates from same entity in 1 s windows (DDIA Ch. 11 stream).
  • Snapshot consistency: copy-on-write tree pointers; readers continue while a snapshot writer iterates.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
S2 cells (level 8 partitioning, level 12 query covering) 50 M moving entities at 100 K wps need geographic partitioning that keeps neighbours co-located so radius queries hit one shard 95% of the time. S2's hierarchical equal-area cells let level-8 shard routing match level-12 query covering naturally — and equal-area means a 1 km radius behaves identically in Helsinki and Singapore. Geohash (rectangle, 2× area distortion at high latitudes); H3 (hex shape, harder radius cover math); Quadtree (no equal-area).
H3 (Uber-style hex grid) — alternative Documented as the index option for ride-dispatch flavours of this problem. Hexagonal cells have uniform neighbour distance (every neighbour is exactly one ring away), which simplifies driver-rider matching where "expand search by one ring" is the dominant primitive. S2 (square neighbours have 4 vs 8 distinct distances); custom Voronoi (heavy to maintain at 50 M churn).
Geohash buckets (v1 simplicity) HashMap<geohash7, List<Entity>> gives O(1) update and 9-cell radius cover for under 200 lines of code — perfect for an MVP serving <10 K wps. We document it as the v1 index because boundary issues only bite at scale. Skip directly to S2 (overkill for v1); R-tree (more code than v1 deserves); PostGIS (operational dependency for a hot-path service).
KD-tree (best for static-ish data) For workloads where churn is <5%/min (e.g., POI lookup, scooter inventory), KD-tree's O(log n) kNN with tight pruning beats bucket scans. Build cost is O(n log n) so we rebuild nightly into a memory-mapped flat file and hot-swap atomically. Ball-tree (better in high dim, but we're in 2D); Cover tree (academic, less tooling); R*-tree (better for moving data, worse for kNN on static).
R-tree / R*-tree (mobile entities + bbox queries) For drivers/couriers moving at 100 K wps, R-tree mutations are O(log n) and it natively supports the "bbox over a city" range query that KD-tree handles awkwardly. R* with bulk reinsertion smooths split costs at the price of ~30% more memory. KD-tree (degrades on churn, no bbox); plain quad-tree (worse splits under skew); BVH (overkill for axis-aligned queries).
Kafka (entity_id-keyed update stream) Producer→shard fan-in at 100 K wps with strict per-entity ordering needs Kafka's per-key ordering guarantee. Keying by entity_id ensures a driver's pings stay sequential even across producer retries; idempotent upserts on the consumer side keep state correct under at-least-once delivery. Kinesis (1 MB/s/shard, would need 100s of shards); RabbitMQ (no key-ordered partitions); SQS (no ordering).
In-process RAM index (per shard) The 30 ms p99 / 10 ms update SLA leaves no budget for network hops to a separate index service. With 50 M × 100 B = 5 GB total, we can fit a region (1 GB) in process and serve 83 K rps per shard locally; an R-tree mutation touches ~25 nodes in 40 µs. Redis GEO (~1 ms RTT plus serialisation = 3-5 ms baseline); ES (~30 ms p99); external KV with custom geo (extra hop).
Write-ahead log to S3 + periodic snapshots In-process indexes lose state on crash. WAL flushed every 5 s + 60 s snapshot lets us replay <65 s of history on restart, hitting the 99.99% availability target with hot-standby and a 30 s recovery window. S3 is durable, cheap, and global. Persist on every write (kills 100 K wps latency); EBS-backed local disk (lost on instance failure); Cassandra-backed (extra hop on hot path).
Dynamic cell splitting (hot-cell mitigation) Times Square's 50 K entities in a single S2 level-12 cell create a head-of-line bottleneck. Splitting overflowing cells to a finer level (12→13) keeps per-cell candidate lists bounded; Uber published H3-based papers describing the same dynamic approach. Static finer level globally (wastes RAM in rural shards); rejection sampling (loses entities); shard hash on lat/lng directly (loses geographic locality).
Off-heap structures (Chronicle Map / mmap) At 50 M entries in JVM/Go runtimes, GC pauses on the entity table can blow past the 10 ms update SLA. Off-heap or mmap-backed flat tables move the bulk allocation outside the GC root set and bring p99 GC pauses from ~50 ms to <2 ms. Plain Java HashMap (GC pauses); Native (Rust/C++) rewrite (months of work); Shenandoah/ZGC (better but still adds tail latency).
Min-heap top-K merge across shards Cross-shard radius queries (≤4 shards at boundary) return ≤500 candidates each; merging into a global top-K via a bounded min-heap costs microseconds. We deliberately cap candidates per shard at 500 to bound coordinator work even when the disk lands on dense urban cells. Sort-then-truncate (O(n log n) when O(n log K) suffices); pull-K-per-shard (loses correctness on skewed shards).
Tombstones with 60 s staleness expiry Drivers who stop pinging shouldn't appear forever. A 60 s tombstone window matches typical GPS drop-out tolerances and triggers explicit removal so radius queries never surface ghost entities — a cheap correctness primitive at this scale. Hard delete on every silence (false positives during tunnel passes); LRU eviction (no time guarantee); client-side filtering (wastes bandwidth).
Copy-on-write tree pointers (snapshot consistency) Snapshots must capture a consistent view without blocking 100 K wps. CoW lets the snapshot writer iterate a frozen root while readers and writers continue against new versions — a textbook MVCC-on-an-index pattern that costs <5% memory overhead. Read-write lock during snapshot (kills latency SLA); STW pause (violates 10 ms update SLA); double-buffered indexes (2× RAM).
