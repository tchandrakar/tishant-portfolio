Design Routing Engine (Dijkstra → contraction hierarchies for road graphs)

Very Hard maps-location

1. Problem statement

Compute the shortest/fastest A→B path on the global road graph (~150 M nodes, ~400 M edges) for 10 B requests/day. Naive Dijkstra is O(E log V) — too slow. Walk through the algorithmic ladder: Dijkstra → bidirectional Dijkstra → A* → ALT → Contraction Hierarchies (CH) → Customizable Contraction Hierarchies (CCH). Discuss live-traffic-aware re-weighting, turn restrictions, multi-modal extensions, and how to keep p99 query < 100 ms with sub-second updates after live-traffic changes.

2. Functional requirements

  • Driving / walking / cycling / transit modes; each is a separate weighted graph.
  • Alternative routes (top-3) with detour ratio constraints.
  • ETA with live traffic; arrival-time-dependent routing (rush hour models).
  • Turn restrictions, one-ways, vehicle restrictions (truck weight, EV charging).
  • Re-route on deviation in <200 ms.
  • Bulk waypoint optimisation (delivery TSP up to 25 stops).

3. Non-functional requirements

  • Latency: p99 route compute < 100 ms, isochrone < 500 ms.
  • Scale: 10 B routes/day ≈ 115 K rps avg, 400 K peak.
  • Freshness: traffic edge updates visible to queries < 60 s.
  • Footprint: CH-augmented graph 80 GB/continent in RAM.
  • Correctness: bit-exact equivalence with plain Dijkstra under static weights (validation suite).

4. Capacity estimation

  • Plain Dijkstra on 400 M edges: ~10 s/query — ✗.
  • Bidirectional Dijkstra: ~1 s — ✗.
  • A* with great-circle heuristic: ~200 ms cross-country — borderline.
  • CH: ~1 ms typical, ~10 ms continental — ✓.
  • Pre-compute: CH on a continent ≈ 6 h on a 32-core box; output ≈ 2× edge memory.
  • Servers: 400 K rps / 1 K rps per node = 400 nodes (CH does ~3 K rps per core).
  • Memory: 80 GB graph + 4 KB per concurrent search = 80 GB + 16 GB workspace.

6. High-level architecture

[Client]──▶[Route API GW]──▶[Region Router] │ ┌─────────────────┼─────────────────┐ ▼ ▼ ▼ [CH Shard NA] [CH Shard EU] [CH Shard ASIA] ┌──────────┐ ┌──────────┐ ┌──────────┐ │CH graph │ │CH graph │ │CH graph │ │+ traffic │ │+ traffic │ │+ traffic │ │overlay │ │overlay │ │overlay │ └──────────┘ └──────────┘ └──────────┘ ▲ ▲ ▲ │ │ │ [Live Traffic Pub/Sub: edge_id → speed_factor] ▲ │ [Probe Aggregator (Flink)] ▲ │ [Kafka: probe pings] [Cross-region Overlay Graph (~5K boundary nodes)] ── used for inter-shard routes

The graph is partitioned by continent; cross-region trips use a small "overlay" of well-chosen border nodes whose pairwise CH distances are precomputed. Live traffic is a pub/sub topic that updates an edge-multiplier table; the CH stays valid because we re-rank only candidate routes via on-the-fly recomputation against the multiplier (Customizable CH).

7. Data model & storage

7.1 Static graph (file-format on RAM)

nodes  : node_id u32 → (lat f32, lng f32, country u8)
edges  : edge_id u32 → (src u32, dst u32, len_m u16, speed_class u8, restrict u16)
fwd_csr / rev_csr : compressed-sparse-row adjacency for cache locality

7.2 CH augmentation

ch_rank[node_id] u32          // contraction order
shortcuts: array of (u, v, weight, via_node)  // added during preprocessing

7.3 Live traffic overlay

edge_factor[edge_id] f16      // multiplicative slowdown, 1.0 = free flow
mmap'd, atomic per-edge writes; readers see consistent snapshot via epoch counter

7.4 Why this layout?

Graph algorithms are memory-latency bound: CSR + 32-bit ids + sequential traversal saturate L2 cache lines. We deliberately avoid pointer-chasing object graphs.

8. Detailed algorithm walk-through

8.1 Bidirectional Dijkstra

Search forward from s and backward from t simultaneously; stop when their settled sets meet. Halves visited nodes vs unidirectional. Foundation for CH.

8.2 A* with landmarks (ALT)

Pick K landmarks (~16); precompute distance from every node to each landmark. h(v,t) = max_l |d(l,v) − d(l,t)| satisfies triangle inequality. Speedup ~10×, but landmarks weak in long routes.

8.3 Contraction Hierarchies (CH)

Pre-compute: order nodes by importance; iteratively "contract" lowest-importance node by adding shortcut edges between its neighbours that preserve shortest paths. Query: bidirectional Dijkstra that only relaxes upward in the hierarchy. Result: 1000× speedup, 1-2 ms typical query on a continent.

8.4 Customizable CH (CCH)

Pre-compute hierarchy structure independent of weights; "customisation" stage applies fresh weights in ~5 s. Lets us push live-traffic edge factors and re-customise every 1-2 minutes — keeps CH valid under dynamic weights.

8.5 Turn restrictions

Edge-based graph: nodes become (edge, turn) pairs. Doubles graph size; required for "no left turn" rules. Done before CH.

9. Scaling & bottlenecks

  • Cross-continent routes: overlay graph of 5 K border nodes; CCH within each continent + Dijkstra on overlay.
  • Customisation cost: 5 s per continent every minute = continuous churn; we pipeline two graph copies (active + warming).
  • Memory bandwidth: NUMA-aware allocation per socket; pin threads.
  • Long routes (NYC→LA): CH search visits ~1 K shortcuts; still < 10 ms.
  • Time-dependent routing: store departure-time profiles per edge; query becomes time-dependent CH (TCH).

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Contraction Hierarchies (CH) — query algorithm Plain Dijkstra on 400 M edges is ~10 s/query, bidirectional ~1 s, A*+ALT ~200 ms — none meet the 100 ms p99 at 400 K rps peak. CH precomputes shortcuts that cut visited nodes 1000× to 1-2 ms typical, fitting 80 GB/continent into RAM and serving ~3 K rps/core. Plain Dijkstra (10 s); bidirectional Dijkstra (1 s); A* with great-circle heuristic (200 ms); Hub Labels (faster but 10× memory).
Customisable Contraction Hierarchies (CCH) Pure CH freezes weights, but live traffic shifts edge times every minute. CCH separates structure pre-compute from a 5 s "customisation" pass that re-applies fresh weights, letting us push traffic factors continuously without redoing 6 h of preprocessing. CH + edge multipliers only (breaks shortcut validity under non-monotone changes); recomputing CH from scratch (6 h, impossible per minute); A* online (200 ms, blows SLA).
ALT (A* + Landmarks + Triangle inequality) Documented as a fallback for graphs too dynamic for CCH. Pre-computing distances from 16 landmarks lets the heuristic h(v,t)=max_l |d(l,v)−d(l,t)| satisfy admissibility, giving ~10× speedup over plain A* without preprocessing the entire hierarchy. Plain A* (heuristic too weak across continents); CH (preprocessing inflexible); arc-flags (similar speedup, harder to update on traffic changes).
Bidirectional Dijkstra (foundation) Halves visited nodes vs unidirectional and is the algorithmic foundation CH builds on. We keep it as the validation oracle: under static weights, CH must produce bit-exact results matching bidirectional Dijkstra — that test catches preprocessing bugs. Unidirectional (2× slower); A* alone (heuristic-dependent); IDA* (memory wins don't matter when we have 80 GB RAM).
CSR (compressed sparse row) graph layout Routing is memory-latency bound — every shortcut relax is a cache-line load. CSR packs neighbours sequentially so a ~5-neighbour-per-node iteration prefetches cleanly into L2; combined with 32-bit node IDs, the working set per query stays under 4 KB. Adjacency-list-of-pointers (cache misses on every hop); adjacency matrix (1.6×10¹⁷ entries, impossible); halfedge mesh (overkill for 1D edges).
RocksDB + RAM (graph storage) The routing graph is read-heavy and rarely mutated; RocksDB serves the cold-load path (initial mmap from SSD) while the hot CH-augmented graph stays in RAM as flat arrays. This split keeps cold-start under 60 s on a 80 GB shard while query path stays sub-ms. Spanner (50-100 ms write latency, expensive); pure on-disk (kills query latency); LevelDB (single-writer bottleneck under preprocessing churn).
Spark (nightly CH preprocessing) Computing CH on a continent's 400 M-edge graph takes ~6 h on a 32-core box. Spark parallelises the importance ordering and shortcut generation across hundreds of nodes, cutting wall time to ~1 h and integrating with the OSM-derived feature pipeline. Single-node OSRM build (6 h, no parallelism); custom MapReduce (reinvent the wheel); Flink batch (less mature for graph algorithms).
Kafka + Flink (CCH customisation pipeline) Probe aggregation produces edge_factor updates that must be customised into the CH every 1-2 min. Flink's keyed state per edge_id, exactly-once checkpoints, and integration with Kafka tiered storage give us replayable customisation without losing updates during a customiser restart. Spark Streaming (microbatch latency); custom ZeroMQ pipeline (no exactly-once); Kinesis (1 MB/s/shard cap awkward at our scale).
Edge-based graph (turn restrictions) "No left turn" rules cannot be expressed on node-based graphs. We expand each (edge, turn) into a node, doubling graph size to ~800 M edges, but enabling correct routing through real-world intersections — a non-negotiable correctness requirement. Node-based with turn cost penalties (incorrect for hard restrictions); post-hoc filtering (route may not exist after filtering).
Atomic graph swap with epoch counter CCH customisation produces a new edge_factor table every minute; query threads must see a consistent snapshot. A single-writer / many-readers epoch counter lets readers grab a stable pointer at query start and lets the writer atomically flip to the new graph without locking — DDIA Ch. 7 single-shot transactions. Read-write locks (10s of µs added per query); double-buffered swap (2× memory at 80 GB cost); copy-on-write structures (complex for CSR).
Region sharding + overlay graph (cross-continent routes) NYC→LA naively spans both NA shards' 80 GB graphs. We precompute pairwise CH distances for ~5 K boundary nodes; cross-region queries do CH within each continent + Dijkstra on the overlay, keeping query under 10 ms even for trans-continental paths. Single global graph (320 GB+, doesn't fit); naive cross-region Dijkstra (seconds); transit-node routing (better but heavier preprocessing).
Time-dependent CH (TCH) — for departure-time queries Rush-hour ETAs need departure-time-aware weights: a 9am edge in Manhattan is 3× slower than 3am. TCH stores piecewise-linear speed profiles per edge and runs CH against the time-dependent function — the only way to honour "depart at 5 PM Friday" correctly. Static weights with rush-hour multiplier (loses intra-hour shape); per-hour CH copies (24× memory); rerouting at run-time (latency hit).
OSRM (MVP off-the-shelf) Documented as the v1 path: open-source CH-based routing engine running on a single beefy box per continent ships in a month. We move custom only after proving real workload patterns — premature optimisation on routing engines wastes 6 months easily. GraphHopper (similar, less driving-mode tuning); Valhalla (great for multimodal, slower for cars); custom from scratch (engineer-years).
NUMA-aware memory pinning An 80 GB graph spans multiple NUMA sockets; uncontrolled allocation causes 2× memory-latency penalty on cross-socket loads. Pinning shards to local NUMA nodes and binding worker threads to the same socket keeps per-query CSR scans on local memory, recovering ~30% throughput. Default OS allocator (cross-socket penalty); single-socket per node (limits throughput per box); larger boxes with single NUMA (rarely available, cost prohibitive).
