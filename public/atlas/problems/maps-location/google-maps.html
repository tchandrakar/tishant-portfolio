Design Google Maps (tile serving + routing + Places)

Very Hard maps-location search cdn

1. Problem statement

Design a planet-scale mapping platform that renders interactive vector and raster tiles for the entire Earth, computes turn-by-turn driving/walking/transit routes in under a second, indexes ~250 M points-of-interest (Places), and ingests live traffic from a billion mobile probes. Three sub-systems must coexist: tile pipeline, routing engine, and Places search. Optimise for read-mostly traffic, sub-200 ms cold-tile latency, and cost-per-tile (a single global re-render is petabytes).

2. Functional requirements

  • Pan/zoom map at 22 zoom levels, both raster (PNG/WebP) and vector (Mapbox MVT) tiles.
  • A→B routing across driving, walking, biking, transit; alternative routes; ETA with live traffic.
  • Places autocomplete (<100 ms p50), reverse-geocode lat/lng → address.
  • Street-level imagery (Street View) with 360 panoramas.
  • User contributions: edits, photos, reviews, "Your Timeline" location history.

3. Non-functional requirements

  • Latency: tile p99 < 200 ms, route p99 < 800 ms, autocomplete p99 < 150 ms.
  • Scale: 1B+ MAU, 25 B tile requests/day, 10 B route requests/day, 5 B autocomplete/day.
  • Availability: 99.99% for tile/Places, 99.95% for routing.
  • Freshness: traffic < 2 min, Places edits < 24 h, base map < 14 days.
  • Footprint: petabytes of tiles, exabytes of Street View imagery.

4. Capacity estimation

  • QPS: 25 B tiles/day ≈ 290 K rps avg, 1 M peak. Routes ≈ 115 K rps avg.
  • Tile storage: 22 zooms, > 4^22 ≈ 17 T tiles theoretical; sparse populated set ≈ 50 B tiles × 30 KB = 1.5 PB per render of vector; raster pyramid ≈ 5 PB.
  • Road graph: ~150 M nodes, ~400 M edges globally; CH-augmented graph ≈ 80 GB in RAM per region shard.
  • Probe ingest: 1 B devices × 1 ping/30 s peak = 33 M events/s; compressed at 30 B = 1 GB/s into Kafka.
  • Bandwidth: avg vector tile 18 KB → 25 B × 18 KB ≈ 450 TB/day egress (CDN absorbs >95%).

6. High-level architecture

[Mobile/Web] → [Edge CDN: 150+ POPs] ──cache miss──▶ [Tile API GW] │ ┌──────────────────────────┬──────────────────────┼──────────────────────┐ ▼ ▼ ▼ ▼ [Tile Renderer] [Routing Svc] [Places Svc] [Probe Ingest] ↑ Mapnik/Vector ↑ CH road graph ↑ ES + Bigtable ↓ Kafka │ │ │ │ [Tile Cache: Bigtable] [Graph shards: RAM] [Place index: ES] [Flink → traffic.tiles] ▲ ▲ ▲ │ │ │ │ ▼ └──────────────[Spanner: master OSM-like edits]───┴──────────[Live traffic store]

Tiles: client requests /z/x/y; CDN serves 95%+ from edge. On miss, Tile API checks Bigtable; if absent, dispatches to renderer pool that pulls geometry from Spanner-backed feature store and writes back. Routing: API hashes origin region, forwards to a CH graph shard; live-traffic edge weights merged via in-memory overlay updated every 60 s. Places: ES handles autocomplete with edge-ngrams; full details from Bigtable keyed by placeId.

7. Data model & storage

7.1 Tile metadata (Bigtable)

row_key = sha1(layer)#z#tile_xy_morton
columns: blob (mvt), etag, version, rendered_at
TTL: derived layers 14d; static base layer infinite

7.2 Road graph (custom on RocksDB + RAM)

nodes(node_id, lat, lng, country_shard)
edges(src, dst, length_m, free_flow_s, road_class, restrictions_bitmap)
ch_shortcuts(src, dst, weight, contracted_via)

7.3 Places (Elasticsearch + Bigtable)

ES doc: {place_id, name_ngrams, geohash7, category, popularity}
BT row: place_id → {photos[], hours, reviews_ref}

7.4 Why this mix?

Bigtable for blob-ish tiles + Places (write-once-read-many, cheap range scans by Morton code). Spanner for transactional map edits — global ACID matters when one editor splits a road. ES purpose-built for ngram autocomplete with geo-filtering. RocksDB+RAM for routing graph because traversals must be sub-millisecond.

8. Detailed component design

8.1 Tile pipeline (DDIA Ch. 10 batch)

Nightly Spark job reads OSM-derived feature tables, runs Mapnik-style stylesheet, emits MVTs partitioned by Morton key into Bigtable. Incremental "dirty tile" path: editors mark bbox → SQS-fan-out → only affected z=14 parents re-render, children invalidated lazily by ETag bump.

8.2 Routing engine

Contraction Hierarchies pre-compute (1-time, ~6 h per continent) shrinks bidirectional Dijkstra from 400 M-edge graph to ~3 ms p99 lookups. Live traffic injects multiplicative edge factors via shared-memory ring buffer; CH validity is preserved because shortcuts stay correct under monotonic weight increases (we re-rank only top-k candidates).

8.3 Places autocomplete

Two-stage: ES prefix lookup with geo-filter (5 km radius around device) returns top-50; ranker (LightGBM) re-scores with personalisation, click-through priors, and live business hours.

8.4 Traffic ingest (DDIA Ch. 11 stream)

Probes → Kafka (sharded by tile_id) → Flink windowed aggregator (1 min tumbling) → per-edge speed median → publishes to traffic.tiles topic consumed by routing front-ends.

9. Scaling & bottlenecks

  • Hot tiles (Times Square z=18): push-pin to every CDN POP, shielded origin, separate "hot tier" with 30 s TTL.
  • Cross-region routes (NYC → LA): overlay graph of ~5 K boundary nodes pre-computed; query stitches 2 region searches + overlay.
  • Bigtable hot rows: Morton ordering creates locality but also hotspots — salt with 4-bit prefix on writes to spread across tablets.
  • Probe storms (rush hour): client-side coalescing, server back-pressure via Kafka quotas.

  • "I'd separate the three sub-systems with hard API boundaries; tiles are the cheap, cacheable workhorse and pay for the long tail of routing."
  • "The hardest part is keeping CH valid under live traffic — I'd discuss the 'turn restrictions + monotonic weight' invariant explicitly."
  • "Places at 250 M docs is where ES starts hurting; I'd shard by country and front it with a learned-bloom prefix filter."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Bigtable (tile blobs + Places details) Morton-coded row keys keep neighbouring tiles physically adjacent on tablets, so a 9-cell viewport scan hits ~3 tablets instead of 9. At 25 B tile reads/day and 5 PB raster pyramid, we need cheap range scans on a write-once-read-many blob, which is exactly Bigtable's sweet spot. S3 (no range scans, 100 ms p99 cold); Cassandra (worse compaction at petabyte scale); HBase (Bigtable is the managed equivalent).
Spanner (master OSM-like edits) When an editor splits a road or moves a POI, we need global ACID — two simultaneous edits on the same edge in NYC and Tokyo must serialise or we corrupt the graph. TrueTime gives external consistency across continents at the cost of 50-100 ms write latency, which is fine for editor workflows. CockroachDB (similar but smaller ecosystem); MySQL with cross-region replication (no global serialisability); DynamoDB (no SQL joins for editor UI).
S2 cells (geo indexing for Places) Equal-area hierarchical cells let us cover a 5 km autocomplete radius with ~9 cells regardless of latitude. Geohash distorts area near poles by 2x and breaks at antimeridian; for 250 M Places with a global query distribution this matters for relevance fairness. Geohash (rectangle distortion); H3 (hex shape complicates radius cover math); Quadtree (no equal-area guarantee).
Contraction Hierarchies (CH) on RocksDB+RAM Plain Dijkstra on 400 M edges costs ~10 s/query — at 115 K rps avg that's 1.15 M cores. CH preprocesses (~6 h per continent) into shortcuts that cut visited nodes 1000×, hitting 1-3 ms p99 on continental routes within an 80 GB RAM footprint per shard. A* with great-circle heuristic (~200 ms — too slow at 400 K peak rps); ALT with 16 landmarks (~10× speedup, still 50 ms); plain bidirectional Dijkstra (~1 s).
Customisable CH (CCH) for live traffic Pure CH freezes weights, but live traffic shifts edge times every 60 s. CCH separates structure pre-compute from a 5 s "customisation" pass that re-applies fresh weights, letting us push traffic factors continuously without invalidating the 1000× speedup. Re-running CH from scratch (6 h, impossible per minute); real-time A* (200 ms p99 violates SLA); arc-flags (similar speedups but harder to update).
Elasticsearch (Places autocomplete) 5 B autocomplete/day requires sub-100 ms prefix matching against 250 M docs with geo-filter and category facets. Edge-ngrams + geo_distance + custom scoring is exactly the workload ES is tuned for; we accept the painful 250 M-doc reindex cost (sharded by country) as the price of feature richness. Trie-based custom service (no facets); Solr (ecosystem smaller); FoundationDB+custom ngram (rebuild it ourselves — not worth two engineers).
Kafka + Flink (probe ingest pipeline) 33 M probes/s = 1 GB/s sustained, with bursty rush-hour spikes. Kafka sharded by tile_id provides locality so per-edge aggregators stay on one node; Flink's 1-min tumbling windows + checkpoints give exactly-once aggregation, hitting the 2-min freshness SLA. Kinesis (1 MB/s/shard limit forces 1000s of shards); Spark Streaming (microbatches add latency); Pulsar (ops overhead, smaller talent pool).
Mapnik vector tiles (MVT) MVT averages 18 KB vs ~80 KB for raster PNG at z=14, cutting CDN egress from ~2 PB/day to 450 TB/day at our 25 B request volume. Client-side restyle also lets us serve dark mode and language overlays without re-rendering 50 B tiles server-side. Raster PNG (4× bandwidth, server restyle); WebP raster (smaller but still server-side restyle); 3D tiles (overkill for 2D pan/zoom).
CDN (150+ edge POPs) 95%+ of tile requests are repeats at popular zooms — Manhattan z=14 gets millions of hits/hour. Pushing the long-tail-flattened 50 GB hot working set to every POP brings p99 from 200 ms (origin) to 30 ms (edge) and absorbs >95% of 290 K rps off the origin tier. Origin-only with bigger Bigtable cluster (10× cost, worse latency); single-region CDN (cross-Atlantic RTT kills mobile UX).
Spark (nightly tile re-render) OSM-derived feature tables are TB-scale; Mapnik render across 50 B tiles is embarrassingly parallel. A nightly Spark job partitions by Morton parent and writes back to Bigtable in 6-8 h, using DDIA Ch. 10 batch semantics — idempotent, replayable, no streaming complexity for static base layers. Flink batch (less mature for TB shuffles); Beam on Dataflow (vendor lock-in); custom MapReduce (reinvent the wheel).
LightGBM (Places ranker) Two-stage retrieval-then-rank: ES returns top-50 candidates, LightGBM re-scores with 30+ features (distance, click-priors, hours, personalisation) in <5 ms per query. Trees handle non-linear feature interactions cheaply; we don't need a deep model when the candidate set is already tight. Deep neural ranker (50 ms inference, 10× cost); pure ES BM25 (no personalisation); XGBoost (similar but LightGBM trains 2× faster on our 1B click logs).
Morton (Z-order) curve for tile keys Bigtable row-key sort order determines locality. Morton interleaves x/y bits so a 2D viewport (a square in tile space) maps to a small number of 1D row ranges — typically 2-4 — making "fetch the 16 tiles around me" a couple of range scans instead of 16 random reads. Hilbert curve (~10% better locality but 3× more code); plain row-major (column scan is 1D, terrible for viewport); raw geohash strings (no native int support).
