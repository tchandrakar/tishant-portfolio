Design Autocomplete / Typeahead (sub-100 ms, top-K trie)

Hard search storage-cache

1. Problem statement

Build a typeahead service that, on every keystroke, returns the top-K (typically K=10) most likely query completions in under 100 ms end-to-end. Suggestions come from historical query logs ranked by frequency, recency, and personalization signals. Think Google search box, YouTube search, or e-commerce suggest.

2. Functional requirements

  • Return top-10 completions for a prefix in real time as the user types.
  • Support multilingual prefixes, accents, case-insensitive matching.
  • Continuously learn from new queries (hourly or faster).
  • Personalization: boost completions the user has typed before.
  • Filter: profanity, spam, trademark/legal blocklist.

3. Non-functional requirements

  • Latency: p50 < 30 ms, p99 < 100 ms server-side; perceived < 150 ms over the wire.
  • Scale: 5 B searches/day → ~10× keystrokes ≈ 50 B suggest calls/day ≈ 580 K QPS avg, 2 M QPS peak.
  • Freshness: new trending query indexable in < 1 hour.
  • Availability: 99.99%.

4. Capacity estimation

  • QPS: 2 M peak suggest QPS. Each box debounces ~50 ms client-side, so amplification is bounded.
  • Vocabulary: ~500 M unique queries with non-trivial frequency; trie nodes ≈ 5 B chars × 8 B avg = ~40 GB raw, ~10 GB after compression — fits per-node memory.
  • Storage: query log 5 B/day × 100 B = 500 GB/day → S3/HDFS for batch.
  • Cache: top-K per prefix for the top 10 M prefixes ≈ 10 M × 10 × 32 B = 3.2 GB → trivially fits in Redis.

5. API design

GET /suggest?q=harr&hl=en&userId=...&k=10
   → { suggestions: [{text, score, source}], queryId }
POST /feedback {queryId, accepted: "harry potter"}   // implicit signal

6. High-level architecture

[Client]--debounced 50ms--> [Edge LB] -> [Suggest Service] -> [Redis hot-prefix cache] | (miss) v [In-memory Trie shards] ^ | atomic swap (every 1h) [Query Log] -> [Kafka] -> [Spark/Flink Job] -> [Aggregator: count+rank] -> [Trie Builder] -> [GCS/S3] | downloaded by suggest pods

7. Data model & storage

7.1 Trie node (compressed)

node {
  children: map<char, nodeId>     // or radix-compressed edge labels
  topK: [(query_id, score)]       // precomputed top-10 for this prefix
}
strings table: query_id -> normalized_text, original_text, lang

7.2 Why precompute topK at every node?

Walking the subtree at query time is O(prefix_subtree). Precomputing top-K per node makes lookup O(prefix_length) — typically < 20 chars — independent of how popular the prefix is. The cost is build-time RAM and a slightly larger artifact (~3× over a plain trie).

7.3 Sharding

Partition by first 1–2 characters of the prefix. Each suggest pod loads only its shards. Hot shards (e.g., "a*", "the*") get more replicas.

8. Detailed component design

8.1 Offline build pipeline

Hourly Spark job: read last 24 h of query logs from Kafka → normalize (lowercase, NFKC, strip diacritics) → aggregate (count, last_seen) → score = log(count) · decay(now − last_seen) · quality_filter → top-K rollup at every prefix using a bottom-up merge → serialize to a compact binary trie (FST or DAWG). Artifact pushed to GCS, suggest pods pull and atomic-swap (DDIA Ch. 10 — batch processing produces a derived dataset consumed by online services).

8.2 Online serving

Suggest pod: load mmap'd trie at startup → on request, normalize prefix → look up node → return its top-K → personalization layer reorders by user's recent queries (small per-user Bloom filter / Redis hash) → blocklist filter.

8.3 Hot-prefix cache

Redis L1 cache keyed on (prefix, lang, userBucket) with 60 s TTL absorbs the long tail of duplicate prefixes — typically 80 %+ hit rate.

8.4 Real-time trending boost

Side stream: a Flink job computes a rolling 5-minute count-min sketch of queries; the top-1 K trending queries are pushed to suggest pods as a small in-memory overlay merged at query time.

9. Scaling & bottlenecks

  • Per-keystroke amplification: debounce client-side to 50 ms and cancel in-flight requests on new keystrokes.
  • Memory: use FST (Lucene-style) or DAWG to dedup suffixes — typically 5–10× smaller than naive trie.
  • Cold start: stagger pod restarts; warm-load mmap pages by walking common prefixes once.
  • Tail latency: avoid GC by allocating the trie off-heap or in a flat byte array.
  • Personalization cost: cap user history to 100 recent queries; merge with global top-K via a heap.

12. Talking points for the interviewer

  • "The unlock is precomputing top-K at every node — it converts an open-ended subtree problem into an O(prefix) lookup."
  • "I keep the online path read-only and immutable; everything dynamic lives in two small overlays (trending + personalization)."
  • "The most under-appreciated win is client-side debouncing; without it the back end pays 10× for the same UX."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
FST / DAWG (compact trie)500 M unique queries × ~16 chars would be ~40 GB as a naive trie; FST suffix-dedup compresses it to ~10 GB so the entire artifact fits in a single pod's RAM, giving O(prefix) lookups in <1 µs against mmap'd bytes.HashMap<prefix, topK> (3–5× memory, no prefix iteration); ternary search tree (worse cache locality at this size).
Radix-compressed trie nodesAverage suggested query is ~20 chars but most internal trie nodes have one child; radix compression collapses single-child chains, cutting node count ~5× and bringing lookup hops from ~20 to ~4 — critical for the <30 ms p50 budget.Plain character trie (5× more pointers, cache miss on every step).
Precomputed top-K per nodeWalking the subtree under "har" at query time would touch millions of nodes; precomputing top-10 at every node converts it to a constant-time read at the cost of ~3× memory — the only way to hit p99 <100 ms at 2 M QPS.On-the-fly subtree scan (10–100× slower for popular prefixes); approximate top-K via sketches (recall regression).
Apache Kafka (query log transport)5 B searches/day = ~58 K msg/s sustained, bursty during news events; Kafka's durable log lets the trie builder replay 24 h windows for retraining and decouples the suggest pods from the offline pipeline.Kinesis (vendor lock-in, lower replay window); direct DB writes (no replay, no backpressure).
Apache Spark (hourly trie builder)Aggregating 5 B queries/day into prefix→top-K rollups requires sorting + reducing 500 GB; Spark's in-memory shuffle finishes the job in ~15 min on a 50-node cluster, well under the 1 h freshness SLA.MapReduce (re-shuffles, 3× wall-time); Flink batch (less mature for this batch shape).
Apache Flink (trending overlay)Trending queries must surface in <5 min — far below the 1 h batch cycle; Flink's event-time windows + count-min sketches over a 5-min rolling window catch news spikes and push a small overlay (top 1 K) to suggest pods.Spark Streaming (mini-batch latency); Storm (no exact-once, weaker state).
Redis (hot-prefix cache)10 M hot prefixes × top-10 × 32 B = 3.2 GB, trivially fits one Redis node; 60 s TTL absorbs the long tail of duplicate prefixes at >80 % hit rate, cutting load on the trie pods by 5×.In-process Caffeine (no cross-pod sharing); Memcached (no TTL granularity needed but no replication).
GCS / S3 (artifact distribution)Hourly trie artifact is ~10 GB; object storage gives 99.999999999% durability, multi-region replication, and HTTP range reads so suggest pods can mmap-stream the file at startup without a custom pipeline.HDFS (extra cluster); Bittorrent-style P2P (operational complexity for marginal speed-up).
Count-min sketch (trending counts)Tracking exact counts for billions of distinct queries in a 5-min window would cost ~50 GB; CMS at ε=0.001 keeps it to ~50 MB per Flink operator with a controlled overcount bias that the top-K extractor tolerates.HashMap (50 GB+ state); HyperLogLog (gives cardinality, not counts).
Bloom filter (per-user history)Personalization needs "has user typed this prefix before" for 100 M users at <1 ms; a 1 KB Bloom per user (1 % FP) costs 100 GB total in Redis and avoids storing exact history per pod.Exact set per user (10× RAM); Redis ZSET per user (slower reads at 2 M QPS).
off-heap / flat byte-array storageA 10 GB on-heap trie would generate >500 ms G1 pauses, blowing the latency budget; a flat byte-array layout gives zero GC pressure and lets the JIT vectorize prefix walks.On-heap Java objects (GC pauses); off-heap via Unsafe (less portable, harder to debug).
HTTP/2 + client-side debouncing (50 ms)Without debouncing, "harry potter" sends 12 keystroke requests; debouncing to 50 ms cuts that to ~3 and HTTP/2 multiplexing reuses one connection so we don't pay TLS handshake per keystroke — 10× backend savings.HTTP/1.1 (head-of-line blocking); WebSocket (heavier connection state per pod).
