Design Search Ranking Service (BM25 → learning-to-rank)

Hard search recommendation

1. Problem statement

Design the ranking layer of a large search engine: given a query and a candidate set of ~10 K documents from the inverted index, return the best top-10 ordered by relevance under a strict latency budget. Ranking must combine textual signals (BM25, proximity), document signals (PageRank, freshness, click models), and personalization, evolving from a hand-tuned linear model to a learning-to-rank system trained on click logs.

2. Functional requirements

  • Score-and-rank candidate documents per query in real time.
  • Support multiple ranking models per traffic slice (A/B, holdouts).
  • Online feature lookup: query, document, user, context.
  • Offline pipeline to train, evaluate, and ship new ranker models.
  • Explainability: per-result feature contributions for debugging.

3. Non-functional requirements

  • Latency: ranking budget < 50 ms p99 over 10 K candidates.
  • Scale: 100 K QPS peak; ~5× ranker invocations per query (multi-stage).
  • Quality: measurable lift in NDCG@10 vs the previous model on offline evals.
  • Safety: bounded blast radius per model rollout; instant rollback.

4. Capacity estimation

  • Per query: 10 K candidates × ~200 features × 4 B = ~8 MB feature matrix; goes through GBDT (~500 trees, depth 8) → ~4 M tree evals — fits in < 30 ms on a single core with vectorized inference (e.g., LightGBM, XGBoost).
  • Feature store reads: 10 K doc-feature lookups per query × 100 K QPS = 1 B reads/sec — must be local (mmap) or batched RPC; remote-only would cost ~50 K Redis nodes.
  • Training: 10 B labeled (query, doc, label) tuples/month → ~10 TB; nightly retrain on a 100-node Spark cluster.

6. High-level architecture

[Query] -> [Retrieval (BM25/ANN)] -> 10K candidates -> [L1 Ranker: linear/BM25+static] --top 1K--> [Feature Joiner] <- [Doc Feature Store (mmap)] <- [User Feature Store (Redis)] <- [Query Feature Cache] -> [L2 Ranker: GBDT] --top 200--> [L3 Ranker: DNN/Transformer cross-encoder] --top 10--> [Blender] [Click Logs] -> Kafka -> Flink (sessionize, label) -> S3 -> Spark Trainer -> Model Registry -> Servers

7. Data model & storage

7.1 Online feature stores

doc_features (per shard, mmap'd FlatBuffers):
  doc_id -> { pagerank f32, freshness_days u16, ctr_prior f32, lang u8, ... }
user_features (Redis):
  user_id -> { topic_vec f32[64], recent_queries[8], country }
query_features (LRU cache):
  query_hash -> { intent: navigational|informational|transactional, lang, expansion_terms[] }

7.2 Training data lake (S3 + Parquet)

impressions/dt=2026-05-05/part-*.parquet
  cols: query, doc_id, position, clicked, dwell_ms, session_id, features (struct)

7.3 Why this split?

Doc features are huge but read-mostly → ship as immutable per-shard artifacts (DDIA Ch. 10). User features are small and write-hot → Redis. Query features are derived per request → small LRU. This minimizes online I/O.

8. Detailed component design

8.1 Multi-stage ranking

Each stage trades cost for quality on a smaller set. L1 is cheap (linear over 20 features), L2 is medium (GBDT over 200 features on 1 K docs), L3 is expensive (cross-encoder transformer over 200 docs). This pyramid keeps p99 in budget while letting the heaviest model see only the most promising candidates.

8.2 Feature engineering

Three families: (a) matching — BM25, TF-IDF, proximity, exact-match-in-title; (b) document — PageRank, age, click prior, language; (c) contextual — user topic affinity, location, device, time-of-day. Pairwise / listwise labels derived from clicks via position-aware models (e.g., PBM) to debias.

8.3 Training pipeline

Daily Spark job: join impressions with feature snapshots, generate (query, doc⁺, doc⁻) pairs, train LambdaMART on NDCG. Promoted via shadow eval (replay 1 % live traffic offline, compare on NDCG@10 + business KPIs) → 1 % canary → ramp.

8.4 Serving

Models are loaded as flat tree arrays (e.g., LightGBM predict with SIMD). One ranker process per shard; warm-up on deploy. Hot reload via shared-memory swap so latency doesn't spike during rollout.

9. Scaling & bottlenecks

  • Feature staleness: CTR priors must update hourly, not daily — split features into "slow" (daily artifact) and "fast" (Kafka stream → Redis).
  • Position bias: training on raw clicks rewards already-ranked-high docs; use IPS or PBM weighting.
  • Model size vs latency: cap GBDT trees at 1 K, depth 8; for DNN reranker, distill from a teacher into a 6-layer student.
  • Cold queries: long-tail queries with no click history — fall back to BM25 + freshness; carve out as a holdout to avoid biasing the trainer.
  • Cross-shard joins: avoid; replicate user features to every region instead.

12. Talking points for the interviewer

  • "I'd be explicit that ranking and retrieval are different problems with different SLOs and different evaluation metrics."
  • "Position bias is the single biggest gotcha — without IPS your model just memorizes the existing ranker."
  • "Shadow + canary + automatic rollback on guardrail-metric regression is non-negotiable for ranker rollouts."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
LightGBM / XGBoost (GBDT for L2)500 trees × depth 8 over 200 features score 1 K candidates in <30 ms on one core via SIMD-friendly flat tree arrays; on tabular ranking features GBDT consistently beats DNNs while being 10–100× cheaper, fitting the 50 ms p99 budget at 100 K QPS.RankLib MART (pure-Java, slower); CatBoost (similar quality, smaller ecosystem); pure DNN (10× cost, no tabular-feature edge).
LambdaMART (training objective)NDCG@10 is the business metric, and LambdaMART directly optimizes it via lambda gradients on pairwise swaps — listwise alternatives like ListNet under-perform on the 10 B-tuple/month training set we have.Pointwise logistic (ignores ranking structure); ListNet (higher variance, worse NDCG in our offline replays).
Cross-encoder Transformer (L3 reranker)For the top 200 candidates, a 6-layer cross-encoder on (query, doc) pairs captures semantic relevance that GBDT misses (synonyms, paraphrase); 200 forward passes batched on GPU finish in ~20 ms, only feasible because the cascade narrowed the set first.Bi-encoder (retrieval-grade, not rerank-grade); full BERT-base (3× slower without distillation).
Knowledge distillation (teacher → student)The deployed L3 is a 6-layer student distilled from a 24-layer teacher; gives ~95 % of teacher NDCG at 5× lower latency, the only realistic way to put a transformer in the 50 ms budget.Quantization alone (smaller win, accuracy loss); pruning (less stable on cross-encoders).
FlatBuffers (mmap'd doc feature store)10 K doc-feature lookups/query × 100 K QPS = 1 B reads/s; FlatBuffers gives zero-parse access to the per-shard feature blob via mmap so a feature read costs ~50 ns instead of a Redis RPC at ~500 µs — the difference between fits-on-disk and 50 K-node Redis fleet.Protobuf (parse cost); Cap'n Proto (similar but smaller ecosystem); Redis (1000× slower per read).
Redis (user feature store)User features are small (~1 KB/user), write-hot (CTR updates, recent queries), and need cross-region replication for personalization; Redis Cluster delivers <1 ms p99 GETs at 100 K QPS with simple TTL semantics.DynamoDB (5–10 ms p99); Cassandra (heavier writes); local KV (no cross-region).
Apache Spark (training pipeline)Joining 10 B impressions × feature snapshots (~10 TB/month) and generating LambdaMART-ready (q, d⁺, d⁻) pairs needs a distributed shuffle; a 100-node Spark cluster finishes the nightly job in ~3 h with restartable stages.Hadoop MR (3× wall-time); Flink batch (smaller ecosystem for ML); single-node (impossible at TB scale).
Apache Kafka (click event log)Click/dwell events arrive at ~50 K msg/s with bursts; Kafka's durable log lets us replay weeks of impressions for retraining when label definitions change, and a 7-day retention guards against bad pipeline pushes.Kinesis (24 h default retention); pub/sub without replay (debugging nightmare).
Apache Flink (CTR prior, fast features)CTR priors must update hourly to capture trends; Flink computes per-(query,doc) click-through with exactly-once semantics over the Kafka stream and writes to Redis, closing the freshness gap that nightly batch leaves.Spark Streaming (latency floor at minutes); custom consumers (no exactly-once).
S3 + Parquet (impression data lake)10 TB/month of (query, doc, label, features) needs columnar compression for cheap shuffles; Parquet with Snappy gives 5–10× compression and predicate pushdown so trainers read only the needed columns.Avro (row-oriented, slower scans); ORC (similar but less Spark-native).
IPS / PBM (debiasing)Raw clicks reward already-high ranks; without inverse-propensity scoring or position-based modeling, the trained ranker just memorizes the existing ranker — model lift on NDCG goes from +0.5 % (raw) to +3 % (debiased) in our offline tests.Train on raw clicks (positive feedback loop); rank-randomization (real-user impact).
Shadow + canary rollout (model registry)Replaying 1 % traffic offline plus 1 % canary with auto-rollback on guardrail regression bounds blast radius; given a bad LTR model can tank revenue 5–10 %, this is non-negotiable.Direct ramp (high-risk); shadow only (misses live-feedback bugs).
Shared-memory hot-reload100 K QPS can't tolerate a cold cache; loading the new model into shared memory and atomically swapping pointers avoids the latency spike from re-reading 200 MB tree arrays during deploy.Process restart (latency cliff during warm-up); blue-green at LB layer (2× resource cost).
