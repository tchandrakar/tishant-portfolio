Design Image / Visual Search (embeddings + ANN: FAISS / HNSW)

Hard search recommendation

1. Problem statement

Build a visual search service: given a query image (uploaded photo or a doc-id), return the visually most similar images from a catalog of ~1 B items in < 200 ms. Use cases: reverse image search, "shop the look", duplicate detection. The core is approximate nearest neighbor (ANN) search over high-dimensional embeddings produced by a vision model.

2. Functional requirements

  • Index 1 B catalog images with metadata (source, owner, category).
  • Search by image upload, image URL, or existing doc-id.
  • Filter results by category, marketplace, language, safety flags.
  • Continuous ingestion: 10 M new images/day indexed within 1 hour.
  • Visual + text hybrid query ("red running shoes" + image).

3. Non-functional requirements

  • Latency: p99 < 200 ms including embedding inference and ANN search.
  • Scale: 1 B vectors of 512 dims × 4 B = 2 TB raw; with PQ → ~64 GB; 10 K QPS peak.
  • Recall: ≥ 95 % recall@10 vs exact nearest neighbor.
  • Availability: 99.95%; safe-content filters always on.

4. Capacity estimation

  • Embedding inference: 1 B × 50 ms on GPU = 50 M GPU-sec ≈ 580 GPU-days per full re-embed; in practice incremental, so ~500 GPUs sustained.
  • ANN index: 1 B × 64 B (PQ-compressed 512-dim) = 64 GB → fits in RAM of 4–8 nodes; with replication = 16–32 nodes.
  • Online query: 1 GPU inference (~30 ms) + ANN lookup (~10 ms) + rerank (~30 ms) ≈ 70 ms internal.
  • Storage: original images on S3, ~500 KB avg → 500 TB.

5. API design

POST /v1/visual-search
   multipart: image OR { docId | imageUrl }, filters, k=20
   → { results: [{docId, score, thumbnail, metadata}] }
POST /v1/index/image  (internal)  {docId, imageBytes, metadata}

6. High-level architecture

[Client] | v [API GW] -> [Query Embedder (GPU)] -> [Vector Router] | v [ANN Shards (FAISS IVF-PQ / HNSW)] x N | v [Reranker (full-precision dot)] | v [Filter / Safety / Hydrator (S3 thumbs)] [Image Ingest] -> Kafka -> [GPU Embedder Workers] -> [Vector Writer] | v [Online ANN Shard (mutable)] | nightly merge into IVF-PQ

7. Data model & storage

7.1 Vector record

doc_id (u64) | embedding (f32[512])  -- mmap'd flat file per shard
codes        : pq_codes (u8[64])      -- product-quantized for ANN
metadata DB  : Cassandra row keyed by doc_id (category, owner, safety, ts)

7.2 ANN structures

FAISS IVF-PQ for billion-scale: vectors clustered into K=65 K coarse cells (IVF), each vector stored as a 64-byte product-quantized code. Query visits nprobe nearest cells (e.g., 32). Memory ~64 B/vec; recall tunable via nprobe. HNSW for < 100 M vectors: hierarchical small-world graph, < 5 ms query, but ~200 B/vec — too expensive past a few hundred million.

7.3 Sharding

Random doc-id-hash partitioning across N shards; each shard runs its own IVF-PQ. Coordinator scatter-gathers, merges top-K. (Avoid clustering by category — it concentrates load.)

8. Detailed component design

8.1 Embedding model

Vision tower (e.g., a ViT or CNN) produces a 512-dim L2-normalized embedding. Trained with contrastive loss on (anchor, positive, hard-negative) image triplets, or jointly with a text tower (CLIP-style) for hybrid queries. Distilled student model used at serving time for latency.

8.2 Indexing pipeline

Ingest events → embed in batches on GPU → append to a mutable HNSW shard kept in RAM. Nightly Spark job pulls all new vectors, retrains coarse centroids if drift exceeds a threshold, rebuilds the IVF-PQ artifact, and hot-swaps. (DDIA Ch. 10 — bulk derived dataset; Ch. 11 — stream feeds the mutable tier.)

8.3 Query path

Embed query image (or look up cached vector for known doc-id) → scatter to all ANN shards → each returns top-K candidate ids with PQ-approximate distances → coordinator merges → reranker pulls full-precision vectors for top 200 and recomputes exact distance → filter (category, safety, ownership) → hydrate thumbnails from S3 via CDN.

8.4 Hybrid text+image queries

Compute text embedding from a shared CLIP-style space, weighted-sum with image embedding (e.g., 0.6·image + 0.4·text), then ANN as usual. Linear in vector ops, no extra index needed.

9. Scaling & bottlenecks

  • Embedding GPU cost: dominant cost — distill model, batch aggressively (32–64), serve on dedicated GPU pool with admission control.
  • PQ recall cliff: at very high QPS, drop nprobe dynamically; recover lost recall via the full-precision rerank stage.
  • Drift: as the catalog grows, IVF centroids drift; retrain weekly or on KL-divergence trigger.
  • Hot updates: mutable HNSW tier handles new vectors; queries hit both tiers and merge.
  • Safety: NSFW classifier runs at index time; safety flag enforced as a hard post-filter.

12. Talking points for the interviewer

  • "At a billion vectors the only realistic option is IVF-PQ; HNSW is a second-tier trick for the freshness window."
  • "Always rerank with full precision after PQ — it recovers most of the recall lost to compression."
  • "Embedding cost typically dwarfs ANN cost; spend optimization budget on the model, not the index."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
FAISS IVF-PQ (billion-scale ANN)1 B vectors × 512 dims raw = 2 TB; IVF-PQ with 64-byte codes compresses to 64 GB so the entire index fits across 4–8 RAM-backed nodes, and nprobe=32 keeps recall at >90 % with <10 ms search per shard.HNSW (~200 B/vec × 1 B = 200 GB → 5–10× more RAM); Annoy (slower at billion scale, weaker tuning knobs); ScaNN (similar quality, smaller community when this design was made).
HNSW (mutable fresh tier)10 M new images/day must be searchable in <1 h; HNSW supports incremental insertion and gives <5 ms queries with very high recall on the <100 M-vector mutable tier, before nightly merge into IVF-PQ.Mutable IVF-PQ (centroid drift on every insert); brute-force on fresh tier (O(N) per query, blows latency).
Product Quantization (PQ codes)Reduces 2 KB float32 vectors to 64 B codes (32× compression) at ~5 % recall loss, recovered by full-precision rerank on top 200; without PQ the 64 GB→2 TB blow-up forces SSD-backed indexes, killing the <200 ms p99 latency.Scalar quantization (only 4× compression); OPQ (marginal improvement, more complex training); no compression (RAM cost prohibitive).
CLIP-style dual encoder (vision+text)Hybrid text+image queries need a shared embedding space; a CLIP-style trained tower lets us weighted-sum 0.6·image + 0.4·text vectors and ANN as usual, without a separate text index.Separate vision + text indexes with late fusion (2× index cost, harder ranking); single modality only (loses hybrid use case).
Distilled student ViT50 ms full-precision teacher is 5× too slow on the GPU at 10 K QPS; the distilled 6-layer student runs at ~30 ms with ~2 % recall regression, cutting GPU fleet from ~2500 to ~500.Quantization-only (smaller win); pruning (less stable on attention).
Apache Kafka (image ingest)10 M images/day with bursty uploads buffer through Kafka; the durable log decouples ingest spikes from GPU embedder workers, supports replay for re-embedding when the model changes, and provides backpressure when the GPU pool saturates.SQS (no replay); direct S3-event triggers (no buffering, GPU thrash).
GPU pool (embedder workers)Embedding inference is the dominant cost (50 M GPU-sec for full re-embed); a dedicated pool with batch=32–64 and admission control keeps GPU utilization >70 % — CPU inference would need 100× more nodes.CPU inference (10–100× cost); shared inference cluster (head-of-line blocking from other workloads).
Apache Spark (nightly IVF-PQ rebuild)Retraining 65 K coarse centroids over 1 B vectors needs distributed k-means; Spark MLlib finishes in ~4 h on a 100-node cluster and produces a single immutable artifact for atomic swap.FAISS native trainer (single-node, OOM at 1 B); custom MPI (no fault tolerance).
S3 + CDN (image storage / hydration)500 TB of original images at 500 KB/each needs cheap, durable storage; S3 + CloudFront caches thumbnails at edge with <50 ms hydration, letting the result page render fast even when the result list spans cold images.HDFS (no edge caching); per-region NAS (operational overhead, no global CDN).
Cassandra (image metadata)1 B doc_id → metadata rows with 200 K writes/s on ingest; Cassandra's wide-row model and tunable consistency give <5 ms p99 reads keyed by doc_id with multi-region replication for safety flag enforcement.Postgres (sharding pain at 1 B rows); DynamoDB (vendor lock-in, higher cost at this scale).
FlatBuffers (vector + metadata records)Per-shard mmap'd flat files of doc_id + embedding allow zero-copy reads at >1 GB/s for the rerank stage where we pull full-precision vectors for top 200 — Protobuf parse cost would dominate the 30 ms rerank budget.Protobuf (parse cost); raw struct arrays (no schema evolution).
NSFW classifier (index-time)Safety must be hard-enforced; running a small NSFW CNN at index time tags each image once and stores a flag in metadata, letting the query path apply a cheap post-filter rather than re-running the classifier per query.Query-time NSFW (10× cost at QPS); manual moderation (won't scale to 10 M/day).
L2-normalized embeddings + dot-productNormalizing to unit length lets dot product = cosine similarity, which FAISS implements with the fastest SIMD kernel; saves ~20 % of distance-computation cost vs L2 distance and matches what the contrastive loss optimized.Raw L2 (slower kernels); learned distance metric (training complexity).
