Design Spell Checker / Did-You-Mean (edit distance, ngrams)

Hard search

1. Problem statement

Build the "Did you mean: ___?" service that fronts a large search engine: given a possibly misspelled query (single word or full phrase), return the most likely intended query within ~50 ms. The system must learn from the query log itself — most "correct" spellings are simply the more popular variants — and handle typos, transpositions, missing/added characters, and phonetic confusions across multiple languages.

2. Functional requirements

  • Single-word correction: "recieve" → "receive".
  • Multi-word context-aware correction: "new york pissa" → "new york pizza".
  • Suggest only when confidence exceeds a threshold; otherwise stay silent.
  • Continuously update from the query log (new product names, slang).
  • Multilingual; respect language hint or auto-detect.

3. Non-functional requirements

  • Latency: p99 < 50 ms server-side.
  • Scale: piggybacks on search QPS — 100 K QPS peak; ~10 % of queries trigger a suggestion.
  • Quality: precision > 90 % at the chosen confidence threshold; never suggest something offensive.
  • Freshness: new popular terms learnable within ~1 day.

4. Capacity estimation

  • Vocabulary: 10 M unique terms with frequency > 5 in last 90 days; ~5 chars avg → ~50 MB raw, ~15 MB FST-compressed.
  • Bigram model: 200 M (w₁, w₂) pairs × 24 B = 4.8 GB → fits in RAM per node.
  • Candidates per misspelled token: ~50 within edit distance 2; ~200 with phonetic neighbors.
  • Storage of query logs for training: 5 B queries/day × 100 B = 500 GB/day → 30-day window = 15 TB on S3.

5. API design

POST /spell/correct
{ query: "new york pissa", lang: "en", k: 3 }
   → { suggestions: [{text:"new york pizza", confidence:0.93}, ...],
       trigger: true }
POST /spell/feedback {originalQuery, suggestionShown, accepted}

6. High-level architecture

[Query] -> [Tokenizer/Normalizer] -> [Candidate Generator] | v [Levenshtein Automaton over Term FST] [Phonetic / SymSpell deletes index] | v [Noisy Channel Scorer] P(c) · P(q|c) uses [Term LM] + [Bigram LM] + [Char-LM] | v [Threshold & Filter] -> Suggestion [Query Log] -> Spark daily -> term/bigram counts -> FST + KenLM -> push to servers

7. Data model & storage

7.1 Term dictionary (Lucene FST)

FST: term -> (count, frequency_rank)
   - O(L) lookup
   - supports Levenshtein automaton intersection for k-edit candidates

7.2 SymSpell delete index (alternative)

map: deletes(term, k) -> [original_terms]
   e.g., delete("receive",1) ⊇ ["eceive","rceive","reeive","recive","receie","receiv"]
   query lookup: enumerate deletes(q,k), join → fast O(1) candidate gen

7.3 Bigram language model

map: (w_{i-1}, w_i) -> log P(w_i | w_{i-1})   (Kneser-Ney smoothed)

8. Detailed component design

8.1 Candidate generation

Two strategies, used together for recall: (a) Levenshtein automaton intersected with the term FST yields all dictionary terms within edit distance ≤ 2 in O(|q|·|states|); (b) SymSpell precomputes delete-only neighborhoods so generation becomes a hash lookup. Phonetic codes (Double Metaphone) catch "fone" → "phone" that pure edit distance misses.

8.2 Noisy channel scoring

Pick candidate c maximizing P(c|q) ∝ P(c) · P(q|c) where P(c) is a language model probability (unigram for single-word, bigram/trigram for phrases) and P(q|c) is an error model — typically learned from (typed, accepted) pairs in the click log; falls back to weighted edit distance with confusion costs (e.g., adjacent-key substitutions cost less).

8.3 Multi-word correction

Tokenize → independently generate candidates per token → Viterbi decode over the bigram LM to pick the joint best path. Allows context to fix "piza" → "pizza" in "new york piza" but leave "piza" alone in a Spanish context.

8.4 Training

Daily Spark job aggregates query frequencies, builds the FST and KenLM bigrams, mines (typo → click → reformulation) pairs from sessions for the error model. Outputs are pushed as immutable artifacts and atomically swapped (DDIA Ch. 10).

8.5 Trigger logic

Show suggestion only when P(c)/P(q) > τ AND original query has low result quality (few results / low click-through). Avoids the embarrassing case of "correcting" a correctly-spelled rare brand name.

9. Scaling & bottlenecks

  • Long words: edit-distance candidate space explodes; cap at 2 edits per token, 3 only for tokens > 8 chars.
  • Brand and proper nouns: mine from query log so popular brands self-correct upward and don't get "fixed".
  • Multilingual: separate FSTs and LMs per language; route by detected language to avoid cross-lingual nonsense.
  • Latency budget: generate candidates and score in parallel across tokens; cap at top 50 candidates per token before Viterbi.
  • Safety: never suggest profane or trademark-protected terms — apply a blocklist after scoring.

12. Talking points for the interviewer

  • "The most important model isn't the spell checker — it's the trigger. Showing a wrong suggestion is far worse than showing none."
  • "Almost all signal comes free from the query log: popular = correct. The error model is a refinement, not the foundation."
  • "For multi-word, Viterbi over a bigram LM is the cheapest way to get context-aware correction without a transformer."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Lucene FST (term dictionary)10 M terms × 5 chars compresses from ~50 MB raw to ~15 MB FST, fitting in L3-cache range; the FST natively supports Levenshtein automaton intersection so we get all k≤2 candidates in O(|q|·|states|) without scanning the dictionary.HashMap<term, count> (3× memory, no fuzzy match); Trie (1.5× memory, no automaton intersection).
Levenshtein automatonExact k≤2 candidate generation against the FST yields ~50 candidates per token in <1 ms with provably correct edit distance; pure brute-force enumeration of 26^5 deletes/inserts would be intractable for the 50 ms p99 budget.Brute-force edit enumeration (exponential); BK-tree (O(log) average but worse worst case).
SymSpell delete index (parallel path)For ultra-low-latency single-word correction, SymSpell precomputes delete neighborhoods so candidate generation is an O(1) hash lookup; we run it in parallel to the Levenshtein path and union for recall, paying 5–10× memory for ~3× lower median latency.Levenshtein-only (slower median, fine tail); neural seq2seq (50× cost, hallucinations).
Double Metaphone (phonetic codes)"fone"→"phone" has edit distance 2 but is a single phonetic substitution; Double Metaphone catches ~15 % of corrections that pure edit distance misses, especially for proper nouns and brand names where users spell phonetically.Soundex (English-only, weaker); custom phonetic table (maintenance burden).
KenLM (bigram language model)200 M bigrams × 24 B = 4.8 GB fits in RAM; KenLM's Kneser-Ney smoothing and trie-based mmap give <5 µs P(w₂|w₁) lookups, which is what makes Viterbi over a 5-token query finish in ~2 ms.SRILM (slower, larger memory); neural LM (10× cost, GPU dependency for the <50 ms target).
Viterbi decoder (multi-word path)For "new york piza", per-token candidates × LM scoring needs the Viterbi DP to find the joint best path in O(T·K²) with K=50 — finishes in <3 ms and gives context-aware correction without a transformer.Beam search (lossy, similar cost); brute-force joint enum (exponential in tokens).
Apache Spark (daily build pipeline)Aggregating 5 B queries/day × 30 days = 15 TB to compute term/bigram counts, mine reformulation pairs, and rebuild FST + KenLM artifacts; Spark's shuffle handles the joins and finishes nightly in ~2 h on a 50-node cluster.MapReduce (3× wall-time); Flink batch (less mature ML support).
Apache Kafka + sessionization (Flink)The error model trains on (typo → reformulated query) pairs mined from sessions; Flink reads the Kafka query log, sessionizes by user/30-min gap, and emits (q1, q2) pairs into S3 for the trainer — exactly-once semantics matter so we don't double-count.Spark Streaming sessionization (mini-batch latency, weaker session windows); offline batch sessionization (re-reads everything daily).
Confusion-cost weighted edit distanceQWERTY-adjacent substitutions ("e"→"r") are 5× more likely than diagonal ones; learned per-key confusion costs replace uniform cost-1 edits and lift correction precision from ~85 % to ~92 % at the same recall.Uniform Levenshtein (lower precision); per-user confusion model (data-sparse).
Trigger threshold modelP(c)/P(q) > τ combined with low-result-quality signal prevents the embarrassing case of "correcting" a rare brand name; precision > 90 % is enforced by tuning τ on a held-out set, since a wrong suggestion is worse than no suggestion.Always-show (precision tanks); learned trigger model (more data, marginal lift).
S3 + atomic artifact swapFST + KenLM artifacts are ~5 GB; pushing to S3 with versioned keys and having servers download + atomic-swap via symlink rename gives zero-downtime ranking-data updates and instant rollback to the previous version.Hot-reload via RPC push (per-server flakiness); rolling restart (latency spike).
Bloom-filter blocklistProfanity / trademark blocklist has ~500 K entries; a Bloom filter (0.1 % FP, ~1 MB) gives O(1) post-scoring filtering and false positives are acceptable because we just suppress a suggestion — the blocklist is the last gate before user-visible text.HashSet (3× memory); regex matcher (slower, harder to update).
