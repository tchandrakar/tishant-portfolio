Design Elasticsearch architecture (inverted index, sharded, near-real-time)

Hard search storage-cache database-ddia

1. Problem statement

Design a distributed search and analytics engine in the spirit of Elasticsearch / OpenSearch: a sharded, replicated cluster sitting on top of Lucene that ingests JSON documents, builds inverted indexes (plus doc-values for aggregations), and serves search and aggregation queries with near-real-time visibility (1 s default refresh). Target use cases: log analytics, full-text product search, observability.

2. Functional requirements

  • Index JSON documents into named indexes; auto-discover or explicitly mapped fields.
  • Search: full-text, term, range, bool, phrase, fuzzy; aggregations (terms, date histogram, percentiles).
  • Near-real-time: documents searchable within ~1 s of indexing.
  • Scale horizontally by adding nodes; rebalance shards automatically.
  • Snapshots and restores to object storage.

3. Non-functional requirements

  • Latency: simple term query < 50 ms p99 on a warm shard; aggregations < 1 s on 1 B docs.
  • Scale: 1 PB stored, 100 B docs, 200 K index ops/sec, 10 K search QPS.
  • Durability: no data loss on single-node failure; RPO ≈ 0 with replicas.
  • Availability: 99.95%; cluster tolerates loss of any one zone.

4. Capacity estimation

  • Doc size 1 KB avg → 100 B docs ≈ 100 TB raw; inverted index + doc-values ≈ 1.3× → ~130 TB; with 1 replica → 260 TB; on i3.4xlarge (3.8 TB NVMe) ≈ 70 data nodes.
  • Indexing throughput: 200 K docs/s × 1 KB = 200 MB/s; partitioned over 50 primary shards → 4 K docs/s/shard.
  • Search QPS: 10 K × fan-out 50 shards = 500 K shard-queries/sec; per node ~7 K shard-q/s — well within Lucene budget.

6. High-level architecture

[Client] -> [Coordinating Node] --routing--> [Primary Shard] -> [Replica Shard] | | | v v v fan-out search Lucene segments Lucene segments | merge top-K + aggs [Master-eligible nodes] -- cluster state, shard allocation -- [Data nodes] [Ingest nodes] [Coordinating-only nodes]

Coordinating node hashes _id to a shard, forwards index ops to the primary, which replicates to in-sync replicas before acking. Search is scatter-gather across one copy of each shard.

7. Data model & storage

7.1 Lucene segment layout (per shard)

segment_N/
  .tim/.tip   term dictionary + index (FST)
  .doc/.pos   postings, positions
  .dvd        doc-values (column store for sort/agg)
  .fdt/.fdx   stored fields
  .liv        soft-delete bitset

7.2 Translog & refresh

Every index op is appended to a per-shard translog (fsync per request or per 5 s). A refresh every 1 s flushes the in-memory buffer to a new searchable segment (visibility = NRT). A flush commits segments and truncates the translog. Compaction merges small segments into bigger immutable ones — exactly the LSM pattern from DDIA Ch. 3.

7.3 Sharding

shard = hash(_routing or _id) % num_primary_shards. Primary count is fixed at index creation; resize via _split/_shrink or reindex.

8. Detailed component design

8.1 Cluster coordination

Master-eligible nodes elect a master via a Raft-like protocol (since 7.x). The master owns cluster state: index metadata, shard allocation table, mappings. Data nodes pull state diffs. Split-brain is avoided by quorum: at least (N/2)+1 master-eligible nodes must agree.

8.2 Indexing path

Coordinating node → primary: append to translog, add to in-memory buffer, replicate to replicas in parallel (wait_for_active_shards). Ack only after all in-sync replicas ack — synchronous replication, leader-based (DDIA Ch. 5).

8.3 Search path

Two phases: query (each shard returns top-K doc IDs + scores, coordinator merges to global top-K) then fetch (coordinator pulls stored fields for the final K). Aggregations use doc-values column store; partial results merge associatively in the coordinator.

8.4 Failure handling

If a primary dies, the master promotes an in-sync replica. Translog replays uncommitted ops on recovery. Allocation awareness keeps replicas in different zones.

9. Scaling & bottlenecks

  • Hot shard: caused by skewed routing or oversized index — mitigate with custom routing, ILM rollover, time-based indices (one per day).
  • Mapping explosion: dynamic mapping on log fields creates millions of fields → cluster state bloat. Cap with index.mapping.total_fields.limit and use flattened type.
  • Deep pagination: from+size is O(shards × from) — use search_after or scroll/PIT.
  • GC pauses: heap > 30 GB hits compressed-oops cliff; cap heap, prefer more nodes.
  • Refresh storms: for write-heavy workloads, raise refresh_interval to 30 s during bulk loads.

12. Talking points for the interviewer

  • "I'd push back on default 1 s refresh for log workloads — bumping to 30 s alone often doubles ingest throughput."
  • "The mental model I always use: each shard is a Lucene index, period. Everything else is plumbing."
  • "Cluster state is the hidden bottleneck at scale; mapping discipline matters more than CPU sizing."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Apache Lucene (per-shard engine)Each of the 50 primary shards is literally a Lucene index storing ~2 B docs at ~1 KB; Lucene's segment-based LSM with .tim/.doc/.dvd files gives sub-50 ms term queries on warm shards and is the only mature engine that bundles inverted index, doc-values, and stored fields with NRT semantics.Tantivy (Rust, fast but ecosystem is thin); custom inverted index (years of work to match Lucene's scorer/rerank code paths).
FST term dictionary (.tim/.tip)For 100 B docs across 50 shards, each shard holds tens of millions of unique terms; the FST encodes them at ~6 B/term and lets the .tip index sit fully in heap, avoiding disk seeks on the hot path that would blow the 50 ms p99 budget.Hash-based term dict (memory blow-up); B-tree on disk (extra seek per query).
Doc-values (column store .dvd)Aggregations on 1 B docs in <1 s require columnar access; doc-values store one value per doc per field in a packed column file, enabling SIMD-style scans at >1 GB/s — postings alone would force 10–100× more random reads.Fielddata in heap (OOM at 100 GB+); external columnar store (cross-process latency).
Translog + fsyncRPO ≈ 0 needs a per-shard write-ahead log; translog fsync per request gives durability under node loss, while batched fsync every 5 s is the escape hatch for 200 K ops/s ingest where 5 s is acceptable RPO.No WAL (data loss on crash); group commit only (no per-request durability).
Raft-like cluster coordination (master-eligible)Cluster state (mappings, shard allocation) for thousands of shards must be linearizable; the post-7.x quorum protocol prevents the split-brain that haunted Zen discovery and tolerates single-zone loss.ZooKeeper (extra dependency; ES dropped it); gossip-only (no linearizability for shard allocation).
Synchronous replica ack (leader-based)"No data loss on single-node failure" requires that the primary block until all in-sync replicas ack; this matches DDIA Ch. 5 leader-based sync replication and survives one zone failure when replicas are zone-spread.Async replication (data loss window); quorum-based (more complex, harder to reason about consistency).
Kafka (upstream ingestion buffer)200 K docs/s with bursty producers needs a durable buffer ahead of bulk indexers; Kafka absorbs spikes, enables replay for reindex, and lets bulk consumers pace themselves so refresh storms don't tip nodes into GC.RabbitMQ (no rewindable log at 200 K/s); direct HTTP _bulk (no shock absorber for spikes).
Roaring bitmaps (filter cache)Bool/term filters over billion-doc indices intersect at >1 GB/s with Roaring; cached per shard, this is what makes filtered aggregations sub-second instead of multi-second.Plain BitSet (4× RAM at 100 B docs); CRoaring with custom Java bindings (already used internally).
S3 / object storage (snapshots)1 PB cluster needs cheap, geo-redundant cold storage for snapshots; S3 at $0.023/GB/mo is ~10× cheaper than block storage and the snapshot diff format only ships changed segments.Local-disk backups (no DR); HDFS (operational overhead, no native ES integration).
i3.4xlarge (NVMe SSD nodes)260 TB total storage / 3.8 TB NVMe ≈ 70 nodes; NVMe gives the random-read IOPS Lucene needs for postings and stored fields, where EBS-only would saturate at 16 K IOPS per volume.EBS gp3 (lower IOPS, latency cliff); HDD-backed (kills query latency on cold segments).
ILM (Index Lifecycle Management)Time-based indices with hot→warm→cold→delete phases keep the active write index small (~50 GB primary) so refreshes are cheap, while old data ages onto cheaper hardware — essential for 100 B-doc log workloads.Single giant index (hot-shard problem); manual rollover scripts (error-prone at this scale).
JVM G1GC, <30 GB heapHeap above 30 GB loses compressed-oops, doubling pointer size and GC pressure; capping heap and adding more nodes keeps p99 GC pauses under 200 ms — directly bounded by the <50 ms p99 query goal.ZGC (lower pauses but worse throughput at the time of design); larger heap (latency cliff).
