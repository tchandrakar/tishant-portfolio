Design Google Web Search (crawler + indexer + serving)

Very Hard search

1. Problem statement

Build a planet-scale web search engine: crawl ~100B unique URLs on the open web, build a fresh inverted index that covers tens of trillions of postings, and serve ~100B queries per month at sub-300 ms p99 with relevance that matches a senior IR system. The system must keep the index reasonably fresh (minutes for news, days for the long tail) and survive the inevitable spam, soft-404s, infinite calendars, and adversarial SEO.

2. Functional requirements

  • Crawl: discover, fetch, and re-fetch HTML across the public web while respecting robots.txt and politeness windows.
  • Parse and normalize: extract anchor text, links, titles, structured data, language, charset.
  • Build an inverted index keyed on normalized terms with positions and per-document features (PageRank, freshness, anchor weights).
  • Serve keyword + phrase + filter queries with ranking (BM25 + LTR + freshness signals).
  • Snippets, spellcheck/did-you-mean, autocomplete, knowledge panels.

3. Non-functional requirements

  • Latency: p50 ~120 ms, p99 < 300 ms end-to-end including snippet generation.
  • Scale: ~100B queries/month (~38 K QPS avg, ~150 K QPS peak), ~100B URLs crawled, ~10T postings.
  • Freshness: news indexed in < 5 min; long-tail re-crawl < 30 days.
  • Availability: 99.99% serving; crawler may degrade gracefully.

4. Capacity estimation

  • QPS: 100B/month / 2.6M sec ≈ 38 K avg, 150 K peak. Each query fans out to ~50 index shards.
  • Storage: 100B docs × 50 KB compressed HTML ≈ 5 PB raw; inverted index ~1–2 PB after compression.
  • Crawl: 100B URLs / 30 days ≈ 38 K fetches/sec sustained; 5 PB/month egress on the crawl fleet.
  • Index shards: 50 KB postings × 100B docs / 1B postings per shard ≈ 5 K serving shards, 3 replicas → 15 K serving nodes.

5. API design

GET  /search?q=...&hl=en&num=10            → results, snippets, related
GET  /complete?q=...                         → autocomplete suggestions
POST /crawl/seed   {urls[]}                  → enqueue seeds (internal)
GET  /index/doc/{docId}                      → forward doc record (internal)

6. High-level architecture

[Seeds] -> [URL Frontier]--politeness/robots--> [Crawler Fleet] -> [Raw HTML Store/GFS] | v [Parser/Linker] -> [Link Graph (PageRank job)] | v [MapReduce Indexer] -> [Inverted Index Shards] | [User] -> [Edge/GFE] -> [Query Mixer] -> [Index Servers x N shards] -> [Doc Servers] -> [Snippet] |-> [Spell] [Autocomplete] [Knowledge]

Walk-through: the Mixer fans a query out to every index shard, each returns top-K doc IDs with feature vectors, the Mixer merges, calls Doc Servers for snippets, and the LTR ranker reorders.

7. Data model & storage

7.1 Forward doc record (Bigtable / GFS)

row_key   = fingerprint(url)
columns   = content:html, meta:title, meta:lang, links:out[], rank:pagerank,
            crawl:last_fetch, crawl:etag, crawl:status

7.2 Inverted index (per shard, on local SSD)

term -> { df, postings: [(docId, tf, positions[], field_mask, score_hint)] }
        -> skip lists every 128 postings, delta+varint encoded

7.3 Why this split?

Forward store is write-heavy and immutable per crawl; Bigtable's LSM matches that pattern (DDIA Ch. 3 — log-structured storage engines). Inverted shards are read-mostly, mmap'd, rebuilt offline, and swapped atomically.

8. Detailed component design

8.1 Crawler & URL frontier

Frontier is partitioned by registrable domain so politeness is local to one worker. Each domain has a token bucket (e.g., 1 req/sec by default, overridden by Crawl-Delay). Per-URL priority = f(PageRank, change-rate, freshness debt). Bloom filter (~1% FP) gates "have we seen this URL".

8.2 Indexer (offline, batch)

Classic MapReduce: map over forward docs emitting (term, posting); reduce sorts and writes per-shard segment files. Re-indexing the whole web is a multi-day batch job; incremental updates ride a real-time path that builds small in-memory segments and merges them log-structured-style (DDIA Ch. 10 — batch processing for the bulk job, Ch. 11 — stream processing for fresh-tier).

8.3 Serving

Two-tier: cheap candidate generation per shard (BM25 + static rank cutoff to top 1 K), then a heavier LTR model on the merged ~10 K candidates. Snippet generation reads the forward doc and highlights matched terms; capped at 30 ms.

8.4 Freshness pipeline

News and trending URLs go through a low-latency path: Pub/Sub → near-real-time indexer → small "fresh" shards queried in parallel with the main index and merged at the Mixer.

9. Scaling & bottlenecks

  • Tail latency: shard fan-out of 50 means p99 of any shard becomes p~p50 of the query — use hedged requests after p95.
  • Hot terms ("the"): truncate impact lists, use static rank early termination (WAND/BMW).
  • Crawl traps: per-host budgets, content-hash dedup, URL canonicalization.
  • Index size: compress with delta + varint + Roaring bitmaps; tier cold shards on slower storage.
  • Spam: separate ranking signals trained on labeled spam; demote at LTR stage.

  • "I would deliberately separate the bulk and fresh paths because their SLOs differ by three orders of magnitude."
  • "The tail-latency problem from 50-way fan-out is the silent killer; hedged requests and WAND are the levers I'd pull first."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Bigtable / GFS (forward doc store)Stores 100 B docs at ~50 KB each = 5 PB of write-once HTML; LSM matches the append-mostly crawl pattern, row-key fingerprint(url) gives O(1) lookup, and column families let us evolve crawl/meta/links independently without rewriting petabytes.HBase (operationally heavier at this scale); Cassandra (no native column-family scans); plain HDFS files (no random row lookup for 100 B keys).
Lucene-style inverted index segments~10 T postings compressed via delta + varint into ~1 PB across 5 K shards × 1 B postings/shard; mmap segment files give zero-copy reads at <5 ms per shard, and immutable segments enable atomic rebuild-and-swap.Postgres GIN (no shard fan-out at PB scale); building on raw RocksDB (would re-implement skip lists + scoring).
FST (term dictionary)Stores tens of billions of unique terms in ~6 bytes/term, an order of magnitude smaller than a hash map; supports prefix iteration for query expansion and intersects cleanly with Levenshtein automata for fuzzy match.Hash map of term→postings (3–5× memory); B-tree (pointer chasing kills cache locality).
MapReduce (bulk indexer)Re-indexing 100 B docs is fundamentally an embarrassingly parallel sort: map emits (term,posting), reduce sorts/writes per-shard segments. The job runs for ~24 h on ~10 K mappers and is restartable, which matters when crashes are routine.Spark (better dev ergonomics but higher per-node memory and weaker disk-shuffle scheduling for PB-shuffle); custom MPI (no fault tolerance).
Spark (PageRank, link graph)PageRank needs ~30 iterations over a 100 B-edge graph; Spark's RDD caching avoids re-reading 5 PB on each iteration, cutting wall-time from days to ~6 h on a 1 K-node cluster.MapReduce iterative (re-shuffles every iteration, 5–10× slower); Pregel/Giraph (operationally rare).
Kafka / Pub/Sub (fresh-tier ingestion)News URLs need to be indexed in <5 min; a 100 K-msg/s durable log buffers crawl-to-index hand-off, decouples crawler bursts from the NRT indexer, and provides replay for backfill.RabbitMQ (no rewindable log at 100 K msg/s sustained); direct RPC (no replay, brittle backpressure).
Flink (real-time spam & trending signals)Computes 5-minute tumbling counts over the click+query stream at 150 K events/s with exactly-once semantics — needed because click-CTR drives the LTR features and double-counting biases ranking.Spark Streaming (mini-batch latency 30 s+, no event-time windowing); Storm (no built-in state backend).
GBDT / LambdaMART (LTR L2 reranker)~500 trees × depth 8 over 200 features score 10 K candidates in <30 ms on one core via SIMD; pairwise NDCG loss directly optimizes the metric we care about, and feature contributions are explainable for debugging quality regressions.Logistic regression (under-fits non-linear interactions); pure DNN (10–100× slower without distillation).
Bloom filter (URL frontier dedup)Tracks "have we seen this URL" for 100 B URLs; at 1 % FP it costs ~120 GB of RAM total (~10 bits/URL) versus ~2 TB for an exact set, and false positives only mean we skip a fresh-but-already-seen URL — acceptable.Cuckoo filter (slightly better but harder to merge across crawl shards); exact set in RocksDB (disk lookup per URL kills crawl rate).
WAND / Block-Max WAND (early termination)For hot terms like "the" with billions of postings, WAND prunes >90 % of postings using upper-bound scores, keeping per-shard query latency at ~30 ms regardless of term frequency.Full-postings scan (10× slower on common terms); static caching of top-K per term (memory blowup, stale scores).
Roaring bitmaps (filter sets)For boolean filters (lang, country, safe-search) over billion-doc shards, Roaring runs AND/OR at >1 GB/s with 5–10× space savings vs uncompressed bitmaps.Plain bitsets (4× memory); compressed-skip-list intersect (slower AND).
gRPC + hedged requests (Mixer→shards)50-way fan-out means p99 of any one shard becomes the query's p~50; hedged requests fired after p95 cut tail latency by ~3×, which is the difference between a 300 ms and 900 ms p99.HTTP/1.1 (no multiplexing); fire-and-cancel without hedging (no tail mitigation).
