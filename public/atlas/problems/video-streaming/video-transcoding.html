Design Video Transcoding Pipeline (multi-bitrate, multi-codec)

Hard video-streaming file-storage

1. Problem statement

Design the transcoding pipeline that takes a raw mezzanine file (the producer-supplied master) and emits a fan-out of encoded variants for adaptive streaming: multiple codecs (H.264, HEVC, VP9, AV1), multiple resolutions (240p → 4K), multiple audio tracks, and packaged HLS/DASH/CMAF manifests. The pipeline must handle an ingest rate of ~500 hours of source video per minute (YouTube-scale) at peak, while keeping per-asset turnaround under 30 minutes for the standard ladder.

2. Functional requirements

  • Accept mezzanine in many input formats (MOV, MP4, MXF, ProRes).
  • Produce an ABR ladder per the codec/resolution policy.
  • Generate thumbnails (storyboard sprite + posters).
  • Extract closed captions, generate auto-captions.
  • Package into HLS & DASH; encrypt with DRM keys.
  • Re-encode (back-fill) when a new codec is introduced (e.g., AV1).

3. Non-functional requirements

  • Throughput: 30,000 hours/hour ingest sustained.
  • Latency: standard ladder ready < 30 min; "fast publish" 720p variant < 5 min.
  • Reliability: exactly-once output (no duplicate variants); idempotent retries.
  • Cost: encoding is the second-largest infra line after CDN; per-hour cost matters.
  • Quality: VMAF target ≥ 93 for the top rung, ≥ 70 for the bottom rung.

4. Capacity estimation

  • Compute: a 1080p H.264 encode runs ~2× real-time on a CPU box; AV1 ~0.3× real-time. With 30k hours/hr ingest and ~10 variants each, raw encoding need ≈ 100k vCPU-hrs/hr → ~50k VMs steady state, GPU-accelerated where possible.
  • Storage: each finalized asset = ~10× source size aggregated across rungs. 720 TB/day source × 10 = 7 PB/day encoded.
  • Queue depth: at peak we accept 100k jobs/min; backlog must drain in < 1 hour.

6. High-level architecture

[Upload Svc] ──► [Job Submit API] ──► [SQS / Pub-Sub: job queue] │ ▼ [Orchestrator (DAG engine)] │ splits into chunks ┌───────────────────────────────┼───────────────────────────────┐ ▼ ▼ ▼ [Probe worker] [Chunk encoder pool] [Caption worker] (ffprobe, scene detect) (CPU + GPU, spot fleet) (ASR + format) │ │ └────────► [Chunk store (S3)] ◄─┘ │ ▼ [Mux / package / DRM worker] │ ▼ [Final asset bucket] → [CDN origin] │ ▼ [Manifest Svc + Catalog]

The orchestrator splits the source on GOP boundaries (typically 2–6 s chunks), fans out parallel encodes across a worker pool, then re-muxes the results. This distributed-encode-then-stitch pattern is what gives us short turnaround on long videos.

7. Data model & storage

7.1 Job table (DynamoDB / Cassandra)

CREATE TABLE jobs (
  job_id uuid PRIMARY KEY,
  asset_id uuid,
  state text,        -- queued|probing|encoding|packaging|ready|failed
  attempts int,
  created_at timestamp,
  updated_at timestamp
);
CREATE TABLE chunk_results (
  job_id uuid, chunk_idx int, rung text,
  s3_key text, sha256 text, vmaf double,
  PRIMARY KEY ((job_id), rung, chunk_idx)
);

7.2 Why Cassandra/Dynamo?

Job state is write-heavy and TTL-friendly; we never query across jobs by anything other than job_id. Chunk_results clustered by (rung, chunk_idx) so the muxer can range-scan in stitch order.

7.3 Object storage

Source mezzanine and output variants live on S3/GCS with intelligent-tiering. Frequently re-accessed mezzanines stay on Standard; older ones tier down to Glacier-IR (instant retrieval) so we can re-encode for AV1 backfill without restoring from cold archive.

8. Detailed component design

8.1 Orchestrator

Modeled as a DAG: probe → split → encode (fan-out per rung × chunk) → mux → package → publish. Built on a workflow engine (Temporal / Airflow / Cadence) so each step is idempotent and retriable. Failed chunks retry with exponential backoff; the asset isn't published until all rungs reach 100%.

8.2 Worker pool

Stateless containers running ffmpeg/SVT-AV1/x265. Scheduled on Kubernetes with mixed instance types: GPU nodes for AV1/HEVC, CPU spot fleet for H.264. We bid on spot/preemptible because chunks are short and idempotent — losing a chunk costs < 1 vCPU-min.

8.3 Per-title encoding

Rather than a fixed bitrate ladder, we run a "complexity probe" pass and choose target bitrates per scene. Saves 15–25% bandwidth at equivalent VMAF — Netflix popularized this approach and it's now industry-standard.

8.4 Quality gate

After mux, we compute VMAF against the source. If any rung scores below threshold we re-encode that rung with a higher bitrate. Prevents publishing visually broken assets due to encoder bugs.

9. Scaling & bottlenecks

  • Long single videos: a 3-hour movie, encoded serially, would take hours. Chunking parallelizes it down to wall-clock = chunk_size / parallelism.
  • Spot eviction: chunks are stateless; lost chunks just retry. The orchestrator tracks attempts and circuit-breaks after N failures.
  • Storage I/O: chunk read/write to S3 is the next bottleneck; we pipe directly between containers when chunks are co-located.
  • AV1 backfill: a sweep across the catalog could DoS the encoder fleet. We rate-limit the backfill at 5% of capacity and prioritize titles by view-count.

12. Talking points for the interviewer

  • "I'd start with chunked encoding — it's the lever that turns wall-clock from O(duration) to O(duration/parallelism)."
  • "The hardest part is exactly-once: spot evictions retry, encoders are non-deterministic across versions, and we need idempotent output keys keyed on (asset, rung, chunk, encoder_version)."
  • "For new codecs like AV1 I'd ship hardware-accelerated encoders (NETINT, NVidia) before doing a catalog backfill — software AV1 is too slow for sustained throughput."
