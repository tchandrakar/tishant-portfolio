Design Google Meet (browser-based VC, WebRTC)

Hard video-streaming messaging

1. Problem statement

Build a browser-first video conferencing service that requires no install. All clients use WebRTC over getUserMedia/MediaStream APIs. Must integrate with Google Calendar, support Workspace SSO, and provide enterprise-grade compliance (SOC2, HIPAA-eligible). Live captions, noise cancellation, and meeting recordings are first-class.

2. Functional requirements

  • Calendar-driven meeting creation; one-click join from invite.
  • Up to 1000 participants in Enterprise; 500 audio+video tiles.
  • Live closed captions and translation in 30+ languages.
  • In-meeting chat, polls, Q&A, hand raise, breakout rooms.
  • Recording to Google Drive; auto-generated transcript in Docs.
  • Companion mode (2nd device for chat while in conf room).
  • Phone dial-in (Google Voice / PSTN gateway).

3. Non-functional requirements

  • Latency: mouth-to-ear < 200 ms p95 inter-region; < 100 ms intra-region.
  • Scale: 100M DAU, peak 10M concurrent meetings.
  • Browser-only path: no plugin, must work on Chrome/Safari/Edge.
  • Compliance: data residency by domain; E2E option for premium.

4. Capacity estimation

  • Concurrent participants: 10M meetings * avg 4 = 40M peak peers.
  • Bandwidth: 40M * 1 Mbps up + 1 Mbps down (avg) = 80 Tb/s.
  • SFU servers: at 2K participants/host → 20K SFU instances.
  • Captioning: 40M streams piped to ASR; ASR throughput needs ~400K GPUs assuming 100 streams/GPU; in practice batched on TPUs.
  • Recording: assume 5% recorded * 30 min avg * 0.6 Mbps = 100 PB/year stored to Google Drive.

5. API design

POST /v1/spaces                {calendarEventId}    → spaceId, joinUri
POST /v1/spaces/{id}/join       {ssoToken}          → mediaServer, iceCreds
WS   /v1/signal/{spaceId}       (per-meeting)       ↔ SDP/ICE/control
GET  /v1/captions/{spaceId}     (SSE)               ↔ {ts, lang, text}
POST /v1/recordings             {spaceId, layout}   → recordingId

6. High-level architecture

[Chrome]--WebRTC-->[Google Front End (GFE)]--->[Signaling/Spanner] | +--->[SFU PoP (Borg)]<---->[SFU PoP (peer region)] | +-->[Captioner: Speech-to-Text] +-->[Recorder]-->[Drive/Colossus] +-->[NoiseSuppressor (Lyra)]

Walk-through: client establishes DTLS/SRTP with the nearest SFU PoP via GFE-routed signaling. SFU forwards packets among peers; ASR taps the audio stream for captions; recorder is a virtual participant that composites and uploads to Drive.

7. Data model & storage

7.1 Spaces (Spanner)

CREATE TABLE spaces (
  space_id STRING(36) NOT NULL,
  org_id STRING(36), calendar_event_id STRING(64),
  data_region STRING(8), e2e_enabled BOOL,
  created_at TIMESTAMP,
) PRIMARY KEY (space_id);
CREATE TABLE participants (
  space_id STRING(36), user_id STRING(36),
  joined_at TIMESTAMP, role STRING(16),
) PRIMARY KEY (space_id, user_id), INTERLEAVE IN PARENT spaces;

Spanner gives multi-region strong consistency for org policies (data residency, recording opt-in) which need transactional updates across replicas.

7.2 Captions store

Streamed to Bigtable keyed by (space_id, ts) for fast retrieval; flushed to Drive transcript on meeting end.

8. Detailed component design

8.1 WebRTC stack in browser

Uses VP9 SVC (3 spatial * 3 temporal layers) for video and Opus for audio. SVC enables single-encode that the SFU can forward partial layers per subscriber — saves CPU vs simulcast.

8.2 SFU on Borg

Stateful service; participant pinned to SFU for meeting lifetime; if SFU crashes, client reconnects, ICE-restart picks new SFU within 2s (one tile flicker).

8.3 Captioner

Audio forked at SFU into Speech-to-Text streaming API; partial transcripts pushed via SSE to all participants. Translation pipelined by Google NMT for non-source languages.

8.4 Lyra noise cancellation

Runs client-side via WASM/WebGL — zero server cost, but a server-side fallback exists for low-power devices via the SFU's audio mixer.

9. Scaling & bottlenecks

  • Large meetings (1000-person): SFU subscribes only to top-N speakers via dominant-speaker detection (DSD); other tiles show static avatars.
  • Cross-region meetings: cascade SFU mesh — each region runs its own SFU, peer-to-peer relay between SFUs over Google's backbone.
  • ICE/TURN: Google's TURN ridges run on the same Borg cluster; ~15% of users need them.
  • Captioning latency: ASR adds ~300 ms; mitigate by streaming partials every 100 ms.
  • Data residency: Spanner placement rules pin space metadata to org's region; SFUs chosen accordingly even if it costs latency.

12. Talking points for the interviewer

  • "WebRTC in the browser dictates a lot — I work backwards from the supported codecs (VP9, Opus, H.264) and the JS APIs."
  • "Dominant-speaker detection is what lets me serve a 1000-person meeting on a single SFU — I'm not sending 999 streams to each client."
  • "Compliance is the underrated hard part: data residency means picking SFU PoP based on policy not just latency, and recordings must land in the org's region of Drive."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason. Here's the why behind each one:
