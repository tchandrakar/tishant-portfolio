Design Low-latency Live Streaming (LL-HLS, CMAF, sub-3s glass-to-glass)

Hard video-streaming cdn

1. Problem statement

Build a CDN-scale live streaming pipeline with glass-to-glass latency under 3 seconds, suitable for sports, auctions, and live events. Must support millions of concurrent viewers (CDN-scaled), adaptive bitrate, and DRM. Cannot rely on WebRTC because the audience size demands HTTP-cacheable delivery. Standard HLS at 6-second segments yields 20–30s latency — unacceptable.

2. Functional requirements

  • Ingest RTMP/SRT/WebRTC-WHIP from broadcasters.
  • Transcode to ABR ladder; package as LL-HLS and LL-DASH (CMAF).
  • Deliver via CDN with HTTP/2 + chunked transfer.
  • DRM (Widevine, FairPlay, PlayReady) for premium content.
  • DVR window (rewind 6 hours).
  • Live captions and metadata insertion (SCTE-35 markers).

3. Non-functional requirements

  • Latency: glass-to-glass 2–3s p95 (vs ~20s standard HLS).
  • Scale: 10M concurrent viewers per event, 100K events.
  • Quality: rebuffer ratio < 0.5%, video start time < 1s.
  • Availability: 99.95%; survive single-region failure.

4. Capacity estimation

  • Ingest: 100K streams * 8 Mbps = 800 Gbps; load balanced across 50 ingest PoPs.
  • Transcode: 100K * 5 renditions = 500K outputs; GPU/ASIC fleet ~50K nodes.
  • Egress: peak 10M viewers * avg 3 Mbps = 30 Tb/s; CDN handles 95%, origin shield ~1.5 Tb/s.
  • Origin storage: rolling 6-hour DVR window * 100K streams * 8 Mbps total = ~1.7 PB live working set on hot tier.
  • Manifest QPS: 10M viewers refreshing every 200 ms = 50M req/s on manifests; absolute necessity to serve manifests from edge with ETag/cache.

5. API design

POST /v1/channels                       {name, drm}      → ingestUrls{rtmp,srt,whip}
GET  /hls/{ch}/master.m3u8                               → master playlist
GET  /hls/{ch}/{rendition}/playlist.m3u8?_HLS_msn=N&_HLS_part=K   → media playlist (blocking)
GET  /hls/{ch}/{rendition}/seg{n}.fmp4                   → CMAF segment
GET  /hls/{ch}/{rendition}/seg{n}.part{k}.fmp4           → CMAF chunk (~200ms)

6. High-level architecture

[Encoder/OBS] --RTMP/SRT/WHIP--> [Ingest PoP] --> [Transcoder (GPU)] | v [LL-HLS Packager] -- CMAF chunks --> [Origin Shield (HTTP/2)] | | v v [DRM License Svc] [CDN Edge Tier] | v [Player]

Walk-through: encoder pushes to ingest. Transcoder produces CMAF with 200 ms chunks aligned across renditions. Packager writes manifests with EXT-X-PART and EXT-X-PRELOAD-HINT directives. Edge serves chunks with HTTP/2 chunked transfer; player uses blocking playlist reload for sub-second updates.

7. Data model & storage

7.1 Live segment store (Redis + S3 tiered)

Latest 30s of chunks live in Redis (key: ch:rendition:msn:part). After 30s, chunks flush to S3 and remain there for the DVR window. After 6 hours, optionally move to VOD bucket for replay or expire.

7.2 Manifest cache

edge_cache_key = (channel, rendition, msn, part)
TTL = 1s for live chunks, immutable=true once segment closes
s-maxage on master = 5s

7.3 Why not store in HDFS or Cassandra?

Chunk read pattern is hot-then-cold; in-memory + object storage matches the access pattern. Cassandra adds compaction overhead with no benefit for blob-style payloads.

8. Detailed component design

8.1 Encoder + transcoder

Encoder configures GOP=2s, keyframe-aligned, B-frames=0 to avoid B-frame reorder latency. Transcoder runs h264_nvenc with low-latency preset; per-rendition bitrate fixed via CRF + cap-VBV.

8.2 LL-HLS packager

Emits CMAF chunks every 200 ms inside 2-second segments. Manifest contains EXT-X-PART-INF, EXT-X-PART for the latest 3 segments, and EXT-X-PRELOAD-HINT for the next chunk, enabling the player to pre-issue blocking GETs.

8.3 Origin shield + edge

HTTP/2 chunked transfer is critical: a partial chunk being written by the packager must stream to the player as it grows. Shield holds open the connection, edges proxy via H/2 push or simple H/2 streaming. Range-coalesced requests deduplicate concurrent edge fetches at the shield.

8.4 Player tuning

Buffer target 1.5–2 chunks (~400 ms); jump-back if buffer stalls > 500 ms. Use Common Media Client Data (CMCD) headers so CDN can prioritize bandwidth-starved clients.

9. Scaling & bottlenecks

  • Manifest QPS: 50M+ rps; cache at edge with 1s TTL or use blocking-playlist hold-time so client requests stay open until new part is ready (saves 80% of QPS).
  • Thundering herd on segment publish: origin shield collapses N concurrent edge fetches to one origin pull (request coalescing).
  • DRM licensing: keep license requests at <2% of player count via long-lived per-session tokens (1 hour).
  • Cross-region failover: dual-ingest from broadcaster (or origin replication); CDN re-routes on origin health-check failure.
  • Latency budget breakdown: capture 50ms + encoder 300ms + ingest+transcode 500ms + packager 200ms + CDN buffer 1000ms + player buffer 500ms ≈ 2.5s.

12. Talking points for the interviewer

  • "The single most important design choice is CMAF chunked transfer over HTTP/2 — it lets the player consume bytes before a segment is fully written."
  • "Standard HLS latency is dominated by the player's 3-segment buffer (18s); LL-HLS replaces that with parts and blocking-playlist hold-time, dropping it to ~2s."
  • "I'd benchmark a 5M-viewer event with synthetic load and watch tail-latency on manifest QPS — that's almost always the breaking point before video bandwidth."
