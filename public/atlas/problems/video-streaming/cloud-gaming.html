Design Cloud Gaming (Stadia / GeForce Now — sub-50ms input-to-pixel)

Very Hard video-streaming cdn

1. Problem statement

Build a cloud gaming service that runs full-fat AAA games on GPU servers in datacenters and streams the rendered video to thin clients (browser, phone, TV). Input (controller, mouse, keyboard) travels back to the server. The hard constraint is end-to-end input-to-pixel latency under 50–80 ms — otherwise FPS games are unplayable. Must support millions of concurrent sessions and hundreds of game titles.

2. Functional requirements

  • Launch any supported game in <30s from click.
  • 1080p60 default, 4K60 / 120fps premium tiers.
  • Controller, keyboard, mouse, touch input.
  • Save game state to cloud; resume across devices.
  • Multiplayer matchmaking integrated with publisher backends.
  • Family/parental controls, session time limits.

3. Non-functional requirements

  • Latency: input-to-pixel < 80 ms p95 within region; <50 ms p50.
  • Scale: 1M concurrent sessions globally.
  • Reliability: session resume after transient disconnects up to 30s.
  • Image quality: 35+ Mbps effective at 4K, 15 Mbps at 1080p; PSNR > 40 dB.

4. Capacity estimation

  • GPU fleet: 1M sessions; one GPU per session at peak (some lighter games can multi-tenant). 1M GPUs is impractical → ~250K NVIDIA A10G/L4 tier multiplexed at avg 4 sessions/GPU = 250K GPUs.
  • Bandwidth: 1M * 20 Mbps = 20 Tb/s egress; this is comparable to all of Twitch peak.
  • Storage: game library 200 titles * 80 GB avg = 16 TB; replicated to every PoP. Save states ~50 MB each * 100M users = 5 PB.
  • PoP density: need a PoP within ~30 ms RTT of every user → ~50 PoPs globally for tier-1 markets.
  • Provisioning: session warm-pool 1% of capacity (always-on idle VMs) to keep launch latency under 5s.

6. High-level architecture

[Client] [Edge PoP] | controller --WebSocket/UDP--> [Input Forwarder] --> [Game VM (GPU)] | | | v | <----H.265/Opus over SRTP--- [HW Encoder (NVENC)] <-- [Rendered frames] | v [Save Sync] -> [S3] [Auth/Session API] -> [Scheduler] -> [GPU Pool Mgr] -> [Game Image Cache]

Walk-through: client requests session; scheduler picks the nearest PoP with available GPU and a warm VM with the requested game image. Client establishes a WebRTC data channel for input and an SRTP receiver for video/audio. Game runs in a Windows/Linux VM with a virtual GPU; frames are HW-encoded and streamed.

7.2 Save game blobs

Per-user, per-game; stored in S3 with versioning. Synced on game save callbacks and on session end. Cache last save in Redis for fast resume.

7.3 Game images

Distributed read-mostly: a master copy in object storage; each PoP has a local NVMe-backed cache. Use content-addressed dedup so 200 titles sharing engine binaries don't bloat storage.

8. Detailed component design

8.1 GPU multi-tenancy

NVIDIA MIG (Multi-Instance GPU) or vGPU partitions a single A100/L4 into 4–7 slices. Each slice gets dedicated VRAM and SMs, isolated from neighbors. Lighter games (indie, mobile ports) share more aggressively; AAA gets dedicated GPU.

8.2 Encoder pipeline

Game renders to a back-buffer; capture path uses NVFBC (NVIDIA Frame Buffer Capture) or Vulkan capture extension to grab without a GPU-CPU round-trip. NVENC encodes H.265 at low-latency preset (P1, lookahead off) in <2 ms per frame.

8.3 Input path

Client samples controller at 250 Hz; sends each event over WebRTC data channel (unreliable ordered) — UDP avoids head-of-line blocking. Server replays event into the game's HID via virtual gamepad driver.

8.4 Latency budget (target 60 ms total)

input capture 4ms + network up 20ms + game tick 16.6ms (1 frame) + encode 4ms + network down 15ms + decode 5ms + display ~ 65ms.

9. Scaling & bottlenecks

  • GPU is the bottleneck cost: CapEx dominates. Drive utilization via diurnal load balancing (Asia peak fills NA daytime capacity).
  • Cold start: Windows + AAA game cold boot is 60–120s. Mitigate with warm pools (per-title) and snapshot/resume of running VMs.
  • Network jitter: dynamic FEC (5–20% redundancy) and adaptive bitrate; on 5%+ loss, drop from 1080p to 720p inside one frame.
  • NAT/firewall: 100% UDP; need TURN-over-TCP fallback for restrictive networks (extra 10–20 ms).
  • Hot title spikes (game release day): pre-stage image to all PoPs and pre-warm 5K VMs the day of launch.

12. Talking points for the interviewer

  • "Cloud gaming is fundamentally a real-time hardware-rendering problem disguised as a streaming problem — the hardest sub-system is the GPU pool manager."
  • "I'd treat the 16.6 ms render frame as fixed and squeeze every other phase: NVENC at 4 ms, capture-zero-copy via NVFBC, kernel-bypass UDP — anything more is luxury."
  • "Stadia's failure was not technology — latency was actually solid — but business: the library, the per-game purchase model, and CapEx pricing didn't fit consumer expectations."
