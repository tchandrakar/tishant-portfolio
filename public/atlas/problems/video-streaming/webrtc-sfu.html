Design WebRTC SFU vs MCU architecture for group video calls

Hard video-streaming messaging

1. Problem statement

Design a media server backend for group video calls (4–500 participants) using WebRTC. Compare and pick between Selective Forwarding Unit (SFU), Multipoint Conferencing Unit (MCU), and hybrid topologies. The system must scale horizontally, handle NAT traversal, support recording, and keep mouth-to-ear latency under 200 ms.

2. Functional requirements

  • Group calls 2–500 participants with audio, video, screen share.
  • Adaptive bitrate per subscriber based on viewport and bandwidth.
  • Server-side recording into a single MP4.
  • Compatible with browser WebRTC and native SDKs.
  • End-to-end encryption option (Insertable Streams / SFrame).
  • Cascade across regions for geographically dispersed groups.

3. Non-functional requirements

  • Latency: < 200 ms p95 mouth-to-ear, < 250 ms intercontinental.
  • Scale: 1M concurrent participants across 100K rooms.
  • Loss tolerance: seamless playback up to 10% loss via NACK + FEC.
  • Cost: < $0.005 per participant-minute.

4. Capacity estimation

  • Mesh (P2P): N-1 uploads per participant; at N=10, each uploads 9× → 9 Mbps up at 720p. Falls apart beyond ~6 peers.
  • SFU: 1 upload, N-1 downloads per peer; server forwards N(N-1) pkts but does no decode — ~50 Mbps + 1 vCPU per 100 participants.
  • MCU: 1 upload, 1 mixed download per peer; server decodes and re-encodes — ~10× CPU vs SFU; ~1 GPU per 30 participants for HD.
  • Cluster sizing for 1M peers via SFU: 100 peers/box * 30% headroom → ~14K SFU bare-metal nodes.
  • Cluster sizing for 1M peers via MCU: ~140K nodes — 10× cost, ruled out for general use.

6. High-level architecture

+-------- MCU (mix & re-encode) -----+ | pros: 1 stream out | [Peer]--RTP----->+ cons: high CPU, lossy quality +-->[Peer] +-------------------------------------+ [Peer A] --RTP-->[ SFU ]-- selective forwarding --> [Peer B] | [Peer C] +--simulcast/SVC layer choice--> [Peer D] Hybrid: SFU + MCU recorder (mix only the recording, forward live)

Walk-through: clients publish via simulcast (3 layers) or SVC. SFU receives RTP, demuxes into producer streams, and per-subscriber chooses the best layer to forward. Recording is offloaded to a virtual subscriber (a small MCU) that mixes only when needed.

7. Data model & storage

EXPIRE room:{id} 3600

7.2 ICE/TURN credentials

Short-lived HMAC creds (5 min TTL) generated by signaling server; stored ephemerally; never persisted. TURN servers themselves stateless.

7.3 Recording metadata (Postgres)

CREATE TABLE recordings (
  id uuid PRIMARY KEY, room_id text, started_at timestamptz,
  duration_s int, s3_uri text, layout text, status text
);

8. Detailed component design

8.1 SFU internals

RTP demuxer → per-producer FIFO → per-subscriber forwarder. The forwarder picks simulcast layer based on (subscriber bandwidth, viewport tile size, dominant speaker score). Maintains NACK buffers (~500 ms) for retransmission.

8.2 Simulcast vs SVC

Simulcast: encoder emits 3 independent streams (e.g., 1080p/720p/180p). Cheap to switch, but 3× upload bandwidth.
SVC (VP9/AV1): single layered stream with spatial + temporal layers; SFU drops layers per subscriber. Saves bandwidth but heavier encoder.

8.3 Cascading SFUs

For multi-region rooms, regional SFUs peer over a mesh. A producer in EU publishes to EU-SFU, which relays to NA-SFU only if NA subscribers exist. Saves transatlantic bandwidth and gives each subscriber a low-latency local SFU.

8.4 Recording bot

A headless Chromium or libwebrtc subscriber connects to the room as a viewer, mixes video tiles via GPU compositor, encodes to H.264, uploads to S3 in 30s chunks.

9. Scaling & bottlenecks

  • Hot rooms (500+ peers): use dominant-speaker detection; only forward top 8–16 video streams.
  • UDP throughput per box: kernel-bypass (DPDK / io_uring) needed beyond 10 Gbps.
  • NAT/firewall: 15–25% of clients need TURN relay; budget separate TURN cluster sized at ~25% of SFU capacity.
  • Failover: SFU is stateful; on crash, signaling re-issues offer/answer with new SFU; ICE restart in <2s. Use connection draining for planned migrations.
  • E2E encryption (SFrame): SFU forwards opaque encrypted frames; can still drop layers because metadata is in plaintext header.

12. Talking points for the interviewer

  • "For nearly every modern group call, SFU is the default — MCU is a fallback for thin clients or PSTN bridging."
  • "The interesting layer is per-subscriber stream selection: dominant speaker, viewport size, and bandwidth jointly drive what the SFU forwards."
  • "E2E encryption is achievable in an SFU world via SFrame because the SFU only needs the RTP header, not the payload — that's a key insight that lets us keep zero-trust media."
