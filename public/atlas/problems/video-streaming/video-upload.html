Design Video Upload & Processing Pipeline (chunked, resumable, virus scan, thumbnails)

Hard video-streaming file-storage

1. Problem statement

Design the end-to-end upload path for a video platform: from a creator pressing "Upload" on flaky mobile data, all the way through chunked transfer, integrity verification, virus/malware scanning, thumbnail extraction, and handoff to the transcoding pipeline. The system must support multi-GB files (a 4K hour-long video can be ~20 GB), survive network drops, deduplicate identical re-uploads, and reject malicious content before it ever becomes visible.

2. Functional requirements

  • Resumable, chunked upload with byte-range continuation.
  • Client-side and server-side hash verification (SHA-256).
  • Pre-signed direct-to-object-store uploads (bypass app servers).
  • Virus / malware scan on the assembled file.
  • Probe (codec, duration, resolution) and reject malformed files.
  • Thumbnail extraction (poster + storyboard sprite).
  • Deduplication: if hash already exists, skip transcoding.
  • Webhook / callback when ready for transcode.

3. Non-functional requirements

  • Reliability: tolerate up to 100 retries / connection drops per upload session; sessions valid for 7 days.
  • Throughput: ingest 720 TB/day raw at YouTube-scale; spike to 2× during creator events.
  • Security: 100% of files virus-scanned before transcode kickoff; no execution path that bypasses scan.
  • Latency: finalize-to-thumbnail < 30 s for a 1 GB file.
  • Cost: upload bytes are billed once; avoid double-write.

4. Capacity estimation

  • Sessions: 5 M creators/day × 1.5 uploads = 7.5 M sessions/day → ~90 sessions/s avg, ~1k/s peak.
  • Chunks: avg 8 MB chunk × 100 chunks/file × 7.5 M files = 750 M chunks/day = ~9k chunks/s.
  • Storage staging: 7-day retention on raw uploads = ~5 PB rolling.
  • Scan compute: ~1 s/GB on a modern engine × 1 PB/day = ~12 scan-CPU-yrs/day → ~12k scanner cores.

5. API design

POST  /v1/upload/init   {filename,size,sha256,mime}    → {uploadId, partSize, urls[]}
PUT   {urls[i]}         (presigned S3 multipart part)   → ETag header
POST  /v1/upload/{id}/complete  {parts:[{n,ETag}]}      → {assetId, status:scanning}
GET   /v1/upload/{id}/status                            → {state, missingParts[]}
POST  /v1/upload/{id}/abort                             → 204
POST  /v1/webhook/transcode-ready (internal)            → 204

6. High-level architecture

[Creator] │ ▼ init ┌────────────────┐ [Edge GW] ─►│ Upload Init Svc│─► allocates uploadId, returns presigned URLs └───────┬────────┘ ▼ [DynamoDB: sessions] │ │ PUT chunks (presigned, direct) ▼ [S3 multipart staging bucket] │ ▼ complete [Upload Finalize Svc] ─► assemble (S3 CompleteMultipartUpload) │ ├─► [Probe Worker] (ffprobe → metadata) ├─► [Hash check] (SHA-256 vs client-provided) ├─► [Virus Scanner] (ClamAV / commercial) ├─► [Thumbnail Worker] (poster + sprite) │ ▼ [Asset DB] ── on success ──► [Transcode Job Queue] │ ▼ [Transcode Pipeline]

Direct-to-S3 multipart uploads with presigned URLs is the architectural cornerstone — bytes never traverse app servers, so the upload service stays cheap and the bandwidth bill is paid only once.

7. Data model & storage

7.1 Upload session (DynamoDB)

CREATE TABLE upload_sessions (
  upload_id   string PK,
  user_id     string,
  asset_id    string,
  size        bigint,
  sha256      string,
  state       string,    -- init|uploading|assembling|scanning|ready|failed
  parts       list<{n:int,etag:string,size:int}>,
  created_at  timestamp,
  ttl         number     -- 7-day TTL
);

7.2 Why DynamoDB / KV?

Single-key access pattern (upload_id), TTL for cleanup, predictable single-digit-ms reads. No relational queries needed.

7.3 Object layout

Staging bucket: uploads/{userId}/{uploadId}/parts/... with 7-day lifecycle to delete unfinished sessions. Finalized objects move (server-side copy) to raw/{assetId}/master.mp4 with object lock.

8. Detailed component design

8.1 Resumable chunking

Client splits the file into N parts (S3 multipart minimum 5 MB except last). On a network drop, client calls /status, receives missingParts, and PUTs only those. Each PUT is idempotent because S3 keys parts by partNumber.

8.2 Hash verification

Client provides full-file SHA-256 at init. After CompleteMultipartUpload the finalize worker streams the assembled object through SHA-256 and compares. Mismatch → reject; this catches both transmission corruption and tampering between init and complete.

8.3 Virus scan

Scan in a sandbox VPC with no egress to the public internet, results published to a Kafka topic. Quarantine bucket holds infected files for forensic review; scanner version is recorded per asset so rescans can be triggered when sigs update. Crucially: nothing reads from raw/ until the scan is green.

8.4 Probe + thumbnails

ffprobe extracts duration, codec, container, AV streams. Reject if duration > max, codec is unknown, or no video stream. Thumbnails generated at 0% / 25% / 50% / 75% + a sprite sheet (10×10 grid at uniform intervals) for scrubber preview.

8.5 Deduplication

Lookup sha256 in an asset_hash index. On hit, return the existing assetId without re-running transcode — saves enormous compute on identical re-uploads (common for multi-account creators or restoration jobs).

9. Scaling & bottlenecks

  • Hot creator burst: presigned URLs are stateless — the upload service horizontally scales without coordination.
  • Scanner backlog: autoscale on queue depth; ClamAV is CPU-bound, so spot fleets work well.
  • Mobile flakiness: default chunk size = 8 MB to minimize re-PUT cost on retries; client adaptively shrinks chunk size on bad networks.
  • Abandoned sessions: 7-day TTL on session row + S3 lifecycle on parts prefix prevents storage leak from sessions that never call complete.
  • Replay attacks: presigned URLs scoped to (uploadId, partNumber) and expire in 1 hour.

12. Talking points for the interviewer

  • "The single biggest design lever is presigned direct-to-store uploads — without it, you're paying for upload bandwidth twice."
  • "I'd treat the virus scan as a hard gate, not a side-effect — assets are unreadable by transcode until the scan emits a green event."
  • "For mobile creators, adaptive chunk size on the client is more impactful than any server-side change; an 8 MB chunk on 3G is a disaster."
