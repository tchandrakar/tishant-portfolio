Design Twitch (live streaming with chat at 10M concurrent)

Very Hard video-streaming messaging cdn

1. Problem statement

Build a global live-streaming platform where creators broadcast game/IRL video, viewers watch with sub-5-second glass-to-glass latency, and millions interact via real-time chat. The system must support a long tail of small streams plus mega-events (Twitch Rivals, Marbles tournaments) that hit 6M+ concurrent viewers on a single channel. Core challenge: ingest millions of RTMP streams, transcode in real time, fan out HLS segments through CDN, and run a chat system that delivers messages with <500ms p99.

2. Functional requirements

  • Creators ingest via RTMP/SRT; viewers watch via HLS/LL-HLS.
  • Adaptive bitrate ladder: source, 1080p60, 720p60, 720p30, 480p, 360p, 160p audio-only.
  • Real-time chat per channel, with slow-mode, sub-only, emote support, moderation.
  • Channel pages, browse/discovery, follow, raid, host, clip creation.
  • VOD recording: archive past broadcasts, generate highlights.
  • Subscriptions, bits, ads insertion.

3. Non-functional requirements

  • Latency: glass-to-glass 2–5s with LL-HLS; chat p99 < 500ms.
  • Scale: 10M concurrent viewers, 100K concurrent broadcasts, 30M DAU.
  • Availability: 99.95% for video plane; 99.9% for chat.
  • Throughput: peak 25 Tb/s egress globally.

4. Capacity estimation

  • Ingest: 100K streams * 6 Mbps source = 600 Gbps ingress.
  • Transcode: each stream → 5 renditions; 100K * 5 = 500K transcode slots; ~50K GPU/ASIC nodes (each handles ~10 streams).
  • Egress: 10M viewers * avg 2.5 Mbps = 25 Tb/s; assume CDN offload 95% → 1.25 Tb/s back to origins.
  • Chat: 10M concurrent, avg 1 msg/viewer/min, 10× fan-out per msg = ~1.6M msg/s send; 16M msg/s deliveries.
  • VOD storage: 100K streams * avg 4h * 6 Mbps = 1 PB/day raw; deduped + tiered to S3 Glacier after 14d.

6. High-level architecture

[Broadcaster] --RTMP--> [Anycast Ingest PoP] --> [Transcoder (GPU)] --> [Origin Shield] | v [HLS Packager] --> [CDN Edge] --> [Viewer] | v [VOD Recorder] --> [S3] [Viewer] --WS--> [Chat Edge (haproxy)] --> [Channel Shard (Erlang)] --> [Redis pubsub] | v [Kafka chat-log] --> [Moderation ML] --> [Spam Filter]

Walk-through: Broadcaster sends RTMP to nearest anycast PoP. Ingest forwards the source to a transcoder cluster that emits an ABR ladder. Packager produces HLS/LL-HLS segments uploaded to origin shield; CDN edges pull-through and serve to viewers. Chat is a separate plane: WebSocket connections terminated at chat edges, sharded per-channel onto Erlang/Elixir nodes that fan out via Redis pub/sub.

7. Data model & storage

7.1 Channel metadata (DynamoDB)

channel_id (PK) | streamer_id | title | category | started_at | viewer_count | sub_only_chat (bool)

7.2 Chat history (Cassandra, TTL 30d)

CREATE TABLE chat_msgs (
  channel_id text, bucket_5m bigint, msg_id timeuuid,
  user_id text, body text, badges set<text>,
  PRIMARY KEY ((channel_id, bucket_5m), msg_id)
) WITH CLUSTERING ORDER BY (msg_id DESC);

7.3 VOD index (Postgres + S3)

Master manifest in Postgres; segments and thumbnails in S3 with lifecycle to Glacier Deep Archive after 60 days for non-partner streams.

8. Detailed component design

8.1 Transcoder

NVIDIA NVENC ASICs on bare-metal; one source stream → 5 outputs in parallel. Use 2-second GOPs aligned across renditions for clean ABR switching. Failover: if a transcoder dies mid-segment, viewer skips one segment (≤2s glitch); supervisor reschedules within 1s.

8.2 LL-HLS packager

CMAF chunks of 200ms, 10 chunks per 2s segment; uses HTTP/2 push and #EXT-X-PRELOAD-HINT to keep latency under 3s. Falls back to standard HLS for older clients.

8.3 Chat fanout

Channel sharded by consistent hash on channel_id. Mega-channels (>100K viewers) auto-split into a tree: leader receives sends, fans to N regional sub-shards which deliver to viewers. Slow-mode and follower-mode enforced at the channel leader.

8.4 Moderation

Async pipeline: every chat msg published to Kafka. ML classifier flags slurs/spam; heavily flagged users auto-timed-out via API back to chat. Mods get instant override authority.

9. Scaling & bottlenecks

  • Mega-events: pre-warm CDN PoPs and origin shields based on category trends; subscribe streamers go through reserved capacity pool.
  • Thundering herd: when a big streamer goes live, viewer count can ramp 0→1M in 60s. Use jittered manifest TTLs and connection-rate limiters at edges.
  • Chat hot channel: Erlang VM per channel; if mailbox queue >50K, switch channel to read-only sample-mode where only 10% of msgs are delivered to non-subs.
  • Cross-region replication: chat is single-leader per channel; reads served from regional replicas to avoid WAN RTT.

12. Talking points for the interviewer

  • "I'd start by separating the video plane (CDN-served HLS) from the chat plane (WebSocket pub/sub) — they have totally different scaling characteristics."
  • "Hottest channel sharding is the killer problem; Drake or a Worlds final pulls 6M concurrent into one logical channel — I'd use a fanout tree rooted at the leader Erlang node."
  • "For LL-HLS I trade away the sub-second latency of WebRTC because I cannot serve 10M concurrent WebRTC peers without a forest of SFUs — CDN economics win."
