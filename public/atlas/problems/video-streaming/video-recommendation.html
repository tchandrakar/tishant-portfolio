Design Video Recommendation System (candidate gen + ranking)

Hard video-streaming recommendation

1. Problem statement

Design the recommendation system that powers the "home feed" and "Up Next" surfaces of a video platform. The system must turn a corpus of ~1 B videos and a viewer's history into an ordered list of 20–50 items, in < 200 ms, and improve a north-star metric like watch-time-per-session. YouTube reports that ~70% of watch time comes from recommendations, making this the highest-leverage system in the entire product. The classic two-stage architecture — candidate generation + ranking — is what makes the latency and quality tractable at this scale.

2. Functional requirements

  • Personalized home-page feed per user/profile.
  • "Up Next" recommendations during/after playback.
  • Exploration vs exploitation balance (avoid filter bubbles).
  • Real-time response to user signals (skip, like, "Don't recommend").
  • Cold-start handling for new users and new videos.
  • Diversity, freshness, and policy constraints (no clickbait promotion).

3. Non-functional requirements

  • Latency: < 200 ms p95 end-to-end ranking call.
  • Scale: 1 B users × tens of impressions/day = ~10 B ranking calls/day = ~120k QPS avg, ~500k peak.
  • Freshness: a video uploaded an hour ago should be reachable in candidate generation.
  • Coverage: the system must be able to surface the long tail, not just the top 1%.
  • A/B-able: every model and every business rule must be experimentally swappable.

4. Capacity estimation

  • Candidate gen: ~120k QPS × 5 retrievers × 1k candidates each = 600 M candidate lookups/s aggregate; ANN index must serve > 100k QPS per shard.
  • Ranking: 120k QPS × ~500 candidates per request × ~1 µs per scoring op = ~60 M scoring ops/s; served by a TF-Serving / TorchServe fleet.
  • Feature store reads: ~120k QPS × 200 features × 8 bytes = ~200 GB/s feature traffic — needs in-memory KV.
  • Training data: 10 B impressions/day × 1 KB = 10 TB/day into the data lake.

6. High-level architecture

┌──────────────────────────────┐ │ Online Serving Path │ [Client] ─► [Edge GW] ─► [Reco GW] ─► [Candidate Generators] ├─ ANN retriever (user→video embed) ├─ Subscription/follow retriever ├─ Trending retriever └─ Personalized rules retriever │ ▼ ~500 candidates [Ranker (DNN)] ◄── [Feature Store (Redis/Bigtable)] │ ▼ re-rank for diversity / policy [Final 20–50 items] Offline path: [Kafka: impressions, plays, likes] → [Flink: realtime features] → [Spark/Beam: training data] → [Trainers] → [Model Registry] → [Serving] → [Embedding builder] → [ANN index build] → [Vector store]

The split between candidate gen (recall) and ranking (precision) is what makes the latency budget work: candidate gen reduces 1 B → 500 with cheap retrievers, then a heavy DNN scores only those 500.

7. Data model & storage

7.1 User & video embeddings (vector store)

user_embed[userId]    -> float32[256]
video_embed[videoId]  -> float32[256]
ann_index             -> HNSW over video_embed, sharded by random hash

Stored in a dedicated vector DB (Vespa, ScaNN, Faiss-shard) for sub-10ms top-K queries.

7.2 Feature store

fs:user:{uid}:lastN_videos        -> list<videoId>
fs:user:{uid}:topic_affinity      -> map<topic,float>
fs:video:{vid}:ctr_24h            -> float
fs:video:{vid}:watchtime_p50      -> int

Redis Cluster for low-latency online features; a parallel offline view in Bigtable/Bigquery for training. Both are populated from the same Kafka stream so training-serving skew stays bounded.

7.3 Impression log

Every served item is logged with ranker version, candidate sources, and outcome. This is the fuel for training and for counter-factual evaluation.

8. Detailed component design

8.1 Candidate generation

Multiple retrievers in parallel: (1) ANN over user/video embeddings — captures "users like you watched this", (2) graph-based co-watch — videos co-watched in the same session, (3) subscription/follow — fresh uploads from followed channels, (4) trending bucket — locale & topic. Each emits ~200 candidates; we union and dedupe.

8.2 Ranker

Multi-task DNN with shared trunk and heads for click probability, expected watch time, and satisfaction (1-5 survey). The final score is a learned linear combo of heads. Inputs: user features, video features, cross features, context features. ~50 ms inference budget per request.

8.3 Re-ranker

After scoring, we apply diversity (MMR), freshness boosting, and business/policy constraints (e.g., suppress titles flagged by Trust & Safety). This is the place where product intuition lives — it's intentionally simple and rule-based so PMs can reason about it.

8.4 Cold start

New users: bootstrap from registration signals (locale, age band, declared interests) and content-based candidates. New videos: rely on content embeddings + a "exploration boost" that gives every new video a small impression budget for ~24 hours; the system collects engagement signal to update embeddings.

9. Scaling & bottlenecks

  • ANN throughput: shard the index by random hash; query fan-out + scatter-gather. Each shard runs HNSW in-memory.
  • Feature fanout: co-locate user and video features in same Redis shard via consistent hashing on userId; fetch 500 video features in one batched mget.
  • Training-serving skew: compute features once in Flink and write to both stores; never compute features differently in the trainer vs the server.
  • Model rollout: shadow-eval new models on 1% of traffic for a week before promotion; canary at 5%.
  • Filter bubbles: cap the share of any single channel/topic in the final feed (e.g., max 3 videos from any one channel in the top 20).

10. Trade-offs

12. Talking points for the interviewer

  • "I'd commit to two stages on day one — anything else doesn't fit the latency budget at billion-item scale."
  • "The hardest part is training-serving skew — most production reco regressions I've seen come from a feature being computed slightly differently online vs offline; my mitigation is a single Flink pipeline writing both stores."
  • "For metrics, I'd optimize watch-time per session but guard with satisfaction surveys, otherwise the model will learn to maximize sessions full of clickbait."
