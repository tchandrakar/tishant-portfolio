Design Zoom (video conferencing)

Very Hard video-streaming messaging

1. Problem statement

Build a video conferencing platform supporting 1:1 calls, small huddles, and large webinars (up to 10K active viewers, 1K speakers). Audio/video must be real-time (<200ms mouth-to-ear), resilient to packet loss, and run on phones, browsers, desktop apps. The system must also handle screen share, recording, breakout rooms, chat, and PSTN bridging — at planet scale with 300M daily meeting participants.

2. Functional requirements

  • Schedule and join meetings with Meeting ID + passcode or SSO.
  • Multi-party audio/video; up to 49 tiles in gallery view, simulcast streams.
  • Screen share, virtual backgrounds, raise-hand, reactions.
  • Cloud recording to MP4 with searchable transcript.
  • Breakout rooms (split N participants into M sub-meetings).
  • PSTN dial-in/out, SIP gateway for room systems.
  • Webinar mode: hosts, panelists, view-only attendees.

3. Non-functional requirements

  • Latency: end-to-end audio < 150 ms p95; video < 200 ms p95.
  • Scale: 300M daily participants, peak 30M concurrent.
  • Reliability: packet-loss tolerance up to 30% via FEC + retransmit.
  • Availability: 99.99% control plane, 99.95% media plane.

4. Capacity estimation

  • Concurrent meetings: 30M participants / avg 5 = 6M meetings.
  • Bandwidth: 30M * avg 1.5 Mbps up + 1.5 Mbps down = 90 Tb/s aggregate.
  • SFU footprint: a single SFU box handles ~500 participants → 60K SFU nodes globally; ~120 PoPs.
  • Recording: 10% of meetings recorded * avg 45min * 0.5 Mbps = ~150 PB/year stored to S3.
  • Signaling QPS: 6M meetings * 4 control msgs/sec = 24M msg/s on the control plane.

6. High-level architecture

[Client] --HTTPS--> [Edge LB] --> [Signaling Cluster] --> [Meeting Coordinator] | | | v | [SFU Selector] | | +----UDP/SRTP----------------------------------------> [SFU Edge PoP] | +--> [Recording Pod] --> [S3] +--> [Transcribe Svc] --> [Postgres] +--> [PSTN Gateway]

Walk-through: Client signs in, requests to join a meeting; the coordinator picks a SFU PoP near the centroid of participants. Media flows over UDP/SRTP via the SFU which forwards (not mixes) packets. Recording, transcription, and PSTN are independent subscribers that pull from the SFU.

7. Data model & storage

7.1 Meeting metadata (Postgres, sharded by meeting_id)

CREATE TABLE meetings (
  meeting_id bigint PRIMARY KEY,
  host_id bigint, topic text, scheduled_at timestamptz,
  passcode text, sfu_pop text, started_at timestamptz, ended_at timestamptz
);
CREATE TABLE participants (
  meeting_id bigint, user_id bigint, joined_at timestamptz, role text,
  PRIMARY KEY (meeting_id, user_id)
);

7.2 Recording index (DynamoDB)

recording_id (PK) | meeting_id | s3_uri | duration_s | transcript_id | retention_days

7.3 Why SFU vs MCU?

SFU forwards encrypted streams without decoding — 10× cheaper CPU than MCU mixing, and preserves E2E options. Trade-off: client must decode N streams (Zoom mitigates with simulcast and SVC layer dropping).

8. Detailed component design

8.1 SFU

Built on libwebrtc forks; runs on bare-metal with kernel-bypass UDP (DPDK). Each participant publishes 3 simulcast layers (180p, 360p, 720p); SFU subscribes others to the layer matching their viewport. SVC is used for screen-share where layered SVC bitrate adaptation reduces CPU vs simulcast.

8.2 Congestion control

Google Congestion Control (GCC) per stream; SFU passes REMB/TWCC feedback. On 5%+ loss, client downgrades to next simulcast layer; on 30% loss, audio-only fallback.

8.3 Breakout rooms

Logically a parent meeting spawns N child meetings; SFU keeps the same physical session and re-routes pub/sub maps server-side, so participants don't reconnect.

8.4 Recording

Recording bot subscribes as a virtual participant; mixes layouts server-side via headless renderer; uploads MP4 in 30s chunks to S3 multipart, finalized when meeting ends.

9. Scaling & bottlenecks

  • Hot meetings (10K+ webinars): use a cascaded SFU tree — one origin SFU forks to N relay SFUs that each serve up to 1K viewers.
  • Geographic optimization: coordinator picks SFU minimizing weighted RTT to participants; live-migrates if centroid shifts (e.g., majority join late from another region).
  • NAT traversal: 80% of clients reach SFU directly via STUN; 20% need TURN relay → TURN cluster sized at 25% of SFU bandwidth.
  • Signaling scale: Kafka-backed pub/sub per meeting room; meeting-level state in Redis with lease; failover via Raft (5-node etcd) for coordinator.

12. Talking points for the interviewer

  • "I'd separate signaling (TCP, control) from media (UDP, real-time) — they have totally different reliability and latency profiles."
  • "SFU vs MCU: SFU wins on CPU and quality but pushes complexity to the client; for webinars I cascade SFUs into a tree to scale viewers without exploding upload bandwidth."
  • "PSTN is a hard problem — I'd model it as a transcoding bridge that ingests PCM, runs WebRTC, and joins as a virtual participant."
