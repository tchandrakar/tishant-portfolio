Design S3-style Object Storage (durability 11-9s, erasure coding)

Very Hard file-storage database-ddia

1. Problem statement

Design a globally distributed object store with the durability and API surface of Amazon S3: 11 nines of durability, virtually unlimited capacity, single-key strong read-after-write consistency, and an HTTP REST interface. Objects range from 1 byte to 5 TB. The system must internally use erasure coding (not 3x replication) to keep storage cost competitive at exabyte scale.

2. Functional requirements

  • PUT / GET / DELETE / HEAD / LIST on objects keyed by (bucket, key).
  • Multipart upload for objects above 5 GB, up to 10,000 parts.
  • Versioning, lifecycle policies (move to IA / Glacier after N days), object lock (WORM).
  • Bucket-level IAM and signed URLs.
  • Server-side encryption (SSE-S3, SSE-KMS, SSE-C).

3. Non-functional requirements

  • Durability: 99.999999999% (11-9s) annual.
  • Availability: 99.99% standard tier.
  • Latency: first-byte under 100 ms p99 in-region for warm GET.
  • Scale: 280 trillion+ objects in production; 100 M+ requests/s peak globally.
  • Cost: < 1.5x raw bytes including all overhead at the largest tier.

4. Capacity estimation

  • Objects: 280 T total; assume 200 KB median → ~56 EB user data.
  • Erasure coding 10+4 → 1.4x storage overhead = ~78 EB physical.
  • Request mix: 90% GET, 8% PUT, 2% LIST; 100 M req/s peak → 90 M GET/s, 8 M PUT/s.
  • Index size: 280 T x ~250 B per row = 70 PB of metadata (sharded across thousands of nodes).
  • Cross-region replication: ~3 PB/day for customers who opt in.

6. High-level architecture

[Client] ──► [GSLB / Anycast] ──► [Front-End Web Servers] │ ┌─────────────────────┼──────────────────────┐ ▼ ▼ ▼ [Index Service] [Storage Nodes] [Auth / IAM] (sharded KV) (chunk servers) │ │ ▼ ▼ [Replicated index DB] [Erasure-coded shards across 3 AZs] │ ▼ [GC + Repair workers]

Walk-through: a PUT lands on a stateless front-end which streams the body through a chunker (8 MB chunks), erasure-codes each chunk 10+4, and writes the 14 shards to 14 different storage nodes spread across 3 availability zones. Once a quorum of writes succeeds, the front-end commits a single row to the Index Service mapping (bucket, key, version) → manifest of chunk locations. Only after the index commit succeeds does the client see HTTP 200.

7. Data model & storage

7.1 Index (sharded LSM KV, like FoundationDB / DynamoDB)

row key:  bucket || 0x00 || key || 0x00 || version_id
value:    {
  size, etag, content_type, storage_class,
  chunk_manifest: [{chunk_id, ec_geometry, node_set, offset}],
  encryption: {alg, key_id, iv},
  created_ts, lifecycle_state
}

Range-partitioned by bucket||key so LIST is a single-shard prefix scan.

7.2 Storage layer

Each chunk = 8 MB user data → 10 data shards of 800 KB + 4 parity shards via Reed-Solomon. Tolerates loss of any 4 of the 14 shards. Shards are placed on different failure domains (rack + AZ aware). Stored on log-structured ext4 with bitrot scrubbing every 14 days.

8. Detailed component design

8.1 Why erasure coding, not 3x

3x replication = 3.0x cost. 10+4 EC = 1.4x cost for the same fault tolerance against AZ loss. At 56 EB user data this is the difference between 168 EB and 78 EB of physical disk — billions of dollars.

8.2 Strong read-after-write consistency

The index is the single source of truth. A GET first reads the index row (linearizable single-key read via Paxos/Raft on that shard), then fetches chunks. As long as the index commits the new manifest before returning 200 to the writer, every subsequent read sees it. This is exactly the model AWS shipped in December 2020.

8.3 Multipart upload

Each part is independently chunked and EC'd. The final POST glues part manifests into one object manifest in a single atomic index write — no data movement.

8.4 Background workers

Garbage collection (delete tombstoned chunks after the lifecycle window), repair (regenerate missing shards from any 10 survivors), scrubber (periodic CRC), and cross-region replicator (tails the index change log and replays PUTs to the peer region).

9. Scaling & bottlenecks

  • Hot keys: S3 historically had 3,500 PUT / 5,500 GET per prefix. Solution: range partitions auto-split when QPS exceeds threshold; clients are advised to randomize key prefixes.
  • Small object overhead: a 1 KB object with 14 shards costs more in metadata than data. Pack many small objects into 64 MB extents on disk; the index points to (extent_id, offset, length).
  • Repair storms: a rack failure can require regenerating petabytes. Token-bucket per rack so repair never starves user IO.
  • LIST performance: LIST is a range scan; pagination of 1,000 keys keeps it bounded.

12. Talking points for the interviewer

  • "The bet that defines this design is splitting the index plane from the data plane and demanding linearizability only on the index — that's how you get 11-9s without paying 3x for replication."
  • "At 280 trillion objects the metadata is itself a 70 PB database; small-object packing is not optional."
  • "I'd staff a separate team just for the repair / scrubber pipeline because that's where durability is actually earned."
