Design Dropbox / Google Drive (file sync + delta encoding)

Hard file-storage collaboration

1. Problem statement

Build a cloud file-sync product where any change made on one device (Mac, Windows, Linux, iOS, Android, web) is reflected on every other linked device within seconds. Files range from a 12 KB .txt note to a 50 GB Final Cut Pro project. Users expect offline edits, selective sync, shared folders, version history, and bandwidth-cheap updates when only a few bytes change inside a multi-gigabyte file.

2. Functional requirements

  • Upload, download, rename, move, delete files and folders.
  • Bi-directional sync across N devices per user; offline writes reconcile on reconnect.
  • Shared folders with per-user ACLs (viewer / commenter / editor).
  • Version history (30 days free, 1 year paid) and undo of any change.
  • Delta sync: only changed byte ranges are transferred.
  • Cross-user deduplication (server-side block dedup).

3. Non-functional requirements

  • Latency: change-event fan-out under 1 s p95 within a region; first-byte download under 200 ms p95 from edge.
  • Scale: 700 M registered users, 100 M DAU, 100 B files, 5 EB stored.
  • Durability: 11-9s on object storage; metadata RPO < 1 s.
  • Availability: 99.95% on the metadata plane.
  • Bandwidth efficient: a 1 KB edit in a 1 GB Keynote should transfer ~4 KB, not 1 GB.

4. Capacity estimation

  • QPS: 100 M DAU x 50 metadata ops/day / 86,400 = ~58 K metadata QPS avg, 250 K peak.
  • Storage: 5 EB raw + 30% version overhead = ~6.5 EB; with 35% global dedup savings, ~4.2 EB physical.
  • Bandwidth: 8 PB/day uploaded, 24 PB/day downloaded → 280 Gbps and 850 Gbps respectively.
  • Notification fan-out: 100 M DAU x 5 devices = 500 M long-lived WebSocket connections at peak.

5. API design

POST /v1/files/commit       {path, parent_rev, blocklist}    → new_rev
GET  /v1/files/download     {file_id, rev}                    → stream
POST /v1/blocks/upload      {hash, size, data}                → block_id
GET  /v1/delta              {cursor}                          → {entries, cursor}
POST /v1/share              {folder_id, email, role}          → share_id
GET  /v1/longpoll_delta?cursor=...                            → {changes:true}

6. High-level architecture

[Client FS Watcher] ── chunker ──► [Block Server] ──► [S3 / blob store] │ │ │ metadata │ block_id ▼ ▼ [Metadata API] ──► [Sharded MySQL: namespace] ──► [Kafka delta log] │ │ ▼ ▼ [Notification Svc / WebSockets] ◄──── [Indexer / Search] [Version GC]

Walk-through: the client watcher detects a changed file, runs rolling-hash chunking (4 MB target), uploads only new blocks to the Block Server, then commits metadata referring to the new block list. The metadata commit appends to a per-namespace Kafka log; the Notification service tails it and pushes a "something changed" ping over the user's WebSocket. Each peer device then pulls a /delta page and downloads only the missing blocks. This split (block plane vs metadata plane) is the core Dropbox design and exactly what the original Magic Pocket paper describes.

7. Data model & storage

7.1 Metadata (sharded MySQL)

CREATE TABLE fs_node (
  ns_id        BIGINT,
  node_id      BIGINT,
  parent_id    BIGINT,
  name         VARBINARY(1024),
  type         TINYINT,        -- file / dir
  rev          BIGINT,
  block_list   VARBINARY,      -- ordered SHA-256 block hashes
  size         BIGINT,
  mtime        BIGINT,
  PRIMARY KEY (ns_id, node_id)
) ENGINE=InnoDB;

Sharded by ns_id (namespace = user or shared folder). Joins always live on a single shard.

7.2 Blocks (object store)

Content-addressed: object key = SHA-256(block). Immutable, never overwritten. Erasure coded 10+4 across 4 AZs. Reference-counted by metadata commits; GC sweeps unreferenced blocks after 7 days.

8. Detailed component design

8.1 Delta encoding

Variable-size chunking with a 64-byte rolling hash (Rabin or Buzhash) avoids the boundary-shift problem of fixed chunking. A 1-byte insert near the front of a 1 GB file shifts only the chunk it lands in; subsequent chunk boundaries re-align. Average 4 MB chunks, min 1 MB, max 8 MB.

8.2 Notification & longpoll

500 M WebSockets are served from a sharded edge tier (one Go process per CPU, ~50 K conns each). The server holds the longpoll for up to 90 s and replies with {changes:true} when the user's namespace cursor advances.

8.3 Conflict resolution

Two devices commit against the same parent rev → second commit fails with HTTP 409. The losing client renames its copy file (Alice's conflicted copy 2026-05-06).docx and re-commits. No silent overwrite.

8.4 Shared folders

Each shared folder is its own namespace. Joining a share is a metadata mount, not a copy — disk usage is counted against the folder owner.

9. Scaling & bottlenecks

  • Hot namespaces (large shares): a 5,000-person engineering folder gets sharded by sub-tree; commit lock is per sub-tree, not per namespace.
  • Block put thundering herd: client first calls HEAD /blocks/{hash}; if present, skip the upload. Saves bandwidth on cross-user dedup.
  • WebSocket fan-out: notification service is stateless and uses consistent hashing on user_id so reconnects land on the same node 99% of the time, preserving the cursor cache.
  • Metadata cross-shard moves: moving a 1 M-file folder between namespaces is implemented as a logical re-parent + lazy copy.

12. Talking points for the interviewer

  • "I'd separate the block plane from the metadata plane on day one — that single decision is what lets Dropbox dedup across users and run them on totally different storage backends."
  • "The hardest part is delta sync on huge files; Rabin chunking is the textbook answer but I'd benchmark it against FastCDC for our workload."
  • "Version history is just retention on immutable blocks — it costs almost nothing extra because of dedup."

Every technology choice in this design exists for a concrete reason.
