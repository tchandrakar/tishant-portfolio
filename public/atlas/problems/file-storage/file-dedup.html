Design File Deduplication System (content-defined chunking, hash dedup)

Hard file-storage storage-cache

1. Problem statement

Design a global deduplication tier that sits behind a backup product (Veeam, Druva, Backblaze) or a sync product (Dropbox, OneDrive). Its job is to take an arbitrary byte stream from a client, split it into chunks, and store only the chunks that have never been seen before — anywhere in the fleet. Typical workloads (VM backups, OS images, build artifacts, photo libraries) yield 5x – 30x dedup ratios; getting close to that ratio at exabyte scale without making the index a bottleneck is the whole problem.

2. Functional requirements

  • Ingest a byte stream and return a manifest (ordered list of chunk hashes).
  • Reconstruct the original byte stream from a manifest.
  • Reference-count chunks; safely garbage-collect chunks reachable by zero manifests.
  • Support both fixed-size and content-defined chunking; expose the choice per tenant.
  • Per-tenant or global dedup pool (configurable for security).

3. Non-functional requirements

  • Throughput: 10 GB/s ingest per region.
  • Dedup ratio: > 10x on typical backup workloads.
  • Index lookup: p99 < 5 ms per chunk.
  • Scale: 500 B unique chunks across the fleet, 5 EB physical storage.
  • Durability: 11-9s on chunks, 13-9s on the index (it's the root of trust).

4. Capacity estimation

  • Avg chunk size 8 KB (variable, content-defined). 5 EB / 8 KB = ~625 trillion chunks pre-dedup; with 10x dedup → 62 T unique.
  • Index row: 32 B hash + 8 B ref-count + 16 B location = 56 B; 62 T x 56 B = 3.5 PB index.
  • 10 GB/s ingest / 8 KB = 1.25 M chunk lookups/s sustained, 5 M peak.
  • Bloom filter to skip storage: 62 T entries x 10 bits = 78 TB filter (sharded).

5. API design

POST /v1/manifest/begin                             → {manifest_id}
POST /v1/chunks/exists  body=[hash...]               → bitmap
POST /v1/chunks/put     body=[{hash, bytes}]         → ok
POST /v1/manifest/append {manifest_id, hashes[]}     → ok
POST /v1/manifest/seal  {manifest_id}                → {root_hash}
GET  /v1/manifest/{root_hash}                        → ordered hash list
GET  /v1/chunks/{hash}                               → bytes

6. High-level architecture

[Client] ─chunker─► [Bloom Filter Tier] ─miss─► [Hash Index (sharded LSM)] │ │ │ if "exists"=false │ commit ref-count ▼ ▼ upload chunk ──► [Chunk Store (object/EC)] ◄── [GC: ref=0 sweeper] ▲ │ [Manifest Store] ──── decrements on delete

Walk-through: the client (or a server-side ingest worker) chunks the stream, computes BLAKE3 of each chunk, and asks /chunks/exists in batches of 256. The server first probes a sharded Bloom filter (in RAM); on a probable hit it consults the persistent hash index; on a definite miss the client uploads the chunk. Manifests are themselves chunked and dedup'd, so a backup that's 99% identical to yesterday's stores almost no new bytes.

7. Data model & storage

7.1 Hash index (sharded LSM, RocksDB-class)

key:    chunk_hash (32 B BLAKE3)
value:  { ref_count u32, location { extent_id u64, offset u32, len u32 }, created_ts u32 }
sharded by: first 2 bytes of hash (65,536 shards)

7.2 Chunk store (extent-packed)

Chunks are too small to store as individual objects (8 KB avg). Pack into 64 MB extents on EC 10+4 storage; the index points at (extent_id, offset, len).

7.3 Manifest store

Manifests are also dedup'd content (recursive Merkle tree). Root hash uniquely identifies a backup snapshot.

8. Detailed component design

8.1 Content-defined chunking (CDC)

FastCDC over a 64-byte rolling window with a min-chunk gate of 2 KB and max of 64 KB. The boundary depends only on local content, so an inserted byte at offset 0 only invalidates the chunk it lands in — every later chunk re-aligns. Fixed-size chunking, in contrast, would invalidate everything after the insert.

8.2 Bloom filter tier

10 bits/element, 1% false-positive rate. Sits in front of the LSM and absorbs ~99% of "this chunk doesn't exist" lookups without touching disk.

8.3 Reference counting & GC

Naive ref-counts have a race between "manifest commits" and "GC scans". Solution: epoch-based GC. Each manifest commit is tagged with epoch E; GC only reclaims chunks whose ref_count reached zero in epoch < E - 2. This avoids deleting a chunk that is being referenced by an in-flight manifest.

8.4 Per-tenant vs global dedup

Global dedup gives 10x – 30x ratios but lets a tenant prove what another tenant has stored (a privacy hole). Compromise: dedup within a "trust pool" (e.g., per enterprise customer). Hash-key salt is per-pool.

8.5 Hash collisions

BLAKE3 is 256-bit. Probability of one collision in 62 T chunks is ~2^-180 — negligible. We do not verify by full byte compare on dedup hits; SHA-256/BLAKE3 strong-collision resistance is the contract.

9. Scaling & bottlenecks

  • Index hot shards: a popular blob (e.g., a default OS install ISO) gets every backup hitting the same hash. Cache top-K hot keys in Redis.
  • Small-chunk overhead: 8 KB chunks → 56 B index entry → ~0.7% metadata overhead. Acceptable.
  • GC cost: sweeping 62 T entries is expensive; partition GC by hash prefix and run continuously.
  • Network round-trips: batch /chunks/exists at 256 hashes per RPC — collapses 256 RTTs into 1.

12. Talking points for the interviewer

  • "The single biggest lever for dedup ratio is content-defined chunking — fixed-size chunking with a 1-byte insert collapses to 1x."
  • "At 62 trillion unique chunks, the index is the system. I'd treat its design (Bloom + LSM + epoch GC) as the centerpiece, not the data plane."
  • "Global dedup is a privacy decision, not a performance one — I'd default to per-pool unless the customer explicitly opts in."
