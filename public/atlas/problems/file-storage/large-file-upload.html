Design Large File Upload (chunked, resumable, multipart, virus scan)

Hard file-storage

1. Problem statement

Design the upload pipeline behind products like WeTransfer, Slack file-share, GitHub LFS, or video-platform ingest. The system must accept files from 1 MB up to 5 TB over flaky residential and mobile networks, survive disconnects without restarting from byte 0, scan every byte for malware before the file becomes downloadable, and bill the customer accurately for storage and egress.

2. Functional requirements

  • Resumable uploads: client can pause / network can drop / browser can crash, then continue.
  • Multipart parallel uploads (8–32 concurrent parts) for throughput.
  • Pre-signed direct-to-storage upload URLs (do not proxy bytes through app servers).
  • Server-side virus scan + content-type sniffing before publishing the file.
  • Per-user upload quotas and abuse rate-limits.
  • Webhook on upload completion for downstream consumers.

3. Non-functional requirements

  • Throughput: saturate a 1 Gbps client link; sustain 5 GB/s aggregate per region.
  • Latency: first part accepted within 50 ms after handshake.
  • Scale: 50 K concurrent uploads, 5 PB ingested per day.
  • Durability: 11-9s once the upload is finalized.
  • Cost: abandoned uploads must be GC'd within 24 h.

4. Capacity estimation

  • Ingest: 5 PB/day = ~58 GB/s sustained; peak 4x = ~230 GB/s.
  • Parts: 8 MB part size, 5 PB / 8 MB = 625 M parts/day → ~7 K parts/s.
  • Init / complete control-plane RPS: 50 K uploads/min x 2 = ~1.7 K RPS.
  • Virus scanner CPU: 5 PB/day / 200 MB/s per worker = ~290 worker-days/day → ~290 always-on workers.
  • Abandonment: assume 12% of started uploads never complete → 600 TB/day GC.

5. API design

POST /v1/uploads                          {filename, size, mime}    → {upload_id, part_size, urls[]}
PUT  {presigned-storage-url}/?part=N      body=bytes                → ETag(part)
POST /v1/uploads/{id}/complete            {parts:[{n, etag}]}       → {file_id, status:"scanning"}
POST /v1/uploads/{id}/abort                                          → 204
GET  /v1/uploads/{id}/status                                         → {bytes_received, parts_received[]}

6. High-level architecture

[Client] ──init──► [Upload API] ──► [Upload State DB (Redis+Postgres)] │ │ │ pre-signed URLs│ ▼ ▼ [Object Store: staging bucket] ◄─── parts ──── [Client uploads in parallel] │ │ on /complete ▼ [Assembler] ──► [Virus Scanner pool] ──► [Final bucket (durable)] ──► [Webhook fanout] ▲ │ [GC: abandoned > 24h]

Walk-through: the client calls POST /uploads with file size; server creates an upload_id, picks a part size (8 MB for < 100 GB, 64 MB for larger), and returns a batch of presigned URLs. The client uploads parts directly to the staging bucket — no bytes traverse the app tier. Each PUT is idempotent on its part number, so retries are safe. /complete validates ETags, triggers the assembler, then scans, then publishes.

7. Data model & storage

7.1 Upload state (Postgres + Redis cache)

CREATE TABLE upload (
  upload_id   UUID PRIMARY KEY,
  user_id     BIGINT NOT NULL,
  filename    TEXT,
  size_bytes  BIGINT,
  part_size   INT,
  state       TEXT,    -- INIT | UPLOADING | ASSEMBLING | SCANNING | DONE | ABORTED
  parts_received JSONB,
  staging_key TEXT,
  final_key   TEXT,
  created_at  TIMESTAMPTZ,
  expires_at  TIMESTAMPTZ
);
CREATE INDEX ON upload (state, expires_at);  -- for GC sweep

7.2 Staging vs final bucket

Staging bucket has 24 h lifecycle policy; final bucket is durable. Two buckets so we never accidentally serve unscanned bytes.

8. Detailed component design

8.1 Resumability

The state of "what parts have arrived" lives both in the upload-state DB and is also derivable from ListParts on the staging bucket — that redundancy means a crashed app server cannot lose progress.

8.2 Idempotency

Every part PUT carries the part number in the URL. The same part can be uploaded twice with different bytes; the last write wins. Final commit verifies size + per-part hash sum against the manifest.

8.3 Virus scan pipeline

On /complete the file moves into a "scanning" state. A pool of ClamAV (or commercial) workers stream the object via range reads, scan in 64 MB windows, and on success rename it (server-side copy) into the final bucket. On detection, the file is quarantined and the user is notified. We never run the scanner in-line with the upload — that would force the client to wait for hundreds of MB/s of scan time.

8.4 Abuse / quotas

Token-bucket per user_id in Redis — caps bytes/hour and concurrent uploads. The POST /uploads handler also rejects if Content-Length exceeds the user's tier.

9. Scaling & bottlenecks

  • Tail-end stragglers: on 1,000-part uploads, one slow part stalls /complete. Solution: client speculatively re-uploads the slowest 2 parts in parallel.
  • Hot upload-state shard: one viral campaign can hammer one user's row. Cache hot rows in Redis with write-through.
  • Scanner backlog: autoscale on queue depth; SLO is "scanned within 5 min of complete".
  • GC correctness: sweeper must respect "still uploading" — uses Redis lease that the client refreshes every 60 s.

12. Talking points for the interviewer

  • "Two design choices win you 80% of the points: presigned direct uploads, and idempotent parts. Everything else is operational hygiene."
  • "I'd not block /complete on the AV scan — it pushes p99 latency through the roof. Instead I'd make 'shareable' a separate state."
  • "GC of abandoned uploads is where this system bleeds money in production; I'd put SLOs on staging-bucket size."

Every technology choice in this design exists for a concrete reason.
