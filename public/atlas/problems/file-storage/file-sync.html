Design Real-time File Sync (multi-device, conflict resolution)

Hard file-storage collaboration

1. Problem statement

Build the real-time sync engine behind products like Notion, Obsidian Sync, Apple Notes, or the file-tier of Google Workspace. Unlike a backup tool (which can tolerate minutes of lag) and unlike an object store (which has no notion of identity-per-device), this system must keep N devices for the same user — and M users sharing a folder — converged within a couple of seconds while users are actively editing, possibly offline, on flaky networks.

2. Functional requirements

  • Continuous sync of file create/modify/rename/delete across all devices and users with access.
  • Offline edits queue locally and replay on reconnect.
  • Deterministic, automatic conflict resolution for concurrent edits to the same file.
  • Per-folder ACLs; revocation must drop a device's access within seconds.
  • Bandwidth-efficient updates (delta sync) and battery-friendly mobile clients.

3. Non-functional requirements

  • Latency: change-event propagation p95 < 2 s within a region; p99 < 5 s globally.
  • Scale: 200 M users, 1 B linked devices, 10 B files, 50 B sync events/day.
  • Availability: 99.95% on metadata; degraded read-only mode below that.
  • Consistency: causal consistency per file; eventual convergence across all devices.
  • Battery: idle mobile client drains < 1% / hour while connected.

4. Capacity estimation

  • 50 B events/day = ~580 K events/s avg, 2.5 M/s peak.
  • WebSocket fan-out: 200 M users x 5 devices x 60% online = 600 M live connections.
  • Event payload ~500 B → 290 GB/s at peak fan-out (read-amplified).
  • Metadata DB: 10 B files x ~600 B/row = 6 TB hot, ~60 TB with versions.
  • Delta blocks: assume 30% of changes are partial → 8 PB/day delta upload.

6. High-level architecture

[Device A] [Device B] [Device C] │ │ │ └─────── WebSocket ───────┼──── WebSocket ───────────┘ ▼ [Sync Gateway tier] │ ▼ [Op Log per file_id (Kafka)] │ ┌──────────────────┼──────────────────┐ ▼ ▼ ▼ [Metadata KV] [Delta Block Store] [Notification fanout] │ ▼ [ACL service] ◄── [Audit log]

Walk-through: every device holds an open WebSocket to a sync gateway. Each file has a per-file op log (a Kafka partition keyed by file_id) that totally orders all commits. A device commits ops with the parent vector clock it last observed; the server appends, rebroadcasts the new vector + ops to every other connected device with access, and updates the metadata snapshot for late joiners.

7. Data model & storage

7.1 File metadata (sharded KV)

key: namespace_id || file_id
value: {
  name, parent_id, size, mtime,
  vector_clock: {device_id -> seq},
  current_blocklist: [hash...],
  acl_id, version_head_op_id
}

7.2 Per-file op log

Append-only sequence of {op_id, device_id, parent_vector, op_payload, ts}. Retain the last N ops in hot storage; older ops are compacted into snapshots. The op log is the source of truth — metadata KV is a materialized view.

7.3 Delta blocks

Same content-addressed block store as the Dropbox design — variable-chunked via FastCDC, EC 10+4, dedup'd globally per namespace.

8. Detailed component design

8.1 Conflict resolution model

We distinguish three classes: 1) Whole-file binary (PDF, DOCX): last-writer-wins on the op log; the loser's edit becomes a "Conflict copy" sibling. Same as Dropbox. 2) Plain text / Markdown: 3-way merge using the common ancestor recorded in the parent vector. Falls back to conflict copy on textual conflicts. 3) Structured doc (CRDT-aware product): ops are CRDT operations (Yjs-style), commute by construction, no conflicts possible. Per-file metadata declares which model applies.

8.2 Vector clocks & causality

Each device maintains {device_id → seq} per file. A commit carries its parent vector; the server rejects commits whose parent is > 16 ops behind the head (forces resync). This keeps memory usage bounded.

8.3 Notification & mobile

Foreground devices use WebSockets. Backgrounded mobile devices fall back to APNs/FCM "silent" pushes that wake the app for at most 30 s every 15 min — tuned to OS budgets.

8.4 Revocation latency

ACL changes write to a per-namespace revocation epoch. Every WebSocket frame carries the device's known epoch; sync gateway drops the connection when epochs diverge, forcing the client to re-auth and re-fetch the (possibly empty) tree.

9. Scaling & bottlenecks

  • Hot files (everyone editing one shared doc): the per-file Kafka partition is the bottleneck. Mitigate by batching ops every 50 ms and folding redundant ops on the gateway before append.
  • Reconnect storms after a regional blip: exponential backoff + jittered reconnect; gateway tier autoscales on connection rate.
  • Cold devices catching up: a laptop that's been off for a month may have 100 K ops to replay — server returns a snapshot + tail of recent ops instead.
  • Multi-region: Kafka MirrorMaker between regions; a single file is "homed" in one region, others read replicas with ~150 ms lag.

12. Talking points for the interviewer

  • "I'd model this as a per-file op log first; everything else (snapshots, conflict policy, mobile push) is layered on top."
  • "The interesting choice is conflict resolution: I'd ship LWW + conflict-copy on day one, then add CRDTs format-by-format because they're not free."
  • "Revocation latency is the security SLO most teams forget — I'd put a hard 5-second budget on it and design the epoch protocol around that number."
