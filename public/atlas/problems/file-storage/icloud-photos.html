Design iCloud Photos (cross-device sync, ML tagging, encryption)

Hard file-storage recommendation

1. Problem statement

Build a photo & video library that automatically syncs every shot from a user's iPhone, iPad, and Mac, while running on-device ML to tag faces, scenes, objects, and pets. The library must be searchable in natural language ("dog at beach 2023"), end-to-end encryptable, and storage-optimized so a 256 GB iPhone can act on a 2 TB library.

2. Functional requirements

  • Auto-upload originals from camera roll; download on demand on other devices.
  • "Optimize device storage": keep only thumbnails + low-res locally if disk is tight.
  • On-device face/scene/object recognition; sync the resulting tags across devices.
  • Natural-language search ("red car"); shared albums; Memories (auto-curated reels).
  • End-to-end encryption (Advanced Data Protection) where the server cannot read pixels.

3. Non-functional requirements

  • Latency: tag-sync between devices p95 < 30 s; thumbnail load p99 < 200 ms.
  • Scale: 850 M users, 4 trillion photos, 100 PB net new per month.
  • Durability: 11-9s on stored originals.
  • Privacy: all ML must run on-device; server holds only ciphertext & opaque tag tokens.
  • Battery: background indexing < 5% device battery/day.

4. Capacity estimation

  • Ingest: 850 M users x 30 photos/day x 3 MB = 76 PB/day uploaded.
  • Storage: 4 T photos x 3 MB + videos at 30 MB avg → ~14 EB net of dedup.
  • Tag volume: 30 tags/photo x 4 T = 120 T tag rows; encrypted ≤ 100 B each = 12 PB.
  • Thumbnail derivative storage: 3 sizes x 200 KB avg x 4 T = 2.4 EB.

5. API design

POST /v1/asset                {client_uuid, ciphertext, hash}    → asset_id
GET  /v1/asset/{id}?size=thumb|preview|orig                       → ciphertext
POST /v1/derivative           {asset_id, kind, ciphertext}        → ok
POST /v1/index/upsert         {tokens[], asset_ids[]}             → ok  (E2EE-tokenized)
GET  /v1/index/query          {token_query}                       → asset_ids[]
GET  /v1/changes              {since_cursor}                      → {changes, cursor}

6. High-level architecture

[iPhone Neural Engine] ──► tags/embeddings ──► encrypt locally │ │ ▼ ▼ HEIC original ─► upload ─► [Asset Service] ─► [Object store EC 12+4] │ ▼ [Metadata KV: per-user tree] │ ▼ [APNs / push] ◄── [Change-log Kafka] ──► [Search index (encrypted)] │ ▼ [iPad / Mac pulls deltas]

Walk-through: the iPhone runs Core ML over each new asset, extracts face/scene embeddings, encrypts them with the per-user CloudKit key, and uploads both the ciphertext original and the encrypted tag bundle. The server never sees pixels or tag plaintext under Advanced Data Protection — it only stores opaque blobs keyed by deterministic search tokens (hashed with the user's key). Other devices receive a push, fetch the change log, and decrypt locally.

7. Data model & storage

7.1 Asset metadata (CloudKit / FoundationDB-like KV)

row key: user_id || asset_id
value:   {
  capture_ts, device_id, hash_blake3,
  derivatives: {thumb_id, preview_id, orig_id},
  ciphertext_size, encryption_record,
  tombstoned: bool, change_seq: u64
}

7.2 Encrypted search index

Per user, an inverted index from HMAC(user_key, term) → list of asset_id ciphertexts. Server can answer "give me all asset_ids whose token bucket equals X" without ever knowing what X means in plaintext.

7.3 Originals

Stored as ciphertext objects in an S3-class object store, erasure-coded 12+4 across 4 AZs.

8. Detailed component design

8.1 On-device ML pipeline

Background task triggered when device is on Wi-Fi + charging + locked. Runs Vision framework: face detection (MTCNN-style) → 128-dim FaceNet embedding, scene classifier (1,000 classes), object detector. Embeddings are clustered locally to form "People" albums.

8.2 Tag sync across devices

Each device produces tags independently. Sync uses a CRDT-style OR-set per asset so tags merge without conflicts. Priority-based de-dup ensures the same face cluster identity propagates (the device with the lowest UUID wins ties).

8.3 Optimize device storage

When local free space drops below threshold, originals are evicted and replaced with previews. Re-fetched lazily on view; LRU + recency-of-shoot weighting.

8.4 Memories & recommendation

Memory pipeline runs entirely on-device: scoring fn over (face quality, scene rarity, recency, manual favorites) selects 8–12 photos per "Memory". Music selection is server-suggested by genre token (a non-PII signal).

9. Scaling & bottlenecks

  • Ingest spikes: Christmas/New Year ingest is 4x baseline; pre-warm regional capacity, throttle non-favorites first.
  • Cross-device tag drift: two devices may compute slightly different face clusters; resolve with a deterministic global salt + Hungarian matching.
  • Encrypted search recall: deterministic tokens leak frequency; add per-user dummy tokens to flatten the histogram.
  • Battery cost: ML jobs gated by Core ML scheduler; never run on cellular.

12. Talking points for the interviewer

  • "Pushing ML to the device is the strongest privacy lever; once you commit to it, the rest of the architecture follows because the server is now blind."
  • "The thorny bit is tag sync: I'd model it as a per-asset OR-set CRDT and define a deterministic face-cluster identity so devices converge."
  • "For encrypted search I'd accept the frequency side channel and pad it with dummy tokens — it's a classic privacy-vs-utility trade."

Every technology choice in this design exists for a concrete reason.

Apple Neural Engine + Core ML on-devicePrivacy NFR forbids server-side pixel access; with 76 PB/day uploaded across 850 M devices, server GPU inference would also cost a fortune.
