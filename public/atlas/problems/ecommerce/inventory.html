Design Inventory Management at Scale (oversell prevention, multi-warehouse)

Hard ecommerce database-ddia

1. Problem statement

Design an inventory system that tracks stock for 100M SKUs across 500 warehouses, exposes a single "available-to-promise" (ATP) number per SKU to the storefront, and prevents oversell during flash sales. It must reconcile with WMS (warehouse management) events, allow soft holds during checkout, and support multi-warehouse fulfilment optimization.

2. Functional requirements

  • Read ATP for any SKU at p99 < 30 ms.
  • Reserve N units (soft hold, TTL 10 min) for a cart.
  • Commit a reservation atomically with order creation.
  • Release reservation on cart abandon or checkout failure.
  • Adjust on receipts (PO arrivals), shipments, returns, cycle counts.
  • Multi-warehouse allocation: pick best-cost warehouse for an order.

3. Non-functional requirements

  • Correctness: zero oversell — physical stock >= committed reservations at all times.
  • Latency: read p99 < 30 ms; reservation p99 < 100 ms; ATP staleness < 1 s.
  • Scale: 100M SKUs * 500 warehouses = 50B (sku, wh) rows; peak 100k reservations/s.
  • Availability: 99.99%; degrades to "out of stock" rather than oversell on partition.

4. Capacity estimation

  • QPS: 1M ATP reads/s peak (Black Friday), 100k reservations/s, 30k commits/s.
  • Storage: 50B rows * 64 B = 3.2 TB hot; ledger of all events 30 TB/yr.
  • Hot keys: top 1000 SKUs see 60% of reservation traffic.
  • WMS event rate: 10M events/day from 500 warehouses.

5. API design

GET   /v1/inv/atp/{sku}?region=US        → {available, leadTimeDays}
POST  /v1/inv/reserve  {sku,qty,cartId}  → {reservationId, expiresAt}
POST  /v1/inv/commit   {reservationId, orderId, idemKey}
POST  /v1/inv/release  {reservationId}
POST  /v1/inv/adjust   {sku, wh, delta, reason, eventId}  (WMS only)

6. High-level architecture

[Storefront] [Cart] [Checkout] [WMS / Receiving] | | | | v v v v [Read API] [Write API] [Adjust API] | | | v v v [ATP Cache] [Reservation Svc] [Adjust Svc] (Redis) | | v v [Postgres shard "stock" table -- per-SKU row] | v [Kafka inv-events] | +-------------+-------------+ v v v [ATP Mat View] [Analytics] [Recon Svc]

Single source of truth: the per-(sku, warehouse) row in Postgres. Everything else is derived.

7. Data model & storage

7.1 Stock row (Postgres, sharded by sku_id hash)

CREATE TABLE stock (
  sku_id     bigint,
  wh_id      int,
  on_hand    int  NOT NULL,            -- physical count
  reserved   int  NOT NULL DEFAULT 0,  -- soft + hard holds
  committed  int  NOT NULL DEFAULT 0,  -- shipped not yet decremented
  version    bigint NOT NULL,
  PRIMARY KEY (sku_id, wh_id),
  CHECK (on_hand - reserved - committed >= 0)
);

7.2 Reservation ledger

reservations(resv_id PK, sku_id, wh_id, qty, cart_id, status,
             created_at, expires_at, idem_key UNIQUE);

7.3 Why a relational ledger?

We need atomic CAS on the stock row plus a durable record of who holds what. DDIA Ch.7 makes the case explicit: anything weaker than a serializable update on the count column allows the lost-update anomaly, which is precisely oversell.

8. Detailed component design

8.1 ATP read path

Most reads are cached in Redis (TTL 1 s, populated by a Kafka consumer reading inv-events). On cache miss, we fall through to a read replica. Storefront tolerates ~1 s staleness — overstating stock is OK at browse time; we re-check at reserve time.

8.2 Reservation write path

BEGIN;  -- SERIALIZABLE
SELECT on_hand, reserved, version FROM stock
  WHERE sku_id=$1 AND wh_id=$2 FOR UPDATE;
-- check on_hand - reserved - committed >= qty
UPDATE stock SET reserved = reserved + $qty, version = version+1
  WHERE sku_id=$1 AND wh_id=$2;
INSERT INTO reservations (...);
COMMIT;

Idempotency: idem_key unique on reservations; a duplicate insert returns the same resv_id without changing stock.

8.3 Multi-warehouse allocation

For an order with multiple SKUs, an allocator solves a min-cost flow problem: which warehouse fulfils which line, minimizing split-shipment cost + distance. Soft-holds across warehouses, then commit on success or release.

8.4 Reconciliation

Daily job compares physical counts (cycle count from WMS) with logical on_hand. Discrepancies emit corrective adjust events with reason="recon". Drift > 0.5% pages on-call.

8.5 Hot-key strategy

For the top-N SKUs we use bucketed counters: pre-split a SKU into 16 sub-rows; reservation picks a bucket (random or hash(cart_id)); ATP reads sum buckets. This linearly scales write throughput at the cost of read complexity.

9. Scaling & bottlenecks

  • Single-row contention: bucketed counters; co-located reservation worker per shard; batching.
  • Long checkout TTLs: short hold (10 min) + reserve-on-checkout-start (not on add-to-cart) reduces in-flight holds.
  • Cross-region writes: stock rows pinned to home region; a SKU's "owner region" is fixed.
  • Recovery from a botched WMS feed: ledger replay from Kafka with idempotent eventId dedup.

12. Talking points for the interviewer

  • "The check-then-act in inventory is the textbook lost-update; only serializable isolation or compare-and-set with a version column makes it safe."
  • "I distinguish soft holds (cart, Redis-only) from hard holds (checkout, persisted) so we don't pay for ledger writes on browse traffic."
  • "For hot SKUs, bucketed counters are pragmatic; the cost is read aggregation, but it preserves correctness while scaling writes."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Postgres SERIALIZABLE (stock row)Zero-oversell at 100k reservations/s requires the strongest isolation. SERIALIZABLE single-row update with a CHECK constraint (on_hand - reserved - committed >= 0) makes oversell impossible at the database level. DDIA Ch.7's lost-update is the textbook anomaly we're defeating.Snapshot isolation (write-skew oversells); Redis INCR (loses the consistency battle under partition); Spanner (latency overhead for in-region writes).

Each shard handles ~5k reservations/s, well below the Postgres write ceiling, and ATP reads use replicas of the same shard.Range partitioning (hot SKUs cluster on one shard); per-warehouse sharding (top 1000 SKUs cross all warehouses, no isolation).
Bucketed counters (top SKUs)Top 1000 SKUs absorb 60% of reservation traffic — single-row CAS becomes a serial bottleneck. Linear write scaling preserves zero-oversell.Pure single-row (chokes at 1000 reservations/s per hot SKU); Redis-only counter (no durability for financial-grade correctness).
Redis (ATP cache)1M ATP reads/s at p99 < 30 ms is impossible against Postgres. Redis hash, populated from Kafka inv-events with 1 s TTL, absorbs read traffic. Reservation ledger (durable)Hard holds at checkout need durability for audit and recovery. Min-cost flow allocatorMulti-warehouse allocation across 500 warehouses for a multi-line order is an optimization problem (split-shipment cost vs distance vs SLA). Min-cost flow on a small graph (lines × candidate warehouses) runs in < 50 ms and is provably optimal — heuristics ship 20% more split orders.Greedy heuristic (suboptimal splits); ILP solver (overkill latency for line-count < 20).
Postgres read replicasEven with Redis fronting, ATP cache misses fall through to read replicas. Replicas absorb leftover read load and serve as the recovery target if the cache is cold. Replication lag is tolerable because cache fronts the hot path.Single primary for reads (write traffic competes); CDC into Cassandra (more moving parts).
Reconciliation job (daily)WMS feeds drift; physical cycle counts diverge from logical on_hand. A daily diff job emits corrective adjust events with reason="recon"; drift > 0.5% pages on-call. Without this, the CHECK constraint will eventually fire on a real customer's reserve and show "out of stock" wrongly.Per-event reconciliation (too noisy); manual ops (doesn't scale to 500 warehouses).
Idempotency on adjust eventIdWMS retries are routine — receiving an event twice without dedup would double-count a 1000-unit pallet. Each adjust carries a UUID stored in a dedup table; replays during Kafka recovery are no-ops. Critical at 10M events/day where even a 0.001% double-application is 100 wrong rows.Best-effort delivery (corrupts inventory); upstream dedup only (we don't trust the WMS).

Each SKU's authoritative shard lives in one region; other regions read via async replication for browse but reserve through the home region.Multi-master write (oversell risk multiplies); Spanner global writes (latency too high for 100 ms reservation budget).
