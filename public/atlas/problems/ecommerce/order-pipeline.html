Design Order Pipeline / Order Lifecycle (state machine, saga, returns)

Hard ecommerce payments

1. Problem statement

Design the end-to-end order pipeline: from "Place Order" through payment, inventory commit, fulfilment, shipping, delivery, and return/refund. The system must coordinate 6+ downstream services (payments, inventory, fraud, fulfilment, carrier, notifications, accounting) without distributed transactions, recover cleanly from any step's failure, and provide a queryable, auditable order history.

2. Functional requirements

  • Create order, validate items + price + address, reserve inventory, charge payment.
  • Allocate to warehouse, generate picklist, ship, track, deliver.
  • Cancel before ship; return + refund after delivery.
  • Customer self-service: "Where's my order?", invoice download.
  • Accurate order state visible to customer, support, and analytics within seconds.

3. Non-functional requirements

  • Correctness: no orphaned charges, no orphaned shipments, no double refunds.
  • Throughput: 30k orders/s peak; 200k events/s on the bus.
  • Latency: place-order p99 < 800 ms (synchronous portion).
  • Durability: every state transition durable; recoverable from any single component failure.
  • Auditability: append-only history; replayable.

4. Capacity estimation

  • QPS: 30k order creates/s, 200k state-change events/s.
  • Storage: 1B orders/yr * 4 KB = 4 TB/yr hot; 40 TB with items + events.
  • Kafka: 200k * 1 KB = 200 MB/s ingress; retain 7 days = 120 TB.
  • Search index (orders): 1B docs, ~500 GB.

5. API design

POST /v1/orders               {cart, addr, payMethod, idemKey} → orderId, status
GET  /v1/orders/{id}                                          → order JSON
POST /v1/orders/{id}/cancel    {reason, idemKey}              → status
POST /v1/orders/{id}/return    {items, reason}                → returnId
POST /v1/internal/orders/{id}/transition  {event, payload}    -- saga internal

6. High-level architecture

[Checkout] --POST /orders--> [Order Svc] --insert--> [Postgres orders] | | v v [Outbox table] -----> [Kafka order-events] | +--------+---------+---------+----------+----------+----------+ v v v v v v v [Payment] [Inv] [Fraud] [Fulfil] [Notif] [Accounting] [Search] | | | | v v v v publishes publishes publishes publishes -> [Saga Orchestrator] -> updates [Order Svc]

Hybrid: a saga orchestrator listens to all service events and drives the order's state machine; services remain decoupled but the orchestrator owns the lifecycle invariants.

7. Data model & storage

7.1 Orders (sharded Postgres by user_id hash)

orders(order_id PK, user_id, status, total_cents, currency,
       created_at, updated_at, version);
order_items(order_id, sku, qty, unit_price, seller_id);
order_events(event_id PK, order_id, type, payload jsonb, ts);  -- append only
outbox(id PK, agg_id, type, payload, published_at);

7.3 State machine

CREATED -> PAYMENT_PENDING -> PAID -> ALLOCATED -> PICKED -> SHIPPED -> DELIVERED
                          \                                              \
                           -> PAYMENT_FAILED -> CANCELLED                 -> RETURN_REQUESTED -> RETURNED -> REFUNDED
   any -> CANCELLED (with compensating actions)

Every transition is recorded in order_events; current status is a derived materialization for fast reads. DDIA Ch.11's "log as system of record" applies.

8. Detailed component design

8.1 Place-order synchronous path

Order Svc validates, inserts the order row + outbox row in one local transaction. The outbox publisher tails the table and writes to Kafka with the order_id as the partition key. Total sync latency: ~150 ms.

8.2 Saga orchestrator

Reads order-events, looks up the order's current state, decides next command, and publishes to the responsible service. Implements timeouts: e.g., if PAYMENT_PENDING > 60 s, fire payment-status-poll; if SHIPPED > 14 days, mark stuck. Idempotency: every command carries a deterministic key derived from (order_id, transition).

8.3 Compensations

Failures emit compensating commands: cancel reservation, refund payment, regenerate invoice. Compensations are themselves idempotent — re-running them is safe.

8.4 Returns

Return is a sub-saga: RETURN_REQUESTED → RMA_ISSUED → ITEM_RECEIVED → INSPECTED → REFUND_INITIATED → REFUNDED. Refund uses the original payment idempotency key + a sub-key so the PSP doesn't double-credit.

8.5 Read paths

Customer order list reads from a denormalized "order summary" view in DynamoDB or Elasticsearch, populated from order-events via Kafka. The OLTP DB is shielded from query traffic.

8.6 Idempotency & exactly-once-effect

End-to-end idempotency key originates at /orders POST. The orchestrator stores (idemKey → orderId) so a retry returns the same order. Each downstream call carries a derived key. We get effective exactly-once via dedup at every boundary, not via 2PC.

9. Scaling & bottlenecks

  • Saga state hot rows: shard orchestrator state by order_id; multiple orchestrator instances with consistent-hash leases.
  • Kafka partition skew: order_id partition key gives even spread.
  • Stuck orders: a watchdog scans for orders without a transition in N hours and re-drives.
  • Reporting load: CDC into a column store (BigQuery/Snowflake); never query OLTP for analytics.

12. Talking points for the interviewer

  • "I never use 2PC across service boundaries; sagas with idempotent compensations are the industry standard for a reason."
  • "The transactional outbox is non-negotiable — it's how you avoid the dual-write problem between DB and Kafka."
  • "Returns are deceptively hard; the refund key has to be idempotent against the original charge so PSPs don't double-credit on retries."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Sharded Postgres (orders)30k orders/s with multi-row writes (orders + order_items + outbox) needs ACID. Sharding by user_id hash co-locates a customer's orders for fast list reads and confines a checkout transaction to one shard. Each shard handles ~3k orders/s with headroom.DynamoDB (no multi-row tx, joins painful for order_items); CockroachDB (cross-shard latency hurts the 800 ms place-order budget).
Transactional outboxThe dual-write problem (write DB then publish Kafka) loses events on crash. Outbox row in the same local Postgres tx + tail-publisher gives atomic-with-DB publish. Without it, at 200k events/s some orders silently never reach Inventory or Payments.Two-phase commit (no Kafka support, blocking); fire-and-forget publish (drops events on crash); CDC-only without outbox (publishes uncommitted writes).
Kafka (order-events)200k state-change events/s with 7-day retention (120 TB) feeds 6+ downstream services. Partition by order_id gives even spread and per-order ordering. 7-day retention enables saga replay after orchestrator failure without losing in-flight orders.RabbitMQ (no replay; can't rebuild materialized views); SQS (no per-key ordering); Pulsar (less mature ecosystem).
Saga orchestrator (vs choreography)Lifecycle invariants — "PAID requires inventory reserved AND payment captured" — need a single owner. Orchestration centralizes timeout policy (60 s payment timeout, 14 d shipped-stuck) and compensations. With 6+ services, choreographed sagas become an undebuggable mesh of "who reacts to what".Choreography (no central audit, hard to evolve); 2PC (PSPs and carriers don't support it).
Append-only event log (order_events)Auditability and replay are explicit non-functional requirements. Every transition stored as an immutable event; current status is a derived materialization. Replays reproduce any historical state, critical for support, fraud, and regulatory audit. DDIA Ch.11 "log as system of record".Mutable status column only (loses history); event sourcing without snapshot (slow current-state reads).
Idempotency-key storeEnd-to-end exactly-once-effect is the single most important correctness property — no double charges, no double shipments. Client-supplied idemKey at /orders POST stored as (idemKey → orderId); duplicate POSTs return the same orderId without side-effects. Each downstream call carries a derived deterministic key.Database unique constraints alone (works for inserts, harder for cross-step retries); best-effort retry (orphaned charges in production).
Consistent-hash leases (orchestrator)Hot saga state rows would serialize transitions if multiple orchestrator instances raced. Consistent-hash leases pin order_id ranges to one orchestrator instance with fencing tokens — no split-brain, no double transitions. Fencing tokens (DDIA Ch.9) protect against zombie instances after GC pauses.Single-instance orchestrator (SPOF, won't survive 30k orders/s); multi-master with locks (deadlock risk).
DynamoDB / ES (read views)Customer "where's my order" reads at scale would crush the OLTP shard. Denormalized order summary view in DDB (by user_id) or ES (text search) is populated from order-events via Kafka. OLTP is shielded; reads are sub-100 ms.OLTP read replicas only (still amplify load); query OLTP directly (kills checkout p99 during traffic spikes).
Watchdog (stuck orders)Sagas occasionally lose events (PSP timeout with unknown outcome, carrier silence). A watchdog scans for orders without a transition in N hours and re-drives. Idempotent compensations make re-driving safe; without this, < 0.01% of orders silently get stuck — at 1B orders/yr that's 100k stuck orders.Manual ops triage (doesn't scale); per-step retries only (won't catch silent gaps between steps).
BigQuery / Snowflake (analytics)1B orders/yr × 4 KB = 4 TB/yr; reporting queries (cohorts, fraud rings, finance close) are columnar workloads. CDC into a column store decouples analytics from OLTP completely. Querying OLTP for analytics has caused at least one outage at every e-commerce company.Read replica querying (locks, latency); per-shard rollups (fragile, slow joins).
State machine (explicit transitions)CREATED → PAYMENT_PENDING → PAID → ALLOCATED → PICKED → SHIPPED → DELIVERED is enforced in code; illegal transitions throw. Without an explicit FSM, services would bypass states (e.g., SHIPPED before PAID) under retry storms. The FSM is the contract every service signs.Implicit status flags (drift; spaghetti); workflow engine like Cadence/Temporal (great alternative, considered for v2 — adds runtime dep).
Refund sub-saga w/ derived idem-keyRefunds re-use the original payment's idempotency key plus a sub-key (paymentId|refundId) so PSPs don't double-credit on retry. Returns are a sub-saga RETURN_REQUESTED → RMA_ISSUED → ITEM_RECEIVED → INSPECTED → REFUND_INITIATED → REFUNDED, each step idempotent and resumable.Generate fresh refund key (PSP can't dedup against original charge); manual refund (doesn't scale to 1B orders/yr).
CDC (Debezium)Outbox tail-publisher uses Debezium-style CDC to stream Postgres WAL into Kafka. Catches every committed write, never reads uncommitted. At 200k events/s the alternative (polling outbox table) is too slow and creates write amplification on the OLTP shard.Polling publisher (lag and table bloat); application-level publish (loses atomicity guarantee).
