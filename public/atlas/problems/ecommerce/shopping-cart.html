Design Shopping Cart (multi-device, persistence, abandoned-cart)

Hard ecommerce storage-cache

1. Problem statement

Design a shopping cart service for a global e-commerce site: 200M users, multiple devices per user, guest carts that merge on login, durable persistence across sessions, and an abandoned-cart pipeline that nudges buyers via email/push within minutes of inactivity. The cart is read 10x more than it's written and must feel instant on mobile.

2. Functional requirements

  • Add, update qty, remove items; apply coupons; estimate totals.
  • Guest cart (cookie-based) merges with user cart on login.
  • Multi-device sync: change on phone visible on desktop within 1 s.
  • Survive across sessions for 90 days.
  • Abandoned-cart detection and notification (15 min, 24 h, 72 h triggers).
  • Real-time price/stock refresh when cart is opened.

3. Non-functional requirements

  • Latency: read p99 < 50 ms, write p99 < 100 ms.
  • Scale: 200M MAU, 30M DAU, 500k cart writes/s peak, 5M reads/s peak.
  • Durability: never lose an item due to server failure.
  • Availability: 99.99% — cart unavailability tanks conversion.

4. Capacity estimation

  • QPS: 5M reads/s (cart preview on every page nav), 500k writes/s peak.
  • Storage: 200M users * 2 KB avg cart = 400 GB hot; 90-day TTL ⇒ ~5 TB warm.
  • Abandonment job: 30M DAU * 60% abandon = 18M nudges/day.
  • Network: 100 Gbps internal at peak.

5. API design

GET    /v1/cart                  → {items, totals, coupons}
POST   /v1/cart/items   {sku, qty}        → cart
PATCH  /v1/cart/items/{sku}  {qty}        → cart
DELETE /v1/cart/items/{sku}              → cart
POST   /v1/cart/merge   {guestCartId}     → cart
POST   /v1/cart/coupon  {code}            → cart
WS     /v1/cart/subscribe                 → push updates

6. High-level architecture

[Phone] [Desktop] [Tablet] | | | +----+----+----+-----+ v v [API GW] [WS GW] -- multi-device push | | v v [Cart Svc] -------- [Pub/Sub: cart-changed] | +----+----+ | | [Redis] [DynamoDB] [Pricing Svc] [Stock Svc] (cache) (durable) | v [Kafka cart-events] -> [Abandon Detector] -> [Notif] -> [Analytics]

Cart svc is stateless. DynamoDB is the durable store; Redis is a write-through cache. Pub/Sub broadcasts to other open sessions of the same user.

7. Data model & storage

7.1 DynamoDB cart

PK = userId   (or guestCartId for guests)
SK = "META"   → {coupon, currency, updatedAt, version}
SK = "ITEM#{sku}"  → {qty, addedAt, priceSnapshot}
TTL on guest carts: 7 days; user carts: 90 days.

7.2 Redis cache

cart:{userId} HASH  -- mirror of DDB items
TTL 1h, refreshed on read; write-through on update.

7.3 Why DynamoDB?

Single-key access pattern (always by userId), predictable p99, native TTL for guest cart cleanup, and global tables if we go multi-region. We don't need joins; a relational DB would be over-engineering.

8. Detailed component design

8.1 Cart writes

Optimistic concurrency: client sends If-Match: version=N. Server does a conditional write on DynamoDB version attribute. On conflict, return 409 with the latest cart so the client merges and retries. Total latency budget: 1 round trip to DDB + cache fill.

8.2 Multi-device sync

On every successful write, publish {userId, version} to Pub/Sub. WS gateway holds a per-user channel; clients connected on other devices receive an "invalidate" signal and re-fetch. Bandwidth-efficient (no item payload over WS).

8.3 Guest-to-user merge

On login, server reads guest cart by cookie ID and user cart, then merges by SKU (sum qty, capped by ATP). Conflict resolution: prefer most recent addedAt for non-numeric fields. Atomic via a DDB transaction.

8.4 Abandoned cart pipeline

Each cart-event in Kafka updates a per-user "last activity" timestamp in Redis. A scheduled job (every 60 s) scans for users idle > 15 min with non-empty carts; emits a notification job. Cooldowns prevent spamming. ML model selects channel and copy.

8.5 Price/stock refresh

On cart open, async fan-out to Pricing & Stock services for each SKU; cart svc updates price/availability in cache and notifies the client over WS if anything changed (e.g., "price dropped" badge).

9. Scaling & bottlenecks

  • Hot users (test accounts, scrapers): per-key write rate limiter; backoff response.
  • Cache stampede on viral SKU: request coalescing in Redis (single-flight); negative caching of "no cart".
  • WS connection storms: sticky-by-user routing + connection limits; fall back to long-poll.
  • Cross-region carts: DDB global tables; conflict resolved by max(version) + LWW per item.

12. Talking points for the interviewer

  • "I treat the cart as a single-key document with optimistic concurrency — it's the simplest correct model and scales linearly."
  • "The trickiest UX is multi-device sync; I push invalidation, not data, to keep the WS layer cheap."
  • "Abandoned-cart is a streaming pipeline, not a cron over a DB scan; it's how you stay real-time and avoid hot-spotting your OLTP store."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
DynamoDB (durable cart)Cart is a single-key document accessed by userId — exactly DynamoDB's sweet spot. Predictable single-digit ms p99, native TTL for guest cart cleanup (7 days) and user cart (90 days), and Global Tables for multi-region. 200M users × 2 KB = 400 GB hot fits comfortably without re-sharding.Postgres (no joins needed, would be over-engineering at 500k writes/s); Cassandra (more ops overhead for the same shape).
Redis (write-through cache)5M cart reads/s on every page nav cannot hit DynamoDB directly without breaking budget — Redis at sub-ms p99 absorbs this. Write-through to DDB on every update keeps cache and durable store consistent. TTL 1h auto-expires cold carts.Memcached (no atomic ops for cart-version CAS); local in-process cache (no cross-pod consistency).
Optimistic concurrency (version)Multi-device sync on the same userId can produce concurrent writes. Conditional writes on the DDB version attribute with If-Match: version=N let us avoid distributed locks; on 409 the client merges and retries. Scales to 500k writes/s without locks.Pessimistic locks (kills throughput, adds latency); LWW (loses items on race).
WebSockets (multi-device sync)1-second freshness across phone/desktop/tablet requires push, not poll — polling at 5M MAU would melt the API. WS gateway pushes only an "invalidate {userId, version}" signal (not the cart payload), keeping bandwidth tiny while devices re-fetch fresh state from the cache.Long-poll (extra round trips, higher tail latency); SSE (one-way is fine but WS already supports both directions for typing/etc); push payloads (10x bandwidth cost).
Pub/Sub (per-user channel)The WS gateway is horizontally sharded; a write on Pod A must reach a client connected to Pod B. A Pub/Sub topic per userId fans the invalidate signal to whichever gateway holds the user's WS. Stateless gateways scale linearly with users.Sticky-by-user load balancer only (rebalances drop connections); direct service-to-service (full mesh, fragile).
Kafka (cart-events)The abandoned-cart pipeline needs every cart change as a stream so we can update last-activity in < 1 s and trigger 15-min/24-h/72-h nudges. 18M nudges/day at 60% abandonment requires partition-by-userId so per-user ordering is preserved through the abandon detector.RabbitMQ (no replay for retroactive analytics); cron over DDB scan (kills OLTP, latency hours).
DynamoDB Transactions (merge)Guest-to-user merge on login must atomically read both carts, merge by SKU, write the union, and delete the guest cart. DDB transactions give multi-item atomicity in one call without a distributed coordinator; idempotent on retry by transaction-token.Two-phase app-level merge (race window leaves orphan items); Saga (overkill for a single user-action).
API Gateway + token bucketHot users (test accounts, scrapers, bots) can blow through 500k writes/s budget. Per-key write rate limit at API GW (e.g., 20 writes/s per user) returns 429 cheaply before pods waste DDB capacity. WAF blocks crude bot floods upstream.App-layer rate limit (already burned a hop); no rate limit (hot key burns shard capacity).
DynamoDB TTLGuest carts (7 days) and user carts (90 days) auto-expire via DDB native TTL — no scheduled job, no manual cleanup. At 200M users, a sweep job would itself be a SPOF. TTL is async but fine because expiry is a soft contract.Custom cleanup cron (operational burden); never expire (storage cost grows linearly).
DDB Global Tables (multi-region)Cross-region carts (user shops on US laptop then opens cart on EU phone) need geo-replicated state. Global Tables replicate within seconds; conflict resolved by max(version) + LWW per item, acceptable for cart UX. Cart is not financial-grade so LWW is fine here.Single-region with cross-region read (high latency); manual sync via Kafka (bug-prone).
Pricing/Stock svc fan-outOn cart open, we fan out async per-SKU to Pricing + Stock so the cart-open response is fast (cached cart) while a "price dropped" badge updates over WS. Avoids the blocking N+1 lookup that would push read p99 past 50 ms.Synchronous pricing on cart-read (kills p99); precomputed prices in cart (stale, must invalidate on every price change).
Single-flight in RedisCache stampede on a viral SKU (everyone clicks the deal at once) would multiplex thousands of concurrent DDB reads through one cold Redis key. Single-flight (one fetch, others wait) and negative caching of "no cart" cap origin reads at one per key per refresh.Naive per-request fetch (DDB throttles); pre-warming (we don't know what's viral in advance).
