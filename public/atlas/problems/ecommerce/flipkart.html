Design Flipkart (Big Billion Day flash-sale handling)

Very Hard ecommerce payments

1. Problem statement

Design Flipkart with the central design constraint of surviving Big Billion Day (BBD): traffic spikes 10–20x normal, hot SKUs (a Rs.10k phone limited to 50k units) are reserved within seconds, and the payment pipeline must stay correct under partial UPI/PSP outages. The system must protect against oversell, payment double-charge, and bot abuse while keeping good customers in the funnel.

2. Functional requirements

  • Browse, search, PDP — same as a normal e-comm site.
  • Flash-sale mode: server-side rate-limited add-to-cart, queue-based checkout.
  • Coupons, EMI, Pay Later, COD, UPI, cards — all must idempotently charge.
  • Order tracking, returns, replacements, marketplace seller flow.
  • Bot/scalper detection on hot inventory.

3. Non-functional requirements

  • Latency: PDP p99 < 250 ms; ATC p99 < 400 ms even at peak.
  • Scale: 350M users, BBD peak 1.5M concurrent users, 200k ATC/s, 30k orders/s.
  • Availability: 99.99% browsing; checkout degrades gracefully (queue) before failing.
  • Correctness: 0 oversell, 0 double-charge, <1 ppm duplicate orders.

4. Capacity estimation

  • QPS: peak 800k catalog reads/s, 200k ATC/s, 30k orders/s, 60k payment calls/s.
  • Storage: 200M SKUs, 800M orders/yr, ~3 PB cumulative orders + events.
  • Bandwidth: 400 Gbps CDN egress at peak; ~10 Gbps API traffic.
  • Inventory hot-keys: 100 SKUs receive 50% of all reservation traffic.

5. API design

POST /v1/sale/queue/enter   {userId, saleId}      → token, position
POST /v1/cart/reserve       {sku, qty, token}     → reservationId, ttl
POST /v1/checkout/init      {reservationId}       → checkoutId
POST /v1/payment/charge     {checkoutId, method, idemKey} → paymentId
POST /v1/order/confirm      {paymentId}           → orderId

6. High-level architecture

[Mobile/Web] | [Akamai CDN] -- static assets, PDP HTML | [Edge WAF + Bot Mgmt] | [API GW] -- token-bucket per user (10 req/s) | [Sale Queue Svc] --> [Redis Streams] (virtual waiting room) | [Cart/Reserve Svc] --> [Inventory Svc] --> [Postgres "stock" row, FOR UPDATE] | \ v \--> [Kafka inv-events] [Checkout Svc] --> [Payment Orchestrator] | +---------+---------+--------+ | | | | [UPI] [Cards] [PayLtr] [COD] | v [Saga Coordinator] --> [Order Svc] --> [Sharded Postgres] \ --> [Notif] [Fulfilment] [Refund]

The waiting-room pattern is the headline: clients can't even reach checkout without a queue token, so backend pressure is bounded.

7. Data model & storage

7.1 Inventory (Postgres + Redis cache)

CREATE TABLE stock (
  sku_id  bigint PRIMARY KEY,
  available int NOT NULL CHECK (available >= 0),
  reserved  int NOT NULL DEFAULT 0,
  version   bigint NOT NULL
);

Single-row decrement under SERIALIZABLE or SELECT ... FOR UPDATE; Redis hash mirrors counts for fast reads only — never authoritative.

7.2 Reservations (Redis with TTL)

HSET resv:{id} sku 9988 qty 1 user 42 exp 1715000000  EX 600

7.3 Orders (sharded by userId hash)

orders, order_items, payments — same shard so checkout is local-tx.

8. Detailed component design

8.1 Virtual waiting room

Before sale start, users park in a queue keyed by (saleId, joinTs). Tokens are admitted at a controlled rate (e.g., 5k/sec) matching downstream capacity. Token JWT carries an expiry and an admit-by timestamp.

8.2 Inventory service

Authoritative single-row decrement is the chokepoint. We shard by sku_id across N Postgres primaries; hot SKUs get their own shard. The decrement is wrapped in a serializable transaction (DDIA Ch.7) — anything weaker risks oversell.

8.3 Payment orchestrator

State machine: INIT → CHARGING → CAPTURED | FAILED → REFUNDING. Every PSP call carries an idempotency key derived from checkoutId. On PSP timeout we poll status before retry to avoid double-charge.

8.4 Bot defense

Edge fingerprinting + per-account velocity limits + CAPTCHA on second attempt. Suspicious tokens are silently shadow-queued.

9. Scaling & bottlenecks

  • Hot SKU row: dedicated shard; co-located reservation worker pinned to that primary; batch-decrement (group N requests, one tx).
  • PSP outage: circuit-breaker per PSP, automatic failover (Razorpay → BillDesk → bank direct).
  • Thundering herd at sale start: randomized client-side jitter + queue admission.
  • Read amplification on PDP: price/stock pulled separately so price page is cacheable; only the small "stock pill" hits origin.

12. Talking points for the interviewer

  • "My single biggest correctness risk is oversell on a hot SKU; I'd rather slow the queue than relax isolation."
  • "Payments must be idempotent end-to-end; the idempotency key is generated client-side at checkout init and survives retries."
  • "I'd ship the waiting room first; it converts a reliability problem into a UX problem, which is far easier to solve incrementally."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Akamai CDNBBD pushes 400 Gbps of static and PDP HTML egress. Akamai's India POPs cut TTFB for 350M domestic users; surrogate-key purges let us invalidate just the price/stock fragment without flushing full PDP HTML, preserving the 800k catalog reads/s cache hit rate.CloudFront (weaker India presence historically); self-hosted Varnish (cannot match 400 Gbps without massive ops).
Redis Streams (waiting room)The virtual waiting room admits 5k tokens/sec from a 1.5M concurrent backlog at sale start. Redis Streams give us O(1) append + ordered consumer groups; tokens with admit-by timestamps survive across cells. Without this, 200k ATC/s would crush downstream Postgres.Kafka (heavier; consumer-group rebalances are painful for short-lived sale flows); SQS FIFO (throughput too low for sale-start spikes).
Postgres SERIALIZABLE (inventory)The Rs.10k phone with 50k units sees 100 SKUs taking 50% of reservation traffic. Serializable single-row decrement with FOR UPDATE is the only isolation level that provably prevents the lost-update anomaly that causes oversell. Anything weaker can sell a phantom 51,000th unit.Snapshot isolation (write-skew oversells); Redis INCR (loses correctness under partition); Spanner (latency too high in India region).
Redis (soft holds + ATP cache)HSET resv:{id} with EX 600 gives 10-minute soft holds at sub-ms latency, freeing the Postgres primary for authoritative decrements only. Mirrors stock counts for storefront reads at 800k/s — never authoritative, always reconciled against Postgres on reserve.Memcached (no TTL on hash fields, no eviction policy fit); in-process cache (no cross-instance view of holds).
Kafka (order/inv events)30k orders/s and 60k payment calls/s feed the saga, refund pipeline, and analytics. Partition by sku_id for inventory and by user_id for orders; 7-day retention enables saga replay after a coordinator failure without losing in-flight BBD orders.RabbitMQ (no log replay); Pulsar (less Indian operational expertise on team).

Orchestration centralizes timeout policy (poll PSP before retry) and compensations, preventing the <1 ppm duplicate-order budget from being blown.Choreography (no single owner of timeouts); 2PC (PSPs don't support it).
Multi-PSP (Razorpay/BillDesk)Indian UPI/PSP outages happen during BBD (NPCI, bank gateways flap). Circuit breaker per PSP with auto-failover keeps checkout 99.99% available even when one PSP is down. Idempotency key per checkoutId stops double-charge across the failover path.Single PSP (single point of failure during sale); direct bank integrations only (slower onboarding, weaker fraud).

Suspicious tokens shadow-queued so attackers don't learn they're flagged.Captcha-everywhere (tanks legit conversion); app-layer-only (already too late, queue token spent).
API Gateway + token bucket10 req/s per user at the gateway is the first line of defense before the 200k ATC/s pressure point. JWT-bound buckets let abusive users get a 429 without burning Postgres connection slots. Cheap, correct, fast.NGINX rate limit (works, but token-bucket on logical user-id is cleaner via API GW); in-app rate limit (too late).
Sharded Postgres (orders)Sharding by userId hash co-locates orders, order_items, and payments on one shard so checkout becomes a local transaction — no 2PC needed for a 30k orders/s peak. Each shard handles ~3k orders/s, well below Postgres ceiling.CockroachDB (cross-region tax); single Postgres (won't survive 30k orders/s peak).

Perfect for 1.5M concurrent users where stateful token store would cost millions.Opaque tokens with Redis lookup (extra hop on every request); cookie-only (replayable across users).
