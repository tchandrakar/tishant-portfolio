Design Amazon (product catalog + checkout + recs at scale)

Very Hard ecommerce search recommendation payments

1. Problem statement

Design a planet-scale e-commerce platform like Amazon: 600M+ SKUs, 300M+ active customers, multi-region marketplaces, full-text product search, personalized recommendations, a transactional cart/checkout flow, and a payments pipeline that survives Black Friday traffic. Reads dominate (~95%) but the writes (orders, payments, inventory) must be ACID-correct.

2. Functional requirements

  • Browse and search the catalog by keyword, category, brand, faceted filters.
  • Render product detail pages (PDP) with price, inventory, reviews, recs.
  • Add-to-cart, checkout, apply promotions/coupons, charge payment, place order.
  • Personalized recommendations on home page, PDP, and post-purchase.
  • Order tracking, returns, refunds, seller marketplace, Prime entitlements.

3. Non-functional requirements

  • Latency: PDP p99 < 200 ms; search p99 < 300 ms; checkout p99 < 800 ms.
  • Scale: 300M MAU, peak 600k RPS catalog reads, 50k orders/sec on Prime Day.
  • Availability: 99.99% for browse; 99.995% for checkout/payments.
  • Durability: 11 nines for orders & payments (no lost transactions).
  • Consistency: strong for inventory decrement & payments; eventual for reviews/recs.

4. Capacity estimation

  • QPS: 600k catalog reads/s peak, 8k cart writes/s, 50k orders/s peak (Prime Day).
  • Storage: 600M SKUs * 4 KB metadata = ~2.4 TB hot; reviews ~50 TB; orders ~5 PB lifetime.
  • Bandwidth: 200 Gbps egress for images/CSS/JS via CDN; ~5 Gbps for JSON APIs.
  • Cache: 95% PDP hit rate at edge → origin needs ~30k RPS only.

6. High-level architecture

[Client] -> [CloudFront CDN] -> [API Gateway] -> [BFF / GraphQL] | +--------------+--------------+----------------+----------------+ | | | | | [Catalog] [Search] [Cart Svc] [Checkout Svc] [Reco Svc] | (ES/Solr) | | | [DynamoDB] [Index] [Redis+DDB] [Order Saga] [Feature Store] | [Kafka bus] | +--------+---------+--------+---------+ | | | | | [Payments][Inv Svc][Fulfil] [Notif] [Analytics] | | [PSPs] [Postgres + Redis lock]

Walk-through: client hits the edge CDN; cacheable PDP HTML/JSON is served from POPs. Dynamic calls fan out from a BFF to micro-services. Orders flow as a saga over Kafka so failures roll back cleanly.

7. Data model & storage

7.1 Catalog (DynamoDB)

PK = asin (String)
SK = "META" | "PRICE#region" | "OFFER#sellerId"
attrs: title, brandId, categoryPath, dims, images[], gsi_brand_id

7.2 Orders (sharded Postgres + S3 cold tier)

orders(order_id PK, user_id, status, total_cents, created_at)
order_items(order_id, asin, qty, unit_price_cents, seller_id)

7.3 Why this split?

Catalog is read-heavy, key-value, eventually consistent → DynamoDB with global tables. Orders need ACID multi-row writes (item, payment, inventory) → relational. Search needs inverted indexes → Elasticsearch. Recs need vectors → feature store + ANN.

8. Detailed component design

8.1 Search

Catalog updates publish to Kafka; an indexer denormalizes into Elasticsearch (sharded by category hash, replica factor 3). Query service rewrites with synonyms, runs BM25 + a learned-ranker re-rank step on the top 200. Facets are bitmap-cached.

8.2 Cart

Authoritative copy in DynamoDB keyed by userId; Redis write-through cache (TTL 7d) for fast reads. Guest carts use a signed cookie cart-id and merge on login.

8.3 Checkout saga

Choreographed saga: ReserveInventory → ChargePayment → CreateShipment → ConfirmOrder. Each step emits events; compensating events (release-inventory, refund) handle failure. Idempotency keys on every step.

8.4 Recommendations

Two-stage: candidate generation via ANN over user/item embeddings (FAISS), then a GBDT ranker scoring CTR * margin. Online features in Redis, offline in S3+EMR.

9. Scaling & bottlenecks

  • Hot SKUs (iPhone launch): request coalescing at edge + per-key rate limit + pre-warmed shards.
  • Inventory contention: single-item reservation row protected by DDIA Ch.7 transactional patterns; fall back to Redis Redlock for soft holds.
  • Search shard hotness: route by (categoryHash, asin) so popular categories spread across shards.
  • Checkout spikes: queue-based load leveling — orders enqueued to Kafka and processed at a stable rate downstream.

  • Ch. 7 Transactions — serializable inventory decrement to prevent oversell.

12. Talking points for the interviewer

  • "I split the system by read/write profile: catalog/search optimized for reads at the edge; checkout/orders optimized for correctness via a saga."
  • "The hardest part is preventing oversell on hot items during a flash sale; my mitigation is a single-row authoritative reservation with idempotency keys, not Redlock alone."
  • "I would start single-region and add cross-region active-active for catalog only; orders stay home-region with async replication for DR."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
CloudFront CDNWith 200 Gbps egress and 600k catalog reads/s peak, edge caching delivers a 95% hit rate so origin sees only ~30k RPS. Surrogate keys per ASIN allow targeted purges when price/stock changes, keeping PDP p99 under 200 ms globally.Self-hosted Varnish (operationally heavy at this scale); Cloudflare (fine, but AWS-native integration with S3/Lambda@Edge wins).
DynamoDB (catalog)600M SKUs at 4 KB each is a key-value workload with predictable single-digit ms p99. Global tables replicate across regions for 300M MAU. ASIN PK + SK pattern (META/PRICE/OFFER) avoids joins entirely and scales horizontally without re-sharding.Cassandra (similar shape, more ops overhead); MongoDB (weaker p99 tail at this scale).
Elasticsearch (search)Inverted-index BM25 with faceted filters over 600M SKUs needs sub-300 ms p99 search. Sharded by category hash with replica factor 3 spreads hot categories. Bitmap-cached facets and learned re-rank on top 200 are first-class citizens.Solr (similar but smaller community); Postgres FTS (no learned ranking, weak faceting at this scale).
Sharded Postgres (orders)Orders need ACID multi-row writes (order + items + payment) with 11 nines durability. Relational integrity prevents orphaned charges; serializable isolation prevents oversell at 50k orders/s on Prime Day. Hash-by-user_id keeps a checkout local to one shard.DynamoDB (no multi-row tx); CockroachDB (cross-region latency hurts checkout p99).
Redis (cart + features)Cart needs read p99 < 50 ms across 8k cart writes/s with write-through to DynamoDB. Also stores online recommendation features for sub-10 ms feature lookup during the GBDT ranker step. Redlock is used only for soft holds, never authoritative.Memcached (no data structures, no persistence); local in-process cache (no cross-instance coherence).
Kafka (event bus)The order saga's ReserveInventory → ChargePayment → CreateShipment → ConfirmOrder steps need durable, ordered, partitioned events. Partition by order_id gives even spread; 7-day retention enables saga replay after orchestrator failure without losing in-flight orders.RabbitMQ (no log replay, weaker throughput); SQS (no ordering per partition).
FAISS (recs ANN)Two-stage recommender candidate generation needs k-NN over hundreds of millions of user/item embedding vectors at sub-50 ms. FAISS IVF+PQ on a GPU box recalls 1000 candidates fast enough that a downstream GBDT can score in the latency budget.Annoy (slower at billion-scale); Elasticsearch dense_vector (much higher tail latency).
S3 + EMR (offline ML)Reviews data (~50 TB) and offline feature engineering for recs run on cheap object storage with Spark/EMR. Decouples slow batch training from the online serving path; online features go to Redis, offline embeddings to FAISS index.HDFS (operational tax); Snowflake (great for BI, expensive for raw ML feature pipelines).
Saga (choreographed)2PC across Payments, Inventory, Fulfilment is impractical at 50k orders/s. Choreographed saga with compensating events (release-inventory, refund) tolerates partial failures; idempotency keys per step give exactly-once-effect without locking distributed resources.2PC (blocks under failure); orchestrated saga (we'd choose this if lifecycle ownership mattered more than coupling).
API Gateway + BFF (GraphQL)A PDP needs catalog + price + reviews + recs in < 200 ms. The BFF parallelizes fan-out to micro-services and returns one payload. Per-route token buckets enforce 50k orders/s ceiling and protect downstream from thundering herds during launches.REST aggregation in client (chatty over mobile); direct service-to-service from client (auth/security nightmare).
Postgres + Redis lock (inventory)Hot SKUs (iPhone launch) need a single authoritative reservation row with serializable isolation to prevent oversell. Redis Redlock is layered on top for soft holds only; the relational row is always the source of truth — never trade correctness for cache speed on stock.Pure Redis counters (lost-update under partition); DynamoDB conditional writes (works, but Postgres SERIALIZABLE is simpler reasoning here).
WAF + bot management at edgeHot-product launches attract scalpers who can drain inventory in seconds. Edge rate limiting (per-IP, per-account) and fingerprinting drop bots before they hit the API GW, protecting the 50k orders/s budget for real customers.App-layer rate limit only (too late, origin already burned); CAPTCHA-everywhere (kills conversion for legitimate users).
