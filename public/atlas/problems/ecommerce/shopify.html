Design Shopify (multi-tenant SaaS for 1M+ merchants)

Very Hard ecommerce payments

1. Problem statement

Design a multi-tenant e-commerce platform that hosts 2M+ merchants ranging from a single-product store to enterprises selling 1M units/day. Each merchant gets their own storefront, theme, products, checkout, and admin, but the underlying infrastructure is shared. Tenants must be isolated for performance and blast-radius, yet operationally the platform is one fleet.

2. Functional requirements

  • Storefront rendering with themes (Liquid templates), custom domains, SEO.
  • Admin panel: products, inventory, orders, customers, discounts, apps.
  • Multi-currency, multi-region checkout with Shopify Payments + 3rd-party PSPs.
  • App ecosystem (webhooks, GraphQL Admin API, embedded apps via OAuth).
  • Black Friday Cyber Monday (BFCM) survival across all merchants simultaneously.

3. Non-functional requirements

  • Latency: storefront p99 < 300 ms TTFB; admin API p99 < 500 ms.
  • Scale: 2M merchants, 50M MAU buyers, BFCM peak 50k checkouts/min globally.
  • Tenant isolation: a noisy-neighbor merchant cannot starve others.
  • Availability: 99.99%; checkout 99.995% (merchant revenue depends on it).

4. Capacity estimation

  • QPS: 700k storefront reads/s peak globally, 80k admin API/s, 5k checkouts/s.
  • Storage: 2M merchants * 1 GB avg = 2 PB; orders ~10 PB lifetime; assets ~50 PB on object store.
  • Bandwidth: 1 Tbps CDN egress at BFCM peak.
  • Webhook fan-out: 2B events/day to apps.

6. High-level architecture

[Buyer] [Merchant Admin] [3rd-party App] | | | [Fastly CDN] -- per-shop key cache | | | [Shop Router] -- maps shop.com -> pod-id | | | v v v +---- Pod (cellular) ------- Pod ------- Pod -------+ | [Storefront] [Admin API] [Checkout] [Jobs] | | [MySQL primary + replicas] [Redis] [Memcache] | | [Object store refs] [Search ES] | +---------------------------------------------------+ | v [Kafka platform bus] -> [Webhooks] [Reporting] [Fraud] [Analytics]

Cellular architecture (a la "pods"): each pod hosts ~100k shops with its own DB and stateful services. Failure of one pod can't take down the platform.

7. Data model & storage

7.1 Tenant routing

shops(shop_id PK, domain UNIQUE, pod_id, plan, created_at)

7.2 Per-pod schema (MySQL, schema-per-tenant or shared with shop_id)

products(shop_id, product_id, ...) PK(shop_id, product_id)
orders  (shop_id, order_id, ...)   PK(shop_id, order_id)

All tables prefixed by shop_id so every query is naturally tenant-scoped.

7.3 Why shared MySQL with shop_id keying?

Schema-per-tenant explodes at 2M shops (DDL becomes glacial). Shared schema with mandatory shop_id in every WHERE clause + linter at the ORM layer keeps tenants logically isolated while operationally manageable.

8. Detailed component design

8.1 Pod (cell) routing

Edge layer looks up domain → pod_id from a globally-replicated KV (DynamoDB-like). Requests are pinned to the home pod. Rebalancing migrates a shop with a copy-and-cutover approach: dual-write window + final delta replay.

8.2 Storefront rendering

Liquid templates compiled per-shop and cached in Memcache. Surrogate keys per shop allow targeted CDN purges on theme/product update.

8.3 Checkout

Shared checkout service but per-pod data. PSP integration uses idempotency keys; payment state machine identical to the order saga in the order pipeline doc.

8.4 Webhook delivery

Events to Kafka → fan-out worker → outbound HTTP with exponential backoff (up to 48h). Per-app circuit breaker prevents one slow app from exhausting workers.

8.5 Noisy-neighbor controls

Per-shop token bucket at API GW; per-pod global limit; "shop in trouble" mode that shifts the shop to a quarantine pod.

9. Scaling & bottlenecks

  • Mega-merchants (Kylie, Allbirds): graduate to dedicated "whale" pods.
  • Cross-shop search/reporting: async ETL into a column store (BigQuery) — never on the OLTP path.
  • App webhooks during BFCM: back-pressure with bounded queues per app; drop-with-DLQ rather than backlog forever.
  • DB hot rows: orders and inventory partitioned by shop_id; hot shop fits in one pod's primary or splits to its own.

12. Talking points for the interviewer

  • "Multi-tenant SaaS lives or dies by isolation; pods are the unit of failure containment."
  • "I'd never run cross-tenant joins on the OLTP store — that path leads to a single SPOF."
  • "Migration tooling is a first-class system: moving a shop between pods must be safe and observable, not a heroic exercise."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Pod (cellular) architecture2M merchants on one fleet creates a single blast radius. Pods of ~100k shops each, with their own MySQL primary and stateful services, mean a hot-row meltdown affects 5% of merchants, not 100%. BFCM peak of 50k checkouts/min spreads naturally across pods.Single global cluster (any noisy neighbor takes everyone down); micro-VM-per-tenant (2M VMs is operationally infeasible).
MySQL primaries (per pod)Shopify's bet: mature, ACID, well-understood, easy DBA recruiting. Per-pod primary handles a pod's shops at p99 < 500 ms admin API. Shared schema with shop_id PK prefix avoids 2M-schema DDL hell while keeping every query naturally tenant-scoped.Postgres (also fine; team familiarity tipped it); CockroachDB (cross-pod scaling unnecessary, costs more); DynamoDB (Liquid templates need joins).
Fastly CDN1 Tbps egress at BFCM peak with 700k storefront reads/s. Fastly's surrogate-key purges (per-shop key) let a theme update purge only that merchant's HTML in < 150 ms globally — critical for SEO and merchant trust. Edge VCL allows shop-domain routing without origin hops.CloudFront (purge granularity weaker historically); Akamai (more expensive at this scale).
Liquid (server-rendered)2M storefronts need SEO-friendly HTML and fast TTFB on slow 3G. Liquid templates compile per-shop, cache in Memcache, render on the pod. SSR keeps p99 TTFB < 300 ms; client-side React would push first-paint past 1 s on low-end devices.React/Next.js SSR (heavier runtime, cold starts); pure SPA (kills SEO, slow TTFB).
MemcacheCompiled Liquid templates and per-shop hot product fragments cache in Memcache for sub-ms reads. Pure cache (no persistence needed), high-throughput, cheap RAM. Per-pod cluster keeps blast radius local; eviction is fine because MySQL is authoritative.Redis (more features than needed; we want compiled-template byte cache, not data structures); local in-process cache (pod-wide invalidation hard).
Redis (per pod)Per-shop token buckets (noisy-neighbor control), session state, and short-lived workflow data live in Redis. Per-pod isolation means one shop's burst can't starve another pod's Redis. Lua scripts implement atomic rate-limiter increments.Memcached (no atomic ops for token buckets); DynamoDB (latency too high for per-request rate-limit check).
Elasticsearch (search)Per-pod ES index for storefront product search. 80k admin API/s and storefront search both need facets + typo tolerance. Indexing per-pod limits cross-tenant blast radius and keeps shard count manageable. Cross-shop search forbidden on this path — that's BigQuery's job.Algolia (would 2M tenants and SaaS-on-SaaS economics work? expensive); Postgres FTS (no facets at scale).
Kafka (platform bus)2B webhook events/day fan out from a Kafka platform bus. Partition by shop_id keeps webhook ordering per-shop. Outbox pattern ensures DB write + Kafka publish are atomic — you can't get a webhook for an order that doesn't exist.RabbitMQ (no replay for backfills); Kinesis (vendor lock to AWS, Shopify is GCP-heavy).
DynamoDB-like KV (shop routing)The domain → pod_id lookup must be globally replicated and sub-10 ms — every request hits it. A KV store (DynamoDB or BigTable) replicates to every region; failure of the routing tier would freeze the platform, so we replicate aggressively rather than rely on a Postgres primary.Postgres (single-region latency); DNS-based routing (TTLs too coarse for fast cutover).
OAuth + GraphQL Admin API3rd-party app ecosystem must connect on behalf of merchants without sharing credentials. OAuth scopes per app limit blast radius; GraphQL lets apps fetch exactly the fields they need, reducing API egress vs REST overfetch at the 80k admin API/s ceiling.API keys (no per-scope revocation); REST (overfetch problem at this scale).
Object store (S3/GCS)50 PB of merchant assets (images, videos, theme files) is impractical on block storage. S3-class with CDN-fronting handles 1 Tbps egress; per-shop prefix isolation simplifies billing and quota. MySQL stores only object refs, never blobs.Block storage (50 PB cost prohibitive); HDFS (operational tax).
BigQuery (cross-shop ETL)Cross-tenant reporting (admin dashboards, fraud, finance) cannot run on per-pod OLTP MySQL — it would re-introduce the SPOF that pods exist to prevent. Async CDC into BigQuery decouples analytics from the request path; minutely freshness is acceptable for reporting.Snowflake (similar fit; GCP-native ergonomics tipped it); per-pod OLAP queries (kills isolation).
Per-app circuit breakerWebhook delivery to slow apps could exhaust HTTP worker pools during BFCM. Circuit breaker per app + bounded queue + exponential backoff (up to 48h) + DLQ keeps one buggy app from blocking the other 999 apps' webhooks. Critical at 2B events/day.Single global queue (head-of-line blocking); fire-and-forget (loses delivery guarantees).
