Design Slack (workspaces, channels, threads, search)

Hard messaging search

1. Problem statement

Build a team-collaboration messenger: workspaces contain channels, channels contain messages and threads, with rich text, files, reactions, mentions, and full-text search across years of history. Slack serves ~20M DAU across millions of workspaces, sending ~1.5B messages/day. Unlike WhatsApp, server-side history is the product — every message is persisted forever, indexed, and searchable.

2. Functional requirements

  • Workspaces with channels (public, private, DM, MPDM up to 9 users).
  • Threads off any message; reactions; mentions; pinned messages.
  • File uploads with previews; rich text via markdown/blocks.
  • Real-time delivery with typing/presence per workspace.
  • Full-text search scoped to user's accessible channels.
  • Mobile / desktop / web with multi-device sync and unread counts.

3. Non-functional requirements

  • Latency: p99 send-to-receive under 500 ms; search p95 under 800 ms.
  • Scale: 20M DAU, 1.5B msgs/day (≈18k/sec avg, peak 60k/sec).
  • Durability: never lose a sent message; history retained per workspace policy.
  • Availability: 99.99% chat plane; 99.9% search.
  • Tenancy: hard isolation per workspace; per-workspace export and compliance.

4. Capacity estimation

  • QPS: 18k msgs/sec avg, 60k peak; reads ~10x writes ⇒ 600k reads/sec.
  • Sockets: ~5M concurrent WebSockets; 100k/box ⇒ 50 edge boxes.
  • Storage: 1.5B msgs/day * 1KB = 1.5 TB/day raw; 5 TB/day with metadata + index = ~2 PB/year.
  • Search index: Solr/ES, 200B docs total, sharded per workspace; ~1 PB on SSD.
  • File storage: 50 PB on S3 with CDN.

6. High-level architecture

[Client]──WSS──[Edge Gateway]──┐ ├──[Channel Svc]──[Vitess/MySQL shards] ├──[Thread Svc] ├──[Presence Svc (Redis)] ├──[Indexer]──Kafka──[Solr Search Cluster] ├──[File Svc]──[S3]──[CDN] └──[Notification Svc]──APNs/FCM/Email

Walk-through: client opens one WSS per workspace. Posting a message hits Channel Svc, which persists to MySQL (Vitess-sharded by workspace), publishes to Kafka, and pushes via the gateway to all online channel members. The Indexer consumes Kafka, writes to Solr. Reads of recent history come from MySQL + Memcached; older history and search go to Solr.

7. Data model & storage

7.1 Messages (Vitess / MySQL, sharded by workspace_id)

CREATE TABLE messages (
  workspace_id BIGINT,
  channel_id   BIGINT,
  ts           DECIMAL(16,6),    -- "1714938472.001200"
  user_id      BIGINT,
  text         MEDIUMTEXT,
  thread_ts    DECIMAL(16,6) NULL,
  blocks       JSON,
  PRIMARY KEY (workspace_id, channel_id, ts)
);

7.2 Why MySQL/Vitess?

Slack famously runs on Vitess. Per-workspace shard keys give linear scale, transactional updates for edits/reactions, and mature operational tooling. Channels-within-workspace fit a clustered index naturally (range scans for history are cheap).

7.3 Search (Solr)

doc {
  id: "W123:C456:1714938472.001200",
  workspace_id: 123, channel_id: 456,
  user_id: 99, text: "...", ts: 1714938472,
  acl: ["U99","U100","U101"]   // members at index time
}

8. Detailed component design

8.1 Edge gateway

Stateless WSS terminator. Maps (workspace_id, user_id) → socket. Subscribes the socket to channel topics in a pub/sub bus (Redis Streams or NATS). Backpressure: per-socket bounded queue, drop typing events first under load.

8.2 Channel service

Validates ACL, allocates monotonic ts (microsecond timestamp doubles as message id and sort key), writes to the workspace shard, fans out via pub/sub. Edits update the same row; an audit table tracks history for compliance.

8.3 Threads

A thread is messages sharing thread_ts. Loaded lazily; thread participants get notifications. Reply count maintained on the parent row, updated atomically.

8.4 Search

Indexer consumes Kafka, denormalizes channel members into the doc's ACL field at index time. Query path: rewrite to (text:foo) AND acl:Uxxx AND workspace:Wnnn. ACL changes (user leaves channel) trigger reindex of affected docs — bulk job, eventually consistent within minutes. Per-workspace shards mean a noisy tenant doesn't tank others.

8.5 Unreads

Per (user, channel) cursor stored in Redis with periodic flush to MySQL. Counts computed as messages-since-cursor; capped at 99+ to avoid expensive scans.

8.6 Notifications

Mention or DM triggers a job: if user has no live socket within 30s, route via APNs/FCM, else only update unread count. Email digest for 24h-stale mentions.

9. Scaling & bottlenecks

  • Mega-workspaces (500k users): single shard becomes hot. Sub-shard by channel_id within workspace, or move giant tenants to dedicated shards (multi-tenant + dedicated hybrid).
  • #general fan-out: a 100k-member channel broadcast = 100k push events. Use fan-out workers behind Kafka, batch by region, drop typing events under pressure.
  • Search ACL drift: reindex amplification when an admin renames a channel; mitigate with versioned ACL doc and lazy filter at query time.
  • Hot threads: a viral thread gets 10k replies; cap real-time fan-out and switch to pull-based after threshold.

12. Talking points for the interviewer

  • "I'd treat workspace_id as the universal shard key from day one — every query carries it, which makes Vitess routing trivial."
  • "Search ACL is the subtle hard part: stamp ACL into docs and reconcile on membership change; pure query-time filtering doesn't scale."
  • "For 500k-user enterprise tenants, I'd offer a dedicated-shard tier rather than try to make one fit-all shard size work."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
MySQL + VitessWorkspace-sharded MySQL gives ACID for edits, reactions, and unread cursors — needed because every message can be mutated, unlike WhatsApp. Vitess automates resharding when an enterprise tenant outgrows a shard, sustaining 60k peak msgs/sec across millions of workspaces.Postgres + Citus (less mature for this scale), Cassandra (no transactions for edits), DynamoDB (cost + vendor lock-in)
WebSockets (WSS) per workspaceOne socket per workspace gives clear ACL isolation — the gateway only subscribes a socket to topics that workspace member can read, simplifying the 5M concurrent socket fleet on ~50 boxes. Bidirectional frames are required for typing, presence, and message_event push under the 500 ms p99 budget.HTTP long-poll (extra latency, no presence semantics), Server-Sent Events (one-way only), gRPC streaming (browser support gaps)
Solr (search cluster)Full-text search across 200B docs with the 800 ms p95 NFR demands an inverted index, which neither MySQL nor Vitess can serve. Per-workspace shards give tenant isolation so a noisy enterprise can't tank others, and ACL-stamped docs let the query filter as a cheap term.Elasticsearch (similar; Slack chose Solr historically for tenant-shard control), Postgres FTS (doesn't scale to 1 PB SSD), MeiliSearch (too young for this scale)
Kafka (indexing pipeline)Decouples the 18k-avg/60k-peak msgs/sec write path from the eventually-consistent search index. Replayable log lets us rebuild Solr from scratch after schema changes, and ACL reindex jobs (channel renames, member churn) consume the same stream.RabbitMQ (no replay), Kinesis (vendor lock-in), direct dual-write (loses messages on Solr outage)
Redis (presence + pub/sub bus)Per-user, per-channel pub/sub fan-out at 600k reads/sec needs sub-millisecond latency — only an in-memory store qualifies. Per (user, channel) unread cursors with periodic flush to MySQL ride on the same fleet, capping cost.NATS (less mature ops at Slack's vintage), Memcached (no pub/sub), in-DB pub/sub (puts hot path on disk)
Memcached (recent-history cache)Reads are 10x writes (~600k/sec); the last few hundred messages of every active channel cache cleanly with LRU. Avoids hammering Vitess shards on common "open channel" UX, hitting the 500 ms p99 receive target.Redis (overkill for plain key/value), in-app LRU (doesn't share across gateways), CDN cache (data is per-user-ACL-filtered)
S3 + CDN (file storage)50 PB of files at compliance retention dwarfs the 2 PB/year message store; durable object storage with edge caching is the only economical fit and offloads bandwidth from the chat plane. Pre-signed URLs let clients upload/download direct.Self-hosted Ceph (ops burden), HDFS (not edge-friendly), embedding files in MySQL (insanity at 50 PB)
APNs / FCM / Email (notification svc)30 s "no live socket" window triggers mobile push for mentions and DMs; only the OS push channels reach a backgrounded device. Email digest layer covers 24 h-stale mentions for users who never opened the app.SMS (cost), web push only (no iOS background), polling (drains mobile battery)
microsecond-timestamp message id (ts DECIMAL(16,6))Doubles as primary key, sort key, and pagination cursor in one column — a single clustered index serves both history range scans and edits. Avoids the cost of a separate snowflake service for ~18k inserts/sec.Snowflake-style 64-bit IDs (extra service), UUIDs (random writes hurt Vitess), auto-increment (gaps under failover)
ACL stamped at index time (denormalized)Query-time ACL join across 200B docs would blow the 800 ms p95 search NFR. Writing the channel member list into each document lets Solr filter as a cheap term — paid back by background reindex on membership change.Pure query-time ACL filter (slow), per-user index shards (storage explosion), trusted client filtering (security risk)
JSON (blocks column for rich text)Slack's block kit is hierarchical and optional fields evolve frequently; a flexible JSON column avoids an ALTER TABLE every time a new block type ships. MySQL 5.7+ JSON support is enough for the rare server-side path queries.Per-block tables (write amplification), MEDIUMTEXT serialization (no querying), Protobuf BLOB (opaque to ops tools)
Compliance audit tablePer-workspace export and retention policies are an explicit NFR; an audit table tracks every edit/delete so the original content survives the live row's mutation. Required for regulated tenants (HIPAA, FINRA).Soft-delete flag (loses content history), event sourcing the whole DB (overkill), trusting Kafka log (retention too short)
