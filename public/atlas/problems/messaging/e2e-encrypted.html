Design End-to-End Encrypted Messaging (Signal protocol, double ratchet)

Very Hard messaging

1. Problem statement

Design an E2E-encrypted messaging service: only sender and recipient devices can read message content; the server only sees opaque ciphertext, routing metadata, and what it must to deliver. Implement the Signal protocol — X3DH for key agreement, double ratchet for forward secrecy and post-compromise security. Support multi-device, group chats, and asynchronous delivery (the recipient can be offline when sent). Real Signal serves ~50M MAU; WhatsApp uses the same protocol at 100B msgs/day.

2. Functional requirements

  • Identity registration; per-device key bundles uploaded to server.
  • Async first message: sender derives a session without recipient online (X3DH).
  • Forward secrecy and post-compromise security via double ratchet.
  • Group chat with sender keys (Signal group v2 / MLS for larger groups).
  • Multi-device: message encrypted to all of recipient's devices.
  • Safety numbers / key verification for users to detect MITM.
  • Sealed sender (server doesn't know who sent a message).

3. Non-functional requirements

  • Latency: p99 send-to-deliver under 500 ms.
  • Scale: 100M users, 1B msgs/day, peak 30k msgs/sec.
  • Availability: 99.95% delivery plane.
  • Privacy: server learns minimum metadata; "private contact discovery" via SGX or PIR.
  • Crypto correctness: deterministic, audited, no homegrown primitives.

4. Capacity estimation

  • QPS: 1B/day ≈ 12k/sec avg, peak 30k/sec.
  • Prekey storage: 100M users * 100 one-time prekeys * 32B ≈ 320 GB; per-device, multi-device 4x = ~1.3 TB.
  • Mailbox: ciphertext queued until delivery; assume 10% recipients offline ⇒ 100M msgs * 1.5KB = 150 GB transient.
  • Sealed-sender tokens: 100M * 64B = 6.4 GB rotated weekly.
  • Network: 1.5KB * 30k/sec = 360 Mbps text.

6. High-level architecture

[Sender]──TLS──[Edge]──[Auth (sealed sender)]──[Key Server] │ │ ▼ [Prekey Store] [Mailbox Queue]──push──[Recipient Edge]──[Recipient] │ [Cassandra] │ [Contact Discovery (SGX)]

Walk-through: at install, client publishes its identity key, signed prekey (rotated monthly), and 100 one-time prekeys to Key Server. To start a session, sender fetches recipient's bundle, runs X3DH to derive a shared secret, initializes a double-ratchet state, encrypts the first message, and POSTs ciphertext to the Mailbox. Recipient's edge pushes ciphertext when its socket is up; otherwise the Mailbox holds it until reconnect. Server never sees plaintext.

7. Data model & storage

7.1 Prekey store

CREATE TABLE prekeys (
  user_id BIGINT, device_id INT, key_id INT,
  pubkey BLOB, signature BLOB, used BOOLEAN,
  PRIMARY KEY ((user_id, device_id), key_id));
CREATE TABLE signed_prekey (
  user_id BIGINT, device_id INT,
  key_id INT, pubkey BLOB, signature BLOB, ts TIMESTAMP,
  PRIMARY KEY ((user_id, device_id)));

7.2 Mailbox

CREATE TABLE mailbox (
  user_id BIGINT, device_id INT, msg_id TIMEUUID,
  sender_blind BLOB,         -- sealed sender opaque to server
  ciphertext BLOB, ts TIMESTAMP,
  PRIMARY KEY ((user_id, device_id), msg_id));

7.3 Why Cassandra?

Per-(user, device) partition is naturally bounded; rows are deleted on delivery. Multi-DC with LOCAL_QUORUM gives durability without cross-ocean RTT. Append-only writes match LSM well.

8. Detailed component design

8.1 X3DH (Extended Triple Diffie-Hellman)

Sender computes 3 (or 4) DH operations: DH(IK_a, SPK_b), DH(EK_a, IK_b), DH(EK_a, SPK_b), DH(EK_a, OPK_b) — concatenated and HKDF'd into a 32-byte root key. The signed prekey gets verified with the recipient's identity key, preventing key substitution. The one-time prekey is consumed once; the server marks it used.

8.2 Double ratchet

Two interleaved ratchets: (a) DH ratchet — every reply, generate a new ephemeral DH key pair, derive a new root key; provides post-compromise security (an attacker who learns state can't decrypt future messages once you reply). (b) Symmetric KDF chain — within a sending streak, derive each message key from the previous; provides forward secrecy (compromise of current key doesn't reveal past messages). Out-of-order delivery handled by skipped-message keys cached up to a bound.

8.3 Sender keys for groups

Naive E2E group with N members = N pairwise ciphertexts per send. Sender keys: each member generates a chain key per group, shares it pairwise once. Subsequent sends are one ciphertext encrypted to that chain, server fans out. Member churn (someone leaves) requires the remaining members to rotate sender keys.

8.4 MLS for big groups

Sender-key approach is O(N) for membership change. MLS (Messaging Layer Security, RFC 9420) uses a binary tree (TreeKEM): membership change is O(log N). Worth adopting for groups > ~50.

8.5 Sealed sender

The sender packages their identity (with signature) into the ciphertext envelope. The outer envelope is anonymous to the server — only the recipient decrypts and learns who sent it. The server still rate-limits via short-lived "delivery tokens" issued by an unlinkable token service.

8.6 Private contact discovery

Naive "send me your phonebook hashes" leaks the social graph. Use Intel SGX enclaves (Signal's approach) or oblivious PRF / PIR so the server can match contacts without learning the input set. The enclave attestation is verified by the client.

9. Scaling & bottlenecks

  • Prekey exhaustion: heavy senders consume one-time prekeys faster than recipient replenishes. Server warns at 20%, falls back to signed prekey only (still secure but less PCS guarantees on first message).
  • Skipped-key memory: reordered messages cause keys to be cached. Cap at 2000 per session, drop oldest (sender retransmits).
  • Group churn storms: members leaving a 200-person group force ratchet rotations; batch with a 10s window.
  • Multi-device fan: 4 devices/recipient * group fan = 4x ciphertexts; acceptable but plan for it.
  • Metadata leak: traffic volume + timing remains visible — pad messages and add cover traffic for high-threat models.

12. Talking points for the interviewer

  • "I would never implement crypto from scratch — I'd use libsignal or an audited MLS library and treat the protocol as a black box."
  • "The hardest non-crypto part is contact discovery: privacy-preserving set membership at scale needs SGX or oblivious primitives, neither is free."
  • "Sealed sender plus rate-limit tokens is the cleanest way to keep anti-abuse working without giving the server identity metadata."
