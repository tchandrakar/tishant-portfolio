Design Discord (low-latency voice + text + servers)

Hard messaging video-streaming audio-streaming

1. Problem statement

Design Discord: a chat platform where "guilds" (servers) host text channels, voice channels, and screen-share / video channels. Real Discord serves ~200M MAU and ~14M concurrent users at peak, with per-guild member counts up to 250k+ and voice channels of up to ~99 simultaneous speakers. The design must couple cheap durable text storage with a hard-realtime voice pipeline (UDP, jitter buffers, SFU).

2. Functional requirements

  • Guilds (servers) with channels: text, voice, video/screen-share, threads, stage.
  • Real-time text with reactions, replies, edits, embeds.
  • Voice channels with 99-user cap, push-to-talk, low-latency mute/deafen.
  • Video / screen-share with up to ~25 active streams per channel.
  • Roles, permissions, mentions; presence and typing.
  • Direct messages and group DMs (up to 10).

3. Non-functional requirements

  • Voice latency: p99 mouth-to-ear under 150 ms.
  • Text latency: p99 send-to-receive under 300 ms.
  • Scale: 14M concurrent, 4B msgs/day, 1M concurrent voice users.
  • Availability: 99.95% chat; voice region-isolated to limit blast radius.
  • Cost: commodity hardware; voice priced in cents/user/month.

4. Capacity estimation

  • Text QPS: 4B/day ≈ 46k/sec avg, peak 150k/sec.
  • Sockets: 14M concurrent WSS at 100k/box ⇒ 140 gateway boxes.
  • Voice: 1M concurrent at ~32 kbps Opus = 32 Gbps upstream; SFU fan-out ratio ~15x (avg listeners per speaker) ≈ 480 Gbps egress.
  • Storage: 4B msgs/day * 1 KB ≈ 4 TB/day raw; ~5 PB/year.
  • Cassandra cluster: tens of thousands of nodes (Discord publicly reports ~177 nodes for messages historically, growing with ScyllaDB migration).

6. High-level architecture

[Client]──WSS──[Gateway]──[Session Cache]──[Guild Service] │ │ │ [ScyllaDB messages] │ │ │ [Redis presence] │ └──UDP/RTP──[Voice Edge / SFU]──[Voice Coordinator] (region pinned)

Walk-through: client connects to a regional Gateway WSS; gateway joins the user to all their guild channels via a fan-out bus. Text messages persist to ScyllaDB and broadcast over Redis pub/sub to all gateways holding members of that channel. Voice is bootstrapped by the gateway returning a voice-edge endpoint; client opens a UDP/RTP session to that SFU, which forwards packets to the other speakers/listeners in the channel.

7. Data model & storage

7.1 Messages (ScyllaDB / Cassandra)

CREATE TABLE messages (
  channel_id BIGINT,
  bucket     INT,           -- 10-day buckets
  message_id BIGINT,        -- snowflake
  author_id  BIGINT,
  content    TEXT,
  PRIMARY KEY ((channel_id, bucket), message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

7.2 Why ScyllaDB?

Discord migrated from Cassandra to ScyllaDB primarily for tail-latency: GC pauses on JVM Cassandra caused p99 spikes that user-perceptible chat traffic could not tolerate. ScyllaDB (C++, shard-per-core) delivers consistent low p99. Bucketing by 10-day windows prevents unbounded partitions on hot channels.

7.3 Guild metadata (Postgres)

guilds(id, name, owner_id, region, ...)
roles(id, guild_id, name, permissions BIGINT)
members(guild_id, user_id, roles JSONB, joined_at)

8. Detailed component design

8.1 Gateway

Elixir/BEAM-based; one process per session. Subscribes to Redis pub/sub channels per guild and per DM. On reconnect, replays missed events from a per-session ring buffer keyed by sequence number — clients send last-seen seq to resume.

8.2 Voice SFU (Selective Forwarding Unit)

WebRTC-based, but Discord built a custom Rust SFU. Each speaker uploads one Opus stream; SFU forwards that single stream to N listeners — no transcoding, no mixing. For video, SVC (scalable video coding) layers let listeners on bad networks pull a lower-bitrate substream from the same upload.

8.3 Voice region affinity

Each guild picks a region; voice edge servers are pinned to that region. Cross-region calls degrade gracefully via inter-region relays. UDP NAT punching first, TURN fallback for symmetric NATs.

8.4 Permissions

Guild roles encode a 64-bit permission bitfield. Channel-level overrides stored as (channel_id, role_id, allow, deny). On every action, server computes effective permission = base ∪ role grants − denies. Cached per (user, channel) in Redis with invalidation on role change.

8.5 Fan-out for huge guilds

A 250k-member guild can't push presence to every member. Discord uses "lazy guilds": a member only receives presence updates for users currently visible (member list scrolled into view, or in the same voice channel). This caps fan-out at the visible window (~100 users) regardless of guild size.

9. Scaling & bottlenecks

  • Hot channel partition: a 1M-message-per-day channel breaks Cassandra; bucketed partition key (channel, 10-day-bucket) keeps each partition under ~1M rows.
  • Presence storms: at game launch, millions update presence; lazy guild + Redis pipelines + coalesce updates into 5s windows.
  • Voice SFU CPU: packet forwarding is cheap; congestion control (TWCC) and packet pacing are the real cost. Use kernel-bypass (DPDK / io_uring) at scale.
  • Video bandwidth: uplink from a screen-sharer at 8 Mbps × 25 listeners = 200 Mbps egress; SVC + simulcast lets listeners subscribe to a 500 Kbps layer.

12. Talking points for the interviewer

  • "Voice and text are two completely different runtimes joined by a gateway — I'd resist the temptation to unify them."
  • "The hardest part is huge guilds; lazy fan-out and view-windowed presence are the only ways to keep N*M from melting the gateway."
  • "I'd choose ScyllaDB over Cassandra specifically for p99 — chat UX is unforgiving of GC tail latency."
