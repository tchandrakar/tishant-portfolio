Design Telegram (cloud-stored chats + channels at hundreds of M)

Hard messaging

1. Problem statement

Design Telegram: cloud-first messenger where regular chats and groups are server-stored (not E2E by default), supplemented by E2E "secret chats". Channels broadcast to up to 200k+ subscribers; supergroups support 200k members. Telegram serves ~900M MAU with ~70B messages/day. Unlike WhatsApp, the server holds the user's entire history and any device can sync the full archive.

2. Functional requirements

  • 1:1 cloud chats, supergroups (200k members), channels (unlimited subs).
  • Messages: text, media, polls, stickers, voice/video notes; edits within 48h; pinning.
  • Server-side full history sync to any logged-in device.
  • Optional E2E "secret chats" tied to one device-pair only.
  • Bots, inline queries, channel admins, scheduled messages.
  • Search across the user's entire cloud history.

3. Non-functional requirements

  • Latency: p99 send-to-deliver under 300 ms in-region.
  • Scale: 900M MAU, ~70B msgs/day, peak 1M msgs/sec.
  • Durability: permanent server-side history; no message loss.
  • Availability: 99.99% chat plane.
  • Geo: users pinned to a "home DC" near them; resilient to regional failures.

4. Capacity estimation

  • QPS: 70B/day / 86400 ≈ 810k/sec avg, peak 2.5M/sec.
  • Concurrent sockets: 900M MAU * 30% ≈ 270M; 1M sockets/box ⇒ 300 edges.
  • Storage: 70B msgs * 1KB = 70 TB/day raw; with media ~500 TB/day. ~180 PB/year hot + cold.
  • Channel fan-out: a 10M-sub channel posting once = 10M deliveries; mitigated via pull model.
  • Bandwidth: 1KB * 2.5M/sec = 20 Gbps for text; media dominates at hundreds of Gbps.

6. High-level architecture

[Client]──MTProto──[Frontend (CDN edge)]──[DC Router] │ ┌───────────────────┴──────────────┐ ▼ ▼ [Home DC: User] [Home DC: Channel] │ │ │ │ │ [Auth][Msg][File] [Channel][Index] │ │ [PostgreSQL/Custom KV] [Inverted index]

Walk-through: every user is assigned a home DC (DC2 Amsterdam, DC4 Miami, etc.) at signup. Reads/writes for that user route to that DC. Each chat lives in the DC of one of its participants; cross-DC chats use replication. Channels (broadcast) live in their own DC sharded by channel id. Clients see one logical service via the front-end edge, which proxies to the right DC.

7. Data model & storage

7.1 Messages (custom storage / sharded Postgres)

CREATE TABLE messages (
  dialog_id   BIGINT,        -- (user, peer) or channel id
  message_id  BIGINT,         -- monotonic per dialog
  author_id   BIGINT,
  body        BYTEA,
  ts          TIMESTAMPTZ,
  edit_ts     TIMESTAMPTZ NULL,
  PRIMARY KEY (dialog_id, message_id)
);

7.2 Why custom?

Telegram famously runs a hand-rolled storage layer in C++ on top of Linux raw block devices, with PostgreSQL for some metadata. The shape — append-mostly per dialog, range-scan for history — fits an LSM-style log per dialog. Per-dialog message ids are monotonically increasing integers, doubling as cursors and pagination tokens.

7.3 Update sequence (pts/qts)

Every user has a personal update counter (pts); every channel has its own (qts in the user's view, channel pts in storage). Clients store the last seen value; on reconnect they call getDifference to fetch missed updates incrementally.

8. Detailed component design

8.1 Home DC routing

On login, server returns a redirect to the user's home DC. All subsequent reads/writes for that user pin to it. Cross-DC messaging: when A (DC2) messages B (DC4), A's DC writes to A's outbox, replicates to B's DC inbox; B reads from DC4. Each side has a self-consistent view, with eventual cross-DC convergence.

8.2 Channels (broadcast)

Channels use a pull model: the channel writes once to its own append log; subscribers fetch via getDifference using their last pts. This makes a 10M-sub channel post O(1) on write, O(N) on read but spread across reconnects. Live "new post" notifications come via small fan-out updates that just say "channel X advanced to pts Y" — fat content stays in the log.

8.3 Supergroups (200k members)

Hybrid: write to a single per-group log, fan out lightweight notifications, members pull bodies on demand. Mentions and replies are surfaced via per-user mention index so a member doesn't miss them in a huge group.

8.4 Secret chats (E2E)

Per-device-pair MTProto E2E using DH key exchange and AES-256-IGE. No server-side history; messages drop after delivery. Tied to the originating device — secret chat does not sync across devices, by design.

8.5 Search

Per-user inverted index in the user's home DC indexes all cloud chats and channels they're in. Channel-side index for public channels. Sharded by user_id for personal index, by channel_id for channel index.

8.6 Media

Files chunked at 512 KB, deduplicated by content hash, stored in a custom file store with CDN edges. Reuse across forwards is free — forwarding a 1 GB video copies a reference, not bytes.

9. Scaling & bottlenecks

  • Cross-DC latency: A messaging B in another continent needs replication; mitigate with regional pairs and async replication, surface "delivered" only after target DC ACK.
  • Mega-channel posts: a viral channel with 50M subs; pull model keeps write cost O(1), but reconnect storms can hammer indexes — rate-limit getDifference per user.
  • Hot dialog partitions: a celebrity DM gets 100k messages from fans; per-dialog log must shard by message_id range.
  • Edit/delete amplification: editing an old message requires updating index docs; bounded by 48h edit window.

12. Talking points for the interviewer

  • "Pull-model channels are the key insight: you can't push to 50M subs in real time, so you push the metadata and let clients pull the body."
  • "Home-DC routing is the simplest correct multi-region model — it trades cross-region latency for a strongly consistent local view."
  • "Cloud chats vs secret chats is a deliberate UX/security trade — the platform is honest about it instead of pretending E2E everywhere."
