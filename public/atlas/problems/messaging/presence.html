Design Presence Service (online/offline/typing across 1B users)

Hard messaging

1. Problem statement

Build a presence subsystem usable by chat, social, and gaming products: report whether a user is online, when they were last seen, and whether they are currently typing in a particular conversation. Must scale to ~1B registered users, ~300M concurrent connections, and serve up to 5M presence reads/sec from clients rendering contact lists. Presence is soft state — strict durability is not required, but consistency tail behaviors are.

2. Functional requirements

  • States: online, offline, away (idle > N min), typing (per conversation).
  • Last-seen timestamp; per-user privacy (everyone, contacts only, nobody).
  • Subscribers receive push when a watched user changes state.
  • Bulk query: "give me state of these 5000 users I follow."
  • Multi-device: a user is online if any device is online.

3. Non-functional requirements

  • Latency: p99 propagation under 1 s for state changes.
  • Scale: 300M concurrent sockets; 50M state changes/sec at peak (mass disconnect events).
  • Read QPS: 5M/sec for bulk presence fetch.
  • Availability: 99.9% (presence outages must not break chat).
  • Cost: mostly in-memory; one box per few million users.

4. Capacity estimation

  • State storage: 1B users * (24B per record: id8 + state1 + lastSeen8 + flags4 + pad) ≈ 24 GB hot — fits in a small Redis/in-mem ring across ~20 shards.
  • Heartbeats: 300M sockets * 1 heartbeat / 30s = 10M/sec inbound.
  • Subscriptions: avg 200 contacts watched per user * 300M concurrent = 60B subscription edges. Sharded.
  • Fan-out events: a celebrity goes online with 5M followers ⇒ 5M fan-out events; cap and batch.
  • Network: 50 bytes per event * 50M/sec = 2.5 GB/s ≈ 20 Gbps internal.

5. API design

// From chat-edge to Presence Svc:
HEARTBEAT     {userId, deviceId, ts}
DISCONNECT    {userId, deviceId, reason}
SET_TYPING    {userId, peerId, ttl=6}

// Read APIs:
GET  /v1/presence/{userId}                  → {state, lastSeen}
POST /v1/presence/bulk  {userIds[]}          → map[userId]→{state,lastSeen}
WS   /v1/presence/subscribe {userIds[]}      → push events

// Internal events bus:
PRESENCE_CHANGED {userId, state, ts}

6. High-level architecture

[Chat-Edge]──heartbeats──┐ ▼ [Presence Shard Ring]──events──[Pub/Sub Bus] (in-mem, sharded by userId) │ ▲ ▼ │ [Subscriber Index] [Bulk Read API] │ ▲ ▼ [App Servers] [Push to watchers via Chat-Edge]

Walk-through: chat-edge boxes already terminate the user TCP socket, so they're the source of truth for "is this user connected." They emit heartbeats / disconnects to the presence shard hashed by userId. Each shard maintains a hashmap (userId → {state, lastSeen, deviceCount}). State changes publish events on a topic; the Subscriber Index turns "who watches userId X?" into a list of recipient sockets, and pushes to them via the chat-edge fleet.

7. Data model & storage

7.1 Hot store (in-memory shard)

struct PresenceRow {
  uint64 user_id;
  uint8  state;          // 0=offline,1=online,2=away
  uint32 device_mask;    // bitmap of connected devices
  uint64 last_seen_ms;
  uint32 privacy;        // visibility flags
}

7.2 Cold store (Cassandra)

CREATE TABLE last_seen (
  user_id BIGINT PRIMARY KEY,
  last_seen TIMESTAMP,
  privacy INT
);  -- async flush every 30s; survives restarts

7.3 Subscriber index

(watcheeId → list of (watcherId, edgeBoxId)). Sharded by watcheeId. Built lazily: when a chat opens, edge subscribes the viewer to the peer's presence. Eviction on chat close keeps the index proportional to active conversations, not total contacts.

8. Detailed component design

8.1 Heartbeat & failure detection

Phi-accrual style detector: an absent heartbeat for 2x interval marks the device offline. Only when the last device of a user goes offline does the user state change to offline — multi-device coalescing happens on the shard.

8.2 Typing indicator

Pure soft state; never written to disk. TTL'd at 6s. Stored on the receiver's chat-edge as an ephemeral row keyed by (peerId, conversationId). Lossy by design: if a typing packet is dropped the worst outcome is the dots disappear, which self-heals within 6s.

8.3 Bulk read path

Client opens a chat with 200 contacts. Edge issues one bulk fetch sharded by userId hash. Each shard returns its slice; results stitched. Cached at the edge with 10s TTL.

8.4 Subscription fan-out for hot users

Celebrity has 10M watchers. Naive fan-out is a thundering herd on every blink. Mitigations: (a) coarsen the celeb's state to a 30s ticker (no flap), (b) move from push-to-watchers to pull-by-watcher (watchers poll every minute on idle, push only when chat is open), (c) sample fan-out — only push to currently-online watchers with that conversation focused.

8.5 Privacy

Per-user privacy flag is checked on read, not write. "Last seen recently" / "Last seen within a week" are coarsenings applied at read time. Blocked-by-recipient lookup runs against a per-user blocklist Bloom filter to fast-reject.

9. Scaling & bottlenecks

  • Mass disconnect (regional outage): 50M users go offline in seconds; coalesce events at the shard with a 5s smoothing window before publishing.
  • Hot watchee: celebrity user blows out the subscriber index — cap subscribers per user (latest N) and degrade to pull-only.
  • Cross-region presence: a user in EU watching contacts in US needs cross-region pub/sub; use regional brokers with asynchronous bridges, accept ~1s extra latency.
  • Cardinality of typing: keep typing entirely on the receiver's edge; never let it hit the central presence ring.

12. Talking points for the interviewer

  • "Presence is soft state — I'd resist any pull to put it in a SQL database. Memory + async snapshot to Cassandra is the right shape."
  • "Celebrities break presence in the same way they break news feeds — the answer is the same: switch to a pull model past a follower threshold."
  • "Typing indicator stays on the recipient's edge; it never deserves a round trip to a central service."
