Design WhatsApp (1:1, groups, E2E encryption, presence)

Hard messaging notifications

1. Problem statement

Design a globally distributed instant messenger that supports 1:1 chat, groups up to ~1024 members, voice notes, media, presence (online / last-seen / typing), read receipts, and end-to-end encryption. The real WhatsApp delivers about 100B messages a day to 2B+ MAU using a famously small server fleet, so the design must lean hard on long-lived TCP connections, simple stores, and offload work to clients.

2. Functional requirements

  • Send / receive text, media, voice notes in 1:1 and group chats.
  • Delivery and read receipts (single tick, double tick, blue tick).
  • Presence: online, last-seen, typing indicator.
  • End-to-end encryption (Signal protocol, double ratchet) for all conversations.
  • Multi-device: phone primary plus up to 4 linked devices.
  • Offline delivery: messages queued until recipient reconnects.
  • Push notifications via APNs / FCM when client socket is closed.

3. Non-functional requirements

  • Latency: p50 send-to-deliver under 200 ms intra-region, p99 under 1 s globally.
  • Scale: 2B MAU, 1B DAU, 100B msgs/day, peak 2M msgs/sec.
  • Durability: never lose an acknowledged message; deliver at-least-once.
  • Availability: 99.99% for the chat plane.
  • Privacy: server cannot read message bodies.

4. Capacity estimation

  • QPS: 100B msgs/day / 86400 ≈ 1.16M msgs/sec average; peak ~3x = 3.5M/sec.
  • Concurrent sockets: 1B DAU * 30% online = 300M long-lived TCP sockets.
  • Per server: 1M sockets/box (FreeBSD + Erlang) ⇒ 300 chat-edge boxes.
  • Storage: encrypted ciphertext is ephemeral on server (deleted after delivery). Media blob storage ~5 PB/day at 50 KB avg * 100B msgs * 1% media.
  • Bandwidth: 1 KB avg ciphertext * 3.5M/sec ≈ 28 Gbps for text alone.

5. API design

// Persistent XMPP-like binary frame over TLS:443
CONNECT  {clientId, authToken, deviceId}
SEND     {msgId, toJid, ciphertext, ts}        → ACK {msgId, serverTs}
RECEIVE  {msgId, fromJid, ciphertext}          → DELIVERED {msgId}
READ     {msgIds[]}                            → broadcast to sender
PRESENCE {state: online|offline|typing}        → fan-out to subs
GROUP_CREATE {groupId, members[]}              → groupJid

// REST sidecar (HTTP/2):
POST /v1/media/upload   → blobId, mediaKey
GET  /v1/media/{blobId} → encrypted bytes

6. High-level architecture

[Phone]──TLS──┐ │ (persistent socket) [Chat-Edge / Erlang]──┬──[Auth Svc] │ ├──[Mailbox Store: Mnesia/Cassandra] │ ├──[Group Svc] │ ├──[Presence Svc (in-mem)] │ └──[Push Bridge → APNs/FCM] │ └──[Media Svc]──[S3 / Object Store]──[CDN]

Walk-through: client opens one TLS socket to its nearest chat-edge. Sender encrypts ciphertext per recipient device, ships it through the socket. Edge enqueues into the recipient's mailbox in Mnesia/Cassandra and pushes to the recipient's edge if online; otherwise sends a thin "you have mail" wake-up via APNs/FCM. Recipient pulls and ACKs, server then deletes the row.

7. Data model & storage

7.1 Mailbox (Cassandra)

CREATE TABLE mailbox (
  user_id   bigint,
  msg_id    timeuuid,
  sender_id bigint,
  ciphertext blob,
  ts        timestamp,
  PRIMARY KEY (user_id, msg_id)
) WITH CLUSTERING ORDER BY (msg_id DESC);

7.2 Why Cassandra / Mnesia?

Per-user partition is naturally bounded (queue drains on delivery), writes are append-only, no read-modify-write. WhatsApp historically used Mnesia (Erlang's in-VM store) for hot mailboxes. Multi-DC replication with LOCAL_QUORUM gives sub-region durability without cross-ocean RTT.

7.3 Identity / prekeys

CREATE TABLE prekeys (
  user_id bigint, device_id int, key_id int,
  pubkey blob, used boolean,
  PRIMARY KEY ((user_id, device_id), key_id));

8. Detailed component design

8.1 Chat-edge (Erlang/BEAM)

One Erlang process per connected user. The BEAM scheduler handles 1M+ lightweight processes per box; preemptive scheduling means a slow user can't block others. Frames are routed via a global registry (gproc / consistent-hash on userId) to find the recipient's edge box.

8.2 E2E encryption (Signal double ratchet)

On install, client generates an identity key, signed prekey, and 100 one-time prekeys, uploads pubkeys to server. Sender fetches recipient bundle, runs X3DH to derive a session, then double-ratchets per message: DH ratchet on every reply, symmetric KDF ratchet per send. Server only sees opaque ciphertext.

8.3 Group fan-out (sender keys)

Naive pairwise encryption for a 256-member group = 256 ciphertexts per send. WhatsApp uses "sender keys": one symmetric chain per (group, sender). Sender ships one ciphertext to the server, server fans out to N members, each member already has the chain key from a one-time pairwise distribution. Server-side fan-out is plaintext-of-routing, not plaintext-of-content.

8.4 Presence service

Strictly in-memory Redis-style ring keyed by userId. TTL 60s, refreshed via heartbeat. Subscribers (contacts who have the user's chat open) get pub/sub notifications. Typing is a soft signal with 6s TTL, never persisted.

8.5 Multi-device

Each linked device has its own identity key. Sender encrypts once per recipient device (4 devices = 4 ciphertexts), server fans out. History sync for new devices uses encrypted backup keyed off the primary device.

9. Scaling & bottlenecks

  • Connection storms: after a regional outage, millions reconnect. Mitigate with jittered exponential backoff in the client and SYN-cookie + LB rate limits.
  • Hot groups (1024-member): serialize fan-out per group through a single Erlang process; cap group size; pre-shard fan-out workers.
  • Push bridge backpressure: APNs/FCM are external; use a bounded queue per device, drop duplicate "you have mail" wakes.
  • Mailbox hot partitions: a celebrity getting 10k DMs has a fat partition. Bucket by msg_id day, drain fast.

12. Talking points for the interviewer

  • "I'd start single-region with one persistent-socket tier and a Cassandra mailbox; multi-region adds a routing layer that knows each user's home DC."
  • "The hardest part is presence at 300M concurrent sockets because naive fan-out is N*M; my mitigation is sub-graph filtering — only contacts who currently have your chat open subscribe."
  • "I would not put message bodies in any analytics pipeline — privacy is a contract, not an afterthought."

Every technology choice in this design exists for a concrete reason.
