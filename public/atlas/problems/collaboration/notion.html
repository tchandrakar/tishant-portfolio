Design Notion (block-based docs, multi-tenant, sync engine)

Hard collaboration file-storage

1. Problem statement

Build Notion: a multi-tenant workspace where every page is a recursive tree of blocks (paragraph, todo, table, embed, sub-page, database). Users edit blocks individually and concurrently, drag-reorder blocks across pages, and link them. Lower real-time bar than Google Docs (block-level, not character-level), but harder data modeling: a block can appear in many places (transclusion), and a database row is itself a page. Notion's actual sync engine is Postgres + a custom optimistic CRUD-op log per block.

2. Functional requirements

  • Block-level edits with conflict resolution (last-writer-wins per property).
  • Tree of blocks per page, drag-and-drop reorder, transclusion.
  • Databases (tables, boards, calendars) where rows = pages.
  • Realtime co-editing of text inside a single block (CRDT for that block only).
  • Multi-tenant workspaces, granular sharing, comments, mentions.
  • Offline support on web/desktop/mobile with merge on reconnect.

3. Non-functional requirements

  • Latency: p50 block save 100 ms, p99 page open 800 ms.
  • Scale: 100 M users, 10 M paying workspaces, 50 B blocks total.
  • Durability: RPO ≤ 1 s, RTO ≤ 5 min.
  • Tenancy isolation: noisy-neighbor proof; per-workspace QPS caps.
  • Availability: 99.9%.

4. Capacity estimation

  • 50 B blocks × avg 1.5 KB JSON = 75 TB hot, multiplied 3× (replicas + indexes) ≈ 225 TB.
  • Edit ops: 10 M DAU × 200 block ops/day = 2 B ops/day, ~25 K ops/s avg, 200 K peak.
  • Realtime sessions: 1 M concurrent, 5 ops/s/user → 5 M ops/s during peak (mostly transient).
  • Search index: 50 B blocks × 200 B inverted entries = 10 TB on Elasticsearch.

5. API design

POST /v1/block/{id}/op            { op:"update", props:{...}, baseVersion }
POST /v1/page/{id}/move-block     { blockId, newParent, newIndex }
GET  /v1/page/{id}?cursor=        → blocks[] (paginated)
WS   /v1/workspace/{id}/sync      → { type:"op", blockId, op, version }
GET  /v1/search?q=&workspace=     → results
POST /v1/db/{id}/row              → page (rows are pages)

6. High-level architecture

[Client]──HTTPS/WS──>[Edge]──>[API Gateway]──>[Sync Service]──>[Postgres (sharded by workspace)] │ │ │ └─>[WAL]──>[Kafka]──>[Search Indexer / Reco / Audit] ▼ [Realtime Hub (per workspace shard)] │ [Block CDN cache] (presigned) [Object Store: S3 for files/embeds]

Workspace is the sharding unit. Each shard = a Postgres logical database + a Realtime Hub process. The Sync Service is the single write path; it appends to the WAL which Kafka mirrors to downstream consumers (search, audit, AI).

7. Data model & storage

7.1 Postgres

CREATE TABLE blocks (
  id              uuid PRIMARY KEY,
  workspace_id    uuid NOT NULL,
  parent_id       uuid,
  type            text,           -- paragraph|todo|table|...
  props           jsonb,          -- type-specific
  position        numeric,        -- fractional index
  version         bigint,         -- monotonic per block
  updated_at      timestamptz,
  deleted_at      timestamptz
);
CREATE INDEX ON blocks (workspace_id, parent_id, position);

7.2 Why Postgres?

Notion famously sharded one giant Postgres into 480 logical shards, then 1,000+. Per-tenant ACID transactions are gold for the "move 50 blocks at once" UX; jsonb gives schema flexibility per block type. DDIA Ch.6 covers exactly this hash-by-tenant partitioning.

7.3 Why not a CRDT for blocks?

Block-level conflicts are rare (only one user usually edits a given block); LWW per property suffices. Inside a block (text), we run a Yjs YText CRDT — narrow scope, small payload.

8. Detailed component design

8.1 Sync engine

Optimistic client: applies op locally with a tentative version, sends to server. Server validates baseVersion matches, increments, persists, broadcasts. On conflict, server returns latest state and client rebases — block-level rebase is trivial because props are independent.

8.2 Block ordering

Fractional indexing on position. Reorder = single UPDATE; bulk reorder = multi-row txn. Periodic rebalance when gaps shrink past 1e-9.

8.3 Realtime hub

Workspace-affinity websockets. Sub-channels per page so a 100k-block workspace doesn't spam every client. Presence on a separate ephemeral topic.

8.4 Search

Kafka → indexer → Elasticsearch with workspace_id as routing key. Permission filtering done at query time using ACL bitmaps.

8.5 Sub-pages & transclusion

A block of type page_ref stores target_id only; loading is on-demand and cycle-detected at render time.

9. Scaling & bottlenecks

  • Hot workspace (10k editors): split shard, add read replicas; Realtime Hub fans out via Redis pub/sub.
  • Resharding: dual-write + backfill + cutover. Notion has documented this at length.
  • Large pages (10k+ blocks): paginate; never load full tree. Render virtualized.
  • Long offline: client keeps an op queue; on reconnect replays in order, server LWW resolves.

12. Talking points for the interviewer

  • "I would not use a global CRDT. Block-level LWW + scoped Yjs for text is the pragmatic Notion-style choice."
  • "The interesting design pressure is resharding; I'd commit early to a logical-shard mapping table so I never re-hash the world."
  • "Transclusion + permissions is the subtlest correctness issue — must check ACLs on every dereference, not at link time."
