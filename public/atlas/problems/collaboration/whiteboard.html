Design Real-time Whiteboard (Miro / Excalidraw — strokes, selection sync)

Hard collaboration

1. Problem statement

Build a multiplayer infinite canvas where users draw freehand strokes, drop sticky notes, connect arrows, and pan/zoom — all collaboratively in real time. The data model is a flat set of shapes (no deep tree like Figma), and the dominant op is "append a stroke at 60 Hz." The right tool is a CRDT keyed by shape-id (Yjs Y.Map), but with extra care for streaming partial strokes.

2. Functional requirements

  • Freehand pen strokes, shapes, sticky notes, arrows, text, images.
  • Multiplayer cursors and laser pointer.
  • Pan/zoom on an infinite canvas (no fixed pages).
  • Selection box, group move, copy/paste.
  • Offline drawing with merge on reconnect.
  • Export to PNG/PDF, embed videos / iframes.

3. Non-functional requirements

  • Latency: stroke render echo < 50 ms p99 same-region.
  • Scale: 30 M MAU, 500 K concurrent boards, max 200 editors per board.
  • Board size: p99 = 50 K shapes, 200 MB serialized.
  • Convergence: strong eventual consistency.
  • Availability: 99.9%.

4. Capacity estimation

  • Stroke point rate: 60 Hz × 16 B per point × 50 K active drawers = ~50 MB/s in-flight.
  • Finalized strokes: 50 K drawers × 1 stroke/3 s × 2 KB compressed = ~33 MB/s persisted.
  • Cursor pings: 500 K sessions × 30 Hz × 16 B = 240 MB/s aggregate.
  • Storage: 100 M boards × avg 5 MB = 500 TB.

5. API design

WS /v1/board/{id}/session         → { snapshot, vectorClock }
WS C2S { type:"stroke-begin", id, color, width }
WS C2S { type:"stroke-point", id, x, y, t, pressure }   // 60 Hz, unordered
WS C2S { type:"stroke-end",   id, points:[...] }        // canonical
WS C2S { type:"shape-op",     op, shapeId, props }
WS S2C { type:"op", origin, op, clock }
WS S2C { type:"cursor", peer, x, y }

6. High-level architecture

[Client]──WebTransport──>[Edge PoP]──>[Board Actor] │ (one per active board) ├─>[Yjs Doc in RAM] ├─>[Op log: ScyllaDB] ├─>[Snapshot: S3] └─>[Cursor relay (no persist)]

Each board has a single owning actor. Streaming stroke points are relayed instantly without going through the CRDT (low latency, lossy ok). Only the final stroke (`stroke-end`) is committed to the CRDT and persisted.

7. Data model & storage

7.1 CRDT

Board = Y.Map<ShapeId, Shape>
Shape = {
  type: "stroke" | "rect" | "sticky" | "arrow",
  bounds: {x, y, w, h},
  z: fractional-index,
  data: type-specific (e.g. compressed point array)
}

7.2 Op log (ScyllaDB)

CREATE TABLE board_ops (
  board_id uuid,
  bucket   int,        -- ts/3600
  ts       timeuuid,
  peer_id  uuid,
  op_blob  blob,
  PRIMARY KEY ((board_id, bucket), ts)
);

Bucketing prevents unbounded partitions on long-lived boards. ScyllaDB chosen for write-heavy throughput (~1M ops/s/cluster) at low cost.

7.3 Why Yjs?

Yjs encodes a Y.Map update in ~30–60 B, has a mature WASM build for the browser, and ships with vector-clock-based GC. We don't need OT — concurrent edits to a stroke don't really happen (it's owned by one drawer).

8. Detailed component design

8.1 Streaming strokes

The 60 Hz partial points are pure relay — server forwards to peers, drops on backpressure. The drawing client also sends `stroke-end` with the canonical compressed polyline; that's what becomes a CRDT shape. Trade-off: a viewer who joins mid-stroke sees nothing until end — acceptable.

8.2 Cursors / laser pointer

Pure pub/sub fanout via Redis Streams. 30 Hz, no persistence, server simply rate-limits per peer.

8.3 Selection sync (optional)

Selection state is ephemeral per peer, broadcast at 5 Hz. We do NOT CRDT it — different users have different selections.

8.4 Z-ordering

Fractional index per shape. "Bring to front" = max(z) + 1. Periodic rebalance.

8.5 Garbage collection

Shapes deleted > 30 days ago and acknowledged by all known peers are tombstone-collected via Yjs's GC, shrinking the doc.

9. Scaling & bottlenecks

  • Hot board (workshop with 200 editors): single actor saturates at ~10k shape ops/s; cursor relay scales horizontally separately.
  • Massive boards (100k shapes): client uses spatial index (R-tree) to render only the viewport; server streams shapes by viewport too.
  • Network jitter: WebTransport with QUIC tolerates packet loss on partial strokes; UI uses dead reckoning to interpolate.
  • Image / video embeds: uploaded to S3, only the URL stored in the CRDT (small payload).

12. Talking points for the interviewer

  • "The 60 Hz stroke-streaming is a separate concern from the CRDT; conflating them murders latency."
  • "I'd cap board size with a soft warning at 50k shapes — beyond that the CRDT's RAM footprint hurts mobile."
  • "If asked about precision: stroke points are quantized to 0.1px and delta-encoded, ~5× compression."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Yjs CRDT (Y.Map of shapes)Concurrent edits to a stroke don't really happen — each shape has one owning drawer — so CRDT's strong eventual consistency is plenty. Y.Map encodes a shape update in 30–60 B and supports vector-clock GC, vital for boards that live for years.OT (rejected: requires central writer, no real benefit on flat shape model); Automerge (rejected: heavier wire format than Yjs at our 33 MB/s persisted rate).
WebTransport / QUIC60 Hz partial stroke points (50 MB/s) tolerate loss but not head-of-line blocking. WebTransport datagrams skip TCP retransmits, beating WebSocket by ~30 ms p99 on 5% packet-loss links — the difference between a smooth and a janky stroke.WebSocket (rejected: TCP HoL stalls strokes); WebRTC (rejected: SFU/TURN ops complexity for relay-only).
WebSocket (committed ops)The canonical stroke-end and shape-op need ordered, reliable delivery. WebSocket over TLS is the right tool for the durable path; we run it side-by-side with WebTransport so committed state is never lost while streaming stays lossy-fast.WebTransport reliable streams (viable; redundant given we already need WS for browser fallback); HTTP/2 server push (rejected: no client→server reliable channel).
ScyllaDB (op log)Write-heavy: 33 MB/s persisted ops with bursts during workshops. Scylla's shard-per-core architecture hits ~1 M ops/s/cluster at < $100/TB/mo, far cheaper than Spanner. We don't need multi-key txns — each op is independent.Spanner (rejected: 5× cost, latency overhead unnecessary); Cassandra (rejected: same model, Scylla 5–10× faster on same hardware).
Bucketed partitioning (board_id, bucket)Long-lived boards would create unbounded partitions (Scylla's pain point). Bucketing by ts/3600 keeps partitions to ~hour-sized, capping any single partition at < 100 MB and avoiding hot-row compaction stalls.Single (board_id) partition (rejected: unbounded growth); per-day buckets (rejected: too coarse for hot boards).
Redis Streams (cursor relay)240 MB/s aggregate of cursor pings, ephemeral, sub-50 ms latency required. Redis Streams gives consumer groups for edge fan-out, automatic backpressure via XADD MAXLEN, and zero persistence cost — we throw away history.Kafka (rejected: 5–10 ms publish latency too high); NATS (viable; chose Redis for shared infra reuse).
S3 (snapshots + image embeds)500 TB total, 100 M boards. S3 at 11 9s is the only sane target for "boards from 2018 still load." Snapshot every 5,000 ops keeps S3 GET p99 (50–80 ms) on the cold path off the hot 50 ms render-echo budget.HDFS (rejected: ops cost); GCS (viable; chose S3 for AWS PoP overlap).
Single board actor (Rust/Go)Per-board ordering of CRDT ops requires one writer. A Rust actor saturates at ~10k shape-ops/s, which exceeds our 200-editor cap by 50×. No GC pauses keep p99 stroke-render echo under 50 ms.Multi-writer + CRDT merge (viable but complicates op log layout); Go (rejected: GC pauses on 200-editor hot boards).
Fractional indexing (z-order)"Bring to front" is max(z) + 1, "send between" is midpoint — no global reordering. With ~50 K shapes per board, integer z-rewrites would cause write storms; fractional cuts that to a single shape update.Integer z-index (rejected: write amplification on reorder); linked-list z (rejected: O(n) traversal for render).
R-tree spatial index (client)Boards with 100k shapes can't render fully on screen. Client-side R-tree gives O(log n) viewport queries, rendering only ~500 visible shapes per frame at 60 fps even on the 200 MB p99 board.Linear scan (rejected: 5+ ms per frame at 100k shapes); quadtree (viable; R-tree better for arbitrary-aspect rectangles).
etcd (board-actor lease)Single-writer requires a fencing lease so a stale actor can't append after partition. etcd's lease + revision number gives a 2 s replay-and-resume window with no split-brain risk.Redis Redlock (rejected: not consensus-grade); ZooKeeper (rejected: heavier ops).
Vector-clock tombstone GCBoards live for years; without GC the Y.Map balloons with deleted-shape tombstones, hitting our 1.5 GB RAM mobile limit. Per-peer vector-clock tracking lets us drop tombstones older than the slowest acked clock — bounded RAM, infinite history.No GC (rejected: unloadable boards after a year); time-based purge (rejected: breaks long-offline merge).
Delta + quantize point encodingEach stroke point is 16 B raw at 60 Hz. Quantizing to 0.1 px and delta-encoding shrinks ~5× to ~3 B/point on the wire, cutting our 50 MB/s in-flight bandwidth to ~10 MB/s without visible quality loss.Float32 raw (rejected: 4× wire cost); Bezier-fit (rejected: CPU cost on phone clients).
