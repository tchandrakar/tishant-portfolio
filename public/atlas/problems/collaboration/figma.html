Design Figma (CRDT, multiplayer cursors, vector graphics sync)

Very Hard collaboration

1. Problem statement

Build a browser-based multiplayer vector design tool. Many designers edit the same canvas concurrently — moving rectangles, editing vector paths, nesting frames — while seeing each other's cursors and selections in real time. The data model is a tree of objects with properties, not text, so we use CRDTs (specifically a tree CRDT + LWW maps) instead of OT. Real Figma uses a custom Rust+C++ multiplayer server with a tree CRDT very similar to Yjs.

2. Functional requirements

  • Tree of objects: page → frame → group → shape, arbitrary nesting.
  • Move, resize, restyle, reparent, delete, undo/redo per user.
  • Multiplayer cursors, selection halos, follow-mode.
  • Components / instances with overrides (additional CRDT layer).
  • Branching & merge of design files.
  • Offline-first: edit while disconnected, merge later without a central referee.

3. Non-functional requirements

  • Latency: p50 cursor echo < 30 ms within region; property echo < 80 ms.
  • Scale: 50 M MAU, 5 M concurrent multiplayer sessions, 200 M files.
  • File size: p99 file = 250 MB serialized, 10 M nodes.
  • Convergence: strong eventual consistency; same set of ops → same state regardless of order.
  • Availability: 99.95% on the multiplayer path.

4. Capacity estimation

  • Cursor updates: 5 M sessions × 30 Hz × 16 B = ~2.4 GB/s aggregate (most in-region).
  • Edit ops: avg 4 ops/s/active-user × 1 M concurrently active = 4 M ops/s, ~250 B each → 1 GB/s.
  • File storage: 200 M files × avg 30 MB = 6 PB; hot tier 600 TB on NVMe.
  • Op log retention: 30 days hot, infinite cold (S3 Glacier) for version history.

6. High-level architecture

[Client (WASM CRDT)]──WS──>[Edge PoP]──>[MP Router]──>[Multiplayer Server] │ (per-file actor, Rust) ├─>[CRDT in RAM] ├─>[Op log: FoundationDB] └─>[Snapshot: S3] [Pub/Sub bus] for cursor/presence

Each open file is hosted by exactly one MP Server actor. Edits are applied to an in-memory CRDT and broadcast. The CRDT runs identically in the browser (WASM) so clients can apply remote ops and edit offline.

7. Data model & storage

7.1 CRDT shape

Doc = Map<NodeId, Node>
Node = {
  id: GUID,
  parent: LWW<NodeId>,        // tree CRDT (Kleppmann move-tree)
  index: Fractional<Position>, // ordering siblings
  props: LWWMap<String, Value>,
  text:  Yjs.YText (only for text layers)
}

7.2 Op log (FoundationDB)

Keyed by (file_id, lamport_ts, peer_id). FDB gives ordered range scans + ACID multi-key txn for snapshotting. DDIA Ch.5 covers leaderless replication patterns; here we use it as a serialization point per file.

7.3 Snapshots

Every 5,000 ops we serialize the CRDT (Yjs-style update encoding, ~typically 30% of naive JSON) to S3. Loading = snapshot + ops since.

8. Detailed component design

8.1 Tree CRDT

Naive parent-LWW lets two concurrent moves create a cycle (A→B and B→A). Figma-style fix (Kleppmann 2020 "Move op for tree CRDT"): each move op carries the old parent; on merge, replay ordered by Lamport ts and undo any op that would create a cycle. Conflict-free, deterministic, ~O(n) replay cost — fine because moves are rare.

8.2 Sibling ordering

Fractional indexing (LSEQ-like). Inserting between A(0.5) and B(0.6) yields C(0.55). After ~50 inserts in the same gap we rebalance the file.

8.3 Cursors / presence

Never go through the CRDT. Separate UDP-like channel (WebTransport) at 30 Hz. Server just relays — no persistence, no ordering guarantees.

8.4 Undo

Per-user undo stack of inverse ops. Because CRDT ops are commutative, undoing a remote-modified node still works without "spooky" effects on other users.

9. Scaling & bottlenecks

  • Huge files (10 M nodes): CRDT memory ≈ 1.5 GB; we stream-load by viewport, lazily hydrating subtrees.
  • Hot file (1k editors in a design review): the single MP Server saturates at ~50k ops/s; we relay reads through edge fanout servers.
  • Op log growth: Yjs garbage collection drops tombstones once all peers have acknowledged a vector clock.
  • Failover: actor lease in etcd; on crash, new actor replays last snapshot + tail (under 2 s for 95% of files).

12. Talking points for the interviewer

  • "CRDTs win here because the data is a tree of properties, not a 1-D string — OT's transform functions get exponentially worse on trees."
  • "The killer correctness bug is the move-cycle problem; I'd cite Kleppmann's 2020 paper as my reference solution."
  • "Tombstone GC is the silent killer at scale; I'd build the vector-clock tracker on day 1, not day 100."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Yjs-style CRDTThe data is a tree of properties, not a 1-D string — OT's transform pairs explode combinatorially on trees. CRDT gives strong eventual consistency, true offline edit, and ~30% of naive JSON via run-length encoding, surviving our 250 MB p99 file budget.OT/Jupiter (rejected: tree-move transforms are an open research problem); Automerge (viable; rejected for slower merge perf at 10 M nodes vs Yjs).
Kleppmann move-tree CRDTNaive parent-LWW lets concurrent moves form cycles (A→B, B→A). Kleppmann 2020's algorithm replays moves in Lamport order and undoes any cycle-creating op, deterministic and conflict-free at O(n) replay since moves are rare (< 1% of ops).Parent-LWW (rejected: cycle bug); centralized move serialization (rejected: kills offline edit promise).
Rust multiplayer serverEach MP Server hosts a per-file actor that saturates at ~50k ops/s. Rust's no-GC tail latency keeps p99 property echo < 80 ms even at 1 M concurrent active editors; tokio's per-task scheduling beats Go's GC pauses on hot files.Go (rejected: 5–20 ms GC pauses break 30 ms cursor SLO); Java (rejected: same GC issue + 2× memory).
WASM CRDT in browserIdentical Yjs binary running on client and server means clients apply remote ops, edit offline, and rebase locally without a server roundtrip. WASM build is 80 KB gzipped and 5–10× faster than a JS port for our 4 M ops/s aggregate.Pure JS Yjs (rejected: too slow for 10 M-node files on mobile); custom JSON-patch (rejected: no offline merge guarantees).
FoundationDB (op log)Keys (file_id, lamport_ts, peer_id) need ordered range scans plus ACID multi-key txns to atomically swap a snapshot pointer with op-log truncation. FDB's serializable isolation across keys is rare in NoSQL and matches per-file actor's serialization point.Spanner (rejected: 50–100 ms commit latency hurts our 80 ms echo SLO); DynamoDB (rejected: no multi-key ACID at our scale).
S3 (snapshot store)200 M files × 30 MB = 6 PB. S3 at $0.023/GB/mo and 11 9s durability dominates the cost equation; snapshot-every-5,000-ops keeps recovery to 1–2 s tail replay, masking the 50 ms S3 GET latency.HDFS (rejected: ops cost); GCS (viable, comparable; chose S3 for region overlap with edge PoPs).
Fractional indexing (LSEQ-like)Sibling order under concurrent insert needs cheap conflict-free position keys. Inserting between 0.5 and 0.6 yields 0.55 with no coordination — O(1) and works offline. We rebalance after ~50 inserts in the same gap to bound key length.Linked-list CRDT (rejected: O(n) traversal for "move by 5"); integer indices with reorder ops (rejected: writes amplify under churn).
WebTransport (cursors/presence)5 M sessions × 30 Hz × 16 B = 2.4 GB/s of cursor traffic; loss is tolerable, latency is not. WebTransport's QUIC datagram mode skips head-of-line blocking and beats WebSocket by ~20 ms p99 on lossy mobile links.WebSocket (rejected: TCP HoL on packet loss adds 100 ms+ stalls); WebRTC data channel (rejected: STUN/TURN ops complexity for relay-only use case).
etcd (actor lease)One actor per file means we need a fencing token to prevent split-brain on failover. etcd's lease + revision number gives a 2 s replay-and-resume window, well under our 99.95% availability SLO.ZooKeeper (rejected: heavier ops); custom Redis lock (rejected: no consensus-grade fencing).
Vector-clock GCYjs tombstones grow forever without GC; on a 10 M-node file with 6 months of churn that's 1.5 GB RAM. Tracking per-peer vector clock and dropping tombstones once all known peers ack the clock keeps RAM bounded.No GC (rejected: file becomes unloadable); time-based purge (rejected: breaks long-offline clients merging in).
Pub/Sub bus (cursor relay)Cursors are out-of-band from the CRDT. A topic per file with at-most-once relay (we drop on backpressure) is the right semantics — stale cursors are useless. ~2.4 GB/s spread across regional Pub/Sub keeps fanout cheap.CRDT-encoded cursors (rejected: pollutes op log, 50× write amplification); Redis pub/sub (rejected: per-file shard hotspots).
Viewport-streamed loading10 M-node files exceed mobile RAM. Server streams subtrees by viewport and lazy-hydrates on scroll, keeping initial parse to < 800 ms even on a 250 MB file. CRDT structure makes partial loads safe — sync ops still apply.Full-file load (rejected: OOM on mobile); server-rendered raster (rejected: no offline edit).
