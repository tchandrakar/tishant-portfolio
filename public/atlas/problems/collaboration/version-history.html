Design Document Version History (snapshots + ops-log + branching)

Hard collaboration database-ddia

1. Problem statement

Build the version-history layer that powers "View revision history", "Restore to this version", and "Branch this doc" inside a collaborative editor. We must reconstruct any document state at any past timestamp efficiently, name versions, branch+merge them, and do it for billions of docs without exploding storage. This is essentially Git for live docs, layered on top of an OT or CRDT op log.

2. Functional requirements

  • Time-travel: open the doc as it was at any past instant.
  • Named versions ("Approved draft v3").
  • Diff between two revisions, attribution per change.
  • Restore: replace current head with an old version.
  • Branching: fork a doc, edit independently, merge back.
  • Retention policy per workspace (e.g. 30 / 90 / forever).

3. Non-functional requirements

  • Latency: "open at timestamp T" < 1 s p99 for any T within retention.
  • Scale: 1 B docs, 100 B revisions, 5 PB historical storage.
  • Cost: < $0.005 / doc / month amortized.
  • Durability: 11 9s on the cold tier (S3 std).
  • Consistency: a "named" version is immutable forever.

4. Capacity estimation

  • Op throughput in: 5 M ops/s × 200 B compressed = 1 GB/s ingest.
  • Snapshot frequency: every 1,000 ops or 5 min, whichever first → ~250 K snapshots/s globally.
  • Snapshot avg size: 30 KB compressed → 7.5 GB/s into S3.
  • Cold tier: 5 PB total, lifecycle to Glacier after 90 days.
  • Read pattern: long-tail; only ~0.1% of revisions ever opened.

5. API design

GET  /v1/doc/{id}/history?from=&to=        → revisions[] (id, ts, author, label?)
GET  /v1/doc/{id}/at?ts=                   → materialized doc

6. High-level architecture

[Editor]──ops──>[OT/CRDT engine]──>[Op Log: Spanner / FoundationDB] │ ├─every 1k ops──>[Snapshot Builder]──>[S3 Hot] │ │ │ └─lifecycle──>[S3 IA / Glacier] │ └─named-version──>[Version Table: Postgres] [History Service]──reads──>[Op Log] + [Snapshots] │ └─reconstruct(ts) = nearest snapshot + replay tail

7. Data model & storage

7.1 Op log

doc_ops (doc_id, branch_id, rev BIGINT, parent_rev, author_id, ts, op_blob, snapshot_id?)
  PK (doc_id, branch_id, rev)

7.2 Snapshots

snapshots (snapshot_id, doc_id, branch_id, base_rev, blob_uri, size, created_at)

Snapshot blob is the materialized doc as of base_rev, stored gzip-compressed in S3. Path: s3://snaps/{shard}/{doc_id}/{base_rev}.bin.

7.3 Named versions / branches (Postgres)

versions(version_id, doc_id, branch_id, rev, label, author, created_at)
branches(branch_id, doc_id, parent_branch_id, fork_rev, head_rev, created_at)

DDIA Ch.5 motivates separating durable ordered log (Spanner) from indexed metadata (Postgres) — log-structured + relational hybrid.

8. Detailed component design

8.1 Snapshot strategy

Trigger: every 1,000 ops or 5 min idle. Adaptive: hot docs lengthen the interval (one snapshot per minute max); cold docs do per-1k-op. Goal: bound replay tail to ≤ 1k ops so reconstruction is < 200 ms.

8.2 Reconstruct(ts)

1. Look up branch.head, walk back to find rev with ts ≤ T.
2. Find nearest snapshot s.t. snapshot.base_rev ≤ rev.
3. Load snapshot blob from S3 (or warm cache).
4. Apply ops [snapshot.base_rev+1 .. rev] from op log.
5. Return materialized doc.

Cache layer (Redis) keyed by (doc_id, rev) for popular reconstructions (e.g. "yesterday's snapshot" pinned overnight).

8.3 Diff

Reconstruct A and B; compute structural diff (Myers for text blocks, set-diff for shape props). Cache common diff pairs (head vs each named version).

8.4 Branching

Branch = new branch_id starting at fork_rev. Edits go to its own op-log range. Cheap: no copy. Storage cost is just the divergent ops.

8.5 Merge

For CRDT docs: merge is just union of ops + GC. For OT docs: replay branch's ops onto target, transforming each against intervening ops — same code path as live edit, just bulk.

8.6 Retention & GC

Per-workspace policy. Free tier: keep 30 days of granular ops + named versions forever. Background job rewrites old log ranges as a single coarse snapshot, then deletes ops. Named versions are pinned and never collapsed.

9. Scaling & bottlenecks

  • Snapshot write storm: stagger via jitter (snapshot at rev congruent to doc_hash mod 1000).
  • S3 GET tail latency: 99.9% < 80 ms but p99.99 spikes to 3 s — keep a Redis warm cache for last-day snapshots.
  • Long branches with thousands of divergent ops: snapshot the branch independently after 1k ops.
  • Cold reconstruct from Glacier: minutes-to-hours; UI must show "restoring archive" affordance.
  • Schema evolution: snapshot blobs versioned with codec_version; readers must handle N-2.

12. Talking points for the interviewer

  • "Reconstruct = snapshot + tail replay; everything else is plumbing. The snapshot cadence is the single biggest knob."
  • "I'd argue against full event sourcing alone — replay from rev 0 is fine in theory, brutal at p99 over a million-op doc."
  • "For branching, CRDT docs are massively easier than OT; if I were designing the editor and the history together I'd pick a CRDT just for this."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Spanner (op log, OT-backed docs)OT requires a totally ordered rev counter per (doc, branch). Spanner's TrueTime + linearizable inserts on (doc_id, branch_id, rev) preserve ordering across regions, which is what makes "restore to rev N" deterministic. 50–100 ms commit latency is fine on the history path.FoundationDB (chosen for CRDT-backed docs separately); Postgres (rejected: no native multi-region linearizable writes at 100 B revisions).
FoundationDB (op log, CRDT-backed docs)CRDT ops are commutative, so we don't need TrueTime — we need ACID multi-key txn to atomically swap a snapshot pointer with op-log truncation. FDB's serializable txns across keys at 1 GB/s ingest are exactly that, at half Spanner's cost.Spanner (overkill for CRDT, no ordering needed); DynamoDB (rejected: no multi-key ACID at scale).
S3 Hot + IA + Glacier tiers5 PB total, only 0.1% of revisions ever read. S3 Standard for last 30 days, IA at 90 days, Glacier afterward — lifecycle gives ~75% cost reduction. Hot S3 GET p99 ≈ 80 ms keeps the < 1 s reconstruct SLA.All-hot S3 (rejected: 4× cost, $200K+/mo); HDFS (rejected: ops cost + worse durability than 11 9s); Glacier-only (rejected: minutes-to-hours restore breaks UX).
Postgres (named versions + branch metadata)"Named versions are immutable forever" needs a relational store with secondary indexes on (doc_id, branch_id, label). Postgres jsonb + btree indexes serve millions of named versions with ACID guarantees on the create-version transaction; the op log stays log-structured.Same Spanner table (rejected: mixes concerns, expensive secondary indexes); MongoDB (rejected: weaker txn semantics for "version create must be linearizable").
Redis (snapshot cache)"Yesterday's snapshot" is opened thousands of times by everyone reviewing what changed overnight. Redis warm cache keyed by (doc_id, rev) caches the 30 KB snapshot blob; sub-ms hit beats S3's 50–80 ms p99 GET on the hottest path.CDN edge cache (viable; chose Redis for invalidation control on restore); local in-memory cache (rejected: cold on cross-region failover).
Adaptive snapshot strategyHot docs (high op rate) snapshot every 5 min max; cold docs snapshot every 1k ops. Goal: bound replay tail to ≤ 1k ops, keeping reconstruct < 200 ms. Adaptive cadence saves ~40% of snapshot writes vs always-1k on chatty docs.Fixed 1k-op cadence (rejected: snapshot storm on hot docs); fixed 5-min cadence (rejected: cold docs replay 10k+ ops).
Branch as op-log range (no copy)"Free fork" requires not copying op-log entries. New branch_id starting at fork_rev with edits going to its own range means storage cost is just divergent ops. CRDT-friendly merge is set-union; OT merge replays branch ops onto target via the same transform pipeline.Full doc copy on fork (rejected: 30 KB × N branches = unfeasible); shallow copy + COW pages (rejected: complex storage layer).
Myers diff (text) + set-diff (props)Diff between two reconstructed revisions needs two algorithms: Myers O(ND) for character-level text, set-diff O(n) for shape props. Myers is the right tool because of its quality on small edits — used by git diff for the same reason.Patience diff (viable but slower for our text-heavy case); structural tree diff only (rejected: poor at character-level text).
Snapshot codec versioningSnapshot blobs persist for years across schema changes. Embedding codec_version in each blob and requiring readers to handle N-2 keeps us forward-compatible without rewriting 5 PB of historical data.One-shot migration (rejected: 5 PB rewrite cost); never evolve schema (rejected: blocks editor evolution).
Jittered snapshot writes"Every 1k ops" naively triggers synchronously, causing thundering-herd S3 PUT. Jittering by hashing doc_id mod 1000 spreads 250 K snapshots/s evenly across the wall clock, holding S3 PUT p99 stable.Synchronous threshold trigger (rejected: write storms); per-shard token bucket (viable; jitter simpler).
Kafka (history-event stream)Downstream consumers (audit, analytics, AI summaries) need every op + every snapshot creation. Kafka's at-least-once + ordered partitions per doc_id let consumers replay; retention 7 days bridges any consumer outage without data loss.Direct DB tail (rejected: tightly couples consumers to op log); SNS/SQS (rejected: no ordering, no replay).
Cycle-detected branch graphBranches can fork from branches, and merges create DAG edges. Storing parent_branch_id with explicit cycle detection on merge-back avoids accidental loops; reconstruct walks the DAG backward to find the right snapshot in O(depth).Linear branch chain (rejected: doesn't model real fork-of-fork); flat branches (rejected: no merge ancestry, breaks 3-way merge).
Per-workspace retention policyFree tier (30 days granular) vs paid (90 days or forever) maps directly to lifecycle policy on S3 prefix and to op-log GC schedule. Background job collapses old log into a coarse snapshot, trimming the per-doc < $0.005/mo cost target without losing named versions.Global 90-day retention (rejected: free-tier cost); never delete (rejected: 5 PB grows linearly forever).
