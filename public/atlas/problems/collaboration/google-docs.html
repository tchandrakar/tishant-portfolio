Design Google Docs (Operational Transformation, OT)

Very Hard collaboration file-storage

1. Problem statement

Build a real-time collaborative rich-text editor where dozens of users can simultaneously edit the same document, see each other's cursors, and never lose a keystroke — even across flaky networks. The canonical algorithm here is Operational Transformation (OT), the same family used by Google Docs / Wave. We must converge all replicas, preserve user intent, and survive disconnects of minutes to hours.

2. Functional requirements

  • Rich text: bold/italic/lists/tables/comments, ~30 op types.
  • Real-time concurrent editing for up to 100 active editors per doc.
  • Offline editing with automatic merge on reconnect.
  • Cursors, selections, presence indicators.
  • Revision history, undo/redo per-user, suggestions / track-changes.
  • Comments anchored to character ranges that survive edits.

3. Non-functional requirements

  • Latency: p50 keystroke echo < 50 ms same-region, p99 < 200 ms.
  • Scale: 1 B docs, 500 M MAU, 50 M concurrent editing sessions.
  • Convergence: all replicas reach identical state within 1 s of the last op.
  • Durability: no acked op ever lost (RPO = 0).
  • Availability: 99.99% on the editing path.

4. Capacity estimation

  • Avg op size: ~120 B (insert/delete + attribution + range).
  • Per-user typing rate: ~3 ops/s while active → 50 M sessions × 0.3 active = 15 M ops/s peak.
  • Op-log storage: 15 M ops/s × 120 B × 86400 = ~155 TB/day raw, ~50 TB after compaction.
  • Snapshot store (every 1000 ops or 5 min): 1 B docs × avg 40 KB = 40 PB.
  • Edge fan-out bandwidth: 15 M ops/s × 120 B × avg 4 peers = ~7.2 GB/s.

6. High-level architecture

[Browser]──WS──>[Edge GW]──>[Session Router]──>[Doc Server (sticky shard)] │ │ │ ├─>[OT Engine] (in-mem) │ │ │ ├─>[Op Log: Bigtable/Spanner] │ │ │ └─>[Snapshot Store: GCS/S3] ▼ [Pub/Sub fan-out]──>peer Doc Servers

A document is owned by exactly one Doc Server at a time (single-writer per doc). The Session Router uses consistent hashing on docId. The OT Engine maintains the canonical revision log; every incoming client op is transformed against concurrent ops it hasn't seen, assigned a revision, persisted to the op log, and broadcast.

7. Data model & storage

7.1 Op log (Spanner)

CREATE TABLE doc_ops (
  doc_id   STRING(36),
  rev      INT64,
  client_id STRING(36),
  parent_rev INT64,
  op_blob  BYTES,        -- protobuf of ops[]
  ts       TIMESTAMP,
) PRIMARY KEY (doc_id, rev);

7.2 Snapshots

Every 1,000 ops or 5 min we materialize the doc and write to GCS. Recovery = latest snapshot + tail of op log. Truncate ops older than newest snapshot after a 7-day grace window for history.

7.3 Why Spanner?

Strict single-document linearizability via row-level locks on (doc_id, rev), multi-region sync replication, and TrueTime gives a globally ordered revision number — exactly what OT needs. DDIA Ch.9 (Linearizability) discusses why this matters for total order.

8. Detailed component design

8.1 OT engine

Each op carries (clientId, parentRev). The server holds the canonical sequence S. When op O arrives with parentRev=r and current server rev is R, the server transforms O against S[r+1..R] using transform functions T(a,b) → (a',b') satisfying TP1: apply(apply(s,a),T(b,a)) = apply(apply(s,b),T(a,b)). It then appends O' as rev R+1 and broadcasts.

8.2 Client

Maintains three buffers: acknowledged, in-flight (sent, awaiting ack), pending (queued). On receiving a remote op it transforms against in-flight+pending. This is the Jupiter model — server is the single source of truth, no client-to-client transforms.

8.3 Presence & cursors

Ephemeral, never persisted. Pushed on a separate Pub/Sub topic at 10 Hz, attached to revision so cursors don't jump after a remote insert.

8.4 Comments anchoring

Comments store (startMarkerId, endMarkerId). Markers are sticky positions in the op log; they survive transforms because each insert/delete updates marker offsets atomically.

9. Scaling & bottlenecks

  • Hot doc (10k editors on a launch doc): shard read fan-out via a tree of relay servers; writes still serialize on the owner.
  • Owner failover: lease in Chubby/etcd, 10 s TTL; new owner replays op log tail from last snapshot.
  • Long offline (days): client may need a full snapshot rebase; we cap offline-merge to 30 days.
  • Op log compaction: background compactor merges adjacent inserts from same author within 2 s.

12. Talking points for the interviewer

  • "OT is correct only if every transform pair satisfies TP1; I would lean on a verified library (ot.js / ShareDB) rather than write transforms from scratch."
  • "The hardest part is offline merge over days — I'd cap it and fall back to a 3-way diff if the op log is gone."
  • "For a greenfield design today I'd seriously consider Yjs CRDTs to drop the central writer; the ~2× payload cost is worth the operational simplicity."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
OT (Jupiter model)Rich-text intent preservation is well-studied for OT; payloads stay at ~120 B vs ~250–400 B for CRDT, halving our 7.2 GB/s edge fan-out. Jupiter's "server is single source of truth" eliminates client-to-client transforms, the bug-prone part of OT.Yjs CRDT (rejected: 2× payload at our scale costs ~$3 M/yr extra egress); ShareJS classic OT (rejected: client-to-client transforms add TP2 correctness burden).
Google Spanner (op log)OT correctness requires a totally ordered revision counter per doc. Spanner's TrueTime + row-level locks on (doc_id, rev) deliver linearizable inserts at 50–100 ms commit latency across regions, which is acceptable because we ack to client only after persistence.FoundationDB (rejected: weaker cross-region story for synchronous replication); Postgres + Patroni (rejected: no native multi-region linearizable writes at 1 B docs).
WebSocket transportBidirectional, sticky to the Doc Server shard, and works on every corporate firewall. We need ordered delivery for ops and 50 ms p50 echo; raw TCP framing through WS keeps headers under 10 B per op after permessage-deflate.WebTransport/QUIC (rejected: still maturing in 2024 enterprise browsers); SSE+POST (rejected: half-duplex doubles RTT for keystrokes).
GCS / S3 (snapshots)Every 1,000 ops we materialize a 40 KB doc — 40 PB total. Object storage at $0.02/GB/mo is 15× cheaper than keeping that in Spanner. Recovery is "latest snapshot + tail" so cold-read latency of 50–80 ms is fine.Spanner blobs (rejected: $0.30/GB/mo × 40 PB = unviable); HDFS (rejected: ops burden and worse durability than S3's 11 9s).
Pub/Sub (op fan-out)After the Doc Server commits rev R+1 it must push to ~4 peer servers and N edge connections. Pub/Sub gives at-least-once delivery with 10 ms intra-region latency; idempotency comes for free because rev numbers are monotonic.Kafka (rejected: per-doc topic explosion at 1 B docs); raw gRPC fanout (rejected: no buffering during peer hiccups).
etcd / Chubby (owner lease)Single-writer-per-doc requires a fencing lease so a stale owner can't append after partition. etcd's 10 s TTL leases with mod-revision fencing match our < 5 s failover SLO without paying for full Paxos in the data path.ZooKeeper (rejected: heavier ops, comparable semantics); Redis Redlock (rejected: not a true consensus system, known correctness gaps).
Consistent-hash session router50 M concurrent sessions must land on the right Doc Server shard. Hashing on doc_id with a virtual-node ring keeps re-shuffling at < 1/N of keys when a node dies, preserving the in-RAM OT state for unaffected docs.Random LB + global lookup (rejected: every op pays a Redis hop); Maglev (viable but added complexity for marginal balance gain).
Protobuf op encoding~120 B per op needs a tight schema. Protobuf gives 30% smaller wire size than JSON, schema evolution for our 30 op types, and zero-copy decode in the C++ Doc Server's hot path.JSON (rejected: 1.5–2× size, parse cost dominates at 15 M ops/s); FlatBuffers (rejected: poorer language support for the JS client).
Sticky markers (comments)Comments anchored to character ranges must survive arbitrary inserts/deletes. Sticky markers updated atomically in the OT engine cost O(1) per op vs recomputing offsets, keeping comment-anchor latency invisible at 15 M ops/s.Recompute on read (rejected: O(log n) per comment per render); CRDT range markers (rejected: incompatible with OT's mutable rev numbering).
Region-local sync replicasp99 keystroke echo budget is 200 ms; cross-region commit alone is 80–120 ms. Spanner sync within region + async cross-region keeps the hot path local while still hitting RPO=0 on majority loss.Global sync replication (rejected: blows past 200 ms p99 budget); Async-only (rejected: violates RPO=0 durability requirement).
Background op-log compactor15 M ops/s × 86400 s = 1.3 T ops/day raw. Merging adjacent inserts from the same author within 2 s shrinks log to ~50 TB/day. Saves ~$50 K/day in Spanner storage without altering the doc state — we replay merged ops identically.No compaction (rejected: 3× storage in 30 days); log-truncate at every snapshot (rejected: kills history granularity).
Per-user undo (client-side)Undo must invert only the local user's ops, not the document's, while remote ops keep flowing. Client tracks its in-flight + acknowledged stack and emits inverse ops; the server treats them as ordinary ops, no undo opcode needed.Server-side undo opcode (rejected: needs per-user state in the OT engine, breaks single-writer simplicity); CRDT-style "undo as new op" (already what we do, just framed differently).
