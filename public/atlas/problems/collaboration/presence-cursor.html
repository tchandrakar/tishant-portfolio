Design Presence + Live Cursor Service (ephemeral state, broadcast)

Hard collaboration messaging

1. Problem statement

Build a horizontal "presence" service that any product (Docs, Figma, Slack, Linear) can use: who is online, who is in this room, where are their cursors, what are they typing? The state is purely ephemeral (lost on disconnect), but fan-out is enormous and latency-critical. This is the Liveblocks / Pusher Channels / Slack Presence problem.

2. Functional requirements

  • Track online/offline status per (user, app, room) tuple.
  • Broadcast cursor (x,y), selection, "typing…", custom JSON state at up to 60 Hz.
  • Subscribe to a room and get the full presence snapshot on join.
  • Detect disconnects within 5 s.
  • Auth: only members of a room may publish or subscribe.
  • Multi-tenant: thousands of customer apps share infra.

3. Non-functional requirements

  • Latency: broadcast p99 < 50 ms within region, < 150 ms cross-region.
  • Scale: 50 M concurrent connections, 5 M rooms, 10 GB/s aggregate cursor traffic.
  • Durability: none required — losing presence is acceptable.
  • Availability: 99.95%; partial outages must degrade per-region, not globally.

4. Capacity estimation

  • 50 M conns × avg 2 rooms each = 100 M (conn, room) subscriptions.
  • Cursor publish: ~30 Hz × avg 10 active publishers / room × 5 M rooms × 16 B = 24 GB/s pre-fanout. Most rooms idle so realistic peak ~3 GB/s.
  • Fanout multiplier: avg 8 viewers / room → 24 GB/s egress.
  • Memory: 50 M conns × ~4 KB state = 200 GB across the fleet.

5. API design

WS  /v1/connect?token=jwt           → { connId }
WS  C2S { type:"join",   roomId }
WS  C2S { type:"presence", roomId, state:{cursor,sel,typing} }   // 60 Hz
WS  C2S { type:"leave",  roomId }
WS  S2C { type:"snapshot", roomId, peers:[{id,state}] }
WS  S2C { type:"update",   roomId, peerId, state }
WS  S2C { type:"left",     roomId, peerId }
GET  /v1/rooms/{id}/presence         → snapshot (REST)

6. High-level architecture

[Client]──WS──>[Edge LB (anycast)]──>[Edge WS Server (Rust/Go)] │ ├─local fanout for same-edge subscribers │ └─>[Room Router (consistent hash)]──>[Room Shard] │ └─pub/sub to peer Edge servers [Redis Cluster] = subscription index, KV for snapshots [Heartbeat watchdog] for disconnect detection

7. Data model & storage

7.1 In-memory only

RoomState {
  roomId,
  members: Map<peerId, { state, lastSeen, ownerEdge }>,
  subscribers: Set<edgeId>          // which edges have subscribers
}

Each room is owned by one Room Shard process. Edge servers locally cache the slice of state relevant to their connected peers and receive deltas from the owner.

7.2 Redis usage

Redis is the source-of-truth for the routing table only: room:{id}:owner → shardId, conn:{id}:rooms → set. Cursor state itself never goes to Redis (too hot). DDIA Ch.5 (gossip / multi-leader) describes the same trade-off.

8. Detailed component design

8.1 Connection layer

Stateless edge WS servers behind anycast. Use SO_REUSEPORT + io_uring on Linux to handle ~500K conns/box. JWT auth on connect; token includes app_id and allowed roomId glob.

8.2 Room ownership

Consistent hash on roomId picks a Room Shard. Owner shard maintains the canonical member list and broadcasts updates. On shard failure, watchdog reassigns ownership in < 2 s; clients see at most one snapshot replay.

8.3 Fanout strategy

Two-tier: Room Shard sends one message per Edge that has subscribers (not per subscriber). Edge then locally fans out to its peers. With 1k subs spread across 50 edges, this is 50× cheaper than naive fanout.

8.4 Rate limiting

Per-conn token bucket: 60 publish/s for cursors, 5/s for non-cursor. Drops oldest on overflow (cursor freshness > completeness).

8.5 Disconnect detection

Bidirectional ping every 5 s, declared dead at 12 s. Edge proactively notifies Room Shard, which broadcasts left. Without this, ghost cursors linger.

8.6 Cross-region

Each region has its own Room Shards. A room with members in multiple regions gets a "primary" region (where the room was created) and other regions subscribe via a tunneled WS. Adds ~80 ms cross-region but bounded.

9. Scaling & bottlenecks

  • Hot room (livestream w/ 100k viewers): Room Shard becomes CPU-bound on fanout; degrade to read-only mode (snapshot every 1 s instead of per-update).
  • Thundering herd on shard failover: 500K conns try to resnapshot — solved by jittered backoff (random 0–5 s).
  • Cursor flood: per-publisher coalescing — only the last cursor in a 33ms window survives; others dropped.
  • Cost: persistent WS conns are expensive. Idle rooms downgrade to long-poll after 60 s no activity.

12. Talking points for the interviewer

  • "Presence is the canonical 'big fanout, no durability' problem; treating it like a normal CRUD service is the #1 mistake."
  • "I'd lean hard on Rust/Go for the edge — GC pauses on 500K conns are death; tail latency dominates everything."
  • "For the interview, the key insight is: shard by roomId, fan out by edge, coalesce in time."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.
