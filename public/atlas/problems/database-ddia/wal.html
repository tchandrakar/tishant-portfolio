Design a Write-ahead Log (durability, replay, group commit)

Hard database-ddia

1. Problem statement

Design a write-ahead log: the append-only durability primitive at the heart of every serious database, message broker, file system journal, and consensus log. The contract is simple — "before any in-memory state change is acknowledged to the client, the corresponding log record must be flushed to non-volatile storage" — but the engineering is dense: file layout, fsync semantics, group commit, segment rotation, checksumming against torn writes, replay, truncation, and how the log is consumed (recovery, replication, change-data-capture). DDIA Ch. 3 frames this under "Making B-trees reliable" and "ARIES"; Ch. 5 reuses the same log for replication.

2. Functional requirements

  • append(record) -> LSN; durable when fsync returns.
  • flush(upto LSN) -> force durability barrier.
  • read(fromLSN) -> iterator for recovery and replication followers.
  • truncate(beforeLSN) -> reclaim space after checkpoint.
  • Crash-safe replay: idempotent application of records to state machine.

3. Non-functional requirements

  • Throughput: > 500 MB/s sequential on NVMe; > 100k ops/s with group commit.
  • Sync latency: NVMe ~50-200 us per fsync; cloud EBS gp3 ~1-2 ms.
  • Durability: survive process crash, OS crash, and (with stable cache or O_DSYNC + barrier) power loss.
  • Recovery: replay 1 GB log in < 30 s.

4. Capacity estimation

  • 50k tx/s * 250 B avg record = 12.5 MB/s; 1 day = 1 TB.
  • Segments of 64 MB -> 16 segments/min.
  • Index of (LSN -> segment, offset): 16 B per checkpoint entry, in-memory.
  • Buffer for in-flight commits: 8 MB ring buffer per writer thread.

5. API design

append(payload, [waitForSync]) -> lsn
flush(lsn)                     -> blocks until durable
sync()                         -> fsync()/fdatasync()
read(fromLsn)                  -> Iterator<Record>
checkpoint(lsn)                -> persist a fence
truncate(beforeLsn)            -> drop segments older than fence

6. High-level architecture

writers ---append()---> +-----------------------+ | In-memory log buffer | (ring, lock-free SP/MC) +-----------------------+ | (writer thread, group commit) | v +-------+ +-------+ +-------+ +-------+ | seg N | + | N+1 | + | N+2 | + | active| append-only files +-------+ +-------+ +-------+ +-------+ ^ | | v checkpoints fsync() | +---> truncation marker Consumers: - recovery (single pass replay from checkpoint) - replication followers (tail the log via offset) - CDC (Debezium/Maxwell-style decoders)

Walk-through: a writer copies its record into the buffer, gets an LSN, and either spins on a condvar until the leader thread fsyncs past its LSN (synchronous), or returns immediately (asynchronous). The leader batches all pending records into one I/O, calls fsync() once, and notifies all waiters. This is the textbook group commit from Helland/Mohan and is why a database can hit 100k tx/s with sync durability on a single disk.

7. Data model & storage

7.1 Record format

+------+--------+-------+----------+--------+--------+
| len  | type   | LSN   | term/xid | crc32c | bytes..|
+------+--------+-------+----------+--------+--------+
4 B    1 B      8 B     8 B        4 B
- LSN is monotonically assigned on append (atomic counter).
- crc32c covers header+payload to detect torn or partial writes.
- Records never cross a 4 KB sector boundary (or padding inserted) to make tearing detectable.

7.2 Segment layout

Fixed-size segments (e.g., 64 MB) named 0000000016AB.wal. Header carries magic, format version, base LSN. Active segment is preallocated and zero-filled to avoid metadata churn during fsync. Closed segments are immutable and may be compressed/archived.

7.3 Checkpoints

State machine periodically fsyncs its in-memory state and records a checkpoint record (checkpointLsn, dirtyPageTable, activeXacts). After successful checkpoint, segments below checkpointLsn can be truncated. ARIES three-pass recovery (analysis/redo/undo) starts from the most recent checkpoint.

8. Detailed component design

8.1 Group commit protocol

One thread is elected leader (lowest LSN). Followers register their commit LSN and sleep on a condvar. Leader writes the buffer, calls fsync(), advances durableLsn, and broadcasts. The waiter wakes when durableLsn >= myLsn. Effective fsyncs/sec drops from N (one per tx) to a small constant.

8.2 fsync semantics & failure modes

  • fsync() can fail and not retry: Linux's "fsyncgate" — many DBs now panic on fsync error to avoid silent data loss.
  • O_DIRECT bypasses page cache for predictability; requires aligned buffers.
  • Disk write cache: if not battery-backed, a sync may lie. Issue FUA or disable volatile cache on critical drives.
  • Partial-page tearing: sectors are atomic at 4 KB on NVMe, but file systems may split. CRCs + record-doesn't-cross-sector rules detect tears at replay.

8.3 Replay

Open the latest valid checkpoint. Read forward; for each record, verify CRC; if CRC fails or LSN regresses, that's the end-of-log marker. Apply records to the state machine using pageLSN >= recordLSN idempotency (ARIES). Truncate the file at the last valid LSN, then accept new appends.

8.4 Replication consumers

Followers tail the log by LSN. Postgres exposes this as the streaming-replication protocol; MySQL exposes it as the binlog; Kafka is a WAL. The single-source-of-truth principle from DDIA Ch. 11 ("Databases and Streams") means the WAL doubles as a CDC bus.

8.5 Truncation & archiving

Cannot truncate past the oldest LSN required by (i) the most recent checkpoint, (ii) the slowest replica, (iii) any retained snapshot. Postgres replication_slots hold the LSN floor; oversights here are the most common cause of disk-full incidents.

9. Scaling & bottlenecks

  • Single fsync stream: on cloud disks fsync is the limit. Mitigations: group commit, multiple WAL streams (TiKV "raft engine"), commit pipelining.
  • Lock contention on append: use a lock-free ring or per-core buffers merged at flush.
  • Replication lag: async followers fall behind; use semi-sync to bound RPO.
  • Disk full from retained slots: alarm on retained_wal_bytes.

12. Talking points for the interviewer

  • "I'd lead with the contract — append-then-flush before ack — and only then talk about group commit, because all the latency tricks are optimizations of that single rule."
  • "I'd call out fsyncgate explicitly: fsync errors are unrecoverable in many kernels, so the engine must crash-and-recover, not retry."
  • "I'd separate physical (page-image) and logical (row) logs; logical logs power CDC and cross-version replication, but recovery uses physical for speed."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Append-only sequential file layout Sequential writes are 10-100x faster than random on every storage medium — the foundational reason DDIA Ch. 3 introduces logs as a primitive. No seeks, no fragmentation, and replay is a single forward pass that the OS readahead can prefetch. In-place log (random writes destroy throughput); B-tree-indexed log (defeats the point — log doesn't need lookup, only append + scan).
Group commit (leader/follower pattern) The Helland/Mohan classic: one fsync amortizes hundreds of waiting writes, taking sync throughput from ~1k tps (per-tx fsync) to 100k+ tps. This is the #1 lever DDIA implies whenever it says "synchronous durability is fast on modern disks." Per-tx fsync (caps tps at 1/fsync_latency); async commit (different durability contract — RPO > 0 by design).
fsync() / fdatasync() with fsyncgate-aware error handling fsync is the only POSIX primitive that crosses the kernel-page-cache to durable storage barrier. Linux's "fsyncgate" means errors are not retryable — most engines now panic on fsync error to avoid silent data loss, exactly as DDIA Ch. 8 warns about unreliable storage primitives. fdatasync alone (skips file metadata — fine if size is preallocated); sync_file_range (advisory, no durability guarantee); msync (only useful with mmap, opaque ordering).
O_DIRECT + aligned 4 KB buffers Bypasses the OS page cache so latency is predictable (no cache-thrash interference) and writes don't double-buffer through dirty pages. Required for tight tail-latency SLAs on NVMe where fsync ~50-200 us is dominated by software path. Buffered I/O (page cache pollution from log streaming evicts hot user data); mmap (page-fault stalls on write barrier — see DDIA Ch. 3's critique of mmap-based engines).
Pre-allocated, zero-filled segments Avoid filesystem block-allocation latency spikes during fsync (extending a file forces metadata journal updates). Recycling closed segments by rename is O(1) and keeps fsync latency on the active segment stable. Append to single growing file (each fsync may extend the file — variable latency); per-record file (inode storm).
CRC32C per record + sector-aligned records Detects torn writes and silent corruption at replay. Records that don't cross 4 KB sector boundaries make tearing detectable: a partial sector write produces a CRC mismatch which is the canonical end-of-log marker. Hardware-accelerated on x86/ARM so cost is near zero. No checksum (silent corruption surfaces months later); SHA-256 (overkill for tear detection, costs 10x); xxHash (faster but ecosystem standardized on CRC32C).
LSN as monotonic 64-bit counter A single global sequence number ties every log record, page, and replication offset together. ARIES idempotency rule (apply only if pageLSN < recordLSN) and follower tailing both depend on LSN monotonicity — the abstraction that makes DDIA Ch. 5's replication-from-WAL strategy possible. Per-table or per-shard sequences (cross-shard ordering becomes a separate problem); timestamp-based (clock skew breaks monotonicity).
ARIES three-pass recovery (analysis, redo, undo) Handles partial transactions, crashes during recovery, and fine-grained locking simultaneously. Analysis rebuilds the dirty-page table and active-tx table from the checkpoint forward; redo replays everything; undo rolls back losers using CLRs to make undo idempotent across nested crashes. Redo-only recovery (cannot abort losers, requires force-at-commit which kills throughput); shadow paging (no fine-grained locking).
Lock-free ring buffer (or per-core buffers) for in-flight records Multi-producer concurrent appends without a global mutex — a single mutex on append becomes the throughput ceiling above ~16 cores. Per-core buffers merged at flush time (TiKV's raft-engine pattern) push the ceiling further. Mutex-protected buffer (lock contention dominates above ~16 cores); pipe/socket (extra copy, no bulk fsync).
Checkpoints with dirty-page table fence Bound recovery time = log volume since last checkpoint. The checkpoint record carries the dirty-page table and active-tx list so analysis pass starts cheaply, and segments below the checkpoint LSN become safe to truncate. No checkpointing (replay time grows unbounded); fuzzy checkpoint without DPT (analysis must scan entire log).
Physical (page-image) log for recovery, logical (row-based) for replication/CDC Physical logs are fast to apply (just memcpy a page) so recovery hits its 1 GB / 30 s target. Logical logs (row before/after images, like MySQL ROW binlog or Postgres pgoutput) decouple primary and replica versions and double as a CDC bus — DDIA Ch. 11's "WAL is the change-data feed" insight. Statement-based (broken by NOW(), nondeterministic functions — DDIA Ch. 5); physical-only (couples replicas to primary engine version, no CDC).
Replication slots / retention floor for followers Truncation must respect the slowest replica's read offset; otherwise a slow follower has to be rebuilt from a base backup. Postgres replication_slots hold the LSN floor exactly for this reason — the DDIA Ch. 5 "replication lag" pattern surfaced as an operational primitive. Truncate by time/age (slow replicas force rebuild — most common cause of disk-full incidents goes the other way); no follower tracking (correctness violation).
FUA / disabled volatile write cache on critical drives Disk firmware write caches lie about durability on power loss unless they're battery-backed (PLP). FUA (Force Unit Access) bypasses the cache for individual writes; alternatively the cache can be disabled cluster-wide. Without this, fsync gives only "process-crash" durability, not "power-loss" durability. Trust consumer disk caches (silent data loss on power cut); UPS-only (doesn't help if the host kernel panics with cached writes).
Multiple parallel WAL streams (TiKV raft-engine style) One sequential fsync chain is the global throughput ceiling on cloud block storage where fsync is ~1 ms. Sharding the WAL by Raft group (or hash) and fsyncing in parallel lifts the ceiling linearly with disk bandwidth — the only way to scale a Raft cluster past ~30k ops/s/group as flagged in §9. Single WAL (caps throughput at one disk's fsync rate); WAL on faster disk (helps but doesn't scale past one device).
