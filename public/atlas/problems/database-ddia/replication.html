Design Replication: leader-based vs leaderless vs multi-leader

Hard database-ddia

1. Problem statement

Replicate state across multiple nodes for durability, availability, and read scaling. The interview wants you to navigate the three families catalogued in DDIA Ch. 5 — single-leader (Postgres, MySQL, MongoDB), multi-leader (Active/Active across regions, BDR, CouchDB), and leaderless (Dynamo/Cassandra/Riak) — and explain when each is correct, what consistency anomalies they expose, how failover works, and how lag manifests to applications.

2. Functional requirements

  • Tolerate single-node failures with no data loss (RPO=0 ideal, configurable).
  • Reads from any replica with explicit consistency knobs.
  • Online membership change (add/remove replicas).
  • Backfill new replicas without downtime.

3. Non-functional requirements

  • Durability: survive loss of any 1 node (3-node cluster) or any 2 (5-node).
  • RPO/RTO: sync mode RPO=0; async mode RPO usually < 1 s; failover RTO < 30 s.
  • Read latency: closest replica; cross-region < 100 ms.
  • Replication lag: p95 < 100 ms intra-DC, < 1 s cross-region.

4. Capacity estimation

  • 1 TB primary -> 1 TB per follower; 3 replicas = 3 TB total.
  • Write fan-out N = number of followers; sync writes wait for max(follower latency).
  • WAL ship rate = primary write rate; size network for 2x peak.
  • Backfill 1 TB at 100 MB/s = ~3 hours.

5. API design

// Client knobs
write(k, v, {durability: ack|sync|quorum})
read(k,  {consistency: any|leader|read-your-writes|monotonic|linearizable})
// Admin

status() -> {leader, lag[], lsn[]}

6. High-level architecture

-------- Single-leader (Postgres / MySQL / MongoDB primary) -------- writes | client -> Leader --(WAL ship)--> Follower1 | --(WAL ship)--> Follower2 (sync) | --(WAL ship)--> Follower3 (async) reads<----+----------------------- any replica -------- Multi-leader (geo Active/Active) -------- USR <-> LeaderA(EU) <== async repl ==> LeaderB(US) <-> USR conflict resolution: LWW / CRDT / app -------- Leaderless (Dynamo / Cassandra) -------- client | coordinator (any node) / | \ N1 (R) N2 (R) N3 (R) W=2, R=2, N=3 -> W+R>N quorum read-repair + hinted handoff + Merkle anti-entropy

Walk-through: in single-leader, the leader is the sole serializer of writes; followers are passive WAL appliers. In multi-leader, both sides accept writes and replicate asynchronously, so concurrent writes to the same key require conflict resolution. In leaderless, every node can accept reads/writes and consistency emerges from quorums (W+R>N) plus background repair.

7. Data model & storage

7.1 Replication log formats

  • Statement-based: ship SQL — broken by NOW(), nondeterministic functions, triggers (DDIA Ch. 5).
  • WAL shipping (physical): bytes-of-page-changes; tightly couples versions and engines (Postgres streaming).
  • Logical (row-based): per-row before/after images (MySQL ROW binlog, Postgres pgoutput); decouples versions, enables CDC and selective replication.
  • Trigger-based: application-managed, slow but flexible.

7.2 Conflict resolution metadata

Multi-leader and leaderless need extra metadata: vector clocks or version vectors per key, Lamport timestamps for LWW, or CRDT state. Cassandra uses (timestamp, value) per cell with last-write-wins; Riak uses dotted version vectors.

8. Detailed component design

8.1 Single-leader replication

Followers connect, request "from this LSN," and stream WAL. Three durability modes: async (fastest, possible loss on leader crash), semi-sync (one follower must ack — bounded RPO), sync (all followers ack — slowest, strongest). Failover requires fencing (STONITH) plus a consensus-driven leader election to avoid split-brain. DDIA Ch. 5 "Handling Node Outages" details the failure modes: lost writes, broken auto-increment, ghost writes from the old leader.

8.2 Replication lag & consistency reads

  • Read-your-writes: route reads after a write to the leader (or to a replica caught up past the write's LSN).
  • Monotonic reads: sticky session pinned to one replica.
  • Consistent prefix: required when reads span partitions; provided by causal ordering.

8.3 Multi-leader replication

Each leader has the full data set. Writes propagate asynchronously; concurrent edits to the same key produce conflicts detected by version vectors. Resolution choices: LWW (lossy), application merge function (CouchDB/PouchDB), CRDT semantics. Topology choices: all-to-all, circular, star — circular has high latency and SPOFs; all-to-all needs loop detection (origin-stripping). Use cases: multi-DC HA, offline-first clients, geo-write-locality.

8.4 Leaderless replication

Coordinator forwards write to N replicas; ack from W. Read from R replicas; reconcile via timestamps, returning newest. W+R > N guarantees overlap so a read sees at least one node with the latest write — but only if the same N nodes are reachable; sloppy quorums (Dynamo) accept writes on temporary nodes via hinted handoff and forfeit this property. Anti-entropy via Merkle trees compares replicas in O(log n).

8.5 Failover & fencing

Detect leader failure via gossip/lease. Elect a new leader (raft-style or via an external coordinator like ZooKeeper/etcd). Fence the old leader with epoch numbers checked at every storage write so a zombie leader cannot mutate state. Reset replicas that diverged via point-in-time recovery from a base backup + WAL.

9. Scaling & bottlenecks

  • Write fan-out: sync writes scale poorly past 5 followers; chain replication amortizes.
  • Lag spikes under load: long transactions on the leader cause replicas to fall behind during apply. Mitigation: parallel apply (MySQL MTS, Postgres logical workers).
  • Cross-region cost: 50-150 ms RTT eats sync budgets; switch to semi-sync within region + async cross-region.
  • Hot keys in leaderless: a single hot key bottlenecks one replica set; precompute or shard the key.

12. Talking points for the interviewer

  • "I'd default to single-leader sync within a region and async cross-region; that gives RPO=0 locally and bounded lag globally."
  • "For mobile/offline I'd reach for multi-leader with CRDTs because LWW silently drops user edits — the sweet spot is per-field merge."
  • "Leaderless looks symmetric but has subtle failure modes (sloppy quorums, concurrent writes); I'd only choose it when the workload is large, write-heavy, and tolerant of eventual consistency."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Single-leader replication (default within a region) One serializer eliminates concurrent-write conflicts entirely — DDIA Ch. 5's simplest-correct option. With sync replication to one same-DC follower, RPO=0 and the model stays linearizable per partition without consensus on every write. Multi-leader (conflicts on every concurrent edit, requires resolution policy); leaderless (no transactional semantics, sloppy quorum loopholes).
Logical (row-based) replication log — pgoutput / MySQL ROW binlog Decouples primary and replica versions (different majors can replicate), enables selective table replication, and doubles as a CDC bus per DDIA Ch. 11. Carries before/after row images so it's deterministic regardless of nondeterministic SQL functions. Statement-based (broken by NOW(), triggers, nondeterminism — DDIA Ch. 5); physical WAL shipping (tightly couples engine versions, no per-table filtering); trigger-based (slow, app-managed).
Semi-synchronous replication (one follower must ack) Bounded RPO without paying for sync to all followers. The sweet spot DDIA flags: tail latency tracks one closest follower, not the slowest, while still surviving leader crash without data loss. Async (RPO > 0 — possible lost writes on failover); sync-all (latency = max follower; one slow replica halts the cluster).
Async cross-region replication (RPO < 1 s) 50-150 ms RTT eats any sync latency budget; async lets writes commit at intra-DC speed while shipping bytes across regions in the background. Bounded lag is acceptable when alternative is a 100 ms+ commit latency floor on every transaction. Sync cross-region (every commit pays inter-region RTT — kills throughput); leaderless multi-region (operational nightmare per DDIA Ch. 5).
External coordinator for leader election (etcd / ZooKeeper / Raft sidecar) Failover requires consensus on who the new leader is — otherwise split-brain corrupts state. DDIA Ch. 8 hammers this point: fencing tokens from the coordinator must be checked at every storage write to neutralize a zombie old-leader. Manual failover (RTO measured in minutes, error-prone); gossip-only election (no global ordering — split-brain risk); built-in election without fencing (zombie leader corrupts data).
Fencing tokens (epoch numbers) checked at storage writes The DDIA Ch. 8 prescription: every leader-issued write carries a monotonic epoch; the storage layer rejects writes from a smaller epoch, so a partitioned old leader cannot mutate state even if it doesn't know it lost leadership. Lease-based timeout only (clock skew defeats it); STONITH alone (works for hardware but slow, doesn't help with network partitions).
Read-your-writes via LSN routing After a write returns LSN L, route subsequent reads from that session either to the leader or to a replica whose applied-LSN ≥ L. Solves the most user-visible replication-lag anomaly DDIA Ch. 5 catalogues without forcing every read to the leader. Sticky-to-leader for N seconds (over-routes; doesn't help across sessions); ignore (users see their own writes disappear — visible bug).
Monotonic-reads via session pinning Pin a user session to one replica so reads never go backward in time. Cheap, requires no consistency machinery, and addresses the second DDIA-listed lag anomaly (jumping between replicas at different LSNs). Quorum reads (overkill, costs latency); ignore (users see "newer then older" data — undermines trust).
Parallel apply on replicas (MySQL MTS / Postgres logical workers) A single-threaded apply lags during write spikes; parallel apply by row-key hash or commit timestamp keeps p95 lag < 100 ms. Critical for read-replica freshness DDIA flags as a failure mode of long primary transactions. Single-threaded apply (lag spikes under load); parallel by transaction (loses ordering across rows — breaks consistency).
Vector clocks / version vectors (multi-leader, leaderless) Detect concurrent edits to the same key without a global clock — a happens-before partial order. Riak's dotted version vectors fix the false-concurrency bug of plain VVs. Required because LWW with wall-clock silently drops one user's edits per DDIA Ch. 5. Last-Write-Wins on wall clock (data loss on clock skew or true concurrency); single Lamport timestamp (no concurrency detection).
CRDTs for multi-leader application data Per-field merge with mathematical convergence guarantees, no resolution UI required. The right choice for offline-first mobile and collaborative editing where LWW would silently destroy concurrent edits to different fields of the same document. LWW (lossy); operational transform (only practical for sequential text — Google Docs); app-defined merge (every change creates a new bug).
Quorum reads and writes (W+R > N) for leaderless The Dynamo formula: as long as the same N nodes are reachable, W+R > N guarantees a read sees at least one node with the latest write. The compositional consistency knob DDIA Ch. 5 introduces — tune R and W per call to trade latency vs staleness. R=1, W=N (writes block; one slow node stalls all writes); R=N, W=1 (reads block similarly); arbitrary R+W (no overlap guarantee).
Hinted handoff + Merkle-tree anti-entropy Hinted handoff captures writes to temporarily down nodes so they can be replayed (sloppy quorum); Merkle trees diff replicas in O(log n) so divergent ranges are repaired without scanning the whole keyspace. Both are Cassandra/Dynamo essentials called out by DDIA Ch. 5. Read-repair only (slow keys never reconciled); full keyspace scan (O(N) per repair — infeasible at TB scale); manual repair (operational burden).
Streaming base backup + WAL catchup for replica bootstrap Adding a new replica requires a consistent base snapshot plus the WAL since that snapshot — Postgres's pg_basebackup model. Backfilling 1 TB takes hours; doing it without blocking the leader requires snapshot isolation on the source. Stop-the-world dump (downtime — unacceptable); copy individual files (no consistency point); logical dump+restore (10x slower for large data).
