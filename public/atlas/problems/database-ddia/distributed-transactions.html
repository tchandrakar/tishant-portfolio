Design Distributed Transactions (2PC, Saga, XA, Spanner-style TrueTime)

Very Hard database-ddia payments

1. Problem statement

Coordinate atomic, isolated, durable updates across multiple shards, services, or databases — the canonical "transfer money from account A on shard X to account B on shard Y" or "place an order that decrements inventory, charges payment, and records ledger entries." The interview wants you to compare the four main techniques: classic 2PC/XA, Saga with compensations, Percolator-style snapshot-isolation 2PC, and Spanner's TrueTime external-consistency approach (DDIA Ch. 7 "Distributed Transactions" and "Atomic Commit").

2. Functional requirements

  • All-or-nothing across N participants (atomicity).
  • Configurable isolation: read-committed -> snapshot -> serializable / external consistency.
  • Durable commit even if any participant or coordinator crashes.
  • Bounded blocking on coordinator failure (or escape via timeouts/heuristics).

3. Non-functional requirements

  • Commit latency: 2-3 RTT for 2PC; 1 RTT + clock-wait for Spanner; async for Saga.
  • Throughput: 10-50k cross-shard tx/s with batching; far higher for sagas.
  • Availability: tolerate any single participant failure; coordinator HA via Raft/Paxos.
  • Recovery: in-doubt transactions resolve within seconds, not days.

4. Capacity estimation

  • 10k commits/s * 5 participants = 50k participant-fsyncs/s.
  • Coordinator log: 200 B/record * 10k = 2 MB/s; 1 day = ~170 GB.
  • Lock memory at participants: avg 200 B/lock * 10k concurrent tx * 5 keys = 10 MB.
  • Spanner-style commit-wait: ~7 ms with TrueTime epsilon = 7 ms; bounded by GPS+atomic clock.

5. API design

// 2PC / XA
xa_start(xid)        on each participant
xa_prepare(xid)      vote yes/no
xa_commit(xid)       or xa_rollback(xid)

// Saga
beginSaga()
step1.execute(); step1.compensate = ...
step2.execute(); step2.compensate = ...
on failure: run compensations in reverse

// Spanner
beginTxn(readTimestamp)         // read at TT.now().latest
commit() -> commitTs;            // s = TT.now().latest at coordinator
                                 // wait until TT.now().earliest > s

6. High-level architecture

7. Data model & storage

7.1 Coordinator log (2PC)

{xid, participants[], state, commitTs, lastUpdate}
states: STARTED -> PREPARED -> COMMITTED | ABORTED

Coordinator force-writes PREPARED before sending commit messages; this is the durability commitment that makes the protocol blocking-safe (DDIA Ch. 7 "Three-Phase Commit" notes that 3PC tries to remove the blocking property but is unsafe under network partitions).

7.2 Saga state

Saga orchestrator persists {sagaId, currentStep, status, completedSteps[], compensationsRun[]}. Each step's command and compensation are idempotent and stored in the workflow definition (Temporal, Cadence, Camunda, AWS Step Functions).

8. Detailed component design

8.1 Two-phase commit (2PC)

Phase 1 — Prepare: coordinator asks each participant to validate constraints, write a prepare record, and lock the rows. A "yes" vote is a durable promise to commit if asked. Phase 2 — Decide: if all yes, coordinator durably writes COMMIT and tells everyone; if any no, ABORT. The blocking problem: if the coordinator crashes after some participants prepare but before deciding, those participants hold locks until the coordinator recovers — they cannot unilaterally abort (they promised). Mitigations: HA coordinator via Raft, heuristic decisions with reconciliation, escalation to operators after timeout.

8.2 XA transactions

The X/Open standard: a transaction manager (TM) drives 2PC across resource managers (RMs — DBs, MQs). Plagued by historical issues: heuristic outcomes leaving inconsistent state, "in-doubt" transactions blocking shutdowns, prepared-state recovery files. DDIA Ch. 7 explicitly warns that XA is widely deployed but operationally painful; modern systems prefer in-protocol consensus.

8.3 Saga pattern

Decompose a long-lived business transaction into local transactions S1..Sn each with a compensation C1..Cn. On failure at step k, run Ck-1..C1 in reverse. No isolation between sagas — interleaving exposes intermediate states; mitigations: semantic locks (reservation/confirm), commutative updates (CRDTs), business-level invariants. Two flavors: orchestrated (central coordinator, easier to reason about — Temporal) or choreographed (event-driven, looser coupling — Kafka events). Sagas trade strict ACID for availability and decoupling and are the dominant pattern in microservice architectures.

8.4 Percolator (snapshot-isolation 2PC)

Used by Google ads, TiKV, CockroachDB. Uses a global timestamp oracle for ordering. Prewrite places intent records and locks on each row; primary-key write commits the whole transaction atomically (a single Bigtable cell flip). Other participants observe the committed primary and roll forward. Lock cleanup is lazy and safe. Achieves serializable / SI semantics over a sharded KV store without needing a heavyweight TM.

8.5 Spanner & TrueTime

Spanner gives external consistency (linearizability for transactions) by combining Paxos per shard with TrueTime — an API exposing [earliest, latest] bounds on real time. The coordinator picks a commit timestamp s after acquiring locks and acknowledging prepare from all groups; it then performs commit wait until TT.now().earliest > s so no future transaction can pick a smaller timestamp. The wait absorbs the clock-uncertainty epsilon (~7 ms thanks to GPS + atomic clocks). Locks are released after the wait, but reads at any timestamp before s are immediately consistent. Reads use safe time on each Paxos group, allowing globally consistent snapshot reads across shards without locking.

8.6 Choosing a strategy

  • Within one DB cluster: use the engine's native distributed tx (Spanner, CockroachDB, YugabyteDB).
  • Across heterogeneous services: Saga with idempotent steps and outbox pattern.
  • Across two DBs in same DC: XA still viable if availability tolerates blocking; otherwise wrap in Saga.
  • Cross-region strict consistency: Spanner-style or accept higher latency for 2PC + consensus per participant.

9. Scaling & bottlenecks

  • Coordinator hotspot: shard coordinators by transaction id; co-locate with leading participant.
  • Lock holding during prepare: minimize keys per tx, batch reads before lock acquisition (read-only tx use SI without locks).
  • Long sagas: use durable workflow engines so steps survive process restarts.
  • Outbox pattern: avoid dual-write inconsistency by writing local DB + event in one local tx, then having a relay publish to the bus (CDC).
  • TrueTime epsilon: bigger epsilon -> longer commit-wait; invest in time infrastructure.

12. Talking points for the interviewer

  • "For payments I'd push toward Saga + ledger + idempotency keys: 2PC across a payment processor and a bank ledger is operationally fragile, and money systems already think in terms of authorize/capture/refund — that is a saga."
  • "If the interviewer pushes back with 'but I want strict consistency,' I'd describe Spanner: TrueTime + commit-wait gives external consistency without giving up scale, but it requires real clock infrastructure."
  • "The single biggest reliability lesson is the outbox pattern — never dual-write a database and a message bus; let CDC turn one durable write into both."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Two-phase commit (2PC) with HA coordinator The canonical atomic-commit protocol for heterogeneous participants — DDIA Ch. 7 prescribes it when you genuinely need all-or-nothing across systems that don't share a transaction layer. The PREPARED durability promise is what makes the protocol blocking-safe even though it's blocking on coordinator failure. Three-phase commit (DDIA Ch. 7 explicitly warns 3PC is unsafe under network partitions despite removing blocking); Paxos commit (more complex; rarely worth the engineering for a 2-3 participant case).
Coordinator backed by Raft/Paxos The DDIA Ch. 9 insight: 2PC reduces to consensus on the commit decision. Replicating the coordinator log via Raft eliminates the single-coordinator SPOF that historically made 2PC operationally fragile, so a coordinator failover takes seconds not hours of operator intervention. Single-node coordinator (in-doubt transactions block participants until the coordinator returns — XA's classic pain point); pair-of-nodes failover (split-brain risk without consensus).
Saga (orchestrated) for cross-service business transactions Microservice architectures rarely share a TM and locking across services for the duration of a 2PC is a liveness disaster. Sagas trade isolation for availability and are the right fit when steps map naturally to existing API operations (authorize/capture/refund) — DDIA Ch. 11's "compensations are normal in business processes." Cross-service 2PC/XA (locks held across HTTP latencies; one slow service stalls all participants); choreographed saga (event-driven but harder to reason about — debugging is a nightmare).
Saga orchestrator (Temporal / Cadence / Step Functions) Durable workflow engines persist saga state and survive process restarts, providing exactly-once-effectively semantics on each step via idempotency keys. Without durable orchestration, a coordinator crash mid-saga leaves the system in an inconsistent state. App-managed saga state in DB (every team rebuilds workflow primitives, badly); choreography via Kafka events (no central state — debugging requires correlating cross-service logs).
Percolator-style snapshot-isolation 2PC for sharded KV Used by Google Ads, TiKV, CockroachDB. A primary-key intent flip commits the whole transaction atomically over a sharded KV store, with lazy lock cleanup that's safe under crashes. Achieves SI/SSI without a heavyweight TM and scales horizontally with the underlying KV store. Classical 2PC + global lock manager (the lock manager becomes the scaling bottleneck); Spanner-style TrueTime (requires real clock infrastructure).
Spanner + TrueTime (external consistency at geo scale) Combines Paxos per shard with a clock API exposing [earliest, latest] real-time bounds. Commit-wait until TT.now().earliest > s ensures no future tx picks a smaller timestamp — gives linearizability across shards without distributed locking. The DDIA Ch. 8/9 reference for "doing strict consistency at planet scale." NTP timestamps + 2PC (clock skew breaks ordering — DDIA Ch. 8); 2PC + global lock service (lock service becomes a global bottleneck).
TrueTime-class infrastructure (GPS + atomic clocks) or AWS TimeSync Bounded epsilon (~7 ms) is what makes commit-wait short enough to be practical. The smaller the epsilon, the lower the commit-wait — directly trades hardware investment for transaction latency. NTP only (epsilon measured in seconds — kills throughput); Marzullo's algorithm on commodity NTP (no real bound on skew).
Global timestamp oracle (TSO) for Percolator Hands out monotonic timestamps to give transactions a total order. Single-machine TSO scales to millions of timestamps/second via batched allocation; replicated via Raft for HA. Avoids the clock-skew problem TrueTime solves with hardware by centralizing time-allocation. Per-shard timestamps (no cross-shard ordering); Lamport/HLC clocks (fine for ordering but don't give SI snapshots cheaply).
Outbox pattern + CDC relay The canonical fix for the "dual-write to DB + message bus" anti-pattern. Write the row + outbox event in one local transaction; a CDC relay (Debezium, Maxwell) tails the WAL and publishes to the bus. Turns one durable write into both with no distributed transaction needed — the DDIA Ch. 11 "exactly-once via dedupe" pattern in production form. Direct dual-write (one side may fail — silent inconsistency); 2PC across DB + broker (XA over Kafka is rare and operationally heavy); transactional outbox without CDC (requires a poller — duplicate work).
Idempotency keys at every step Every saga step (and every retry of a 2PC participant) must be idempotent because crashes/timeouts cause replays. Idempotency keys (client-supplied UUIDs stored in a dedupe table) make "exactly-once-effectively" achievable without 2PC — Stripe/AWS API style. Best-effort dedupe by content (collisions cause incorrect dedupe); no dedupe (replays double-charge or double-credit — money bugs).
Semantic locks (reservation/confirm) inside sagas Sagas have no isolation, so concurrent sagas can observe intermediate states. A two-step "reserve seat" / "confirm seat" with TTL on the reservation gives application-level isolation — the standard pattern for inventory and seat-booking workflows. Pessimistic locks across the whole saga (saga can take seconds to minutes — locks are too long-lived); ignore (overselling, race conditions).
Heuristic decision recording for in-doubt 2PC If the coordinator is permanently lost, an operator can force commit/abort on a participant — the XA standard's escape hatch. The participant records that the decision was heuristic so reconciliation can detect the divergence later (DDIA Ch. 7 explicitly calls this out as XA's safety-vs-liveness trade-off). Block forever waiting for coordinator (locks held indefinitely — operational disaster); auto-abort on timeout (violates the prepare promise — corrupts atomicity).
Read-only transactions via SI without locks Snapshot-isolation reads at a fixed timestamp need no locks and no 2PC participation; serve them from any replica with safe-time ≥ snapshot. Eliminates the lock-holding cost flagged in §9 for the read-mostly fraction of the workload. 2PC even for read-only (acquires locks on every shard — wasteful); pessimistic reads (blocks writers — kills throughput).
Outbox + Kafka topic per aggregate (DDIA "stream as truth") Long sagas span minutes/hours; a durable event log lets steps survive process restarts and lets new consumers replay history for analytics or fraud detection. Aligns with DDIA Ch. 11's "databases and streams" — the WAL/event log becomes the source of truth. Stateless message queue (events lost on consumer crash); RabbitMQ-style ack queues (no replay for new consumers).
