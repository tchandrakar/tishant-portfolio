Design a B-tree Storage Engine (page-based, WAL, buffer pool)

Hard database-ddia

1. Problem statement

Design the workhorse storage engine behind PostgreSQL, MySQL/InnoDB, Oracle, SQL Server, and SQLite — a disk-resident B+-tree with a fixed-size page abstraction, a buffer pool, and a write-ahead log (ARIES-style). The interview here is about data structure honesty: page splits, latch coupling, recovery, partial page writes, and why B-trees still win for OLTP point-and-update workloads even though LSM trees win sequential write throughput (DDIA Ch. 3, "B-Trees" and "Comparing B-Trees and LSM-Trees").

2. Functional requirements

  • Indexed point lookups, range scans, inserts, updates, deletes on a primary key.
  • Crash recovery via redo + undo logs (ARIES).
  • Multi-row transactions with row-level locking (or MVCC via undo segments).
  • Online index build / vacuum without long write blackouts.

3. Non-functional requirements

  • Read latency: in-memory tuple in < 50 us; disk-resident in 1 SSD seek (~100 us NVMe).
  • Write throughput: 50-100k tx/s on a single node; bound by WAL fsync and page flushes.
  • Durability: redo log fsync before COMMIT; recovery from any crash < 1 min.
  • Cache hit ratio: > 99% for OLTP working set in the buffer pool.

4. Capacity estimation

  • Page size 8 KB (Postgres default 8 KB; InnoDB 16 KB). Tree fan-out ~200-500 keys/page.
  • 1 B rows -> ceil(log_400(1e9)) = 4 levels; 4 page reads per cold lookup.
  • Buffer pool 64 GB hosts ~8 M pages — typical OLTP working sets fit fully.
  • WAL: 50k tx/s * 200 B/record = 10 MB/s; 1 hour = 36 GB.

5. API design

BEGIN;                                  -- assigns txId
SELECT v FROM t WHERE k = ?;            -- shared latch on leaf, S row lock
INSERT INTO t VALUES (?,?);             -- X latch on leaf, possible split
UPDATE t SET v = ? WHERE k = ?;         -- in-place; undo record written
DELETE FROM t WHERE k = ?;              -- mark deleted; vacuum reclaims
COMMIT;                                 -- WAL flush + commit record

6. High-level architecture

SQL -> Executor -> Access methods (heap, B+tree) | v +----------------------------+ | Buffer Pool (LRU) | pages pinned/latched +----------------------------+ | | page-in (read) page-out (checkpoint/eviction) | | v v +---------+ +---------+ | Disk | | Disk | | data | | data | +---------+ +---------+ ^ | redo log replay on crash | +------------ WAL (sequential, fsync per commit)

Walk-through: every page modification is preceded by a WAL record (write-ahead). The dirty page sits in the buffer pool; a periodic checkpoint flushes dirty pages and emits a checkpoint record, bounding recovery time. After a crash, recovery does redo from the last checkpoint to the end of the log, then undoes losers (uncommitted transactions) using undo records — the canonical ARIES trio of analysis, redo, undo.

7. Data model & storage

7.1 B+-tree page

+--------+------------------+--------------+----------+
| Header | Slot array (off) | Free space   | Tuples   |
+--------+------------------+--------------+----------+
Header: pageId, LSN, level, slot count, right-sibling pointer.
Leaf:   (key, tid) sorted by key; tid points into heap pages.
Internal: (key, childPageId) routing entries.

7.2 Heap (Postgres) vs clustered (InnoDB)

Postgres stores tuples in an unordered heap with the B-tree storing TIDs into it; InnoDB clusters the row in the leaf of the primary B-tree. Clustered indexes give cheaper PK scans but expensive secondary indexes (which must store the PK and re-traverse).

7.3 Splits and merges

On insert into a full leaf, allocate a new sibling, split keys, push the median up. Latch-couple from root downward, releasing parent latches when the child has room (the classic "safe node" optimization). Right-sibling pointers (B-link tree) let concurrent readers traverse without holding internal latches (DDIA Ch. 3, "Concurrency control").

8. Detailed component design

8.1 Buffer pool

Hash table pageId -> frame; replacement via clock-sweep or LRU-K. Each frame has pin count and dirty bit. Background bgwriter trickles dirty pages to disk to keep checkpoints short. Double buffering protects against torn pages: Postgres writes a full-page image to WAL on first dirty after checkpoint; InnoDB uses the doublewrite buffer.

8.2 Write-ahead log (ARIES)

Every page update writes a redo (and optionally undo) record carrying the LSN. The page header's pageLSN >= log LSN guarantees the page reflects that record on disk. Recovery: scan from checkpoint, build the dirty-page table and transaction table, redo all records whose LSN > pageLSN, then undo losers with CLRs to make undo idempotent.

8.3 Concurrency control

  • Latches protect physical structure (pages); short-lived, OS mutex-like.
  • Locks protect logical objects (rows); 2PL ensures serializability.
  • MVCC: Postgres keeps multiple row versions in heap (HOT chains); InnoDB keeps the latest in place + undo segments. Snapshot reads do not block writers — DDIA Ch. 7 "Snapshot Isolation."

8.4 Vacuum / purge

MVCC accumulates dead tuples. Postgres VACUUM scans pages, marks dead tuples free, and updates the visibility map; InnoDB purge thread truncates undo segments past the oldest snapshot. Tuning autovacuum is the single most common Postgres production task.

8.5 Group commit

Multiple transactions piggyback one fsync of the WAL; commit latency rises slightly but throughput multiplies 10-50x on rotational/cloud disks.

9. Scaling & bottlenecks

  • WAL contention: single sequential file becomes the bottleneck. Mitigation: bigger WAL buffers, group commit, NVMe with stable cache.
  • Right-edge insert hotspot: monotonic keys (timestamps, auto-inc) cause all inserts on one leaf. Mitigation: key randomization, hash partitioning.
  • Long-running transactions: block vacuum, bloat MVCC chains. Monitor oldest_xmin.
  • Index bloat: non-HOT updates double-write index entries; rebuild concurrently.
  • Buffer pool eviction storms: large analytical scans pollute LRU; use ring buffers or hint BUFFER_USAGE_LIMIT.

12. Talking points for the interviewer

  • "I'd default to a B-tree for OLTP because in-place updates keep read amp constant and ARIES recovery is well-understood — the LSM win only shows up when ingest dominates."
  • "The trickiest correctness story is torn pages on 8 KB writes vs 4 KB sectors; I'd require either FPW in the WAL or a doublewrite buffer."
  • "Under contention I'd reach for B-link tree latching plus optimistic descent; lock-coupling alone limits scalability above ~32 cores."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
B+-tree (vs B-tree) All values live in leaves connected by right-sibling pointers, so range scans are sequential I/O while internal nodes stay small enough to cache. DDIA Ch. 3 highlights this layout as the OLTP workhorse: 4 levels for 1B rows with fan-out ~400. Plain B-tree (values in internal nodes balloon node size, hurts fan-out); LSM (better write throughput but higher read amp; wrong default for point-and-update OLTP).
8 KB / 16 KB fixed pages Aligns with filesystem block size and OS page cache, makes torn-write detection tractable, and gives stable fan-out for capacity math. Postgres 8 KB / InnoDB 16 KB are the canonical defaults DDIA references when sizing buffer pools. Variable-size pages (complicate buffer pool slot management); 4 KB (lower fan-out, more tree levels); 64 KB (wastes buffer pool slots for sparse leaves).
ARIES recovery (analysis/redo/undo + CLRs) The only recovery algorithm in production use that handles partial rollbacks, repeated crashes during recovery, and fine-grained locking simultaneously (DDIA Ch. 3 "Making B-trees reliable"). CLRs make undo idempotent so a crash mid-undo replays cleanly. Shadow paging (LMDB-style — simple but doubles write cost and breaks fine-grained locking); no-undo logging (forces no-steal, kills buffer manager flexibility).
Buffer pool with clock-sweep / LRU-K Caches hot pages in RAM so 99% of OLTP requests never hit disk. Clock-sweep approximates LRU at O(1) without a global lock; LRU-K (Postgres) resists scan pollution by counting K-th-most-recent access. Pin counts and dirty bits coordinate with bgwriter and checkpoints. Pure LRU (mutex on every access — bottleneck above 32 cores); FIFO (poor hit rate); page cache only (loses pin/latch semantics required for ARIES).
Latch coupling + B-link right-sibling pointers Latches are short-lived OS-level mutexes that protect physical page integrity during traversal. B-link pointers (Lehman-Yao) let a reader follow a right-link if the page split mid-descent, so internal latches can be released early — the standard way to scale concurrent readers past lock-coupling limits. Lock-coupling alone (every descent holds parent latch — caps concurrency); copy-on-write tree (LMDB — write serialization bottleneck).
Two-level concurrency: latches (physical) + locks (logical) Latches protect the page; locks protect the row/tuple. This separation is what DDIA Ch. 7 calls out: 2PL on locks gives serializability while latches stay short — long-held latches would tank throughput. Single locking layer (forces row locks to be held during page traversals — disastrous); lock-free tree (research-grade, no production engine ships it for OLTP).
MVCC via heap version chains (Postgres) or undo segments (InnoDB) Snapshot isolation without readers blocking writers — the central DDIA Ch. 7 win for OLTP. Postgres keeps multiple row versions in heap with HOT chains; InnoDB keeps the latest in place plus an undo log for older versions, which makes vacuum cheaper but secondary index updates more expensive. Pure 2PL no MVCC (readers block writers, kills mixed workloads); SI in WAL only (cannot answer reads without log replay).
Full-page writes (Postgres) / doublewrite buffer (InnoDB) 4 KB sectors are atomic on NVMe but 8/16 KB pages are not — a crash mid-write produces a torn page. FPW logs the entire page on first dirty after checkpoint; doublewrite writes the page to a scratch area first. Either survives torn writes that ARIES redo otherwise cannot repair. No torn-page protection (silent data corruption on crash); per-write fsync of the page file (defeats sequential WAL).
Group commit on the WAL Multiple transactions piggyback one fsync, multiplying throughput 10-50x on cloud disks where fsync is ~1 ms. The pattern DDIA Ch. 3 references for keeping commit latency floor near disk-fsync time without serializing commits. Per-transaction fsync (caps tps at 1/fsync); async commit (loses durability — different contract).
Checkpoints with bgwriter trickle flush Bounded recovery time = WAL volume since last checkpoint. Trickling dirty pages between checkpoints (Postgres bgwriter, InnoDB page cleaner) avoids the I/O storm of flushing the entire dirty set at once, keeping commit latency stable. Big-bang flush at checkpoint (latency cliff every N minutes); never-checkpoint (replay time grows unbounded).
Heap (Postgres) vs clustered index (InnoDB) For this design we choose heap to keep secondary indexes cheap (they store TIDs, not the PK + payload). DDIA Ch. 3 notes the trade-off: clustered wins PK range scans but every secondary index becomes a two-step lookup that re-traverses the primary tree. Clustered index (cheaper PK scans but expensive secondary indexes — penalty grows with index count); IOT-only (Oracle) — same trade-off, less flexibility.
Vacuum / autovacuum MVCC accumulates dead tuples; without reclamation, tables bloat indefinitely. Autovacuum scans pages, marks dead tuples free, updates the visibility map (which lets index-only scans skip heap probes). Tuning autovacuum is the #1 Postgres production task DDIA implies but doesn't enumerate. Manual VACUUM only (gets forgotten — bloat outpaces operators); aggressive purge thread always-on (consumes CPU/IO budget that OLTP needs).
Serializable Snapshot Isolation (SSI) Postgres Ch. 7 SSI tracks rw-conflict edges between snapshot transactions and aborts one if a dangerous cycle could form — getting full serializability with the throughput of SI. The right choice when correctness requires no anomalies but workload is mostly read-heavy. 2PL serializable (readers block writers, throughput collapses); SI without SSI (allows write-skew and phantom anomalies).
NVMe with stable write cache (or BBU/PLP) WAL fsync is the commit-latency floor; cheap NVMe with volatile DRAM cache lies about durability on power loss. Power-loss-protected enterprise NVMe (or RAID with battery-backed cache) makes fsync honest at ~50-200 us — the assumption every section of this design rests on. Consumer NVMe (silent data loss on power cut); HDD with NVRAM controller (slower but acceptable for cold tiers).
