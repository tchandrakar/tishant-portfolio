Design an LSM-tree Storage Engine (memtable -> SSTable -> compaction)

Hard database-ddia storage-cache

1. Problem statement

Design the storage engine that powers LevelDB, RocksDB, Cassandra, HBase and ScyllaDB: a Log-Structured Merge tree. Writes go to an in-memory mutable buffer (memtable) plus a sequential write-ahead log; once the memtable fills, it is flushed as an immutable, sorted, on-disk file (SSTable). A background compactor merges SSTables to bound read amplification and reclaim space from overwrites/tombstones. The interview asks you to defend every choice: why log-structured, why sorted, leveled vs size-tiered compaction, how to keep p99 reads bounded, how crash recovery works, and how to tune for read-heavy vs write-heavy workloads (DDIA Ch. 3, "SSTables and LSM-Trees").

2. Functional requirements

  • Single-node embedded KV: put(k,v), get(k), delete(k), scan(start,end).
  • Crash-safe: any acknowledged write survives a process or power failure.
  • Snapshot reads (MVCC) and atomic batch writes.
  • Online compaction without blocking foreground writes.

3. Non-functional requirements

  • Write throughput: 200k+ ops/s/node on NVMe; sequential I/O dominates.
  • Read latency: p50 < 1 ms, p99 < 10 ms with bloom filters and block cache.
  • Durability: fsync on commit (or group commit) for ACK==durable.
  • Space amp: < 1.3x logical size in steady state under leveled compaction.

4. Capacity estimation

  • Memtable: 64 MB; flush rate at 200 MB/s sustained writes = one flush/320 ms.
  • WAL: ~4 GB/min sustained; rolled at memtable flush.
  • SSTables on disk: 6 levels, fan-out 10 -> L0 = 4 files, L1 = 256 MB, L6 = 256 GB.
  • Bloom filter: 10 bits/key -> 1% FPR; 1 B keys = 1.25 GB RAM/replica.

5. API design

put(key, value, [seqNo])           -> ok
get(key, [snapshot])               -> value | NotFound
delete(key)                        -> writes a tombstone
scan(startKey, endKey, [snapshot]) -> iterator
writeBatch([(op,k,v)...])          -> atomic; one fsync

6. High-level architecture

SST] (non-overlapping, 10x size) L2 [SST ............] ... L6 [SST ..............................] Read path: memtable -> imm-memtable(s) -> L0 (newest first) -> L1..L6 (binary-search per level via manifest) -> bloom filter prunes >99% of misses.

Walk-through: a write hits the WAL (a single sequential fsync), then the in-memory skiplist. When the skiplist exceeds the threshold it is sealed, a new one takes over, and a flusher writes the sealed copy as an SSTable into L0. A separate compactor merges files from level L_n with overlapping files in L_{n+1}, producing fresh non-overlapping SSTables and dropping superseded versions/tombstones (DDIA Ch. 3, "Performance optimizations").

7. Data model & storage

7.1 SSTable layout

+----------------+----------------+--------+--------+--------+--------+
| Data Block 0   | Data Block 1   |  ...   | Index  | Filter | Footer |
+----------------+----------------+--------+--------+--------+--------+
Data Block: sorted (key,seq,type,value), prefix-compressed, ~32 KB, CRC.
Index:      (last_key_of_block -> block_offset), binary searchable.
Filter:     bloom or ribbon filter per block.
Footer:     magic, index offset, filter offset, version.

7.2 Manifest & versions

A "manifest" log records edits: file added/removed at level N, smallest/largest key, sequence range. Recovery replays the manifest plus the WAL. Snapshot reads pin a sequence number; the iterator skips entries with seq > snap, giving MVCC without copy-on-write of pages.

8. Detailed component design

8.1 Memtable

Concurrent skiplist (LevelDB) or hash+skiplist hybrid (RocksDB) for O(log n) inserts and ordered scans. Inline-allocate keys/values into an arena to avoid per-insert malloc.

8.2 WAL & group commit

Each write appends (seq, op, k, v) records to the active WAL segment. Multiple writers batch under a leader (group commit) so one fsync amortizes hundreds of writes — critical for synchronous durability throughput.

8.3 Compaction strategies

  • Leveled (RocksDB default): low space amp, higher write amp (~10-30x). Best for read-heavy, mostly-static datasets.
  • Size-tiered (Cassandra default): low write amp, higher space amp (up to 2x). Best for write-heavy, churn-light workloads.
  • Tiered+leveled hybrid (Universal): RocksDB option; few large files, good for time-series.

8.4 Read path optimizations

Bloom filter per SSTable cuts negative lookups to ~1 disk read. Block cache (LRU/CLOCK) keeps hot blocks in RAM. Prefix bloom + partitioned index help range scans. Hyper-clock cache (RocksDB 7+) removes mutex contention at high concurrency.

8.5 Tombstones & deletes

A delete is a write of a tombstone with the latest sequence. Tombstones are dropped only when the compactor proves no older live entry can exist below — i.e., the file participates in a compaction touching the bottom level. Range tombstones (RocksDB) collapse mass deletes.

9. Scaling & bottlenecks

  • Write stalls: if L0 grows too tall, foreground writes are throttled. Mitigations: more flush threads, larger memtable, sub-compaction parallelism.
  • Compaction CPU: dominant cost on NVMe; pin compaction to a thread pool sized to cores - foreground.
  • Read amp on point-lookups: tuned by bloom-bits/key and pinning L0 metadata in memory.
  • Space amp during compaction: need ~1x the file size as transient free space. Plan disks accordingly.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Concurrent skiplist memtable (LevelDB/RocksDB) O(log n) ordered inserts with lock-free readers and a single CAS on writes — exactly what DDIA Ch. 3 prescribes for the in-memory mutable layer because flushes need a sorted iterator and point lookups must be fast on the hot path. Arena allocation amortizes alloc cost. Red-black tree (rebalances cause O(log n) cache-unfriendly pointer chases, harder to make concurrent); hash table (no ordered scan, breaks SSTable flush invariant).
Sorted SSTable file format with prefix-compressed blocks The DDIA Ch. 3 SSTable contract — sorted, immutable, sparse-indexed — enables binary search inside ~32 KB blocks, prefix compression of adjacent keys, and merge-sort compaction in linear time. Immutability also makes the file safely shareable across snapshots and compactions. Unsorted log segments (force full scan on read, no merge-step efficiency); per-key files (filesystem inode pressure, no compression).
Leveled compaction (RocksDB default) For mostly-static read-heavy datasets, leveled keeps space amplification near 1.1x and bounds reads to one SSTable per level (DDIA "size-tiered vs leveled"). Each level holds non-overlapping files so a key lives in at most one file per level. Size-tiered (Cassandra default) — up to 2x space amp and bursty p99 due to large merges; Universal/tiered — fine for time-series but yields huge files that hurt random reads.
Bloom filter (10 bits/key, ~1% FPR) per SSTable Reduces negative lookups from O(levels) disk reads to ~1, which is decisive when the working set spills past block cache. DDIA Ch. 3 calls this out as the single highest-leverage LSM read optimization. Cuckoo filter (better FPR but harder to build incrementally during flush); Ribbon filter (lower bits/key, slower to build — RocksDB ships it as opt-in).
WAL with group commit Sequential append + one fsync per batch turns synchronous durability from a per-write fsync (~50-200 us each) into ~100k ops/s on NVMe. Mirrors the Helland/Mohan group-commit pattern that DDIA Ch. 3 references when discussing reliability of B-trees and LSM-trees alike. fsync-per-write (kills throughput); async/no-fsync (violates the durability contract on power loss).
Manifest log of version edits A separate append-only log of (file added, file removed, smallest/largest key, seq range) per level lets recovery reconstruct the LSM "version" atomically without scanning every SSTable footer. This is how LevelDB/RocksDB implement crash-consistent compaction handoff. Reading every SSTable's metadata at startup (slow with millions of files); embedding state in filenames (race-prone, no atomic multi-file change).
Block cache (LRU/CLOCK or hyper-clock) Pins decompressed hot blocks in RAM so reads avoid both decompression CPU and disk I/O. Hyper-clock (RocksDB 7+) removes the LRU mutex that becomes the bottleneck above ~32 cores — directly addresses the read-amp tail latency DDIA flags for LSMs. OS page cache only (caches compressed blocks, repeats decompression each access); no cache (catastrophic at p99).
Sequence numbers + tombstones for MVCC A monotonic 64-bit seq stamps every write; snapshot reads pin a seq and the iterator skips entries with seq > snap. This delivers DDIA Ch. 7 snapshot isolation without page-level copy-on-write and lets compaction safely drop superseded versions only when they fall below the oldest live snapshot. In-place updates (kills LSM's append-only shape); per-row version chains (hurts scan locality, complicates compaction).
Range tombstones (RocksDB) Mass deletes encoded as one (start, end, seq) record instead of N point tombstones, avoiding compaction blow-up when deleting a million-key range. Critical for time-series TTL workloads. Point tombstones only (each deleted key carries its own marker until bottom level — huge write amp); table-level drop (only works if the range aligns with file boundaries).
Sub-compaction parallelism + thread-pool isolation Splits a single L_n -> L_{n+1} compaction by key range across cores so compaction CPU keeps up with NVMe write throughput. Pinning compaction to its own pool prevents foreground writes from starving (a common cause of L0 stalls flagged in §9). Single-threaded compaction (cannot saturate NVMe — DDIA acknowledges compaction CPU as the modern bottleneck); shared thread pool (compaction blocks foreground reads).
CRC32C on every block Detects silent disk corruption and torn writes during read; CRC32C has hardware acceleration on modern x86/ARM so the check is essentially free per block. Required because LSM files are read months after they were written and may have been touched by firmware bugs or bit rot. No checksum (silent corruption surfaces as wrong query results); xxHash (faster but no SSE4.2 instruction, and the compatibility ecosystem standardized on CRC32C).
Partitioned index + prefix bloom (range scans) Plain bloom filters help only point lookups. Partitioned indexes load only the index sub-block needed for a scan window, and prefix bloom skips SSTables whose key prefix is absent — both keep range-scan latency bounded as the LSM grows. Single monolithic index per file (memory hog at TB scale); no prefix bloom (every range scan probes every level).
Direct I/O (O_DIRECT) for compaction reads/writes Compaction streams hundreds of MB/s; routing it through the page cache evicts hot user blocks (cache pollution). O_DIRECT keeps the block cache for reads and lets compaction use its own buffer pool. Buffered I/O (cache pollution destroys p99 reads during compaction); posix_fadvise(DONTNEED) (advisory, kernel may ignore).
WAL segment recycling Pre-allocated, zero-filled segments avoid filesystem metadata churn on every append, giving stable fsync latency. Closing a segment after memtable flush makes truncation an O(1) unlink rather than a scan. Single growing log file (fsync latency includes filesystem extension cost); per-write file (inode storm).
