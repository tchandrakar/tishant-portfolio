Design Consensus (Raft and Paxos walkthrough)

Very Hard database-ddia

1. Problem statement

Design a fault-tolerant consensus protocol that lets a cluster of N nodes agree on an ordered log of commands despite up to f = floor((N-1)/2) crash failures and an asynchronous, lossy network. This is the substrate of etcd, ZooKeeper (ZAB), Consul, CockroachDB, TiKV, Spanner, MongoDB elections, Kafka KRaft, and every modern strongly-consistent distributed system. The interview wants two artifacts: a precise Raft walkthrough (leader election, log replication, safety, membership change) and a contrast with classic Paxos / Multi-Paxos (DDIA Ch. 9 "Consensus" and Ch. 8 "Trouble with Distributed Systems").

2. Functional requirements

  • Replicated state machine semantics: every replica applies the same commands in the same order.
  • Linearizable reads and writes against the cluster.
  • Tolerate f crash failures with N = 2f+1 nodes; remain safe under arbitrary network partitions.
  • Online membership change without downtime.

3. Non-functional requirements

  • Safety: at most one decision per log slot, ever, even across leadership changes.
  • Liveness: progress when a majority is up and can communicate within ~election timeout.
  • Commit latency: 1 RTT to a majority; ~1-3 ms within a DC, 50-150 ms cross-region.
  • Throughput: 10k-100k ops/s/group with batching+pipelining.

4. Capacity estimation

  • 5-node cluster tolerates 2 failures; 7-node tolerates 3 (diminishing returns due to majority size).
  • Per-op fsync at every replica; batch 100-1000 ops per fsync to amortize.
  • Log storage = state-machine command rate * retention until snapshot.
  • Snapshot every ~100 MB of log; compaction reclaims pre-snapshot entries.

5. API design

6. High-level architecture

clients | v +---------------+ +----------+ | Leader (T=7) | -----> | Follower | AppendEntries(prev, e[]) | | -----> | Follower | | log: [...] | -----> | Follower | +---------------+ +----------+ ^ ^ | | RequestVote on timeout Candidate (term T=8) States: Follower --timeout--> Candidate --majority--> Leader | | +--- higher term ------+ Log: index : 1 2 3 4 5 6 7 ... term : 1 1 2 3 3 3 4 cmd : x y z a b c d commit at majority >= index AND entry term == currentTerm.

Walk-through: Raft elects a leader for a term; the leader is the sole source of log entries. Followers append, the leader commits an entry once a majority has it (in the leader's current term — the famous safety bug fix in figure 8 of the Raft paper). Committed entries are applied to the state machine in order and acknowledged to clients.

7. Data model & storage

7.1 On-disk Raft state

persistent: currentTerm, votedFor, log[]   (must fsync before reply)
volatile  : commitIndex, lastApplied
leader    : nextIndex[i], matchIndex[i]
log entry : (index, term, command, type)

7.2 Snapshots

State machine periodically dumps its state and the included (lastIdx, lastTerm). Log prior to that index is truncated. Slow followers receive InstallSnapshot RPCs to catch up — essential for membership changes and recovery from long partitions.

8. Detailed component design

8.1 Leader election

Each follower starts a randomized election timeout (e.g., 150-300 ms). On expiry it bumps term, votes for itself, requests votes. A vote is granted iff (a) the requester's term >= ours, (b) we haven't voted in this term, and (c) the requester's log is at least as up-to-date as ours (last-log-term then last-log-index). Majority wins. Randomization breaks split votes.

8.2 Log replication & safety

Leader assigns the next index, appends locally, sends AppendEntries to all followers with (prevIndex, prevTerm) as a consistency check. Followers reject if their log doesn't match — leader decrements nextIndex and retries until it finds the divergence point, then overwrites the suffix. Safety invariants (Raft figure 3): election safety, leader append-only, log matching, leader completeness, state-machine safety. The non-trivial bit is "a leader may only commit entries from its current term" to avoid the figure-8 bug.

8.3 Linearizable reads

Three options: (1) commit a no-op via the log (1 RTT, simple), (2) ReadIndex — record commit index, send heartbeat to majority to confirm leadership, then wait for state machine to apply up to that index (no log write), (3) leases — leader proves it holds an unexpired lease and reads locally (clock-skew bound). Etcd defaults to ReadIndex.

8.4 Membership changes

Naive simultaneous swap can split a cluster into two majorities. Raft's solution is joint consensus: transition through C_old,new where decisions need majorities of both configs, then to C_new. Single-server changes are a simpler restricted case used in etcd.

8.5 Paxos contrast

  • Basic Paxos: two phases — Prepare(n) collects promises, Accept(n,v) commits a value for one slot.
  • Multi-Paxos: elect a stable distinguished proposer (effectively a leader) and skip Phase 1 for subsequent slots; otherwise indistinguishable from Raft on the wire.
  • EPaxos: leaderless, exploits commutativity, fast-path 1 RTT to fast quorum (3f/2 for f=2).
  • Flexible Paxos: Q1 (prepare) and Q2 (accept) only need to overlap, not be majorities — enables grid quorums.

Raft chose understandability over flexibility (as the paper title states); Paxos is more general but historically harder to implement correctly. ZAB (ZooKeeper) is a primary-backup variant tuned for state-machine replication.

8.6 Failure scenarios

  • Leader crash mid-replication: entries on a minority may be overwritten by a new leader with longer/equal log; safe by leader-completeness.
  • Network partition: a leader on the minority side cannot commit (no majority); the majority elects a new term and proceeds. The old leader steps down on receiving higher term.
  • Disrupting candidate: a partitioned node bumps terms continuously; PreVote (Raft extension) prevents term inflation.

9. Scaling & bottlenecks

  • Single-group throughput cap: one fsync chain per group; shard the keyspace into many Raft groups (TiKV, CockroachDB).
  • Cross-region commit latency: 1 RTT to majority. Mitigation: place majority in primary region, use witness/learner replicas elsewhere.
  • Heartbeat overhead: O(N) per tick; co-locate groups and batch heartbeats (MultiRaft).
  • Read-mostly workloads: use leases or follower reads with safe-time to scale reads horizontally.

12. Talking points for the interviewer

  • "I'd reach for Raft as the implementation, but I'd describe consensus abstractly as total-order broadcast — that framing makes 2PC, Spanner, and Kafka KRaft fall out as instances."
  • "The figure-8 bug is the gotcha — a freshly elected leader cannot commit old-term entries on its own, only by piggybacking a current-term entry."
  • "For geo deployments I'd shard with Multi-Raft and use Spanner-style read leases; pure single-leader Raft hits a wall around 30k ops/s/group on typical hardware."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Raft (over classic Paxos) The strong-leader model collapses several Paxos correctness conditions into one invariant ("only the leader appends") and gives an explicit leader election protocol. DDIA Ch. 9 explicitly notes that operational consensus systems converged on Raft because invariant-checking is straightforward, while Multi-Paxos is wire-equivalent but harder to reason about. Multi-Paxos (more flexible quorums, but ambiguous leader role makes implementation error-prone — see Chubby paper); EPaxos (1 RTT geo wins but rare in practice, complex conflict graph).
N = 2f+1 odd cluster size (typically 3 or 5) Majority quorum is the smallest reliable failure-detector under DDIA Ch. 8's asynchronous-network assumptions. 5 nodes tolerate 2 failures while keeping the majority small (3) for low commit latency; 7 gives diminishing returns because every commit waits for 4 acks. Even N (e.g., 4) — same fault tolerance as 3 but more commit cost; N=3 (only tolerates 1 failure — risky during rolling restarts).
Randomized election timeout (150-300 ms) Breaks split votes between simultaneous candidates without coordination — the elegant property that makes Raft elections terminate quickly. Range covers typical inter-DC RTT plus jitter so a healthy leader's heartbeat always preempts the timeout. Fixed timeout (split votes recur indefinitely); priority-based election (requires global priority oracle — chicken-and-egg).
PreVote extension A partitioned node otherwise increments its term on every timeout, so when it rejoins it disrupts the live leader by causing a step-down. PreVote checks if a real election would succeed before bumping term — fixes the "disrupting candidate" failure mode flagged in §8.6. Naive election (term inflation causes spurious leadership churn); pre-election barrier (custom solution, less battle-tested than PreVote).
Log matching + leader-completeness invariants The two safety properties from Raft figure 3: identical (index, term) means identical log prefix; an elected leader has every committed entry. Together they guarantee state-machine safety without per-entry consensus, simplifying the proof DDIA Ch. 9 references. Per-entry Paxos (correct but high-overhead); chain replication (simpler but reconfiguration is hard, no leader-driven catch-up).
Current-term commit rule (Raft figure 8 fix) A leader may not commit an entry from a prior term solely on majority replication — it must piggyback a current-term entry. This closes the famous figure-8 safety bug where an old entry could be silently overwritten after re-election. Naive "majority means committed" (the original bug — corrupts the replicated state machine on specific election sequences).
fsync of currentTerm, votedFor, log[] before reply These three fields must survive a crash or election safety breaks (a node could vote twice in the same term). DDIA Ch. 8's "unreliable storage" hammer: trusting OS page cache here is a correctness bug, not a performance issue. Async persistence (election safety violations on crash); separate file per field (more fsyncs per RPC — kills throughput).
Batching + pipelining of AppendEntries One fsync amortizes 100-1000 entries; pipelining lets the leader send the next batch before the previous batch is acked. Together they take a single Raft group from ~1k ops/s to 30k+ ops/s — the §9 throughput target. Per-entry RPC (fsync-bound at ~1k ops/s); batching only without pipelining (RTT-bound; cannot saturate disk).
ReadIndex for linearizable reads Records the current commit index, sends a heartbeat to confirm leadership, then waits for state-machine apply — no log write, no fsync per read. The etcd default and the right balance between correctness (no stale reads) and throughput (no extra log entries). No-op log entry per read (1 fsync per read — kills read throughput); leader leases (faster but require bounded clock drift — DDIA Ch. 8 warns); follower reads without safe-time (stale).
Leader leases (with bounded clock drift) A timed lease lets the leader serve reads locally without a heartbeat round-trip — order-of-magnitude read throughput win when clock infrastructure is solid. Falls back to ReadIndex when the lease is questionable. Pure ReadIndex (1 RTT per read — caps geo read throughput); leases on best-effort clocks (correctness violation under skew — DDIA Ch. 8).
Joint consensus (C_old,new) for membership change The naive simultaneous swap can produce two disjoint majorities under network partition, both committing different values. Joint consensus requires a majority of both old and new configs during the transition — Raft's safety-preserving membership change. Stop-the-world reconfiguration (requires downtime); single-server changes only (limits operational flexibility — though etcd uses this restricted form).
Snapshots with InstallSnapshot RPC Without snapshots, the log grows unbounded and slow followers can never catch up. Snapshotting the state machine and truncating the log keeps storage bounded; InstallSnapshot replaces a stale follower's state in one transfer instead of replaying GBs of log. No snapshots (unbounded log growth — operational blowup); copy-on-write state-machine snapshots only (fine for in-memory state, hard for disk-resident KVs).
Multi-Raft / sharded groups (TiKV / CockroachDB pattern) A single Raft group is bottlenecked by one fsync chain. Sharding the keyspace into thousands of independent groups, each with its own Raft, scales linearly. The §9 mitigation for the single-group throughput cap. Single huge Raft group (caps cluster throughput at one group's rate); EPaxos (leaderless balances load but conflict resolution is complex at scale).
Witness / learner replicas (non-voting) Geo deployments need replicas in remote regions for read locality and disaster recovery, but voting replicas in remote regions add their RTT to every commit. Non-voting learners receive the log without participating in quorum — the standard Spanner / CockroachDB pattern for cross-region reads. All voting replicas across regions (every commit pays max-region RTT); async DC-level shipping outside Raft (loses linearizability for that region's reads).
