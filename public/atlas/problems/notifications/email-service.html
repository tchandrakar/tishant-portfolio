Design Email Service (transactional + marketing at scale, SES-style)

Hard notifications

1. Problem statement

Build an SES-style multi-tenant email platform that sends both transactional (password reset, receipts) and marketing (newsletters, campaigns) email. The system must protect sender reputation, manage IP warm-up, handle bounces/complaints, support DKIM/SPF/DMARC signing, template rendering with personalization, and scale to billions of messages per day across thousands of tenants.

2. Functional requirements

  • Send a single transactional email or a campaign (millions of recipients).
  • Template management with Handlebars-style variables and localization.
  • Bounce, complaint, open, click tracking via webhooks and event streams.
  • Suppression lists per tenant (hard bounces, unsubscribes, complaints).
  • IP pool management: dedicated, shared, warm-up pools.
  • DKIM signing per sending domain; SPF/DMARC validation guidance.

3. Non-functional requirements

  • Latency: transactional accept-to-handoff under 1 s p95; campaign jobs may take minutes.
  • Scale: 5B emails/day, 60k/sec avg, 250k/sec peak across 50k tenants.
  • Availability: 99.95%; durable accept even when MTAs degrade.
  • Deliverability: bounce rate under 5%, complaint rate under 0.1%.

4. Capacity estimation

  • QPS: 5B/86,400 ≈ 58k avg; campaign bursts to 250k/sec.
  • Storage: 5B msgs × 2 KB metadata = 10 TB/day; 90-day retention ≈ 900 TB.
  • Body storage: 5B × 20 KB avg = 100 TB/day in S3; lifecycle to Glacier after 30d.
  • Event volume: opens+clicks ≈ 30% of sent = 1.5B events/day = 17k events/sec.
  • IP pool: 250k/sec ÷ 100 msg/IP/sec rate cap = 2500 sending IPs.

5. API design

POST /v1/email/send         {from,to,subject,html,text,headers}        -> messageId
POST /v1/email/template     {name, html, text, variables}
POST /v1/email/campaign     {templateId, audienceId, scheduleAt}        -> jobId
GET  /v1/email/events?messageId=...                                     -> [open,click,bounce]
POST /v1/suppressions       {tenantId, email, reason}

6. High-level architecture

[Tenant API] -> [Validation/Rate-limit] -> [Kafka: email.accepted] | v [Renderer Worker] (templates + body S3) | v [Kafka: email.ready (sharded by domain)] | v [MTA Pool] --(SMTP)--> [Recipient MX] | v [Bounce/FBL Listener] -> [Kafka: events] | v [Suppression Store] / [Tenant Webhooks]

Acceptance is decoupled from delivery via Kafka so spikes don’t block tenants. Renderers expand templates and offload large bodies to S3. MTAs are sharded by recipient domain to allow per-domain rate control (Gmail/Yahoo/Outlook each have distinct throttles).

7. Data model & storage

7.1 Message metadata (Cassandra)

CREATE TABLE messages (
  tenant_id text, message_id timeuuid,
  to_addr text, from_addr text, status text,
  body_s3 text, sent_at timestamp,
  PRIMARY KEY ((tenant_id, day_bucket), message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

7.2 Suppression list (DynamoDB / RocksDB-backed KV)

PK = (tenant_id, sha256(email))   value = {reason, added_at}

7.3 Why these stores?

Cassandra absorbs 60k+ writes/sec with multi-DC replication and supports day-bucket pagination for tenant dashboards. Suppression lookups are point reads on the hot path and need sub-ms latency, so a managed KV with in-memory cache fits.

8. Detailed component design

8.1 Renderer

Stateless workers pull from email.accepted, resolve templates via a versioned cache (Redis, 30s TTL), expand variables, sign with DKIM, and write final RFC 5322 body to S3. Output is a small pointer message on email.ready.

8.2 MTA pool & reputation

Each MTA owns a subset of IP addresses. A scheduler routes messages to (IP, recipient-domain) lanes with token buckets configured from learned rate caps (e.g., Gmail 200/sec/IP for new senders). Warm-up follows a 30-day curve doubling daily volume. Per-IP reputation is tracked and bad IPs are auto-paused.

8.3 Feedback loop

Bounces arrive via SMTP (synchronous) and async via DSN/FBL mailboxes. Listeners parse, classify (hard/soft/transient), and update suppression. Hard bounces auto-suppress; soft bounces retry with exponential backoff up to 72h.

9. Scaling & bottlenecks

  • Domain throttling: a campaign hitting 10M Gmail addresses must respect Gmail’s per-IP cap → spread across 500 IPs over 2 hours.
  • Reputation poisoning: one tenant’s spam tanks shared-pool delivery; isolate suspect tenants to quarantine pool.
  • Renderer fan-out: personalized campaign of 50M recipients → shard by recipient hash; render at 50k/sec across 200 workers.
  • Event ingestion: 17k events/sec into Kafka; downstream tenant webhooks must not block — buffer and retry with DLQ.

12. Talking points for the interviewer

  • "I'd separate transactional and marketing onto different Kafka clusters and IP pools because reputation cross-contamination is the #1 deliverability killer."
  • "The hardest part is per-recipient-domain rate control — I'd model it as a hierarchical token bucket (tenant → IP → domain)."
  • "For an MVP I'd skip click-tracking and use an off-the-shelf MTA like Haraka, then build the in-house MTA when reputation tooling matters."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Haraka / Postfix MTA poolSending 5B emails/day = 250k peak/sec across 2500 IPs requires a battle-tested SMTP daemon with per-IP queue isolation; Haraka's plugin model lets us hook DKIM signing and per-domain throttling without forking core code.SendGrid SMTP relay (margin-killing per-msg cost at 5B/day); building a new MTA from scratch (years of RFC 5321 edge cases).
DKIM signing (RSA-2048 / Ed25519)Receiver MTAs (Gmail, Yahoo) hard-bounce unsigned mail under DMARC reject; per-tenant private keys signed at the renderer authenticate ownership of the sending domain to keep our 5% bounce SLO.No DKIM (catastrophic for deliverability); shared global key (one tenant compromise leaks everyone).
SPF DNS recordsTenants publish v=spf1 include:_spf.ourservice.com ~all so receiving MTAs verify our 2500 sending IPs are authorized; pairs with DKIM under DMARC alignment to clear inbox-vs-spam thresholds at Gmail/Yahoo/Outlook.SPF-only (no message integrity check); DKIM-only (subdomain spoofing risk).
HSM (CloudHSM / KMS) for DKIM keysRSA-2048 sign at 60k/sec costs ~3 cores per worker; offloading to FIPS-140-2 HSM both protects the per-tenant private keys and frees CPU for template rendering at 50k/sec across 200 workers.In-process software signing (CPU-bound at scale, key exposure on host compromise); shared key in Vault (per-msg latency too high).
Apache Kafka (email.accepted, email.ready, events)60k avg / 250k peak msgs/sec needs a partitioned log; partitioning email.ready by recipient domain lets Gmail-bound traffic flow at Gmail's per-IP cap independently of Yahoo-bound traffic, and 1.5B events/day (17k/s) of opens/clicks land on the same backbone.RabbitMQ (per-msg ack at 250k/sec overwhelms broker); Kinesis (shard rebalance lag during campaign spikes).
S3 (rendered RFC 5322 bodies)5B × 20 KB = 100 TB/day of bodies; S3 at $0.023/GB/month with lifecycle to Glacier after 30 days is 10× cheaper than keeping bodies in Cassandra and offloads body durability to AWS's 11-9s.Cassandra blob storage (write amplification kills cluster); EBS volumes (no inter-AZ durability).
Cassandra (messages metadata)5B writes/day = 60k+/sec sustained; partitioning by (tenant_id, day_bucket) bounds partition size to ~10M rows even for the largest tenant and supports tenant dashboard pagination on message_id DESC.Postgres (60k/sec writes saturate WAL on a single primary); MongoDB (range scans by day are inefficient on hashed shards).
DynamoDB / RocksDB KV (suppression list)Every send checks suppression on the hot path — 250k/s point reads at sub-ms latency; SHA256(email) keys with auto-sharded throughput are exactly the workload DynamoDB is built for.Bloom filter only (false positives drop legitimate mail); SQL table (250k QPS point reads needs heavy caching anyway).
Redis (template cache, 30s TTL)Renderer pulls templates 60k+/sec; a 30-second hot cache with versioned keys serves >99% of reads from RAM and lets template publishers iterate without DB stampedes.In-process cache only (incoherent across 200 renderer workers); no cache (template DB QPS would explode).
Hierarchical token bucket (tenant → IP → domain)Gmail caps new senders at 200/sec/IP; a 3-level bucket enforces tenant rate, IP capacity, and domain politeness simultaneously so a 10M-Gmail-recipient campaign spreads across 500 IPs over 2h instead of bombing one IP into the spam folder.Single global bucket (kills per-domain throttling); fixed-window counters (boundary spikes blacklist IPs).
Dedicated vs shared IP poolsReputation isolation: high-volume tenants (>1M/day) get dedicated IPs to control their warm-up curve, while low-volume tenants share warm pools to avoid a 30-day cold-start; quarantine pool isolates suspect tenants from poisoning shared reputation.All-shared (one bad tenant tanks deliverability for 50k others); all-dedicated (cold IP warm-up takes 30 days, unusable for new tenants).
Bounce/FBL listener (DSN + ARF mailboxes)Hard bounces hit ~3% at our scale and must auto-suppress to stay under the 5% bounce SLO; ARF feedback loop reports from Gmail/Yahoo/Outlook flag complaints that drive the <0.1% complaint SLO.Synchronous SMTP-only (misses async DSN bounces hours later); manual review (250k/s makes it impossible).
Apache Flink / Kafka Streams (event ETL)17k events/sec of opens+clicks need windowed aggregation for tenant dashboards; stateful stream joins on message_id correlate sends → opens → clicks without batch ETL lag.Spark batch (15-minute lag breaks "live" dashboards); Lambda + DynamoDB streams (cost prohibitive at 17k/s).
