Design Notification Scheduler (cron + delayed jobs at billion-scale)

Hard notifications misc

1. Problem statement

Build a durable scheduler for notifications: arbitrary delay (5 s to 1 year), cron-style recurring jobs, and timezone-aware schedules ("send at 9am local time"). It must fire on time within ±2 s p99, never lose a job, deliver exactly-once at the orchestrator boundary, and scale to 1B in-flight scheduled jobs across 100k tenants.

2. Functional requirements

  • Schedule one-shot job at fireAt (UTC or user-local).
  • Schedule recurring jobs with cron expressions and tz.
  • Cancel and reschedule before fire time.
  • Fan-out triggers to a target topic/queue with payload.
  • Idempotent firing — at-least-once with consumer-side dedup support.

3. Non-functional requirements

  • Accuracy: ±2 s p99, ±10 s p99.99 under failure.
  • Scale: 1B active jobs, 200k fires/sec peak.
  • Durability: jobs persisted with RF=3 across AZs; survive single-AZ loss.
  • Availability: 99.99% (clock service), 99.95% scheduling API.

4. Capacity estimation

  • Active jobs: 1B × 256 B = 256 GB; with index ~600 GB shareded across 64 nodes.
  • Fires: 200k/sec × 1 KB = 200 MB/s into trigger Kafka topic.
  • Time-bucket index: 1B jobs ÷ 86,400 s/day = ~12k jobs/sec average; bursts 10×.
  • Memory for next-minute window: 60 s × 200k/s = 12M jobs ≈ 3 GB → easily in RAM per shard.
  • Recurring expansions: 100M cron jobs × next-1-week materialization → 700M derived rows.

5. API design

POST /v1/schedule         {fireAt, target, payload, idempotencyKey}   -> jobId
POST /v1/schedule/cron    {cron, tz, target, payload}                 -> jobId
DELETE /v1/schedule/{id}
PUT  /v1/schedule/{id}    {fireAt}
GET  /v1/schedule/{id}                                                -> status, nextFireAt

6. High-level architecture

[Client] | v [Schedule API] --> [Job Store: Cassandra (sharded by job_id)] | v [Time-bucket Index] (per-minute partitions, sharded by user-hash) | v [Tickers per shard] -- watermark = wall-clock - skew | v [Fire Worker pool] -- emits to Kafka: timer.fired | v [Trigger Topic] -> [Orchestrator / Channel Services]

Storage is partitioned by job_id for cancel/lookup. A separate time-bucket index partitioned by minute-bucket and user-hash is the hot path: each shard owns a contiguous slice of buckets and a goroutine ticks every second to load the next-minute slice into memory and fire as wall-clock crosses each item’s fireAt.

7. Data model & storage

7.1 Job store (Cassandra)

CREATE TABLE jobs (
  job_id uuid PRIMARY KEY,
  tenant_id text, fire_at timestamp,
  cron text, tz text,
  target text, payload blob,
  status text, version int
);
CREATE TABLE bucket_index (
  bucket_minute timestamp, shard int, job_id uuid,
  fire_at timestamp,
  PRIMARY KEY ((bucket_minute, shard), fire_at, job_id)
);

7.2 Why bucket index?

Querying "all jobs firing in the next minute" is the engine’s critical query. Bucketing by minute keeps partition sizes bounded (200k peak ≤ 200k rows per (minute, shard) with shard count tuned to 64). DDIA Ch. 11 motivates time-windowed indexes for stream processing.

8. Detailed component design

8.1 Tickers and leases

Each shard is owned by exactly one ticker (Zookeeper / etcd lease, 30 s TTL). The ticker reads (bucket, shard) for the next two minutes into a min-heap keyed by fire_at. Every 100 ms it pops items with fire_at ≤ now and dispatches them. On lease loss it stops and another node takes over; the heap is rebuilt from the index, idempotent because Cassandra is the source of truth.

8.2 Cron expansion

A separate materializer walks recurring jobs and writes the next N=10 occurrences into bucket_index. A daily compaction trims expired rows. This avoids per-tick cron parsing.

8.3 Exactly-once at the boundary

At fire time, the worker writes (job_id, occurrence_id) to a fired table with a uniqueness constraint, then publishes to Kafka. Replays are detected by the dedup table; downstream consumers also dedupe on idempotencyKey. This implements the at-least-once-with-idempotent-consumer pattern from DDIA Ch. 11.

9. Scaling & bottlenecks

  • Hot minute: "9am Monday" creates a thundering herd. Add jitter at schedule time (uniform 0–60 s) per tenant policy.
  • Tail latency: shard reassignment on ticker crash must complete in <30 s; hot-standby tickers pre-load buckets.
  • Cron explosion: a tenant defining a per-second cron on 1M users generates 86M jobs/day. Enforce per-tenant cron-cardinality caps.
  • Clock skew: NTP-synced; reject schedules with fireAt < now - 5s as bad input.

12. Talking points for the interviewer

  • "I'd start with a single-region scheduler backed by Cassandra and add cross-region replicas in iteration 2; cross-region clock semantics get hairy fast."
  • "The hardest part is exactly-once firing across ticker failover; I'd use a fenced lease and a fired-dedup table."
  • "For very-long delays (months), I'd not keep them in the hot index; nightly job promotes near-due items into the bucket index."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.
