Design Multi-channel Notification Orchestrator (channel choice, dedup, prefs)

Hard notifications

1. Problem statement

Design the brain in front of push, email, SMS, and in-app channels. Given a logical event ("order shipped"), the orchestrator must pick the optimal channel(s) per user, respect quiet hours and channel preferences, deduplicate against recently sent notifications, throttle per category, and fall back across channels on non-delivery. Targets: 2B logical events/day across 200M users.

2. Functional requirements

  • Single API: notify(userId, eventType, payload) — orchestrator decides channels.
  • Per-user channel preferences and quiet hours (with timezone).
  • Per-category frequency caps (e.g., max 3 marketing/day).
  • Deduplication on (userId, eventKey) within configurable window.
  • Channel fallback: if push not delivered in N minutes, fall back to email.
  • A/B testing of channel-mix policies.

3. Non-functional requirements

  • Latency: orchestration decision under 100 ms p95.
  • Scale: 2B events/day, 23k events/sec avg, 100k/sec peak.
  • Availability: 99.95%; user prefs are critical and read-heavy.
  • Consistency: preference reads strongly consistent or near-RT.

4. Capacity estimation

  • QPS: 23k avg, 100k peak.
  • User pref store: 200M users × 1 KB = 200 GB; Redis hot cache 32 GB (top 10% active).
  • Dedup state: 2B events × 1 KB key/value × 24h = ~30 GB; Redis cluster.
  • Decision logs: 2B/day × 500 B = 1 TB/day → S3 + Parquet for offline analysis.
  • Engagement features: 200M users × 100 features × 4 B = 80 GB feature store.

5. API design

POST /v1/notify              {userId, eventType, payload, dedupKey, ttl}
PUT  /v1/users/{id}/prefs    {channels:{push:true,email:true,sms:false}, quietHours, locale, tz}
POST /v1/categories          {category, frequencyCap, fallbackOrder:[push,email,sms]}
GET  /v1/notify/{id}                                                  -> decision, attempts, status

6. High-level architecture

[Caller] -> [Orchestrator API] | v [Decision Engine] -- reads --> [Pref Cache: Redis] | [Feature Store: KV] | [Dedup Store: Redis SETNX] | +---------+----------+----------+ v v v v [Push Topic] [Email] [SMS Topic] [In-app Topic] | | | | v v v v Push Svc Email Svc SMS Svc Inbox Svc \________________________/ | v [Status Aggregator] -> [Fallback Scheduler] -> re-enqueue if needed

Each channel emits delivery events back to a status aggregator. The fallback scheduler is a delay-queue (Kafka + per-event timer or Redis sorted set keyed by deadline) that triggers next-channel attempts when a primary channel doesn’t confirm delivery within its grace window.

7. Data model & storage

7.1 Preferences (DynamoDB / Cassandra)

CREATE TABLE prefs (
  user_id text PRIMARY KEY,
  channels map<text, boolean>,
  quiet_start time, quiet_end time, tz text,
  category_overrides map<text, frozen<pref>>
);

7.2 Dedup & rate-limit (Redis)

SETNX dedup:{user}:{eventKey} 1 EX 86400
INCR ratelimit:{user}:{category}:{day} EX 86400

7.3 Decision log (Kafka -> S3 Parquet)

Every decision is logged with the inputs, chosen channels, and policy version for offline analysis and audit.

8. Detailed component design

8.1 Decision engine

Pipeline of stages: load prefs → dedup check → frequency-cap check → quiet-hour shift → channel selection → policy override. Each stage is a pure function with a typed input/output, enabling unit tests and policy versioning. Decisions are emitted to a Kafka topic before any channel send for replayability.

8.2 Channel selection policy

Default policy: push if device-active in 7d, else email, else SMS for high-priority only. Engagement-aware: a learned model scores channel effectiveness from the feature store and the engine picks argmax above a threshold. Bandit experiments are scoped per category.

8.3 Fallback scheduler

For events with fallbackOrder=[push,email] and grace=10 min: write a delayed message to Kafka via a Redis ZSET keyed by now + 600s. A poller scans due items every 1 s and re-runs decision with channel masked to the next in order. Cancel on success event.

9. Scaling & bottlenecks

  • Pref read amplification: 100k decisions/sec × 1 read = 100k Redis GETs/sec → use a near-cache (process-local LRU) with 10 s TTL to halve traffic.
  • Dedup hot users: a server emitting 1k events for one user/sec; shard dedup keys with user-bucket suffix to spread load.
  • Quiet-hour holdback: events queued for 8h sleep need a durable timer service; use Kafka log compaction + replay rather than huge Redis ZSETs.
  • Policy rollout: bad policy version can drop notifications; canary 1% then 5% then 100%.

12. Talking points for the interviewer

  • "I'd treat the orchestrator as a pure decision service that emits intent to Kafka — channel services are isolated and can degrade independently."
  • "The hardest part is dedup correctness across retries and channel fallbacks; I'd key on (userId, eventKey, channel) with a 24h TTL."
  • "Quiet hours look simple but timezone math is brutal — store IANA tz string per user and reject ambiguous times."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Apache Kafka (decision log + channel topics)2B events/day = 23k avg / 100k peak/sec must flow through the orchestrator without backpressuring callers; Kafka's partitioned log gives durable replay for policy debugging and decouples decision-making from per-channel send latency.RabbitMQ (per-msg ack throttle hurts at 100k/s); Pulsar (operational overhead unjustified vs Kafka here).
Redis cluster (preferences cache)100k decisions/sec × pref read = 100k Redis GETs/sec must clear under 1 ms; sharded Redis with 32 GB hot cache covers the top 10% active users (200 GB total prefs) and a process-local LRU near-cache halves backend traffic.Memcached (no native cluster mode for failover); in-process only (incoherent across orchestrator pods).
Redis SETNX (dedup store)Dedup window of 24h × 2B events/day = ~30 GB of keys; SETNX with EX=86400 atomically claims (userId, eventKey) so retries and channel fallbacks can't double-fire — a critical correctness primitive across the 100k/s peak.Postgres unique index (write contention at 100k/sec); Bloom filter (false positives drop legitimate notifications).
DynamoDB / Cassandra (preferences store)200M users × 1 KB = 200 GB read-heavy with strong-consistency requirement on opt-out; DynamoDB's strongly-consistent reads cost 2× but guarantee no notification after the user toggles off (regulatory + UX).Postgres (200 GB single-primary read scaling caps before 100k/sec); MongoDB (eventual consistency risks unwanted sends).
Redis ZSET (delay queue / fallback scheduler)Fallback grace window of 10 min × 100k/sec = 60M in-flight delayed jobs; Redis sorted sets keyed by deadline give O(log N) ZADD and O(1) ZPOP for the 1-second poller — Kafka delayed delivery isn't native and would need a separate delay topic per grace duration.Kafka per-delay-topic (topic explosion); Quartz / DB-backed timer (DB lock contention at 100k/s).
Process-local LRU cache (10s TTL)At 100k decisions/sec, the orchestrator pod farm is the bottleneck on Redis; a 10-second LRU on userId → prefs cuts Redis traffic by ~50% while staying within the "near-RT" preference-consistency NFR.No cache (Redis QPS doubles); 60-second TTL (post opt-out window too wide).
S3 + Parquet (decision audit log)2B/day × 500 B = 1 TB/day decision logs for offline policy evaluation, A/B analysis, and audit; columnar Parquet on S3 gives 10× compression and Athena/Presto-friendly scan for ad-hoc queries on policy version drift.Postgres OLAP (1 TB/day ingest melts WAL); ClickHouse (great but heavier ops than S3+Athena for offline-only).

Multi-armed bandit (channel A/B)Per-category bandits learn channel effectiveness from real-time engagement events; Thompson sampling auto-balances exploration/exploitation without manual experiment cutovers — at 23k/sec we can converge on the winning policy within hours.Fixed A/B with manual cutoff (slow, requires analyst); rule-based heuristics (no improvement loop).
