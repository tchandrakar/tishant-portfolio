Design Push Notification System (APNs + FCM at 100M+ devices)

Hard notifications messaging

1. Problem statement

Design a push notification platform that fans out application messages to 100M+ mobile devices via Apple Push Notification service (APNs) and Firebase Cloud Messaging (FCM). The system must handle device registration, token rotation, per-app rate limits, priority classes (high/normal), payload templating, and tight feedback loops on undeliverable tokens. Latency target: from publisher API call to device wake-up under 2 s p95 for high-priority alerts.

2. Functional requirements

  • Register/unregister device tokens keyed by (appId, userId, deviceId).
  • Send notification by userId, deviceId, topic, or segment (1..100M recipients).
  • Support APNs HTTP/2 and FCM HTTP v1; abstract provider choice from caller.
  • Templating with localization, badge counts, deep links, silent/data pushes.
  • Feedback ingestion: invalid token, unregistered, quota exceeded.
  • TTL, collapse keys, priority (high vs normal), per-tenant rate limits.

3. Non-functional requirements

  • Latency: publisher-to-provider under 500 ms p95; end-to-end under 2 s p95 for priority=high.
  • Scale: 100M devices, 5B sends/day peak, 250k sends/sec sustained, 1M/sec burst.
  • Availability: 99.95% — providers may be down, we must buffer.
  • Durability: at-least-once for transactional pushes; best-effort for marketing.

4. Capacity estimation

  • QPS: 5B/86,400 ≈ 58k avg, peak 4x ≈ 232k sends/sec.
  • Token storage: 100M devices × 256 B (token + metadata) ≈ 25 GB hot, 100 GB with indexes.
  • Provider connections: APNs HTTP/2 supports ~1500 streams/conn; need 250k/1500 ≈ 170 conns × 4 regions.
  • Kafka throughput: 250k msgs/s × 1 KB = 250 MB/s; with RF=3 → 750 MB/s replicated.
  • Feedback: ~3% bounce rate → 7.5k events/sec into the token-invalidation pipeline.

6. High-level architecture

[Publisher API] | v [Ingest GW] --> [Kafka: push.requests (32 partitions/region)] | v [Fanout Worker] --(lookup)--> [Token Store: Cassandra] | v [Kafka: push.dispatch (256 partitions, by deviceId)] | +-------------+-------------+ v v [APNs Sender Pool] [FCM Sender Pool] HTTP/2 long-lived HTTP v1 long-lived | | v v Apple APNs Google FCM | | v v [Feedback Topic] <-- [Feedback Collector] --> [Token GC]

The ingest layer validates, deduplicates by idempotency key, and persists to Kafka. Fanout workers expand audiences (topic/segment) into per-device messages and re-shard onto a dispatch topic partitioned by deviceId so a single device’s messages stay ordered. Sender pools maintain warm HTTP/2 connections to providers and respect provider rate caps.

7. Data model & storage

7.1 Token Store (Cassandra)

CREATE TABLE devices_by_user (
  app_id text, user_id text, device_id text,
  token text, platform text, locale text,
  app_version text, last_seen timestamp,
  PRIMARY KEY ((app_id, user_id), device_id)
);
CREATE TABLE token_index (
  token text PRIMARY KEY,
  app_id text, user_id text, device_id text
);

7.2 Why Cassandra?

Wide-column store with linear write scaling, multi-DC replication, and TTL on rows for stale tokens. A user’s devices live on one partition for cheap fan-out by userId; token_index supports O(1) inverted lookup from feedback events.

8. Detailed component design

8.1 Sender pool

Each sender holds N HTTP/2 connections to APNs/FCM. APNs allows ~1500 concurrent streams per connection; we run a token-bucket per (provider, app) and a circuit breaker that opens on >5% 5xx in a 10-second window. Backpressure propagates to Kafka via consumer-pause rather than dropping.

8.2 Idempotency & ordering

The idempotencyKey + 5-minute Redis SETNX dedupes retries. Per-device ordering is achieved by hashing deviceId to a dispatch partition; APNs’s collapse-id further coalesces redundant alerts.

8.3 Feedback loop

APNs returns 410/Unregistered and FCM returns NotRegistered/InvalidRegistration. A feedback collector tails sender error streams and emits to token.invalid; the GC worker deletes from Cassandra after a 24-hour grace period to tolerate brief unsubscribes.

9. Scaling & bottlenecks

  • Audience expansion: 1 message → 100M sends. Use a two-stage fanout: shard audience into 1000 chunks via a coordinator job, then per-shard workers.
  • Hot users: celebrity tokens get duplicated across 50 devices; cap per-user fanout and async batch.
  • Provider rate limits: APNs ~9000 streams/conn-set; horizontally add connection pools per region.
  • Storm protection: per-tenant token bucket at the ingress prevents one app exhausting capacity.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
APNs HTTP/2 (provider)Apple's only supported path for iOS push; ~1500 concurrent streams per connection lets us serve 250k sends/sec with ~170 long-lived connections per region instead of 250k short-lived TCP handshakes that would saturate kernel sockets.Legacy APNs binary protocol (deprecated by Apple); third-party relays (extra hop adds latency above the 2s p95 budget).
FCM HTTP v1 (provider)Mandatory for Android delivery; OAuth2-scoped tokens give per-app isolation across thousands of tenants, and JSON payloads simplify the renderer compared to the deprecated legacy XMPP path.FCM legacy server keys (deprecated, single shared key); direct GCM (sunset).
Apache Kafka (push.requests + push.dispatch)250k msgs/sec at 1 KB = 250 MB/s replicated to 750 MB/s with RF=3 — Kafka's sequential-IO log handles this on commodity disks, and partition-by-deviceId preserves per-device ordering across 256 partitions.RabbitMQ (per-message ack overhead caps near 50k/s/node); AWS SQS FIFO (300 msg/s/group ceiling kills 250k/s target).
Cassandra (devices_by_user, token_index)Linear-scale wide-column writes for 100M tokens; PRIMARY KEY ((app_id, user_id), device_id) co-locates a user's devices on one partition for cheap fan-out, and built-in TTL auto-expires stale tokens.DynamoDB (item-size limits and cross-region cost); Postgres sharded (manual resharding pain at 100M rows).
Redis (idempotency SETNX, token bucket)SETNX with 5-minute TTL dedupes retries on the hot path under 1 ms — at 250k sends/sec we cannot afford a Cassandra round-trip per request just to check duplicates; per-tenant token buckets also need atomic INCR.Memcached (no atomic SETNX-with-TTL semantics); in-process LRU (incoherent across fanout workers).
HTTP/2 multiplexing (sender pool)APNs caps at ~1500 streams per connection; HTTP/2 multiplexing is the only way to hit 250k sends/sec without thousands of TLS handshakes — each handshake is ~100 ms which would blow the 500 ms publisher-to-provider budget.HTTP/1.1 (one request per connection ruins throughput); QUIC (not supported by APNs/FCM yet).
Token bucket rate limiter (per-tenant)Storm protection so one tenant's campaign cannot drain the 1M/sec burst capacity from transactional traffic; bucket state in Redis is shared across ingest replicas.Leaky bucket (less burst-friendly for legitimate spikes); fixed window (boundary effects double the effective rate).
Circuit breaker (sender → provider)Opens on >5% 5xx in 10s windows so a regional APNs outage doesn't accumulate a 5B-msgs-deep backlog of doomed retries; pairs with consumer-pause backpressure into Kafka.Pure exponential retry (amplifies provider outage by retry storms); fail-open (sends garbage to a known-down APN).
Consistent hashing (deviceId → dispatch partition)Hashing deviceId across 256 dispatch partitions distributes 250k/s evenly; keeps per-device ordering for collapse-id semantics so a user doesn't see "deleted" before "received".Random partitioning (breaks ordering); modulo by userId (celebrity hot-key problem amplified).
Two-stage fanout (coordinator + workers)For 100M-target campaigns, a single fanout worker would take 400 seconds at 250k/sec; sharding the audience into 1000 chunks lets 1000 workers finish in <1 s parallelism while sharing the same Kafka dispatch topic.Single-stage fan-out (sequential bottleneck); pre-materialized per-segment topics (storage explosion for 50k tenants).
Multi-DC Cassandra replication (NetworkTopologyStrategy)100M tokens must survive a regional APNs outage redirected to another region; multi-DC RF=3 in each DC gives local-quorum reads at single-digit ms while tolerating a full DC loss.Single-DC + nightly backup (RTO of hours violates 99.95%); cross-region Postgres logical replication (lag spikes break correctness).
Feedback collector + token GC worker~3% bounce rate at 250k/s = 7.5k invalid tokens/sec; a dedicated stream worker reading APNs 410 / FCM NotRegistered keeps the active token base clean and avoids wasting provider quota on dead devices.Inline GC in sender (CPU contention with the hot path); periodic full scan (250 GB scan is too expensive daily).
