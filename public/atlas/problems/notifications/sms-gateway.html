Design SMS Gateway (multi-carrier, deliverability, fallback)

Hard notifications

1. Problem statement

Build an SMS gateway that delivers application-to-person (A2P) text messages globally via multiple carrier aggregators (Twilio, Sinch, Vonage, MessageBird) plus direct SMPP connections in select countries. The system must route by least-cost-and-best-deliverability, fallback when a route fails, handle DLRs (delivery receipts), respect country regulations (sender-ID rules, opt-out), and cap spend per tenant.

2. Functional requirements

  • Send SMS by E.164 number; auto-segment long messages (160 GSM-7 / 70 UCS-2 chars).
  • Multi-provider routing with health-aware failover.
  • DLR ingestion: queued, sent, delivered, failed, expired.
  • Two-way SMS: receive inbound and route to tenant webhook.
  • STOP/HELP keyword handling, opt-out registry, TCPA compliance.
  • Per-tenant budget caps, per-country pricing tiers.

3. Non-functional requirements

  • Latency: publisher accept under 200 ms p95; carrier handoff under 1 s p95.
  • Scale: 500M SMS/day, 6k/sec avg, 30k/sec peak.
  • Availability: 99.95% — no single carrier outage causes message loss.
  • Cost: route selection must minimize $/msg subject to deliverability SLA.

4. Capacity estimation

  • QPS: 500M/86,400 ≈ 5,800 avg; peak 30k/sec on auth-OTP spikes.
  • Storage: 500M × 1 KB metadata = 500 GB/day; 1-year retention ≈ 180 TB.
  • DLRs: ~2 per send (sent + delivered) → 1B events/day, 12k/sec.
  • Outbound bandwidth to SMPP: 30k msgs/sec × 200 B = 6 MB/s; trivial.
  • Routing decisions: 30k/sec route lookups served from in-memory weighted table.

6. High-level architecture

[Tenant API] -> [Validate/Opt-out check] -> [Kafka: sms.accepted] | v [Routing Engine] (cost, success-rate, country) | +---------------+---------------+---+---------------+ v v v v [Twilio Adapter] [Sinch Adapter] [Direct SMPP] [Vonage Adapter] | | | | +---------------+-------+-------+-------------------+ v [Carrier Networks] | v [DLR Listener] -> [Kafka: sms.events] | v [Status Store: Cassandra] / [Webhook Dispatcher]

The routing engine consults a per-(country, carrier) rolling-window success-rate computed on a 5-minute Flink job and a price table to choose the cheapest carrier above a configured deliverability floor. Failover is automatic on 5xx or timeout.

7. Data model & storage

7.1 Message store (Cassandra)

CREATE TABLE sms (
  tenant_id text, day text, message_id timeuuid,
  to_e164 text, from_id text, route text,
  status text, segments int, cost_usd decimal,
  PRIMARY KEY ((tenant_id, day), message_id)
);

7.2 Opt-out registry (Redis + DynamoDB)

SET optout:{tenant}:{e164} = 1   (TTL: forever; sourced from STOP keyword)

7.3 Why this split?

Opt-out is on the hot path of every send — Redis gives sub-ms checks. Cassandra holds the durable history with day-buckets to keep partition size bounded.

8. Detailed component design

8.1 Routing engine

For each message: compute candidates = carriers supporting destination country and sender type. Rank by score = w1 * success_rate - w2 * normalized_cost. Health-check via 1-minute rolling p95 and short-circuit a carrier on >3% failure for 60s. Sticky-route per recipient for 24h to maximize delivery on numbers that worked previously.

8.2 SMPP adapter

Maintains long-lived TCP connections to carriers with bind_transceiver. Window size 100, throughput throttled per binding. Handles enquire_link keepalives and reconnects on link failure within 2 s.

8.3 DLR pipeline

DLRs arrive minutes to hours later. The listener correlates via carrier message-id stored at send time, updates the message row, and emits sms.delivered events. Webhooks to tenants use exponential backoff with a 24h DLQ.

9. Scaling & bottlenecks

  • OTP storms: auth flows generate synchronized bursts; isolate OTP traffic on a dedicated lane with reserved carrier capacity.
  • Sender-ID rotation: India DLT and US 10DLC require pre-registered sender IDs; route engine must filter by registration state.
  • Cost runaway: tenant budget cap enforced via Redis token bucket atomic decrement.
  • Hot countries: Brazil/India dominate volume; pre-allocate SMPP windows per region.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
SMPP v3.4 (direct carrier)The only open carrier-grade protocol for Brazil/India where direct SMPP is 30-50% cheaper than aggregators; bind_transceiver with window size 100 sustains 30k peak/sec across long-lived TCP sessions per binding.HTTP REST to aggregator only (high $/msg at 500M/day); SMPP v5.0 (poor carrier support).
Twilio / Sinch / Vonage adapters (HTTP)Multi-aggregator routing means no single carrier outage causes message loss (99.95% SLO); HTTP REST adapters give a fast path to coverage in 100+ countries without per-country regulatory work.Single aggregator (vendor lock-in, no failover); SMPP-everywhere (regulatory burden in 100+ countries blocks launch).
Apache Kafka (sms.accepted, sms.events)30k peak msgs/sec accepted decoupled from carrier handoff so a Twilio spike doesn't block tenant API; partition-by-tenantId preserves per-tenant ordering for OTP flows where sequence matters.SQS standard (out-of-order kills OTP UX); Redis Streams (no multi-region replication for DR).
Redis (opt-out registry, budget bucket)Every send hits opt-out check on the hot path — 30k/sec SUBSTR-style key lookups must be sub-ms; atomic INCRBY on tenant budget prevents cost runaway when a tenant misconfigures a campaign.Postgres (point-read latency too high at 30k/sec); DynamoDB (acceptable but 5× cost for the same QPS).
DynamoDB (durable opt-out store)STOP keyword opt-outs are TCPA-regulated and must survive Redis cluster loss; DynamoDB gives 11-9s durability and TTL=forever semantics with global tables for cross-region failover.Cassandra (more ops burden); RDS (single-primary risk for a compliance-critical store).
Cassandra (sms message store)500M writes/day = 5,800/s avg with 30k bursts; PRIMARY KEY ((tenant_id, day), message_id) bounds partitions to ~5M rows for the biggest tenant and supports day-paginated tenant dashboards.Postgres sharded (resharding pain at 180 TB/year); ClickHouse (great for analytics, weak for point-read by message_id).
Apache Flink (5-min rolling success rate)Routing decisions need a real-time materialized view of (country, carrier) success rate; Flink's event-time windows on the DLR stream produce p95 success rate at 12k events/sec without batch ETL lag that would lock in a degraded carrier.Spark Streaming micro-batches (1-min lag misses fast outages); in-memory counters per box (incoherent routing across replicas).
Sticky routing per recipient (24h)Same recipient on the same successful carrier maximizes deliverability — porting + carrier preferences mean once a number works on Sinch in the US, it'll likely work again; cuts retry rate by ~15%.Random per-send (worse delivery); permanent stick (bad on number porting events).
Token bucket (tenant budget cap)Atomic Redis DECR enforces $/day spend caps before SMS is enqueued; without this a misconfigured campaign at $0.05/msg × 30k/sec = $1500/sec tenant burn.Post-hoc billing alerts (damage already done); fixed daily quotas (no fairness across hours).
Webhook dispatcher with DLQTenant webhooks for DLRs cannot block the 12k events/sec pipeline; exponential-backoff with 24h DLQ accommodates flaky tenant endpoints without dropping delivery receipts.Synchronous webhook (one slow tenant blocks all DLR ingestion); fire-and-forget (loss of regulatory audit trail).
Per-prefix anomaly detection (fraud)Pumping attacks target premium-rate prefixes (e.g., +234, +252) where attackers profit per send; velocity rules on (tenant, country prefix) running in Flink catch the attack within minutes instead of after the bill.Static blocklist (attackers rotate prefixes); manual review (30k/sec overwhelms ops).
Carrier sender-ID registry (DLT, 10DLC)India DLT and US 10DLC reject unregistered sender IDs at the carrier — routing engine must filter candidate routes by registration state per tenant or messages disappear silently with no DLR.Best-effort send (silent drops, no audit trail); single shared sender ID (banned by carriers).
OTP-isolated lane (separate Kafka topic + reserved IPs)OTP-spike is bursty and latency-sensitive (under 1s critical for auth UX); isolating it from marketing on dedicated lanes with reserved carrier capacity guarantees auth flows never get queued behind a 1M-msg campaign.Single shared queue (auth blocked by marketing burst); priority queue inside one topic (still shares carrier rate limit).
