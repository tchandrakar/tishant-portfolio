Design In-app Notifications (real-time inbox, read/unread, badge counts)

Hard notifications messaging

1. Problem statement

Design the in-app notification feed: a per-user inbox showing notifications (likes, mentions, system messages), a real-time push of new items via WebSocket/SSE, accurate unread badge counts that update across devices, and read/seen state synchronized in seconds. Targets: 300M MAU, 50M DAU concurrent across web and mobile, 10B feed items/day.

2. Functional requirements

  • Append a notification item to a user’s inbox.
  • List inbox with pagination, ordered by time DESC.
  • Real-time delivery to currently connected clients.
  • Mark single/all as read/seen; sync across devices in <5 s.
  • Badge count: unread total per user, per category.
  • Server-side TTL/archival of old items.

3. Non-functional requirements

  • Latency: publish to client under 1 s p95; read-state sync under 5 s p95.
  • Scale: 10B items/day, 115k writes/sec, 5M concurrent WebSocket connections per region.
  • Availability: 99.95%; degrade to polling if WebSocket fleet is unhealthy.
  • Consistency: eventual on feed order; monotonic on badge count (never decreases unexpectedly).

4. Capacity estimation

  • Writes: 10B/86,400 ≈ 115k/sec; peak 4× = 460k/sec.
  • Inbox storage: 10B × 1 KB × 90-day retention = 900 TB; tiered (hot 7d in Cassandra, cold in S3).
  • Connection fleet: 5M conns × 100 KB mem each = 500 GB → ~50 nodes × 100k conns each.
  • Fanout msgs: 50M DAU × 50 notifs/day = 2.5B WebSocket pushes/day = 29k/sec avg.
  • Badge cache: 300M users × 16 B counter = 5 GB → Redis cluster.

5. API design

POST /v1/inbox/items                {userId, type, payload}    -> itemId
GET  /v1/inbox?cursor=...           paginate                  -> [items], next
POST /v1/inbox/read                 {itemIds[]} or {all:true}
GET  /v1/inbox/badge                                          -> {total, byCategory}
WS   /v1/realtime/inbox             (auth via JWT)            -> stream

6. High-level architecture

[Producer Svc] -> [Kafka: inbox.events] | v [Inbox Writer Workers] | | v v [Inbox Store [Redis: badge counts Cassandra] INCR per user] | v [Pub/Sub: user.{id}] | v [WebSocket Edge Fleet] | v [Clients] [Read API] -> [Read-state writer] -> Cassandra + Redis (DECR badge) -> publish read-event to Pub/Sub for cross-device sync

Producers emit logical events; an inbox writer materializes per-user rows with a timestamp+ulid sort key, increments the unread badge atomically, then publishes to a per-user Pub/Sub channel that the WebSocket edge translates into a frame.

7. Data model & storage

7.1 Inbox (Cassandra)

CREATE TABLE inbox (
  user_id text, item_id timeuuid,
  category text, payload text,
  read boolean, seen boolean,
  created_at timestamp,
  PRIMARY KEY (user_id, item_id)
) WITH CLUSTERING ORDER BY (item_id DESC)
  AND default_time_to_live = 7776000;  -- 90 days

7.2 Badge counters (Redis)

HINCRBY badge:{userId} total 1
HINCRBY badge:{userId} cat:{category} 1
-- on read:
HINCRBY badge:{userId} total -N (with floor at 0 via Lua script)

7.3 Connection registry (Redis)

SADD conn:{userId} {edgeNodeId}:{connId}  EX 60

8. Detailed component design

8.1 WebSocket edge

Stateful nodes terminate WebSocket; on connect, a node registers (userId, nodeId, connId) in Redis and subscribes to user.{id} on the pub/sub layer. Heartbeats every 30 s; idle timeout at 90 s. On scale-out, sticky load balancing via consistent hashing on userId minimizes re-registrations.

8.2 Read-state synchronization

Marking read is a write to Cassandra (read=true) and an atomic Redis Lua script that decrements the badge with a non-negative floor. A read-event is also published to the user’s pub/sub channel so other connected devices update their badge instantly. Conflict: if device A and B mark the same item simultaneously, the second write is a no-op (CAS on read flag).

8.3 Badge correctness

Counters can drift under failures. A nightly reconciler scans inbox rows where read=false per user and rewrites the Redis counter. Cheap because we only run reconcile for users with reported drift (client sends server count + local count; mismatches enqueue a fix).

9. Scaling & bottlenecks

  • Celebrity follow events: 1M followers receiving the same producer event → fan-out via Kafka with per-user keys; throttle low-priority categories.
  • Reconnect storms: after edge restart, 100k clients reconnect within 1 s — use jittered exponential backoff and admission control on the edge.
  • Hot users: 10k items/day for power accounts; cap inbox at 10k rows + spillover archive, paginate with cursor.
  • Pub/Sub fan-out: Redis pub/sub doesn’t scale to 300M channels; use NATS or a sharded pub/sub keyed by user-shard.

12. Talking points for the interviewer

  • "I'd build the inbox first as a polling-only feed, then add WebSocket push as a thin overlay because polling correctness is the safety net."
  • "The hardest part is badge consistency across devices and crashes — I rely on Redis for fast path and a reconciler against Cassandra as the truth."
  • "At 300M users, Redis pub/sub doesn’t cut it; I'd shard channels or use NATS JetStream."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
WebSocket (RFC 6455) edge5M concurrent connections per region with sub-1s push latency requires persistent bidirectional channels; WebSocket frames have ~2-byte overhead vs SSE's HTTP framing and survive on mobile background-tab better than long-poll.SSE (one-way only, costs more bytes per message); HTTP long-polling (5M concurrent half-open requests strains LB).
Cassandra (inbox storage)10B writes/day = 115k/s avg with 460k/s peak; PRIMARY KEY (user_id, item_id) with descending clustering gives O(1) cursor pagination of newest-first inbox and built-in TTL=90d auto-archives without batch jobs.DynamoDB (cost at 900 TB hot tier prohibitive); Postgres (115k/s sustained writes saturate WAL on a single primary).
Redis (badge counters with HINCRBY)300M users × 16 B counter = 5 GB easily fits a Redis cluster; HINCRBY on badge:{userId} gives O(1) atomic per-category bumps and a Lua floor-at-zero script handles the read decrement without races.Cassandra counter columns (looser consistency, drift); Postgres counter (write contention at 460k/s peak).
Redis pub/sub sharded by user-shardPer-user channel pattern fans out to the right WebSocket edge node; sharding 300M channels across N Redis pub/sub clusters by hash(userId) avoids the single-Redis bottleneck where 300M channels would melt one node.Single Redis pub/sub (300M channels overwhelm a single instance); Kafka per-user topic (topic explosion).
NATS JetStream (alternative pub/sub)NATS subject-based routing scales beyond Redis pub/sub limits with built-in persistence; at 29k WebSocket pushes/sec average it provides "at-least-once" semantics so a brief edge-node restart doesn't lose deliveries that arrived during reconnect.Redis pub/sub (no persistence, lost messages during reconnect); RabbitMQ (per-channel overhead at 300M channels).
Apache Kafka (inbox.events)Producers push 115k events/sec with peak 460k; Kafka's partitioned log keyed by userId preserves per-user ordering for celebrity follow events while allowing 1M-follower fan-out via downstream workers without backpressuring producers.SQS (out-of-order kills monotonic feed); Kinesis (shard rebalance lag during follower-spike).
Consistent hashing (LB → edge nodes)Sticky LB on userId means a user's WebSocket lands on the same edge node across reconnects, so the connection registry SADD doesn't churn and the pub/sub subscription is stable; minimizes re-registration on the 50-node fleet.Round-robin LB (each reconnect rebuilds subscriptions); IP-hash (NAT/CGNAT collapses to few nodes).
JWT auth on /v1/realtime/inboxWebSocket upgrade can't carry per-message auth headers; signed JWT issued at login carries userId + scopes so the edge can authenticate 5M connections without per-request DB lookup.Session cookie (stale revocation on logout); per-message OAuth (latency/CPU killer at 29k pushes/sec).
Redis SADD connection registry (60s TTL)Multi-device sync needs to know which edge node holds a user's connections; SADD conn:{userId} {nodeId}:{connId} with 60s TTL self-heals on edge crashes when heartbeats stop refreshing the entry.Permanent registry rows (stale entries after edge crash); ZooKeeper ephemeral nodes (5M ephemerals overwhelms ZK).
Lua atomic floor-at-zero (read decrement)Marking 50 items read requires a single round-trip atomic decrement that doesn't dip below 0 — Lua scripts on Redis run inside the single-threaded loop so the badge count stays monotonic across simultaneous A/B device reads.WATCH/MULTI optimistic txn (more retries under load); client-side check-then-decrement (race condition).
Nightly reconciler (Cassandra → Redis)Counter drift is inevitable under failures; a reconciler scans read=false rows for users with reported drift and rewrites Redis counters — only running for clients that report mismatch keeps it cheap (drift < 0.1% of users daily).Per-write reconciliation (write amplification at 115k/s); never reconcile (drift accumulates user complaints).
Tiered storage (hot Cassandra 7d, cold S3 90d)900 TB at 90-day retention in Cassandra would cost ~$200k/mo in instances; spilling 7-day-old items to S3+Parquet cuts hot tier to 70 TB and keeps the cold tier at $0.023/GB-month, paginated via cursor when the user scrolls past day 7.All in Cassandra (cost prohibitive); all in S3 (read latency for the hot path unacceptable).
Polling fallback endpoint (degrade path)WebSocket fleet outages must not lose the inbox UX; a 30-second client poll on GET /v1/inbox?cursor= guarantees the inbox always works, and clients auto-promote back to WebSocket when the edge fleet recovers.WebSocket-only (full outage on edge failure); always-poll (5M × 30s polls = wasted backend QPS).
