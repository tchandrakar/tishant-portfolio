Design Driver Matching Engine (geospatial + supply/demand)

Hard ride-sharing maps-location

1. Problem statement

Build the dispatch engine that decides which driver gets which rider. It must consume a stream of driver location pings, maintain a real-time spatial index, and on every ride request return the optimal driver in well under one second. "Optimal" is multi-objective: minimize rider ETA, maximize driver utilization, balance global earnings, and respect surge-pricing incentives.

2. Functional requirements

  • Ingest driver location pings (4s cadence) and maintain status (online, on-trip, off-duty).
  • On a ride request, return a ranked list of candidate drivers within configurable radius.
  • Send dispatch offer to top driver with a 10–15s acceptance window; fall through to next on decline/timeout.
  • Support pooled rides: assign a second rider to an in-progress trip if detour cost is acceptable.
  • Honor preferences: car class, accessibility, pets, language, rider rating threshold.

3. Non-functional requirements

  • Latency: p50 dispatch decision < 200 ms; p99 < 1s.
  • Scale: 5 M concurrent online drivers, 1.25 M location writes/sec, 3 K dispatches/sec peak.
  • Correctness: a driver is offered to at most one rider at a time (linearizable on driver state).
  • Fault tolerance: a single AZ failure must not stall a city's dispatch; degrade gracefully (fallback to last-known location).

4. Capacity estimation

  • Location ingest: 1.25 M wps × 100 B = 125 MB/s; with 3× replication, ~10 TB/day.
  • Geo lookups: 3 K dispatch/sec × ~50 candidates each = 150 K spatial queries/sec.
  • In-memory state per city: ~50 K active drivers × 200 B = 10 MB; trivially fits in a single node.
  • Match decision compute: 50 candidates × road-network ETA (~5 ms each) → must parallelize: 8-way fan-out → ~30 ms.

6. High-level architecture

+------------------+ [Driver app] -> | Ping Ingest | --(Kafka: pings)--> +------------------+ | v +------------------+ | Geo Index Shards | (per-city, in-mem) | S2 cell -> set | +------------------+ ^ | [Rider app] -> [Match API] --> [Dispatch Coordinator] --+ | v [Scorer (ETA + ranker)] | v [Offer Manager] --(push)--> [Driver app] | v [Kafka: dispatch_events]

Geo Index is a sharded in-memory service, partitioned by city. Each shard owns the full set of drivers for its city (typically 30–80 K) — fits comfortably in a single 16 GB box, replicated 3-way. Dispatch Coordinator is stateless; it queries the index, scores candidates, and hands off to Offer Manager which owns the offer FSM.

7. Data model & storage

7.1 Driver state (Redis cluster)

HSET driver:{id} status ONLINE lat 37.78 lng -122.41 cell 9q8yyz ts ...
ZADD geo:cell:9q8yyz {score=ts} {driver_id}     # for fast scan + TTL eviction
SET  offer:{driver_id} {match_token} EX 15      # NX guards uniqueness

7.2 Dispatch decision log (Kafka -> S3 + ClickHouse)

Every (request, candidates considered, offer outcome) tuple is logged. ClickHouse for ad-hoc analytics; S3 + Parquet for ML training (rider acceptance models, ETA models).

8. Detailed component design

8.1 Geo index

S2 at level 12 (~1.2 km cells). Cells map to a sorted set of driver IDs ordered by last-update timestamp. On a query, we fetch the rider's cell and 8 neighbors — typical worst case 9 sets × ~80 drivers = 720 candidates, which we narrow to top-50 by haversine distance, then call the routing engine for true road ETA on those 50.

8.2 Scorer

Multi-objective: score = w1*(1/eta) + w2*(driver_acceptance_rate) + w3*(driver_idle_minutes) - w4*(detour_cost_for_pool). Weights tuned per city via offline RL. Returns ranked list to the offer manager.

8.3 Offer manager

Holds offer state (SET offer:driver_id NX EX 15 for linearizable single-offer guarantee). On accept, atomically transitions driver state to ASSIGNED; on decline/timeout, removes the lock and re-queries the next candidate. This is the only strongly-consistent path; the rest is eventually consistent.

9. Scaling & bottlenecks

  • Hot cells: downtown rush hour can push 5 K drivers into one S2 cell; subdivide dynamically into level-13 children when count > 1 K.
  • Repeated retries: a flaky driver app re-pinging at 1 Hz can DoS the index; rate-limit per driver.
  • Cross-AZ failover: use Redis sentinel + ping replay from Kafka to rebuild a hot replica in <30s.
  • ETA service overload: only score top 50 by haversine; use cached road-network ETAs for popular cell pairs.

12. Talking points for the interviewer

  • "I'd start with greedy per-request matching and add a 2-second batch window only in cities where we measure measurable utilization gains."
  • "The hardest bug class is double-dispatch under concurrent retries; my mitigation is the offer NX-lock with TTL and a Kafka decision log for forensics."
  • "I'd treat the index as a materialized view over the ping stream — that lets me rebuild it from Kafka after any failure."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.
