Design Lyft (multi-modal: rideshare + bikes + scooters)

Hard ride-sharing maps-location

1. Problem statement

Build a multi-modal mobility platform: classic rideshare (car + driver), shared rides (Line/Pool), and dockless micromobility (bikes, e-bikes, scooters). A single rider session must seamlessly compare modes, reserve a vehicle, navigate to it, and complete the trip — with one wallet, one map, one trip history. Internationally rolled out, with strict per-city regulatory zones (geofences) for micromobility.

2. Functional requirements

  • Unified trip planning: show car, shared, bike, scooter options with price + ETA + walking distance.
  • Reserve a bike/scooter for 10 minutes; unlock via QR/BLE; lock at trip end.
  • Driver-based rideshare: request, match, track, pay (Uber-like flow).
  • Geofence enforcement: speed-throttle scooters in pedestrian zones, prevent end-trip outside service area.
  • Single wallet across modes, ride credits, business profiles.

3. Non-functional requirements

  • Latency: map of nearby vehicles loads p99 < 500 ms; unlock command p99 < 1.5s end-to-end (server → IoT modem → vehicle).
  • Scale: ~2 M rideshare rides/day, ~1 M micromobility trips/day across 200 cities; 200 K active vehicles.
  • Availability: 99.95% for trip planning; 99.99% for unlock (a stuck rider is a support nightmare).
  • Reliability: IoT command idempotent; offline scooter must reconcile state on reconnect.

4. Capacity estimation

  • Vehicle telemetry every 30s for 200 K vehicles → 6.7 K writes/sec; 60 B/event → 25 GB/day.
  • Search-nearby QPS: 50 K/sec peak (every map pan triggers one).
  • Trip records: 3 M/day × 1.5 KB ≈ 4.5 GB/day; 1.6 TB/year.
  • Unlock commands: 1 M/day, must round-trip in <1.5s — sized for 100 RPS sustained, 1 K RPS peak.

5. API design

GET  /v1/nearby?lat=..&lng=..&modes=car,bike,scooter   → [{vehicleId, mode, eta, price, lat, lng}]
POST /v1/trips/plan   {origin, dest, modes}             → planId, options[]
POST /v1/trips/reserve{planId, optionId}                → trip, holdExpiresAt
POST /v1/vehicles/{id}/unlock {tripId, deviceId}        → 200 (idempotency-key required)
POST /v1/vehicles/{id}/end    {endLoc, photoUrl}        → fare
POST /v1/drivers/location     {driverId, lat, lng}      → 200

6. High-level architecture

[Rider App] -> [API GW] -> [Trip Planner] -> [Mode Routers] | | | +---------------+ | +---------------+ v v v [Rideshare Svc] [Bike Svc] [Scooter Svc] | | | v v v [Dispatch + S2 idx] [Geo Index (H3)] [Geofence Svc] | | | +--------+-----------+--------+-----------+ v v [Kafka: events] [IoT Gateway / MQTT] | | v v [Wallet/Payments] [Vehicle (modem)]

Trip Planner is the orchestrator. It calls each mode service in parallel, merges results, and returns a unified option list. Per-mode services own their geo index and lifecycle. The IoT Gateway terminates MQTT/AMQP from physical vehicles and translates to internal RPC.

7. Data model & storage

7.1 Vehicle (DynamoDB)

PK: vehicle_id   (string)
SK: -            (single-row)
attrs: mode, status (IDLE|RESERVED|INUSE|OOS),
       last_lat, last_lng, last_seen, battery_pct,
       firmware_v, geofence_region
GSI1: status + h3_cell  (for nearby queries)

7.2 Trip (Postgres, partitioned by month)

CREATE TABLE trips (
  trip_id UUID PRIMARY KEY,
  user_id BIGINT, mode TEXT, vehicle_id TEXT,
  start_loc GEOGRAPHY, end_loc GEOGRAPHY,
  fare_cents INT, started_at TIMESTAMPTZ, ended_at TIMESTAMPTZ
) PARTITION BY RANGE (started_at);

7.3 Geofences (PostGIS)

Polygons stored with GiST spatial index; lookups in <10 ms. Versioned so regulatory changes don't break in-flight trips.

8. Detailed component design

8.1 Trip planner

Fan-out to mode services with a 250 ms hard timeout. Merges results, applies user preferences, sorts by composite (price, eta, mode_pref). Caches per-(origin-cell, time-bucket) for 30s.

8.2 Geofence service

Hot polygons loaded into memory as an R-tree; uses point-in-polygon tests on every telemetry ping. Emits Kafka events on zone enter/exit. Scooter firmware receives a speed-cap command via downlink within 2s.

8.3 IoT gateway & unlock

MQTT broker (EMQX) with persistent sessions. Unlock command is idempotent: server stores (trip_id, command_id) and dedups. If MQTT downlink fails, fallback to LTE-direct via vehicle modem; if both fail, surface "manual unlock code" within 8s.

9. Scaling & bottlenecks

  • Map-pan storms: CDN-cache the nearby endpoint at coarse H3 resolution; precompute per-cell vehicle snapshots every 5s.
  • Hot geofence polygons: bloom filter on cell membership before doing the expensive PIP test.
  • Vehicle clock skew: server stamps every event; never trust client time for billing.
  • Battery-driven supply collapse: rebalance ops surfaces low-battery clusters; predictive charging dispatch.

12. Talking points for the interviewer

  • "I'd push the planner to be stateless and per-mode services to own their index — that decoupling lets us add a new mode (mopeds) without touching rideshare."
  • "The unlock path is the trickiest because it bridges cloud and a flaky LTE modem; idempotency + dual-channel commands are non-negotiable."
  • "Geofencing is a regulatory product, not a feature — I'd version polygons and replay them in a staging zone before rolling out."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
H3 (resolution 8–9)200 K active vehicles plus 50 K nearby-search QPS need uniform hex neighbors so a single map-pan covers the same area regardless of latitude — important when Lyft operates in 200 cities from Toronto to São Paulo.Geohash → rectangular cells skew badly at high latitudes; S2 → uniform-ish but neighbor distance still varies on the cube projection.
S2 (rideshare path)Per-mode rideshare service inherits the Uber-style dispatch loop; S2's 1D Hilbert key is faster for cell range scans during the 250 ms trip-planner fan-out budget.H3 only → fine but no native 1D ordering for sorted-set scans.
DynamoDB (vehicle KV)200 K vehicle rows updated every 30s = 6.7 K wps with single-digit ms p99; GSI on (status, h3_cell) handles the nearby query without a separate cache; auto-scales across 200-city footprint.Cassandra → equally fine but more ops; Postgres → no managed multi-region active-active.
PostGIS (geofences)Polygon point-in-polygon for regulatory zones (NYC pedestrian-only blocks, Paris no-park areas) needs GiST spatial indexing for <10 ms lookups on every telemetry ping; SQL JOINs let us version polygons.Pure H3 cells → can't represent arbitrary regulator-drawn polygons exactly; Mongo geospatial → weaker polygon ops.
Postgres (trips, partitioned by month)3 M trips/day × 1.5 KB = 4.5 GB/day; range partition by started_at keeps the hot month small and lets us drop old partitions cheaply for the 1.6 TB/year retention.MySQL → fine; Cassandra → no JOINs for support/billing queries.
EMQX (MQTT broker)1 M unlock commands/day with 100 RPS sustained, 1 K RPS peak need persistent sessions and QoS 1 for idempotent delivery; the 1.5s end-to-end unlock budget cannot tolerate handshake on every command.Mosquitto → single-node only; HiveMQ → fine, EMQX is cheaper at scale.
Kafka (mode events)Decouples Trip Planner from Wallet, Surge, and Geofence; replayable log lets us reprocess a regulatory polygon change against the last 24 h of telemetry without disrupting live traffic.SQS → no replay; Pulsar → smaller Flink connector ecosystem.
R-tree (in-memory geofences)Hot polygons loaded into the Geofence Service's process memory: a point-in-polygon test on every 30s telemetry ping × 200 K vehicles = 6.7 K PIP/sec, but bloom-filter pre-check + R-tree keeps the median <1 ms.PostGIS for hot path → 10 ms × 6.7 K = 67 K rps load on Postgres, too expensive.
gRPC (internal)Trip Planner fan-out to 3 mode services within a 250 ms hard timeout; protobuf keeps payloads small enough to parallelize without exhausting the budget.REST → JSON parsing eats 30–50 ms of the budget; Thrift → smaller polyglot ecosystem.
QUIC + LTE fallbackVehicle modems on flaky carrier networks get 0-RTT resumption; if the MQTT downlink fails the same payload is retried over LTE-direct so the rider never sees an unlock failure beyond 8s.TCP-only → reconnects after every cell handoff break the unlock SLA.
CDN (nearby endpoint)Map-pan storms generate 50 K QPS of geographically-clusterable reads; caching at coarse H3 cell for 5s collapses 90%+ of traffic into edge hits.Origin-only → would 10× backend cost; client-only cache → no benefit on cross-user pans.
Redis (planner cache)Per-(origin-cell, time-bucket) trip-plan cache for 30s absorbs the bursty user behavior of comparing modes 3–5 times before reserving; sub-ms read keeps the planner under 250 ms.Memcached → no TTL-on-element granularity; in-process cache → no cross-pod hit.
Spark (rebalance jobs)Nightly low-battery cluster prediction joins 25 GB/day telemetry with charging operations data; batch is fine here because rebalance ops act on hourly cadence.Flink batch → less mature for ML pipelines; Trino → no Python ML libs.
