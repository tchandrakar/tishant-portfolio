Design ETA Prediction Service (ML-based, real-time traffic)

Hard ride-sharing maps-location recommendation

1. Problem statement

Build an ETA service that, given an origin/destination pair, returns an accurate travel-time estimate in single-digit milliseconds. Used by ride matching, fare estimation, food delivery, and live trip tracking. Must adapt to live traffic (incidents, weather, events) and learn from billions of historical trip traces.

2. Functional requirements

  • POST /eta with origin/dest → travel_time, distance, polyline.
  • Bulk ETA: 1×N (one origin, many candidates) for dispatch fan-out.
  • Live updates: while a trip is in progress, refresh ETA every 30s using current traffic.
  • Mode-aware: car, bike, walk, scooter speed profiles.
  • Confidence interval: return p50 and p90 ETAs, not just a point estimate.

3. Non-functional requirements

  • Latency: single ETA p99 < 50 ms; bulk-50 ETA p99 < 80 ms.
  • Accuracy: MAE < 60s for trips under 30 min, < 10% relative error overall.
  • Scale: 500 K QPS globally at peak (dispatch fan-out dominates).
  • Freshness: traffic state lag < 90s; model refresh hourly.

4. Capacity estimation

  • 500 K QPS × ~5 KB request/response avg → 2.5 GB/s ingress; CDN can't help (per-request unique).
  • Road graph: ~50 M edges globally, ~5 GB compressed; per-region shard ~100–500 MB → fits in memory.
  • Live traffic snapshots: every 60s, 50 M edges × 8 B speed value = 400 MB/min → 24 GB/hour pushed to read replicas.
  • Trip traces for training: 25 M trips/day × 1 KB = 25 GB/day → 9 TB/year, kept 2 years for seasonality.

5. API design

POST /v1/eta             {origin, dest, mode, depTs}     → {p50, p90, distM, polyline}
POST /v1/eta/bulk        {origin, dests[], mode}         → [{dest, p50, p90}]
POST /v1/eta/live/{trip} {currentLoc}                    → {p50, p90, etaTs}

Internal:
  GET /v1/traffic/snapshot/{region}/{t}                  → edge_speeds[]
  POST /v1/route/shortest {o, d, weights}                → path, time

6. High-level architecture

[Client] -> [API GW] -> [ETA Service] | \ v v [Router] [Ranker (GBM/DNN)] / \ ^ v v | [Road Graph] [Live Traffic State] <-- [Kafka: gps_pings] ^ | [Map-matcher / Speed agg] ^ | [Drivers + open ridehailing fleets]

Two-stage pipeline: a classical graph router (Contraction Hierarchies / CRP) returns a candidate path and base time; a learned ranker corrects this using context (time-of-day, weather, event tag, recent observed speeds for these edges). The traffic state is a materialized view over the GPS ping stream.

7. Data model & storage

7.1 Road graph (custom binary, mmap'd)

edge {
  u: int32, v: int32, length_m: float,
  cls: u8 (highway|arterial|residential),
  free_flow_s: float, lanes: u8, oneway: bit
}

Stored as compressed adjacency lists, mmap'd per region into a stateless router process. Reload via blue/green every map update (weekly).

7.2 Live traffic (Redis + RocksDB)

Hot: edge → recent observed speed window (last 5 min, EWMA). Warm: per-edge time-series in RocksDB for the past 24h, used by the ranker as a feature.

7.3 Trip traces (Parquet on S3, partitioned by day/region)

Ground-truth labels for offline training: (origin, dest, departure_ts, actual_time, path).

8. Detailed component design

8.1 Router

Contraction Hierarchies precomputed weekly: shrinks 50 M-edge query to ~hundreds of hops, gets sub-ms shortest path. Edge weights = base free-flow time × live speed multiplier.

8.2 ML ranker

Gradient-boosted tree on features: base ETA, hour-of-day, day-of-week, weather code, edge-class mix, recent edge-speed deltas, origin/dest cluster embeddings. Trained nightly on the previous 90 days. Served via a thin C++ predictor; ~200 µs/inference.

8.3 Map-matcher & speed aggregator

Stream job (Flink) consumes GPS pings, snaps each to the nearest road edge using a Hidden Markov Model, and emits (edge_id, speed). A per-edge tumbling window (1 min) computes mean speed and pushes to Redis. Outliers filtered by Kalman smoothing.

9. Scaling & bottlenecks

  • Bulk fan-out: a 50-candidate dispatch ETA call must reuse one source-side Dijkstra (one-to-many) — drops cost from 50× to ~1.5×.
  • Cold edges: rural roads have no recent pings; fall back to historical-time-of-day model.
  • Map updates: a global graph rebuild every week; use blue/green per region to avoid stalls.
  • Hot regions: shard router by region; a single SF instance handles ~50 K QPS.
  • Tail latency: bound router with a 20 ms deadline; on timeout, return haversine × class multiplier as fallback.

12. Talking points for the interviewer

  • "The two-stage design (graph router + learned ranker) gives me cheap inference while still capturing context like weather and events."
  • "I'd treat traffic as a materialized view over GPS pings — that means a clean replay path when I change the speed-aggregation logic."
  • "For dispatch, a one-to-many Dijkstra is the single biggest cost saver: 50× fewer searches per ride request."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Contraction Hierarchies (CH)50 M global edges shrink to a few hundred hops per query; the 50 ms p99 single-ETA budget and 80 ms bulk-50 budget at 500 K QPS are only feasible because CH delivers sub-ms shortest-path lookups.Live Dijkstra → 50–100 ms per query, blows the SLO; A* → faster than Dijkstra but still 5–10× slower than CH.
mmap'd binary road graphPer-region 100–500 MB shards mmap into stateless router processes; OS page cache shares pages across forks, and blue/green reload on weekly map updates avoids stalling the 500 K QPS read path.Postgres/PostGIS in-line → 10–100× slower for shortest-path; in-process JVM heap → GC pauses break tail latency.
S2/H3 (cluster embeddings)Origin/dest cluster embeddings as ranker features need consistent geographic bucketing; H3 res 8 (~0.7 km²) gives the GBM a stable categorical even when the road graph rebuilds.Free-form k-means → embeddings drift on retrain; raw lat/lng → ranker overfits to coordinate noise.
Redis (live edge speeds)EWMA over 5-min windows keyed by edge_id needs sub-ms reads from the router on every shortest-path expansion; 50 M edges × 8 B = 400 MB hot fits comfortably with sharding.Memcached → no atomic ops for EWMA update; in-process cache → no cross-pod consistency.
RocksDB (warm 24h history)Per-edge time-series for the past day feeds the ranker as a feature; LSM keeps writes cheap as Flink emits 50 M edge updates/min, and range scans by edge_id are fast.InfluxDB → fine, RocksDB embeds in the ranker process for 0-network-hop reads; Cassandra → overkill for embedded use.

S3 + Parquet (trip traces)25 M trips/day × 1 KB = 25 GB/day; 2-year retention = ~18 TB. Day/region partitioning means a Spark training job over the past 90 days reads only ~2 TB and parallelizes across 1 K cores.HDFS → no managed multi-region durability; row-store SQL → poor compression and full-scan cost.
Spark (CH rebuild + GBM training)Weekly CH preprocessing and nightly GBM training are batch-shaped (hours of compute on TB of data); Spark's broadcast joins handle the road-graph + traces join naturally.Flink batch → less mature ML tooling; Hadoop MR → 5–10× slower wall-clock.
GBM (gradient boosted trees)200 µs/inference at 500 K QPS = 100 cores aggregate; interpretable feature importances let us debug regressions per-city. MAE < 60s target is achievable without DNN-scale hardware.DNN → 5–20× more inference cost, opaque; linear regression → too biased for non-linear weather/event interactions.
gRPC (router + ranker)Two-stage pipeline must complete in 50 ms p99; protobuf and HTTP/2 multiplex the router and ranker calls in parallel from the ETA service.REST → JSON parse eats 5–10 ms of the budget; co-located library → harder to scale router and ranker independently.

CDN bypass (per-request unique)500 K QPS × 5 KB = 2.5 GB/s ingress lands directly on the ETA service because origin/dest pairs rarely repeat; CDN caching wouldn't help and would just add a hop.Edge cache → near-zero hit rate, wasteful; client-side dedup → no benefit for unique journeys.
