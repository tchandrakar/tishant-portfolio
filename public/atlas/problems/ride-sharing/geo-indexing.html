Design Geo-Indexing with S2 / H3 / Geohash (city quad-trees)

Hard ride-sharing maps-location storage-cache

1. Problem statement

Build a generic geo-indexing service that supports billions of moving points (drivers, scooters, delivery riders, IoT sensors). It must answer two queries fast: (1) "give me the K nearest entities to this point" and (2) "give me everything inside this polygon," at QPS levels that ride-sharing requires (hundreds of thousands of queries per second).

2. Functional requirements

  • Insert/Update/Delete a point with TTL.
  • Range query: bounding box / circle / polygon.
  • kNN query for a given lat/lng.
  • Aggregations: count points per cell at a given resolution.
  • Multi-tenant: namespaces per fleet (drivers, scooters, restaurants).

3. Non-functional requirements

  • Latency: kNN p99 < 10 ms for K=50; range p99 < 30 ms for cells with up to 10 K points.
  • Scale: 10 M live points across all tenants; 1.25 M writes/sec; 200 K reads/sec.
  • Freshness: writes visible within 200 ms.
  • Durability: live state can be lost (rebuild from stream); historical traces are durable in object storage.

4. Capacity estimation

  • 10 M points × ~120 B = 1.2 GB hot state; replicated 3× across AZ → 3.6 GB. Easily fits in a Redis cluster.
  • Writes: 1.25 M/s × 200 B Kafka envelope = 250 MB/s ingest.
  • Reads: 200 K/s × avg fan-out 9 cells × ~50 entries = 90 M entry reads/sec → must be in-memory.
  • Per-cell density: dense urban cell at S2 level 12 contains 200–5 K points; sparse rural cells < 10.

5. API design

POST /v1/geo/{ns}/upsert {id, lat, lng, attrs, ttlSec}    → 200
GET  /v1/geo/{ns}/knn?lat=..&lng=..&k=50&filter=...      → [{id, distM, attrs}]
GET  /v1/geo/{ns}/within?bbox=lat1,lng1,lat2,lng2         → [{id, lat, lng}]

DELETE /v1/geo/{ns}/{id}                                   → 204

6. High-level architecture

[Producers] --> [Kafka: geo_updates] --> [Indexer pods] | (in-mem index per shard) v [Redis Cluster] key = ns:cell -> ZSET(id by ts) key = ns:point:id -> hash ^ | [Clients] -> [Geo API] -> [Query Planner] --+ | +--> [PostGIS] (durable, polygon ops) +--> [S3 / Parquet] (cold history)

Hot path uses Redis for cell-set storage, fronted by a stateless Geo API that knows the indexing strategy (S2 vs H3 vs Geohash). PostGIS is consulted only for arbitrary polygon queries that don't decompose cleanly to cells. Writes flow through Kafka so the index is rebuildable from the stream.

8. Data model & storage

8.1 Cell index (Redis)

ZADD geo:{ns}:cell:{cellId}  {score=ts}  {pointId}
HSET geo:{ns}:point:{id}     lat lng attrs cellId
EXPIRE geo:{ns}:point:{id}   {ttl}

The ZSET orders points by last-update timestamp so we can cheaply expire stale entries and run a windowed scan. The point hash holds full attributes for filtering.

8.2 Polygon support (PostGIS)

CREATE TABLE geofences (
  id UUID PRIMARY KEY, ns TEXT, name TEXT,
  region GEOGRAPHY(POLYGON, 4326),
  effective_at TIMESTAMPTZ, expires_at TIMESTAMPTZ
);
CREATE INDEX ON geofences USING GIST (region);

9. Detailed component design

9.1 kNN algorithm

Compute the rider cell at level R. Scan the cell + 8 neighbors. If total candidates < K, expand to ring 2 (24 more cells). Always cap at ring 3 (48 cells) — anything farther is not useful for dispatch. Sort the gathered candidates by haversine and return top-K. Typical: 9 cells × ~80 points = 720 candidates → 50 returned in <5 ms.

9.2 Adaptive resolution

Use H3 resolution 8 (~0.7 km²) globally. In dense cells (count > 1000), the indexer dynamically also writes the point to the resolution-9 child cell, so dense queries can use the finer index without affecting sparse ones. Reads pick resolution based on observed density per cell.

9.3 Writer pipeline

Kafka consumer group with one partition per shard. Each writer batches updates per cell (50–200 per batch, 20 ms flush) using Redis pipelining to keep the network round-trips down. Idempotent on (ns, id, ts) — out-of-order pings are dropped.

9.4 Polygon query

For an arbitrary polygon, decompose to a covering set of H3 cells using polygonToCells; union-fetch from Redis; finally do a precise point-in-polygon filter only for cells that straddle the boundary.

10. Scaling & bottlenecks

  • Hot cells: a single Redis key holding 5 K members can cause CPU stalls — split via dynamic resolution upgrade.
  • Cluster slot skew: use {ns:cell} hash tags so neighbor lookups land on contiguous shards; rebalance using consistent hashing.
  • TTL stampedes: stagger TTLs (±10s jitter) so a million keys don't expire together.
  • Polygon-to-cells fan-out: a city-sized polygon at res 9 yields ~50 K cells; cap and use coarser res for large polys.
