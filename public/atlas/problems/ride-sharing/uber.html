Design Uber (rider/driver matching at city scale)

Hard ride-sharing maps-location payments

1. Problem statement

Build a global ride-hailing platform that connects riders with nearby drivers, dispatches the closest viable driver in under a second, tracks the trip in real time, computes fare on completion, and settles payment to the driver. Must work across thousands of cities with very different supply/demand profiles, comply with local regulations, and remain available during city-wide demand spikes (NYE, sports events, airport surges).

2. Functional requirements

  • Rider requests a ride from origin to destination; system returns ETA and fare estimate.
  • Driver matching: assign closest available driver within ~5 km; rider sees driver location updating every 4s.
  • Live trip tracking, route updates, on-trip ETA refresh.
  • Fare computation, surge multiplier application, split fare, tipping.
  • Payments: charge rider, payout to driver, refunds, receipts.
  • Ratings, support tickets, driver onboarding, KYC.

3. Non-functional requirements

  • Latency: dispatch decision p99 < 1s; driver location ingestion p99 < 200 ms.
  • Scale: 25 M trips/day (~290 trips/sec average, ~3 K trips/sec peak); 5 M concurrent online drivers globally.
  • Availability: 99.99% for matching, 99.999% for payments ledger.
  • Consistency: strong on the trip lifecycle (one driver per request); eventual on analytics.

4. Capacity estimation

  • Driver location pings every 4s: 5 M drivers / 4s = 1.25 M writes/sec at peak.
  • Each ping ~100 B → 125 MB/sec ingest, 10 TB/day raw, 90-day retention > 900 TB compressed.
  • Match QPS: 3 K dispatches/sec peak × 50 candidate evaluations = 150 K geo-queries/sec.
  • Trip metadata: 25 M/day × 2 KB = 50 GB/day; 18 TB/year per region.

6. High-level architecture

[Rider App] [Driver App] | | | HTTPS/gRPC MQTT/QUIC | (location pings 4s) v v [Edge / API GW + WAF] [Location Ingest Svc] | | v v [Ride Svc]----+---->[Match/Dispatch Engine]<---[Geo Index (S2/H3)] <--+ | | | | v v v | [Pricing][Trip State FSM][Notification Svc] | | | | | v v v | [MySQL] [Kafka: trip_events] -----> [Payments] -> [Ledger (Postgres)] | | | +---> [Stream Proc / Flink] -> [Analytics, Surge]------+

Rider hits the API GW, which routes to the Ride Service. Ride Service writes a pending trip record (FSM: REQUESTED), pushes a match request to the Dispatch Engine. Dispatch queries the Geo Index for nearby drivers (S2 cell scan + filter by status), scores them, and offers to the top driver via MQTT push. On accept, the FSM transitions and Kafka emits trip_events that downstream payments, surge, and analytics consume.

7. Data model & storage

7.1 Trip (MySQL, sharded by city_id)

CREATE TABLE trips (
  trip_id BINARY(16) PRIMARY KEY,
  city_id INT, rider_id BIGINT, driver_id BIGINT,
  state ENUM('REQ','MATCHED','ARRIVED','ONTRIP','DONE','CANCELED'),
  origin POINT, dest POINT,
  fare_cents INT, surge_x DECIMAL(3,2),
  created_at DATETIME(3), ended_at DATETIME(3)
) PARTITION BY HASH(city_id);

7.2 Driver location (Redis + Cassandra)

Hot path: Redis sorted set per S2 cell, TTL 30s, used by Dispatch. Cold path: Cassandra time-series keyed by (driver_id, day) for forensics and ML training. Cassandra is chosen for high write throughput and TTL support (DDIA Ch. 3).

8. Detailed component design

8.1 Dispatch engine

Per-city service. On a match request, it computes the rider's S2 cell at level 12 (~1.2 km), expands neighbors, queries the geo index for ~50 candidates, scores them by ETA (road-network not haversine), driver acceptance rate, and surge balancing. Sends offer with 10s TTL; on timeout, falls to next candidate. Uses optimistic locking on driver state in Redis to prevent double-dispatch.

8.2 Trip state FSM

Implemented as a Saga over Kafka. Each transition is an event with idempotency key = trip_id+to_state. Compensating events handle driver no-show or payment failures.

8.3 Payments

Double-entry ledger in Postgres. Authorization at trip start (hold), capture at trip end. Driver payout aggregated daily and pushed to bank rails. Strong consistency (read-after-write) is non-negotiable here.

9. Scaling & bottlenecks

  • Location ingest hot spots: partition Kafka by driver_id hash, not by city, to avoid mega-city skew; back-pressure via QUIC.
  • Geo-index hot cells: Manhattan at rush hour packs 10 K drivers in a few S2 cells; subdivide cells dynamically using H3 resolution 9–10 in dense zones.
  • Match thundering herd: use exponential back-off and per-rider single-flight to prevent retries from doubling load.
  • MySQL hot shard: NYC alone is ~8% of trips; split city_id 1 into multiple logical shards.

12. Talking points for the interviewer

  • "I would start with one region and scale by sharding on city_id, because dispatch is inherently local — a driver in Mumbai never matches a rider in NYC."
  • "The hardest part is the geo-index hot-cell problem; my mitigation is dynamic cell subdivision plus a top-k heap maintained in Redis."
  • "For payments I'd use a separate, smaller, strongly-consistent service — the rest of the system can be eventually consistent."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
S2 (level 12)Hilbert-curve 1D keys make 1.2 km cell scans a single Redis range read; the 50 candidates per dispatch (3 K dispatches/sec × 50 = 150 K geo-queries/sec) come back in <2 ms. Hierarchical levels let Manhattan subdivide on demand.Geohash → uneven cells at high latitudes; H3 → no native 1D ordering, harder to range-scan.
H3 (resolution 9–10)Reserved for dense urban subdivision when an S2 cell exceeds ~1 K drivers; uniform hex neighbor distance matters when scoring a 5 km radius around a rider in midtown.S2 only → uneven hot-cell shapes; quadtrees → no edge-uniform neighbor ring.
Redis (sorted sets, TTL 30s)1.25 M location writes/sec land in a sharded cluster keyed by S2 cell with sub-ms latency; TTL auto-expires offline drivers; CAS via SET NX gives the linearizable single-offer guarantee on driver state.Memcached → no sorted set or TTL-on-element; Hazelcast → heavier ops footprint for the same access pattern.
Cassandra (ping cold path)Time-series of 1.25 M wps × 100 B = 10 TB/day with 90-day retention requires a write-optimized LSM store; partition (driver_id, day) + native TTL fits forensics and ML training without compaction pain.HBase → harder ops; Postgres → would need aggressive partitioning, no native TTL.
Kafka (trip_events)Replayable log decouples Ride Service from Payments, Surge, and Analytics. Partitioning pings by driver_id avoids the NYC-as-8%-of-trips skew and lets Flink consume at 1.25 M msg/sec.RabbitMQ → no replay, weaker throughput; Pulsar → fine but smaller ecosystem for Flink connectors.

MQTT (driver downlink)Persistent sessions over QUIC keep 5 M concurrent driver phones connected with low battery cost; offers reach the driver in <200 ms so the 10s TTL window stays useful.WebSocket → heavier per-connection state at 5 M scale; APNs/FCM only → 2–10s tail, unusable for dispatch.
QUIC (location ingest)0-RTT resume and head-of-line-blocking-free streams matter when a driver phone hops cell towers mid-trip; keeps p99 ping ingest < 200 ms even under packet loss.TCP+TLS → reconnect cost on tower handoff; raw UDP → no congestion control, would melt under 1.25 M wps.
MySQL (sharded by city_id)Trip FSM needs single-leader writes for the "one driver per request" invariant; 25 M trips/day × 2 KB = 50 GB/day per region fits comfortably with hash partitioning by city.Postgres → equally fine, but Vitess/MySQL gives more proven sharding tooling at city scale.
Postgres (payments ledger)Double-entry ledger needs serializable isolation and read-after-write; 99.999% availability target on payments justifies a smaller, stronger store separate from the trip DB.MySQL → fine, but Postgres' SERIALIZABLE and rich constraint support reduce ledger bugs.
