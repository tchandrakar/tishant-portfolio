Design Real-time Bidding (RTB exchange — sub-100ms auction)

Very Hard ad-tech

1. Problem statement

1. Problem statement

Build an OpenRTB 2.x compliant ad exchange. When a publisher serves an ad slot, the exchange must broadcast a bid request to dozens of DSPs (Demand-Side Platforms), collect their bids, run a second-price auction, return the winning creative, and reconcile the win-notification — all under a hard 100 ms wall-clock budget. The exchange handles billions of bid requests per day, persists every auction for billing and audit, and exposes near-real-time analytics.

2. Functional requirements

  • Receive bid requests from publishers (OpenRTB JSON or Protobuf over HTTP/2).
  • Fan out to N DSPs (typically 20–50) with deadlines < 80 ms.
  • Run second-price auction with reserve, deal-ID priority, brand-safety filtering.
  • Return a creative URL or VAST tag to the publisher; fire win-notice to the winner.
  • Track impression beacons, deduplicate, bill the DSP at the cleared price.
  • Provide DSP throttling, QPS quotas, and time-out blacklists.

3. Non-functional requirements

  • Latency: p99 end-to-end < 100 ms; DSP timeout 80 ms; internal auction < 5 ms.
  • Scale: 5 M QPS peak, 400 B bid requests/day, 100 PB raw logs/year.
  • Availability: 99.99%; degraded mode (skip DSPs, serve fallback) instead of failing.
  • Throughput per DSP: 100k–1M QPS sustained.

4. Capacity estimation

  • QPS: 5 M peak × 30 DSP fan-out = 150 M outbound RPCs/s.
  • Storage: 400 B requests/day × 1 KB compressed = 400 TB/day; tiered to S3/Glacier.
  • Network: 30 Gbps inbound, 200 Gbps east-west to DSPs.
  • Servers: ~5,000 bid routers (1k QPS each at 100 ms p99 with 100 in-flight).

5. API design

POST /rtb/v2.5/bid    (publisher → exchange)   body: BidRequest{id,imp[],site,user,device}
POST /rtb/v2.5/bid    (exchange → DSP)         deadline: 80ms
POST /rtb/v2.5/win    (exchange → DSP)         body: {bidId, clearingPrice}
GET  /img/imp?aid=... (browser → exchange)     fires impression beacon
GET  /img/clk?aid=... (browser → exchange)     fires click, 302 to landing

6. High-level architecture

[Publisher] ──► [Edge LB / TLS term] ──► [Bid Router (Netty/gRPC)] │ fan-out (parallel async) ┌───────────────────────────┼─────────────────────────┐ ▼ ▼ ▼ [DSP-1] [DSP-2] ... [DSP-N] └───────────────────────────┼─────────────────────────┘ ▼ [Auction Engine (in-proc)] │ ┌──────────────────────┼──────────────────────┐ ▼ ▼ ▼ [Win Notify] [Kafka log] [Creative cache] │ ▼ [Flink → ClickHouse + S3 (Parquet)]

The Bid Router holds the request in memory, fans out to DSPs with a hard deadline (80 ms), collects bids as they arrive, picks the winner at deadline, returns the response. Everything else (logging, billing, beaconing) is asynchronous.

7. Data model & storage

7.1 Auction log (Kafka → ClickHouse)

CREATE TABLE auctions (
  auction_id String, ts DateTime64(3),
  pub_id UInt32, imp_type Enum,
  bids Nested(dsp_id UInt32, price_cpm Decimal(10,4), latency_ms UInt16),
  winner_dsp UInt32, clearing_cpm Decimal(10,4)
) ENGINE = MergeTree PARTITION BY toYYYYMMDD(ts) ORDER BY (pub_id, ts);

7.2 DSP config (Redis)

Endpoint, QPS quota, current QPS counter, timeout SLA, blacklist categories — hot-loaded into the router process every 30 s.

7.3 Cold storage

Raw bid requests Parquet on S3, 90-day retention for ML training and audit.

8. Detailed component design

8.1 Bid Router

Single-threaded event-loop per core (Netty / Vert.x). Async HTTP client with connection pooling (1 conn per DSP × cores). Deadline timer cancels late bids. CPU-pinned, GC-tuned (ZGC or off-heap pools).

8.2 Auction engine

Filters: brand-safety category match, advertiser blocklist, deal-ID priority, reserve price. Then second-price: winner pays max(secondHighest, reserve) + $0.01. Tie-break by latency, then DSP score.

8.3 Throttling and dynamic timeout

Each DSP gets an adaptive timeout based on rolling p95 (PID-controlled). DSPs that exceed timeout 5x in 1 min are temporarily downsampled. Token-bucket per-DSP enforces QPS quota.

8.4 Cookie sync & user matching

Pre-auction: pixel sync builds (exchange-uid ↔ dsp-uid) mapping in a ScyllaDB. Lookup < 2 ms p99 to enrich BidRequest with audience segments.

9. Scaling & bottlenecks

  • Tail latency: DSP at p99=120 ms wrecks budget; use hedged requests only as last resort due to traffic doubling.
  • GC pauses: off-heap object pools, low-allocation parsing (jsoniter / Protobuf direct).
  • Network jitter: co-locate router and DSP in same metro; cross-region adds 30+ ms.
  • Bid log throughput: sample 10% raw, 100% summary; Kafka with snappy compression, 1 MB batches.

  • "The 100 ms budget is the system's prime constraint — every architectural choice is in service of tail latency."
  • "I'd be conservative on hedged requests; doubling DSP traffic is expensive and causes feedback loops."
  • "For first-price migration we'd add bid-shading services, which is a clean extension."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
OpenRTB 2.5 (Protobuf over HTTP/2)The protocol is the industry contract with 50+ DSPs; HTTP/2 multiplexing avoids TCP handshake within the 80 ms DSP timeout, and Protobuf parsing is 3-5x faster than JSON which matters when each of 5 M QPS allocates parsers.OpenRTB JSON (rejected: GC pressure at 5 M QPS); gRPC native (DSPs don't all support it).
Netty / Vert.x bid routerSingle-threaded event-loop per core handles the 100 ms hard deadline with predictable latency; one router box at 1k QPS holds ~100 in-flight requests, and ~5,000 boxes serve the 5 M QPS without thread-context-switch overhead.Tomcat / Spring (rejected: thread-per-request collapses at 100k connections); Go net/http (viable, but JVM ecosystem here for ZGC + jsoniter).
ZGC / off-heap pools (JVM)A 50 ms GC pause inside an 80 ms deadline drops the bid; ZGC keeps p99 pauses under 1 ms even with 32 GB heaps, and off-heap pools for BidRequest objects eliminate per-request allocation in the 5 M QPS hot path.G1GC (rejected: 10-50 ms pauses); CMS (deprecated); Shenandoah (similar to ZGC, ZGC chosen for predictability).
jsoniter / Protobuf direct parsingStandard Jackson allocates per field; jsoniter and Protobuf direct parse onto pre-allocated buffers so a 1 KB BidRequest avoids ~30 small allocations — multiplied by 5 M QPS that's the difference between 80 ms p99 and missing the deadline.Jackson (rejected: GC churn); Gson (slower); fastjson (security history).
Apache Kafka (auction log)400 B requests/day = 1 KB compressed produces 5 GB/s sustained; Kafka with snappy compression and 1 MB batches absorbs this off the hot path, and transactional producers give exactly-once into ClickHouse for billing.Pulsar (smaller ops community); Kinesis (per-shard limits force excessive sharding at 5 GB/s).
ClickHouse (auction analytics)Bid analytics need slice/dice on (publisher, dsp, time) over 400 TB/day; ClickHouse's MergeTree partition-by-day with sparse index serves p99 dashboard queries in <500 ms where Druid's segment overhead would explode at this cardinality.Druid (rejected: too many segments at 400 B rows/day); BigQuery (per-scan cost).
S3 + Parquet (cold storage)90-day retention of raw bid requests = 36 PB; Parquet column pruning makes ML training on bid features economical, and S3 lifecycle to Glacier moves day-90 files automatically without ops work.HDFS (operational overhead at 36 PB); GCS (similar, AWS chosen for closer to DSP infra).
ScyllaDB (cookie sync / user matching)The (exchange-uid ↔ dsp-uid) lookup before each auction must complete <2 ms p99 at 5 M QPS; Scylla's shard-per-core C++ implementation hits sub-ms p99 on point lookups where Cassandra would 5-10x that under load.Cassandra (rejected: JVM tail latency); Aerospike (viable, ScyllaDB cheaper at this scale); Redis (RAM cost prohibitive for 2 B users).
Redis (DSP config / QPS quotas)20-50 DSPs × per-DSP timeout / quota / blacklist config is small (~MB) but read every request; Redis hot-loaded into router process every 30 s gives nanosecond reads with bounded staleness — a config DB query per auction would burn the latency budget.etcd (rejected: read latency 1-2 ms); ZooKeeper (heavier).
Apache Flink (post-auction ETL)Windowed aggregations on auction outcomes (DSP win rate, latency p99) need exactly-once into ClickHouse; Flink's Kafka transactional sink + RocksDB state cleanly handles late bid notifications and billing-grade accuracy.Spark Streaming (rejected: micro-batch lag); Kafka Streams (state size limits).
Second-price auctionTruth-telling incentive minimizes DSP bid-shading complexity; though the industry is shifting to first-price, second-price is simpler to onboard new DSPs and the engine logs both clearing and second-bid for future migration.First-price (would force bid-shading services); VCG (overkill for K=1 slot).
Hard 80 ms deadline timerWithout a deadline, slow DSPs would bloat p99 across the fleet; the timer cancels late bids deterministically so the 100 ms publisher SLA holds even when 1-2 DSPs are degraded — predictable failure beats variable latency.Adaptive deadline (used too — but a hard floor must exist); hedged requests (rejected: doubles outbound traffic).
PID-controlled adaptive DSP timeoutEach DSP's rolling p95 informs its per-request deadline within the 80 ms cap; PID smoothing avoids flapping where a DSP is briefly slow then fast, and 5x-timeout-in-1-min downsampling protects budget from a hung DSP.Static per-DSP timeout (rejected: doesn't adapt to ops issues); EWMA only (no integral term, slow to converge).
