Design Budget Pacing & Frequency Capping (avoid overspend, cap frequency)

Hard ad-tech storage-cache

1. Problem statement

Build the system that enforces (a) budget pacing: spend the daily budget evenly through the day rather than burning it in the first hour, and (b) frequency capping: limit how many times a given user sees the same ad in a window. Both must be checked inline in the auction (sub-millisecond) at 1 M+ QPS, but with strict enforcement so that no campaign overspends by more than ~1% and no user sees an ad more than the cap.

2. Functional requirements

  • Per-campaign daily & lifetime budgets; intra-day pacing (ASAP, even, accelerated).
  • Frequency caps: per ad, per campaign, per advertiser, in windows (1 h, 1 day, 7 days).
  • Real-time admission decision: pass / drop in < 1 ms.
  • Reconcile asynchronous spend events (clicks, impressions) against budgets.
  • Recover gracefully from late events without permanent overspend.
  • Reset frequency caps at window boundaries; fast lookup at 5 M QPS read.

3. Non-functional requirements

  • Latency: p99 admission < 1 ms; cap lookup p99 < 0.5 ms.
  • Scale: 5 M auction QPS, 2 B users × 1 KB cap state = 2 TB.
  • Accuracy: overspend ≤ 1% of daily budget; cap exceedance ≤ 0.5%.
  • Availability: 99.99% — failures must "fail closed" (don't serve) rather than overspend.

4. Capacity estimation

  • Auction QPS: 5 M, each does ~3 cap lookups → 15 M Redis GETs/sec.
  • Spend events: 25k clicks/sec + 1.2 M impressions/sec → 1.25 M decrements/sec.
  • Frequency state: 2 B users × 100 ads/user × 16 B = 3.2 TB; sharded across 64 Redis clusters.
  • Budget state: 10 M campaigns × 1 KB = 10 GB, fits in a single Redis Cluster.

5. API design

GET  /admit?campaign=...&user=...&ad=...   → {ok:bool, reason}
POST /spend  {campaign_id, amount_micros, event_id}   → idempotent decrement
POST /freq/incr {user_id, ad_id, ts}                  → updates HLL/window
POST /budget/reset (cron daily)                       → roll daily counters
GET  /pacing/curve?campaign=...                       → expected vs actual

6. High-level architecture

┌─────── Auction Hot Path ───────┐ [Ad Mixer] ──► [Pacer Sidecar (in-mem)] ──► [Redis Cluster: budgets] │ │ │ │ └─────► [Redis Cluster: freq caps (HLL/Bitmap)] │ ▼ [Kafka: spend events] ──► [Flink: aggregate, dedup] ──► [Redis decrement] │ ▼ [Spanner: source-of-truth ledger] │ ▼ [Pacing Controller (PID)] tunes per-campaign admission rate

The auction path checks an in-memory token bucket replenished by the Pacing Controller. Spend events flow asynchronously through Kafka → Flink → Redis decrement, then are persisted to Spanner as the audit ledger. The PID controller adjusts each campaign's admission probability every 1 minute based on actual vs target spend.

7. Data model & storage

7.1 Budget state (Redis hash, per campaign)

HSET cmp:{id}
  daily_budget_micros 100000000
  spent_micros 23450000
  pacing_rate 0.85          # admission probability
  reset_at 1714521600
  status active

7.2 Frequency cap (Redis bitmap or HLL)

Key = freq:{user_id}:{ad_id}; value = small list of (timestamp, count) within window. Or per-user bitmap of ads seen, expired by window.

7.3 Spend ledger (Spanner)

Append-only event_id-keyed table for idempotent reconciliation; daily reconcile vs Redis to detect drift.

8. Detailed component design

8.1 Token-bucket pacer

Each campaign has a bucket capacity = budget × overshoot-tolerance. Refill rate = budget / day_seconds × multiplier(t) where multiplier(t) follows the expected impression curve (more during prime time). Decrement is best-effort; exact spend reconciles via Flink.

8.2 PID-controlled pacing

Target spend by minute t = budget × cdf(t). Error = actual − target. Output is a 0..1 admission probability that throttles auctions to keep error small. Anti-windup clamps so depleted budgets don't get a burst when refilled.

8.3 Frequency capping

Three options ranked by cost: (1) sorted-set of impression timestamps trimmed by window — exact, ~50 B/user/ad; (2) sliding-window counter via Redis Cell — approximate, ~16 B; (3) HLL per (user, window) for cap-by-distinct-creatives — very compact. Choose per cap type.

8.4 Reconciliation

Hourly Flink job compares Spanner ledger to Redis. If drift > 0.5%, reset Redis from ledger. Catastrophic loss of Redis is recoverable by replaying spend topic.

9. Scaling & bottlenecks

  • Hot campaigns: shard a single big-spender across multiple Redis keys; sum on read or split bucket.
  • Cap explosions: bots cause huge user keys; cap key size and quarantine outliers.
  • Time skew: use NTP-synced PTP-grade clocks at the pacer; reset windows on Redis-server time.
  • Cold start of new campaigns: aggressive pacing first hour with low cap to learn CPM, then ramp.

12. Talking points for the interviewer

  • "The trick is to admit that strong consistency at 1 ms is impossible at this scale — so I design for bounded overspend (~1%) and reconcile."
  • "Pacing is a control problem; I'd reach for a PID rather than ad-hoc rules because it generalizes across pacing modes."
  • "Frequency caps are where data structure choice matters most: HLL vs sorted set is a 10× memory difference at 2 B users."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis Cluster (budget hot path)5 M auctions/sec × 3 cap lookups = 15 M Redis GETs/sec at p99 <0.5 ms; Redis's single-threaded in-memory model gives deterministic sub-ms reads, and Lua scripts make decrement+check atomic without the round-trip cost of WATCH/MULTI.Memcached (rejected: no Lua atomicity); Postgres (rejected: 100x too slow); DynamoDB (cross-region latency).
Aerospike (frequency cap state)2 B users × 100 ads × 16 B = 3.2 TB sharded; Aerospike's hybrid RAM/SSD storage holds frequency-cap state at sub-ms p99 reads cheaper than pure-RAM Redis at this scale, with auto-sharding across 64 nodes.Pure Redis (rejected: RAM cost for 3.2 TB prohibitive); Cassandra (rejected: p99 too high for 0.5 ms target).
Lua atomic decrement (Redis)Spend decrement at 1.25 M decrements/sec must be idempotent (event_id-keyed) and atomic (no overspend race); Lua's atomic execution within Redis avoids the GET-CAS-SET race that would let two consumers double-decrement on Kafka rewind.WATCH/MULTI (rejected: retry storms); client-side CAS (race window); external lock service (latency).
HyperLogLog (HLL freq cap)Cap-by-distinct-creatives uses HLL ~12 KB per (user, window) for ~2% error vs 5 KB exact sorted-set per cap; at 2 B users this is the difference between 24 GB and 10 TB — the approximation is acceptable for cap-by-distinct.Exact sorted-set (rejected: 10TB+ memory cost); count-min sketch (similar, HLL chosen for distinct-count fit).
Redis sorted-set (exact cap)For exact cap-by-impression-count, sorted-sets keyed by (user, ad) trim by score (timestamp); ~50 B per (user, ad) is exact-correct and supports sliding-window via ZREMRANGEBYSCORE in <0.3 ms.Bitmap (rejected: ad-id sparse); Redis Cell module (approx, not exact); list (rejected: no efficient trim).
Token-bucket pacer (per campaign)Refill rate = budget/day_seconds × multiplier(t) gives smooth pacing without admit/deny flapping; the bucket model is well-understood, easy to reason about, and admits in O(1) — critical at the 5 M QPS auction rate.Leaky bucket (similar perf, less flexible refill); naive rate-limit per second (rejected: bursts).
PID controller (pacing)Target spend by minute t = budget × cdf(t); PID's P-I-D terms react to current error, accumulate to fix bias, and dampen oscillation — a hand-tuned threshold rule would over- or under-pace under traffic shifts (Black Friday, time zone roll).Bang-bang control (rejected: oscillates); ML-based pacer (overkill, slower to debug).
Apache Kafka (spend events)1.25 M decrements/sec from clicks + impressions need a durable replayable log so that a Redis cluster failure can replay missed decrements; partition-by-campaign avoids hotspots on big spenders.RabbitMQ (rejected: no replay); Kinesis (shard limits at this rate).
Apache Flink (spend aggregator)1-minute aggregation windows of spend events keyed-by-campaign feed both Redis decrement and Spanner ledger; Flink's exactly-once + windowed sums prevent double-counting on Kafka rewind, and idempotent Redis decrement (keyed by event_id) survives retries.Spark Streaming (rejected: micro-batch lag for 1-min PID loop); Kafka Streams (state-size limits at TB).
Spanner (spend ledger / source-of-truth)Append-only event_id-keyed ledger needs ACID for daily billing reconciliation; Spanner's external consistency means finance reads the same total all advertiser dashboards saw, even cross-region — and the ledger reconciles Redis drift hourly.Postgres (rejected: no global ACID); Cassandra (no ACID); BigQuery (no idempotent UPSERT).
Fail-closed on Redis outageIf Redis is unreachable, the auction admission decision returns "no" — a brief revenue loss is recoverable, but uncapped overspend is not (advertisers can't recover their dollars from a vendor); the SLA explicitly trades availability for accuracy.Fail-open (rejected: catastrophic overspend); fallback to Spanner (rejected: too slow at 5 M QPS).
NTP-PTP synced clocksPacing windows reset on a wall clock; clock skew >100 ms across pacers causes per-shard drift in admission probability; NTP/PTP keeps hosts within ~1 ms, and reset operations use Redis-server time as the canonical clock.Local clocks only (rejected: skew up to seconds); centralized time service (extra dependency).
Hourly Flink reconcilerDrift between Redis (decrement) and Spanner (ledger) accumulates from late events / retries; an hourly reconciler resets Redis from ledger if drift >0.5%, bounding total error to the <1% SLA without blocking the auction path.Real-time reconcile (rejected: latency cost); daily only (rejected: drift can exceed SLA mid-day).
