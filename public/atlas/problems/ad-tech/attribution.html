Design Multi-touch Attribution System (last-touch → MTA models)

Hard ad-tech analytics

1. Problem statement

Build an attribution system that, given the full sequence of ad touchpoints (impressions, clicks, video views) a user saw before converting (purchase, signup, install), apportions credit across those touchpoints. Support last-touch as a baseline, plus first-touch, linear, time-decay, position-based, and data-driven multi-touch attribution (MTA) using Markov / Shapley models. Outputs feed advertiser ROAS dashboards and bidding optimization.

2. Functional requirements

  • Ingest impressions, clicks, conversions from web, mobile, CTV pixels and SDKs.
  • Stitch user journeys across devices (deterministic + probabilistic).
  • Apply attribution windows (e.g., 1-day view, 7-day click, 28-day install).
  • Run multiple attribution models in parallel; expose per-model credit.
  • Recompute when late conversions arrive (e.g., a 14-day delay).
  • Per-advertiser tenant isolation; PII compliance (GDPR / CCPA delete).

3. Non-functional requirements

  • Latency: conversions surface in dashboards within 1 hour; full MTA recompute nightly.
  • Scale: 200 B touchpoints/day, 500 M conversions/day, 2 B users.
  • Accuracy: deterministic attribution for click-based models; calibrated probabilistic for view-through.
  • Compliance: 30-day max retention for raw user paths; right-to-be-forgotten in 30 days.

4. Capacity estimation

  • Touch volume: 200 B/day × 0.5 KB = 100 TB/day raw.
  • User journeys: 2 B users × avg 100 touches × 100 B = 20 TB hot state.
  • Compute: nightly MTA on 7-day window ~= 1 PB scan; Spark cluster ~5,000 cores for 4 h job.
  • Conversion ingest: 6k/sec average, 50k/sec peak (Black Friday).

6. High-level architecture

[Pixels/SDK] ──► [Ingest GW] ──► [Kafka: touches] │ ▼ [Flink: Sessionize per userKey] │ ┌────────────────────┼────────────────────┐ ▼ ▼ ▼ [User Journey Store] [Identity Graph] [S3 raw] (Cassandra, 30d TTL) (Spanner) │ ▼ [Conversion Joiner] ─── matches conversion ts back through window │ ▼ [Attribution Engine] ├─ Last-touch (real-time) ├─ Linear / Time-decay (real-time) └─ Markov / Shapley (Spark nightly) │ ▼ [Druid + BigQuery] → Advertiser dashboards

Real-time path computes rule-based models (last-touch, time-decay) within minutes. Data-driven MTA runs as a Spark batch over the previous 7-day journey log nightly because Markov fitting and Shapley value enumeration need the full graph.

7. Data model & storage

7.1 User journey (Cassandra, wide row)

CREATE TABLE journey (
  user_key text, day date,
  ts timestamp, ad_id text, campaign_id text, type text,
  PRIMARY KEY ((user_key), day, ts)
) WITH default_time_to_live = 2592000;  -- 30 days

7.2 Identity graph (Spanner)

Edges: (deviceId, cookieId, hashedEmail) with confidence weights. Updated by login events and probabilistic graph models.

7.3 Conversion log (BigQuery + Kafka)

Conversions stored with auctionId references when known, plus user_key for join on attribution.

8. Detailed component design

8.1 Sessionization

Flink keyed-by user_key, gap-session = 30 min. Emits ordered touch sequence; out-of-order events buffered up to 10 min watermark.

8.2 Conversion joiner

For each conversion at time T, range-scan journey for user_key in [T - 28d, T]. Rule-based credit applied immediately; result written to Druid for the real-time dashboard.

8.3 Markov MTA (nightly)

Build a Markov chain over channel transitions including a "conversion" absorbing state. Each channel's removal effect = Δ(P(conversion)) when its transitions are deleted; normalize to credit. O(C²) per advertiser where C = channel count (~50).

8.4 Shapley value

For an ordered touch sequence, credit channel i = average marginal contribution over all subsets. Approximated by Monte-Carlo sampling — 1k permutations per journey is empirically sufficient.

8.5 Identity stitching

Deterministic via login (hashedEmail). Probabilistic via (IP × UA × geo) co-occurrence with confidence threshold; respect privacy regulations and per-region opt-out.

9. Scaling & bottlenecks

  • Hot users (bots): cap journey length at 10k touches; spillover sampled.
  • Late conversions: 14-day reprocess window; advertisers see "preliminary" vs "final" labels.
  • MTA compute: partition by advertiser; co-locate journey + conversion shards in Spark to minimize shuffle.
  • GDPR delete: tombstone in Kafka; downstream consumers honor on next read.

  • "I'd ship last-touch + linear in v1; advertisers can compare. Data-driven MTA is v2 because the modeling and explainability work is non-trivial."
  • "Privacy is the hard constraint: 30-day TTL is built in, not bolted on, and identity stitching has region-specific gates."
  • "Late conversions are inevitable — the dashboard surfaces 'preliminary vs final' so advertisers don't get confused by restated numbers."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

Apache Kafka (touches ingest)200 B touchpoints/day = 100 TB/day raw needs a durable replayable log for nightly MTA recompute; Kafka's partition-by-userKey lets Flink sessionize without shuffle, and 30-day retention spans the longest 28-day install attribution window.Kinesis (rejected: shard limits at 100 TB/day); Pulsar (smaller ops community for this scale).
Apache Flink (sessionization)Keyed-by user_key with 30-min gap session and 10-min watermark needs exactly-once stateful processing at 200 B events/day; Flink's RocksDB state and watermark mechanics deliver bounded-lateness with idempotent sinks.Spark Structured Streaming (rejected: micro-batch lag for sessionization); Kafka Streams (state size limit at TB scale).
Cassandra (user journey, 30d TTL)2 B users × 100 touches × 100 B = 20 TB hot state with strict 30-day TTL for GDPR; Cassandra's TTL-aware tombstones auto-expire rows, wide-row partitioning by user_key keeps per-user touch retrieval to one disk seek.HBase (rejected: weaker TTL story); Bigtable (viable on GCP); ScyllaDB (better latency, smaller community).
Spanner (identity graph)(deviceId, cookieId, hashedEmail) edges need ACID merge when login events fire; Spanner's external consistency prevents split-brain identity merges across regions, and stored confidence weights inform probabilistic stitching.Postgres (rejected: no global ACID for cross-region writes); Neo4j (graph queries unneeded, edges are flat).
