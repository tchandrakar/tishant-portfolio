Design Ad Auction Engine (second-price / VCG, reserve pricing)

Hard ad-tech

1. Problem statement

Design the core auction engine that, given a list of N candidate ads with bids, predicted CTRs, and quality scores, selects K winners for K slots and computes the price each winner pays. Support GSP (generalized second-price), VCG (Vickrey–Clarke–Groves), reserve prices, deal-ID priority, and squashing. The engine is invoked inline by the ad server with a strict 5 ms compute budget per request.

2. Functional requirements

  • Rank N candidates (typically 50–500) into K slots (1–10).
  • Apply hard filters: brand safety, frequency cap, advertiser block lists.
  • Compute clearing price under GSP or VCG depending on tenant config.
  • Honor reserve prices (per slot, per advertiser tier).
  • Emit a deterministic auction record (auctionId, ranked candidates, prices, reasons).
  • Support A/B traffic splitting on auction logic itself.

3. Non-functional requirements

  • Latency: p99 < 5 ms for N=200, K=3 on a single core.
  • Throughput: 1.2 M auctions/sec across the fleet.
  • Determinism: same input → same output (auditable for billing disputes).
  • Auditability: every auction record retained 13 months.

4. Capacity estimation

  • QPS: 1.2 M × avg 100 candidates = 120 M candidate scorings/s.
  • Audit log: 1.2 M × 1 KB = 1.2 GB/s, 100 TB/day.
  • Memory: stateless engine; per-tenant config ~10 MB total in-process.
  • CPU: ~5 µs per candidate including filter chain → 1 ms for N=200.

5. API design

POST /v1/auction/run
  body: { auctionId, slots:K, reserves:[..], candidates:[
            {adId, advertiserId, bid, pCTR, qualityScore, dealId?, categories:[..]}
          ], rules:{type:"GSP"|"VCG", squash:1.0} }
  response: { winners:[{slot, adId, price, reason}], allowed:[..], filtered:[..] }

6. High-level architecture

[Ad Server] ──► [Auction Engine (in-process library)] │ ├─► [Filter Chain] (brand-safety, freq-cap, blocklist) │ ├─► [Scorer] score = (bid × pCTR)^squash × QS │ ├─► [Top-K Selector] (partial sort / quickselect) │ ├─► [Pricing Module] GSP | VCG | first-price │ └─► [Audit Emitter] → Kafka → S3 (Parquet)

The engine is a library linked into the ad server, not a remote service: an RPC hop costs more than the entire compute budget. Tenant rules are loaded from a versioned config service and hot-swapped atomically.

7. Data model & storage

7.1 Auction record (Parquet on S3 via Kafka)

message AuctionRecord {
  string auction_id = 1;
  int64  ts_micros  = 2;
  repeated Candidate candidates = 3; // ranked, with score & price
  string rules_version = 4;
  int32  k = 5;
}

7.2 Tenant config (Spanner + in-memory cache)

Auction type, squash factor, reserve prices, blocklists. Versioned; engine reloads on a 30 s tick or push notification.

8. Detailed component design

8.1 GSP pricing

Sort by score = bid × pCTR (or Quality Score). Winner of slot i pays the price that would have made score equal score of i+1: price_i = score_{i+1} / pCTR_i + $0.01, floored by reserve. Below reserve → no fill.

8.2 VCG pricing

Each winner pays the externality they impose on others: sum over j ≠ i of (score_j_without_i − score_j_with_i), divided by their pCTR. More truthful than GSP but harder to explain to advertisers and sensitive to pCTR calibration.

8.3 Squashing

score = bidα × pCTR (α < 1) reduces the influence of high bids on poor-quality ads, improving long-term marketplace health (Yahoo / Lahaie 2008).

8.4 Determinism & replay

Inputs are captured before randomization. Tie-breaking uses a hash of (auctionId, adId) — same input produces same output for replay tools used in billing disputes.

9. Scaling & bottlenecks

  • Sort cost: use std::nth_element / partial_sort — O(N log K), not O(N log N).
  • Filter chain order: cheap filters first; advertiser blocklist as a Bloom filter.
  • Audit log bandwidth: Snappy compression, batched flushes; Kafka producer with linger 5 ms.
  • Config reload: RCU pattern to avoid locks on the hot path.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
In-process C++/Rust libraryThe 5 ms p99 budget for N=200 candidates leaves no room for network hops; a same-process call costs ~50 ns vs ~500 µs for the cheapest RPC, so the engine ships as a linked library inside every ad-server binary across the 1.2 M QPS fleet.gRPC microservice (rejected: RPC overhead alone exceeds budget); JNI shim (GC pauses unacceptable at p99).
std::nth_element / quickselectFor Top-K selection where K=3 and N=200, partial sort is O(N log K) ≈ 200 ops vs full sort O(N log N) ≈ 1500 ops; over 1.2 M QPS this saves CPU equivalent to ~50 cores and tightens p99 by ~1 ms.std::sort (rejected: wasted work); heap-based Top-K (similar perf, more code).
Bloom filter (advertiser blocklist)Per-tenant blocklists can have millions of entries but blocklist checks happen N times per auction; a 16 MB Bloom filter at 0.1% FPR rejects 99.9% of blocked candidates in <100 ns each, with the rare false positive verified via exact lookup.Hash set (rejected: cache-unfriendly at this size); cuckoo filter (similar, Bloom simpler).
RCU config reload patternTenant rules (squash factor, reserves) reload every 30 s without locking the 1.2 M QPS hot path; readers atomically dereference an immutable config pointer and writers swap atomically — zero contention, zero stalls.RWLock (rejected: cache-line bouncing at this QPS); event-loop reload (delays config propagation).
Apache Kafka (audit log emit)1.2 GB/s of auction records must be durable for 13-month billing audit; Kafka's batched producer with 5 ms linger absorbs this without blocking the 5 ms compute, and snappy compression cuts the wire by 60%.Direct S3 writes (rejected: per-request latency); Kinesis (shard limits at 1.2 GB/s).
S3 + Parquet (auction archive)13 months × 100 TB/day = 40 PB cold storage with column pruning for billing replays; Parquet's predicate pushdown lets a billing dispute scan one campaign's records without reading the full day's 100 TB.HDFS (operational cost at 40 PB); Avro on S3 (no column pruning).
Spanner (tenant config)Auction rules per tenant need ACID updates with global readers; Spanner's external consistency lets advertiser console writes be visible to all engine replicas within seconds, and versioned rules enable rollback on a bad squash deploy.etcd (rejected: write throughput at 10 M tenants); MySQL (no global ACID).
GSP pricing (default)price_i = score_{i+1}/pCTR_i + ε is computable in 4 floating-point ops per slot — well under the 5 ms budget; near-truthful in practice and explainable to advertisers, reducing billing-dispute volume.VCG (rejected for v1: more ops, harder to explain, sensitive to pCTR error); first-price (forces bid-shading externally).
VCG pricing (opt-in)For tenants prioritizing strict truth-telling, VCG offers dominant-strategy bidding; cost is O(K·N) externality computation, still <5 ms for N=200 K=3 because cached score totals make removals O(1).Ironclad VCG with pricing curves (overkill); first-price (different incentive model).
Squashing α=0.85score = bid^α × pCTR dampens bid relative to quality; empirically (Lahaie 2008) improves marketplace health by ~5-15% revenue on quality-tilted slots, configurable per tenant for A/B testing.α=1 pure bid (rejected: degrades user experience); α=0 ignore bid (rejected: ignores advertiser value signal).
Deterministic hash tie-breakTie-breaking by hash(auctionId, adId) yields same output for same input — required for replay tools used in billing disputes 13 months later; alternative random tie-breaks would make replays inconsistent.Random tie-break (rejected: non-deterministic); first-seen (depends on candidate order, not stable).
Snappy compression (audit)1.2 GB/s raw → ~480 MB/s wire with snappy at ~3 GB/s decompression; faster than gzip (~150 MB/s decompress) which matters when Flink replays 13 months for a billing dispute.gzip (rejected: slow decompress); zstd (close, snappy chosen for stability).
