Design Google Ads (auction + serving + reporting)

Very Hard ad-tech recommendation

1. Problem statement

Design a search and display advertising platform like Google Ads. Advertisers create campaigns with keyword targeting, budgets, and bids. When a user issues a query, the system runs an auction across millions of candidate ads in <100 ms, serves the winners, tracks impressions and clicks, charges advertisers under a generalized second-price (GSP) model, and reports near-real-time performance to advertisers.

2. Functional requirements

  • Advertiser console: campaign / ad-group / keyword CRUD, bid management, budgets.
  • Real-time auction at query-time: candidate selection, scoring, ranking, pricing.
  • Ad serving with click and impression tracking, fraud filtering.
  • Budget pacing across the day; stop serving when daily cap is exhausted.
  • Reporting: dashboards with hourly aggregates, daily attribution, conversion tracking.
  • Quality Score model that combines bid, predicted CTR (pCTR), and landing-page experience.

3. Non-functional requirements

  • Latency: auction p99 < 80 ms; total ad-serving budget < 120 ms inside the SERP.
  • Scale: 100 B searches/day → ~1.2 M QPS peak; 10 M active advertisers; 50 B keywords.
  • Availability: 99.99% — every dropped auction is lost revenue.
  • Consistency: eventual for reporting, strong for budget enforcement.

4. Capacity estimation

  • QPS: 1.2 M peak auctions/s × ~30 candidate ads scored each = 36 M scoring ops/s.
  • Storage: campaigns ~10 TB; click/impression log ~50 TB/day raw, ~5 TB after compaction.
  • Bandwidth: 100 B impressions/day × 2 KB beacon = 200 TB/day.
  • Memory: keyword inverted index ~2 TB sharded across ~200 serving replicas.

5. API design

POST /v1/campaigns                     {name, dailyBudget, bidStrategy}    → campaignId

6. High-level architecture

[User Query] → [SERP FE] → [Ad Mixer] ─────────────┐ │ │ ▼ ▼ [Candidate Selector] [pCTR Model Server] (inverted index) (TF Serving / GPU) │ │ ▼ │ [Auction / GSP] ◄────────────┘ │ ▼ [Ad Server + Beacons] │ ▼ [Kafka click/impr log] → [Flink ETL] → [BigQuery + Druid] │ ▼ [Budget Pacer (Redis token bucket)]

The Ad Mixer fans out to candidate selection (keyword + audience match), gets pCTR predictions, runs the GSP auction, and writes the auction record to Kafka. Beacons fire on render and click; Flink performs deduplication, bot filtering, and budget decrement.

7. Data model & storage

7.1 Campaign metadata (Spanner)

CREATE TABLE campaigns (
  campaign_id INT64, advertiser_id INT64,
  daily_budget_micros INT64, status STRING,
  PRIMARY KEY (advertiser_id, campaign_id)
) INTERLEAVE IN PARENT advertisers;

7.2 Inverted index (Bigtable + in-memory)

Keyword → posting list of (adGroupId, bid, matchType). Replicated to each serving shard; reload via versioned snapshots every 5 minutes.

7.3 Reporting (BigQuery + Druid)

Druid for sub-second slice/dice on advertiser dashboards; BigQuery for ad-hoc and billing reconciliation.

8. Detailed component design

8.1 Auction (GSP)

Score = bid × pCTR × QualityScore. Top-K by score; price for slot k = (score_{k+1} / pCTR_k) + ε. Reserve price floors enforced. Auction record (auctionId, candidates, prices) is logged for audit and billing.

8.2 pCTR model

Logistic regression with feature crosses for candidate generation, GBDT/DNN for ranking. Features: query × ad embeddings, historical CTR with smoothing, device, geo, dayparting. Online updates via streaming TF training.

8.3 Budget pacer

Redis token-bucket per campaign. Pacer admits an auction only if a token is available; refills at budget / day_seconds with smoothing to avoid early-day exhaustion.

9. Scaling & bottlenecks

  • Index hot keywords ("iphone"): partition by keyword hash with a small replicated hot-key set served from RAM.
  • pCTR latency: batch scoring per request, INT8 quantization, locality with the candidate selector.
  • Click fraud: stateful Flink job correlating IP, UA, click cadence; suspicious clicks marked non-billable.
  • Budget overshoot: Lua atomic decrement in Redis; reconcile end-of-day in Spanner.

12. Talking points for the interviewer

  • "I'd separate the hot-path auction from the cold-path reporting — the latter can absorb minutes of lag, the former cannot afford 10 ms."
  • "Budget enforcement is the trickiest tension: strong consistency would slow the auction, so I use a token bucket per shard plus periodic global reconciliation."
  • "Quality Score keeps incentives aligned — purely highest-bidder pricing degrades user experience and long-term revenue."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Google Spanner (campaign metadata)10 M advertisers and 50 B keywords need globally-consistent edits with strict ACID for billing reconciliation; Spanner's external consistency lets us bill against the same row read by the auction without reconciliation drift across regions.Sharded MySQL (rejected: manual resharding pain at 10 TB); CockroachDB (less mature billing-grade ACID at this scale).
Bigtable (inverted keyword index)The 2 TB keyword posting list is read-heavy and sharded by keyword hash; Bigtable's row-key range scans serve posting lists at <5 ms p99 while the in-memory replicas absorb the 1.2 M auction QPS hot path.Cassandra (rejected: GC stalls hurt p99); Elasticsearch (heavyweight, slower for simple posting-list lookups).
Redis token bucket (budget pacer)Budget admit/decrement happens inside an 80 ms auction at 1.2 M QPS; Redis Lua atomic decrement gives sub-millisecond enforcement with idempotent semantics, while Spanner reconciles end-of-day to bound overspend to ~1%.In-process counters (rejected: cross-shard inconsistency); Spanner direct (too slow for hot path at 50 ms+ commit latency).
TensorFlow Serving on GPU (pCTR ranker)Per auction we score ~30 candidates inside a 30 ms ranking budget; TF Serving with INT8 quantization handles 36 M scoring ops/s on a GPU fleet co-located with the candidate selector to avoid an RPC hop.CPU-only XGBoost (rejected: 3x slower at this batch size); ONNX Runtime (similar perf, weaker TF ecosystem fit).
GBDT for CTR (ranking model)Logistic regression with crosses for cheap candidate generation, GBDT for final ranking — GBDT captures non-linear feature interactions (query × ad × geo × daypart) better than LR while training nightly on 100 B impressions stays tractable.Pure DNN (rejected: training cost, latency); Pure LR (under-fits non-linear ad-quality signals).
Apache Kafka (impression / click log)200 TB/day of beacon traffic plus auction records need a durable replayable log; Kafka's partitioned append-only model lets Flink replay last-N-days for billing recompute and feeds both BigQuery (cold) and Druid (hot) sinks.Pulsar (smaller ecosystem); AWS Kinesis (lower throughput per shard, harder for 5 GB/s peak).
Apache Flink (ETL + click fraud)Stateful click fraud detection correlates IP / UA / cadence over 5-minute keyed windows; Flink's exactly-once + RocksDB state backend supports this at click volumes (~100 B/day) with bounded watermarks for late beacons.Spark Structured Streaming (micro-batch lag); Kafka Streams (state size limits at TB scale).
Apache Druid (advertiser dashboards)Advertisers slice/dice campaign × hour × geo on dashboards expecting <1 s response; Druid's bitmap-indexed columnar segments deliver sub-second OLAP on the last 30 days where BigQuery would charge per scan and add latency.BigQuery (high per-query cost for dashboards); ClickHouse (viable, but Druid's ingestion + lambda support more mature here).
BigQuery (billing & ad-hoc)Daily billing reconciliation scans 5 TB compacted logs; BigQuery's serverless model lets finance run ad-hoc audit queries without managing a cluster, and federated queries against Spanner cross-check spend vs ledger.Hive on HDFS (operational overhead); Snowflake (similar capability, more expensive at GCP-native ingest).
GSP auction (vs VCG)GSP scoring (bid × pCTR × QualityScore) and pricing (next-score / pCTR + ε) is well-understood by 10 M advertisers; near-truthful in practice and avoids the pCTR-calibration sensitivity that breaks VCG revenue.VCG (rejected: complex, advertiser-confusing, revenue volatile); First-price (forces bid-shading complexity).
Quality Score squashing (α≈0.85)Pure bid×pCTR rewards rich advertisers with bad creative; raising bid to power α<1 dampens the bid signal so high-quality ads win when scores are close, protecting long-term marketplace health and SERP quality.No squashing (rejected: degrades user experience); reserve-only floors (blunt instrument).
Versioned snapshot reload (5 min)Hot-reloading the 2 TB keyword index without serving stalls requires atomic versioned swaps; 5-minute snapshots balance index freshness for new campaigns against the cost of rebuilding posting lists across 200 replicas.Live diff streams (rejected: complex correctness for posting list edits); hourly reload (too stale for new campaign launches).
