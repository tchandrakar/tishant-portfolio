Design Ad Click Tracking (clickstream → idempotent counter)

Hard ad-tech analytics

1. Problem statement

Build a click-tracking pipeline that ingests every ad click, deduplicates retries and replay attacks, filters bot traffic, attributes the click to the original auction, decrements the advertiser's budget, and emits accurate billing-grade aggregates. The pipeline must be idempotent end-to-end: a redirect retry, a Kafka rewind, or a downstream re-run must never double-count.

2. Functional requirements

  • Click endpoint that issues a 302 redirect to the advertiser landing page.
  • Sign click URLs so they cannot be forged or replayed.
  • Deduplicate on (auctionId, userId) within a 24-hour window.
  • Bot filtering (signature-based + ML-based, both online and offline).
  • Decrement campaign budget; settle billing daily against the audit log.
  • Hourly and daily aggregates per campaign, ad-group, creative, geo.

3. Non-functional requirements

  • Latency: redirect p99 < 50 ms (user is waiting on the browser).
  • Scale: 100 B impressions/day, ~2 B clicks/day, ~25k clicks/sec peak.
  • Durability: zero data loss on click; impressions sampled 100% but tolerant.
  • Accuracy: billing aggregate within 0.01% of audit log.

4. Capacity estimation

  • QPS: 25k clicks/sec, 1.2 M impressions/sec.
  • Storage: 2 B clicks × 1 KB = 2 TB/day raw; 100 B impressions × 0.5 KB = 50 TB/day.
  • Kafka: 200 MB/s sustained on click topic, 5 GB/s on impression topic.
  • Dedup state: 2 B keys × 64 B = 128 GB Bloom filter (24 h TTL).

5. API design

GET  /clk?aid=...&auc=...&u=...&sig=HMAC   → 302 to landing
GET  /imp?aid=...&auc=...&u=...&sig=HMAC   → 1×1 GIF
POST /v1/click/replay  (internal)             → reprocesses from S3 with idempotency
GET  /v1/reports/clicks?campaign=...&range  → aggregated counts

6. High-level architecture

[Browser] ─302──► [Click Edge (Go/Rust)] ──► [Kafka: click_raw] │ HMAC verify │ ▼ ▼ [Redirect] [Flink: dedup + bot] │ ┌────────┴────────┐ ▼ ▼ [Kafka: click_clean] [Kafka: dlq_bot] │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ [Budget Dec] [Druid Realtime] [S3 Parquet] │ ▼ [Billing batch]

The edge tier does only HMAC verification and Kafka write before redirecting — it must be brutally simple to hit 50 ms p99. All filtering, dedup, and aggregation happen downstream where they don't block the user.

7. Data model & storage

7.1 Click event (Avro on Kafka)

{
  "click_id":  "uuid-v7",
  "auction_id":"...",
  "ad_id":"...", "campaign_id":"...",
  "user_id":"...", "ip":"...", "ua":"...",
  "ts_ms":1714,  "geo":"US-CA",
  "sig_valid":true, "edge_pop":"sjc-3"
}

7.2 Dedup state (RocksDB in Flink)

Key = (auctionId, userId). TTL = 24 h. Backed by RocksDB state with incremental checkpoints to S3.

7.3 Aggregates (Druid + S3 Parquet)

Druid for sub-second dashboards (last 30 days). Parquet on S3 partitioned by date for billing and historical analytics.

8. Detailed component design

8.1 HMAC-signed URLs

Click URL contains a signature: sig = HMAC_SHA256(secret, auctionId|adId|userId|ts). Edge verifies signature and freshness (≤ 30 min). Forged or replayed-after-TTL clicks rejected pre-Kafka.

8.2 Idempotent dedup

Flink keyed stream by clickId; first occurrence emits to click_clean, subsequent are dropped. clickId is a UUIDv7 generated at edge → monotonic, sortable, no central coordinator.

8.3 Bot filtering

Stage 1 (online, in Flink): rule list — bad UAs, datacenter IP ranges, click cadence > 10/s. Stage 2 (offline, hourly): GBM model on session features → backfill suspicion scores, recompute aggregates.

8.4 Exactly-once budget decrement

Flink → Kafka transactional sink (read-process-write). Budget service consumes click_clean with consumer group offsets committed in the same Kafka transaction as the budget Redis Lua decrement (idempotency token = clickId).

9. Scaling & bottlenecks

  • Hot campaigns: partition click_raw by hash(adId) — 256 partitions; rebalance via cooperative sticky assignor.
  • Late events: Flink watermark with 5-minute allowed lateness; very-late events go to a side output for nightly correction.
  • Edge GC: stack-allocated parsers, no JSON; protobuf only, pre-allocated buffers.
  • Reprocessing: S3 raw is the source of truth; full pipeline can replay deterministically.

12. Talking points for the interviewer

  • "Edge tier does the minimum: verify, log, redirect. Anything more would risk the 50 ms p99."
  • "Idempotency comes from a stable clickId generated at the edge. Without it, retries and replays poison the count."
  • "Billing differs from the dashboard by design — billing waits for the offline bot model; dashboards show the real-time view."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Go / Rust click edge50 ms p99 redirect requires zero GC stalls and minimal allocation; Go's goroutines or Rust's tokio handle 25k clicks/sec per box without the JVM warmup or GC pauses that make Java unfit for this redirect-critical hot path.Java (rejected: GC pauses risk 50 ms p99); Node.js (single-thread bottleneck); nginx + Lua (less observable).
HMAC-SHA256 signed URLsForged or replayed clicks must be rejected pre-Kafka or they pollute aggregates; HMAC-SHA256 (~1 µs verify) over (auctionId|adId|userId|ts) with a 30-min freshness window blocks tampering at the edge before any state is written.JWT (rejected: heavier, JWS attack surface); plaintext tokens (forge-able); RSA sigs (~100 µs, too slow at 25k QPS).
UUIDv7 (clickId at edge)Time-ordered 128-bit UUIDs let the edge generate idempotency keys with zero coordination — no Snowflake server, no DB sequence — and downstream Flink dedup gets sortable keys for efficient RocksDB iteration.UUIDv4 (rejected: random, kills RocksDB locality); Snowflake IDs (extra service hop); DB sequence (round-trip kills 50 ms budget).
Apache Kafka (click_raw topic)2 B clicks/day (200 MB/s) plus 100 B impressions/day (5 GB/s) need durable replay for billing reprocess; Kafka's partitioned log lets us hash-partition by adId across 256 partitions to absorb hot campaigns.Kinesis (rejected: shard limits at 5 GB/s); Pulsar (smaller ops community); RabbitMQ (no replay).
Avro (click event format)Schema evolution matters across 10+ downstream consumers (Flink, Druid, Spark); Avro's schema-registry-backed compatibility lets us add fields without breaking existing jobs, and the binary format is 60% smaller than JSON.Protobuf (rejected: weaker schema-registry tooling for streaming); JSON (3x bandwidth at 5 GB/s).
Apache Flink (dedup + bot)Keyed-stream dedup over a 24-h window needs exactly-once with sub-second latency at 25k clicks/sec; Flink's RocksDB state with incremental S3 checkpoints handles 128 GB dedup state with bounded recovery time.Spark Streaming (rejected: micro-batch latency); Kafka Streams (state size limit at 128 GB problematic).
RocksDB (Flink dedup state)2 B keys × 64 B Bloom = 128 GB doesn't fit in heap; RocksDB's LSM with TTL compaction stores keyed dedup state on local SSD, with incremental S3 checkpointing recovering from box failure in <5 min.Heap state (rejected: GC + size); Redis (network hop per dedup); standalone HBase (operational overhead).
Kafka transactional sinkRead-process-write exactly-once into click_clean + budget Redis Lua decrement requires atomic commit; Kafka transactions tie consumer-offset-commit + producer-write into one atomic step so a crashed Flink job can't double-decrement budget.At-least-once + idempotent consumers (more app logic); 2PC (rejected: latency hit).
Apache Druid (real-time dashboard)Last-30-day click counts sliced by campaign × hour × geo need <1 s response; Druid's bitmap-indexed segments with real-time ingestion from click_clean serves dashboards even as new events stream in.ClickHouse (similar, Druid's lambda-pattern integration richer); BigQuery (per-query cost on dashboards prohibitive).
S3 + Parquet (billing source-of-truth)Billing aggregates within 0.01% require recomputability from the immutable raw log; Parquet on S3 partitioned by date supports column-pruned scans for monthly billing reconciliation against Druid's real-time view.HDFS (operational at PB scale); Avro on S3 (no column pruning, slower scans).
GBM bot filter (offline stage)Two-stage filtering: simple rules online catch fast attacks; nightly GBM on session features (click cadence, IP reputation, UA entropy) catches sophisticated bots and backfills suspicion scores into billing aggregates.Pure rule-based (rejected: misses adversarial bots); deep neural net (overkill for tabular features, slower training).
Redis Cluster (budget Lua decrement)Budget decrement after dedup needs idempotency token + atomic decrement in <1 ms; Redis Lua executes the get-check-decrement script atomically and stores last-clickId per campaign to reject replays.Postgres (rejected: too slow at 25k decrements/s); in-memory only (loses durability on failover).
Watermark + 5-min latenessLate events (mobile reconnects, edge replays) arrive minutes after click; Flink's watermark with 5-min allowed lateness routes very-late events to a side output for nightly correction without blocking real-time.Strict event-time (rejected: drops late events); processing-time (loses billing accuracy).
Snappy on Kafka5 GB/s impression topic compressed to ~2 GB/s saves disk + network; Snappy's ~3 GB/s decompression keeps Flink workers from being CPU-bound on decompression.gzip (rejected: too CPU-heavy at 5 GB/s); lz4 (close, snappy chosen for ecosystem fit).
