Design Image CDN with on-the-fly resize / format conversion

Hard cdn file-storage

1. Problem statement

Design an image-optimization CDN (Cloudflare Images / Imgix / Akamai Image Manager). Customers upload one master image; consumers request arbitrary derivatives — width, height, crop, quality, format (WebP/AVIF/JPEG XL) — via URL parameters. The system must transform on the fly, cache aggressively, and never let a viral image melt the transformer fleet.

2. Functional requirements

  • Upload master image (up to 100 MB, JPEG/PNG/HEIC/RAW/TIFF).
  • URL-driven transforms: /img/abc?w=400&h=300&fit=cover&fm=avif&q=80.
  • Auto format negotiation via Accept header (serve AVIF to Chrome, JPEG to Safari ≤14).
  • Smart crop (face/saliency detection), watermarking, signed URLs.
  • Per-customer storage quotas, hotlink protection.

3. Non-functional requirements

  • Latency: warm cache p99 < 50 ms; cold transform p99 < 600 ms.
  • Scale: 5 M req/sec globally, 10 B masters stored, 100 B derivative URLs.
  • Availability: 99.99%; cache-miss must never page-out the origin.
  • Cost: AVIF saves ~30% bytes vs JPEG → directly drops egress bill.

4. Capacity estimation

  • QPS: 5 M derivative req/s; 99% cache-hit → 50 k transforms/sec at the edge.
  • Each transform: ~150 ms CPU (AVIF encode). 50 k/s × 0.15 s = 7,500 cores busy → ~250 transformer boxes (32 vCPU each).
  • Master storage: 10 B × 500 KB avg = 5 PB; replicated 3× across regions = 15 PB.
  • Derivative cache: 100 B URLs × 30 KB avg × 1% hot = 30 PB hot tier across all POPs.
  • Bandwidth: 5 M/s × 30 KB = 150 GB/s = 1.2 Tbps egress.

5. API design

POST  /v1/images                  multipart upload          → {imageId, masterUrl}
GET   /img/{imageId}?w=&h=&fit=&fm=&q=&crop=  Accept: image/avif → bytes
POST  /v1/images/{id}/variants    {name, params}            → variantUrl
DELETE /v1/images/{id}                                       → 204
GET   /v1/images/{id}/metadata                              → {w,h,format,faces[]}

6. High-level architecture

[Client] ─Accept: image/avif─► [Edge POP] │ hit → bytes │ miss ▼ [Derivative Cache (regional)] │ miss ▼ [Transformer Pool] ──► [Master Store (S3)] │ ▲ └─ writes back ──────┘ [Smart-crop / Face-detect svc] (gRPC)

Edge serves the derivative if cached. On miss, a regional derivative cache is consulted (so 300 edge misses don't hit 300 transformers). On second-tier miss, a transformer pulls the master from S3, decodes once, applies operations, encodes to target format, and writes back through both cache tiers.

7. Data model & storage

7.1 Master metadata (Postgres / DynamoDB)

CREATE TABLE images (
  image_id    uuid PRIMARY KEY,
  customer_id uuid,
  s3_key      text,
  width int, height int, format text,
  bytes bigint, created_at timestamptz,
  faces jsonb,        -- precomputed saliency boxes
  variants jsonb      -- named presets
);

7.2 Derivative cache key

key = imageId | sorted(params) | accept_format_class
TTL: 30 days; immutable since master_id changes invalidate.

Master objects sit in S3 / GCS with cross-region replication (DDIA Ch. 5: leader-based async replication, bounded staleness ~seconds). Derivative blobs live in NVMe at edge POPs with TinyLFU admission.

8. Detailed component design

8.1 Transformer

libvips (8–10× faster than ImageMagick, streaming pipeline, low RAM). Per request: decode → resize → color-correct → encode. AVIF encode is the long pole at ~120 ms for a 1080p frame; we cap concurrency per box and shed load via 503 + Retry-After when queue depth > 50.

8.2 Format negotiation

Edge varies on a normalized Accept bucket: {avif, webp, jpeg}. Cardinality stays at 3 instead of millions of raw header strings, keeping cache-key explosion bounded.

8.3 Smart crop

Async ML service (MobileNet-class) runs on master upload, stores face/saliency JSON in metadata. Crop math at request time uses precomputed boxes — no GPU on the hot path.

8.4 Single-flight coalescing

If 1,000 users request the same viral derivative simultaneously, the transformer fleet sees exactly one fetch + one encode. Followers wait on the in-flight future (≤ 600 ms) and stream the same result. Without this, a tweet-storm = pager.

9. Scaling & bottlenecks

  • Combinatorial cache explosion: attackers append ?w=401, 402, ... to bust cache. Mitigation: signed URLs or a per-customer allow-list of param dimensions.
  • AVIF encode CPU: two-pass libavif is too slow; use single-pass + speed=8.
  • Cold master fetch from S3: ~80 ms p99. Pre-warm hot masters into a regional object cache on first hit.
  • Memory pressure: 100 MP TIFFs OOM workers. Hard limit decoded pixel area per request (e.g. 50 MP).

12. Talking points for the interviewer

  • "The system is really a cache in front of a transform — get the cache key right and everything else is engineering."
  • "I'd push back on 'AVIF for everyone'; falling back to JPEG for old Safari is a real cost saver."
  • "The biggest production foot-gun is param-explosion abuse. Signed URLs are not optional at scale."
