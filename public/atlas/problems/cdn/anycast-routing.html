Design Anycast Routing for global CDN (BGP, geo-DNS, latency-based)

Hard cdn

1. Problem statement

Design how a global CDN steers each user to the "best" POP. The naive answer — "GeoDNS sends users to the nearest country" — falls apart on real internet topology where AS-paths, peering, and congestion dominate latency. Real CDNs combine BGP anycast, GeoDNS, and RUM-based latency steering. This problem covers the routing layer end-to-end.

2. Functional requirements

  • Route users to a POP that minimizes RTT, subject to capacity and policy.
  • Failover: a dead POP must be removed from rotation in < 30 s.
  • Honor compliance: EU users must terminate in EU POPs.
  • Support TCP/HTTP and UDP/QUIC equally — no L4 stickiness assumptions.
  • A/B traffic shifting (canary 1% to a new POP).

3. Non-functional requirements

  • Latency overhead: routing decision adds < 5 ms; DNS resolution < 30 ms p99.
  • Scale: 50 M req/s steered globally; 300 POPs, 1000+ peering sessions.
  • Availability: 99.999% routing-plane (anycast survives single POP loss with no DNS change).
  • Stickiness: p99 mid-session POP flap rate < 0.1% of connections.

4. Capacity estimation

  • BGP sessions: 300 POPs × 30 transit/peer per POP ≈ 9,000 active sessions.
  • DNS QPS: 5 M/s authoritative queries (lookups dominated by short TTLs).
  • RUM beacons: 1 in 10,000 page loads, 50 M/s page loads → 5 k/s beacons globally feeding the latency map.
  • Latency map size: ~16 M /24 IPv4 prefixes × 300 POPs × 4 B = ~20 GB; gz to ~2 GB.

6. High-level architecture

[Client] │ ① DNS query (with EDNS client-subnet) ▼ [Authoritative DNS] ── lookup ──► [Latency Map / Geo DB] │ ② returns anycast VIP ▼ [BGP cloud] ── shortest AS path ──► [Nearest POP advertising VIP] │ ③ TCP/QUIC handshake ▼ [Edge POP L4 LB (Maglev)] ──► [L7 proxy] ──► cache / origin ▲ └── ④ RUM beacon feeds back real RTT → updates Latency Map

Layer 1 is BGP anycast: one IP advertised from every POP. Layer 2 is GeoDNS, which picks which anycast pool to return (e.g., EU-only pool for GDPR zones). Layer 3 is RUM-driven steering, which adjusts the DNS answer based on measured RTTs from real users in the same /24.

7. Data model & storage

7.1 Latency map

// keyed by client /24 (IPv4) or /48 (IPv6)
prefix_to_pop_rtt:
  "203.0.113.0/24" → [{pop:"sfo", rtt_ms:18}, {pop:"sea", rtt_ms:32}, ...]
  -- updated every 60s from RUM + active probes

7.2 Capacity / health table

pop_state:
  pop_id, healthy(bool), bps_in, bps_capacity, cpu_pct, withdrawn(bool)

Stored in a globally-replicated KV (DDIA Ch. 5 leader-based async replication, edge reads bounded-stale by ~1 s). DNS resolvers cache the latency map locally and refresh every 30 s.

8. Detailed component design

8.1 BGP anycast

Every POP advertises the same /24 to its transits. Internet routers pick the shortest AS-path; ties broken by MED, local-pref, IGP cost. Failover is automatic: a withdrawn prefix is reconverged across the global table in 30–120 s. We add BFD on transit links for sub-second link-failure detection inside the POP.

8.2 GeoDNS

Authoritative DNS uses EDNS Client Subnet (RFC 7871) to see the resolver's /24 instead of the resolver IP. Returns one of N anycast VIPs based on country/policy. TTL is short (30–60 s) to enable fast steering changes. We never use TTL=0 — that breaks resolver caching and 10×s our QPS.

8.3 RUM-based steering

Browsers post tiny beacons with timing data. A streaming pipeline (Flink/Kafka) aggregates RTTs per (client /24, POP) tuple and updates the latency map. When the empirical RTT to POP B is >20% better than POP A for a /24, DNS starts answering with B for that prefix.

8.4 Capacity-aware steering

If a POP exceeds 80% utilization, the steering controller reduces its weight. At 95%, the POP is removed from DNS for new flows but existing anycast sessions continue. This avoids the BGP-flap stampede that withdrawal would cause.

9. Scaling & bottlenecks

  • Mid-session flap: BGP reconvergence mid-TCP kills the flow. Use anycast only at the L4 LB, then pin the 5-tuple via Maglev consistent hashing inside the POP. For long-lived flows (QUIC), connection-id allows resumption at any anycast POP — DDIA Ch. 8 partial-failure handling.
  • DNS resolver caching: ISPs ignore TTLs <5 min. Solution: keep anycast as the steady state and use DNS only for policy.
  • Latency map staleness: /24 with no recent RUM falls back to GeoIP heuristics.
  • Route hijack risk: RPKI-sign all prefixes, monitor for unauthorized announcements.

12. Talking points for the interviewer

  • "Anycast handles 99% of failover for free; DNS is for policy, not for speed of failover."
  • "GeoIP is a lie at the AS level — RUM is the only honest signal. I'd build the system around RUM from day one."
  • "The subtle bug is mid-session BGP flap on long-lived connections. QUIC's connection-id is the cleanest fix; for legacy TCP we lean on consistent-hashing the L4 LB."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

BGP anycast (one VIP per zone)Steers 50 M req/s without a per-request DNS lookup, and provides 99.999% routing-plane availability through automatic prefix withdrawal. Mandatory at our scale of 9,000 BGP sessions across 1,000+ peers.RPKI-unsigned (hijack-vulnerable); IRR-only filtering (weaker validation).
