Design Edge Cache Invalidation (purge at billions of files)

Very Hard cdn storage-cache

1. Problem statement

Design a CDN purge subsystem. A customer publishes a CMS update and expects the new HTML to appear globally in < 5 seconds, across 300 POPs and tens of thousands of cache servers, without nuking adjacent cache that's still valid. We must support URL purge, prefix/wildcard purge, and tag-based purge over a working set of 100 B+ cache entries.

2. Functional requirements

  • Purge by exact URL, by URL prefix, by surrogate key (Cache-Tag).
  • Purge propagation to all 300 POPs in < 5 s p99.
  • Idempotent and replay-safe (purges can be retried during partition).
  • Customer-scoped: a purge from tenant A cannot affect tenant B's keys.
  • Audit log of purge events for 90 days.

3. Non-functional requirements

  • Throughput: 1 M purge ops/sec aggregate; bursts of 100 M tag-purge fan-outs.
  • Latency: p50 propagation 1.5 s, p99 5 s, p999 30 s globally.
  • Storage: 100 B cache entries; tag index 10× that.
  • Availability: 99.99%; cache must "fail open" — invalidation outage cannot break serving.

4. Capacity estimation

  • Purge events: 1 M/s × 86,400 = 86 B/day. Each event ~200 bytes → 17 TB/day raw.
  • Fan-out per event: 1 → 300 POPs × ~200 boxes/POP = 60 k targets, but we deliver per-POP not per-box (POP-local broadcast).
  • Tag-index: avg 50 URLs per tag, p99 1 M URLs per tag (e.g. "homepage" tag).
  • Bandwidth for purge bus: 1 M/s × 200 B × 300 POPs = 60 GB/s on the global pub/sub.

5. API design

6. High-level architecture

[Customer API] → [Purge Coordinator] → [Global Pub/Sub bus] │ ┌───────────────────────────┼───────────────────────────┐ ▼ ▼ ▼ [POP-1 Relay] [POP-2 Relay] ... [POP-300 Relay] │ │ │ fanout to fanout to fanout to ~200 cache boxes ~200 cache boxes ~200 cache boxes │ ▼ [Local Tag Index] → mark entries tombstoned

The purge coordinator validates the request, assigns a monotonic purgeId (Lamport-style), and writes it to a globally-replicated, ordered log (Kafka-like, or a Quicksilver-style KV per Cloudflare). Each POP runs a relay that subscribes to the log and broadcasts purges intra-POP via UDP multicast or a local gRPC fanout.

7. Data model & storage

7.1 Tag index (per cache box)

// In-memory inverted index, persisted hourly
tag_to_keys: HashMap<tag, RoaringBitmap<cache_key_id>>
key_to_tags: HashMap<cache_key_id, Vec<tag>>

7.2 Tombstone vs eviction

We don't physically delete on purge — that would require a synchronous index write under load. Instead, each cache entry stores a generation counter and a min_valid_purge_id. A purge bumps the per-tag/zone generation; a read compares and treats stale generations as MISS. Physical reclaim happens lazily.

7.3 Wildcard / prefix

Prefix purges materialize into "purge generations" stored in a per-zone trie. Lookup at read time checks each prefix ancestor. This avoids enumerating all matching keys at write time, which would be O(billions).

8. Detailed component design

8.1 Purge coordinator

Stateless tier behind a regional API gateway. Authenticates, rate-limits per zone (e.g. 1k purge/sec), computes the canonical event, and writes to the ordered log with a fsync barrier. Returns purgeId immediately; propagation is async.

8.2 Global ordered log

Multi-region Raft (DDIA Ch. 5 leader-based replication, Ch. 9 linearizability). Single global leader for write order; followers in every region for read-side fanout. RPO 0 for purges by paying ~80 ms write latency. This is acceptable because we acked async to the customer.

8.3 POP relay

Subscribes to its regional follower, batches purges (10 ms windows), and pushes a compact bitmap to all in-POP cache boxes via UDP multicast (best-effort) plus a gRPC reconciliation channel (eventually consistent). Boxes that fall behind catch up by replaying from the log offset they last acked.

8.4 Tag-purge fan-out at scale

A tag like "homepage" may map to 100 M URLs. We never enumerate keys on the wire. We send the tag + a new generation; each box bumps its in-memory generation map (O(1) write). Cost is shifted to read time: a read does a generation lookup per tag (still O(k) for k tags on the entry, k≤8 typical).

9. Scaling & bottlenecks

  • Hot zone purge storms: per-zone rate-limit + coalescing window (combine 50 purges in 100 ms).
  • Log leader bandwidth: shard the log by zone-id hash; cross-shard purges are rare.
  • Box catching up after partition: cache treats unknown generations as MISS — fail safe, but origin load spikes; mitigate with shield POP.
  • Clock skew: never use wall clocks for ordering — use the log's monotonic offset (Ch. 8).

12. Talking points for the interviewer

  • "Purge is a write-amplification problem disguised as a delete problem. Generation counters trade read-time cost for unbounded write-time cost — the right trade for a CDN."
  • "I deliberately keep the customer ack on the regional coordinator, not on global propagation, so that one bad POP doesn't increase your purge p99 to 30 seconds."
  • "The hairy edge case is a partitioned POP returning stale content for an hour. My answer: the cache fails closed on unknown generations, accepting an origin spike over correctness violations."
