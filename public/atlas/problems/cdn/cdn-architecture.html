Design CDN architecture (Cloudflare/Akamai-style, hierarchical caching)

Hard cdn storage-cache

1. Problem statement

Design a global Content Delivery Network in the spirit of Cloudflare or Akamai. The system terminates TLS at the edge, caches static and semi-dynamic assets close to the user, and shields origin servers from request floods. The hard parts are global request routing, multi-tier cache hierarchy, cache fill efficiency at petabyte scale, and surviving regional failures without thundering-herd traffic on the origin.

2. Functional requirements

  • Serve HTTP/1.1, HTTP/2, HTTP/3 (QUIC) from any of N edge POPs.
  • Honor Cache-Control, ETag, Vary, range requests, and stale-while-revalidate.
  • Tiered cache: edge (POP) → regional shield → origin.
  • Per-customer config: cache keys, TTL overrides, custom headers, WAF rules.
  • TLS termination, SNI, automatic certificate provisioning.

3. Non-functional requirements

  • Latency: p50 TTFB < 30 ms, p99 < 150 ms within region.
  • Scale: 50 M HTTP req/sec at the edge (Cloudflare-class), 300+ POPs, 100 Tbps egress.
  • Availability: 99.99% per POP, 99.999% global (anycast removes individual POP failures).
  • Cache hit ratio: > 95% bytes-hit on tier-1 customers.

4. Capacity estimation

  • QPS: 50 M req/sec → ~4.3 trillion req/day. With 95% hit ratio, only 2.5 M req/sec touch shield/origin.
  • Storage per POP: 200 servers × 30 TB NVMe = 6 PB hot cache; global working set ~50 PB.
  • Bandwidth: peak 100 Tbps egress; per-POP 200–800 Gbps.
  • Object size mix: 70% < 100 KB (HTML/JS/CSS), 25% 100 KB–10 MB (images), 5% > 10 MB (video chunks).

5. API design

GET   /any/path                  Host: foo.com   → cached or origin-fetched response
PURGE /api/v4/zones/{z}/purge_cache  {urls|tags} → 202 Accepted
PUT   /api/v4/zones/{z}/cache/rules  {pattern, ttl, key} → 200 OK
GET   /api/v4/zones/{z}/analytics   ?since=...   → req, bytes, hit_ratio

6. High-level architecture

[Client] │ (anycast IP, BGP picks nearest POP) ▼ [Edge POP] ── L4 LB (Maglev/Katran) ── L7 proxy (nginx/envoy/pingora) │ │ │ hit │ miss ▼ ▼ [Local NVMe cache] ◄── consistent hash ──► [Regional Shield POP] │ miss ▼ [Customer Origin]

A request hits the closest POP via BGP anycast. Inside the POP, an L4 load balancer (Maglev-style consistent hashing) pins the 5-tuple to one of ~200 L7 proxies. The proxy looks up the cache key in local NVMe; on miss it asks the regional shield POP, which acts as a single fan-in to origin. This collapses N edge misses into 1 origin fetch.

7. Data model & storage

7.1 Cache key & index

cache_key = sha256(scheme | host | path | normalized_query | vary_headers)
index entry: {key, size, ttl_expiry, etag, last_modified, sfid, offset, hit_count}

Index lives in RAM (~50 GB per box for 500 M objects); object bodies live on NVMe in slab files (SFIDs) to avoid filesystem inode pressure. Eviction is segmented LRU with admission filtering (TinyLFU) to keep one-hit wonders out of hot tier.

7.2 Why tiered cache

If 300 edge POPs each miss independently, origin sees 300× amplification. A regional shield (one per continent, ~10 globally) deduplicates fills. Coalescing in-flight misses on the same key (single-flight) further reduces origin load by 100–1000× during a flash crowd.

8. Detailed component design

8.1 L7 proxy

Built on Pingora/Envoy. Each box runs ~50k concurrent connections, terminates TLS via session-resumption tickets, and serves from NVMe at 40+ Gbps. Hot objects (< 64 KB, > 100 hits) are promoted to RAM tier (kB-class objects). Range requests are served by sparse file indexing.

8.2 Cache fill / shield

Shield POP keeps a larger working set (~30 PB regional). Origin connections are pooled and HTTP/2-multiplexed. stale-while-revalidate returns the stale copy immediately and refreshes asynchronously, so origin RTT never appears on the hot path.

8.3 Control plane

A globally-replicated KV (Cloudflare Quicksilver, ~<1 s convergence to 300 POPs) pushes config changes. Per DDIA Ch. 5, this is a leader-based replication system with read replicas at every edge — strong consistency at the leader, eventual at the edges, and bounded staleness.

9. Scaling & bottlenecks

  • Hot key (e.g. live event): add request-coalescing + shield → 1 origin fetch regardless of edge QPS.
  • NVMe write amplification: append-only slab files, compaction during off-peak.
  • TLS handshake cost: session tickets + 0-RTT (QUIC) cuts CPU 60%.
  • POP failure: BGP withdraws the prefix; anycast routes to next-nearest POP within seconds.
  • Cold-cache thundering herd after POP drain: pre-warm via shield + jittered TTL.

12. Talking points for the interviewer

  • "Origin offload is the product. Everything else — anycast, NVMe, shield — exists to push hit ratio from 90% to 99%."
  • "I'd treat the control plane as a separate, leader-based system. Conflating it with the data plane is how you cause global outages."
  • "The hardest failure mode is a poisoned cache entry replicated globally; my mitigation is signed origin responses + fast tag-purge."
