Design Multi-region Failover (active-active vs active-passive)

Hard cdn database-ddia

1. Problem statement

Design the regional-failover layer that sits behind the CDN edge: when an entire AWS region or origin DC goes dark, traffic should drain to a healthy region with bounded data loss and bounded user-visible disruption. We need to choose, per workload, between active-active (all regions serve writes) and active-passive (one writer, others stand by), and prove the trade-offs with concrete RPO/RTO numbers.

2. Functional requirements

  • Detect region failure in < 60 s (false-positive rate < 0.01%/yr).
  • Drain user traffic from a failed region within < 2 min via DNS + anycast withdraw.
  • Replicate state across regions: stateless cache (best-effort), session store (≤ 1 s lag), source-of-truth DB (configurable RPO).
  • Run game-day drills: planned failover with zero data loss.

3. Non-functional requirements

  • RTO: ≤ 2 min user-visible unavailability for stateless tier; ≤ 5 min for stateful tier.
  • RPO: 0 (active-active payments), 1 s (active-passive sessions), 10 s (analytics).
  • Scale: 5 regions × 10 M req/s; 200 TB hot dataset replicated across regions.
  • Availability target: 99.99% per region → 99.9999% multi-region (independent failures).

4. Capacity estimation

  • Cross-region replication bandwidth: 200 TB dataset × 1% daily change × 5 regions = 10 TB/day per peer = ~1 Gbps sustained, peaks to 20 Gbps.
  • Health-check QPS: 5 regions × 1k probes/s × 100 services = 500 k/s.
  • Failover state machine actions: ~100 actions per drill, each must be idempotent.
  • Cost: active-active = 5× compute + cross-region egress. Active-passive ~1.2× (passive runs warm).

6. High-level architecture

┌───────── Global Health Plane ──────────┐ │ (Raft-replicated control state) │ └───────┬─────────────────────┬──────────┘ ▼ ▼ [GeoDNS / Anycast] ─── steering weights ─── per-region │ ▼ ┌──────────────────┐ async replication ┌──────────────────┐ │ Region A │ ◄───────────────────► │ Region B │ │ Edge → Origin │ │ Edge → Origin │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ │ Postgres │ ─► │ ──── physical WAL ──► │ ►│ Postgres │ │ │ │ (leader) │ │ │ │(replica) │ │ │ └──────────┘ │ ◄── Spanner/CRDT ─► │ └──────────┘ │ └──────────────────┘ └──────────────────┘

The control plane (a small Raft cluster, e.g. etcd across 5 regions) holds region health and steering weights. The data plane uses two replication models: physical-log (Postgres streaming) for active-passive workloads, and Spanner/CockroachDB-style multi-leader Paxos or CRDTs for active-active.

7. Data model & storage

7.1 Active-passive (most workloads)

region_state:
  region_id PK, role enum(leader|replica|drained),
  wal_lsn bigint, lag_ms int, healthy bool,
  last_heartbeat timestamptz

Per DDIA Ch. 5: single-leader async replication. RPO is bounded by replication lag at failover (typical 100–500 ms; spike to 5 s under load). After failover, we promote the lowest-lag replica using a fencing token (Ch. 8) so the old leader can never accept writes again.

7.2 Active-active (payments, inventory)

Use Spanner/CockroachDB or a CRDT-backed store. Each row is replicated by Paxos across 3–5 regions; writes commit at quorum (DDIA Ch. 9 linearizability). RPO is 0 because no committed write can be lost. Write latency > 80 ms cross-region — acceptable only if business permits.

8. Detailed component design

8.1 Health detection

Each region runs N independent prober instances in other regions (cross-region probing avoids the "sick region certifies itself healthy" anti-pattern). A region is declared down when ≥ 3 of 5 external probers agree for 30 s. The Raft control plane records the verdict.

8.2 Traffic drain

Steering weights flow to GeoDNS + the L4 LB control plane. New connections are weighted to 0 first; existing connections drain over 60 s. BGP withdraw is the last resort because it takes longer to reconverge.

8.3 DB failover with fencing

Promote new leader → bump epoch number in Raft → all clients refuse writes signed with the old epoch (DDIA Ch. 8 fencing tokens). Without fencing, a network-partitioned old leader can keep accepting writes ("split brain"), corrupting data.

8.4 Active-active conflict resolution

For inventory: serializable transactions via Spanner (TrueTime). For shopping carts: CRDTs (OR-Set) merging concurrent adds. For counters: PN-Counter CRDT (DDIA Ch. 5 multi-leader replication, Ch. 9 linearizability vs causal).

9. Scaling & bottlenecks

  • Replication lag spikes during traffic peaks → RPO breached. Mitigation: dedicate replication NICs, alert at lag > 500 ms.
  • Failover storm: when a region fails, 1.25× capacity hits each survivor. Pre-provision 25% headroom or auto-scale aggressively.
  • Cross-region write latency (active-active): 60–150 ms per commit. Use regional partitioning (a row's "home region" handles writes; reads are local).
  • Flapping region: Hysteresis — require N consecutive healthy probes before re-admitting traffic.

12. Talking points for the interviewer

  • "Choose the replication model per workload, not per company. Payments demand RPO 0; product images don't."
  • "The most dangerous failure isn't a dead region — it's a half-dead region. External cross-region probing + fencing tokens is non-negotiable."
  • "I default to active-passive with rigorous game-days. Active-active is a strong technical answer but a costly product answer; only adopt where RPO 0 is a real requirement."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
etcd / Raft control plane (5 regions)Region health and steering weights must be linearizable — two regions cannot both believe they are leader. 5-region Raft tolerates 2 simultaneous failures while keeping a quorum, hitting our 99.9999% multi-region availability target. Per-region etcd would risk split-brain.ZooKeeper (operationally heavier); Consul (gossip-based, weaker consistency); single-region DB (single point of failure).
Fencing tokens on DB promotion (DDIA Ch. 8)The dangerous failure is a network-partitioned old leader still accepting writes. Bumping an epoch number in Raft means every storage operation carries a token; the old leader's tokens are rejected, eliminating the split-brain corruption that killed many real-world failovers.STONITH only (slow, manual); no fencing (data corruption inevitable); leases (clock-dependent).
Postgres streaming replication (active-passive)Most workloads tolerate 100–500 ms RPO. Postgres physical-WAL streaming gives this with minimal CPU overhead — far cheaper than Spanner-class infrastructure for the 80% of tables where RPO 0 isn't required.Logical replication (per-row overhead); MySQL (weaker tooling); rsync (no LSN tracking).
Spanner / CockroachDB (active-active payments)RPO 0 with multi-region Paxos commits is non-negotiable for payments. Spanner's TrueTime gives external consistency at quorum write cost (60–150 ms cross-region), acceptable when business permits and required by financial compliance.Multi-leader async (lost writes); single-leader cross-region (write latency cliff); two-phase commit (coordinator failure).
CRDTs (OR-Set, PN-Counter) for shopping carts & countersActive-active cart adds and view counters don't need linearizability — they need "no lost adds." OR-Set merges concurrent additions deterministically; PN-Counter handles concurrent increments. No coordination on the 10 M req/s hot path.Last-write-wins (drops cart items); single-leader (cross-region latency); locking (10× write cost).
Cross-region external probers (3-of-5 consensus)Avoids the "sick region certifies itself healthy" anti-pattern. Probers in 5 other regions agree on a 30 s window before declaring failure — keeps the false-positive rate <0.01%/yr while detecting in <60 s.Self-probing (sick-self-certifies); single external prober (single point of failure); customer-only beacons (slow).

This hits the <2 min RTO without dropping in-flight work — critical for long-running uploads or video sessions.Hard cutover (drops sessions); weight-only (slow drain on long-lived).
GeoDNS + anycast withdraw (dual-layer steering)DNS is for policy (e.g., GDPR EU-only); anycast is for speed of failover. Stacking them gives sub-second for stateless reqs and minutes for full regional drain — covering both ends of the RTO/RPO matrix.DNS-only (TTL stickiness, 5 min worst case); anycast-only (no per-tenant policy).

TrueTime's bounded clock uncertainty (~7 ms) lets Spanner provide external consistency with ~80 ms p99 commits — the only viable path for active-active inventory at 200 TB.Optimistic concurrency (high abort rate); single-leader (cross-region latency).
Dedicated replication NICsReplication lag spikes during traffic peaks because user traffic and WAL ship compete for NIC bandwidth. Dedicated 25/100 Gbps NICs for replication isolate the 1 Gbps sustained / 20 Gbps peak cross-region replication from user load, holding RPO <500 ms.Shared NIC (lag spikes under load); WAN-shared (peer congestion).
Hysteresis (N consecutive healthy probes) for re-admissionFlapping regions cause routing storms. Requiring N consecutive healthy probes before re-admitting traffic dampens oscillation — same logic as BGP route dampening but applied at the steering layer.Single healthy probe (flap chatter); manual re-admit (operator burden).
25% capacity headroom per region5-region failure means 1.25× load on each survivor. Pre-provisioning 25% headroom absorbs the failover storm without auto-scaling latency. Cheaper than over-provisioning to 50% (covers 2-region failure with auto-scaling).No headroom (cascading overload); 100% headroom (2× cost).
Idempotent failover state-machine actions (~100 per drill)Game-day drills replay the same sequence multiple times; without idempotency a partial failure leaves the system in an unknown state. Each action carries an action-id and is safe to retry, enabling automated drills with confidence.Stateful actions (manual recovery); fire-and-forget (silent partial failure).
Regional row-pinning ("home region" reads/writes local)Active-active cross-region writes cost 60–150 ms p99. Pinning each row's writes to its home region keeps the bulk of writes local; cross-region traffic only on home-region failover. Hits the cost target without giving up RPO 0.Global writes (latency cliff); per-customer single region (loses A-A benefits).
