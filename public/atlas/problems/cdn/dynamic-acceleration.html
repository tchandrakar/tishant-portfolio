Design Dynamic Content Acceleration (DSA, edge compute, route optimization)

Hard cdn

1. Problem statement

Static content caches well; APIs and personalized HTML do not. Akamai's DSA (Dynamic Site Accelerator), Cloudflare Argo, AWS Global Accelerator all attack the same problem: how do you speed up uncacheable traffic between user and origin? The levers are TCP/TLS termination at the edge, persistent backbone connections, route optimization on a private overlay, and edge compute (Workers / Lambda@Edge).

2. Functional requirements

  • Terminate TLS at the edge for every API call, even uncacheable ones.
  • Route the second-leg (edge → origin) over an optimized overlay (private backbone or measured public path).
  • Run JS / WASM at the edge to assemble personalized HTML, do A/B tests, rewrite responses.
  • Allow micro-caching of API responses (1–10 s TTL) where safe.
  • Honor request affinity for stateful APIs (sticky to a region).

3. Non-functional requirements

  • Latency improvement: 25–50% TTFB reduction on uncacheable paths vs raw internet.
  • Scale: 5 M dynamic req/s globally; 1 M edge-compute invocations/s.
  • Cold start: < 5 ms for V8 isolates (Cloudflare Workers-style).
  • Reliability: 99.99%; overlay outage falls back to public internet automatically.

4. Capacity estimation

  • 5 M dynamic req/s × ~3 KB request + 8 KB response ≈ 55 GB/s = 440 Gbps overlay backbone.
  • Edge compute: 1 M invocations/s × 5 ms CPU = 5,000 cores busy → ~150 boxes (32 vCPU each).
  • Persistent connection pools: 300 POPs × 50 origins × 50 conns = 750 k long-lived TCP/H2 streams.
  • Private backbone: ~50 backbone POPs across continents, 100 Gbps inter-POP wavelengths, < 80 ms transcontinental.

6. High-level architecture

[Client] │ TLS (sub-10 ms RTT to nearest edge) ▼ [Edge POP] ── V8 isolate (Worker) ── micro-cache (1–10 s) │ │ (uncacheable / cache miss) ▼ [Edge → Backbone Ingress] │ ▼ ⟵ optimized path picker (per-destination, per-time-of-day) [Private Backbone POP-1] → [Backbone POP-2] → ... → [Egress POP near origin] │ ▼ [Customer Origin]

The win comes from three places: (1) TLS terminates near the user, eliminating 1–2 RTTs of public-internet handshake; (2) the edge holds a warm, multiplexed HTTP/2 connection to a backbone POP, so there is no TCP slow-start on the long leg; (3) the backbone leg uses an SDN-style optimized path that avoids congested transit.

  micro_cache_ttl_s int,
  health_path text
);

Path-metrics live in a fast distributed KV (Redis Enterprise / FoundationDB) with eventual consistency across POPs (DDIA Ch. 5 multi-leader async, bounded staleness ~30 s — fine since paths change slowly).

8. Detailed component design

8.1 Edge compute (V8 isolates)

Each POP runs a single V8 process hosting thousands of customer scripts as isolates. Cold-start is ~5 ms (vs 100–500 ms for containers/Lambda) because we skip OS process start. Scripts are immutable, signed, and globally distributed via the same control-plane KV used for cache config.

8.2 Optimized routing

Active probing runs continuously between every backbone POP pair. A path picker (similar to Argo) chooses the lowest-latency multi-hop overlay for each (ingress, egress) pair every 30 s. Paths are encoded as a SR-style label stack carried inside an MPLS or QUIC tunnel header.

8.3 Connection reuse

Edge → origin connections are pinned, HTTP/2-multiplexed (or HTTP/3), and kept warm. A single TCP slow-start cost is amortized across millions of user requests. If the origin supports HTTP/2 + TLS 1.3, we save 2–3 RTTs per request — at 80 ms per RTT, that's 160–240 ms per call.

8.4 Micro-caching

For "uncacheable" GET endpoints that are actually identical for thousands of users in a 5-second window (e.g. trending feed), micro-caching at TTL 5 s slashes origin QPS by 100–1000× during a viral moment, with bounded staleness the customer accepts.

8.5 Edge worker as response composer

A typical pattern: worker fans out to 5 origin services in parallel, assembles HTML, returns in 60 ms instead of 300 ms (each origin fetch from edge is 1 RTT vs from client is 5 RTTs).

9. Scaling & bottlenecks

  • Backbone capacity: a viral event saturates a POP-pair; mitigations are weighted ECMP across multiple wavelengths and graceful fallback to public internet when overlay loss > 1%.
  • Worker noisy-neighbor: single V8 process means a hot script can stall others; enforce CPU time limits (50 ms hard, 10 ms typical) and memory caps (128 MB).
  • Stateful API affinity: if a session must hit one specific region, sticky-key hashing + consistent-hash overlay routing.
  • Path-flap during active probing: hysteresis (require sustained 20% improvement before switching).

12. Talking points for the interviewer

  • "Dynamic acceleration is unsexy compared to caching, but it's where the modern CDN business is. The cache hit ratio on APIs is zero; the levers are TCP, TLS, and routing."
  • "V8 isolates are the right unit of edge compute. Containers are too heavy; functions on cold infrastructure can't hit single-digit-ms cold-starts."
  • "The most underrated trick is micro-caching: a 5-second TTL on a 'dynamic' endpoint can absorb a viral spike that would otherwise page the origin team."
