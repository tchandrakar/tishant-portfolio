Design Memcached (LRU, consistent-hash, multi-tenant)

Hard storage-cache

1. Problem statement

Design a multi-tenant, in-memory look-aside cache fronting OLTP databases for a service with thousands of microservices. Targets: 50M ops/sec aggregate, p99 GET under 500 us, no replication (cache-only), elastic add/remove of nodes without invalidating >1/N of keys, and noisy-neighbor isolation across tenants.

2. Functional requirements

  • Commands: get, gets, set, add, cas, incr/decr, delete, flush_all, touch.
  • Per-tenant namespaces with quota in MB and ops/sec.
  • Consistent-hashing client with virtual nodes; warm rings on deploy.
  • TTL with lazy expiration + background reaper; LRU eviction when slab full.
  • Binary protocol over TCP and UDP for SET-and-forget low-latency reads.

3. Non-functional requirements

  • Latency: p50 GET 100 us, p99 500 us same-AZ.
  • Throughput: 200k req/s/node on 16-core; cluster of 250 nodes = 50M req/s.
  • Memory: 128 GB/node, 32 TB cluster, hit rate > 95%.
  • Availability: Cache is best-effort; node loss degrades but never errors out.

4. Capacity estimation

  • 1B daily active users * 50 reads/user = 50B reads/day = 580k reads/sec average, 5x peak = 2.9M/s — easily handled by <20 nodes; budget 250 for tenant isolation.
  • Object mix: 1 KB avg, 30B keys cached → 30 TB working set.
  • Network: 1 KB * 200k ops = 200 MB/s/node = 1.6 Gbps NIC.
  • Connections: 50k clients * 2 conns/node = mux through proxy or use UDP for reads.

5. API design

get   <key>               → VALUE <key> <flags> <bytes>\r\n<data>\r\nEND
set   <key> <flags> <ttl> <bytes> [noreply]\r\n<data>\r\n  → STORED
cas   <key> <flags> <ttl> <bytes> <cas-token> → STORED|EXISTS
delete <key>              → DELETED|NOT_FOUND
incr  <key> <delta>       → <new-value>

6. High-level architecture

[App tier]──┐ [App tier]──┼─►[mcrouter / smart client]──hash ring──┬─►[memcached node 1] [App tier]──┘ ▲ ├─►[memcached node 2] │ ├─►[memcached node 3] [config registry] └─►[memcached node N] │ slab allocator + LRU

Apps talk to mcrouter (Facebook-style proxy) which owns the consistent-hash ring. mcrouter does request coalescing, lease tokens for thundering-herd protection, and pool failover. Each memcached node is shared-nothing.

7. Data model & storage

7.1 Slab allocator

Memory is split into 1 MB pages, each assigned to a slab class with fixed chunk size (96 B, 120 B, ... 1 MB; growth factor 1.25). Eliminates external fragmentation but causes slab calcification — workload shift starves new sizes; mitigation: slab rebalancer redistributes pages periodically.

7.2 LRU

Per-slab segmented LRU: HOT (20%), WARM (40%), COLD (40%). New items enter HOT; on miss it ages to COLD. This avoids scan-pollution (DDIA Ch.3 — cache replacement policies). Each item has a 4 B last-access ts updated lazily to avoid contention.

8. Detailed component design

8.1 Consistent hashing & vnodes

Ring of 2^32 with ~200 vnodes per physical node. Add a node → only 1/N keys move (DDIA Ch.6 "Consistent Hashing"). Use Ketama hashing for client-library compat. Warm-up: stagger adds 5 min apart; pre-fetch hot keys via gutter pool to avoid stampede.

8.2 Multi-tenancy

Two strategies: (a) per-tenant pools (full isolation, low utilization) or (b) shared cluster + key prefix + slab quotas via cgroup-like accounting. We pick (b) with admission control: token bucket per tenant in mcrouter. Fairness via weighted DRF.

8.3 Stampede protection

Lease tokens: on miss, mcrouter returns a 64-bit lease and stalls subsequent identical misses for 100 ms; only the lease holder fills cache. Reduces DB read amplification by 100-1000x at peak.

9. Scaling & bottlenecks

  • Hot keys: celebrity profile served by one node; replicate hot keys to a "gutter" pool of ~5% capacity; mcrouter shadow-reads.
  • UDP loss: reads only; SET still goes TCP. Allows 2x throughput per NIC at the cost of ~0.01% miss rate.
  • Connection storm: use mcrouter to mux 10k clients → 50 backend conns/node.
  • Slab calcification: rebalance daily; alert on slab imbalance > 30%.

12. Talking points for the interviewer

  • "I'd choose memcached over Redis when the workload is pure look-aside and we want predictable per-thread scaling — memcached is multi-threaded, Redis is not."
  • "Cache-aside has a known race: writer deletes key, reader fills with stale value before DB commit. Mitigation is delayed double-delete or DB CDC into the cache."
  • "The biggest production lesson: invest in mcrouter early. Without it, hot keys and stampedes will be the postmortem you write at 3 AM."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Memcached server Multi-threaded with per-thread LRU sharding scales linearly to 200k req/s on 16 cores — exactly what 50M aggregate ops/sec across 250 nodes requires. No replication code path means simpler binary and lower memory overhead per cached object. Redis (single-threaded, 100k/node ceiling); custom in-process cache (no shared invalidation).
mcrouter proxy Owns the consistent-hash ring so all 50k client conns become 50 backend conns/node — solves the C10k connection storm. Centralizes lease tokens, request coalescing, and pool failover; without it hot keys and stampedes dominate postmortems. Smart client per language (drift across SDKs); twemproxy (no leases, less feature-rich).
Ketama consistent hashing 2^32 ring with ~200 vnodes/physical node ensures adding a node moves only ~1/N keys (DDIA Ch.6). Ketama is the de-facto compat hash across PHP, Java, Go, Python clients — critical for a multi-tenant cache touched by thousands of microservices. Rendezvous (HRW) hashing (slightly better balance, less library support); modulo (every add reshuffles all keys).
Slab allocator (1 MB pages, 1.25 growth) O(1) malloc/free with zero external fragmentation, which matters when running for months with 30 TB working set. Power-of-1.25 chunk classes keep internal waste under ~12% on average — better than malloc's 30% RSS bloat at this scale. jemalloc (fragmentation drift over weeks); tcmalloc (better for short-lived workloads).
Slab rebalancer (background thread) Workload mix shifts (avatars 5 KB → tokens 100 B) cause slab calcification — pages stuck in stale classes while new size starves. Daily rebalance keeps imbalance under 30%, our alert threshold; without it hit rate degrades 5-15% over weeks. Static slab assignment (calcification); restart cycling (cold cache shock).
Segmented LRU (HOT 20% / WARM 40% / COLD 40%) Plain LRU is poisoned by full-table scans (DDIA Ch.3). Segmented LRU keeps recent items in HOT until verified by a second access, protecting the 95% hit rate from one-shot batch jobs that touch billions of keys. Plain LRU (scan pollution); LFU (better hit rate but counter overhead per access).
Binary protocol over TCP Compact framing (24 B header) saves ~40% bandwidth vs ASCII at 1 KB avg values, and the fixed header avoids parser branches that hurt the p99 500us tail. CAS tokens fit naturally in the binary opaque field. ASCII text protocol (legacy, easier debug); gRPC (overkill, +200us serialization).
UDP for reads No conn state means each NIC handles 2x more pps; for 200 MB/s/node read traffic this avoids needing 25 GbE everywhere. Loss is treated as a cache miss (expected ~0.01% — well below the 5% miss budget on a 95% hit-rate cache). TCP-only (conn state explosion); QUIC (good but no mainstream memcached client).
CAS (compare-and-swap) tokens Look-aside cache has a writer/reader race (writer deletes, reader refills with stale DB read). 64-bit CAS token is the optimistic-concurrency analog (DDIA Ch.7) — refill only succeeds if the version matches, eliminating the race without locks. Distributed lock (Redlock complexity, latency); double-delete (still a window).
Lease tokens (mcrouter) On miss, mcrouter issues one lease and stalls duplicate misses for 100 ms — the 100x-1000x DB read amplification reduction is the difference between surviving a hot-key event and DB meltdown. Cheaper than per-key Redlock. Coalescing in app code (per-language drift); request hedging (amplifies, not reduces).
Gutter pool (5% capacity) Hot keys pin to a single node by hash; replicating just hot keys to a small "shadow" pool with mcrouter shadow-reads sidesteps the single-node bottleneck without paying full-cluster replication cost (which would halve effective capacity). Full N-way replication (50% capacity loss); client-side L1 only (per-pod cache pollution).
Token-bucket admission (mcrouter) Multi-tenancy in a shared 32 TB pool needs per-tenant ops/sec quotas to prevent noisy neighbor. Token bucket per (tenant, route) gives bounded burst + sustained rate; weighted DRF guarantees fairness when capacity is contended. Per-tenant pools (low utilization, ~30% wasted RAM); no admission (one tenant kills cache).
Lazy expiration + reaper thread TTL check on GET is O(1) and avoids a global timer wheel. Background reaper sweeps cold expired entries to free slabs, but at low priority (CPU yield) so it never steals from request threads — preserving the p99 500us latency target. Eager TTL timer per key (heap insert overhead); never expire (memory bloat).
Linux: SO_REUSEPORT + RPS, NUMA pinning SO_REUSEPORT spreads accept() across worker threads avoiding the herd. RPS pins NIC queues to cores; NUMA-aware allocation cuts cross-socket memory access (which adds ~100 ns per hop) — measurable on the 100us p50 GET budget. Default networking (single-accept bottleneck); userspace NIC (DPDK is overkill here).
