Design Distributed Rate Limiter (token bucket / leaky bucket, multi-region)

Hard storage-cache

1. Problem statement

Build a globally consistent rate limiter for an API gateway with 100k+ tenants, enforcing per-key, per-route, per-region quotas. Limits range from 5 req/min (auth code) to 1M req/sec (bulk ingest). Decisions must add < 1 ms p99 to request latency, work across regions, and degrade gracefully if the central store is unreachable.

2. Functional requirements

  • Configurable algorithms: token bucket, leaky bucket, fixed window, sliding window log, sliding window counter.
  • Composite keys: tenantId + apiKey + endpoint + ip.
  • Tiered limits with burst allowance (e.g., 100 rps + burst 500).
  • Soft (warn) + hard (429) limits; expose Retry-After.
  • Per-route override; canary tenants get higher limits.

3. Non-functional requirements

  • Latency overhead: p50 100 us, p99 1 ms.
  • Availability: 99.99% — limiter outage must not block traffic; fail-open or fail-closed configurable.
  • Accuracy: ±1% on global limits, exact within a region.
  • Scale: 1M decisions/sec/gateway pod; 1000 pods cluster-wide = 1B/s.

4. Capacity estimation

  • 1B decisions/sec * 50 B counter state = 50 GB cluster state at peak.
  • Per-key footprint: 32 B key + 8 B count + 8 B reset_ts = 48 B; 100M keys = 4.8 GB in Redis.
  • Sliding window log worst-case: 1000-rps key * 60s window = 60k entries * 16 B = 1 MB/key — too costly; use sliding window counter instead.
  • Cross-region replication: 10k limit-state changes/s * 100 B = 1 MB/s, fits async link.

5. API design

POST /v1/check  {keys:[...], cost:1}        → {allowed:true, remaining, resetMs}
GET  /v1/quota?key=...                       → {limit, remaining, refillRate}
PUT  /v1/policy  {tenant, route, algo, limit, burst}
                                              → 200 (admin only)

6. High-level architecture

┌─────────────────────────────┐ │ API Gateway pod │ │ ├─ in-proc bucket (L1) │ ← refilled from L2 every 100 ms │ └─ admission decision │ └────────────┬────────────────┘ │ async incr ▼ ┌──────────────────────┐ Lua script: atomic CAS+INCR+TTL │ Redis cell (per-AZ) │ ◄── per-shard, 16 shards/region └──────────┬───────────┘ │ change-log ▼ ┌──────────────────────┐ multi-master CRDT counters │ Cross-region sync │ (1-2s lag) └──────────────────────┘

Two-tier design: each gateway pod owns a local "lease" of N tokens fetched from Redis. Decisions are made locally; a low-frequency reconcile call to Redis renews/returns leftover tokens. Cross-region truth converges via CRDT-style PN counters.

7. Data model & storage

7.1 Redis state

HSET rl:{key} tokens 95 lastRefill 1714900000 limit 100 refill 100/s
PEXPIRE rl:{key} 60000
// Sliding window counter
ZADD swc:{key} <ts> <eventId>     # capped per window
ZREMRANGEBYSCORE swc:{key} 0 (now-window)

7.2 Why Redis

Single-threaded Lua atomicity → race-free token bucket math. Sub-ms latency. Cluster mode with hash-slots (DDIA Ch.6). For multi-region we run one Redis cell per region and sync counters as CRDTs (DDIA Ch.5 — convergent replicated data types).

8. Detailed component design

8.1 Token bucket Lua script

-- KEYS[1]=rl key, ARGV={now, cost, refill, capacity}
local b = redis.call('HMGET', KEYS[1], 'tokens','last')
local tokens, last = tonumber(b[1]) or ARGV[4], tonumber(b[2]) or ARGV[1]
tokens = math.min(ARGV[4], tokens + (ARGV[1]-last)*ARGV[3])
if tokens >= ARGV[2] then
  tokens = tokens - ARGV[2]
  redis.call('HMSET', KEYS[1], 'tokens', tokens, 'last', ARGV[1])
  return 1
else return 0 end

8.2 Two-tier (lease) optimization

Pod requests a lease of L tokens (e.g., L=100) via single Redis call; serves L locally; renews when 20% remain. Reduces Redis QPS by Lx (50-100x). Trade-off: short bursts can over-deliver up to (L * #pods) tokens during partition.

8.3 Sliding window counter

Hybrid of fixed window + linear interpolation: count = current_window * elapsed_fraction + previous_window * (1-fraction). Constant memory (2 ints) and ±1% accuracy vs sliding-log.

8.4 Multi-region semantics

For "global 1000 rps" enforce per-region budget = 1000/N + 10% headroom; sync excess via async CRDT. Hard global limits use a single-leader Redis with cross-region reads (higher tail latency but exact). Strict global limits → DDIA Ch.9 linearizability cost: at least one cross-region RTT.

9. Scaling & bottlenecks

  • Redis hot key: a single tenant's bucket at 1M rps is a hot shard; mitigate via tier-1 leases or split key into rl:{tenant}:{N} with each shard owning capacity/N.
  • Clock skew across pods: use Redis-side now (TIME command) to compute refill, not pod clocks.
  • 429 storms: add jittered Retry-After (1-5s) and per-tenant circuit breaker.
  • Failure mode: Redis outage → fail open with local lease for 30s; alarm and decay capacity to 50% to prevent abuse.

  • Ch.7 Transactions — atomic CAS via single-threaded Redis Lua (analog of single-shard tx).

12. Talking points for the interviewer

  • "I default to token bucket + two-tier lease — it's the right balance of strictness and overhead at internet scale."
  • "Truly global limits cost a cross-region RTT; for SaaS billing limits I prefer per-region budgets reconciled hourly."
  • "The most critical decision is fail-mode: APIs serving payments fail closed; public read APIs fail open with degraded capacity."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis (cluster mode, 16 shards/region) Single-threaded Lua atomicity makes token-bucket math race-free in one round-trip — the only way to hit p99 1 ms at 1M decisions/sec without distributed locks. 48 B/key * 100M keys = 4.8 GB fits 1 shard, sharding is for QPS not memory (DDIA Ch.6). Cassandra counters (eventual, no atomic CAS); Postgres advisory locks (10x latency).
Lua scripting for token bucket HMGET + math + HMSET in one atomic script eliminates the get-modify-set race that classic CAS would loop on. Single round-trip preserves the 1 ms budget; Lua runs in the Redis event loop so no lock acquisition cost (DDIA Ch.7). WATCH/MULTI/EXEC (retry storms under contention); app-side mutex (cross-pod doesn't work).
Two-tier lease optimization Pod fetches L=100 tokens locally, makes 100 admission decisions before next Redis call → reduces Redis QPS by ~50-100x. Critical at 1B decisions/s cluster-wide; without it Redis hot shards hit ceiling. Trade: up to L*pods over-serve during partition. Per-decision Redis call (200x QPS pressure); fully local (no cross-pod budget enforcement).
Sliding window counter algorithm Hybrid of fixed window + linear interpolation needs only 2 ints per key (16 B) vs sliding-log's 60k entries * 16 B = 1 MB/key at 1000 rps. Hits ±1% accuracy SLA without exploding memory — the right point on the precision/cost curve. Sliding log (memory blows up at scale); fixed window (boundary burst 2x limit).
Token bucket as default algorithm Allows bursts up to capacity (good UX for retry-with-jitter clients) while enforcing sustained rate. Math is constant memory and constant time. Most APIs in production accept burstiness gracefully — leaky bucket is too strict for typical traffic. Leaky bucket (no burst tolerance); fixed window (boundary issues).
CRDT PN-counter for cross-region sync Independent regional Redis cells converge on global truth via PN-counter merge (DDIA Ch.5 CRDTs) without a leader hop. 10k changes/s * 100 B = 1 MB/s async replication is trivial. Sacrifices strict global limit for AP — acceptable for soft quotas. Single global leader (cross-region RTT every check); manual reconciliation (drift).
In-process L1 token cache (per gateway pod) Refilled from Redis every 100 ms; serves admission locally at p50 100us — Redis call alone would be 200-500us. The L1+L2 design is what makes the <1 ms p99 budget achievable at 1M decisions/s/pod. Pure remote (latency budget gone); pure local (no global enforcement).
Redis TIME command for refill clock Pod clocks drift up to 100 ms even with NTP, which would cause refill miscalculation (DDIA Ch.8 unreliable clocks). Using Redis-side TIME inside Lua makes refill math deterministic across all callers; the script reads server clock once. Pod local clock (skew gives wrong refill); chrony-strict NTP (still ms-level drift).
Per-cell sharding (16 shards/region) Hash-slot routing on Redis Cluster spreads 1B decisions/s across shards so no single Redis hits its 100k ops/s ceiling. Tenant key suffix rl:{tenant}:{N} further splits hot tenants — adaptive split when shard QPS >80%. Single Redis instance (CPU-bound); proxy-based routing (extra hop).
Configurable fail-open / fail-closed Payment APIs fail closed (refuse on Redis outage) — security critical; public read APIs fail open with 30s local lease + decay to 50% capacity to prevent abuse during outage. The single most important policy decision per route. Always fail open (abuse risk); always fail closed (false outages).
Jittered Retry-After (1-5s) Without jitter, 429 storms cause synchronized retry waves that re-trigger limit at the next second boundary — classic thundering herd. Random jitter spreads retries; per-tenant circuit breaker prevents repeat offenders from poisoning the limiter shard. Fixed Retry-After (synchronized retries); no Retry-After (clients don't back off).
Weighted DRF for tenant fairness When Redis shard is contended, dominant resource fairness across tenants prevents one noisy tenant from starving others — works correctly even with heterogeneous request costs (CPU vs memory). Token bucket alone allows greedy tenants to hog capacity. FIFO (head-of-line); strict priority queue (starvation risk).
Composite key (tenant + apiKey + endpoint + ip) Hierarchical limits at multiple granularities catch abuse missed by any single dimension (tenant-level limit ok, but a single IP DDoSing one endpoint). Composite hash is 32 B; cost is 4x key count which still fits in 4.8 GB. Single-dim key (gameable); per-attempt deep introspection (latency cost).
HSET + PEXPIRE for state TTL Hash structure groups tokens, lastRefill, limit, refill in one Redis op; PEXPIRE 60s auto-cleans dormant tenants without a sweep job. Memory footprint stays bounded as tenants churn — critical for the 100M-key quota at steady state. Separate keys per field (3x ops); no TTL (memory growth unbounded).
