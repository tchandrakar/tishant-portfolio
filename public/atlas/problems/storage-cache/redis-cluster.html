Design Redis Cluster (16384 hash slots, replication, failover)

Hard storage-cache database-ddia

1. Problem statement

Design a horizontally-sharded, in-memory key-value store compatible with Redis semantics. The cluster must distribute keys across N shards via 16384 hash slots, replicate each shard for HA, detect node failure via gossip, and promote a replica with sub-10s impact. The system targets 10M+ ops/sec aggregate with p99 GET latency under 1 ms inside the DC.

2. Functional requirements

  • Standard Redis commands: GET/SET/DEL/EXPIRE, INCR, LPUSH/RPOP, ZADD, HSET, SUBSCRIBE.
  • Multi-key ops within the same hash slot via hash-tags {user:42}:profile.
  • Online resharding: move slots between nodes without downtime.
  • Replica failover within 10s of master loss; CLUSTER MEET/FORGET for membership.
  • Pub/Sub broadcast across all nodes; client-side smart routing via MOVED/ASK.

3. Non-functional requirements

  • Latency: p50 GET under 200 us, p99 under 1 ms intra-AZ.
  • Throughput: 100k ops/sec/node; 100-node cluster ~ 10M ops/sec.
  • Availability: 99.99% with 1 master + 2 replicas per shard.
  • Durability: AOF every 1s = ~1s data-loss window on power-loss.
  • Memory: 64-256 GB per node, 6.4-25.6 TB cluster.

4. Capacity estimation

  • Active users: 200M, 5 cache reads/sec/user peak = 1B ops/sec aggregate; we shard to ~100 shards.
  • Avg value 512 B, working set 5B keys = 2.5 TB; with 3x replication factor = 7.5 TB RAM.
  • Cluster bus traffic: gossip 10 packets/sec/node, ~200 B each = 2 KB/s/node, negligible.
  • Replication stream: 50 MB/s/master at sustained writes, 100 nodes = 5 GB/s intra-AZ.
  • Failover budget: NODE_TIMEOUT 5s + election quorum 1s + slot reconfig 1s = ~7s downtime.

Client computes CRC16(key) mod 16384; on first MOVED response it caches the slot map. Masters gossip via the cluster bus on port 6379+10000. A replica wins election if it has the most-recent replication offset and a majority of masters vote.

7.2 Persistence

RDB snapshot every 5 min + AOF append. AOF rewrite is fork+COW, doubling RSS briefly — size nodes to 50% memory headroom. Disable swap; THP off.

8. Detailed component design

8.1 Slot routing & resharding

16384 slots is a sweet spot: bitmap fits in 2 KB per gossip packet (DDIA Ch.6 "Partitioning by Hash of Key"). To move slot 3999, source enters MIGRATING, target enters IMPORTING; client GETs return ASK redirect for keys not yet migrated. MIGRATE moves keys atomically; finally CLUSTER SETSLOT NODE updates ownership.

8.2 Replication & failover

Async leader-follower (DDIA Ch.5 "Single-Leader Replication"). PSYNC2 supports partial resync via repl-id + offset. On master loss, a replica with min(rank) and majority votes from other masters wins election. Split-brain risk: a stale master in a network partition keeps accepting writes until NODE_TIMEOUT, so set min-replicas-to-write 1 for stronger guarantees.

8.3 Hot key mitigation

Use hash tags so related keys colocate; for a single hot key (celebrity timeline), front with client-side L1 LRU + Redis pub/sub invalidation. Optionally enable Redis 6 client-side caching (RESP3 tracking).

9. Scaling & bottlenecks

  • Single-thread per shard: CPU-bound at ~100k ops/sec; scale by adding shards, not threads. Redis 6 I/O threads parse only.
  • Big keys: a 100 MB list freezes the event loop on DEL; use UNLINK and avoid >10k elements per key.
  • Cluster bus storm: 1000+ nodes generate O(N²) gossip; cap at ~500 masters.
  • Failover oscillation: tune cluster-node-timeout 5-15s; lower → faster failover but flaps; higher → longer outage.

12. Talking points for the interviewer

  • "I'd default to 16384 fixed slots over consistent hashing because the bitmap fits in gossip and lets clients cache routing in 2 KB."
  • "Failover is async + best-effort; for write-once-or-fail semantics I'd front Redis with a Raft-backed sequencer or use Redis-Raft."
  • "The hardest operational issue is big keys and forking AOF on 64 GB RSS — I size nodes ≤ 25 GB and shard aggressively."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis 7.x server Single-threaded event loop gives predictable sub-200us p50 GETs and lock-free atomic ops; in-memory hash index (DDIA Ch.3) at 100k ops/sec/node hits the 10M aggregate target without exotic hardware. Native cluster mode ships the 16384-slot routing we need. KeyDB (multi-thread, but fork costs worse on 64GB); Dragonfly (newer, less battle-tested at 100-node scale).
16384 hash slots (fixed) Slot bitmap fits in 2 KB, so each gossip packet carries full ownership info — keeps cluster bus under 2 KB/s/node at 100 nodes (DDIA Ch.6 partitioning by hash). CRC16(key) mod 16384 is deterministic and clients cache the map. Consistent hashing rings (gossip payload grows O(N) with vnodes); range partitioning (hot ranges, manual splits).
CRC16 hash function Cheap (~1 ns/key), uniform distribution across 16384 slots, fits in 14 bits. Hash-tags {user:42} let multi-key ops colocate without sacrificing distribution. Good enough since collisions just mean same-slot, not same-key. SHA-1 (overkill, 30x slower); xxhash (excellent but not in client compat matrix).
Cluster bus gossip (port +10000) O(N²) message complexity is acceptable up to ~500 masters at 10 packets/sec each (~200 B). Detects PFAIL/FAIL in <NODE_TIMEOUT (5-15s) without a central coordinator (DDIA Ch.8). PING/PONG carries slot bitmap piggyback. ZooKeeper/etcd-based membership (extra dependency, 2-3 hop coordination); SWIM (better scaling but custom).
PSYNC2 async replication 1 GB ring-buffer backlog allows partial resync after replica reconnects, avoiding full RDB transfer (which would saturate the 5 GB/s intra-AZ stream). Single-leader async (DDIA Ch.5) keeps write latency at sub-ms. Sync replication (10x write tail latency); chain replication (more complex failover).
AOF + RDB hybrid persistence AOF every 1s caps data loss to ~1s per the NFR; RDB snapshot every 5 min gives fast restart (load 64 GB in ~90s vs ~10 min for AOF replay). Fork+COW for rewrite needs the 50% RSS headroom we sized for. AOF-only (slow restart); RDB-only (5-min loss window violates NFR); no persistence (cold cache stampedes).
Replica-based failover (1 master + 2 replicas) Quorum vote among masters (DDIA Ch.5 leader election) completes in ~1s. Two replicas survive AZ loss + ongoing maintenance. Election uses replication offset rank — replica with most-recent data wins, minimizing rollback. Sentinel-only (no automated slot rebalance); Raft-per-shard like Redis-Raft (CP but 2-3x latency).
Smart client (slot map cache) One MOVED redirect bootstraps the entire slot→node map (2 KB), so steady-state requests are 1-hop. 100k ops/sec/node math assumes no proxy hop. ASK redirects handle in-flight migration without invalidating the cache. Twemproxy/envoy proxy (extra hop = +200us p50, 2x conn count); dumb client (every op risks MOVED).
UNLINK + lazy free Big keys (100 MB lists) would freeze the single-threaded event loop on DEL for >100 ms, blowing p99 budget. UNLINK offloads memory reclamation to a background thread, keeping the main loop <1 ms. Synchronous DEL (event-loop stalls); manual sharding of big keys (app burden).
Hash-tag colocation {user:42}:profile Forces related keys into the same slot so MULTI/EXEC, Lua, and pipelined ops work atomically. Without it, multi-key ops error with CROSSSLOT — the routing layer cannot coordinate across shards (no 2PC in Redis). Client-side fan-out + reconciliation (no atomicity); cross-slot transactions via Redis-Raft (heavy).
min-replicas-to-write 1 Refuses writes if no replica is connected, preventing the split-brain stale-master scenario during a NODE_TIMEOUT-sized partition (DDIA Ch.8). Trades a sliver of availability for bounded data loss on failover. Default (accept writes always — risk lost writes); WAIT command per write (5x latency).
RESP3 client-side caching Redis tracks which keys each client cached and pushes invalidations on change, eliminating round-trips for hot keys (celebrity timeline at 100k+ QPS) without app-level pub/sub plumbing. Cuts hot-key shard load 10-100x. App-level LRU + pub/sub invalidation (works but more code); read-replica scaling (replication lag risk).
Linux: THP off, swap disabled, vm.overcommit=1 Transparent huge pages cause >100 ms latency spikes during fork-on-AOF-rewrite. Swap on a 64 GB RSS = unbounded latency. overcommit=1 lets fork() succeed even when nominal memory looks short. These tunings preserve the p99 < 1 ms budget. Default kernel settings (THP causes pauses); cgroup memory limits (forks fail at worst time).
Prometheus + redis_exporter Tracks cluster bus messages, replication lag offset, slowlog, fork time, RSS — the four leading indicators for the failure modes listed in section 9. Pull-based scrape every 10s is cheap relative to 100k ops/sec. Redis INFO polling only (no time-series); Datadog (SaaS cost at 100 nodes).
