Design Distributed Session Store (sticky vs shared, eviction, TTL)

Hard storage-cache

1. Problem statement

Design a session store for a global web platform with 200M MAU and 20M concurrent sessions. The store must serve session reads in sub-ms, persist auth state across pod restarts, support sliding TTL, sign tokens to prevent forgery, and survive a single-AZ failure. Compare sticky-LB sessions, shared-cache sessions, and stateless JWTs and pick a hybrid.

2. Functional requirements

  • create(userId, deviceId, scopes) → sessionId, refreshToken.
  • get(sessionId) → {userId, exp, scopes, csrf}; sliding window TTL.
  • revoke(sessionId | userId | deviceId).
  • List active sessions per user; force-logout-all.
  • Multi-region read-anywhere with eventual consistency on revocation.

3. Non-functional requirements

  • Latency: p50 200 us, p99 1 ms read; create/refresh p99 5 ms.
  • Throughput: 500k reads/sec, 20k writes/sec (every page-view rolls TTL).
  • Availability: 99.99% — session store outage must not log everyone out.
  • Durability: revoke must propagate within 5s globally.
  • Memory: 20M sessions * 1 KB = 20 GB resident.

4. Capacity estimation

  • 500k QPS / 100k QPS-per-Redis-shard = 6 shards minimum; over-provision to 12 across 3 AZs.
  • Write amplification: every page-view extends TTL → 20k writes/s * 2 replicas = 40k phys writes/s.
  • Refresh-token DB: 20M * 200 B = 4 GB in MySQL/Postgres + replicas.
  • Cross-region replication bandwidth: 20k writes/s * 1 KB = 20 MB/s; trivial.
  • JWT alternative: 1 KB token sent every request * 50k req/s/region = 50 MB/s ingress overhead.

5. API design

POST /v1/sessions      {userId, deviceId, scopes}    → {sid, refresh, exp}
GET  /v1/sessions/{sid}                               → {userId, scopes, exp}
POST /v1/sessions/{sid}/refresh {refresh}             → {sid, refresh, exp}
DELETE /v1/sessions/{sid}                             → 204
DELETE /v1/users/{u}/sessions  ?reason=password_reset → 204
GET  /v1/users/{u}/sessions                            → [{sid, ua, geo, lastSeen}]

6. High-level architecture

[client] ── cookie sid ──►[L7 LB]──►[stateless app pods] │ ▼ ┌────────── Redis (per-region) ──────────┐ │ primary ──async repl──► replica(AZ2) │ │ cluster of 12 shards, 3 replicas/shard│ └─────────────┬──────────────────────────┘ │ change-stream ▼ [Kafka revocation topic]──►other regions │ ▼ [Refresh-token DB (Postgres) — durable record]

App pods are stateless. Session ID is an opaque random 128-bit token in an HttpOnly cookie. Read-path: cookie → Redis GET. Write-path (login/refresh): write to Redis + durable Postgres for refresh tokens. Revocations broadcast via Kafka so all regions invalidate within seconds.

7. Data model & storage

7.1 Redis session record

HSET sess:{sid} userId 42 scopes "read,write" csrf abc123 exp 1714903600
PEXPIRE sess:{sid} 1800000      # 30 min sliding
SADD user:42:sessions sid
ZADD revoked:bloom <ts> sid     # short-term denylist for rotated sessions

7.2 Postgres durable record

CREATE TABLE refresh_tokens (
  token_hash bytea PRIMARY KEY,
  user_id bigint, device_id text,
  created_at timestamptz, expires_at timestamptz,
  revoked_at timestamptz NULL,
  rotated_to bytea NULL  -- for token rotation chain
);

Refresh tokens are durable for audit + revocation. Hash with SHA-256; never store plaintext (DDIA Ch.12 — security principles).

8. Detailed component design

8.1 Sticky vs shared vs JWT

ModeProsCons
Sticky LBIn-process cache, zero hopsPod loss = logout; uneven load
Shared cache (Redis)Pods stateless, easy revokeCache outage = global logout
Stateless JWTNo store reads, scales freeRevoke is hard; JWT bloat in cookies

Hybrid: short-lived signed JWT (5 min) + opaque refresh token in Redis. Most reads are JWT signature checks (no I/O). Revoke uses denylist + short JWT TTL bound. This is the "best of both" pattern.

8.2 Sliding TTL

On every authenticated request, app calls EXPIRE sess:{sid} 1800 (Redis O(1)). To avoid 500k EXPIRE/s, batch: only refresh if remaining TTL < 50% (probabilistic). Cuts writes 5x with no UX impact.

8.3 Eviction policy

Redis maxmemory-policy volatile-lru evicts the least-recently-used keys with TTL — perfect for sessions. Avoid allkeys-lru to protect non-session keys. Set maxmemory to ~80% RAM.

8.4 Revocation propagation

Local revoke = DEL in Redis + insert into Kafka topic. All regions consume and DEL their copies. JWTs use a short denylist of revoked-token JTIs (24h max), checked on the auth path. Bounded size since JWTs auto-expire.

9. Scaling & bottlenecks

  • Hot user (celebrity logged in many devices): store SADD user-sessions; cap 1k devices/user.
  • Login storm after outage: rate-limit /sessions; warm Redis with replicas; pre-shard by userId.
  • Cross-region read-after-write: session created in region A, read in region B 20 ms later → may miss. Mitigation: write-through to global Redis active-active (Redis Enterprise CRDB) for tier-1 routes only.
  • Cookie size: opaque 128-bit ids beat JWT for cookie hygiene; minimize Set-Cookie traffic.

12. Talking points for the interviewer

  • "I'd ship the JWT+refresh hybrid: it's auditable, revocable within 5 min, and removes one Redis hop from 95% of traffic."
  • "For payments and admin endpoints I'd force opaque-session reads with a stricter denylist — defense in depth."
  • "The trickiest tradeoff is read-your-writes after login across regions; I'd cookie-pin to the login region for 60s to avoid 'why am I logged out' bug reports."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis cluster (12 shards, 3 replicas/shard) Sub-ms p50 GET hits the 200us session-read SLA; 12 shards give 1.2M ops/s headroom over the 500k QPS target. 3 replicas survive AZ loss without forcing global logout — the most user-visible failure mode (DDIA Ch.5 single-leader async replication). Memcached (no replication = full logout on node loss); DynamoDB (5-10ms latency too slow).
Opaque 128-bit session ID Random token in HttpOnly cookie is unforgeable, fixed-size (16 B encoded), and revocable by single Redis DEL. Compare to JWT which bloats cookies to 1KB+ and is non-revocable without an extra denylist hop. Best cookie hygiene at scale. JWT-only (revocation hard); UUID v4 string (longer cookie); hash-of-userId (predictable).
JWT + opaque hybrid Short-lived (5 min) signed JWT carries claims so 95% of reads avoid Redis entirely (CPU verify only). Refresh via opaque token in Redis happens once per 5 min — cuts session reads from 500k/s to ~100k/s while keeping revoke window bounded. Pure JWT (no revoke); pure opaque (Redis hop on every request); Macaroons (overkill, complex).
Postgres for refresh tokens Refresh tokens need durable audit + revocation history (token_hash, rotated_to chain) — 4 GB fits comfortably in Postgres with replicas. ACID transactions matter for rotation: old token invalidated and new one issued atomically (DDIA Ch.7). Redis-only (no audit trail); MySQL (similar but Postgres has better partial indexes for revoked_at).
SHA-256 hashed token storage Plaintext tokens in DB = breach exposes all sessions. SHA-256 hash is cheap (1 us) and one-way; rotated_to chain references hashes so audit trail is rebuildable. Defense in depth (DDIA Ch.12 security principles). Plaintext (catastrophic on breach); bcrypt (slow, unnecessary for high-entropy random tokens).
HttpOnly + Secure + SameSite=Lax cookies HttpOnly blocks JS access (XSS can't steal); Secure forces HTTPS; SameSite=Lax mitigates CSRF on state-changing routes. The minimum bar for a session cookie at 200M MAU — anything less is a CVE waiting to happen. localStorage (XSS-readable); SameSite=Strict (breaks OAuth redirects).
Sliding TTL via probabilistic EXPIRE Naive every-request EXPIRE = 500k writes/s → 5x cluster overhead. Refreshing only when remaining TTL <50% (probabilistic) cuts writes to 100k/s with no UX impact since nobody notices a session that re-extended at 16 min vs 14 min. Always EXPIRE (write storm); fixed TTL (forces re-login mid-session); per-request lambda update.
volatile-lru eviction policy Evicts only TTL-bearing keys when memory hits 80% — protects non-session keys (rate limits, denylist) from session pressure. Pure LRU on access timestamp aligns with usage pattern: idle users get evicted first, active users stay resident. allkeys-lru (evicts denylist, security risk); volatile-ttl (evicts about-to-expire — wasteful).
Kafka revocation topic Cross-region revoke must propagate within 5s SLA — Kafka producer→consumer chain is ~1-2s p99 globally. Decouples revoke fan-out from request path; durable retention handles brief regional consumer outages without losing revocations. Direct cross-region Redis writes (RTT cost on every revoke); polling DB (lag too high).
Bloom-style JWT denylist Bounded by JWT max TTL (5 min) so the denylist stays small (~10k entries at typical revoke rate). Checked on auth path with O(1) lookup. Without it, a stolen JWT is valid until expiry — unacceptable for password-reset flows. No denylist (5-min stolen-JWT window); long denylist (memory unbounded).
SADD user:{id}:sessions index Secondary "list all my sessions" + "force-logout-all" requires a userId→[sessionId] index. SADD/SREM is O(1); cap 1k devices/user prevents pathological celebrity accounts from blowing memory. DDIA Ch.6 secondary partition strategy. Scan all sessions (unscalable); separate scan-index DB (more moving parts).
HMAC-SHA256 JWT signing HMAC is 10x faster than RSA verify (~5us vs 50us) — at 500k req/s with JWT verify on hot path that's 50% CPU saved. Symmetric key shared across regions via secrets manager; rotation via overlapping key set with kid header. RS256 (slower verify, public key benefit not needed for first-party); none (insecure).
Region-pinned cookie for 60s post-login Read-your-writes after login fails when login writes hit region A and next read hits region B before 20ms async replication catches up (DDIA Ch.5 lag). Cookie pin forces sticky reads to login region until propagation settles, eliminating the bug. Sync cross-region replication (latency cost); always sticky (uneven load).
CSRF token in session record Double-submit cookie pattern needs a per-session CSRF token returned via header and matched on state-changing requests. Storing in HSET sess:{sid} keeps it co-located with auth state — single Redis read covers both checks. Per-request CSRF (extra round-trip); origin-only check (insufficient for legacy clients).
