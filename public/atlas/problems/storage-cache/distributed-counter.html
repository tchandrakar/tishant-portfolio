Design Distributed Counter (per-tweet view counter at 1M+ events/sec)

Hard storage-cache analytics

1. Problem statement

Build a sharded counter service that increments per-object counters (tweet views, like counts, ad impressions) at 1M+ events/sec globally, returns a "good enough" value with sub-100 ms staleness, and an exact value within 1 minute. Must survive node loss, deduplicate retried events, and remain cheap (no per-event ACID writes to OLTP).

2. Functional requirements

  • incr(objectId, delta=1, eventId) — idempotent on eventId.
  • get(objectId) → approxCount (fresh ≤ 1s) and exactCount (≤ 1 min).
  • Time-bucketed reads: count in last 1 min / 1 hr / 1 day.
  • Per-tenant rate limits and admission control.

3. Non-functional requirements

  • Throughput: 1M increments/sec sustained, 5M peak (live event).
  • Latency: incr p99 < 20 ms (fire-and-forget); get p99 < 50 ms.
  • Accuracy: approx within 1% of true; exact within 60s SLA.
  • Durability: No more than 1s of events lost on broker failure.
  • Cost: < $0.10 per million increments end-to-end.

4. Capacity estimation

  • 1M events/sec * 100 B = 100 MB/s; partitioned across 100 Kafka partitions = 1 MB/s/partition.
  • Active hot objects: 100k tweets with QPS > 100; long-tail: 10B objects with QPS < 1.
  • Storage: 10B counters * 32 B (id+count+ts) = 320 GB resident in Redis, plus rollups 1 TB in Cassandra.
  • Aggregator memory: 100k partial counters/window/aggregator * 64 B = 6.4 MB; 100 aggregators fit in 1 box.
  • Network: 100 MB/s in, 10 MB/s flush to KV every 1s = 10x reduction.

5. API design

POST /v1/incr  {objectId, delta, eventId, ts}      → 202 Accepted
GET  /v1/count/{objectId}?mode=approx|exact         → {count, freshnessMs}
GET  /v1/count/{objectId}/window?w=1m|1h|1d         → {bucket:[{ts,count}...]}
GET  /v1/topk?ns=tweet&w=5m&k=100                   → [{id,count},...]

6. High-level architecture

[client]──incr──►[edge svc]──►[Kafka topic: events, 100 partitions] │ ▼ [Flink/Samza aggregator]── pre-aggregate 1s tumbling │ ┌───────────┼────────────┐ ▼ ▼ ▼ [Redis HLL/INCR][Cassandra ts-rollup][HDFS exact archive] ▲ ▲ │ │ fast read nightly batch reconcile

Producers append to Kafka with key=objectId for partition stickiness. Stream job sums per-key in 1s windows and emits deltas to Redis (fast path) and to a time-series rollup in Cassandra (durable). Batch job reconciles exact totals from raw events.

7. Data model & storage

7.1 Redis fast layer

// One key per object, INCRBY is atomic
counter:{objectId} = int64
// Bucketed for windowed reads
window:{objectId}:{minute_epoch} = int64, TTL = 2h

7.2 Cassandra time-rollup

CREATE TABLE counts_by_min (
  object_id text, bucket_min timestamp,
  delta counter,
  PRIMARY KEY ((object_id), bucket_min)
) WITH CLUSTERING ORDER BY (bucket_min DESC);

Cassandra counter columns use vector-clock-style increment merging — eventual consistency, but commutative (DDIA Ch.5 "Conflict-Free Replicated Datatypes"). For exact dedupe we keep a Bloom filter of recent eventIds in Redis (24h TTL).

8. Detailed component design

8.1 Sharding hot keys

A single tweet at 100k incr/s exceeds one Redis shard. Split into K logical shards: counter:{tweetId}:{shard0..K-1}; client picks random shard for write, sums all K on read. Adaptive K based on observed QPS (1k QPS/shard target).

8.2 Idempotency & dedupe

Producer assigns eventId = uuid. Aggregator keeps last 5 min eventIds in a per-window Bloom filter (1% FP, ~10 bits/event). Cross-window exactness comes from batch reconciliation (Lambda architecture, DDIA Ch.10/11).

8.3 Approximate count via HLL

For "unique viewers" counters use HyperLogLog with 14-bit registers (16 KB) → 1% standard error at billions of cardinalities. Fits in a single Redis key; supports PFMERGE for windowed aggregation.

8.4 Backpressure

If Kafka lag > 60s, drop fidelity: switch to count-min sketch (CMS) sampling at 10% with extrapolation. Cheaper than dropping events; configurable per-tenant.

9. Scaling & bottlenecks

  • Hot key (Elon tweets): K-way sharding + client-side L1 LRU of read counts (1s TTL).
  • Kafka partition skew: partition by hash(objectId)+rand%K for hot keys; consumer rebalances are expensive — pin assignments.
  • Cassandra counters anti-pattern: retried writes can double-count; we mitigate by never retrying the counter write — instead replay from Kafka.
  • Clock skew: 1-min buckets tolerate ~10s skew; use ingestion ts (broker) not client ts.

12. Talking points for the interviewer

  • "For 99% of products an approximate count via Redis INCR with 1s lag is enough; exact reconciliation is a $/SLA decision."
  • "The hardest bug at scale is double-counting on retry — we anchor exactness in Kafka offsets, not in the counter store itself."
  • "I'd build with Kappa (stream-only) if Flink exactly-once is acceptable, dropping the batch path entirely (DDIA Ch.11)."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Kafka (100 partitions on events topic) Durable log absorbs 1M events/sec * 100 B = 100 MB/s with 1s lost-write SLA via leader ack=1 + replication. Partition-by-objectId gives stickiness so all increments for a tweet land at one consumer (DDIA Ch.11 stream processing). Offsets anchor exactly-once. Pulsar (similar, less ecosystem); RabbitMQ (no durable log replay for batch reconcile).
Flink stream aggregator Tumbling 1s windows pre-aggregate per-key sums, cutting downstream KV writes 10x (100 MB/s in → 10 MB/s out). Exactly-once via Kafka source + checkpoint barriers prevents the double-count bug that kills Cassandra counter retries. Samza (similar, smaller community); Kafka Streams (lacks robust event-time windowing at this scale).
Redis INCRBY for fast path Atomic single-threaded increment at sub-ms latency hits the get p99 <50 ms target with massive headroom. INCR is O(1) and race-free without app-level locks. 320 GB working set fits across ~6 shards comfortably. Memcached incr (no persistence); App-level CAS loop (10x latency, contention).
Cassandra counter columns Counter columns merge via CRDT-style commutative increment (DDIA Ch.5) — perfect for multi-replica eventual consistency. Time-series partition key (object_id, bucket_min) supports windowed reads with 1-row-per-minute density. DynamoDB ADD (vendor lock, $/op); Postgres counters (write-lock contention at 1M/s).
HyperLogLog for unique counts 14-bit HLL registers fit in 16 KB and give 1% std error at billion-scale cardinalities — exact set membership would need TBs. PFMERGE supports windowed unique-viewer aggregation across shards. Constant memory regardless of input. Set/HashSet (memory blows up); KMV sketch (worse error-per-byte ratio).
Bloom filter for eventId dedupe (24h TTL) Producer-assigned UUID + Bloom check eliminates duplicate increments from retried events. 1% FP at ~10 bits/event covers 5 min of 1M events/s in ~600 MB; cross-window exactness comes from batch reconciliation (DDIA Ch.10). Exact KV dedupe (10x memory); no dedupe (over-count on retry).
K-way logical sharding for hot keys Single tweet at 100k incr/s exceeds one Redis shard's 100k ops/sec ceiling. Splitting into {tweetId}:{shardN} with random write + sum-on-read trades read cost (K GETs) for write throughput (K-fold). Adaptive K targets 1k QPS/shard. Vertical scaling Redis (single-thread ceiling); pre-sharding all keys (overhead for cold tail).
Count-min sketch (backpressure mode) When Kafka lag >60s, switch to CMS sampling at 10% with extrapolation — cheaper than dropping events outright. Preserves the within-1% accuracy SLA at the cost of momentary fidelity. Per-tenant configurable. Drop events (violates SLA); slow path with full fidelity (cascading lag).
HDFS exact archive + nightly batch Lambda architecture (DDIA Ch.10/11): raw events archived for ground-truth reconciliation when the 60s exact SLA matters. Batch sums over 24h are deterministic and replayable, catching any stream-side drift before it pollutes long-term rollups. Stream-only Kappa (acceptable but no recovery from stream bugs); S3 only (HDFS preferred for batch locality).
Broker ingestion timestamp (not client clock) 1-min buckets tolerate ~10s skew, but client clocks drift wildly across mobile devices. Using Kafka broker timestamp (NTP-disciplined, ~10ms) keeps bucket assignments deterministic and avoids late-data tail (DDIA Ch.8 unreliable clocks). Client timestamp (skew breaks bucketing); ingestion-time at edge svc (extra hop variance).
Edge service (HTTP → Kafka producer) Decouples client retry semantics from Kafka. 202 Accepted returns immediately (sub-20 ms p99) so callers don't wait for downstream aggregation. Owns rate limiting + admission control per tenant on the request hot path. Direct Kafka produce from clients (auth complexity, version drift); sync ack (latency cost).
Per-tenant token bucket admission Multi-tenant 1M ops/s service needs noisy-neighbor isolation; token bucket per tenant drops excess at the edge before Kafka sees it, protecting downstream from runaway producers and preserving budget for fair tenants. No admission (one tenant DOSes pipeline); fixed quota (no burst tolerance).
Pinned consumer assignments Consumer rebalances stall the entire group for 5-30s — unacceptable at 1M events/s. Pinning partitions to consumer instances keeps Flink checkpoints cheap and avoids the partition-skew amplification when consumers shuffle. Default range/round-robin (frequent rebalance); cooperative sticky (better but Kafka 2.4+).
Prometheus metrics on Kafka lag, Redis QPS, CMS error The three leading indicators for the failure modes in section 9. Lag >60s triggers CMS fallback; Redis shard QPS >80% triggers K-way reshard; CMS error >1% alerts ops. Pull-scrape every 15s is cheap relative to event volume. Push StatsD (counter loss on agent restart); polling INFO only (no time-series).
