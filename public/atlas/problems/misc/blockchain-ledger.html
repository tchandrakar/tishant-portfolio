Design Blockchain-style Append-only Ledger (Merkle tree, BFT consensus)

Very Hard misc database-ddia payments

1. Problem statement

Build a permissioned, append-only ledger of financial transactions across N independently-operated organizations (~20 banks, ~5 regulators) with cryptographic auditability and Byzantine-fault tolerance. Inspired by Hyperledger Fabric, R3 Corda, and Diem — not a public PoW chain. Throughput target: 10 K tx/s, finality < 2 s, no double-spend, immutable history with Merkle proofs for any record.

2. Functional requirements

  • Submit a transaction (payer, payee, amount, signed); validators agree on inclusion and ordering.
  • Query account balance at any historical block height.
  • Generate a Merkle inclusion proof for a given tx — verifiable offline by a regulator.
  • Smart-contract-style chaincode for conditional logic (escrow, multi-sig).
  • Membership service: enroll/revoke organizations, rotate validator keys.

3. Non-functional requirements

  • Throughput: 10 K tx/s sustained, 30 K tx/s burst.
  • Finality: deterministic, < 2 s p99 (no probabilistic confirmations).
  • Fault model: tolerate f Byzantine validators with N ≥ 3f+1 (PBFT/HotStuff).
  • Auditability: every block links its predecessor via SHA-256; tampering with any historical record breaks the chain.

4. Capacity estimation

  • 10 K tx/s × 86400 s = 864 M tx/day; tx ~500 B → 432 GB/day, ~160 TB/yr (per replica).
  • Block size 4 MB at 1 s cadence holds ~8 K tx; pad with empty if low traffic.
  • Validators: 25 (3f+1 with f=8 tolerated). Cross-DC, low-latency mesh between them.
  • State DB (UTXO/account model): 100 M accounts × 1 KB = 100 GB live state.

6. High-level architecture

[Client SDK] --signed tx--> [Org peer (gateway)] --gossip--> [Mempool] | v +--> [Orderer cluster: HotStuff BFT] --> block proposed | [25 validators vote: prepare → pre-commit → commit] | v [Block committed] | +-----------------------------+-----------------------------+ v v v [Org peer A: ledger DB] [Org peer B: ledger DB] [Regulator: read-only] [State DB (RocksDB)] [State DB (RocksDB)] [Merkle index] [Merkle index]

Walk-through: clients sign txs with their private key. Org peers gossip into a shared mempool. The orderer cluster runs HotStuff (3-phase: prepare, pre-commit, commit) to agree on block n's contents and order. Once committed, every peer applies the block to its local state DB. The Merkle root of the block's tx set is in the header — anyone can prove inclusion in O(log n).

7. Data model & storage

7.1 Block / chain (immutable log)

Block:
  header:
    height, timestamp, prev_hash, merkle_root, validators_sig[]
  txs[]:
    tx_id, payload, signatures[]

Chain rule: header.prev_hash = SHA256(prev_block.header)

7.2 Merkle tree of txs

          merkle_root
         /            \
       h01            h23
      /   \          /   \
     h0   h1        h2   h3        (hN = SHA256(tx_N))
Proof for tx_2 = [h3, h01]; verifier hashes upward, compares to root.

7.3 State DB (RocksDB)

key = account_addr (20 B), value = {balance, nonce, code_hash}
Append-only changeset per block; snapshot every 10 K blocks for fast cold-start.

8. Detailed component design

8.1 Consensus: HotStuff

Linear-view-change PBFT variant used by Diem/Aptos. Each round: leader proposes block, validators send PREPARE votes, then PRE-COMMIT, then COMMIT — all aggregated via BLS threshold signatures so a single 96 B sig represents 17 votes. Throughput scales to 10 K tx/s on commodity 25-node networks; finality is one round (~500 ms). View-change runs in O(1) message complexity, unlike classic PBFT's O(n²).

8.2 BFT vs CFT

Raft (CFT) trusts all nodes — fine for one company. Across orgs we assume Byzantine actors (compromised key, malicious admin), so we need ≥ 3f+1 nodes and signed votes. See DDIA Ch. 9: "Byzantine fault tolerance" section.

8.3 Smart-contract execution

Endorsement-then-order model (Hyperledger Fabric): chaincode runs on a quorum of endorsing peers, produces a read-write set, signed; orderer just orders those — keeps consensus deterministic and side-effect-free. Avoids the Ethereum problem of re-running every contract on every node.

8.4 Anti-double-spend

State DB holds nonce per account; tx with stale nonce is rejected at endorsement. UTXO alternative (Bitcoin-style) trades simpler concurrency for harder querying.

8.5 Privacy

Channel-based isolation: per-deal sub-ledger visible only to participants; main chain stores only a salted hash of the off-chain payload. Optional zk-SNARK range proofs hide amounts while letting validators check non-negativity.

9. Scaling & bottlenecks

  • Validator count: BFT message complexity grows quickly; HotStuff's linear-view-change keeps 25 nodes feasible — beyond 100, throughput drops sharply.
  • State growth: snapshot + prune; archive nodes keep full history, validators keep last 30 days.
  • Cross-org WAN latency: co-locate validators in 3 regions max; 100 ms RTT round between regions caps round latency.
  • Block-leader hotspot: rotate leader every block (HotStuff pipelined), spreading egress load.

12. Talking points for the interviewer

  • "I'd start by drawing the line between permissioned and public chains — public PoW will never hit 10 K tx/s with 2-s finality, and naming that up front saves 10 minutes."
  • "The hardest real-world part is key management — losing a validator's BLS key requires a coordinated rotation, and I'd dedicate slide time to HSMs and threshold-sig DKG."
  • "For privacy, I'd default to channel isolation + payload hashing; zk-SNARKs are tempting but the prover cost (seconds per tx) blows the SLO unless we batch."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
HotStuff BFT consensusLinear-view-change PBFT variant hits 10 K tx/s on 25 validators with O(1) message complexity per view; classic PBFT's O(n²) would cap us at ~2 K tx/s and miss the < 2 s finality target.PBFT (n² messages); Tendermint (n² in worst case); Raft (CFT only, no Byzantine tolerance).
BLS threshold signaturesAggregating 17 of 25 validator votes into a single 96-byte signature keeps block headers small; without aggregation, header sigs alone would balloon block size past the 4 MB target at 8 K tx/block.ECDSA per-validator (25× signature bloat); Schnorr (no production-hardened threshold).
SHA-256 (Merkle & chain hash)Hardware-accelerated on x86/ARM (~1 GB/s/core) keeps Merkle root computation under 5 ms per 4 MB block; FIPS-approved is mandatory for the regulator audit story.BLAKE3 (faster but not FIPS-approved); SHA-3 (slower in HW).
Merkle tree (tx inclusion)O(log n) proof of any tx in a block — for 8 K tx/block that's 13 hashes (~416 B); regulators verify offline without trusting any single peer, satisfying the audit-anywhere requirement.Patricia trie (overkill for ordered set); flat hash list (O(n) proof size).
RocksDB (state DB)100 GB live state with append-only changesets per block fits LSM-tree's write-amplification budget; Hyperledger and Diem both use RocksDB for the same reason — block-replay performance.LMDB (B-tree write amp on hot accounts); LevelDB (RocksDB's older, less-tuned fork).
gRPC + Protobuf (peer mesh)Cross-org WAN at 100 ms RTT needs HTTP/2 multiplexing to keep the 3-phase HotStuff round under 500 ms; Protobuf's deterministic encoding is critical so all peers hash identical bytes.JSON-RPC (non-deterministic field order); custom TCP (years of edge cases).
HSMs (validator keys)Validator BLS keys are the security root; HSM-stored keys with FIPS 140-2 Level 3 prevent extraction even with root access — losing one key only requires DKG rotation, not a chain fork.Software keys (catastrophic on host compromise); KMS (worse latency on per-vote signing).
Ed25519 (client tx signing)Clients (banks, end-users) need fast, small (64 B) signatures with batch-verifiable semantics; Ed25519's deterministic nonces also avoid the Sony PS3-style randomness leak.ECDSA (non-deterministic, larger sig); RSA (256 B sigs blow block size).
Endorsement-then-order (Fabric model)Chaincode runs on endorsing peers, produces signed read-write set, orderer just orders — keeps consensus deterministic and side-effect-free, avoiding Ethereum's "re-execute everywhere" CPU bottleneck at 10 K tx/s.Order-then-execute (every node re-runs every contract); off-chain execution (loses determinism).
WASM (chaincode runtime)Sandboxed deterministic execution across heterogeneous validators (Linux, BSD, Windows); < 100 µs invocation overhead per tx fits the 10 K tx/s budget, and language-agnostic onboarding for the 20 banks.EVM (Solidity-only, slower interpreter); Docker chaincode (non-deterministic resource limits).
Channel isolation (per-deal sub-ledger)Two banks doing a private deal don't want all 25 validators to see amounts; per-channel sub-ledgers with hash-on-main-chain give privacy without sacrificing the global audit trail.Plain-text on chain (privacy violation); zk-SNARKs (multi-second prover cost breaks 2 s finality).
Distributed Key Generation (DKG)BLS threshold sig requires no single party knows the master key; DKG protocols (Pedersen-VSS) let 25 validators jointly generate shares — necessary for the "no trusted setup" property regulators demand.Trusted dealer (single point of compromise); centralized KMS (defeats permissioned model).
Apache Kafka (mempool gossip)Pending tx pool must survive validator restarts and replay missed proposals; durable mempool on Kafka gives leaders deterministic batching and lets joiners catch up without re-asking peers.In-memory gossip (lost on restart); BitTorrent-style (overkill for 25-node mesh).
