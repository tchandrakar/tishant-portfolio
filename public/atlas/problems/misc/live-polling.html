Design Live Polling / Voting at Scale (1M+ votes/sec, real-time aggregation)

Hard misc analytics

1. Problem statement

Design a real-time polling system used during live TV (e.g., Eurovision, IPL fan-poll) and global product surveys. Tens of millions of viewers cast votes during a 30-minute window with peaks above 1 M votes/sec. Aggregate counts must update on screen within 2 seconds, votes must be one-per-user (or N-per-user with rate limit), bot-and-fraud flagged, and final tallies auditable.

2. Functional requirements

  • Cast a vote for option O on poll P.
  • Real-time aggregate (per option, optionally per region/age bucket).
  • Per-user vote rules: one-and-only-one, or N-per-window.
  • Live leaderboard for multi-option polls (top-K options by votes).
  • Auditable final tally with signed Merkle root.
  • Fraud detection: bot rings, geographic anomalies, mass-vote scripts.

3. Non-functional requirements

  • Latency: ack < 200 ms p99; aggregate visible < 2 s p99.
  • Scale: 1.5 M votes/sec peak, 30 min event window, 100 M total votes.
  • Durability: zero vote loss after ack; survive AZ failure.
  • Integrity: tamper-evident final tally signed by 3 independent attesters.

4. Capacity estimation

  • 1.5 M votes/sec × 256 B = 400 MB/s ingress; Kafka 1024 partitions × ~400 KB/s/partition.
  • If aggregator is single-counter per option × 10 options = 1.5 M increments/s on 10 keys ⇒ classic hot key. Solution below.
  • Per-region rollups: 200 regions × 10 options × 4 buckets = 8K counters ⇒ trivial after sharding.
  • Storage: 100 M votes × 200 B = 20 GB raw events; Cassandra retains 1 yr.
  • Bandwidth: 30 min × 1.5 M × 256 B = 720 GB total event volume.

5. API design

POST /v1/polls/{id}/vote   {option, idempotencyKey}   → {accepted, voteId}
GET  /v1/polls/{id}/live                                → {totals:{A:..,B:..}, ts}
GET  /v1/polls/{id}/final                               → {totals, merkleRoot, signers}
WS   /v1/polls/{id}/stream                              → push aggregate every 1 s

6. High-level architecture

[App]──HTTPS──►[Edge GW]──►[Vote Ingest (stateless)]──►[Kafka: votes (1024 parts)] │ │ │ ▼ │ [Idempotency Cache (Redis)] │ ▼ [Anti-fraud: rate limit + JWT + device attest] [Kafka: votes]──►[Flink Pre-Agg (per-partition local count)]──►[Kafka: agg-deltas] │ ▼ [Flink Global Agg]──►[Redis Live Counters] │ │ ▼ ▼ [Cassandra: vote_log] [WS Fanout] │ ▼ [Audit + Merkle builder]

Votes append to Kafka; per-partition local counters pre-aggregate (combiner pattern), emit deltas every 200 ms; global aggregator merges into Redis counters; WS fanout pushes 1 Hz updates to all viewers.

7. Data model & storage

7.1 Vote log (Cassandra)

CREATE TABLE votes (
  poll_id UUID, ts_bucket BIGINT, vote_id timeuuid,
  user_id BIGINT, option SMALLINT, region TEXT,
  device_attestation TEXT,
  PRIMARY KEY ((poll_id, ts_bucket), vote_id)
);

7.2 Live counters (Redis)

INCRBY poll:{id}:opt:{o}:s{shard} 1; 64 shards per option to dodge hot key. Read-side sums all shards every 1 s and caches.

7.3 Idempotency (Redis)

SET dedup:{userId}:{pollId} 1 NX EX 86400: returns OK only on first call. Reject 429 otherwise (and don't write to Kafka).

7.4 Audit Merkle tree

Each minute, Cassandra reader scans the minute's votes, computes a Merkle root, signs with HSM. Final root = root over per-minute roots, co-signed by 3 attesters at poll close.

8. Detailed component design

8.1 Hot-counter elimination via sharded INCR

Even Redis flinches at 150 K INCR/s on one key. We shard each option counter into 64 keys; ingest writes to a random shard, reader sums shards (64 GETs ~ 0.5 ms pipelined). This is the same pattern as Cassandra counter-shards.

8.2 Combiner pre-aggregation

Flink per-partition operator buffers counts in memory, emits delta-snapshots every 200 ms (~5 events/s instead of 1.5 M). Global aggregator handles ~5 K events/s — trivial. This is DDIA Ch. 11's "stream-stream join with windowed reduce" pattern.

8.3 Approximate live, exact final

Live UI shows the Redis-derived count, accepting up to 2 s lag; final tally is recomputed from Cassandra vote_log after dedup and fraud removal. Discrepancy displayed with a "preliminary results" banner during the window.

8.4 Fraud detection

Online: per-IP rate limit (10/min), device-attestation token, suspicious-UA list. Offline: count-min sketch tracks per-user vote frequency; HLL counts unique users per option per region — a sudden HLL/total-vote ratio drop indicates botting (1M votes from 100 users). Suspicious votes are kept in Cassandra but excluded from final tally.

8.5 Backpressure & load shedding

If Kafka ingress lag exceeds 5 s, Vote Ingest sheds: 5xx + Retry-After to non-priority cohorts (clients in regions already heavily over-quota). This preserves global representativeness during overload.

9. Scaling & bottlenecks

  • Hot Kafka partition (one option): partition by hash(userId), not option; combiners do per-option counting locally.
  • Idempotency Redis hot key: dedup key includes userId, naturally distributed.
  • WS fanout to 30 M viewers: SSE-over-CDN with shared aggregate snapshot every 1 s — only 86 KB/s aggregate egress at origin (one snapshot per second × ~64 KB).
  • Audit replay: Cassandra full scan of 100 M rows runs in 10–15 min on parallel readers; runs once after poll close.

  • Ch. 11 Stream Processing — Kafka log + Flink windowed aggregation; combiner pattern; exactly-once via idempotent producer + dedup key.
  • Ch. 9 Consistency — final tally requires a strict serialization point (poll close timestamp); we use Kafka offset as the cut.
  • Ch. 5 Replication — Cassandra LOCAL_QUORUM keeps the audit log durable across AZs.
  • Sketch data structures — count-min sketch for per-user vote frequency, HyperLogLog for unique-voter counts, Bloom filters for fast "have we seen this idempotency key?" check before Redis hit.

12. Talking points for the interviewer

  • "The interesting subproblem isn't ingestion — Kafka eats 1.5 M/sec for breakfast — it's the single hot counter. Sharding INCR is the move."
  • "Live and final must be different views of the same log. Confusing them is how you ship a tally that mismatches what people saw on screen."
  • "Bot votes are 80% of operational pain. I'd budget 30% of design effort on attestation + offline anomaly scoring, not just the happy path."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Apache Kafka (1024 partitions) 1.5 M votes/s × 256 B = 400 MB/s ingress; 1024 partitions hold per-partition rate at ~400 KB/s with comfortable headroom for the 30-min event; partition by userId (not option) avoids the catastrophic single-option hotspot during a polarised vote. Kinesis (rejected: shard rebalance painful at 1.5 M/s), RabbitMQ (rejected: not partition-scalable).
Sharded Redis INCRBY (64 shards/option) Single Redis key flinches at 150 K INCR/s; 64 shards per option drop per-key load to ~2.3 K/s — well within sub-ms territory; reader sums shards via 64 GETs pipelined in 0.5 ms, total scatter-gather still <5 ms. Single counter (rejected: hot-key meltdown), Cassandra counters (rejected: 4× write cost via Paxos).
Apache Flink (combiner pre-agg) Per-partition local counters emit 200 ms delta-snapshots — DDIA Ch. 11's combiner pattern; reduces 1.5 M events/s to ~5 K/s downstream, making the global aggregator trivial and capping aggregate visibility lag at 200 ms (well under the 2 s SLA). Direct write to global counter (rejected: hot key), Spark Streaming (rejected: micro-batch lag).
Redis SETNX + 24h TTL (idempotency) SET dedup:{userId}:{pollId} 1 NX EX 86400 returns OK only on first call, atomically — keys naturally distributed by userId (no hot key), and 24h TTL covers the entire poll lifetime plus retries without unbounded RAM growth. Database UNIQUE (rejected: hot row contention), client-side guard (rejected: trivially bypassed).
Cassandra (vote audit log) 100 M votes × 200 B = 20 GB per event, append-only with 1-yr retention; partition (poll_id, ts_bucket) with timeuuid clustering keeps partitions under ~1 GB; LSM trees swallow 1.5 M writes/sec across the cluster easily. Postgres (rejected: vacuum cost), DynamoDB (rejected: write-unit cost at 100 M votes).
Count-min sketch (per-user vote frequency) Tracking exact per-user vote counts across 30 M users would cost gigabytes; CMS in <5 MB flags users hitting unusual rates (e.g., 100 votes/min), feeding the offline fraud-removal pass before final tally. Per-user counters (rejected: GB memory), sampling (rejected: misses small bot rings).
HyperLogLog (unique voters per option) Detects bot rings by HLL/total-vote ratio — 1 M votes from only 100 unique users is the signal; 12 KB per option per region gives 0.81% error vs storing 30 M userIds, critical at 1.5 M votes/s ingest. Redis SETs of userIds (rejected: GBs of memory), no signal (rejected: leaves bot detection blind).
Bloom filter (idempotency pre-check) Cheap "have we seen this idempotency key?" filter sits in front of Redis SETNX — kills round-trip on duplicates from retry storms; with 1% false-positive rate the worst case is a redundant Redis call, no correctness risk. Direct Redis call (rejected: round-trip on every retry), perfect dedup (rejected: cost-prohibitive in path).
Merkle tree + HSM signing Per-minute Merkle root over the minute's votes signed by HSM, final root co-signed by 3 attesters — gives tamper-evident integrity for the final tally; 100 M leaf hashes computable offline in <15 min on parallel readers. Append-only log only (rejected: no integrity proof), full blockchain (rejected: massive overkill for 30-min event).
SSE over CDN (WS fanout) 30 M viewers × 1 Hz aggregate = unmanageable WS fan-out at origin; SSE-over-CDN serves a 64 KB shared snapshot every 1 s — origin egress drops to ~64 KB/s while every viewer sees fresh totals within 1 s of refresh. WebSockets per-viewer (rejected: 30 M connections), polling (rejected: 30 M GET/s thundering herd).
JWT + device attestation Vote ingest validates a short-lived JWT minted at app launch plus device-attestation token (Play Integrity / DeviceCheck) — kills the trivial scripted-vote attacker before they ever reach Kafka, and the per-IP rate limit handles the residue. Captcha (rejected: friction), no edge auth (rejected: bot floor unbounded).
Token bucket rate limit (edge) Per-IP 10 votes/min at edge gateway sheds the bulk of script-driven traffic before it consumes Kafka partitions; combined with backpressure shedding when Kafka lag >5 s, preserves global representativeness during 1.5 M/s overload. No edge limit (rejected: Kafka saturates), per-user only (rejected: anonymous abuse path open).
Monotonic clock cut (poll close) Final tally requires a strict serialization point — Kafka offset at the close instant defines the cut, not wall-clock; consumers read up-to-offset and stop, eliminating any "did this vote sneak in 50 ms late?" ambiguity that could invalidate the audit. Wall clock cut (rejected: NTP skew across consumers), no cut (rejected: tally drift).
Cassandra LOCAL_QUORUM 2-of-3 replica acks per write keeps audit-log durable across AZs without paying cross-region latency; survives single-AZ failure with zero vote loss after ack — directly hits the durability NFR for the live broadcast event. EACH_QUORUM (rejected: 100 ms cross-region per write), ONE (rejected: data loss on replica crash).
