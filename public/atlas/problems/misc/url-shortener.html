Design URL Shortener (TinyURL/Bitly — base62, click analytics)

Medium misc storage-cache analytics

1. Problem statement

Build a service that converts long URLs (up to 2 KB) into compact 7-character codes (e.g. bit.ly/3xZk9Qp), redirects on hit with sub-50 ms latency, and emits a click-event stream for analytics. Bitly handles roughly 600 M new shortens per month and serves ~10 B redirects/month, so reads dominate writes by ~100:1.

2. Functional requirements

  • POST a long URL, receive a short code; optional custom alias and expiry date.
  • GET /{code} issues 301/302 to the original URL.
  • Per-link analytics: total clicks, geo, referrer, device, time bucket.
  • Soft-delete or expire links; user dashboards with auth and team workspaces.
  • Spam/malware checks against Google Safe Browsing before issuing the code.

3. Non-functional requirements

  • Latency: redirect p99 < 50 ms at edge; shorten p99 < 200 ms.
  • Scale: 600 M writes/month (~230 wps avg, 2 K wps peak), 10 B reads/month (~3.8 K rps avg, 40 K rps peak).
  • Availability: 99.99% on the redirect path (revenue-critical).
  • Durability: never lose a mapping; codes are forever-stable.

4. Capacity estimation

  • 5 yr retention: 600 M × 12 × 5 = 36 B rows. Row ~500 B → ~18 TB primary, 36 TB with replicas.
  • Code space: 62^7 = 3.5 × 10^12, plenty for decades of growth at current rate.
  • Hot cache: top 20% of codes serve 80% of traffic — ~7 B keys × 200 B = 1.4 TB Redis cluster (sharded).
  • Click events: 10 B/month × 300 B = 3 TB/month into the analytics warehouse.

5. API design

POST /v1/shorten           {longUrl, customAlias?, expiresAt?}  → {code, shortUrl}
GET  /{code}                                                    → 301 Location: longUrl
GET  /v1/links/{code}/stats?from=&to=&bucket=hour              → time-series
DELETE /v1/links/{code}                                         → 204
POST /v1/bulk-shorten      {urls[]}                             → {results[]}

6. High-level architecture

[Client] -- HTTPS --> [Anycast CDN/Edge] --> [Redirect Svc] --> [Redis] --> [DynamoDB/MySQL] | +--> [Kafka click-events] --> [Flink] --> [ClickHouse] | [Dashboard] <-- [Analytics API] <-------------------------------------------------------+ ^ [Shorten Svc] <-- [API GW] <-- [Client] | | | +--> [ID Generator (Snowflake/zk-counter)] + +--> [Safe-Browsing checker]

Walk-through: redirect path is a single Redis GET; on miss, fall through to the KV store and back-fill cache. Writes go through a stateless Shorten Service that pulls a 64-bit ID, base62-encodes it, double-checks for collision (custom aliases only), and persists. Click events are fire-and-forget into Kafka — never block the 302.

7. Data model & storage

7.1 Mappings (DynamoDB)

Table: links
  PK: code (string)        -- e.g. "3xZk9Qp"
  long_url: string
  owner_id: string         -- GSI for "list my links"
  created_at: number
  expires_at: number       -- TTL attribute
  status: enum(active, disabled, malware)

7.2 Why DynamoDB?

Single-key lookup, predictable single-digit-ms p99, native TTL for expiry, on-demand scaling for the 40 K rps peak. Multi-region global tables give us active-active redirects. MySQL would work but partitioning 36 B rows manually is painful — see DDIA Ch. 6.

7.3 Click events (ClickHouse)

CREATE TABLE clicks (
  ts DateTime, code String, country LowCardinality(String),
  referrer String, ua_bucket LowCardinality(String)
) ENGINE = MergeTree PARTITION BY toYYYYMM(ts) ORDER BY (code, ts);

8. Detailed component design

8.1 ID generator and base62

A Snowflake-style 64-bit ID (timestamp || worker-id || seq) is base62-encoded — 11 chars max, but we trim to 7 by allocating from a sharded counter where each shard hands out 1 M-block ranges to workers (tracked in ZooKeeper). This keeps codes short and roughly k-sortable for partition affinity.

8.2 Custom aliases

Optimistic insert with a conditional attribute_not_exists(code) check; on conflict return 409. Reserved-word blocklist guards against impersonation (/login, /admin).

8.3 Redirect service

Stateless Go service, 1 ms cache hit budget. Serves 302 by default (so we can change targets and revoke); 301 only when caller opts in. Response includes Cache-Control: private, max-age=90 so browsers do not poison shared caches.

8.4 Analytics pipeline

Kafka topic clicks partitioned by code-hash (64 partitions). Flink job sessionizes by (code, country, hour) and writes pre-aggregated rollups into ClickHouse. Raw events kept 30 days for ad-hoc.

9. Scaling & bottlenecks

  • Hot codes (viral links): consistent-hash + per-key local LRU on the redirect host shields Redis; CDN edge cache for ultra-hot links with 30 s TTL.
  • Thundering herd on cache miss: request coalescing (singleflight) at the redirect tier.
  • Custom alias contention: in-memory bloom filter at the API tier rejects 99% of duplicates before hitting DynamoDB.
  • Geographic latency: active-active multi-region with last-write-wins on writes (acceptable since codes are immutable post-creation).

12. Talking points for the interviewer

  • "I'd start with a Snowflake-ID + DynamoDB MVP and only add ZooKeeper-coordinated counters when collision rate on custom aliases crosses 1%."
  • "The hardest part is unbounded growth of analytics — I separate hot path (Redis + Kafka) from cold path (ClickHouse) so a warehouse outage never breaks redirects."
  • "For abuse, I lean on Safe-Browsing at write time plus per-IP rate limits; revocation must propagate in < 60 s, hence 302 over 301."
