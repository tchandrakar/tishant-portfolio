Design Distributed Cron / Scheduler (Quartz at scale, exactly-once trigger)

Hard misc database-ddia

1. Problem statement

Design a horizontally scalable scheduler that fires millions of jobs per minute on cron expressions or one-shot timestamps, dispatches them to user-defined workers (HTTP webhook, Kafka, gRPC), and guarantees at-least-once with effectively-once semantics — even across host failures, network partitions, and clock drift. Think AWS EventBridge + Quartz + Airflow's scheduler.

2. Functional requirements

  • Create/update/delete schedules with cron expression or epoch timestamp.
  • Targets: HTTP, Kafka topic, gRPC, AWS SQS, custom adapter.
  • Exactly-once trigger semantics from the user's perspective (idempotent dispatch).
  • Per-job retry policy with exponential backoff and DLQ.
  • Pause/resume jobs; backfill missed runs.
  • Tenant isolation: 100 K jobs per tenant, no cross-tenant interference.

3. Non-functional requirements

  • Trigger accuracy: p99 within 1 s of scheduled instant.
  • Scale: 10 M scheduled jobs total, 500 K firings/minute steady, 5 M/min burst.
  • Durability: survive AZ failure with zero job loss.
  • Availability: 99.99%; partial degradation acceptable but no missed fires.

4. Capacity estimation

  • 10 M jobs × 500 B row = 5 GB hot working set (fits in RAM across shards).
  • 500 K firings/min ≈ 8.3 K/s; with 1 ms write per fire-event ⇒ 8 K writes/s, well under sharded Postgres limits.
  • Horizon scan window 60 s; per shard (256 shards) avg 33 due jobs/s.
  • Audit log: 500 K × 1 KB × 60 = 30 GB/hr ⇒ rolled to S3 hourly.

5. API design

POST /v1/jobs        {name, cron, target, retry, payload}   → jobId
PATCH /v1/jobs/{id}  {paused: true}
GET  /v1/jobs/{id}/runs?from=...                             → [{runId, firedAt, status, attempts}]
POST /v1/jobs/{id}/trigger                                   → manual fire
DELETE /v1/jobs/{id}

6. High-level architecture

[Client]──►[API GW]──►[Job Service]──►[Postgres (sharded by tenant)] │ ▼ [Schedule Index (per-shard)] │ ┌─────────────────┼─────────────────┐ ▼ ▼ ▼ [Scheduler-1] [Scheduler-2] [Scheduler-N] (leases via etcd) │ │ │ └────────►[Kafka: due-jobs (256 partitions)] │ ▼ [Dispatcher Workers] │ ┌─────────────────┼─────────────────┐ ▼ ▼ ▼ [HTTP] [Kafka] [gRPC] │ ▼ [Run Log (Cassandra)]

Schedulers own a slice of shards (etcd lease); each scans its index for jobs due in the next 10 s, emits a "due" record to Kafka keyed by jobId, and updates nextFireAt atomically. Dispatcher consumers fire targets idempotently.

7. Data model & storage

7.1 Jobs (Postgres, sharded by tenant)

CREATE TABLE jobs (
  job_id UUID PRIMARY KEY,
  tenant_id UUID NOT NULL,
  cron TEXT, next_fire_at TIMESTAMPTZ NOT NULL,
  target JSONB, payload BYTEA, retry JSONB,
  paused BOOLEAN DEFAULT false,
  version BIGINT NOT NULL DEFAULT 0
);
CREATE INDEX idx_due ON jobs (next_fire_at) WHERE paused = false;

7.2 Run log (Cassandra, time-bucketed)

CREATE TABLE runs (
  job_id UUID, fire_bucket bigint, run_id timeuuid,
  status text, attempts int, response int, error text,
  PRIMARY KEY ((job_id, fire_bucket), run_id)
) WITH default_time_to_live = 7776000;  -- 90 days

7.3 Why Postgres for schedule, Cassandra for runs?

Schedule needs ACID UPDATE+SELECT for "claim and advance"; B-tree on next_fire_at is ideal. Run log is append-only at huge volume — Cassandra wins on write throughput and TTL.

8. Detailed component design

8.1 Atomic claim & advance

To prevent two schedulers firing the same job, the scheduler uses optimistic concurrency: UPDATE jobs SET next_fire_at = $new, version = version+1 WHERE job_id = $id AND version = $v. Only the winner emits the Kafka due-record. The Kafka record key = jobId:fireEpoch; downstream uses this as the idempotency key.

8.2 Effectively-once dispatch

Dispatcher writes (jobId, fireEpoch) → INSERT-IF-NOT-EXISTS into a fire_log table (Cassandra LWT) before invoking the target. Duplicate Kafka messages no-op. Target HTTP call carries Idempotency-Key: jobId:fireEpoch for the customer to dedupe end-to-end.

8.3 Sharding & rebalance

256 logical shards mapped to physical schedulers via Rendezvous hashing. Lease in etcd with 10 s TTL; on lease loss, scheduler stops claiming. Total claim window ≤ 5 s ahead, so a 10 s outage leaves jobs late by ≤ 15 s p99 — recovered by the new lease holder running its first scan immediately.

8.4 Cron expansion

Standard 5-field plus optional seconds; expansion uses ical-rrule semantics. nextFireAt recomputed from fireEpoch after each fire to avoid drift.

8.5 Backfill & catch-up

If a job missed N fires (e.g., paused), policy chooses: skip, fire-once-with-latest-payload, or fire-each-with-bounded-rate. Bounded-rate uses a token bucket per job to avoid thundering herd.

9. Scaling & bottlenecks

  • Hot minute (00:00): millions of jobs fire same instant. Solution: jitter cron parses by ≤ 30 s if user opts in; load-shed via per-target rate limit at dispatcher.
  • Postgres index hot spot: partial index WHERE paused=false + per-shard table keeps B-tree small.
  • Clock drift: all schedulers use monotonic clocks for "due now" checks (DDIA Ch. 8); wall clock only for cron expansion.
  • Long-running webhooks: dispatcher hands off to async worker pool with 30 s timeout; outcome reconciled by status callback.

12. Talking points for the interviewer

  • "True exactly-once across user webhooks is impossible; what we sell is at-least-once with a stable idempotency key the customer can dedupe on."
  • "The schedule store is the heart — I'd benchmark Postgres vs CockroachDB early; the cron index is small per-shard, so Postgres scales further than people expect."
  • "For the 00:00 thundering herd I'd push optional jitter at job-create time; otherwise the dispatcher must throttle and we miss SLA."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Quartz scheduler (engine) Battle-tested cron-expression parser with iCal-rrule semantics — the parsing layer is the boring 5% you don't want to reinvent; we wrap it with our own claim-and-advance store rather than its JDBC backend, which doesn't scale past one node. cron4j (rejected: thinner ecosystem), home-grown parser (rejected: edge cases around DST and 29 Feb).
PostgreSQL (sharded) 10 M jobs × 500 B = 5 GB working set fits in RAM across 256 shards; partial B-tree index WHERE paused=false on next_fire_at serves the every-second "due now" scan in <5 ms with hot pages cached, and CAS on version gives lock-free claim semantics. CockroachDB (rejected: 5× write latency for the CAS path), DynamoDB (rejected: cron-range scan unfriendly).
etcd (lease-based shard ownership) Strong consistency on lease ownership prevents two schedulers from claiming the same shard during a partition — the only place we accept an external Raft dependency, because exactly-one-leader-per-shard is the safety boundary for "no missed fires". ZooKeeper (rejected: heavier operationally), Redis (rejected: no real Raft, lease bugs at scale).
Apache Kafka (256 partitions) Decouples scheduler from dispatcher and absorbs the 5 M/min burst at midnight; partition by jobId preserves per-job ordering, key jobId:fireEpoch lets downstream LWT dedupe — same record key flowing through the whole effectively-once chain. RabbitMQ (rejected: ordering across partitions weaker), SQS (rejected: 256-partition fan-out cumbersome).
Cassandra LWT (fire_log) INSERT-IF-NOT-EXISTS keyed on (jobId, fireEpoch) achieves effectively-once dispatch; LWT's 4× write cost is acceptable here because we only LWT 8.3 K times/sec on the dispatcher, not on the hot ingestion path. Postgres unique constraint (rejected: cross-region replication awkward), Redis SETNX (rejected: not durable enough).
Cassandra (run log) 500 K firings/min × 1 KB = 30 GB/hr append-only writes with 90-day TTL — LSM trees, time-bucketed clustering keys, native TTL all match perfectly; partition (job_id, fire_bucket) keeps wide-partition risk bounded. Postgres TOAST (rejected: vacuum cost at 30 GB/hr), Elasticsearch (rejected: cost-per-byte).
Rendezvous hashing Maps 256 logical shards to N schedulers with minimal disruption on rebalance — only ~1/N of shards move when a scheduler joins/leaves, vs ~half with naïve modulo; lets us add capacity without missing a 1 s trigger SLA during the rebalance window. Modulo hashing (rejected: massive churn on resize), consistent hashing (rejected: equivalent here, more code).
Optimistic CAS (version column) Lock-free claim-and-advance — losing schedulers retry without holding row locks, so a 256-partition scan completes in <100 ms even under contention; eliminates a lock manager, which is critical at 8 K writes/s shared across schedulers. SELECT FOR UPDATE (rejected: lock contention spikes), advisory locks (rejected: no fence token).
Monotonic clocks (Java System.nanoTime) Wall-clock skew across schedulers can mis-decide "due now"; monotonic clocks for elapsed measurements + a single authoritative wall-clock per scheduler at boot gives us the 1 s p99 trigger accuracy without trusting NTP sync. System.currentTimeMillis (rejected: NTP slew artifacts), TrueTime (rejected: cloud lock-in).
Token bucket (per-job, per-target) Bounded-rate backfill catches the user who paused 100 K jobs and resumed them at midnight; per-target limit at the dispatcher prevents a customer's webhook endpoint from being DDoS'd by their own backfill — same lever for the 00:00 thundering herd. Leaky bucket (rejected: less burst-friendly), no rate cap (rejected: customer outages).
Count-min sketch (per-tenant fairness) 10 M jobs across 100 K tenants: tracking exact per-tenant fire rates needs a hash map keyed on tenant; CMS gives the same fairness signal in <2 MB per scheduler with O(1) lookups, allowing weighted-fair queueing without a heavy counter store. Exact counters (rejected: memory + GC pressure), sampling (rejected: misses small tenants).
S3 (audit log archive) Run-log Cassandra rows roll to S3 hourly (30 GB/hr) for 7-year compliance retention; 100× cheaper than keeping it in Cassandra and the rare-read pattern (post-incident forensics) tolerates S3 latency easily. Cassandra long-term (rejected: cost), Glacier-only (rejected: investigation latency).
OpenTelemetry + Prometheus Trigger-accuracy SLA requires per-job latency histograms (fire-time minus scheduled-time); OTel traces follow a job from API to scheduler claim to dispatcher to webhook ack, so we can prove which hop ate the 1 s budget when SLA breaches. Logs only (rejected: no histograms), proprietary APM (rejected: lock-in).
