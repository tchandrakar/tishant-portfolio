Design Online Judge (LeetCode/HackerRank — sandboxed code execution)

Hard misc

1. Problem statement

Build a competitive programming platform that accepts user-submitted code in 15+ languages, compiles and executes it against hidden test cases inside a hardened sandbox, returns verdicts (AC, WA, TLE, MLE, RTE, CE) within seconds, and powers contests with thousands of concurrent participants. The system must be tamper-proof: malicious submissions cannot escape the sandbox, exhaust the host, or leak test data.

2. Functional requirements

  • Submit source code for a problem; receive a verdict + runtime/memory stats.
  • Support C, C++, Java, Python, Go, Rust, JS, Kotlin, Swift, C#, Ruby, PHP, Scala, Haskell, TypeScript.
  • Run against N test cases; first failure aborts (configurable).
  • Display sample I/O diff on Wrong Answer; hide hidden tests.
  • Live contest mode: leaderboard updates, frozen scoreboard last hour.
  • Plagiarism detection (MOSS-style fingerprinting) on contest submissions.

3. Non-functional requirements

  • Latency: p50 verdict in 2 s, p99 in 15 s (excluding TLE-bound problems).
  • Scale: 5 M users, 200 K daily submissions, peak 2 K submissions/sec during weekly contests.
  • Isolation: zero host compromise; every submission runs in a fresh micro-VM.
  • Availability: 99.9% (contests demote to read-only if judge fleet degrades).

4. Capacity estimation

  • QPS submit: 200K/86400 ≈ 2.3/s avg, 2,000/s peak (contest start).
  • Avg test cases per submission: 25; avg run 500 ms ⇒ 12.5 s CPU per submission.
  • Peak CPU demand: 2,000 × 12.5 = 25,000 CPU-sec/sec ⇒ 25 K cores; with 30 s queue tolerance and 5× burst absorbtion via queue, steady fleet ~5,000 cores (62 × 80-vCPU boxes).
  • Storage: source 5 KB avg × 200 K/day × 365 = 365 GB/yr; test data ~2 TB total (S3).
  • Bandwidth: stdout capped 64 KB/test ⇒ 5 MB/submission max ⇒ 1 GB/s peak egress to result store.

5. API design

POST /v1/submissions          {problemId, lang, source}     → submissionId
GET  /v1/submissions/{id}                                   → {verdict, runtime, memory, testsPassed}
GET  /v1/submissions/{id}/stream  (SSE)                     → progressive verdict per test
GET  /v1/contests/{id}/leaderboard?frozen=true              → ranked list
POST /v1/admin/problems       {statement, tests, limits}    → problemId

6. High-level architecture

[Web/IDE]──HTTPS──►[API GW]──►[Submission Svc]──►[Kafka: submissions] │ │ ▼ ▼ [Postgres] [Judge Workers (autoscale)] │ ┌───────────┼───────────┐ ▼ ▼ ▼ [Firecracker] [gVisor] [nsjail] │ ▼ [Test S3]──►[Result Topic]──►[Verdict Svc] │ ▼ [Leaderboard (Redis)]

Submission goes to Kafka; judge workers pull, fetch tests from S3, run inside an isolated micro-VM, publish per-test results, and the verdict service aggregates and updates Postgres + Redis sorted set for leaderboard.

7. Data model & storage

7.1 Submissions (Postgres)

CREATE TABLE submissions (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT, problem_id BIGINT, contest_id BIGINT,
  lang TEXT, source_s3 TEXT,
  verdict TEXT, runtime_ms INT, memory_kb INT,
  tests_passed INT, total_tests INT,
  submitted_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX ON submissions (contest_id, user_id, submitted_at);

7.2 Test cases (S3 + CDN)

Per-problem tarball at s3://judge-tests/p/{id}/v{n}.tar.zst; signed URLs minted per submission. Workers cache locally with LRU keyed by (problemId,version).

7.3 Leaderboard (Redis)

One sorted set per contest: ZADD contest:42 score user:7. ZREVRANGE for top-K. Per-user score recomputed on AC.

8. Detailed component design

8.1 Judge worker lifecycle

Worker pulls one job, fetches source + tests, spins a Firecracker microVM (boot ~125 ms), copies code via vsock, compiles inside the VM with stricter limits (mem 512 MB, time 30 s), then iterates tests: writes stdin, reads stdout, enforces wall/CPU time + RSS via cgroup v2. On verdict, snapshot is destroyed.

8.2 Sandboxing layers

Defense in depth: (1) Firecracker microVM (KVM hardware isolation), (2) seccomp-bpf allowlist (≤40 syscalls), (3) cgroups v2 caps (CPU, memory, pids, IO), (4) rootless user with read-only rootfs, (5) network namespace = none. For Python/JS where boot cost dominates, gVisor sandbox is reused (lighter, weaker).

8.3 Backpressure & queueing

Submission Kafka topic partitioned by userId for fairness; consumer group autoscales on lag (KEDA). A token-bucket on submission API throttles abusers (10 subs/min/user).

8.4 Plagiarism

Winnowing fingerprint (k-gram=5, w=4) stored in Postgres with GIN index on int8[]. Pairwise compare runs offline post-contest using count-min sketch to skip pairs below threshold — same sketch family as DDIA Ch. 11 cites for streaming top-K.

9. Scaling & bottlenecks

  • Cold-start: warm-pool of 200 idle Firecracker VMs per worker keeps p99 boot < 50 ms.
  • Hot problem at contest start: tests pre-pulled to every worker 5 min before contest.
  • Compiler cache: sccache + per-language base image with toolchain pre-installed.
  • Disk thrashing: tmpfs scratch, 256 MB cap, destroyed with VM.

12. Talking points for the interviewer

  • "I'd start with gVisor for everything to ship fast, then graduate C/C++ to Firecracker once we hit the first sandbox-escape report — defense in depth, not all at once."
  • "The hardest bit is fairness during a 100K-user contest start: token-bucket per user plus per-user Kafka partition keeps one fast solver from starving everyone."
  • "Test data is the real secret; signed S3 URLs minted only inside the worker, never reaching the user's browser."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Firecracker microVM KVM-backed isolation with ~125 ms boot keeps p99 verdict under 15 s while running 2 K untrusted submissions/sec; warm pool of 200 idle VMs per worker drops effective boot to <50 ms, far safer than container escape paths for C/C++. runc (rejected: weak syscall isolation), Kata Containers (rejected: 600 ms boot, heavier RAM).
gVisor User-space kernel intercepts syscalls cheaply (~10% overhead) for Python/JS where Firecracker's RAM cost dominates; reused across 100 execs per pod, matching our $0.0002/exec target on the 5 K-core fleet. Bare runc (rejected: no syscall layer), Nabla (rejected: language-runtime restrictive).
seccomp-bpf + cgroups v2 Allowlist of ≤40 syscalls plus memory.max=512M, pids.max=64, cpu.max caps stop fork bombs and kernel-attack surface inside the VM; enforced per-test so a 2 s TLE problem doesn't burn 30 s of worker CPU. AppArmor only (rejected: weaker syscall granularity), no kernel limits (rejected: forkbomb risk).
Apache Kafka Submission topic partitioned by userId (256 partitions) provides per-user ordering and absorbs the 2 K/s contest-start burst; KEDA autoscales judge consumers on lag, draining a 30 s queue before SLA breach. RabbitMQ (rejected: weaker partition semantics at 2 K/s), AWS SQS (rejected: no replay for re-judging).
PostgreSQL (submissions) 200 K submissions/day fit comfortably in a single primary; ACID matters for verdict + leaderboard recomputation correctness, and the (contest_id, user_id, submitted_at) index serves "my submissions" reads at <5 ms p99. Cassandra (rejected: cross-row contest queries awkward), MySQL (rejected: weaker JSON for limits config).
S3 + Zstd tarballs 2 TB of test data versioned per problem (v{n}.tar.zst) with signed URLs minted per submission; pre-pulled to all workers 5 min before contest to avoid 1 GB/s download cliff at start. NFS (rejected: bandwidth ceiling at scale), local SSD (rejected: cache invalidation pain).
Redis sorted sets (ZSET) One ZSET per contest gives O(log N) ZADD on AC and O(log N + K) ZREVRANGE for top-100 leaderboard at <5 ms p99; eventual freshness (~1 s) is acceptable per the leaderboard SLA. Postgres rank() (rejected: 50-100 ms for 100 K-row contests), Elasticsearch (rejected: overkill).
vsock (KVM) Source code and stdin transferred to the microVM over vsock instead of network namespace, sidestepping veth setup latency and keeping network=none guarantee — no packet ever reaches the VM's network stack. 9p mount (rejected: filesystem-level escape risk), virtio-net (rejected: violates network-isolation invariant).
sccache + per-language base images Compiler caching cuts repeated C++/Rust compile time from ~3 s to ~200 ms; pre-installed toolchain images amortize image-pull cost across the 200 K-submission daily volume. ccache (rejected: weaker remote cache), build-from-scratch (rejected: 10× compile latency).
Server-Sent Events (SSE) Per-test verdicts streamed progressively via SSE — natural fit for one-way server push, simpler than WebSockets, plays well with HTTP/2 multiplexing through the API gateway for the 2 K concurrent contest streams. WebSockets (rejected: bidirectional overkill), polling (rejected: misses the snappy UX win).
Count-min sketch (plagiarism) Pairwise winnowing fingerprint compare is O(n²) — 50 K contest submissions = 1.25 B pairs. CMS pre-filters candidate pairs with shared k-grams in fixed memory, cutting the heavy compare to <1% of pairs. Full pairwise (rejected: prohibitive), MinHash only (rejected: no frequency signal).
etcd / Kubernetes HPA + KEDA Judge worker fleet is bursty (2 K/s contest peak vs 2.3/s average); KEDA scales on Kafka consumer lag, hitting 5 K cores within 30 s of contest start without paying for idle capacity off-peak. Static fleet (rejected: 4× cost waste), AWS Lambda (rejected: no microVM control).
