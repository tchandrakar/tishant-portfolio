Design Pastebin (text + code paste with syntax highlight, expiry)

Medium misc file-storage

1. Problem statement

Build a service to publish text/code snippets up to 10 MB, addressable by a short ID, with optional syntax highlighting (~180 languages), expiry (10 min — never), visibility (public/unlisted/private), and a raw endpoint usable by curl. Pastebin.com peaks at ~17 M monthly pastes and roughly 1.5 B raw views/month.

2. Functional requirements

  • Create a paste with body, language hint, expiry, visibility, optional password.
  • Read by short ID — rendered HTML view and /raw/{id} endpoint.
  • Server-side syntax highlight rendering, cached.
  • Search by user, tag, language for logged-in users.
  • Spam/abuse pipeline: keyword scan + dedupe-by-hash flag for credential dumps.

3. Non-functional requirements

  • Latency: read p99 < 150 ms; write p99 < 400 ms incl. 10 MB body.
  • Scale: 17 M writes/month (~7 wps avg, 200 wps peak); 1.5 B reads/month (~580 rps avg, 8 K rps peak).
  • Availability: 99.9% reads, 99.5% writes.
  • Durability: 11×9s for non-expired pastes (S3 standard).

4. Capacity estimation

  • Avg paste 10 KB, p99 1 MB, max 10 MB. 17 M × 10 KB = 170 GB/month metadata-adjacent; bodies in object store.
  • 5-yr retention (paid tier) = ~10 TB body store.
  • Highlighted-HTML cache: 80% hit ratio over hot 5% of pastes → 50 GB Redis is enough.
  • Egress: 1.5 B reads × 15 KB rendered = ~22 TB/month outbound.

5. API design

POST /v1/pastes        {body, lang?, expiresIn?, visibility, password?}  → {id, url}
GET  /{id}                                                                → HTML render
GET  /raw/{id}                                                            → text/plain
DELETE /v1/pastes/{id}  Authorization: Bearer ...                         → 204
GET  /v1/users/{u}/pastes?cursor=                                         → list

6. High-level architecture

[Browser] --> [CDN] --> [API GW] --> [Paste Svc] --> [Postgres metadata] | | | +--> [S3 bodies] | +--> [Highlight Svc (Pygments/Tree-sitter)] | +--> [Redis rendered-HTML cache] | +--> [Kafka] --> [Abuse-scan worker] | +--> [Quarantine bucket]

Walk-through: the Paste Service writes metadata to Postgres and the body to S3 in a single transaction (write S3 first, then commit row referencing the object key). Render path: Postgres lookup, body fetch from S3 if not cached, Pygments highlight, store HTML in Redis with 1 h TTL.

7. Data model & storage

7.1 Postgres metadata

CREATE TABLE pastes (
  id            CHAR(8) PRIMARY KEY,        -- base62
  owner_id      BIGINT,
  s3_key        TEXT NOT NULL,
  lang          TEXT,
  size_bytes    INT,
  visibility    SMALLINT,                   -- 0 public, 1 unlisted, 2 private
  pw_hash       TEXT,
  created_at    TIMESTAMPTZ,
  expires_at    TIMESTAMPTZ                 -- partial index WHERE expires_at IS NOT NULL
);
CREATE INDEX ix_owner ON pastes(owner_id, created_at DESC);

7.2 Why split metadata + S3?

Postgres rows stay slim (~200 B), keeping the hot working set in RAM. S3 handles the long tail of 10 MB bodies cheaply with built-in lifecycle expiration. Storing big blobs in Postgres TOAST works initially but vacuum cost grows unbounded — see DDIA Ch. 3 on storage engines.

8. Detailed component design

8.1 ID generation

8-char base62 = 218 trillion IDs. Sample randomly from CSPRNG; collision rate negligible at our scale (birthday-bound after ~14 M unlisted-only pastes). Insert with ON CONFLICT DO NOTHING and retry on the rare miss.

8.2 Highlight pipeline

Tree-sitter parses lazily for the top-20 languages (covers 92% of traffic); fall back to Pygments for the long tail. Render output keyed by sha256(body) + lang — different pastes with identical content share one cache entry.

8.3 Expiry

Postgres partial index drives a sweeper that runs every 60 s, deletes rows past expires_at, and queues their s3_key for async S3 deletion. S3 lifecycle policy is the safety net (delete after expires + 7 days).

8.4 Private + password pastes

Argon2id over the password; never serve from CDN edge cache (force Cache-Control: private, no-store). Body decrypted server-side per request — paste-level keys wrapped by KMS.

9. Scaling & bottlenecks

  • Viral pastes (post on Hacker News): CDN with 60 s TTL absorbs the spike; one paste can do 100 K rps without touching origin.
  • Highlight CPU: 10 MB Rust file + Tree-sitter is ~600 ms; cap to 256 KB rendered preview, link to /raw for the rest.
  • Abuse: async hash-against-known-leaks (HIBP, GitHub secret scanner) within 5 min of write.
  • Postgres write hot spot: sequential ID would cause btree right-edge contention; random base62 spreads writes uniformly.

12. Talking points for the interviewer

  • "I deliberately do not server-render bodies > 256 KB inline — interviewer will probe TTFB, and partial-render + lazy-load is the answer."
  • "Hardest part is abuse: leaked credentials and phishing make pastebins a magnet. I'd outline the takedown SLO (15 min) and the human-in-the-loop queue."
  • "For multi-region, I'd start single-region with cross-region S3 replication and add read replicas later — strong consistency on writes is more valuable than 50 ms of EU latency."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
PostgreSQL (metadata)17 M pastes/month is small for a single Postgres primary; partial indexes drive the expiry sweeper efficiently and 200 B rows keep the working set in RAM, supporting 8 K rps peak read with sub-150 ms p99.DynamoDB (no partial indexes for expiry sweep); MySQL (worse partial-index support).
Amazon S3 (paste bodies)Bodies range 10 KB avg to 10 MB max; 11×9s durability, $0.023/GB-month pricing on ~10 TB of 5-yr retention, and built-in lifecycle rules give us the safety-net expiry deletion without app code.Postgres TOAST (vacuum cost grows unbounded past TB); GCS (similar but lock-in to AWS stack).
Random base62 IDs (8 char)62^8 = 218 trillion IDs gives birthday collisions only after ~14 M inserts; random distribution avoids B-tree right-edge contention that sequential IDs would cause at 200 wps peak.Snowflake (sequential, hot btree page); UUIDv4 (longer URLs, ugly to share).
Tree-sitter (top-20 langs)92% of traffic is JS/Python/Go/Rust/etc.; Tree-sitter's incremental parser highlights 100 KB in < 50 ms — fast enough to render inline without blocking the 150 ms read p99.highlight.js (browser-side, breaks raw /raw use case); ANTLR (slower, heavier).
Pygments (long-tail langs)Covers 180+ obscure languages where Tree-sitter has no grammar; only invoked off the hot path for the 8% tail, so the slower 600 ms render on 10 MB doesn't hurt aggregate latency.Rouge (Ruby-only ecosystem); chroma (Go port lacks coverage).
Redis (rendered HTML cache)80% hit ratio over hot 5% of pastes fits in 50 GB; serving cached HTML straight from Redis hits the < 150 ms read SLO and shields Pygments CPU from viral spikes.Memcached (no persistence, cold-start cost); CDN-only (rendered HTML changes on lang flip).
SHA-256 content keyRender cache keyed by sha256(body)+lang dedupes identical content across pastes — credential-dump copies share one cache entry, saving CPU during abuse spikes.Paste-id key (no dedup, wastes cache); xxHash (collisions a real risk at 14 M pastes).
Argon2id (password pastes)Memory-hard KDF resists GPU brute-force at the per-paste-password granularity; tuned to 100 ms cost stays under the 400 ms write p99 while making offline cracking economically infeasible.bcrypt (no memory hardness); PBKDF2 (GPU-vulnerable).
AWS KMS (envelope encrypt)Per-paste DEKs wrapped by KMS CMK give defense-in-depth for private pastes; one KMS call per fetch fits the 150 ms latency budget and rotation is centralized.Vault (extra ops burden); app-layer key store (key rotation nightmare).
Apache Kafka (abuse pipeline)Async hash-against-known-leaks needs durable replay if we add a new HIBP-style scanner months later; Kafka's 7-day retention plus replay gives that without re-reading 170 GB/month of bodies.SQS (no replay); RabbitMQ (no log retention semantics).
CDN (CloudFront/Fastly)One viral paste can do 100 K rps without touching origin; 60 s edge TTL on public pastes turns the HN-spike problem into a CDN problem, freeing origin for the long-tail.Origin-only (capacity blowup); longer TTL (breaks delete-now SLO).
HIBP / GitHub Secret ScannerReal-world pastebins are credential-dump magnets; integrating with HIBP and GitHub's secret-scanning service gives us a 5-min takedown SLO without building secret detection from scratch.In-house regex scanner (false-positive hell); manual review only (15-min SLO impossible).
