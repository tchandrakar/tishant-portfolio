Design IoT Data Pipeline (1B devices, MQTT → time-series store)

Very Hard misc analytics

1. Problem statement

Build a telemetry ingestion pipeline for 1 B connected devices (smart meters, vehicles, sensors) emitting one reading every 30 s on average over MQTT. Data lands in a time-series store for queries (last value, last 24 h, monthly aggregates) and is forked into a stream-processing tier for alerts. Comparable scale: AWS IoT Core (trillions of msgs/year), Tesla fleet telemetry, John Deere connected farm.

2. Functional requirements

  • Devices connect via MQTT 5 over TLS, publish to tenants/{t}/dev/{id}/....
  • Bidirectional: server can push commands (firmware update, config) to a device.
  • Per-device shadow (last-known state) accessible by REST.
  • Time-series store keeps 13 months at 1-min raw + 5-yr at 1-h rollups.
  • Real-time rules engine: trigger on threshold (e.g. temp > 80 °C) within 5 s.

3. Non-functional requirements

  • Scale: 1 B devices × 1 msg/30 s = ~33 M msg/s steady-state, ~80 M/s peak.
  • Latency: ingest-to-rule-eval < 5 s p99; shadow read < 50 ms.
  • Connection density: 100 K simultaneous TLS conns per broker pod, 10 K broker pods.
  • Durability: no data loss on broker failure; QoS 1 end-to-end.

4. Capacity estimation

  • 33 M msg/s × 200 B = 6.6 GB/s ingress = ~570 TB/day raw.
  • Compress 8× (numeric, repeated tags) → 70 TB/day, ~25 PB/yr at 13-month raw.
  • Rollups 1-min → 1-h (60×): keep 5 yr = ~2 PB.
  • Kafka cluster: 80 M/s × 1 KB × replication-3 = 240 GB/s → ~2400 brokers (100 MB/s each).
  • ~10 K MQTT broker pods (EMQX/HiveMQ); 100 K conn/pod = 1 B device capacity.

5. API design

# Device side (MQTT)
PUBLISH tenants/acme/dev/1234/telemetry  payload={ts,t,p,batt}  QoS=1
SUBSCRIBE tenants/acme/dev/1234/cmd                                     # for commands

# Cloud REST
GET  /v1/devices/{id}/shadow                              → {state,ts}
POST /v1/devices/{id}/cmd            {op,args}            → {jobId}
GET  /v1/query?dev=&metric=&from=&to=&agg=avg&step=1m      → series

6. High-level architecture

[Device] --MQTT/TLS--> [NLB] --> [MQTT broker pool] --> [Kafka: telemetry] | | v +--> [Flink: rules engine] --> [Alert svc] [Auth + ACL svc] +--> [Flink: rollups] --> [TSDB rollups] +--> [Spark: archival] --> [S3 Parquet] +--> [Shadow updater] --> [Cassandra shadow] +--> [TimescaleDB/Druid raw] [Browser] <-- [Query API] <-------------------------------------------+ [Backend] -- /cmd -- [Cmd svc] --> [Kafka: cmd] --> [Broker] --> [Device]

Walk-through: TLS-terminated NLBs spread connections across MQTT brokers. The broker authenticates the device cert, applies topic ACLs, and republishes payloads onto Kafka. Multiple Flink jobs consume the same Kafka topic — fan-out, not fan-in — so rules, rollups, archival, and shadow updates are independent.

7. Data model & storage

7.1 Time-series (Druid / TimescaleDB)

-- Timescale hypertable
CREATE TABLE readings (
  ts TIMESTAMPTZ, dev_id BIGINT, metric SMALLINT, value DOUBLE PRECISION
);
SELECT create_hypertable('readings','ts',
  chunk_time_interval => INTERVAL '1 hour',
  partitioning_column => 'dev_id', number_partitions => 256);

7.2 Device shadow (Cassandra)

CREATE TABLE shadow (
  dev_id BIGINT PRIMARY KEY, last_seen timestamp, state map<text,text>, version BIGINT
);

7.3 Why these stores?

Druid is unrivaled for scan-heavy time-series queries with high cardinality on dev_id. Cassandra's wide rows + LSM perfectly fit the "1 row per device, frequently overwritten last-state" pattern. S3 Parquet is the cold archive — cheap, scannable from Athena/Trino. See DDIA Ch. 6 on partitioning by composite (tenant, dev_id, time).

8. Detailed component design

8.1 MQTT broker tier

EMQX clustered with shared subscriptions. TLS session resumption keeps reconnect cost low. Each broker maintains an in-memory routing table; cross-broker delivery via internal gossip + Kafka backbone for QoS 1 durability.

8.2 Sharding key

Kafka partition = hash(tenant_id, dev_id) % 4096. Co-locates a device's traffic on one partition so rule state can be local. Avoids hot-shard with random suffix bucketing for a small set of mega-tenants.

8.3 Rules engine (Flink)

RocksDB-backed keyed state per device; checkpoints to S3 every 30 s. Watermark = max(event_time) − 60 s, so out-of-order packets from cellular networks still land in the right window. Materialized rules compiled to bytecode for < 100 µs/event eval.

8.4 Downlink commands

Cmd service writes to Kafka cmd topic; brokers subscribe and route to the connected broker holding the session. If the device is offline, persist in Cassandra and replay on reconnect (LWT for "claimed").

8.5 Backpressure

If Kafka lag > 30 s, brokers downgrade QoS to 0 for non-critical metrics and drop telemetry classes B/C — graceful degradation per the device contract.

9. Scaling & bottlenecks

  • Connection storms (post-outage reconnect): jittered reconnect on device, plus broker-side rate limit on auth (1 K/s/pod).
  • Cardinality explosion: 1 B dev_ids in TSDB tag indexes — partition by tenant, use roaring bitmaps (Druid).
  • Hot tenants: top 1% of customers can drive 50% of traffic — dedicated Kafka topics per VIP tenant + dedicated Flink jobs.
  • Cross-region replication: regional brokers and TSDB; cross-region only for global rollups via S3.

12. Talking points for the interviewer

  • "I'd defend MQTT over HTTP/2 explicitly: at 1 B devices, TCP/TLS overhead and persistent-conn semantics matter more than ecosystem familiarity."
  • "Cardinality is my biggest fear — the moment a customer adds a UUID tag per reading, Druid lookups fall over. I'd reject high-cardinality tags at the schema layer."
  • "For the rules engine, exactly-once is achievable inside Flink, but downstream alert delivery is at-least-once over the internet — same trap as the webhook problem."
