Design Real-time Leaderboard (gaming, top-K via Redis sorted set + sharding)

Hard misc storage-cache

1. Problem statement

Design a real-time leaderboard for a global mobile game with 50 M MAU. Players submit scores after each match; the system must show the global top-100, the player's rank, friends-leaderboard, and per-region/per-time-window views (daily, weekly, all-time) with sub-second latency at peak. Anti-cheat is part of the path: scores must be validated before counting.

2. Functional requirements

  • Submit score; idempotent per match.
  • Get top-K (typical K=100) globally, regionally, by time window.
  • Get my rank (exact for top-1K, approximate beyond).
  • Get friends leaderboard (intersect friend list with sorted set).
  • Tournament mode: time-boxed leaderboard with reset.

3. Non-functional requirements

  • Latency: top-K read p99 < 50 ms; rank-of-user p99 < 100 ms.
  • Scale: 50 M MAU, 5 M DAU, peak 100 K score-submits/sec during weekend events.
  • Freshness: top-K reflects writes within 1 s.
  • Availability: 99.95%; degrade to slightly-stale top-K under partition.

4. Capacity estimation

  • 50 M players × 64 B per ZSET entry ≈ 3.2 GB per leaderboard; 5 leaderboards (global, daily, weekly, monthly, tournament) ⇒ ~16 GB hot data.
  • Writes: 100 K/s × 5 ZSET updates = 500 K Redis ops/s ⇒ shard across 16 Redis nodes (~31 K ops/s each, well within 100 K cap).
  • Reads: 5 M DAU × 10 leaderboard views/day = 50 M/day ≈ 600/s avg, 10 K/s peak — easily served by replicas.
  • Long-term storage in Cassandra: 100 K writes/s × 200 B = 1.7 TB/day, TTL 90 days ⇒ ~150 TB rolling.

6. High-level architecture

[Game Client]──►[API GW]──►[Score Ingest Svc]──►[Anti-cheat]──►[Kafka: validated-scores] │ ┌──────────────────────────────┼──────────────┐ ▼ ▼ ▼ [Redis Cluster] [Cassandra (audit)] [Flink job] (16 shards, ZSET) (windowed agg) ▲ │ [Read API]──►[Replica nodes]

Score path: validate → publish → fan-out consumers update sharded Redis ZSETs and a Cassandra audit log. Reads go to Redis replicas; cross-shard top-K merged in the read API via a heap.

7. Data model & storage

7.1 Redis sorted sets

# Global all-time
ZADD lb:global  {score} {userId}
ZREVRANGE lb:global 0 99 WITHSCORES   # top-100
ZREVRANK   lb:global {userId}          # my rank

# Sharded global (16 shards)
ZADD lb:global:s{hash(userId) % 16}  {score} {userId}

# Time-windowed
ZADD lb:daily:2026-05-06   {score} {userId}    EX 172800

7.2 Audit (Cassandra)

CREATE TABLE score_events (
  user_id BIGINT, match_id UUID,
  score BIGINT, region TEXT, ts TIMESTAMP,
  PRIMARY KEY ((user_id), ts)
) WITH default_time_to_live = 7776000;

7.3 Why Redis ZSET?

Skip-list under the hood: O(log N) insert/rank/range. Cluster mode shards by key hash; replicas serve reads. Persistence via AOF every-second strikes the right durability/perf trade.

8. Detailed component design

8.1 Sharded top-K

Single global ZSET hits a Redis hot key under 100 K writes/s. We shard by hash(userId) % 16. To compute global top-100: query top-100 from each shard (16 × 100 = 1,600 entries), merge in a min-heap of size 100 in the read API. Cost: 16 parallel ZREVRANGE calls; p99 ≈ 5 ms with pipelining.

8.2 Approximate rank for the masses

Exact rank is O(log N) per query but loads Redis. For users beyond top-1K we maintain a per-leaderboard histogram (DDIA Ch. 11 sketch family — t-digest) of score distribution; rank = N × percentile lookup. Updated every 10 s via Flink.

8.3 Friends leaderboard

Friend list (≤ 500 typically) fetched from the social graph; ZSCORE lb:global {friendId} in a pipelined batch — 500 calls in one round trip ~3 ms. Sort in app layer.

8.4 Time-windowed leaderboards

Daily/weekly/monthly are separate ZSETs with Redis TTL. Tournaments use a key like lb:tour:{id} created at start, frozen at end, then snapshotted to Cassandra and deleted from Redis.

8.5 Anti-cheat

Score ingest service validates server-side replay token, checks score against per-game upper bound, and runs a CountMinSketch of recent-score-frequency to flag impossible streaks. Only validated events hit Kafka.

9. Scaling & bottlenecks

  • Celebrity score storm: a streamer's match doesn't fan-out (single user); not a hot key.
  • Tournament start: create the ZSET on all 16 shards eagerly; pre-warm replicas.
  • Cross-region: per-region ZSETs in regional Redis; global view stitched async by Flink and stored in a separate "global-rolling" key.
  • Cold start of replica: RDB snapshot replication; takes < 60 s for 3 GB shard.

12. Talking points for the interviewer

  • "The classic interview answer is a single ZSET; what they're testing is whether you spot the hot key at 100 K/s and shard."
  • "Exact rank for everyone is unnecessary — players past rank 1,000 don't care about the difference between 47,221 and 47,304. t-digest is the right hammer."
  • "For tournaments I'd freeze the final ZSET into Cassandra and serve historical leaderboards from there — Redis is for the hot present."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Redis sorted sets (ZSET) Skip-list-backed ZSET gives O(log N) ZADD/ZREVRANK and O(log N + K) ZREVRANGE for top-100 — directly hits the <50 ms p99 read SLA on a 50 M-player set; ZINCRBY for atomic score updates avoids any read-modify-write race at 100 K/s peak. Postgres + window function (rejected: 50-100 ms per rank query), Elasticsearch (rejected: ~300 ms for ranks).
Sharded ZSETs (16 shards) Single ZSET pegs a Redis shard at ~100 K ops/s — exactly our 100 K writes/s × 5 leaderboards = 500 K ops/s ceiling; hashing by userId % 16 spreads load to 31 K ops/s per shard with comfortable headroom. One ZSET (rejected: hot key meltdown), 64 shards (rejected: scatter-gather overhead).
Min-heap merge (top-K scatter-gather) Cross-shard top-100: pull top-100 from each of 16 shards (1,600 candidates), merge in a size-100 min-heap in the read API — O(1,600 log 100) ≈ 11 K comparisons; with pipelined ZREVRANGE the whole call is <5 ms p99. Full union sort (rejected: O(N log N) overkill), broker-side merge (rejected: no native Redis op).
t-digest (long-tail rank) Exact rank for player #4M loads Redis pointlessly; t-digest holds the score percentile distribution in ~5 KB and answers "your rank ≈ percentile × N" in O(1) — accurate to 0.1% for the bulk of users who don't care about #47,221 vs #47,304. Histogram (rejected: poor tail accuracy), exact ZREVRANK on ZSETs (rejected: load).
Apache Kafka Score events keyed by userId on a 256-partition topic decouple anti-cheat validation from ZSET updates; 100 K/s ingest absorbed into per-partition batches, replayable for tournament reconstruction without touching Cassandra audit log. Direct Redis writes (rejected: anti-cheat coupling), RabbitMQ (rejected: replay weakness).
Apache Flink (windowed agg) Daily/weekly leaderboards are tumbling-window aggregations over the score stream; Flink's keyed state + watermarks correctly handle late events from regional latency, and the cluster-aware top-K snapshots Cassandra at window close. Spark Streaming (rejected: micro-batch lag), Kafka Streams (rejected: weaker watermarking).
Cassandra (audit / cold store) 100 K writes/s × 200 B = 1.7 TB/day with 90-day TTL = 150 TB rolling — pure append-only, partitioned by user_id; LSM trees handle the write rate that would shred any B-tree, and frozen tournament leaderboards land here permanently. Postgres (rejected: vacuum at 1.7 TB/day), DynamoDB (rejected: cost at this volume).
Redis AOF (every-second fsync) Every-second AOF gives ≤1 s data loss on shard failure with negligible perf hit — the right durability point for a leaderboard where a missed score is recoverable from Cassandra; full fsync-per-write would halve our 100 K ops/s ceiling. RDB only (rejected: minute-level loss), AOF always (rejected: 50% throughput loss).
Redis replicas (read scaling) 5 M DAU × 10 reads/day ≈ 600/s avg, 10 K/s peak — replicas eat this whole load while the primary stays free for the 100 K/s write path; ≤1 s replication lag is acceptable per our 1 s freshness SLA. Primary-only reads (rejected: write-side starvation), DAX (rejected: not Redis).
Count-min sketch (anti-cheat) Tracks recent-score frequency per user/match in <1 MB without per-user maps; flags impossible streaks (e.g., 50 perfect scores in 60 s) before scores reach the ZSET — keeps the cheat-suppression hot path off Cassandra. Per-user counters (rejected: 50 M-key memory blow-up), no anti-cheat (rejected: leaderboard credibility).
HyperLogLog (unique players) Counting unique active players per leaderboard at <0.81% error in 12 KB — order of magnitude cheaper than a SET of 50 M userIds, used for "5 M people are competing" UI without any per-user storage. Redis SET (rejected: GBs of memory), Bloom filter (rejected: false positives skew counts).
Pipelined ZSCORE (friends LB) 500 friends × ZSCORE in one Redis pipeline round-trip = ~3 ms vs 500 ms serially; small enough that no pre-aggregation is needed, big enough that naïve per-call latency would blow the SLA. Per-friend GET (rejected: round-trip storm), pre-materialized friends ZSET (rejected: write amplification).
Redis TTL (time-windowed keys) Daily keys lb:daily:2026-05-06 EX 172800 auto-expire 48 hr later — no cleanup cron, no orphaned keys; the 2-day overlap covers timezone reporters who want "yesterday" for their region. Manual sweep (rejected: cron drift), no expiry (rejected: unbounded RAM growth).
Per-region Redis clusters Cross-region writes at 100 K/s would saturate inter-region links; regional clusters localise the hot path and Flink stitches a global "rolling" view async — accepting eventual consistency in exchange for keeping regional p99 <50 ms. Single global cluster (rejected: latency + bandwidth), CRDT replication (rejected: ZSET CRDTs immature).
