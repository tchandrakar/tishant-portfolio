Design Twitter Trending Topics (count-min sketch, top-K streaming)

Hard misc social-network analytics

1. Problem statement

Compute the top-50 trending topics (hashtags, n-grams, named entities) globally and per-region every minute, from a firehose of 500 K tweets/sec, using a small fixed memory footprint and no full per-key counters. Trends should reflect velocity (recent spikes), not just absolute volume; spam & bot-generated trends must be suppressed.

2. Functional requirements

  • Global top-K trends, refreshed every 60 s.
  • Per-region top-K (200 regions), per-locale, per-time-window (5 min, 1 hr, 24 hr).
  • Personalised trends (filter by who-you-follow's network).
  • Spike detection: a hashtag jumping from 100/min to 50,000/min should bubble up immediately.
  • Spam suppression: bot rings, repeated identical tweets, suspended-account amplification removed.

3. Non-functional requirements

  • Latency: end-to-end tweet-to-trend < 90 s p99.
  • Scale: 500 K tweets/sec sustained, 1.5 M/sec peak (world cup final).
  • Memory: trend computation must run in < 4 GB RAM per worker (no full key map).
  • Accuracy: top-50 set match with exact ground truth ≥ 95% across 1 hr windows.

4. Capacity estimation

  • 500 K tweets/s × 5 candidate terms each = 2.5 M counter-bumps/sec.
  • Exact counters: 50 M unique terms/day × 16 B = 800 MB; with bigrams + trigrams ⇒ 10–50 GB. Sketches reduce to ≤ 200 MB per worker.
  • Kafka tweet topic: 500 K × 1 KB = 500 MB/s ingress, 256 partitions.
  • Output: 200 regions × 50 trends × 60 refreshes/hr = 600 K rows/hr to Cassandra (~150 GB/yr).

5. API design

GET /v1/trends/global?window=5m&limit=50          → [{term, score, deltaPct}]
GET /v1/trends/region/{geo}?window=1h&limit=50    → [...]
GET /v1/trends/personalized?userId=...             → [...]
INTERNAL: trend-pipeline consumes Kafka tweets, emits trend-snapshots topic.

6. High-level architecture

[Tweet Ingest]──►[Kafka: tweets (256 partitions)] │ ▼ [Spam Filter (Flink)]──drops──►[bot-events] │ ▼ [Tokenizer + NER (Flink)] │ ┌─────────┼─────────┐ ▼ ▼ ▼ [CMS Worker] [CMS Worker] ... (per region keyed) │ │ │ └────►[Heavy Hitter Top-K (Space-Saving)] │ ▼ [Trend Aggregator]──►[Cassandra: trend_snapshots] │ ▼ [Cache (Redis) ←── Read API]

Flink consumes the tweet topic, tokenizes into terms (hashtags, hashtag-less n-grams, named entities), updates per-window count-min sketch + Space-Saving top-K, then aggregator merges shards into final ranked list.

7. Data model & storage

7.1 Trend snapshots (Cassandra)

CREATE TABLE trend_snapshots (
  scope text,           -- "global" | "region:US-CA"
  window text,          -- "5m" | "1h"
  ts_bucket bigint,
  rank int,
  term text, score double, delta_pct double,
  PRIMARY KEY ((scope, window), ts_bucket, rank)
) WITH CLUSTERING ORDER BY (ts_bucket DESC, rank ASC);

7.2 Sketch state (Flink RocksDB)

Per (scope, window) keyed state holds a count-min sketch (depth=4, width=2^17 → 524 KB) plus a Space-Saving top-K (k=500, ~30 KB). Total ~600 KB per scope × 200 scopes × 4 windows ≈ 480 MB — fits comfortably.

7.3 Cache (Redis)

Top-50 per scope cached as a JSON blob with 60 s TTL; read API never hits Cassandra on cache hit.

8. Detailed component design

8.1 Probabilistic top-K

Count-min sketch supplies frequency estimate for any term in O(d) hashing; Space-Saving (Misra-Gries) maintains the actual top-K candidates with bounded error. Hybrid: every term increments CMS; if its CMS estimate exceeds the current K-th element of Space-Saving, it enters/replaces. This is the canonical sketch combo from DDIA Ch. 11's stream-processing notes on heavy-hitter detection.

8.2 Sliding windows

Tumbling 60 s sub-windows; a 5 min window is the union of 5 sub-windows. Each sub-window gets its own CMS; on advance, oldest is dropped (no cell-by-cell decay). For 1 hr window: 12 × 5 min snapshots merged on read.

8.3 Spike detection (z-score)

For each top-K candidate, compute baseline = EWMA over last 60 minutes of per-minute counts. spikeScore = (currentMinute − baseline) / stddev. Trends sorted by 0.6 × volumeRank + 0.4 × spikeRank. This catches "X just got mentioned 50× normal" even when total count is modest.

8.4 Spam & bot suppression

Pre-pipeline filter: drops tweets from suspended accounts, near-duplicate text (SimHash within 3 bits), accounts with < 24 h age + > 100 tweets/day. Per-term unique-author HyperLogLog: a hashtag with 100 K mentions but only 200 unique authors is demoted (bot ring).

8.5 Personalised trends

Per-user trends are too expensive to compute live (300 M users). We compute per-cluster (1,000 follower clusters from offline clustering) and assign each user to their cluster; live read is a single cluster's top-K.

9. Scaling & bottlenecks

  • Skewed partitions: mega-event hashtag floods one partition. Fix: secondary salt + map-side combine before shuffle.
  • Late events: Flink watermark with 30 s allowed lateness; later events dropped from current trend.
  • Cold start of new region: bootstrap from last hour replay (Kafka log compaction off; retention 24 h).
  • Cassandra wide row: ts_bucket clustering keeps partition bounded; older snapshots TTL'd at 30 days.

12. Talking points for the interviewer

  • "This problem is a classic excuse to talk sketches: CMS for frequency, Space-Saving for top-K, HLL for unique authors. Each saves an order of magnitude of RAM."
  • "Volume alone is wrong — 'iPhone' would always trend. The interesting metric is z-score against the term's own baseline."
  • "Spam is half the work. Without bot suppression you'll trend 'win-iPhone-RT-now' every hour."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Count-min sketch (CMS) Per-region sketch with depth=4, width=2^17 = 524 KB holds frequency estimates for tens of millions of terms — exact counters would cost 10–50 GB per worker; CMS keeps memory under our 4 GB/worker NFR while preserving the heavy-hitter signal. Exact hash map (rejected: 50 GB/worker), Bloom filter (rejected: presence-only, no frequency).
Space-Saving / Misra-Gries top-K Maintains k=500 candidate top-K with bounded error proportional to the (k+1)-th item frequency — the canonical pairing with CMS to fix CMS's over-estimation of cold keys; ~30 KB total state per scope. Heavy Keeper (rejected: more complex, marginal accuracy gain), full sort (rejected: not streaming).
HyperLogLog (unique authors) Per-term unique-author counter detects bot rings: a hashtag with 100 K mentions but only 200 unique authors is demoted; HLL gives 0.81% error in 12 KB instead of a SET of millions of userIds — vital at our 500 K tweet/sec rate. Redis SET (rejected: GB-level memory per term), exact dedupe (rejected: prohibitive at 50 M unique terms/day).
Apache Flink Native event-time semantics with watermarks (30 s allowed lateness) handle the firehose's out-of-order tweets correctly; keyed state in RocksDB persists the 600 KB-per-scope sketches with checkpoint-based exactly-once recovery on worker death. Spark Structured Streaming (rejected: micro-batch latency > 90 s p99 budget), Kafka Streams (rejected: weaker watermarks).
Apache Kafka (256 partitions) 500 K tweets/s × 1 KB = 500 MB/s ingress — 256 partitions give per-partition load ~2 MB/s, well within broker comfort; replayable retention (24 h) enables bootstrap of new regions from log without re-ingesting from origin. Kinesis (rejected: shard cost at 500 MB/s), RabbitMQ (rejected: not partitioned for this volume).
RocksDB (Flink state backend) Per-(scope, window) keyed state across 200 regions × 4 windows = 800 keyed entries; RocksDB's LSM keeps memory-mapped active state hot and pushes cold to local SSD — required to checkpoint 480 MB total state to S3 within Flink's 30 s checkpoint interval. Heap-based state (rejected: GC pauses on 480 MB), Redis (rejected: extra hop, no checkpoint).
Tumbling sub-windows (60 s) 5 min window = union of 5 sub-windows, 1 hr = 12 × 5 min — drops oldest sub-window cleanly on advance with no cell-by-cell decay math; matches the 60 s refresh SLA exactly and keeps state size bounded per scope. Exponential decay (rejected: implementation complexity, harder to reason about), session windows (rejected: no fit).
EWMA + z-score (spike detection) Baseline = EWMA over last 60 minutes of per-minute counts; spikeScore = (current − baseline) / stddev catches a hashtag jumping 100/min → 50 K/min within 60 s — pure volume ranks would always elect "iPhone". Absolute volume (rejected: stale-celebrity bias), naïve ratio (rejected: numerical instability).
SimHash (near-duplicate filter) SimHash within 3-bit Hamming distance flags copy-paste spam tweets pre-pipeline; 64-bit fingerprint computed in microseconds, far cheaper than full text comparison at 500 K tweets/s — drops the obvious bot amplification before sketches are touched. MinHash + LSH (rejected: heavier per-tweet cost), exact dedup (rejected: 500 K/s lookup).
NER pipeline (tokenizer) Hashtags alone miss "Lionel Messi"; lightweight on-device named-entity recogniser extracts up to 5 candidate terms per tweet, feeding 2.5 M counter-bumps/sec into the sketch — calibrated so NER cost stays under 1 ms/tweet. Hashtags only (rejected: misses entities), heavyweight ML NER (rejected: blows latency budget).
Cassandra (trend snapshots) 200 regions × 4 windows × 60 refreshes/hr × 50 trends ≈ 600 K rows/hr — partition key (scope, window) with ts_bucket clustering keeps wide partitions bounded; 30-day TTL handles retention without manual cleanup. Postgres (rejected: write rate), Elasticsearch (rejected: cost-per-write at this volume).
Redis (read cache, 60 s TTL) Top-50 per scope cached as JSON with 60 s TTL matching the refresh interval — read API hits Redis 99% of the time, keeping Cassandra free for ingestion writes; cache miss falls back to last snapshot in Cassandra at <20 ms p99. App-level cache (rejected: per-instance staleness), CDN (rejected: 1 min TTL too coarse).
Salt key + map-side combine World Cup hashtags would peg one Flink partition; salting the term key with a 0–15 prefix spreads load across 16 sub-keys, then combines them in a second stage — keeps no partition above 95% CPU during the 1.5 M tweet/s burst. No mitigation (rejected: backpressure on hot keys), random partition (rejected: loses keyed state benefit).
Avro + Schema Registry Tweet schema evolves (new bot-flag fields, attestation columns) — Avro's backward-compatible evolution keeps consumer Flink jobs running through schema bumps without redeploy, critical at 500 K msg/s where consumer downtime quickly becomes data loss. JSON (rejected: 4× size, no schema enforcement), Protobuf (rejected: weaker schema-evolution tooling).
