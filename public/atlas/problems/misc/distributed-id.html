Design Distributed ID Generator (Snowflake, ULID, K-sortable)

Hard misc database-ddia

1. Problem statement

Build a service that issues globally-unique 64-bit (or 128-bit) IDs across thousands of producer hosts in multiple data centers, with no coordination on the hot path. The IDs must be roughly time-sortable so that B-tree-based indexes append on the right edge instead of fragmenting. Twitter Snowflake (10 K IDs/ms/worker, 1024 workers) and Instagram's sharded counters are the canonical references; we want both.

2. Functional requirements

  • Generate 64-bit IDs at > 1 M/sec aggregate, with optional 128-bit ULID variant.
  • K-sortable: monotonic per worker; globally near-monotonic.
  • Embed metadata: timestamp, worker, sequence (so we can shard by ID).
  • Survive worker restarts, clock skew, leap seconds without duplicates.
  • Optional sidecar library (in-process) and standalone gRPC service.

3. Non-functional requirements

  • Latency: in-process < 200 ns; gRPC sidecar p99 < 1 ms.
  • Throughput: 4096 IDs/ms/worker, 1024 workers/region → 4 B IDs/sec/region.
  • Uniqueness: mathematically impossible duplicates within the configured epoch (~69 yr).
  • Availability: 99.999% on the hot path — dependency for every write in the platform.

4. Capacity estimation

  • Bit budget: 41 timestamp ms (~69 yr) + 10 worker (1024) + 12 seq (4096/ms) = 63 + sign.
  • At 4 K IDs/ms × 1024 workers = 4.1 B/sec. One region exhausts 64-bit space in ~70 yr — fine.
  • ZooKeeper for worker-id leases: ~1024 znodes, < 1 MB total state.
  • NTP-disciplined clocks; PTP in racks for ±100 µs.

6. High-level architecture

+---------------------+ [App pod] --rpc--> | Snowflake sidecar | <-- ZooKeeper /workers/{region}/{id} +----------+----------+ | | (in-process lib) [App] --link-time--> libsf.so --+ | v 64 bits = [ 1 sign | 41 ts-ms | 10 wkr | 12 seq ] ^ | NTP/PTP --> monotonic clock guard --> refuse to issue if clock went backward

Walk-through: each worker boots, leases an unused worker-id from ZooKeeper (ephemeral znode); the lease lasts the worker lifetime. Every call advances the millisecond timestamp; if the same ms, increment the 12-bit seq; if seq overflows, busy-wait to next ms. If wall clock moves backward, refuse to issue (or stall) until clock catches up.

7.2 ULID variant (128-bit)

128 bits = [ 48 ts-ms (Unix epoch, ~10889 AD) | 80 random ]
Crockford-base32 encoding -> 26 chars, sortable lexically.

7.3 Why k-sortable?

B-tree primary keys (Postgres, MySQL) suffer page splits and write amplification when keys are random (UUIDv4). K-sortable IDs keep inserts on the right edge — see DDIA Ch. 3, "Comparing B-Trees and LSM-Trees", and Ch. 6 on partitioning by key range vs hash.

8. Detailed component design

8.1 Clock-skew defense

On boot, sync to NTP and refuse to start if offset > 50 ms from peer median. At runtime, monotonic clock drives the timestamp — wall clock is read once per ms and clamped: ts = max(last_ts, now). If now < last_ts by > 5 ms, return error and let upstream fail-fast.

8.2 Worker-id leasing

ZooKeeper SEQUENTIAL+EPHEMERAL ensures unique IDs without manual config. On session expiry, ID is reclaimable after a 30 s TTL safety wait (longer than the 41-bit ms granularity allows the prior owner to drift).

8.3 Batch issue

Sidecar serves NextID(count=100) in one RPC: pre-allocate a 100-wide seq range under the same ms (or roll over). Slashes RPC overhead 100× for bulk inserters (Kafka producers, ETL).

8.4 Alternatives

UUIDv7 (RFC 9562) is essentially Snowflake without the worker bits — relies on entropy for uniqueness. Sonyflake uses 39-bit ts/sec for longer life; MongoDB ObjectId uses 96 bits with machine + counter. Choose based on whether you control all generators.

9. Scaling & bottlenecks

  • Sequence exhaustion in 1 ms: rare in practice; busy-wait is cheap. Go beyond 4096/ms by widening seq bits at the cost of ts range.
  • ZooKeeper as SPOF: 5-node ensemble in each region; in-process clients cache the lease — ZK can be down for hours and existing workers keep issuing.
  • Cross-region monotonicity: not guaranteed — only per-worker. Documented contract; consumers do not assume global ordering.
  • Leap seconds: use CLOCK_TAI or smear (Google-style) so wall clock never goes backward.
