Design Code Execution Sandbox (Docker + cgroups + seccomp, multi-language)

Hard misc

1. Problem statement

Design a multi-tenant remote code execution service that other products (interview platforms, IDEs, AI agents, online judges) call to compile and run untrusted user code in 20+ languages with strict CPU/memory/time/network limits. Returns stdout, stderr, exit code, runtime, peak memory. Must withstand fork bombs, kernel exploits, network exfiltration, and noisy neighbors.

2. Functional requirements

  • Synchronous run: POST source + stdin → response with stdout/stderr/exit/timing.
  • Streaming run: WebSocket REPL session, bidirectional stdin/stdout.
  • Asynchronous batch: queue N jobs with priority, deliver via webhook.
  • Per-tenant quotas: CPU-seconds/day, concurrent slots, memory ceiling.
  • Pluggable language runtimes via versioned base images.

3. Non-functional requirements

  • Latency: p50 cold start < 200 ms, warm start < 30 ms; total p99 sync < 3 s.
  • Scale: 10 K rps sustained, 50 K burst; 1 B executions/month.
  • Isolation: no host kernel CVE escape (assume best-effort), no cross-tenant leakage of fs/network/CPU cache.
  • Cost: < $0.0002/execution at steady state.

4. Capacity estimation

  • 10 K rps × avg 800 ms wall = 8 K concurrent slots; with 1 slot per vCPU, ~8 K vCPU = 100 × 80-vCPU bare-metal hosts.
  • Storage: 1 B execs × 4 KB stored output = 4 TB/month (S3 lifecycle to glacier after 30d).
  • Image registry: 20 langs × 5 versions × 600 MB = 60 GB; cached on every host.
  • Egress per execution capped at 1 MB; aggregate 10 K × 1 MB = 10 GB/s peak (egress-disabled by default).

5. API design

POST /v1/exec        {lang, source, stdin, timeoutMs, memoryMb}  → {stdout, stderr, exit, runMs, peakKb}
POST /v1/exec/async  {...}                                       → executionId
GET  /v1/exec/{id}                                               → status + result
WS   /v1/exec/stream {lang, source}                              → frames {type:"stdout"|"stderr"|"exit", data}

6. High-level architecture

[Client]──HTTPS──►[API GW]──►[Scheduler]──┬──►[Warm Pool: gVisor pods] │ │ │ ▼ │ [Sandbox Runner]──►[cgroup v2 + seccomp + netns] ▼ [Quota Service]──►[Redis] │ ▼ [Kafka: exec-events]──►[Billing/Audit]

Scheduler picks a warm pod from a same-language pool, mounts the source via tmpfs, kicks off run via gRPC to in-pod runner, streams output back. Pod is killed after one execution (or N for warm reuse with strict reset).

7. Data model & storage

7.1 Execution record (Cassandra)

CREATE TABLE executions (
  tenant_id text, exec_id timeuuid,
  lang text, source_hash text,
  exit int, run_ms int, peak_kb int,
  output_s3 text, created_at timestamp,
  PRIMARY KEY ((tenant_id), exec_id)
) WITH CLUSTERING ORDER BY (exec_id DESC);

7.2 Quotas (Redis)

Hash per tenant: HINCRBY quota:{tenant} cpu_ms 850; sliding window via Redis Streams. Hard caps short-circuit at API GW.

7.3 Output blobs

stdout/stderr larger than 4 KB pushed to S3; small outputs inline in Cassandra row to skip S3 round trip.

8. Detailed component design

8.1 Sandbox layers

(1) gVisor (user-space kernel) intercepts syscalls — first line. (2) seccomp-bpf allowlist further strips dangerous syscalls (ptrace, kexec, bpf, userfaultfd). (3) cgroups v2 caps: cpu.max=100000 100000, memory.max=256M, pids.max=64, io.max on tmpfs only. (4) network namespace with no veth (no internet). (5) read-only rootfs; per-exec ephemeral tmpfs at /work. (6) AppArmor profile blocks mount/ptrace.

8.2 Warm pool & pod recycling

Per language, pre-baked Kubernetes pods sit idle with the runtime loaded (JVM warmed via CRaC, Python interpreter pre-imported). After each exec, pod state is reset by truncating tmpfs and signaling runner; after 100 execs the pod is destroyed to bound any state leak.

8.3 Streaming runner

gRPC server inside the pod exposes Run(stream); backpressure via gRPC flow control. stdout is line-chunked, base64-frame over WS to client, throttled to 1 MB/s.

8.4 Multi-tenant fairness

Weighted fair queueing on the scheduler: each tenant has a virtual clock advanced by CPU-ms consumed; lowest virtual clock dequeued next. Prevents one tenant exhausting the warm pool.

9. Scaling & bottlenecks

  • Image pull storm: p2p image distribution (Dragonfly/Spegel) on cluster bootstrap.
  • JVM cold start: CRaC/CDS snapshots cut Java first-exec from 600 ms to 80 ms.
  • Kernel CVE blast radius: per-tenant node pools for high-paying customers; co-tenancy disabled on demand.
  • Forkbomb: pids.max=64 + 30 s wall timer SIGKILL.

  • "The thing that bites in prod isn't sandbox escape — it's noisy neighbors burning CPU cache. CPU pinning + L3 cache partitioning (Intel CAT) is a Phase-2 lever."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Docker (OCI runtime) Standardised image format for the 20 langs × 5 versions × 600 MB = 60 GB registry; layer caching means base+toolchain only pulled once per host, then per-version diffs stay under 50 MB for fast warm-pool refresh across 100 hosts. LXC (rejected: less ecosystem), Podman (rejected: minor delta, weaker K8s integration).
gVisor (runsc) User-space kernel intercepts every syscall — first defence at 10 K rps where Linux kernel CVEs are the real risk; ~10% syscall overhead is acceptable since we're CPU-bound, not syscall-bound, for typical 800 ms execs. runc (rejected: shared kernel = direct CVE blast radius), Kata (rejected: 600 ms boot blows the 200 ms cold-start SLA).
cgroups v2 Unified hierarchy enforces cpu.max=100000 100000, memory.max=256M, pids.max=64, io.max on tmpfs — single source of truth that survives runtime upgrades and gives accurate CPU-ms billing per tenant. cgroups v1 (rejected: split controllers, race conditions), rlimits only (rejected: no IO/PID caps).
seccomp-bpf Allowlist strips ptrace, kexec, bpf, userfaultfd — the syscalls behind most kernel-escape CVEs; runs in-kernel at near-zero overhead, layered atop gVisor for defence-in-depth at the 1 B exec/month volume. SELinux (rejected: distro-specific), no seccomp (rejected: gVisor alone leaves attack surface).
Kubernetes warm pods Per-language pre-baked pods sit idle with the runtime loaded; warm start is 30 ms vs 200 ms cold — critical for the <3 s p99 sync SLA. Pod recycling after 100 execs bounds state leak without paying full cold cost every time. Lambda/Firecracker pool (rejected: no language-runtime warm), VM pool (rejected: 10× RAM).
Apache Cassandra 1 B execs/month = ~400 writes/s sustained, append-only with TTL — Cassandra's LSM tree is ideal; partition by tenant_id with timeuuid clustering keeps recent execs in fewer SSTables, and per-tenant TTL prevents wide partitions for whales. Postgres (rejected: vacuum overhead at this volume), DynamoDB (rejected: cost at 4 TB/month).
Redis (quotas) HINCRBY on quota:{tenant} cpu_ms at 10 K rps with sub-ms latency; sliding-window via Redis Streams gives accurate per-second rate limits without flushing to disk for every check at API gateway. Postgres counters (rejected: 100× slower per HINCRBY), in-memory leaky bucket (rejected: not multi-region).
S3 + Cassandra inline blobs Outputs >4 KB go to S3 (4 TB/month, lifecycle to Glacier at 30d); <4 KB inline in Cassandra row to skip the S3 round-trip — important since 60% of execs produce trivial output and hit the <3 s p99 budget. S3-only (rejected: extra hop hurts p99), Cassandra-only (rejected: blob bloat at 4 TB).
tmpfs (per-exec scratch) Bounded RAM-backed FS at /work destroyed with the pod — zero persistence leak, sub-ms write latency vs disk; the 256 MB cap prevents one exec from starving the warm pool through page-cache pressure. overlayfs on disk (rejected: persistence cleanup race), bind mount (rejected: cross-tenant leakage).
gRPC + flow control In-pod runner exposes Run(stream); gRPC's HTTP/2 backpressure cleanly throttles a chatty exec producing 100 MB/s stdout down to the 1 MB/s WS frame budget without buffering it all in pod memory. Plain TCP (rejected: backpressure DIY), HTTP/1.1 streaming (rejected: head-of-line blocking).
CRaC / CDS snapshots JVM cold start drops from 600 ms to 80 ms by deserialising a pre-warmed heap — direct hit on the <200 ms cold SLA for Java, which is 30% of submitted languages and otherwise dominates p99. Plain JVM (rejected: 600 ms blows SLA), GraalVM native-image (rejected: lang-feature regressions).
Dragonfly / Spegel (p2p image distribution) Cluster bootstrap of 100 hosts × 60 GB images = 6 TB cold pull; p2p amortises that across hosts, avoiding the registry storm that would otherwise rate-limit deploys at 10 Gbps egress per registry node. Single registry (rejected: bottleneck at scale), pre-baked AMIs (rejected: 5 min update lag).
Apache Kafka (exec-events) Idempotent producer keyed on tenant:exec_id gives effectively-once delivery to the billing/audit pipelines off the 10 K rps hot path; replayable for fraud investigation without re-running execs. SQS (rejected: no replay), in-process pub/sub (rejected: lost on crash).
AppArmor profiles Blocks mount, ptrace, raw filesystem ops at the LSM layer — a third deny-list pass after gVisor and seccomp; cheap to enforce and catches profile-violating syscalls that slip past userspace kernel emulation. SELinux (rejected: policy authoring overhead), no MAC (rejected: weaker isolation).
