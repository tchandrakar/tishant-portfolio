Design Webhook Delivery Service (at-least-once, retry, signing)

Hard misc notifications

1. Problem statement

Build the platform-as-a-service component that delivers webhook events from our application (e.g. Stripe-like payment events, GitHub push events) to customer-controlled HTTPS endpoints. Stripe delivers ~5 B webhook attempts/month with at-least-once semantics, exponential backoff over 3 days, HMAC signatures, and replay protection. Customer endpoints fail constantly — we must isolate slow/broken endpoints from healthy ones.

2. Functional requirements

  • Customers register endpoints, choose event types, rotate signing secrets.
  • For each platform event, fan out to all matching subscriptions.
  • HMAC-SHA256 sign each request; include timestamp + idempotency key.
  • Retry with exponential backoff up to 72 h on 5xx/timeout.
  • Dashboard: per-event delivery log, manual replay, dead-letter inspection.

3. Non-functional requirements

  • Latency: first attempt p99 < 10 s after event publish.
  • Throughput: 5 B attempts/month → ~2 K rps avg, 50 K rps peak.
  • Durability: never drop an event before exhausting retry policy.
  • Isolation: a slow endpoint (10 s response) must not slow others.

4. Capacity estimation

  • 1 B events/month × avg 5 subscriptions = 5 B attempts; first-attempt success ~95% → 250 M retries/month.
  • Event payload ~2 KB; queue storage = 5 B × 2 KB × 3 day TTL = ~30 TB.
  • ~5 M outbound HTTPS connections in flight peak; ~10 K worker pods (500 conn/pod).
  • DB: 50 M active subscriptions, 200 B/row → 10 GB; trivial.

6. High-level architecture

[Producers] --> [Kafka events] --> [Fan-out worker] --> [Per-endpoint queue (Redis Streams)] | | v v [Subscription DB] [Delivery worker pool] | v [Customer HTTPS endpoint] | v (on failure) [Retry scheduler (Redis ZSET, score=next_attempt_ts)] | v (after N retries) [DLQ + dashboard]

Walk-through: Kafka is the event spine. A fan-out worker reads each event, looks up matching subscriptions, and pushes one task per endpoint into a per-endpoint queue. Delivery workers pull, sign, and POST. Failures push to a delayed retry ZSET keyed by the next-attempt epoch. A scheduler thread pops due tasks every second.

7. Data model & storage

7.1 Subscriptions (Postgres)

CREATE TABLE endpoints (
  id UUID PRIMARY KEY, account_id UUID, url TEXT, secret_v INT, secret BYTEA,
  event_filter TEXT[], created_at TIMESTAMPTZ, disabled_at TIMESTAMPTZ
);
CREATE INDEX ix_account ON endpoints(account_id) WHERE disabled_at IS NULL;

7.2 Attempt log (Cassandra)

CREATE TABLE attempts (
  endpoint_id uuid, msg_id timeuuid,
  attempt_n int, status_code int, response_excerpt text,
  attempted_at timestamp, next_attempt_at timestamp,
  PRIMARY KEY (endpoint_id, msg_id, attempt_n)
) WITH CLUSTERING ORDER BY (msg_id DESC, attempt_n ASC);

7.3 Why split?

Subscription metadata is small + relational; attempts are append-heavy time-series — Cassandra's LSM tree handles 50 K writes/sec/node easily. See DDIA Ch. 3 (LSM vs B-tree) and Ch. 11 (event log as system-of-record).

8. Detailed component design

8.1 Per-endpoint isolation

Each endpoint has a bounded queue and a token bucket (default 200 in-flight, 50 rps). A slow endpoint backs up its own queue; the global pool stays free. Circuit-break individual endpoints after 50 consecutive failures — pause for 5 min, then probe.

8.2 Retry policy

Backoff schedule: 5 s, 5 min, 30 min, 2 h, 5 h, 10 h, 1 d (×3) — total ~72 h, max 10 attempts. Add 25% jitter to avoid synchronized retry storms (the "thundering herd after we recover" problem).

8.3 Signing

Header: v1,base64(HMAC_SHA256(secret, id + "." + ts + "." + body)). Customers can rotate; we keep prior secret valid for 24 h and emit both signatures so they migrate cleanly. Reject deliveries where ts older than 5 min on inbound replays.

8.4 Idempotency

Every delivery has a stable Webhook-Id. Customer is contractually expected to dedupe — we are at-least-once, not exactly-once. (Exactly-once over the open internet is impossible; see DDIA Ch. 9.)

9. Scaling & bottlenecks

  • Customer-induced HOL blocking: per-endpoint queue + connection pool prevents one bad URL from starving the system.
  • SSRF / internal-IP attack: URL validator blocks RFC1918, link-local, metadata-IP (169.254.169.254), DNS-rebinding via re-resolution at connect time.
  • TLS handshake cost: connection reuse per endpoint via persistent HTTP/2 pool — 10× throughput.
  • Hot endpoints (millions of events/min): shard into N parallel queues by hash(msg_id) so a single endpoint can use all worker capacity.

12. Talking points for the interviewer

  • "The most common interviewer trap is conflating ordering with at-least-once — I'll declare that we don't guarantee per-endpoint ordering during retries, and document it."
  • "SSRF is the number-one production incident category for webhook services; I'd budget real time on URL validation, DNS pinning, and a deny-listed egress proxy."
  • "For the dashboard's 'replay' button, I keep raw payloads in S3 with a 30-day TTL — replay re-enqueues from S3, not from Kafka, so we don't run out of Kafka retention."

Every technology choice in this design exists for a concrete reason.
