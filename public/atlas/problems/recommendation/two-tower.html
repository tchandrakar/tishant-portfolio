Design Two-tower Recommender System (user/item embeddings, ANN serving)

Hard recommendation

1. Problem statement

Build a two-tower recommendation system that powers personalized feeds for a TikTok-scale platform. The user tower encodes user features (history, demographics, context) into a 128-dim embedding; the item tower encodes content features (creator, topic, audio, visual). At serving time, given a user vector, retrieve top-K candidate items via Approximate Nearest Neighbor (ANN) search in under 30 ms, feeding a downstream ranker. Must handle 1B users, 500M items, sub-100ms end-to-end recommendation latency.

2. Functional requirements

  • Train user/item towers offline on (user, item, label) interaction logs with sampled-softmax or in-batch negatives.
  • Materialize all item embeddings nightly; refresh hot items every 15 minutes.
  • Compute user embedding online from latest features (last 50 watches, search queries, dwell signals).
  • Serve top-1000 ANN candidates per user request.
  • Support multiple recommendation surfaces (For You, Following, Search re-rank).
  • A/B test new tower architectures without full re-index.

3. Non-functional requirements

  • Latency: ANN lookup p99 under 30 ms; user-tower inference p99 under 20 ms.
  • Scale: 1B MAU, 200M DAU, peak 500K rec QPS, 500M item index.
  • Freshness: new items discoverable within 15 minutes of upload.
  • Availability: 99.95% with graceful fallback to popularity-based recs.
  • Recall@1000: at least 0.85 vs brute-force ground truth.

4. Capacity estimation

  • QPS: 200M DAU * 50 sessions/day * 5 reqs/session / 86400 ≈ 580K QPS peak.
  • Item index: 500M items * 128 floats * 4 bytes = 256 GB raw; PQ-compressed to ~32 GB.
  • Embedding storage: user emb cache 1B * 128 * 2 (fp16) = 256 GB in Redis cluster.
  • Training data: 200M DAU * 200 events/day * 200 bytes ≈ 8 TB/day to feature store.
  • GPU: 8 A100 nodes for nightly index build (90 min for 500M vectors).

5. API design

POST  /v1/recs/candidates    {userId, context, k=1000}    -> [itemId, score][]
GET   /v1/embeddings/user/{userId}                         -> {vector[128], version}
POST  /v1/embeddings/item/batch  {itemIds[]}               -> {itemId: vector[128]}
POST  /v1/index/rebuild      {modelVersion}                -> jobId  (admin)

6. High-level architecture

[Client] -> [API GW] -> [Rec Orchestrator] | +-> [User Tower Svc] <- [Feature Store (Redis+S3)] | | | v 128-d user vec +-> [ANN Index Shards (FAISS/ScaNN)] -> top-1000 | +-> [Ranker Svc] -> top-50 -> client Offline: [Kafka events] -> [Flink ETL] -> [TFRecord on S3] -> [TPU/GPU Trainer] | item tower -> [Index Builder] -> shards user tower -> [TF-Serving]

Walk-through: client request hits orchestrator, which fetches user features, runs the user tower, queries ANN shards in parallel via scatter-gather, merges top-K, then forwards to the ranker.

7. Data model & storage

7.1 Embedding store (Redis Cluster + S3)

KEY  emb:user:{userId}:v{modelVersion}   VALUE 256-byte fp16 blob   TTL 24h
KEY  emb:item:{itemId}:v{modelVersion}   VALUE 256-byte fp16 blob   TTL none
S3   s3://embeddings/v{ver}/items.parquet  (cold, full snapshot)

7.2 Why this split?

User embeddings change as the user interacts; cache with TTL and lazy recompute. Item embeddings are stable; bake into the FAISS shard at build time and keep S3 as source of truth for re-indexing.

8. Detailed component design

8.1 User tower

MLP with 3 hidden layers (512-256-128). Inputs: 64-dim recent-watch sequence avg-pool, 32-dim profile, 32-dim context. Trained with softmax over in-batch negatives + 1000 sampled negatives. Served via TF-Serving on T4 GPUs, 5ms p50.

8.2 Item tower & ANN index

FAISS IVF-PQ: 16384 inverted lists, PQ16x8 (16 sub-vectors, 8 bits each). nprobe=32 at query time. Index sharded by item_id mod 64 across 64 shards; each shard ~8M vectors, fits in 512 MB RAM. ScaNN is an alternative with better recall/latency Pareto.

8.3 Training pipeline

Daily batch on Spark+TF: 30 days of impressions, label = positive if dwell>3s or like. Negative sampling: 80% in-batch, 20% hard negatives mined from prior model. Loss: sampled-softmax with frequency-correction (log Q correction).

9. Scaling & bottlenecks

  • ANN tail latency: nprobe spikes hurt p99; use deadline-aware nprobe (start 32, drop to 16 if elapsed>15ms).
  • Hot users: celebrities with millions of similar followers thrash cache; pre-compute and pin user embeddings.
  • Index rebuild storm: 90-min nightly job; do rolling shard swap, never full cutover.
  • Model drift: dual-write user/item embeddings during version transitions for 24h.
  • Cold items: items <1h old bypass FAISS, served from a separate "fresh" index on Redis with brute-force scan.

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
FAISS IVF-PQ Compresses 500M 128-d vectors from 256 GB raw down to ~32 GB via PQ16x8, fitting one shard (8M vectors) in 512 MB RAM. With 16384 inverted lists and nprobe=32 we hit sub-30ms p99 ANN at 580K QPS while staying within budget on commodity boxes. Brute force (256 GB scan, 100x too slow); Annoy (worse recall/latency Pareto at this scale).
HNSW Used as fallback for the high-recall (>0.95) source where memory is acceptable. No training step means we can rebuild a 500M shard set incrementally, which matters when index drift forces re-embed during model upgrades. IVF-PQ alone (recall caps at 0.88); ScaNN (similar Pareto, less mature ops tooling internally).
ScaNN Listed as the alternative ANN with a better recall/latency Pareto for the 500M corpus when GPU-class CPUs are available. Anisotropic quantization beats PQ on Recall@1000 at the same ~30ms budget. FAISS (chosen first for ecosystem); HNSW (3x memory cost on a 256 GB raw corpus).
TensorFlow Serving on T4 GPUs User tower MLP (512-256-128) needs sub-20ms p99 inference at 580K QPS; T4 batching at 32 amortizes kernel launches and hits 5ms p50. TF-Serving's model versioning lets us A/B new towers without re-indexing items. TorchServe (weaker batching primitives); ONNX Runtime CPU (cannot meet 20ms at peak QPS).
Redis Cluster (embedding cache) 1B users * 128 fp16 = 256 GB user-embedding cache; Redis sharded gives sub-1ms GET so the cached path never violates the 100ms end-to-end budget. TTL 24h matches the user-tower drift window. Memcached (no native cluster mode for 256 GB); DynamoDB (10ms tail kills the budget).
Apache Kafka Carries the raw interaction events feeding the 8 TB/day training pipeline; partitioned by user_id so Flink ETL keeps user-feature ordering. Replay capability is mandatory for re-training new tower architectures on historic data. AWS Kinesis (lower retention); RabbitMQ (not designed for 8 TB/day throughput).
Apache Flink Streams Kafka events into the feature store with sub-minute lag so user-tower inputs (last 50 watches) are fresh; exactly-once via Kafka transactional sinks prevents double-counting impressions in the training set. Spark Structured Streaming (higher tail latency); Kafka Streams (no native exactly-once cross-topic joins at this scale).
Apache Spark Daily 30-day training-data assembly over impression logs (8 TB/day * 30 = 240 TB) is a textbook batch problem; Spark's Catalyst optimizer and shuffle handle the join between user features and item content efficiently. Beam on Dataflow (locked to GCP); Hadoop MapReduce (worse iterative joins).
S3 (item embedding snapshot) 500M item embeddings * 256 bytes = 128 GB Parquet snapshots per model version; S3 is the durable source-of-truth that ANN shards bulk-load at boot, enabling rolling shard swap rather than a 90-min full cutover. HDFS (operational overhead); EBS (not multi-attach across 64 shards).
NVIDIA A100 GPUs (training) Nightly index build embeds 500M items in 90 minutes on 8 A100 nodes; A100's HBM bandwidth and sampled-softmax-friendly tensor cores cut training time roughly 3x vs V100, keeping the daily window achievable. TPU v4 (vendor-locked); V100 (would push training to 4+ hours and miss the daily refresh).
gRPC + protobuf Scatter-gather to 64 ANN shards for top-1000 candidates needs binary serialization with HTTP/2 multiplexing; protobuf vectors are 4x smaller than JSON, halving wire time inside the 30ms ANN budget. REST/JSON (too verbose); Thrift (smaller ecosystem for ML serving).
Sampled-softmax with log-Q correction Full softmax over 500M items is intractable; in-batch + 1000 sampled negatives with log-Q debiasing gives unbiased gradient estimates and is the empirically dominant choice for two-tower retrieval at this corpus size. Full softmax (compute-prohibitive); triplet loss (worse Recall@1000 in published benchmarks).
