Design A/B Testing Platform (bucketing, stats engine, guardrails)

Hard recommendation analytics

1. Problem statement

Design an experimentation platform powering 1000+ concurrent A/B tests at TikTok scale (1B users, 200M DAU). Must handle deterministic bucketing in microseconds, exposure logging, daily metric computation, sequential testing with guardrails (latency, crashes), and a UI for PMs/scientists to launch and read out experiments.

2. Functional requirements

  • Define experiments with arms, traffic allocation, eligibility predicates.
  • Deterministic, sticky bucketing per (user, experiment).
  • Mutually-exclusive layers and orthogonal layers (factorial designs).
  • Exposure log: every flag check writes (user, exp, arm, ts).
  • Compute daily and cumulative metrics with confidence intervals.
  • Auto-shutdown on guardrail violation (crash rate, p99 latency).
  • Sample ratio mismatch (SRM) detection.

3. Non-functional requirements

  • Latency: bucket assignment p99 under 1 ms in-process; remote eval under 5 ms.
  • Scale: 200M DAU * 1000 flags * 100 evals/session = 20T evals/day = 230M QPS.
  • Metric freshness: daily counts within 4 hours of midnight; near-real-time dashboards within 5 min.
  • Correctness: identical user always gets same arm; bucket changes audited.

4. Capacity estimation

  • Bucketing: 230M QPS, but co-located in app via SDK -> 0 network cost.
  • Exposure events: 200M DAU * 1000 flags * 5 unique daily = 1T events/day, 200 bytes -> 200 TB/day raw, ~50 TB compressed.
  • Metric pipeline: 50 TB/day Spark scan, 4-hour SLO with 200-node cluster.
  • Config distribution: 1000 experiments * 2KB = 2MB config, refreshed every 60s to clients.
  • Storage: exposures retained 90 days = 4.5 PB, on S3 + Iceberg.

5. API design

// SDK
bool inArm(userId, expId, arm)         // local hash, 1 us
Variant getVariant(userId, expId)      // returns control/treatment/...

// Control plane
POST /v1/experiments {name, layers, arms, allocation, eligibility}
POST /v1/experiments/{id}/ramp  {fraction}
GET  /v1/experiments/{id}/readout?metric=ctr  -> {effect, ci, pvalue}
POST /v1/experiments/{id}/halt   (auto or manual)

6. High-level architecture

[App + SDK] | bucket = hash(userId + expSalt) % 10000 | emit exposure event v [Kafka exposures] -> [Flink dedupe + SRM check] -> [S3/Iceberg] | v [Spark daily metrics] -> [Metric Store (BigQuery/Druid)] | [Stats engine: t-test/CUPED/sequential]| v [Readout UI] Config plane: [Console UI] -> [Experiment Svc (Postgres)] -> [Config CDN] -> SDK polls

Walk-through: SDK does local bucketing using a config snapshot; emits one exposure event per (user, exp, day) via dedupe layer. Daily Spark job aggregates metrics, joins guardrails, writes to a metric store. Stats engine runs t-tests / sequential tests, surfaces in UI.

7. Data model & storage

7.1 Experiment metadata (Postgres)

CREATE TABLE experiments (
  exp_id text PRIMARY KEY, name text, status text,
  layer text, salt text, created_at timestamptz);
CREATE TABLE arms (
  exp_id text, arm text, allocation_bps int, params jsonb,
  PRIMARY KEY (exp_id, arm));

7.2 Exposure log (Iceberg on S3)

Partitioned by (date, exp_id). Columns: user_id, exp_id, arm, ts, ctx.
Compaction: daily merge of small files; Z-order on (exp_id, user_id).

7.3 Bucketing

arm = arms[ murmur3_128(user_id || ":" || exp_salt) mod 10000 / 10000 * cumulative_alloc ]. Salt per experiment ensures orthogonality across layers; same salt within a layer ensures mutual exclusion.

8. Detailed component design

8.1 Stats engine

Frequentist: Welch's t-test on per-user mean metric, with CUPED variance reduction using a 30-day pre-period covariate (typically reduces variance ~30%). Sequential: mSPRT for daily peeking without alpha inflation. Bayesian alternative for low-power surfaces.

8.2 Guardrails

Auto-halt when: arm crash rate >control + 0.5pp, p99 latency +30 ms, SRM p<0.001, or revenue -2% with 95% CI. Halt latency under 10 minutes via near-real-time aggregator on the same Flink job.

8.3 SDK

5 KB binary delta config pulled every 60s from a CDN-fronted config service. Falls back to last-good on fetch failure. Bucketing is pure-functional, fully offline-capable.

9. Scaling & bottlenecks

  • Exposure flood: dedupe to one event per (user, exp, day) at the Flink layer cuts volume 100x.
  • SRM false positives: with 1000 experiments and p<0.01, you'll fire 10/day by chance; use Bonferroni or FDR control.
  • Network effects: social/marketplace experiments leak between arms; cluster-randomize by region or social cluster.
  • Config drift: stale SDK keeps using old config; bound to 5 min and reject older with circuit breaker.
  • Metric store hotspots: top experiments queried 100x more; cache readouts for 5 min.

12. Talking points for the interviewer

  • "Bucketing is the single most reliability-sensitive surface; it must be local, deterministic, and never break a user."
  • "The hardest part is trust. PMs need readouts they believe; that means SRM checks, CUPED, sequential testing, and a culture of pre-registered hypotheses."
  • "For network-effect surfaces I'd push for cluster-randomized designs even though they hurt power."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
Murmur3-128 (bucketing hash) 230M QPS of bucketing must be local and 1us per call; Murmur3 is non-cryptographic, well-distributed, and faster than CityHash on common ARM/x86 mobile devices. 128-bit output gives 10000-bucket precision with negligible collision risk. SHA256 (10x slower, overkill); xxHash (great speed but weaker distribution at low bucket counts).
In-process SDK bucketing 200M DAU * 1000 flags * 100 evals = 230M QPS would require thousands of remote eval boxes; local bucketing reduces that to 0 network cost and gives the sub-1ms p99 the design demands. Offline-safe critical for mobile. Remote flag service (network cost, SPOF); edge-cached eval (weaker offline).
Apache Kafka 1T exposure events/day at 200 bytes = 2.3M events/sec peak; Kafka's partitioned log handles the throughput and gives Flink replay-from-offset semantics for the SRM and dedupe pipeline. Kinesis (lower retention, per-shard cap); Pulsar (smaller Flink connector ecosystem).
Apache Flink Dedupe to one event per (user, exp, day) cuts volume 100x; Flink keyed-state with TTL plus Kafka transactional sinks gives exactly-once guarantees per DDIA Ch.11. Same job emits near-real-time guardrails feeding the <10min auto-halt SLO. Spark Streaming (micro-batch lag breaks 10min halt); Kafka Streams (state size unfriendly to 200M-key dedupe).
Apache Spark Daily metrics over 50 TB/day compressed exposure log with a 4-hour SLO needs a 200-node cluster doing wide shuffles for per-experiment aggregations; Spark's Catalyst optimizer plus Iceberg pruning meets the window. Trino (better ad-hoc, weaker scheduled-job tooling); Beam (vendor-coupled).
Apache Iceberg on S3 4.5 PB exposure log retained 90 days needs partition-pruned reads for daily metric jobs; Iceberg's hidden partitioning by (date, exp_id) plus Z-order on (exp_id, user_id) gives O(1) experiment scans without listing all files. Hive tables (no atomic writes, slower planning); Delta Lake (similar; Iceberg picked for engine neutrality).
BigQuery / Druid (metric store) PMs query readouts with sub-5min freshness and slice/dice across 1000 experiments * 100 metrics; columnar OLAP with pre-aggregated rollups is the only architecture that supports interactive readout latency. Postgres (cannot scale to 4.5 PB); ClickHouse (viable; chose Druid for time-series rollup ergonomics).
PostgreSQL (experiment metadata) 1000 active experiments with arms, allocations, eligibility predicates is small (<10 GB) but mutates frequently and requires ACID for the ramp/halt control plane; Postgres is the safe correct choice. etcd (no relational queries for arm config); MySQL (similar; Postgres picked for jsonb richness).
CDN config distribution 2MB binary delta config refreshed every 60s to 200M DAU SDKs needs caching; CDN edge serves 99% of these without origin load, and 60s staleness is bounded by the 5-min circuit-breaker reject threshold. Direct origin (origin overload); push via WebSocket (mobile battery cost).
Welch's t-test + CUPED Per-user mean metric with unequal variances is the realistic distribution; CUPED with a 30-day pre-period covariate cuts variance ~30%, meaning we need ~30% smaller samples to detect the same effect, which is essential at 1000 concurrent experiments. Mann-Whitney U (less power on continuous metrics); raw t-test (variance budget too high).
mSPRT (sequential testing) PMs peek daily; classical fixed-horizon t-tests inflate alpha catastrophically. mSPRT (mixture sequential probability ratio test) preserves alpha under continuous monitoring, which is the only honest way to support the daily readout culture. Bonferroni-on-peeks (over-conservative); always-valid CIs (similar; mSPRT chosen for tooling familiarity).
Bonferroni / FDR control 1000 experiments at p<0.01 fire 10 false SRM alerts/day by chance; multiple-testing correction (Bonferroni for guardrails, BH-FDR for exploratory readouts) is mandatory to keep the alert volume actionable. Per-test alpha=0.05 (alarm fatigue); no correction (loss of trust).
Bloom filter (dedupe assist) Exposure dedupe at 1T events/day uses Flink keyed-state but a per-(user,exp,day) bloom in front avoids most state lookups; small RAM footprint, microsecond check, FP rate acceptable for an optimization layer. Pure Flink state (more RAM, slightly slower); HyperLogLog (counts not membership).
