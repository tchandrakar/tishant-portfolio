Design Candidate Generation Pipeline (ANN, FAISS/HNSW, recall-focused)

Hard recommendation search

1. Problem statement

Design the candidate-generation stage of a large-scale recommender that retrieves a few thousand items from a corpus of hundreds of millions in under 50 ms, optimized for high recall (not precision). Multiple retrieval sources (collaborative-filtering ANN, content ANN, graph walks, popularity, follow-graph) feed a downstream ranker. Numbers: 500M items, 200M DAU, 580K rec QPS at peak, similar to TikTok's For You stack.

2. Functional requirements

  • Retrieve top-2000 candidates per user/request from N sources, deduped and merged.
  • Support pluggable retrieval sources (two-tower, item-CF, graph, trending, follow).
  • Apply hard filters: blocked creators, region, language, age-gate, already-seen.
  • Allow per-source quotas tunable via experiment config.
  • Index new items within 10 minutes of publish.
  • Snapshot indexes nightly; support rollback to previous version.

3. Non-functional requirements

  • Latency: p99 fan-out + merge under 50 ms.
  • Recall: recall@2000 vs brute-force above 0.90 for two-tower source.
  • Scale: 500M item corpus; 580K QPS peak; 8 retrieval sources in parallel.
  • Availability: 99.95%; degrade to popularity when ANN degraded.

4. Capacity estimation

  • QPS per source: 580K (one fan-out per request); each source 580K QPS.
  • Index memory (HNSW, 128-d fp16): 500M * (256 + 64 graph) = 160 GB; sharded 64x = 2.5 GB/shard.
  • FAISS IVF-PQ alternative: ~32 GB; sharded 16x = 2 GB/shard.
  • Ingest: 5M new items/day -> 60 items/sec; embedding inference 2 GPU nodes.
  • Filter set per user: avg 10K already-seen item IDs; bloom filter 12 KB at 1% FPR.

6. High-level architecture

[Rec Orchestrator] | v scatter +------+----------+--------+--------+--------+ | | | | | | v v v v v v [Two-tower ANN] [Item-CF ANN] [Graph walk] [Trending] [Follow] | | | | | shards shards redis-graph redis mysql \________|______________|_____________|__________/ merge + dedupe | v [Filter Svc] -- [Bloom: seen] [Block list] | v top-2000 to ranker Ingest: [Kafka items] -> [Embedding Svc (GPU)] -> [Index Writer] | +-> HNSW shards (in-mem) +-> S3 snapshot (durable)

Walk-through: orchestrator fans out to all configured sources in parallel with a 30 ms deadline, collects results, dedupes by item_id, applies filters, returns top-2000.

7. Data model & storage

  index_id text, shard_id int, host text, PRIMARY KEY (index_id, shard_id));

7.2 Why HNSW for high-recall source?

HNSW gives recall>0.95 at 1ms latency without training. Cost is 3x memory of IVF-PQ, acceptable since the flagship two-tower source justifies the spend. Cheaper sources (CF) use IVF-PQ.

8. Detailed component design

8.1 ANN shard

Each shard is a stateful pod holding ~8M vectors in HNSW. Built from S3 snapshot at boot (5 min). Live writes stream from Kafka into a per-shard delta segment; query merges base + delta. Compaction every 30 min collapses delta into base.

8.2 Scatter-gather orchestrator

Async fan-out using gRPC streaming, hedged at 15 ms (send duplicate to a peer if no reply). Per-source budget configured: two-tower 30ms, CF 25ms, graph 20ms, trending 5ms. If a source misses budget, drop it; never block.

8.3 Filter service

Maintains per-user "seen" bloom filter in Redis (12 KB, 0.5% FPR). Updated by a Flink job consuming impression events with 30s lag. Block-list and creator-block joined inline.

9. Scaling & bottlenecks

  • Tail latency: a slow shard ruins p99; use request hedging and tail-tolerant nprobe.
  • Index hot-spotting: celebrity items pulled by every user; replicate hot shards 4x.
  • Memory pressure: HNSW + delta segment can OOM during compaction; pre-allocate 1.5x headroom.
  • Snapshot stampede: all shards loading from S3 at once saturates network; staggered boot with jitter.
  • New-item recall: recently-added items miss CF signal; rely on content tower until first 100 impressions.

12. Talking points for the interviewer

  • "Recall, not precision, is the goal here. The ranker downstream is precision-optimized. Mixing them confuses tuning."
  • "I split sources by cost: HNSW for the flagship two-tower (recall matters), IVF-PQ for cheap CF sources, hand-rolled lookups for trending/follow."
  • "The hardest production issue is index rebuild stampedes. Rolling shard swap with snapshot pinning is the move."

13. Tech stack & rationale

Every technology choice in this design exists for a concrete reason.

TechnologyWhy we chose it (for THIS problem)Alternatives considered
HNSW The flagship two-tower source needs Recall@2000 above 0.90 against a 500M corpus; HNSW delivers >0.95 at 1ms latency without a training step, fitting 8M vectors per shard in 2.5 GB. Worth the 3x memory because retrieval recall directly bounds downstream ranker quality. FAISS Flat (256 GB, infeasible); Annoy (worse build time and tail latency).
FAISS IVF-PQ Used for cheaper CF retrieval sources where 0.85-0.90 recall suffices; compresses 500M vectors to ~32 GB total (16 shards * 2 GB) so we can run 4 secondary retrieval sources without 4x'ing the memory bill. HNSW everywhere (3x memory waste on cheap sources); brute-force (250x too slow at 580K QPS).
ScaNN On-deck alternative for the IVF-PQ tier; anisotropic quantization gives 5-10% better recall at the same compute, useful when CF source quality becomes the bottleneck after the two-tower source is saturated. FAISS (chosen now for tooling parity); HNSW (memory cost not justified for CF).
Apache Kafka 5M new items/day stream into the embedding service; Kafka partitions by item_id so each ANN shard consumes only its assigned partition for incremental upserts, preserving the 10-minute publish-to-discoverable SLO. AWS Kinesis (per-shard write cap insufficient); Pulsar (smaller ecosystem of Flink connectors).
Apache Flink Maintains the per-user "seen" bloom filter with 30s lag from impression events; keyed-state on user_id keeps the 12 KB bloom local to one operator, avoiding 580K QPS round-trips to Redis on every filter update. Spark Streaming (micro-batch lag too high); Kafka Streams (less mature large-state handling).
Apache Spark (snapshot build) Nightly batch over the full Iceberg item-feature lake produces the S3 HNSW snapshot ANN shards bulk-load at boot; Spark's wide cluster handles the 500M-vector embedding inference parallelism. Beam (vendor lock-in); ad-hoc Python (cannot scale to 500M items in the nightly window).
Redis (bloom filters + trending) Per-user 12 KB seen bloom * 200M DAU = 2.4 TB; Redis Cluster sharded by user_id keeps lookups sub-1ms, fitting inside the 50ms scatter-gather budget. RedisGraph variant powers the graph-walk source. Memcached (no native bloom, no cluster mode at 2.4 TB); DynamoDB (10ms tail).
S3 (snapshot durability) HNSW shard snapshots (500M * 320 bytes = 160 GB) live on S3; staggered boot loads with jitter prevent the snapshot-stampede problem when 64 shards restart simultaneously. EBS (not multi-attach); HDFS (operational overhead).
PostgreSQL (index metadata) Index-version, shard-assignment, and rollback metadata is small (<1 GB) but needs strong consistency for the rolling shard swap; Postgres ACID guarantees no two shards claim the same shard_id during a swap. etcd (overkill for relational data); MySQL (less rich JSONB for shard params).
gRPC streaming + hedged requests 8-way scatter-gather at 580K QPS with a 30ms budget cannot tolerate one slow shard; hedging at 15ms (send dup to peer) cuts p99 by 30-50% per published Google research and fits the DDIA Ch.8 mitigation pattern cited. Plain REST (no streaming primitives); Thrift (smaller hedging tooling).
NVIDIA T4 GPUs (item embedding) 2 T4 nodes embed the 60 items/sec ingest stream with 200ms per item, well within the 10-minute publish SLO; T4's INT8 inference path doubles throughput vs CPU and is cheaper per item than A100 at this rate. CPU inference (would need 30+ nodes); A100 (overprovisioned for 60 items/sec).
Bloom filters (seen-set) Per-user 10K already-seen IDs encoded in 12 KB at 1% FPR is the only structure that meets both memory (200M users * 12 KB = 2.4 TB feasible) and 50ms latency targets; the 1% false positive rate is an acceptable recall cost. Exact set in Redis (10x memory); Cuckoo filter (similar but more complex write path).
RedisGraph Graph-walk retrieval source needs sub-20ms multi-hop traversals over the follow graph; RedisGraph's in-memory adjacency representation hits this where Neo4j on disk would not at 580K QPS. Neo4j (disk-based, 5-10x slower); custom adjacency lists in Redis (no graph query language).
