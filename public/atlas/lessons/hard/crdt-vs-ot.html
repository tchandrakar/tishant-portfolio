CRDTs vs OT for collaborative editing

Hard crdt ot collaboration

The idea in one paragraph

When two people edit the same document at the same time, you have to merge their changes without losing intent. Operational Transformation (OT) sends operations through a central server that transforms each op against concurrent ops so the end state converges. CRDTs (Conflict-free Replicated Data Types) attach unique IDs and partial-order metadata to every character or field so concurrent edits can be merged on any replica with no coordination — same inputs, same final state, regardless of order.

Why it matters

Google Docs, Figma, Notion, Linear, and Apple Notes all need real-time multi-user editing with offline support, undo, and presence. The choice between OT and CRDT decides whether your sync layer is a clever centralized algorithm (OT) or a peer-to-peer-friendly data structure (CRDT). It also drives storage cost, conflict semantics, and how nasty your edge cases get when network partitions heal.

Real-world examples

  • Google Docs: OT (Jupiter algorithm) — central server transforms ops; battle-tested for two decades.
  • Figma: Custom CRDT-like tree with last-writer-wins on properties; server is the source of truth but clients merge optimistically.
  • Apple Notes / iCloud: CRDT — needed for offline iPhone edits that sync later.
  • Automerge / Yjs: Open-source CRDT libraries; Yjs powers many startups doing local-first apps.
  • Linear / Notion: Hybrid — CRDT-ish per-block, with a central server arbitrating structural moves.

The mental model

OT (centralized transform) Client A: insert("X", pos=3) ---\ >--[ Server transforms ]--> ops to B, C Client B: delete(pos=2) ---/ CRDT (commutative merge) Client A: insert(id=A1, after=A0, "X") Client B: insert(id=B1, after=A0, "Y") Any replica merges by ID order: A0 -> A1 -> B1 (deterministic)

OT requires every concurrent op to be transformed against every other op the server has accepted; the server is the linearization point. CRDTs avoid the transform by giving every edit a globally unique ID and a rule (e.g. "insert-after this ID, tie-break by site ID") so any two replicas that have seen the same set of ops converge — even if they saw them in different orders.

Trade-offs

  • Metadata cost: CRDTs carry tombstones for deleted characters and per-character IDs. A 1MB doc with heavy editing can balloon to 5–10MB. Yjs and Automerge have spent years compressing this.
  • Server simplicity: CRDT servers can be dumb relays (great for P2P, offline-first). OT servers are the brain — harder to scale horizontally.
  • Intent preservation: OT can encode richer "user intent" rules (e.g., "if you bolded a word and I deleted it, drop the bold"). CRDTs default to mechanical convergence; intent rules are bolted on.
  • Undo: Undo in OT is famously hard — inverse-transform an op against everything that happened after. CRDTs make undo a first-class op (mark tombstone live again).
  • Rich structure: Trees (like Figma's scene graph) need specialized CRDTs (Move CRDT, RGA-tree). Naive CRDTs corrupt under concurrent moves.

Anti-patterns and failure modes

  • Rolling your own OT: Transform functions must satisfy TP1 (and ideally TP2) properties. Getting this wrong silently corrupts documents days later. Use a library.
  • CRDT without garbage collection: Tombstones never collected; doc grows forever, cold-start gets slower every week.
  • Treating presence cursors as CRDT data: Cursors are ephemeral. Putting them in the merged doc creates massive history bloat. Use a separate awareness/presence channel.
  • Concurrent tree moves: Two users move folder A under B and B under A simultaneously. Without a Move CRDT this creates a cycle or silently drops one move.
  • Assuming CRDT = no server: You still need auth, persistence, snapshotting, and access control. CRDT only solves the merge problem.

Common mistakes

  • Picking CRDT "because it's modern" when a single-region OT server would solve it in 1/10th the code.
  • Forgetting that ACL/permission changes are NOT a CRDT — they need linearization (you don't want eventually-consistent "who can read this").
  • Not benchmarking with realistic edit traces (10k ops over a week) — both approaches look fine on small docs.

Try it yourself

Question: You're designing a collaborative spreadsheet. Cells hold formulas that reference other cells. Two users edit the same cell offline for 30 minutes, then both reconnect. Do you reach for OT or a CRDT, and how do you handle the formula references?
Hint: Think about what "merge" even means for a formula. Is the cell value a string, an AST, or a CRDT register? What if user A renames the column that user B's formula references?
