The hot-key / celebrity-fanout problem

Hard hot-key fanout scaling

The idea in one paragraph

In any sharded or partitioned system, traffic is rarely uniform. A small fraction of keys (a viral tweet, a celebrity's profile, a Black Friday SKU, a single trending hashtag) attract orders of magnitude more reads or writes than the average. That single shard becomes a bottleneck while the others are bored. The "celebrity-fanout problem" is the write-side variant: when a user with 100M followers tweets, naively writing into 100M timelines kills the system. Hot-key handling is the art of detecting these outliers and routing them through different code paths.

Why it matters

Hot keys are the single most common reason "we're sharded, why is it slow?" shows up in postmortems. Capacity planning based on averages is a lie — the p99.9 key drives your infrastructure spend and your outage frequency. Every social, e-commerce, gaming, or analytics system above modest scale eventually rewrites a chunk of itself to handle skew. Knowing the patterns up front saves you from a panicked re-architecture during a viral moment.

Real-world examples

  • Twitter timelines: Justin Bieber-era — fanout-on-write was untenable for top accounts. They use a hybrid: fanout-on-write for normal users, fanout-on-read merge for celebrity follows.
  • Instagram: Similar hybrid; celebrity posts are pulled in at read time and merged with the precomputed feed.
  • YouTube: Video metadata for "trending" videos is cached at every edge, not in a single shard.
  • Stripe / payment processors: A hot merchant (e.g., a major retailer on launch day) has dedicated rate-limit buckets and pre-warmed connection pools.
  • Redis / DynamoDB: Both have well-known "hot partition" failure modes; DAX and DynamoDB Accelerator exist largely to absorb hot-key reads.

The mental model

UNIFORM (theory) ZIPFIAN (reality) shard QPS shard QPS ----- ----- ----- ----- s1 1000 s1 90000 <-- celebrity lives here s2 1000 s2 800 s3 1000 s3 900 s4 1000 s4 750 s5 1000 s5 820 ^ | 100x skew, single shard at limit

Real-world keys follow Zipf or power-law distributions: the top 1% of keys often serve 50%+ of traffic. Hashing into N shards doesn't fix this — you just get one very hot shard.

Mitigations: read-side hot keys

  • Multi-tier caching: Hot keys live in process-local LRU + edge cache + regional cache. Origin sees only cache misses, which by definition aren't hot.
  • Request coalescing: 10,000 concurrent requests for the same key collapse into one origin fetch (singleflight pattern). Standard in CDNs and well-built caches.
  • Read replicas with consistent-hash routing: Promote hot keys to N replicas; route reads round-robin across them.
  • Key splitting (read-side): Store the value under key#0..N-1, write to all, read from a random one. Trades write amplification for read distribution.
  • Probabilistic refresh: Refresh cache entries before TTL expires (with a small probability) so 1M users don't simultaneously stampede on TTL boundary.

Mitigations: write-side hot keys (counters, fanout)

  • Sharded counter: Instead of one row "likes_for_post=N", maintain 100 sub-counters; aggregate on read. Used in YouTube view counts, Reddit votes, etc.
  • Fanout-on-read for celebrities: Don't write to 100M follower inboxes. At read time, merge their precomputed feed with celebrity tail-fetch. The "Twitter trick."
  • Async / batched fanout: Big-fanout writes go on a queue; a fleet of workers spreads delivery over seconds or minutes. Trade freshness for survivability.
  • Pre-aggregation in stream: If 1000 events/sec all increment the same counter, group them in 1-second windows and write 1 update per second per shard.
  • Write through a leader-per-key with batching: Pin the hot key to one node that owns coalescing; dramatically fewer cache invalidations downstream.

Detection

  • Heavy-hitter sketches: Count-Min Sketch or Space-Saving on the request stream identifies hot keys in real time with bounded memory.
  • Per-key metrics on the dataplane: Sample request keys at 1/1000 and emit; aggregate centrally. Cheap and good enough for skew detection.
  • Latency anomalies: A single-shard p99 spike usually means hot key before it means hardware. Alert on per-shard, not global, p99.

Anti-patterns and failure modes

  • "Just add a cache": Cache stampede on TTL expiry can be worse than no cache. Need request coalescing + jittered TTL + async refresh.
  • Hash-then-pray: Assuming consistent hashing solves skew. It distributes keys evenly, not traffic.
  • Synchronous fanout to all followers: A celebrity tweet blocks the API call until 100M writes complete. The user's phone times out, retries, doubles the load.
  • One global rate limiter for hot key: Rate-limiting a viral key at the edge protects the backend but turns into a thundering retry storm. Use queues, not 429s, for elephant flows.
  • No pre-warming for known events: A scheduled product launch at 9am — and ops finds out at 9:00:30 when the shard melts. Pre-warm caches and pre-shard the key the night before.
  • Sharded counter without GC: Counters keep accumulating sub-shards across deploys; reads get slower forever.

Common mistakes

  • Optimizing for the median user when the top 0.01% are the load-bearing tail.
  • Picking partition keys based on what's "natural" (user_id) without checking distribution (10 power users dominate).
  • Designing fanout-on-write because it's "simpler" then having no fallback when an account hits 1M followers.

Try it yourself

Question: You're designing the "like" counter for a video site. Some videos get 5 likes/day; one trending video gets 50,000 likes/second for 10 minutes. Sketch the data path: counter storage, write path, read path, and how you handle the moment of virality without melting Redis.
Hint: Do all writes need to hit the same row? Does the displayed count need to be exact, or "close enough"? When does the answer change for a payment vs a like?
