Approximate algorithms at interview scale

Hard probabilistic streaming data-structures

The idea in one paragraph

When you need a number now from a stream of billions — "is X in the set?", "how many distinct users today?", "what are the top 100 hashtags?", "what's the p99 latency?" — exact answers cost too much memory and CPU. Approximate algorithms trade a bounded, tunable error for orders of magnitude less space. The five you must know cold are Bloom filter (membership), HyperLogLog (cardinality), Count-Min sketch (frequency), Misra-Gries / Space-Saving (top-K), and t-digest (quantiles).

Why it matters

Most "scale" interview questions secretly hinge on one of these. "Track unique visitors per URL" → HLL. "Spam-check before hitting the DB" → Bloom. "Trending hashtags" → Count-Min + Space-Saving. "Latency dashboards across 10k hosts" → t-digest. Knowing the math intuition (not the proof) lets you say "we use a 12 KB HLL per URL with ~2% error" instead of "uh, we'd... store all of them?"

Bloom filter — membership

Question: "is x in the set S?" Answer: definitely not, or probably yes. Use k hash functions to set k bits in an m-bit array. To check, AND those bits. False positives only; never false negatives. The math: with n insertions, optimal k = (m/n) ln 2, giving FPR ≈ 0.6185^(m/n). Rule of thumb: ~10 bits per element ≈ 1% FPR; ~14 bits ≈ 0.1%. So a 1B-item Bloom at 1% costs ~1.25 GB vs. ~24 GB to store the keys themselves.

Variants: Counting Bloom (allows delete, ~4x space), Cuckoo filter (better cache locality, supports delete, slightly tighter), Scalable Bloom (grows when filling).

HyperLogLog — cardinality

Question: "how many distinct items in this stream?" Idea: hash each item to a uniform random bit string. The maximum number of leading zeros you've ever seen is a logarithmic estimate of cardinality (if you've seen k leading zeros, you've probably seen ~2^k distinct items). One register is noisy; HLL splits the hash into p bucket bits + the rest, keeps the max-leading-zero per bucket, and harmonic-means the buckets.

The math: with m = 2^p buckets, standard error ≈ 1.04 / sqrt(m). So m = 2^14 = 16384 buckets uses ~12 KB and gives ~0.81% error — for any cardinality, even billions. HLLs merge trivially (per-bucket max), so you can pre-aggregate per-shard and combine.

Count-Min sketch — frequency

Question: "how many times did key x appear?" Use a d × w matrix of counters and d independent hashes. To increment: hash x with each of the d hashes, increment matrix[i][h_i(x)]. To query: return the min across the d rows. Hash collisions only ever over-count, so the min is the tightest upper bound.

The math: set w = ⌈e/ε⌉ and d = ⌈ln(1/δ)⌉. Then with probability ≥ 1-δ, error ≤ ε · ||a||_1 where ||a||_1 is total stream weight. Picking ε=0.001, δ=0.01: w≈2718, d≈5, total ~14k counters ≈ 56 KB. Use it for heavy-hitter candidates, coupled with a small heap for top-K.

Top-K — Misra-Gries and Space-Saving

Question: "what are the most frequent K items in a stream?" Exact top-K needs O(N) space. Misra-Gries keeps k-1 counters; on each item, increment its counter if present, else if there's an empty slot use it, else decrement every counter by 1 (and evict zeros). Guarantees: any item with frequency > N/k is in the table; counter values lower-bound true counts (errors up to N/k).

Space-Saving (Metwally et al.) is the practical refinement: when the table is full and a new item arrives, replace the minimum-counter item — but inherit its counter and add 1. This gives both lower bounds (counter − error) and upper bounds (counter), and tracks heavy hitters far more accurately at the same memory. Top-K is typically Space-Saving with k×c slots (c=2–10 for safety) plus a min-heap for fast eviction.

t-digest — quantiles

Question: "what's the p50, p99, p99.9 of this stream?" Naive quantile needs all data; reservoir sampling has fixed error per quantile but loses tail accuracy where you need it most. t-digest (Ted Dunning) keeps a sorted set of weighted centroids over the value range, where centroid weight is small near the tails (p1, p99, p999) and large in the middle. Concretely it bounds the size of each centroid by k(q) = δ · q(1-q) for some compression δ; this packs centroids near 0 and 1 and spreads them out near 0.5.

The math/intuition: the relative error on a quantile q is roughly proportional to q(1-q)/δ, so error at p99 (q=0.99) is ~100× smaller than at p50. Typical δ=100 gives ~100 centroids, ~few KB, with sub-1% relative error at the extremes. Like HLL, t-digests merge: per-host digests combine into a global one without re-streaming raw data.

Common mistakes

  • Using a Bloom filter when you need counts (use Count-Min) or distinct counts (use HLL).
  • Sizing HLL by data volume — its size is independent of N. Pick by error tolerance.
  • Forgetting Count-Min over-counts. Pair it with Space-Saving for unbiased top-K.
  • Computing quantiles by averaging per-host p99s. Quantiles do not average; merge t-digests instead.
  • Treating false positives in a Bloom as harmless when downstream is expensive (e.g., DB lookup is cheaper than the wrong path).

Try it yourself

Question: Design a real-time "trending now" service for a social app: 200k posts/sec, find the top 100 hashtags over the last 1 hour with <2% relative error and unique-author counts per tag with ~1% error. What structures, what sizes, how do you merge across shards?
Hint: Per-tag HLL (12–16 KB each) for unique authors. Sliding-window Count-Min (e.g., 10 × 6-minute panes) plus a Space-Saving top-200 per shard. Merge HLLs by per-bucket max; merge top-K by union and re-rank. The whole thing fits in memory on one node.
