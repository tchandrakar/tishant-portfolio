Backpressure & flow control

Hard backpressure flow-control resilience

The idea in one paragraph

Every multi-stage system has a slowest stage. Backpressure is the discipline of letting that slow stage tell upstream to slow down instead of silently dropping work, blowing up memory, or cascading timeouts. Flow control is the implementation: bounded queues, credit-based flow, blocking sends, or explicit "pause/resume" signals. Without backpressure, a 100x burst at the front turns into unbounded queue growth, GC death spirals, and a half-finished mess when the system finally falls over.

Why it matters

Most "the system fell over under load" outages are backpressure failures, not throughput failures. The system kept accepting work it could not finish. Latency exploded, queues grew, retries multiplied, and what started as a 2x spike became a 20x meltdown. Designing backpressure in from day one is one of the highest-leverage habits in distributed systems.

Real-world examples

  • TCP: The classic. Receive window tells the sender how many bytes it can have in flight. No window, no send. Backpressure baked into the wire.
  • Kafka consumers: Consumer pulls; broker never pushes faster than the consumer asks. Built-in backpressure.
  • gRPC streaming: Flow-control credits per stream; matches HTTP/2 framing.
  • Reactive Streams (RxJava, Project Reactor, Akka Streams): The request(n) protocol — subscriber asks for N items, publisher sends at most N.
  • Node.js streams: writable.write() returns false when the buffer is full; you're expected to wait for 'drain'.
  • BBR / congestion control: Datacenter networks (Google, Meta) use bandwidth-delay-product probing for backpressure across the fabric.

The mental model

PUSH (no backpressure) Producer --burst--> [ unbounded queue ] --> Consumer (slow) ^^^ memory grows forever, GC stalls, OOM PULL (backpressure) Producer <--- "give me 100" --- Consumer --- 100 items ----> processes 100, asks for 100 more if slow, asks slowly. Producer naturally blocks. CREDIT-BASED Sender has C credits. Sends 1 -> credit-=1. Receiver processes, returns credit. Sender refills. If C reaches 0, sender stops. Bounded memory by construction.

The unifying principle: somewhere in the pipeline there must be a finite resource (window, credits, queue slots) that the slow stage controls. When it runs out, upstream stops. There is no other way — every alternative (drop, OOM, retry storm) is failure.

Strategies

  • Bounded queues: The simplest, most powerful tool. Pick a size; when full, blocking-put forces upstream to wait. Magic number: bounded = backpressure; unbounded = ticking bomb.
  • Pull-based consumption: Consumer asks for work. Trivially backpressured. Kafka, SQS long-poll, request-driven workers.
  • Credit-based flow: Receiver grants N credits; sender sends at most N. Decoupled and works across async boundaries (HTTP/2, AMQP, RSocket).
  • Token bucket at ingress: Bound the rate at which work enters the system. Doesn't help mid-pipeline pressure, but prevents pathological floods.
  • Load shedding: When backpressure hits the front door, return 429/503 fast. Better to tell 10% of users "try again" than melt for everyone.
  • Adaptive concurrency limits: Netflix's concurrency-limits library: dynamically adjust max-in-flight based on observed latency (Little's Law). Self-tuning backpressure.
  • Backpressure all the way to the client: The browser, mobile app, or upstream service must see the slowdown. If you absorb it silently in a queue, the symptom moves but doesn't disappear.

Anti-patterns and failure modes

  • Unbounded in-memory queue: "Java's LinkedBlockingQueue" with no capacity. Looks fine in load test, OOMs in production. Always set a bound.
  • Async fire-and-forget without bound: executor.submit(...) in a loop, no limit on pending tasks. Memory grows; latency follows.
  • Retry storms: Slow service returns timeouts. Caller retries 3x. Now the slow service has 4x the load. Now it's even slower. Use deadlines + jittered exponential backoff + circuit breakers; never retry on a deadline-exceeded.
  • Backpressure that hides: Service A blocks because B is slow, but A's clients see only "everything is fine, just slow." Healthchecks pass; you're degraded but invisible. Surface queue depth, in-flight count, and saturation as first-class metrics.
  • "We'll just add a Kafka in front": A queue in front of an overloaded service hides the problem until the queue fills disk. You bought time, not capacity. Ship a plan to drain or scale, not just a buffer.
  • Head-of-line blocking on a slow consumer: One slow consumer in a multi-tenant pipeline halts the entire stream. Use per-tenant partitions or bulkheads.
  • Mismatched timeouts: Client timeout 1s, server timeout 30s. Client retries while server is still working. Always nest timeouts: caller < callee.
  • Queue length as the only signal: A short queue can still mean disaster if items are huge or processing is degraded. Watch latency and saturation, not just depth.

Common mistakes

  • Treating backpressure as an optimization rather than a correctness property. It's correctness — without it, the system has no defined behavior under overload.
  • Adding caches and queues to "absorb spikes" without bounding them. You've moved the failure, not prevented it.
  • Forgetting to surface backpressure in API contracts. If a client can't tell that you're saturated, they retry, making it worse.
  • Confusing rate-limiting with backpressure. Rate limits cap input; backpressure responds to actual capacity. You typically need both.

Try it yourself

Question: You're building an image-processing pipeline: HTTP upload → queue → resize worker → ML tagger → persistent store. The ML tagger uses a GPU and processes 50 images/sec. Uploads can spike to 5,000/sec. Where do you place backpressure, what does the user see during a spike, and how do you keep the GPU pegged at 100% without head-of-line blocking small users behind one user's 10,000-image upload?
Hint: The slowest stage owns the truth. Walk backward from the GPU. What does it tell upstream? What does upstream tell the HTTP layer? What does the HTTP layer tell the user — synchronously, asynchronously, or both?
