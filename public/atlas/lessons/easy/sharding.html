Sharding (partitioning): range, hash, directory

Easy sharding partitioning scalability

The idea in one paragraph

Sharding is splitting one big dataset into many smaller pieces — shards — that live on different machines, so reads and writes spread across the fleet instead of crushing a single node. The whole game is choosing the right partition key and partition strategy: range (sort by key, slice into contiguous ranges), hash (hash the key, mod the bucket count), or directory (a lookup service decides which shard owns which key). Pick well and growth is just "add nodes". Pick badly and you get hot shards, cross-shard joins, and resharding nightmares.

Why it matters

Vertical scaling runs out. Replication helps reads but not writes — every replica still holds the full dataset. Once your data or write throughput exceeds what one machine can do, sharding is the only path forward. It's also one of the most error-prone decisions in distributed systems: a bad partition key shows up months later as one shard at 95% CPU while the others sit at 10%.

Real-world examples

  • Range sharding (HBase, Bigtable): rows sorted by key, contiguous ranges per region. Great for range scans; risky for monotonic keys (timestamps -> hot tail shard).
  • Hash sharding (Cassandra, DynamoDB): hash(partition_key) decides the node. Even distribution by default; range scans become expensive.
  • Directory sharding (Vitess, Figma's approach): a metadata service maps tenant -> shard. Flexible re-balancing; the directory itself becomes a critical dependency.
  • Geosharding (Uber): shard by region/city — natural locality, but cross-region trips need special handling.

The mental model

RANGE HASH DIRECTORY keys: A..Z hash(key) mod 4 lookup(key) -> shard +-------+ +-------+ +---+ +---+ +---+ +---+ +-----------+ | A..F | | G..M | ... | 0 | | 1 | | 2 | | 3 | | metadata | +-------+ +-------+ +---+ +---+ +---+ +---+ | service | good for scans even spread +-----------+ bad for hot tail bad for range scans | | | shard shard shard Resharding pain (with naive hash mod N): adding one node remaps ~all keys. Use CONSISTENT HASHING to remap only ~1/N.

Consistent hashing places both nodes and keys on a ring; each key belongs to the next node clockwise. Adding a node only steals keys from its immediate neighbor, not the whole cluster — this is why every modern key-value store uses it.

Common mistakes

  • Choosing a low-cardinality partition key (like country). The biggest country becomes a permanent hot shard.
  • Sharding by timestamp. All writes pile into the most recent shard while older ones sit idle.
  • Forgetting cross-shard queries. A SQL JOIN across shards either fans out (slow) or moves to the application layer (complex).
  • Plain hash-mod-N instead of consistent hashing. The first time you add a node, you reshuffle nearly the entire dataset.
  • No rebalancing plan. Shards grow unevenly over time; without splits/merges, you end up back at "one giant shard".

Try it yourself

Question: You're sharding a tweets table across 16 nodes. Two candidate partition keys: user_id or tweet_id. Which gives more even load? Which makes "show this user's timeline" cheap? What happens to your design when one user has 50M followers and tweets every second?
