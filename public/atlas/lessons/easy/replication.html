Replication: single-leader, multi-leader, leaderless

Easy replication availability consistency

The idea in one paragraph

Replication keeps the same data on multiple machines so reads can fan out, failures don't lose state, and users near different regions get fast service. The hard part isn't copying bytes — it's deciding who is allowed to accept a write. That single decision splits replication into three families: single-leader (one node owns writes), multi-leader (several do), and leaderless (any node will take a write if a quorum agrees).

Why it matters

Every durable system you'll design — Postgres, Kafka, DynamoDB, Cassandra, MySQL behind your favorite app — is one of these three shapes. Picking the wrong shape locks you into either painful conflict resolution, a global write bottleneck, or weaker consistency than you advertised. Knowing the trade-offs is the difference between "we lost an order" and "we recovered cleanly".

Real-world examples

  • Single-leader: Postgres streaming replication, MySQL primary/replica, MongoDB replica sets. Easy to reason about; failover is the hard part.
  • Multi-leader: Multi-region Cassandra with LOCAL_QUORUM in each region, CouchDB, calendar/notes sync. Great for write locality; conflicts must be merged.
  • Leaderless: DynamoDB and Cassandra (within a region), Riak. Writes go to N replicas, R+W>N gives quorum reads.

The mental model

SINGLE-LEADER MULTI-LEADER LEADERLESS client client client client | | | | | | v v v v v v [LEADER] --async--> [L-east] <--> [L-west] [N1][N2][N3] / \ | | write to W, [F1] [F2] [F1] [F2] read from R, followers replicate replicate R+W > N -> quorum

In single-leader, all writes serialize through one node, which gives you a clean total order at the cost of a single write bottleneck and failover risk. Multi-leader spreads writes across regions for latency, but two leaders can accept conflicting edits to the same row — you must resolve them (last-write-wins, CRDTs, app logic). Leaderless skips the leader entirely: clients write to W replicas, read from R, and as long as R+W > N, every read intersects at least one fresh replica.

Sync vs async — the buried decision

Within any of the three shapes, replication can be synchronous (leader waits for follower acks before confirming the write) or asynchronous (leader confirms immediately, followers catch up). Sync gives durability but exposes you to follower latency and outages. Async is fast but can lose recently-acked writes during a leader crash. Most production systems pick a middle ground: semi-sync — wait for at least one follower, not all.

Common mistakes

  • Treating "replicas" as automatic high-availability — without proper failover, a primary crash means downtime, not magic.
  • Reading from async replicas and being surprised by stale data (a classic read-your-writes violation).
  • Choosing multi-leader without a real conflict-resolution plan — "last writer wins" silently drops data.
  • Forgetting that quorum (R+W>N) protects against stale reads but not against concurrent writes — you still need versioning.

Question: You're running a global SaaS where each tenant lives in one region but admins occasionally edit settings from anywhere. You want low-latency writes for admins and zero data loss. Which replication shape — and how do you handle the cross-region admin write?
