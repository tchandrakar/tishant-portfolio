CDN basics: how it works, push vs pull, edge caching

Easy cdn caching latency

The idea in one paragraph

A CDN (content delivery network) is a fleet of caches scattered around the world, sitting close to your users. When a user asks for a file, DNS or anycast routes them to the nearest edge node; if that node has the file, it serves it directly — no trip to your origin. The result: faster page loads, lower bandwidth bills, and a shield in front of your servers when traffic spikes.

Why it matters

Most static assets — images, JS bundles, videos, fonts, even API responses — don't change often but get requested millions of times. Sending every request back to a single origin is wasteful and slow for users far from your data center. A CDN turns one global problem ("how do I serve everyone fast?") into a set of local problems ("how do I serve people in Frankfurt fast?"), each solved by the nearest edge.

Real-world examples

  • Cloudflare / Fastly / Akamai: general-purpose CDNs in front of websites and APIs.
  • CloudFront: AWS-native CDN, often paired with S3 for static hosting.
  • Netflix Open Connect: custom CDN appliances installed inside ISPs to serve video locally.
  • Spotify / YouTube: tiered edge caches for media chunks; popular content is pre-warmed close to users.

The mental model

user (Tokyo) | DNS / anycast | v +---------------------+ | EDGE node (Tokyo) | | cache HIT? --> serve from disk | cache MISS -+ +----|----------+ | miss v [REGIONAL shield] -- miss --> [ORIGIN] | S3 / app

An edge HIT serves the byte directly to the user. A MISS triggers a fetch up the hierarchy: edge → regional shield → origin. Once the asset is fetched, every subsequent request in that region is a cheap HIT until the TTL expires or it gets evicted.

Push vs pull

  • Pull CDN (lazy): the edge fetches from origin on first request, then caches. Easy to set up — just point CNAME and add cache headers. First user in each region pays the latency cost.
  • Push CDN (eager): you upload assets to the CDN ahead of time. Used for huge media libraries (video) where you don't want any user to be the unlucky first-miss, and where origin bandwidth is precious.

In practice, most teams start with pull and reach for push only when content is large, predictable, and expensive to miss.

Cache control — the headers that matter

  • Cache-Control: max-age — how long the edge may serve the asset without revalidating.
  • ETag / Last-Modified — let the edge do a cheap conditional GET on revalidation.
  • stale-while-revalidate — serve a stale copy while fetching a fresh one in the background. Smooths spikes nicely.
  • Vary — split caches by Accept-Encoding, device, etc. Use sparingly: it multiplies cache entries.

Common mistakes

  • Caching personalized HTML at the edge (now everyone sees Bob's account page).
  • Setting TTL=0 "to be safe" — every request becomes a MISS and you've turned the CDN into a slow proxy.
  • Forgetting to bust the cache on deploys — users get stale JS that breaks against new APIs. Use content-hashed filenames.
  • Trusting one CDN provider for 100% of traffic; outages happen. Multi-CDN with DNS failover for high-stakes sites.
