Indexes: B-tree, hash, inverted, secondary

Easy indexes storage databases

The idea in one paragraph

An index is a side data structure that maps a value (a name, an ID, a word) to where the matching rows live. Without one, the database has to scan every row; with one, it goes straight to the answer. Different access patterns demand different index shapes — range scans want B-trees, exact lookups love hash maps, full-text search needs inverted lists, and "the same row by a non-primary column" is what secondary indexes are for.

Why it matters

Most "the database is slow" problems are really "the index is missing or wrong." Choosing the right index turns a 10-second sequential scan into a 1-millisecond seek; choosing too many indexes turns every write into a maintenance tax. Knowing the four common shapes lets you reason about query plans, sharding keys, and storage cost without ever opening a profiler.

Real-world examples

  • B-tree: Postgres, MySQL InnoDB, SQL Server — the default index in every relational database.
  • Hash: Redis hash sets, Postgres HASH index, in-memory caches, partitioning hash for sharding.
  • Inverted: Elasticsearch, Lucene, Solr — every search bar you've ever typed into.
  • Secondary: DynamoDB GSI/LSI, Cassandra materialized views, MongoDB compound indexes.

The four shapes at a glance

B-TREE HASH INVERTED SECONDARY ------ ---- -------- --------- [50] h(k) -> bucket "fast" -> [3,7,9] email->rowID / \ "alice"-> #4 "boat" -> [2,9] name ->rowID [25] [75] "bob" -> #11 "cat" -> [1,8] points to / \ / \ primary key [10][30][60][90] range scans OK exact lookup text / multi-term alt access path O(log n) O(1) token postings same row, new key

A B-tree keeps keys sorted in a shallow tree, so range queries (WHERE date BETWEEN ...) and ORDER BY are cheap. A hash index hashes the key into a bucket — O(1) for equality, useless for ranges. An inverted index lists, for each term, every document that contains it; intersect a few lists and you have search. A secondary index is just any of the above built on a non-primary column, pointing back at the primary row.

Cost: every index taxes writes

Every INSERT, UPDATE, or DELETE must update every index on the table. Five indexes means roughly six writes per row change. So the rule is: index what you query often, not what you might query someday. In NoSQL especially, secondary indexes are async and eventually consistent — your "read by email" can lag behind "read by user_id".

Composite indexes & the leftmost-prefix rule

An index on (country, city, age) can serve queries on country, (country, city), or (country, city, age) — but not a query on city alone. The columns are concatenated left-to-right in the tree. Order them most-selective-first, and put equality columns before range columns.

Common mistakes

  • Indexing every column "just in case" — write amplification eats throughput.
  • Using a hash index when you need range scans (now WHERE age > 30 falls back to a full scan).
  • Forgetting that LIKE '%foo' can't use a B-tree — use trigram or inverted indexes for substring search.
  • Trusting that DynamoDB GSIs are immediately consistent — they're not.
  • Over-relying on covering indexes; they save reads but bloat write paths.
