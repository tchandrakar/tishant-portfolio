Load balancers: L4 vs L7, algorithms

Easy load-balancing networking routing

The idea in one paragraph

A load balancer is the traffic cop sitting in front of your fleet of servers. Clients talk to it; it picks a healthy backend and forwards the request. The two big choices are where in the network stack it operates (L4 = transport / TCP, L7 = application / HTTP) and how it picks a backend (round-robin, least connections, consistent hashing, and friends). These choices decide what your load balancer can see and how evenly traffic actually distributes.

Why it matters

Without a load balancer, horizontal scaling is impossible — clients would have to know every server's IP. With one, you get a single virtual endpoint, automatic failover when a node dies, rolling deploys without downtime, and the ability to slice traffic by URL, header, or cookie. Almost every web architecture you'll design starts with "users hit a load balancer, which routes to..." — so it pays to know exactly which kind to draw.

Real-world examples

  • AWS NLB / GCP TCP LB: classic L4 — fast, blind to HTTP, used for raw TCP, gaming, databases.
  • AWS ALB / NGINX / HAProxy in HTTP mode: L7 — can route by path, host, header; can terminate TLS.
  • Envoy / Istio: L7 service-mesh proxies with retries, circuit breaking, mTLS.
  • Memcached client / Cassandra coordinator: consistent hashing — same key always goes to the same node, so caches stay warm.

The mental model

client | v +---------------+ L4: routes by (src IP, src port, dst IP, dst port) | Load Balancer | L7: routes by URL path, Host header, cookie... +---------------+ / | \ v v v [B1] [B2] [B3] <-- backends, picked by an algorithm: - round-robin: B1, B2, B3, B1... - least-conn: whichever has fewest open conns - hash(key): user 42 -> always B2

L4 is a dumb-but-fast packet forwarder; it can't read the HTTP body. L7 parses the request, so it can do path-based routing (/api → API fleet, /static → CDN origin) at the cost of more CPU.

Common mistakes

  • Using round-robin for long-lived connections (WebSockets, gRPC). One unlucky node ends up with all the heavy clients — use least-connections instead.
  • Putting an L4 LB in front of services that need sticky sessions or path routing. You'll either build your own routing in the app or have to upgrade.
  • Forgetting health checks. A "healthy" port-open check passes even when the app is deadlocked. Use an HTTP /healthz that exercises a real dependency.
  • Ignoring consistent hashing. With plain hash-mod-N, adding one node remaps almost every key — your cache hit ratio collapses.

Try it yourself

Question: You're running a chat service with 1M concurrent WebSocket connections across 20 backend servers. You currently use round-robin DNS. You add 5 more servers and traffic stays unbalanced — the old servers are still pegged. Why? Which algorithm and which LB layer would you switch to?
Hint: Round-robin balances new connections, not active ones. Long-lived connections never rebalance on their own.
