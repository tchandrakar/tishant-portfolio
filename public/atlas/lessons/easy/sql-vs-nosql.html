Database choice: SQL vs NoSQL, ACID vs BASE

Easy databases acid nosql

The idea in one paragraph

SQL databases store data in fixed tables with strong relationships and give you ACID guarantees — every transaction is Atomic, Consistent, Isolated, and Durable. NoSQL databases relax the schema and often the consistency, in exchange for horizontal scale and flexible data shapes; they typically advertise BASE — Basically Available, Soft-state, Eventually consistent. Choosing between them is mostly about what you need more: rock-solid invariants on every write, or the ability to grow data and traffic by 100x without rewriting.

Why it matters

The database is almost always the hardest part of any system to change later — schemas calcify, queries get tuned, ops teams build muscle memory. Picking the wrong category early means you pay for it every quarter. The good news: 80% of the choice is decided by two questions — "do I need multi-row transactions?" and "do I expect the data to outgrow one machine?".

Real-world examples

  • Postgres / MySQL (SQL, ACID): banking, ledgers, anything where two rows must update or neither does.
  • DynamoDB / Cassandra (NoSQL, BASE): shopping cart, user profile, IoT telemetry — huge volume, simple key-based access.
  • MongoDB (document, tunable): flexible schemas; ACID on a single document, multi-doc since 4.0 with caveats.
  • Redis (key-value, in-memory): caches, leaderboards, rate limiters; durability is opt-in.
  • Neo4j (graph): social graphs, fraud detection — relationships are first-class.

The mental model

ACID (SQL world) BASE (NoSQL world) +---------------------+ +---------------------+ | A: all-or-nothing | | Basically Available | | C: invariants hold | | Soft state | | I: txns don't see | | Eventually | | each other's WIP | | consistent | | D: survives crash | | | +---------------------+ +---------------------+ strict, single-master tunable, many nodes scales VERTICALLY scales HORIZONTALLY Decision flow: Need multi-row txns / strict invariants? --> SQL Need >1 TB and >50k QPS at low cost? --> NoSQL Need both? --> NewSQL (Spanner, CockroachDB) or careful sharded SQL

Notice that ACID and BASE aren't opposites so much as endpoints of a spectrum. Modern systems (Spanner, CockroachDB, FoundationDB) try to give you ACID at horizontal scale by paying with latency.

Common mistakes

  • Picking NoSQL "for scale" when you have 50 GB of data and need transactions. You're trading what you have for what you don't need.
  • Picking SQL and then doing application-side joins because you're afraid of relational queries. The DB does that better.
  • Confusing "schemaless" with "no schema". Document stores still have an implicit schema — it just lives in your application code, untyped.
  • Assuming "eventual consistency" means "a few milliseconds". Under partition, eventual can mean minutes — design for it.
  • Ignoring access patterns. NoSQL rewards designs where the query is known up front; if you need ad-hoc analytics, that's a warehouse, not a key-value store.

Try it yourself

Question: You're building a ride-sharing app's two core tables: (1) trips, where money moves between rider and driver and a refund must never produce a partial state, and (2) driver_locations, updated every 4 seconds for 10M drivers worldwide. Would you put both in one database? If you split them, which engine for each — and why?
Hint: Money is ACID-shaped. Location pings are firehose-shaped. They have nothing in common except the user.
