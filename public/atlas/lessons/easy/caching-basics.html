Caching basics: write-through, write-back, write-around, eviction

Easy caching performance consistency

The idea in one paragraph

A cache is a small, fast store that sits between your slow data source (disk, database, remote API) and your application, holding the items most likely to be asked for next. The clever part isn't the cache itself — it's the write strategy (how new data gets in) and the eviction policy (what gets thrown out when the cache fills up). Get those wrong and your cache becomes either stale or useless.

Why it matters

Most reads in real systems are repeats: the same product page, the same user profile, the same trending tweet. Caching those reads can drop database load by 90%+ and cut latency from tens of milliseconds to hundreds of microseconds. Almost every interview design ends with "and we put Redis in front of it" — the question is whether you can defend which cache pattern and why.

Real-world examples

  • Write-through (Facebook TAO): every write goes to cache and DB synchronously. Cache is always fresh; writes are slower.
  • Write-back / write-behind (Linux page cache, some Redis setups): write to cache, flush to DB asynchronously. Fast writes, but a crash can lose data.
  • Write-around (CDN origin caches): writes skip the cache and go straight to DB; cache only fills on read miss. Good for write-heavy, read-rare data.
  • LRU eviction (Memcached default): drop the least recently used key when full.
  • LFU (Redis allkeys-lfu): drop the least frequently used — better for hot-key workloads.

The mental model

WRITE-THROUGH WRITE-BACK WRITE-AROUND app -> cache -> DB app -> cache ~> DB app -----> DB (sync, fresh) (async flush, fast, | risk of loss) v read miss -> DB -> cache EVICTION when cache is full: LRU: kick out who-was-touched-longest-ago LFU: kick out who-was-touched-least-often TTL: kick out anyone whose timer ran out

The strategies trade three things against each other: write latency, read freshness, and risk of data loss. Pick the one that matches your workload — there is no universal winner.

Common mistakes

  • Caching with no TTL. Stale data lives forever; debugging becomes "did you flush the cache?".
  • Cache-stampede on miss: 10,000 requests for the same expired key all hit the DB at once. Fix with single-flight / request coalescing.
  • Using write-back for anything you can't afford to lose. A node crash drops un-flushed writes.
  • Caching the wrong layer. A 100% hit-rate cache in front of a 5ms query saves nothing; cache the 500ms join or the 200ms remote call.
  • Ignoring negative caching. Repeated 404s for the same key still hammer the DB unless you cache the "not found" too.

Try it yourself

Question: You're designing a product-detail page. Reads are 100x writes, but when an item's price changes, customers must never see the old price for more than 1 second. Which write strategy do you pick — write-through, write-back, or write-around — and what TTL do you set? What do you do on a cache miss to avoid a stampede?
Hint: Freshness requirement rules out one option immediately. For stampedes, look up "request coalescing" or "single-flight".
