LSM-trees & SSTables: turning random writes into sequential ones

Medium storage lsm sstable compaction rocksdb

The idea in one paragraph

An LSM-tree (Log-Structured Merge tree) is a write-optimized storage shape that buffers mutations in a sorted in-memory structure (the memtable), persists them as immutable, sorted on-disk files (SSTables — Sorted String Tables), and reconciles the resulting overlap with a background compaction process. Reads check the memtable first, then progressively older SSTables; bloom filters prune levels that can't possibly contain the key. The whole point: turn the workload's random writes into one sequential append (cheap on every storage medium ever invented) and pay the bookkeeping cost off the critical path.

What LSM + SSTables are used for

  • Write-heavy KV stores: RocksDB, LevelDB, Cassandra, ScyllaDB, HBase, BigTable, Riak — all LSM under the hood.
  • Embedded engines inside larger systems: RocksDB powers DynamoDB's storage nodes, CockroachDB, TiKV, Kafka Streams state stores, Flink's RocksDBStateBackend, MySQL MyRocks, MongoDB's WiredTiger LSM mode.
  • Time-series & logging: InfluxDB, Prometheus TSDB, ClickHouse's MergeTree all share LSM DNA — high ingest, range scans on recent data.
  • Anything ingest-bound: ad-click pipelines, IoT telemetry, observability stacks, blockchain state DBs (Geth's Pebble), search index segments (Lucene's segment merge is structurally an LSM).
  • NOT a fit for: read-heavy point lookups on a stable corpus (B-tree wins), workloads with cheap in-place updates (Postgres heap), or anywhere predictable p99 matters more than throughput.

Why it exists: the SSD/HDD reality check

A spinning disk does ~100 random IOPS but ~100 MB/s sequential. SSDs flatten the gap but still pay write amplification on small random writes (the FTL has to erase a 256 KB block to update a 4 KB page). Every storage medium rewards sequential append. B-trees, by contrast, do random in-place page writes — fast for reads, but every update walks from root to leaf and rewrites a page. LSMs trade read amplification (multiple files to consult) for write amplification you can schedule (compaction runs in the background, not on the write path).

The four building blocks

  1. WAL (write-ahead log): every put/delete is appended to a single log file with one fsync (or batched via group commit). On crash, replay the WAL to rebuild the lost memtable.
  2. Memtable: sorted in-memory structure (skiplist in LevelDB/RocksDB). Writes are O(log n); a single sealed flush dumps the whole thing to disk in key order.
  3. SSTable (Sorted String Table): immutable file of sorted (key, sequence#, type, value) records, plus a sparse index, a bloom filter, and a footer. Once written, it's never modified — only deleted by compaction.
  4. Compaction: background job that merges overlapping SSTables, resolves duplicates (newest seq# wins), drops tombstones whose deletion is provably safe, and produces fewer, larger files at the next level.

SSTable file format (the bytes on disk)

+----------------+----------------+--------+--------+--------+--------+ | Data Block 0 | Data Block 1 | ... | Index | Filter | Footer | +----------------+----------------+--------+--------+--------+--------+ ~32 KB each restart points (last_key bloom magic + prefix-compressed every 16 keys -> offset) per SST offsets CRC32C checksum for binary search binary searchable
  • Data blocks: sorted (key, seq, type, value) tuples with prefix compression — adjacent keys share a common prefix, so user:1234, user:1235 store the diff, not the whole key. Restart points every 16 keys let you binary-search inside a block.
  • Index block: one entry per data block — (last_key_of_block -> block_offset). To find a key: binary search the index, load the one block that could contain it.
  • Filter block: a bloom filter (typically 10 bits/key, ~1% false positive rate) per SSTable, or partitioned per data block. If the bloom says "no," skip the file entirely — saves a disk read.
  • Footer: fixed-size trailer with magic bytes + offsets to index and filter, so opening an SSTable is one positional read of the last 48 bytes.

The write path (one put at a time)

  1. Append (seq, op, k, v) to the WAL. fsync (or join a group commit batch).
  2. Insert into the memtable (skiplist). Acknowledge the client.
  3. If memtable size > threshold (e.g. 64 MB): seal it, allocate a new one, hand the sealed copy to the flush thread.
  4. Flusher writes the sealed memtable as one SSTable into L0. The L0 file's range overlaps with other L0 files — that's expected; L0 is the spillover zone.
  5. Once the SSTable is durable, its WAL segment can be deleted.

The read path (with bloom filters)

get(k): 1. Look in active memtable -> hit? return value or tombstone 2. Look in immutable memtable(s) -> hit? return 3. For each L0 SSTable, newest first: bloom says "maybe"? load index, load block, search. 4. For L1..L6 (each level is non-overlapping): binary search the level's manifest for the one file whose key range contains k bloom says "maybe"? load index, load block, search. 5. If we never found it -> NotFound.

Without bloom filters, a missing key forces a probe of every level — that's the famous LSM read amplification. With 10 bits/key bloom filters, ~99% of "key absent" lookups skip the disk read entirely. This is the single highest-leverage optimization in any LSM (DDIA Ch. 3). The cost: ~1.25 GB of RAM per billion keys, per replica.

Compaction: where the real engineering happens

Without compaction, the LSM degenerates into an unbounded pile of overlapping files, reads slow to a crawl, and disk fills with superseded versions. The compactor's job: merge overlapping SSTables, drop dead data, and produce fewer, larger, non-overlapping files at deeper levels.

Tombstones, MVCC, and why deletes are weird

You can't delete a key from an immutable file. So a delete(k) writes a tombstone — a record that says "as of seq N, key k is dead." The tombstone shadows older writes during reads. Compaction can drop the tombstone only when it can prove no older live entry survives below — typically when a compaction reaches the bottom level. Until then, the tombstone occupies space and counts against read amp. Range tombstones (RocksDB) encode mass deletes as one (start, end, seq) record instead of N point tombstones — critical for time-series TTL.

The same machinery delivers MVCC: each write carries a monotonic 64-bit sequence number; a snapshot read pins a seq# and the iterator skips entries with seq > snapshot. Compaction respects the oldest live snapshot — a long-running scan can pin garbage in place, which is why "snapshot leak" is a real ops problem on any LSM.

The three amplifications (the LSM tuning lens)

  • Write amp: bytes the engine writes ÷ bytes the user wrote. Compaction rewrites every byte once per level it traverses. Higher fan-out = higher write amp but fewer levels.
  • Read amp: blocks read per logical lookup. Without bloom filters: O(levels). With them: ~1 in steady state for misses, 1–2 for hits.
  • Space amp: bytes on disk ÷ bytes of live data. Tombstones, superseded versions, and in-flight compactions all inflate this.

You can't minimize all three at once. Leveled trades write amp for space amp; size-tiered does the opposite. Tuning is mostly negotiating the triangle for your workload.

Common production failure modes

  • L0 stalls / write throttle: if compaction can't keep up with flushes, L0 grows tall, the engine throttles foreground writes, and your tail latency explodes. Mitigation: more flush threads, larger memtable, sub-compaction parallelism, faster disk.
  • Tombstone backlog: mass deletes written as point tombstones can't be dropped until they reach the bottom level — meanwhile every scan walks past them. Use range tombstones; consider TTL/manual compaction.
  • Bloom filter blow-up: at 10 bits/key, 10 B keys = 12.5 GB RAM. Use partitioned bloom + ribbon filters for very large datasets.
  • Cache pollution from compaction: compaction streams hundreds of MB/s through the page cache and evicts hot user blocks. Use O_DIRECT for compaction I/O or pin the user block cache.
  • Snapshot leaks: a forgotten iterator pins old SSTables alive; disk fills. Always set snapshot timeouts and watch num_snapshots in the metrics.
  • WAL + data on the same disk: fsync from the WAL contends with sequential compaction writes. Put WAL on a separate fast disk.

Tuning cheatsheet (RocksDB-flavored)

  • Write-heavy, throughput > latency: size-tiered/universal compaction, larger memtable (256 MB+), more flush threads, maybe disable bloom filters on bottom level.
  • Read-heavy point lookups: leveled compaction, 10–16 bits/key bloom, generous block cache, partitioned indexes, prefix bloom if you scan by prefix.
  • Time-series with TTL: universal or FIFO, range tombstones, smaller bloom (or none on cold levels), aligned write-time clustering.
  • Mixed OLTP-ish workload: leveled, group commit on, MVCC snapshots short-lived, separate WAL disk, watch L0 file count as the leading indicator of trouble.

Common mistakes

  • Picking an LSM for a read-heavy point-lookup workload on a stable corpus — Bloom filters help, but B-trees still win on p99.
  • Disabling fsync to "speed things up" — durability evaporates on the first power loss; the correct lever is group commit.
  • Treating compaction as a black box — if you don't watch L0 file count, write amp, and pending-compaction-bytes, you'll only notice the problem when writes stall.
  • Mass-deleting with point tombstones and wondering why scans got slower — use range tombstones.
  • Leaking long-running snapshots and watching disk usage climb — every iterator/snapshot pins SSTables alive.
  • Thinking SSTables can be edited in place — they cannot. Updates go to a new memtable; the old value dies in compaction.

Try it yourself

Question: Your service ingests 5 KB IoT telemetry records at 200k writes/sec, queried mostly by recent (timestamp, device_id) ranges, occasionally by device_id alone. The cluster has 2 NVMe disks per node and 64 GB RAM. Choose a compaction strategy, key encoding, and memtable size. Where does the WAL live, and how do you bound p99 read latency?
Hint: Pick universal compaction with range tombstones for cheap TTL deletion. Encode the key as (device_id, timestamp_desc) so recent ranges are co-located in young SSTables. Memtable 256 MB, 4 flush threads. WAL on the second NVMe to avoid fsync contention with compaction writes. For p99: 10 bits/key bloom on all but the bottom level, partitioned index, block cache 16 GB pinned to user data, O_DIRECT for compaction I/O. Watch num_l0_files and pending_compaction_bytes — they spike before latency does.
