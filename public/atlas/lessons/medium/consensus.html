Consensus: Raft & Paxos in plain English

Medium consensus raft paxos

The idea in one paragraph

Consensus is the problem of getting a group of unreliable machines to agree on a single value, in a single order, even though some of them will crash, lag, or disagree about what time it is. Paxos proved it was possible (1989). Raft (2014) made it teachable. Both elect a leader who serializes writes into a replicated log; followers acknowledge each entry, and an entry is "committed" once a majority (a quorum) has it on disk. From the outside, the cluster looks like one consistent machine that never forgets.

Why it matters

Anything that needs to be true everywhere — a config registry, a primary election, a distributed lock, a row in a CP database — sits on top of consensus. etcd, ZooKeeper, Consul, CockroachDB, Spanner, and Kafka's KRaft mode are all consensus systems wearing different hats. Without consensus, two nodes can both believe they're leader (split-brain) and corrupt your data forever.

Real-world examples

  • etcd / Consul: Raft for cluster metadata; powers Kubernetes' control plane.
  • ZooKeeper: Zab (a Paxos cousin) for coordination, used by HBase and old-school Kafka.
  • Spanner: Multi-Paxos per shard, with TrueTime for external consistency.
  • CockroachDB / TiDB: Raft per-range, hundreds of consensus groups per cluster.

The mental model (Raft)

term=7, leader=N1 +------+ +------+ +------+ | N1 |<------>| N2 |<------>| N3 | +------+ +------+ +------+ | | | log: [x=1, y=2, z=3] [x=1, y=2, z=3] [x=1, y=2] ^^ lagging follower 1. Client write -> leader N1 2. N1 appends to its log, sends AppendEntries to N2, N3 3. Once 2 of 3 (majority) ack -> entry committed 4. Leader applies to state machine, returns to client 5. Followers apply once they learn commitIndex

If N1 dies, N2 or N3 starts an election by bumping the term and asking for votes. A candidate wins only if a majority votes for it AND its log is at least as up-to-date as theirs — that's how Raft guarantees no committed entry is ever lost.

Paxos vs Raft: same shape, different vibes

  • Paxos is two phases: prepare (claim a ballot) and accept (propose a value). Multi-Paxos amortizes phase 1 by keeping a stable leader.
  • Raft bakes leadership in from the start: leader election, log replication, safety. It's Multi-Paxos with the rough edges sanded off.
  • Both need 2f+1 nodes to tolerate f failures. 3 tolerates 1, 5 tolerates 2.
  • Both require persistent state (current term, voted-for, log) — fsync before responding.

Costs and trade-offs

  • Latency: every write is a round-trip to a majority. In a 3-region deployment, that's cross-region RTT on the critical path.
  • Throughput ceiling: a single leader serializes writes. Sharding into many Raft groups (one per range) is the standard fix.
  • Reads: linearizable reads need either a quorum read or a "leader lease." Stale reads from any follower are cheap but eventually consistent.
  • Membership changes (adding/removing nodes) are the hardest part to get right — use joint consensus.

Common mistakes

  • Running an even number of nodes (4 nodes still need 3 votes — same fault tolerance as 3, double the cost).
  • Putting the consensus log on the same disk as the application — fsync contention kills latency.
  • Assuming "leader = always correct"; a partitioned old leader can still serve stale reads without a lease.
  • Trying to scale write throughput by adding nodes — more followers = slower commits, not faster.

Try it yourself

Question: A 5-node Raft cluster is split 2-3 by a network partition. Which side keeps serving writes? What happens to clients connected to the minority side?
Hint: Only the side with a majority (3) can elect a leader and commit. The minority's leader (if any) steps down once it can't reach a quorum on heartbeats; clients there see timeouts or "not leader" errors. When the partition heals, the minority's uncommitted log entries are overwritten.
