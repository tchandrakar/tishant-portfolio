Idempotency & exactly-once delivery

Medium idempotency delivery-semantics retries

The idea in one paragraph

An idempotent operation produces the same result whether you run it once or one thousand times. Distributed systems retry — networks drop packets, clients time out, brokers redeliver — so every write that crosses a wire must either be naturally idempotent (e.g., SET balance = 100) or made idempotent with a dedup key. "Exactly-once delivery" is a marketing phrase: what real systems give you is at-least-once delivery + idempotent processing = effectively-once.

Why it matters

Without idempotency, a flaky network charges your customer twice, a Kafka rebalance double-credits an account, and a retried webhook ships the same package to two warehouses. Payment APIs, order systems, and message consumers all depend on safe retries. Get this wrong and you ship duplicate side effects to the real world — money, packages, emails — that you cannot recall by rolling back a transaction.

Real-world examples

  • Stripe: the Idempotency-Key HTTP header. Stripe stores the response for 24 hours and replays it byte-for-byte on retry.
  • Kafka: the idempotent producer attaches a producer ID + sequence number so the broker discards duplicates within a session.
  • SQS: at-least-once queue; consumers must dedup by MessageId or a business key to avoid double-processing.
  • DynamoDB: conditional writes (attribute_not_exists(orderId)) make insertion idempotent.

The mental model

Client Server | POST /charge | | Idempotency-Key=K | |--------------------->| | | seen[K]? --no--> charge card | | store seen[K] = response | 200 OK (resp R) |<----- write |<---------------------| | | | (timeout, retry) | | POST /charge K | |--------------------->| | | seen[K]? --YES--> return cached R | 200 OK (resp R) | |<---------------------|

The dedup table is the entire trick. The first request goes through the real path; every retry short-circuits to the cached response. The key must be chosen by the client so retries reuse it — a server-generated UUID defeats the purpose.

Designing idempotent operations

  • Use natural keys when possible. UPSERT user(id=42) is safer than INSERT + retry.
  • Single-row dedup table. (idempotency_key PRIMARY KEY, response_blob, expires_at) — write it in the same transaction as the side effect.
  • Scope the key. Per-customer or per-tenant, not global, to avoid one tenant evicting another's keys.
  • Pick a TTL. Long enough to outlive client retry windows (24h is typical), short enough to keep the table small.
  • Hash the request body. If a client reuses a key with a different payload, return 409 Conflict rather than overwrite.

Common mistakes

  • Believing brokers offer "exactly-once." They offer at-least-once; you provide the once-ness.
  • Storing the dedup row after the side effect — a crash in between leaves a duplicate-charge gap.
  • Using auto-incrementing IDs as idempotency keys — every retry gets a new key.
  • Not handling concurrent retries: two in-flight requests with the same key need a row-level lock or unique constraint.
  • Ignoring partial failures inside multi-step handlers — wrap them in a saga or outbox.

Try it yourself

Question: Your payments service consumes a Kafka topic of ChargeRequested events. The broker may redeliver after a consumer crash. The handler calls Stripe and writes a charges row. How do you guarantee a customer is never charged twice, even if the consumer dies between the Stripe call and the DB write?
Hint: Reserve the idempotency key before calling Stripe. Use the Kafka (topic, partition, offset) tuple, or a hash of the event ID, as the Stripe Idempotency-Key. On replay, Stripe returns the original response — no second charge.
