Distributed transactions: 2PC, Saga, Outbox

Medium transactions saga outbox 2pc

The idea in one paragraph

A single-database transaction is easy: BEGIN, do stuff, COMMIT, all-or-nothing. Once your "transaction" spans two services or two databases, atomicity stops being free. Three patterns dominate how real systems get out of this hole: two-phase commit (2PC) coordinates a synchronous all-or-nothing across participants, Saga replaces the rollback with explicit compensating actions, and the Outbox pattern guarantees that "DB write + message publish" happens together by sneaking the message into the same DB transaction.

Why it matters

Every microservice system eventually needs "create order AND charge card AND reserve inventory AND send email" to behave like one operation. Get this wrong and you get ghost charges, double-shipped orders, or rows that swear an event was published when it never was. These three patterns are the toolkit; choosing between them is a tradeoff of latency, blast radius, and operational complexity.

Two-phase commit (2PC)

A coordinator asks every participant: "can you commit?" — phase 1 (prepare). If all say yes, it tells everyone "commit" — phase 2. If any says no, "abort". Strong: real atomic semantics. Weak: blocking — if the coordinator dies after participants prepared, they hold locks until it returns. Latency = max(participants) × 2 round trips.

  • Use it: small fixed set of participants, same data center, low rate, when you genuinely need ACID across services (rare).
  • Don't use it: across the public internet, with flaky participants, or above ~100 tps.

Saga

A saga is a sequence of local transactions where each step has a compensating action. If step 4 fails, you run compensations for steps 3, 2, 1 in reverse. There are two flavors: orchestration (a central state machine drives the steps) and choreography (each service reacts to the previous step's event). No locks, no coordinator outage — but you give up isolation: other transactions can see intermediate state.

  • Use it: long-running business workflows (booking a trip, onboarding a customer), cross-service flows where blocking is unacceptable.
  • Watch out for: compensation isn't always possible (you can't un-send an email). Design idempotent steps and "semantic" compensations (refund instead of un-charge).

The Outbox pattern

Problem: you want to "update the orders table AND publish OrderCreated to Kafka" atomically. Two systems, two failure modes. Outbox solves it by writing the event into an outbox table in the same DB transaction as the order. A separate poller (or CDC stream like Debezium) reads the outbox and publishes to the broker, marking rows as sent. The event is never lost and never published without the row.

The mental model

2PC Saga (orchestrated) Outbox coord ──prepare──► A,B,C orch ──step1──► svc A tx{ ◄──yes/no─── ◄──ok── INSERT order; ──commit/abort─► ──step2──► svc B INSERT outbox; ◄──fail── } blocks if coord dies compensate step1 relay → broker strong atomicity eventual + idempotent at-least-once

How to choose

  • Need real atomicity, few participants, ok with locks? 2PC.
  • Long workflow, need to stay available, ok with eventual + compensations? Saga.
  • Just need "DB write + event publish" to be reliable? Outbox. (Often combined with Saga.)
  • Truly stateless cross-service writes? Often you don't need a transaction at all — just idempotent retries.

Common mistakes

  • Reaching for 2PC across services when a Saga + Outbox would do, then watching the system stall whenever the coordinator hiccups.
  • Writing compensations that aren't idempotent — replays double-refund customers.
  • Publishing to Kafka first, then writing the DB. If the DB write fails, you've leaked a phantom event.
  • Forgetting to mark outbox rows processed atomically — the relay republishes on restart and downstream sees duplicates without dedup.

Try it yourself

Question: A checkout flow needs to: (1) charge the card, (2) decrement inventory, (3) create the order row, (4) send confirmation email. Each is a different service. Sketch the transaction strategy — which pattern, what are the compensations, where does the outbox live?
Hint: An orchestrated saga over (charge → reserve → create → email), with the orchestrator's state in its own DB and an outbox in the order service for the OrderCreated event. Compensation for charge is refund; for inventory is release; email has no rollback so it goes last.
