Distributed time: Lamport, vector clocks, HLC

Medium clocks causality ordering

The idea in one paragraph

Wall-clock time on distributed nodes is a lie — NTP drifts, leap seconds happen, VM hosts hiccup. To order events correctly, systems use logical clocks that capture causality (who happened before whom) instead of trusting the wall. Lamport timestamps give a total order. Vector clocks detect concurrency. Hybrid Logical Clocks (HLC) combine logical safety with human-readable wall-clock values.

Why it matters

You can't say "the latest write wins" without defining "latest." If node A's clock is 200 ms ahead of node B's, a write at A timestamped after B's write but actually before it (in real causality) silently overwrites the truth. Replication conflicts, distributed snapshots, MVCC visibility, debug log alignment — they all need a notion of time that doesn't lie.

Real-world examples

  • DynamoDB / Riak: vector clocks (now version vectors) to detect concurrent updates and surface them as siblings.
  • CockroachDB: HLC for transaction timestamps; close-enough to wall time but provably monotonic.
  • Cassandra: client-supplied microsecond wall timestamps for last-write-wins — famously fragile.
  • Spanner: TrueTime — a physical clock with a known uncertainty interval, backed by GPS and atomic clocks.

Lamport timestamps

Node A: 1 -- 2 -- 3 -- ... -- 7 (recv from B, max(3,6)+1 = 7) \ / send msg ts=3 / \ / Node B: 1 -- 2 -- 3 -- 4 -- 5 -- 6

Each node holds a counter. On every local event, increment. On send, attach the counter. On receive, set local = max(local, received) + 1. Result: if event X causally precedes Y, then ts(X) < ts(Y). The converse is not true — small timestamp doesn't prove causality. Ties are broken by node ID for a total order.

Vector clocks

A Lamport stamp can't tell concurrent events apart. A vector clock is one counter per node: [A:3, B:5, C:2]. Update rules:

  • Local event on A: increment A's slot.
  • Send: attach the whole vector.
  • Receive: take elementwise max, then increment your own slot.

Compare two vectors: if every component of V1 ≤ V2 (and one is strictly less), V1 happened-before V2. Otherwise they're concurrent — the system must surface a conflict (siblings) or merge (CRDT). The cost: vector size grows with cluster size, which is why production systems prune or use dotted version vectors.

Hybrid Logical Clocks (HLC)

HLC is the practical compromise: a 64-bit timestamp where the high bits are a wall-clock time (ms) and the low bits are a logical counter. On every event, take max(physical_now, last_hlc) ; if equal, bump the logical counter. Result: timestamps are within a few ms of wall-clock (so logs and metrics line up), but strictly monotonic and causally consistent across nodes that exchange messages.

HLC = (wall_ms : logical_counter) ---------------------------------- A local event @ wall=1000: (1000, 0) A sends to B, attaches (1000, 0) B's wall=999, receives: (1000, 1) <-- max(physical, recv) + bump B local event @ wall=1001: (1001, 0)

Common mistakes

  • Using System.currentTimeMillis() as a write timestamp — clock skew silently loses data.
  • Assuming Lamport order implies causality — it doesn't, only the reverse.
  • Letting vector clocks grow unbounded as nodes come and go — prune retired entries.
  • Trusting NTP "to the millisecond" — typical skew is 10–100 ms; cloud VMs can spike to seconds.
  • Using HLC across data centers without bounded NTP drift — uncertainty creeps into the physical part.

Try it yourself

Question: A user updates their profile twice in 200 ms. Replica A receives name=Bob at wall=12:00:00.100 (its clock). Replica B receives name=Robert at wall=12:00:00.050 (its clock, but actually after A's write in real time — B's clock is slow). Last-write-wins by wall time picks "Robert." Is that correct? How would vector clocks resolve it?
Hint: Wall-time LWW is wrong: the slow clock loses real updates. With vector clocks, the two writes from the same client would be causally ordered (each carries the previous version vector), so the second write strictly dominates the first regardless of wall time.
