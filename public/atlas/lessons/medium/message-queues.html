Message queues: Kafka vs RabbitMQ vs SQS — when to use which

Medium queues kafka rabbitmq sqs

The idea in one paragraph

A message queue decouples producers from consumers so they can run at different speeds, fail independently, and scale on different axes. The three workhorses each made a different bet: Kafka treats the queue as a durable replayable log, RabbitMQ treats it as a smart router with rich exchange semantics, and SQS treats it as a fully managed dumb pipe with at-least-once delivery and visibility timeouts. Picking the wrong one is a quiet but expensive mistake — you find out months later when you can't replay, can't fan out, or can't keep up.

Why it matters

Almost every distributed system has a queue at the center: order pipelines, analytics ingestion, email sending, audit logs, search indexing. Whether you can do exactly-once, replay last week's traffic, or push 1M msgs/sec depends entirely on which broker is sitting there. Interviewers ask about this because the answer reveals whether you understand throughput vs routing vs ops cost.

Real-world examples

  • Kafka: LinkedIn activity stream, Uber trip events, any event-sourcing or stream-processing pipeline. High throughput, ordered per partition, retained for days.
  • RabbitMQ: Task queues for background jobs (think Celery/Sidekiq workloads), RPC-style request/reply, complex routing where one event needs to land in five different queues with different filters.
  • SQS: AWS-native glue between Lambda, EC2 workers, and S3 events. Standard queues for at-least-once; FIFO queues for ordered, deduplicated processing at lower throughput.

The mental model

Kafka (log) RabbitMQ (router) SQS (managed pipe) ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ P1 P2 P3 P4 …│ ←offset │ exch ─► queue│ │ queue │ │ retained 7d │ │ ─► queue│ │ visibility │ │ replay any │ │ ─► queue│ │ timeout │ └──────────────┘ └──────────────┘ └──────────────┘ pull, ordered push, routed poll, opaque per partition via bindings at-least-once

Kafka stores; consumers track their own offset and can rewind. RabbitMQ delivers and forgets — once acked the message is gone. SQS hides the broker entirely: you poll, process within the visibility timeout, and delete; if you don't delete, the message reappears.

How to choose

  • Need replay or stream processing? Kafka. Offsets + retention give you a time machine.
  • Need rich routing (topic + headers + per-message TTL + priority)? RabbitMQ. Exchanges and bindings express what you'd otherwise hand-code.
  • Need zero ops on AWS? SQS. Pay-per-message, infinite scale on standard queues, no brokers to babysit.
  • Throughput target > 100k msgs/sec sustained? Kafka, by a wide margin. RabbitMQ tops out around 50k/node before you start fighting it.
  • Strict per-key ordering? Kafka (partition by key) or SQS FIFO (group id). RabbitMQ ordering breaks once you add competing consumers.

Common mistakes

  • Using Kafka for a 100-msg/day job queue. You'll spend more on ZooKeeper/KRaft than on the work itself.
  • Using RabbitMQ as an event log — nothing stops a consumer from acking and losing the only copy.
  • Forgetting SQS visibility timeout. If your worker takes 60s but timeout is 30s, the message gets redelivered and you process it twice.
  • Treating "exactly-once" as a checkbox. It's a property of producer + broker + consumer + sink together, not the broker alone.

Try it yourself

Question: You're building a payment-event pipeline. Auditors require replaying any day in the last 90 days into a new fraud model. Risk team wants to fan the same event out to four downstream systems. Which broker, and how do you lay out topics/partitions?
Hint: Replay + fanout + 90-day retention is one broker's home turf. Partition by what gives you ordering where it matters (account id) without creating hot keys.
