Event sourcing & CQRS

Medium event-sourcing cqrs audit

The idea in one paragraph

Event sourcing stores the history of changes instead of the current state. Every mutation is an immutable event — OrderPlaced, ItemShipped, RefundIssued — appended to a log. Current state is a projection: replay the events through a fold and you get the current row. CQRS (Command Query Responsibility Segregation) is the natural partner: commands write to the event log, queries read from one or many derived views — each tuned for its read pattern.

Why it matters

For domains where "what changed and why" is part of the product — banking, accounting, healthcare, audit-heavy SaaS — event sourcing is the difference between a system that can answer "show me account state on March 14 at 2pm" in one query versus one that can never answer it at all. CQRS lets you serve a write-heavy command side from a normalized store and a read-heavy query side from denormalized projections, scaling them independently.

Real-world examples

  • Stock exchanges: the order book is a sequence of NewOrder/Cancel/Fill events. Replay reconstructs any moment.
  • Git: the canonical event-sourced system. Commits are events; the working tree is a projection.
  • Kafka + ksqlDB / Materialize: topics are the log; materialized views are projections.
  • Banking ledgers: double-entry accounting is event sourcing in formalwear — never overwrite a balance, append a transaction.

The mental model

+-------------+ command +-----------------+ | Client |------------------>| Command Side | +-------------+ | (validate + | ^ | append event) | | query +-------+---------+ | | +-----+-------+ v | Query Side | +------------------+ | (read model|<-----project--- | Event Log | | per view) | | OrderPlaced #1 | +-------------+ | ItemShipped #2 | | RefundIssued #3 | +------------------+

Writes go through a small command handler that validates business rules against a loaded aggregate (replay events for that order), then appends one or more new events. A separate process tails the log and updates read models — a "current orders" table, an Elasticsearch index, a daily-revenue rollup — each optimized for one query.

Snapshots and replay

Replaying every event from genesis is slow once you have millions per aggregate. The fix is snapshots: every N events, persist the current state. To rehydrate, load the latest snapshot and replay events after it. Snapshots are an optimization — the event log remains the source of truth, so you can always rebuild a snapshot when the schema evolves.

What you actually gain

  • Audit for free. The log is the audit trail.
  • Time travel. "What did the cart look like at 9:42?" — replay up to that point.
  • New read models without backfill pain. Want a fraud-detection view? Replay the log into a new projection.
  • Natural integration story. Other services subscribe to the same event stream.

What it costs

  • Eventual consistency on reads. Projections lag behind commands by tens to thousands of ms.
  • Schema evolution is hard. Old events live forever; you need versioning + upcasters.
  • Operational complexity. Two stores (log + projections), more moving parts, more rebuild scripts.
  • Privacy/GDPR friction. Immutable logs and "right to be forgotten" don't naturally agree — solve with crypto-shredding (drop the per-user encryption key).

Common mistakes

  • Using event sourcing for a CRUD domain — the audit value is zero, the complexity is real.
  • Letting projections drift from the log — always make them rebuildable from scratch.
  • Embedding business logic in events ("UserBecameVIP") that ages badly — keep events factual ("UserSpent > 1000").
  • Forgetting idempotent projection consumers — at-least-once delivery applies here too.
  • Treating Kafka as the system of record without retention discipline — set retention to "forever" for sourcing topics.

Try it yourself

Question: Your e-commerce app uses event sourcing for orders. Marketing wants a new "average time from PlaceOrder to Shipped, per region" dashboard going back 2 years. The current read model doesn't have it. How do you build it without writing migration scripts or backfill jobs?
Hint: Spin up a new consumer that starts at the beginning of the order-events topic, projects into a new table, then catches up to head. Switch the dashboard to it. The event log is the backfill.
