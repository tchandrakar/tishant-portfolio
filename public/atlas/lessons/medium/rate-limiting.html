Distributed counters & rate limiting (token / leaky bucket)

Medium rate-limiting token-bucket leaky-bucket

The idea in one paragraph

Rate limiting is a distributed counter problem in disguise. You want to allow up to N requests per window per key (user, IP, API token) across a fleet of stateless servers. The two classical algorithms — token bucket (allows bursts up to bucket size, refills at rate r) and leaky bucket (smooths to a steady drip) — give you different shapes of "fair". Implementing them correctly when ten edge servers see the same user simultaneously is where the interesting engineering lives.

Why it matters

Without rate limits, one buggy client or one motivated attacker can consume your entire capacity. With wrong rate limits — counted only locally per server, or with a sliding window that's actually a fixed window — limits leak by 2x at the edges. Every public API (Stripe, GitHub, Twitter) lives or dies by this; in interviews, "design a rate limiter" is its own canonical question.

Token bucket vs leaky bucket

  • Token bucket: Bucket of capacity B, refilled at r tokens/sec. Each request consumes 1 token; if empty, reject. Allows bursts up to B, then settles to r/sec. Used by AWS, Stripe.
  • Leaky bucket (queue): Requests enter a fixed-size FIFO; processed at rate r. Excess requests overflow and are dropped. Output rate is constant — good when downstream cannot tolerate bursts.
  • Fixed window: Count per minute, reset at boundary. Simple, but a user can do 2N requests across the boundary (last 30s of minute 1 + first 30s of minute 2).
  • Sliding window log: Store every timestamp, count those within the last 60s. Accurate but memory-heavy.
  • Sliding window counter: Weighted blend of current and previous fixed window — cheap and ~1% accurate.

Doing it across N servers

Local counters are wrong: ten servers each allowing 100 rps means the user gets 1000 rps. Real options:

  • Centralized Redis with Lua: Atomic INCR + EXPIRE, or a Lua script implementing token bucket. ~1ms per check, single source of truth, single point of failure if not replicated.
  • Sticky routing: Hash user_id to a specific limiter shard. No coordination, but loses requests when a node dies.
  • Approximate / async: Each edge keeps a local counter and gossips deltas every 100ms. Allows ~10% over-limit but stays up under partition.
  • Token-issuer: A central authority hands out batches of tokens to edges; edges spend locally. Bursts well, recovers from edge crashes by not re-issuing.

Edge cases that bite

  • Clock skew: Sliding windows that compare wall clocks across hosts will mis-count. Use the Redis server clock as the single source.
  • Hot keys: A single celebrity user can pin one Redis shard. Pre-shard popular keys (user_id + bucket_index % 4) and sum.
  • Cost-weighted requests: A search endpoint costs 10x a ping. Charge tokens proportional to cost, not 1.
  • Soft vs hard limits: Reject with 429 + Retry-After header so clients back off; don't just drop the connection.

Common mistakes

  • Using fixed window and calling it "100 rps" when it's actually "200 rps at boundaries".
  • Implementing the limiter in app memory and forgetting it resets on deploy.
  • Not having a kill-switch to disable limits during incidents.
  • Returning 500 instead of 429 — clients don't know to slow down and retry storms make it worse.

Try it yourself

Question: An API at 1M rps globally needs per-API-key limits of 100 rps with 200-token burst. Edge fleet is 200 servers across 5 regions. Design the limiter — describe data flow, where state lives, and what happens when one region partitions.
Hint: A pure central Redis won't survive. Consider a token-issuer with regional caches that fall back to local approximate counting under partition.
