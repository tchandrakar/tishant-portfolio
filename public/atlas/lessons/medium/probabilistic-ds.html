Probabilistic data structures: Bloom, Count-min, HyperLogLog

Medium bloom-filter count-min hyperloglog

The idea in one paragraph

Exact answers cost memory proportional to the data. When the data is billions of items but the question is small ("have I seen this before?", "how many uniques?", "what's the top key?"), probabilistic data structures trade a tiny, bounded error for orders-of-magnitude smaller memory. The three you need to know cold: Bloom filter (set membership, no false negatives), Count-min sketch (frequency estimates, never under-counts), and HyperLogLog (cardinality of huge sets in a few KB).

Why it matters

Real systems are full of "is this URL in our 100M crawl-list?" and "how many unique users hit this endpoint today?" type questions. Storing a hash set of every item works at small scale and explodes at large scale. These structures let a cache, a fraud system, or an analytics pipeline answer the same questions in fixed, predictable memory вҖ” usually a few MB regardless of input size.

Bloom filter вҖ” set membership

A bit array of m bits and k hash functions. Insert: set h1(x), h2(x), вҖҰ, hk(x) bits to 1. Query: if all k bits are 1, "probably present"; if any is 0, "definitely absent". No false negatives, tunable false-positive rate (~1% with ~10 bits per item, k=7).

  • Use it: in front of a slow store to skip lookups that definitely miss (Cassandra does this per SSTable).
  • Don't use it: when you need deletes (use counting Bloom) or exact answers.

Count-min sketch вҖ” frequency

A 2D array of d rows Г— w cols, plus d hash functions (one per row). Increment: for each row i, do table[i][hi(x) % w]++. Query: return the min of the d cells. Always overestimates (collisions add), never underestimates. Width controls accuracy, depth controls confidence.

  • Use it: top-K heavy hitters, streaming frequency, anti-abuse counters where exact is overkill.
  • Don't use it: for tail counts вҖ” small frequencies get drowned in collision noise.

HyperLogLog вҖ” cardinality

Hash each item, look at the leading-zero count of the hash, keep the max per bucket. The intuition: in a uniform random stream, seeing a hash with 20 leading zeros means you've probably seen ~2^20 distinct items. Average across many buckets to reduce variance. ~1.6 KB of state estimates cardinality up to 10^9 with ~2% error.

Picking the right one

  • "Have I seen X?" Bloom.
  • "How often is X?" Count-min.
  • "How many distinct?" HyperLogLog.
  • "Top-K heavy hitters?" Count-min sketch + a small min-heap of candidates.
  • All three are mergeable вҖ” you can sketch per-shard and union for a global answer. That's the property that makes them shine in distributed systems.

Common mistakes

  • Sizing the Bloom filter for the wrong N вҖ” once you exceed expected capacity, false-positive rate climbs fast.
  • Using count-min for things with extreme skew without a top-K heap to extract the actual hot items.
  • Storing PII in a "Bloom filter is one-way" mindset вҖ” small input spaces (phone numbers) are trivially brute-forced.
  • Reporting HLL cardinality without disclosing the Вұ2% вҖ” finance teams will hate you.
