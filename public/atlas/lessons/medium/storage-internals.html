Storage internals: WAL, LSM-tree, B-tree

Medium storage lsm btree wal

The idea in one paragraph

Every durable database is built from three primitives: a write-ahead log (WAL) for crash recovery, a main index for fast lookups, and a page cache in RAM. The two dominant index shapes are B-trees (in-place updates, optimized for reads) and LSM-trees (append-only writes flushed to sorted files, optimized for writes). Knowing which one you're sitting on tells you why your queries are slow.

Why it matters

If your workload is write-heavy and Postgres feels strained, you may want RocksDB or Cassandra (LSM). If you're doing range scans and point lookups on a stable corpus, B-trees crush LSMs. The WAL is non-negotiable for both — it's how durability survives a power cut. Confusing these layers leads to bad index choices, surprise write amplification, and 10x storage bills.

Real-world examples

  • B-tree engines: Postgres, MySQL InnoDB, SQLite, MongoDB WiredTiger, Oracle.
  • LSM engines: RocksDB, LevelDB, Cassandra, ScyllaDB, HBase, BigTable, DynamoDB (under the hood).
  • WAL everywhere: Postgres pg_wal, MySQL redo log, SQLite journal, Kafka segment log, etcd Raft log.

WAL: the universal durability trick

client write | v [ WAL append + fsync ] -----> "durable, ack the client" | v [ apply to in-memory page / memtable ] | v (later, batched) [ flush dirty pages / SSTables to data files ]

The trick: writing one sequential append to the WAL is dramatically faster than random-writing the data file in place. After a crash, replay the WAL from the last checkpoint to redo lost mutations. Group commit (batching many transactions into one fsync) is what gets databases to thousands of TPS on commodity SSDs.

B-tree: balanced, in-place, read-friendly

A B-tree is a wide, shallow tree of fixed-size pages (typically 4–16 KB). Each internal node holds dozens to hundreds of keys; height is log_B(N), so a billion rows reach in 3–4 disk seeks. Updates rewrite the page in place — fast for reads, but every write touches the WAL and the page (write amp ≈ 2x baseline plus index maintenance).

  • Strengths: excellent point reads, range scans, predictable latency, mature MVCC.
  • Weaknesses: random write IO; page splits cause fragmentation; concurrency requires careful latching.

LSM-tree: log-structured, append-only, write-friendly

writes -> [ WAL ] + [ memtable (sorted, in RAM) ] | v (when full) [ L0 SSTable ] [ L0 ] [ L0 ] | compaction v [ L1 SSTables: 10x bigger, non-overlapping ] | v [ L2 ... ]

Writes are buffered in a sorted in-memory structure (memtable) and flushed as immutable sorted files (SSTables). Background compaction merges files, drops tombstones, and pushes data down levels. Reads check memtable, then L0, then deeper levels — a Bloom filter per SSTable skips levels that can't contain the key.

  • Strengths: sequential writes, high ingest throughput, great compression (sorted files dedup well).
  • Weaknesses: read amplification (multiple files), write amplification from compaction (often 10–30x), tail-latency spikes during big merges.

The three amplifications

  • Write amp: bytes written to disk ÷ bytes the user wrote. LSMs trade this for sequential IO; B-trees keep it small but random.
  • Read amp: blocks read per logical lookup. LSMs higher (multiple SSTables); B-trees ≈ tree height.
  • Space amp: bytes on disk ÷ live bytes. LSMs hold tombstones until compaction; B-trees fragment after deletes.

Tuning storage is mostly negotiating among these three.

Common mistakes

  • Picking an LSM for a read-heavy point-lookup workload — Bloom filters help, but B-trees still win.
  • Disabling fsync to "speed things up" — your durability guarantee evaporates on the first power loss.
  • Ignoring compaction tuning on Cassandra/RocksDB — default settings rarely match your write rate.
  • Treating SSDs as having infinite endurance — write amp shortens drive life proportionally.
  • Putting WAL and data files on the same disk — fsync contention destroys throughput.

Try it yourself

Question: You're storing 5 KB user-event records at 200k writes/sec, queried mostly by recent timestamp ranges, occasionally by user_id. Which engine — B-tree or LSM — and why? Where do you put the WAL?
Hint: 200k/sec random writes will melt a B-tree (page splits, random IO). Pick an LSM (RocksDB, Cassandra) keyed by (user_id, timestamp) so recent ranges are co-located in young SSTables. Put the WAL on a separate fast disk to avoid fsync contention with compaction.
